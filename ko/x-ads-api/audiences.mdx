## 맞춤 잠재고객

<div id="overview">
  ### 개요
</div>

파트너는 여러 가지 방법으로 [맞춤 잠재고객(Custom Audiences)](https://business.x.com/en/targeting/tailored-audiences.html)을 생성할 수 있습니다.

* [Audience API (CRM)](#crm)
* [웹](#web)
* [모바일](#mobile)
* [Flexible](#flexible)

유사 맞춤 잠재고객(lookalike custom audiences)은 타기팅 대상에서 제외할 수 없습니다. 또한 동일한 광고 라인 아이템(광고 그룹)에서 맞춤 잠재고객과 맞춤 잠재고객의 유사 잠재고객을 동시에 타기팅할 수 없습니다.

**잠재고객 관리**

잠재고객은 audience 파트너와 Ads API 파트너를 통해 관리할 수 있습니다. 맞춤 잠재고객에 액세스하고 이를 유지 관리할 수 있도록 API에 일련의 엔드포인트를 제공합니다.

맞춤 잠재고객 정보를 위해서는 다음 두 개의 엔드포인트를 제공합니다.

* [GET accounts/:account&#95;id/custom&#95;audiences](/ko/x-ads-api/audiences)
* [GET accounts/:account&#95;id/custom&#95;audiences/:custom&#95;audience&#95;id](/ko/x-ads-api/audiences)

잠재고객을 업로드하고 관리하는 방법에 대한 자세한 내용은 [Audience API 가이드](/ko/x-ads-api/audiences)를 참조하세요.

**처리 시간**

일반적으로 잠재고객 변경 사항은 6~8시간마다 실행되는 배치 단위로 처리됩니다. 잠재고객 변경 사항이 처리되는 동안 업데이트 대상이 되는 기존 잠재고객은 영향을 받지 않습니다. 이 시간 범위 내에서 잠재고객별로 추가에 대한 업데이트는 한 번, 제거에 대한 업데이트도 한 번을 초과하지 않도록 하는 것을 권장합니다.

**타기팅**

잠재고객은 X가 소유·운영하는 클라이언트에서 지난 90일 동안 활동한 사용자 100명 이상과 일치하는 경우에만 타기팅할 수 있습니다. [GET accounts/:account&#95;id/custom&#95;audiences/:custom&#95;audience&#95;id](/ko/x-ads-api/audiences)는 잠재고객이 일치하는 사용자가 너무 적어 타기팅이 불가능한지 여부를 표시합니다.

**Q: 아주 많은 데이터를 전송했는데, 왜 오디언스 규모가 TOO&#95;SMALL로 표시되나요?**

A: 현재 데이터는 실시간으로 오디언스에 추가되고 있지만, 오디언스 규모를 계산하는 데이터 처리 작업은 일정 시간이 지난 후에만 실행됩니다. 몇 시간 정도가 지나면 UI에 올바른 오디언스 규모가 표시됩니다.

**Q: 오디언스 데이터를 모두 전송하고 24시간 이상 기다렸는데도 여전히 해당 오디언스를 타깃팅할 수 없습니다. 다음 단계로 무엇을 해야 하나요?**

A: 다음 사항을 확인해 주세요.

* 전달되는 user ID가 올바르고, 형식에 문제가 없는지 확인합니다.
* 전달되는 오디언스 이름이 올바르며 이전 멤버십 업데이트에서 사용한 이름과 일치하는지 확인합니다.
* POST 명령의 응답을 확인합니다.
* ID Sync 픽셀이 올바르게 구현되었는지, 그리고 ID Sync 프로세스에서 설명한 대로 충분한 수의 사용자가 해당 사이트를 방문하여 사용자를 매핑할 수 있는지 확인합니다. 멤버십 업데이트에서 매핑되지 않은 사용자는 타깃팅 가능한 사용자로 전환되지 않습니다.

위의 사항 모두 올바르고 정상 동작하는 것으로 확인되면, 가능한 한 자세한 정보를 포함하여 X 제품 담당자에게 문의해 주세요(선호되는 정보 예시는 [Guide to Partner Inbounds](/ko/x-ads-api/introduction)를 참고).

**Q: 엔드포인트는 얼마나 자주 호출할 수 있으며, 어떤 알고리즘을 사용해야 하나요?**

A: 전체 오디언스 멤버십을 다시 보내지 말고, 증분 델타(incremental delta) 방식으로만 시스템을 호출할 것을 강력히 권장합니다. 이 시스템은 세계에서 가장 큰 웹사이트 중 일부에 대해서도 증분 데이터 업데이트를 처리할 수 있을 정도의 처리량으로 테스트되었습니다. 오디언스의 최초 업로드는 신중하게 속도 제한(throttling)을 적용해야 하며, 첫 업로드를 완료하는 데 상당한 시간이 소요될 수 있습니다.

**Q: 타깃팅에 사용할 수 있는 오디언스의 최소 규모는 얼마인가요?**

* 타깃팅에 사용할 수 있는 오디언스의 최소 규모는 100명(매칭 이후 기준)입니다. 500명 미만의 사용자가 매칭된 오디언스는 X Ads UI에서 타깃팅용으로 사용할 수 없습니다.

**Q: 오디언스 파일을 처리하는 데 얼마나 걸리나요? 그리고 X 사용자 인터페이스에서 해당 오디언스를 사용할 수 있게 되기까지는 얼마나 걸리나요?**

* 일반적으로 오디언스 파일을 처리하는 데 4~6시간이 걸리지만, 파일 크기에 따라 달라질 수 있습니다. 파일이 처리되면 해당 오디언스는 X Ads UI에서 사용할 수 있게 됩니다.

**Q: 매치율은 어떻게 계산되나요?**

* 매치율 = 최근 90일 활성 X 사용자 수 / 제공된 사용자 수

**Q: 오디언스 파일이 정상적으로 작동하는지 테스트하려면 어떻게 해야 하나요?**

* 테스트 오디언스 파일을 제공하고 광고주 핸들로 “keltonlynn”을 사용하면 됩니다. 그러면 해당 파일이 X UI에 정상적으로 수집되어 로드되는지 저희가 검증할 수 있습니다.

**Q: 파트너 사용자 식별자(`p_user_id`)란 무엇인가요?**

* 이는 귀사가 각 고객을 고유하게 식별하기 위해 사용하는 식별자입니다.

**Q: Standard ID란 무엇인가요?**

* 이메일 주소, 디바이스 ID, X @handle 또는 ID일 수 있습니다.

**Q: HMAC 키는 어떻게 받나요?**

* HMAC 키는 암호화된 이메일을 통해 제공됩니다. [mpp-inquiry@x.com](mailto:mpp-inquiry%x.com)으로 귀하의 공개 PGP 키를 보내 주시면, 먼저 테스트 이메일을 보내 모든 것이 정상적으로 동작하는지 확인합니다. 검증이 완료되면 HMAC 키를 발송합니다.

**Q: 제공된 HMAC 키를 사용해 해싱 프로세스가 정상적으로 동작했는지 어떻게 검증하나요?**

* X는 테스트 파일(샘플 이메일 주소, 디바이스 ID 등 포함)과 그 결과로 생성된 해시 파일을 제공하며, 이를 사용해 귀하의 결과를 비교·검증할 수 있습니다.

**Q: 전체 데이터 매치 파일의 파일 크기에 제한이 있나요?**

* 아니요, 전체 데이터 매치 파일에는 크기 제한이 없습니다.

**Q: 전체 데이터 매치 파일이 처리되는 데 얼마나 걸리나요?**

* 파일이 X에 수신된 후, 파일을 처리하는 데는 대략 1일 정도가 소요됩니다.

이 문서에서는 파일 형식 및 데이터 교환 프로세스를 포함하여 Custom Audiences CRM 파트너를 위한 통합 세부 정보를 설명합니다.

**요약**

Company는 고객을 대신하여 해시된 공통 사용자 식별자(예: 이메일 주소) 또는 파트너 사용자 ID 목록을 X에 제공하여 블라인드 매칭(blind match)을 수행하고, 타게팅을 위한 X 사용자 ID 목록을 생성합니다. 타게팅에 사용할 세그먼트는 ads.x.com 캠페인 설정에서 파일명으로 지정된 광고주의 특정 @handle에 제공됩니다.

Company에서 전달하는 모든 파일은 X가 Company에 부여한 전용 계정을 통해 IronBox(www.golockbox.com)의 보안 패키지를 사용하여 X에 제공됩니다. X는 IronBox에 대한 액세스를 제공합니다. IronBox API에 대한 문서는 [https://secure.goironcloud.com/Docs/Help/](https://secure.goironcloud.com/Docs/Help/)에서 확인할 수 있습니다.

<div id="partner-id-matching-requirements">
  #### 파트너 ID 매칭 요구 사항
</div>

회사가 사용자 추적을 위해 자체 표준 ID 시스템(즉, 이메일 주소, 디바이스 id, X user ID 등과 같은 일반적인 사용자 식별자가 아닌 자체 ID)을 사용하는 경우, 다음 프로세스를 권장합니다.

**1. 전체 데이터 매칭**

초기에는 회사가 X와의 공통 사용자 식별자를 포함한 모든 사용자 레코드의 전체 목록을 하나의 파일로 제공하여, 전체 데이터 매칭을 수행하고 파트너 ID(`p_user_id`)와 X ID(`tw_id`) 간의 매핑 정보를 X가 보관하도록 합니다. 이는 적절한 유지 관리를 위해 정기적으로 2~3개월 주기로 수행됩니다. 매칭이 완료되면, X는 이 파일을 기준으로 산출된 초기 매칭률을 이메일을 통해 회사에 공유합니다.

이 파일의 형식은 다음과 같아야 합니다:

이름 규칙: FullDataMatch.[CompanyName].txt

해싱 알고리즘: HMAC&#95;SHA-256

포맷:

Column 1: 공통 사용자 식별자의 HMAC 해시 값

Column 2: 파트너 사용자 ID(사용자별로는 고유하지만, 파일 내에서는 중복 가능)

컬럼 구분자(CSV): 해시된 공통 사용자 식별자와 파트너 ID는 콤마로 구분

줄 단위로 구분된 값

* 예: 사용자 레코드 A의 파트너 사용자 ID가 1이고, 공통 사용자 식별자가 1, 2, 3인 경우:

|     |     |
| :--- | :--- |
| common user identifier 1 | p&#95;user&#95;1 |
| common user identifier 2 | p&#95;user&#95;1 |
| common user identifier 3 | p&#95;user&#95;1 |

*공통 사용자 식별자에 대한 자세한 내용은 아래의 Hashing Directions 섹션을 참고하세요.

**2. 커스텀 세그먼트 리스트**

회사는 `p_user_id` 형식으로 된 사용자 리스트를 제공하여, X에서 타깃팅용 고객 커스텀 오디언스를 생성합니다.

* 줄 단위로 구분된 값
* `p_user_id`
* * (위 1. 전체 데이터 매칭 섹션에서 제공된 값과 동일합니다. 전체 데이터 매칭에서 제공된 값이 해시된 값이라면, 회사는 오디언스 파일에도 동일한 해시 값을 제공합니다. 전체 데이터 매칭에서 제공된 값이 해시되지 않은 값이라면, 회사는 해시되지 않은 값을 제공합니다.)

<div id="standard-matching-requirements">
  #### 표준 매칭 요구 사항
</div>

회사가 모든 고객 사용자 식별자를 매핑하는 데 표준 ID를 사용하지 않는 경우, 아래 프로세스를 따를 것을 권장합니다.

**Custom Segment Lists**

회사는 고객을 대신하여 해시된 공통 사용자 식별자 목록을 X에 직접 제공하여 커스텀 오디언스를 생성합니다.

이 파일의 형식은 다음과 같아야 합니다.

* 줄바꿈으로 구분된 값
* 해시된 공통 사용자 식별자(예: 이메일 주소)
* 아래에 설명된 파일 명명 규칙을 따를 것
* 아래 ‘해싱 지침(Hashing Directions)’에 설명된 이메일 주소 해싱 지침을 따를 것

<div id="custom-segment-list-file-naming-operations">
  #### Custom Segment List File Naming &amp; Operations
</div>

파일의 처리 방식은 파일 이름에 따라 결정되며, 사용 가능한 작업과 일반적인 파일 명명 규칙은 다음과 같습니다: audiencename&#95;partnername.handle.operation.filetype

* audiencename: Custom Audience의 이름입니다. 이 필드는 ads.x.com 캠페인 설정 UI에서 오디언스를 선택할 때 표시되는 이름입니다. 예: brand&#95;loyalty&#95;card&#95;holders.
* partnername: 광고주를 대신해 데이터를 제공하는 회사의 이름입니다. 예: company&#95;name.
* handle: Custom Audiences에 접근할 수 있는 X 계정(@handle)입니다. 예: @pepsi, @dietpepsi
* operation: new, add, remove, removeall, replace (아래에 자세한 설명 있음)
* : 업로드되는 각 오디언스 파일이 고유하도록 보장하기 위해 사용되는, 초 단위의 표준 Unix epoch 시간.
* filetype: 파일은 *.txt 형식이어야 합니다.

<div id="creating-and-updating-audiences">
  #### 오디언스 생성 및 업데이트
</div>

하나의 파일로 새 오디언스를 생성합니다. 예: loyalty&#95;card&#95;holders&#95;partnername.pepsi.new.txt

Add - 리스트와 매칭되는 대상을 기존 오디언스에 추가합니다. 예: loyalty&#95;card&#95;holders&#95;partnername.pepsi.add..txt

Remove - 리스트와 매칭되는 대상을 기존 오디언스에서 제거합니다. 예: loyalty&#95;card&#95;holders&#95;partnername.pepsi.remove..txt

Remove All - 정기적으로 업데이트되는 누적 리스트에서 매칭된 대상을 해당 클라이언트의 모든 오디언스에서 제거합니다(즉, 클라이언트의 옵트아웃 리스트). 예: partnername.pepsi.removeall.txt

* 광고주로부터 옵트아웃한 사용자의 포괄적인 리스트로 사용할 수 있습니다.
* X는 이 파일에 제공된 최신 리스트만을 적용하며, 이 리스트와 매칭되는 모든 기존 및 향후 오디언스에 대해, 이 파일이 제공되고 처리된 시점의 X 사용자에 대해서만 적용합니다.

Replace - 기존 오디언스를 제거하고 새로운 오디언스 리스트로 교체합니다. 예: loyalty&#95;card&#95;holders&#95;partnername.pepsi.replace..txt

전체 회사 옵트아웃 - 회사는 회사의 옵트아웃 정책에 따라 옵트아웃한 사용자를 제거하기 위한 누적 옵트아웃 파일을 제공합니다.

X는 이 회사 옵트아웃 파일에 제공된 최신 리스트만을 적용하며, 이 리스트와 매칭되는 모든 기존 및 향후 오디언스에 대해, 이 파일이 제공되고 처리된 시점의 X 사용자에 대해서만 적용합니다. 회사 옵트아웃 파일의 형식은 다음과 같습니다. 예: partnername.removeall.txt

Delete - 기존 오디언스를 현재 오디언스 리스트에서 제거합니다. 예: loyalty&#95;card&#95;holders&#95;partnername.pepsi.delete.txt

<div id="hashing-directions">
  #### 해싱 지침
</div>

X는 공통 사용자 식별자(예: 이메일 주소)를 해싱하기 위해 base64로 인코딩된 프로덕션 키를 PGP를 통해 안전하게 공유합니다. 회사는 이 키를 base64 디코딩하여 해싱을 수행하는 데 사용할 32바이트 키를 얻습니다.

base64 인코딩된 키 예시:

BrQvOg+dACBUmKjRiNxZgJLh6zydjS0ZOv80FelTNzM=

base64 디코딩된 키 예시:

/:� TшY

정규화: 회사는 해싱 전에 공통 사용자 식별자에 대해 기본적인 정규화를 수행합니다(단, Device ID는 제외되며 Device ID Normalization 섹션을 참조하십시오).

<div id="e-mail-normalization">
  #### E-mail 정규화
</div>

즉, 앞뒤 공백을 제거하고 이메일 주소를 소문자로 변환합니다.

예: 원본 이메일 주소: testemail&#95;Organisational&#95;baseball+884`@`It92I6Ev2B`.`Com

정규화 후: testemail&#95;organisational&#95;baseball+884`@`it92i6ev2b`.`com

해시값: 74d9584eded0ad1e5572a1c1849f3716751d371d6117a6155dad5363f4b4fbec

참고: 인코딩된 HMAC과 키의 문자 수는 입력값과 인코딩 방식에 따라 달라질 수 있으므로, 문자 수가 특정 값으로 고정되는 것은 아닙니다.

<div id="device-id-normalization">
  #### Device ID 표준화
</div>

데이터 파트너에게 제공하는 공통 salt와 SHA-256 해시 알고리즘을 사용해 Device ID를 해싱하는 데 동일한 요구 사항이 적용됩니다. 이메일 주소와 마찬가지로 공백은 제거하지만, IDFA/Android ID에 대해서는 소문자 표준화를 수행하지 않으며 IDFA/Android ID의 정확한 형식을 그대로 사용해야 합니다.

다음은 해싱 전에 iOS 및 Android용 Device ID의 원본 형식 예시입니다.

iOS IDFA: DD99CFF7-6186-4602-9DF2-ED3FD0B2D431

Android ID: b5bf2122961b3595

해시된 iOS IDFA: 134fb8cd95c7fd42e2793f469a447198ca5f990968db2dbadad70e723ed9750b

해시된 Android ID: 130dddff1939f229476f50bc8adab8fcb7e3525b0e9604fe8effc15e68cee4a4

<div id="x-user-id-normalization">
  #### X 사용자 ID 정규화
</div>

X ID는 여전히 해시됩니다. 데이터의 그룹(예: @handle로 구성된 고객 리스트)은 PII가 아니더라도 광고주에게만 비공개로 유지되기 때문입니다. 데이터 파트너에게 제공하는 공통 salt와 SHA-256 해싱 알고리즘을 사용해 X ID를 해시하는 데 동일한 요구 사항이 적용됩니다. X ID 및 `@`username 모두에서 공백은 제거해야 하지만, User ID는 별도의 정규화가 필요하지 않습니다. @username은 정규화를 위해 소문자로 변환해야 하며, @ 기호는 username의 일부로 포함하면 안 됩니다.

원시 ID 형식은 다음과 같습니다.

* User ID: 27674040
* @username: testusername

해시된 User ID: bf6b57d4e861e83bea8bbed2b800b251a64c95468ee6e8cb07c3368c9ed45e85

해시된 @username: 12201ae78ad1afa907c7112d17f498154ffb0bf9ea523f5390e072a06d7d9812

<div id="id-sync-integration">

</div>

`p_id`로 데이터를 전송하는 파트너는 광고주 또는 파트너의 사용자 id를 X 사용자 id와 매핑하기 위한 ID Sync 프로세스를 거쳐야 합니다. 이를 통해 광고주는 X에서 자신의 사용자 세그먼트를 직접 타기팅할 수 있습니다. 파트너는 멤버십 업데이트를 전송할 때 `user_identifier_type` 파라미터 값을 `TALIST_PARTNER_USER_ID` 또는 `TAWEB_PARTNER_USER_ID`로 설정해야 합니다.

* **웹(Web Only)**: 아래에 설명된 대로 광고주의 사이트에 픽셀을 설치하여 진행할 수 있습니다.

<div id="custom-audiences-user-data">
  ### 맞춤 잠재고객 사용자 데이터
</div>

이 문서는 [Custom Audience]/x-ads-api/audiences 사용자 데이터의 형식을 설명합니다.

**데이터 정규화**

**디바이스 ID**:

* IDFA - 모두 소문자, 하이픈 포함; 예: `4b61639e-47cc-4056-a16a-c8217e029462`
* AdID - 디바이스에 저장된 원래 형식 그대로여야 하며, 대문자로 변환하지 말고 하이픈을 포함해야 합니다; 예: `2f5f5391-3e45-4d02-b645-4575a08f86e`
* Android id - 디바이스에 저장된 원래 형식 그대로여야 하며, 대문자로 변환하지 말고 하이픈 및 공백 없이 작성해야 합니다; 예: `af3802a465767e36`

**이메일 주소**:

* 모두 소문자로, 앞뒤 공백을 제거; 예: `support@x.com`

**X 사용자 이름(X Usernames)**:

* @ 기호 없이, 모두 소문자로, 앞뒤 공백을 제거; 예: `jack`

**X 사용자 ID(X User IDs)**:

* 표준 정수; 예: `143567`

**데이터 해싱**

각 행의 데이터는 salt 없이 `SHA256`을 사용해 해시해야 합니다. 또한 최종 출력 해시는 소문자여야 합니다. 예: 49e0be2aeccfb51a8dee4c945c8a70a9ac500cf6f5cb08112575f74db9b1470d 이고, **다음이 아니어야 합니다:** 49E0BE2AECCFB51A8DEE4C945C8A70A9AC500CF6F5CB08112575F74DB9B1470D

```
# Python을 사용하여 사용자 @AdsAPI 해싱하기
import hashlib
hashlib.sha256("adsapi".encode()).hexdigest()

#출력
49e0be2aeccfb51a8dee4c945c8a70a9ac500cf6f5cb08112575f74db9b1470d
```

해싱용 추가 코드 샘플은 [github.com/xdevplatform/ads-platform-tools](https://github.com/xdevplatform/ads-platform-tools)에서 확인할 수 있습니다.

**정보**

파트너는 광고주를 대신하여 타겟팅할 ID 목록(`p_user_ids`)을 전송합니다. 이는 `p_user_ids`와 X 사용자 ID 간의 매핑을 생성하는 ID 동기화(ID Sync) 프로세스를 통해 수행됩니다. 이렇게 생성된 매핑은 타겟팅에 사용할 수 있는 X 사용자 ID 리스트를 생성하는 데 사용됩니다. 이러한 커스텀 오디언스는 ads.x.com의 Custom Audiences Web 캠페인 설정에서 라벨로 지정된 광고주의 특정 @handle에서 사용할 수 있게 됩니다.

X는 파트너 태그와 사이트에 삽입하여 ID(`p_user_ids`)를 X 사용자 ID와 매칭할 수 있는 보안 픽셀(secure pixel)을 제공합니다. ID 동기화 프로세스가 완료된 후 파트너가 타겟팅 파일을 생성하며, 이 파일은 HTTPS 엔드포인트를 통해 X가 정기적으로 가져올 수 있도록 제공됩니다. 이러한 타겟팅 파일은 X에 의해 정기적으로 수집되며, 이후 X UI에서 사용할 수 있게 됩니다.

**X 보안 픽셀**

X 보안 픽셀은 다음과 같은 형태입니다:

[https://analytics.x.com/i/adsct?p&#95;user&#95;id=xyz&amp;p&#95;id=123](https://analytics.x.com/i/adsct?p\\_user\\_id=xyz\&p_id=123)

**`p_user_id`** - xyz는 파트너가 제공한 파트너 사용자 ID를 나타냅니다.

**`p_id`** - 123은 파트너에 대한 고유 ID(X가 제공)를 나타냅니다.

**파트너 HTTPS 엔드포인트 &amp; 타겟팅 사용자 파일**

파트너는 X가 타겟팅 파일을 정기적으로 가져올 수 있도록 HTTPS 엔드포인트와 자격 증명(사용자 이름/비밀번호)을 제공해야 합니다. 샘플 HTTPS 엔드포인트는 다음과 같은 형태입니다:

```
https://<partnerdomain>/twitter/partner_targeting_%Y-%M-%D.tsv.gz
```

%Y - 연도 형식 코드 (YYYY)

%M - 월 형식 코드 (MM)

%D - 일 형식 코드 (DD)

전송되는 데이터는 다음 파일들로 구성됩니다:

1. Partner Targeting User File
2. Targeting Conversion File

모든 파일은 TSV 형식이며, 각 행의 개별 필드는 탭 문자로 구분됩니다. 유효한 필드 값 자체에는 탭 문자가 포함되지 않습니다.

**허용되는 X IP 범위:**

Partner Endpoint에 대한 접속을 허용할 수 있는 IP 범위는 다음과 같습니다.

* 199.16.156.0/22
* 199.59.148.0/22

| 2                 | advertiser id     | string          | “advertiser id”는 광고주의 @handle입니다.                                                                                                                                                       |

| 4                 | segment label     | string          | “segment label”은 선택 사항입니다. 파트너는 예를 들어 상품 카테고리를 지정하기 위해 “segment label”을 사용할 수 있습니다. ads.x.com UI에서 Custom Audiences에 대한 사람이 읽을 수 있는 이름이므로, 이 “segment label”을 사용할 것을 권장합니다.             |

**참고 사항:**

새로운 Partner Targeting File을 수신할 때마다, 별도의 합의가 없는 한 증분 데이터가 아니라 파트너가 타겟팅을 추천하는 전체 사용자 목록일 것으로 간주합니다. 이 Partner Targeting File의 전달 빈도는 각 파트너와 협의하여 정합니다. 예상한 시점에 Partner Targeting File을 받지 못한 경우, 사전에 정의된 만료 시간까지는 이전 버전을 계속 사용할 것입니다.

<div id="audience-api-integration">
  ## Audience API 연동
</div>

<div id="overview">
  ### 개요
</div>

Audience API는 Ads API의 [v4](/ko/x-ads-api/introduction)의 일부로 출시되었으며, 이를 통해 기존 Audiences 엔드포인트에 여러 개선 사항이 도입되었습니다. 이 새로운 엔드포인트는 새로운 Audience 처리 백엔드에 의해 지원되며, 안정성, 견고성, 신뢰성 측면에서 여러 개선 사항을 제공합니다. 이 가이드의 목적은 Audience API와 기존 Audience 업로드 및 관리 프로세스 간의 차이점을 설명하는 것입니다. 

참고 문서는 [Audience API](/ko/x-ads-api/audiences) 참고 문서 페이지에서 확인할 수 있습니다. 

**참고**: 모든 Audience 사용자 데이터는 업로드 전에 SHA-256으로 해시되어야 합니다. 허용되는 사용자 식별자 유형과 데이터 정규화에 대한 자세한 내용은 [user data](/ko/x-ads-api/audiences) 페이지를 참조하세요.

**Audience 기능 변경 사항**

v4 기준으로 Custom Audience에 다음 변경 사항이 도입되었으며, 사용 중단된 엔드포인트는 Ads API v3가 완전히 중단된 이후 더 이상 사용할 수 없습니다.

* **사용 중단됨(Deprecated)** TON 업로드:
  * GET accounts/:account&#95;id/custom&#95;audience&#95;changes
  * GET accounts/:account&#95;id/custom&#95;audience&#95;changes/:custom&#95;audience&#95;change&#95;id
  * POST accounts/:account&#95;id/custom&#95;audience&#95;changes
  * PUT accounts/:account&#95;id/custom&#95;audiences/global&#95;opt&#95;out
* **사용 중단됨(Deprecated)** Real Time Audiences:
  * POST custom&#95;audience&#95;memberships
* Custom Audience:
  * `list_type` 파라미터는 모든 [Custom Audience](/ko/x-ads-api/audiences) 엔드포인트의 요청 및 응답에서 제거됩니다. 이 파라미터는 이전에 Audience의 사용자 식별자 유형(예: 이메일, X User ID 등)을 식별하는 데 사용되었으나, 이제 Audience는 동일한 Audience에 대해 여러 사용자 식별자를 수용할 수 있으므로 이 값은 더 이상 의미가 없습니다.
* 일반:
  * Audience 조회(lookback) 기간은 최근 30일에서 최근 90일 내에 활동한 사용자를 대상으로 일치 여부를 확인하도록 업데이트되었습니다.
  * 타기팅 가능하도록 요구되는 Audience의 최소 매칭 사용자 수는 500명에서 100명으로 감소했습니다.

<Note>
  **필수 조건**

  * Ads API 액세스
  * Audience 엔드포인트에 대한 액세스를 위해서는 허용 목록(allowlist)에 추가되어야 합니다. 2018-08-01 이전에 처음 승인된 경우, 이 양식을 작성하고 새로운 [X Ads Products and Services Agreement](/ko/x-ads-api/introduction)에 동의해야 합니다.
</Note>

**Audience 업로드 프로세스**

다음 표는 이전과 새로운 Audience 생성 플로 간의 주요 차이점을 나열한 것이며, 아래에서 더 자세한 내용을 확인할 수 있습니다.

| 프로세스 단계 | Audience API | (사용 중단됨) TON 업로드 |
| :--- | :--- | :--- |
| 셸 Audience 생성 | [POST custom&#95;audience endpoint]/x-ads-api/audiences 를 통해 생성 가능 | [POST custom&#95;audience endpoint]/x-ads-api/audiences 를 통해 생성 가능 |
| 새 사용자 추가 | [Audience endpoint](/ko/x-ads-api/audiences)에서 `operation_type` `Update` 사용 | [POST custom&#95;audience&#95;changes](/ko/x-ads-api/audiences) 엔드포인트에서 `operation` `ADD` 사용 |
| 사용자 제거 | [Audience endpoint](/ko/x-ads-api/audiences)에서 `operation_type` `Delete` 사용 | [POST custom&#95;audience&#95;changes](/ko/x-ads-api/audiences) 엔드포인트에서 `operation` `REMOVE` 사용 |
| 사용자 옵트아웃 처리 | 사용자가 속한 해당 `custom_audience_id`들과 함께 [Audience endpoint](/ko/x-ads-api/audiences)에서 `operation_type` `Delete` 사용 | [Global opt-out endpoint](/ko/x-ads-api/audiences) 사용 |

**참고** TON 업로드 경로를 통해 업데이트되거나 옵트아웃되는 모든 Audience는 [TON Upload](/ko/x-ads-api/audiences) 엔드포인트를 통해 업로드된 해당 리스트가 있어야 하며, [custom&#95;audience&#95;changes](/ko/x-ads-api/audiences) 엔드포인트를 사용해 Audience와 연계되어야 합니다.

**Rate Limiting**

Audience API 엔드포인트에는 계정당 1분에 1500회라는 요청 한도(rate limit)가 적용됩니다. 단일 페이로드에 포함할 수 있는 사용자 수에는 제한이 없습니다. 페이로드에 대한 제약은 다음과 같습니다.

1. 전체 작업 수: 2500 operations

2. 최대 페이로드 크기: 5,000,000 바이트

<div id="create-a-new-custom-audience">

[POST custom&#95;audience]/x-ads-api/audiences 엔드포인트를 사용해 새로운 맞춤 잠재고객 &quot;셸(shell)&quot;을 생성하고, 해당 맞춤 잠재고객의 `id`를 가져옵니다. 이 단계는 잠재고객을 처음부터 생성하는 경우에 필수입니다. 기존 잠재고객을 업데이트하는 경우에는 다음 섹션으로 건너뜁니다.

<div id="add-users-to-an-audience">

Custom Audience `id`와 다음과 같은 샘플 페이로드를 사용하여 [POST accounts/:account&#95;id/custom&#95;audiences/:custom&#95;audience&#95;id/users](/ko/x-ads-api/audiences) 엔드포인트에 POST 요청을 보냅니다:

          "users": [
            {
              "email": [
                "abc@x.com"
              ],
              "handle": [
                "x",
                "adsapi"
              ]
            },

사용자를 Audience에 추가하려면 `operation_type`로 `Update`를 사용해야 합니다. 새로운 Audience 인터페이스를 사용하면 하나의 사용자에 대해 여러 개의 user key를 전달할 수 있습니다. JSON 객체 배열의 각 객체는 하나의 사용자에 해당합니다. 위의 예시 페이로드를 사용할 경우, 요청은 두 명의 사용자를 Audience에 추가하며, 한 명은 `email`과 `handle`을, 다른 한 명은 `email`과 `twitter_id`를 갖습니다.

<div id="remove-users-from-an-audience">
  #### 오디언스에서 사용자 제거
</div>

사용자를 추가하는 과정과 마찬가지로, 아래와 같이 호출하여 오디언스에서 사용자를 제거할 수 있습니다:

POST [https://ads-api.x.com/11/accounts/18ce54d4x5t/custom&#95;audiences/1nmth/users](https://ads-api.x.com/11/accounts/18ce54d4x5t/custom_audiences/1nmth/users)

```
    # 모든 값은 해시 처리되어야 하며, 이 예제에서는 설명 목적으로 해시되지 않은 값을 사용합니다
    [
      {
        "operation_type": "Delete",
        "params": {
          "effective_at": "2018-05-15T00:00:00Z",
          "expires_at": "2019-01-01T07:00:00Z",
          "users": [
            {
              "email": [
                "abc@x.com"
              ],
              "twitter_id": [
                "783214",
                "1225933934"
              ]
            },
            {
              "email": [
                "edf@x.com"
              ],
              "twitter_id": [
                "121291606",
                "17874544"
              ]
            }
          ]
        }
      }
    ]
```

`operation_type`는 `Delete`로 설정해야 하며, 사용자는 해당 오디언스에 사용자를 추가할 때 사용되었던 키 중 어떤 것으로든 매칭됩니다. 예를 들어, 한 사용자를 `email`과 `twitter_id`를 사용해 오디언스에 추가했다면, 동일한 사용자는 이 키들 중 어느 하나를 사용해 제거할 수 있습니다. 즉, `email` 또는 `twitter_id`만 사용하거나, 둘 다 사용할 수 있습니다.

또한, 하나의 요청 안에서 오디언스에 사용자를 추가하고 제거하는 것도 가능합니다. 이 엔드포인트는 요청당 여러 개의 `operation_type` 값을 지원합니다.

<div id="opt-out-users">
  #### 옵트아웃 사용자
</div>

글로벌 옵트아웃 엔드포인트가 사용 중단됨에 따라, 파트너는 어떤 Audience에서든 옵트아웃한 사용자를 `Delete`해야 합니다. 이를 수행하는 방법은 다음과 같습니다:

1. 어떤 사용자가 어떤 Audience에 속해 있는지를 추적한 후, 각 Audience에서 해당 사용자를 개별적으로 제거합니다.
2. Ads 계정과 연결된 **모든** Audience에서 해당 사용자를 제거합니다.

**일반 모범 사례**

* 큐가 들쭉날쭉해져 처리 시간이 길어지고 시스템에 불필요한 부하를 주는 것을 방지하기 위해, 이 엔드포인트는 근실시간 배치로 호출할 것을 강력히 권장합니다. 이렇게 하면 사용자를 캠페인 타기팅에 더 빨리 활용할 수 있습니다.
* 성공적인 API 호출은 요청에서 수신된 `user` 객체 수에 해당하는 `success_count` 및 `total_count`를 반환합니다.
* 이 엔드포인트는 원자적 특성을 가지므로, 전체 요청이 모두 성공하거나, 오류가 발생하는 경우 전체 요청이 실패합니다. 오류 응답을 받은 경우, API 사용자는 오류를 수정한 후 전체 payload로 다시 요청을 재시도할 것을 권장합니다. 
* 실패하는 경우에는 재시도 시 [exponential backoff](https://en.wikipedia.org/wiki/Exponential_backoff) 방식을 사용할 것을 권장합니다. 예를 들어, 첫 번째 실패 시에는 즉시 재시도하고, 두 번째 실패 후에는 1분 뒤에 재시도하며, 세 번째 연속 실패 후에는 5분 뒤에 재시도하는 식으로 진행합니다.

  #### POST accounts/:account&#95;id/custom&#95;audiences/:custom&#95;audience&#95;id/users[](#post-accounts-account-id-custom-audiences-custom-audience-id-users "Permalink to this headline")

이 엔드포인트를 사용하면 파트너는 지정된 `custom_audience_id`에서 사용자를 추가, 업데이트 및 제거할 수 있습니다. 또한 이 엔드포인트는 하나의 사용자에 대해 여러 사용자 식별자 유형을 함께 제공하는 것도 허용합니다.

요청의 `users` 필드에 제공되는 모든 데이터는 `partner_user_id`를 **제외하고** `SHA256`을 사용해 해시되어야 하며, [정규화](/ko/x-ads-api/audiences#twitter-user-id-normalization)되어야 합니다.

**배치 요청**

* 현재 이 엔드포인트에 대한 최대 배치 크기는 `2500`입니다. 배치 크기는 요청당 작업(`Update`/`Delete`) 개수로 결정됩니다. 예를 들어, 하나의 배열에 2500개를 초과하는 작업 객체(`{"operation_type": "Update/Delete", [..] }`)가 포함되면 오류가 발생합니다.
* 이 엔드포인트가 허용하는 요청 POST 본문의 최대 크기는 `5,000,000`바이트입니다.
* 이 엔드포인트에 대한 요청 한도는 1분 윈도우당 1500건입니다.
* 모든 매개변수는 요청 본문으로 전송되며, `Content-Type`은 `application/json`이어야 합니다.
* 배치 요청은 그룹 단위로 함께 실패하거나 성공하며, 오류와 성공에 대한 모든 API 응답은 초기 요청의 항목 순서를 그대로 유지합니다.

**배치 응답**

Ads API가 반환하는 응답에는 `success_count`와 `total_count` 두 개의 필드가 포함됩니다. 이 값들은 항상 동일해야 하며, 백엔드에서 처리된, 요청에 포함된 레코드 개수를 나타냅니다. 요청 본문에 전송된 레코드 수가 `success_count` 및 `total_count`와 **동일하지 않은** 상황은 오류 상태로 간주해야 하며, 재시도가 필요합니다.

**배치 오류**

* 요청 수준 오류(예: 최대 배치 크기 초과)는 응답의 `errors` 객체 아래에 표시됩니다.
* 항목 수준 오류(예: 필수 매개변수 누락)는 응답의 `operation_errors` 객체 아래에 표시됩니다.
* `operation_errors` 내 오류의 인덱스는 입력 항목의 인덱스를 나타내며, 해당 오류 메시지가 함께 제공됩니다.

**참고:** 이 배치 엔드포인트는 현재 **비공개 베타** 상태로, 일부 광고주만 사용할 수 있습니다. 이 베타 기간 동안에는 모바일 Custom Audiences를 기반으로 한 Flexible Audiences만 생성할 수 있습니다.

**배치 요청**

* 현재 최대 배치 크기는 10입니다.
* 모든 매개변수는 요청 본문에 포함되며, `Content-Type`은 `application/json`이어야 합니다.
* 배치 요청은 하나의 그룹으로 함께 실패하거나 성공하며, 오류 및 성공에 대한 모든 API 응답에서 초기 요청의 항목 순서가 그대로 유지됩니다.

**배치 응답**

Batch API 응답은 순서가 보장된 항목 컬렉션을 반환합니다. 그 외 구조는 해당 단일 항목 엔드포인트와 동일합니다.

**배치 오류**

* 요청 수준 오류(예: 최대 배치 크기 초과)는 응답의 `errors` 객체에 표시됩니다.
* 항목 수준 오류(예: 필수 매개변수 누락)는 응답의 `operation_errors` 객체에 표시됩니다.

<div id="post-batchaccountsaccount_iddo_not_reach_listsdo_not_reach_list_idusers">
  #### POST batch/accounts/:account&#95;id/do&#95;not&#95;reach&#95;lists/:do&#95;not&#95;reach&#95;list&#95;id/users[](#post-batch-accounts-account-id-do-not-reach-lists-do-not-reach-list-id-users "Permalink to this headline")

이 엔드포인트를 사용하면 지정된 `do_not_reach_list_id`에 사용자를 추가, 업데이트 및 제거할 수 있습니다. 이 엔드포인트는 이메일만 유효한 사용자 식별자 유형으로 허용합니다.

요청의 `emails` 필드로 제공되는 모든 데이터는 `SHA256`으로 해시 처리하고, [정규화](/ko/x-ads-api/audiences#e-mail-normalization)해야 합니다.

**Notes**

* 하나의 `account_id`는 최대 하나의 Do Not Reach List만 가질 수 있습니다.
* 이 리스트에 추가되는 사용자는 현재 타임스탬프 기준 13개월 미만으로 설정된 `expires_at` 타임스탬프를 **반드시** 가져야 합니다.
* Do Not Reach List API는 `effective_at` 타임스탬프를 허용하지 않으며, 기본값으로 현재 타임스탬프를 사용합니다.
* Do Not Reach List는 계정의 어떤(또는 모든) 커스텀 오디언스에서 사용자를 제거하지 않으며, 계정에서 송출되는 모든 캠페인에 대한 제외 타깃팅으로 동작합니다.

**Batch Requests**

* 이 엔드포인트에 대한 현재 최대 배치 크기는 `2500`입니다. 배치 크기는 요청당 작업(`Update`/`Delete`) 개수로 결정됩니다. 예를 들어, 하나의 배열에 2500개가 넘는 작업 오브젝트(`{"operation_type": "Update/Delete", [..] }`)가 포함되면 오류가 발생합니다.
* 이 엔드포인트가 수락할 수 있는 최대 요청 POST 본문 크기는 `5,000,000`바이트입니다.
* 이 엔드포인트에 대한 요청 한도는 1분 윈도우당 1500건입니다.
* 모든 파라미터는 요청 본문에서 전송되며, `Content-Type`은 `application/json`이어야 합니다.
* 배치 요청은 그룹 전체가 함께 실패하거나 함께 성공하며, 오류와 성공에 대한 모든 API 응답은 초기 요청의 항목 순서를 그대로 보존합니다.

**Batch Responses**

Ads API가 반환하는 응답에는 `success_count`와 `total_count`라는 두 개의 필드가 포함됩니다. 이 값들은 항상 동일해야 하며, 백엔드에서 처리된 요청 내 레코드 개수를 나타냅니다. 요청 본문에 전송된 레코드 개수가 `success_count` 및 `total_count`와 같지 **않은** 경우는 재시도가 필요한 오류 상태로 간주해야 합니다.

**Batch Errors**

* 요청 수준 오류(예: 최대 배치 크기 초과)는 응답의 `errors` 오브젝트 아래에 표시됩니다.
* 항목 수준 오류(예: 필수 파라미터 누락)는 응답의 `operation_errors` 오브젝트 아래에 표시됩니다.
* `operation_errors` 내 오류의 인덱스는 입력 항목의 인덱스를 가리키며, 해당 오류 메시지가 함께 제공됩니다.
