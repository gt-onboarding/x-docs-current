## 소개

애널리틱스 메트릭은 파트너와 광고주가 X에서 프로모션하는 콘텐츠의 성과를 이해하는 데 도움을 줍니다. 여기에는 노출 수(impressions), 클릭 수, 동영상 조회 수(video views), 집행 비용(spend)과 같은 정보가 포함됩니다. 또한 파트너와 광고주는 도달한 오디언스를 다양한 세그먼트로 나누어 세부 메트릭을 조회할 수 있습니다.

Ads API는 캠페인 성과 메트릭을 상세하게 조회하는 두 가지 방식을 지원합니다. 동기(synchronous) 방식과 비동기(asynchronous) 방식입니다. 동기 애널리틱스 호출에서는 요청한 메트릭이 응답 본문에 바로 반환됩니다. 비동기 애널리틱스 엔드포인트에서는 관련 &#39;job&#39; 처리가 완료된 후, 요청한 메트릭이 다운로드 가능한 결과 파일 형태로 제공됩니다. 동기 엔드포인트는 짧은 기간만 조회를 지원하며 실시간 캠페인 최적화에 적합합니다. 비동기 엔드포인트는 훨씬 더 긴 기간을 지원해 훨씬 더 많은 데이터를 가져오는 데 사용되며, 리포트 생성이나 과거 데이터 백필(historical backfill)에 매우 적합합니다.

  ## 자세한 내용
</div>

<div id="synchronous-vs-asynchronous">
  ### 동기식 vs. 비동기식
</div>

동기식 및 비동기식 analytics 엔드포인트 간의 차이점은 아래 표에 요약되어 있습니다. 이 정보는 어떤 엔드포인트 집합을 사용할지 개발자가 결정하는 데 도움이 되도록 제공됩니다.

| Feature | Synchronous | Asynchronous |
|:---------|:------------|:--------------|
| Rate limiting | 사용자 수준: 15분당 요청 250개 | 계정 수준: 동시에 처리 중인 작업* 최대 100개 |
| Time range | 7일 | 90일 (비세분화)<br />45일 (세분화) |
| Segmentation | 없음 | 있음 |
| Response returns | 메트릭 데이터 | 작업의 처리 상태** |
| Recommended use case | 실시간 최적화<br />사용자 인터페이스 요청 | 정기적인 동기화<br />과거 데이터 백필(보충) |

<Note>
  * 이는 어느 시점에서든 처리 상태에 있을 수 있는 작업의 최대 개수를 의미합니다.

  ** 작업 처리가 성공적으로 완료되면 URL이 반환됩니다. 이 URL에서 압축된(gzip) 결과 파일을 다운로드할 수 있습니다.

  이 밖에는 두 엔드포인트가 동일한 기능을 제공합니다.
</Note>

<div id="use-cases">
  ### 사용 사례
</div>

주요 분석 사용 사례는 세 가지입니다.

1. 실시간 최적화: 성과 메트릭을 사용해 진행 중인 캠페인을 업데이트
2. 동기화: 정기적으로 실행되는 백그라운드 동기화
3. 신규 계정 온보딩: 과거 데이터 백필(backfill)

동기식 분석 endpoint는 최근 5~15분 동안의 메트릭 변화를 기반으로 캠페인을 업데이트하는 실시간 최적화에 사용할 수 있습니다. 분석 동기화에는 어느 endpoint든 사용할 수 있습니다. 원하는 시간 범위와 세분화(segmentation) 필요 여부에 따라 어떤 endpoint를 사용할지가 결정됩니다. 신규 계정 온보딩은 비동기식 분석 endpoint만 사용해야 합니다. (동기식 분석 endpoint는 대량의 데이터를 조회하는 데 절대 사용해서는 안 됩니다.)

비동기식 분석 endpoint는 메트릭을 백엔드 프로세스와 동기화하는 경우 대시보드와 기타 UI 요소를 구동하는 데 사용할 수 있습니다. 구현 시 사용자 인터페이스 요청을 처리하기 위해 비동기식 분석 endpoint를 호출하는 방식은 피해야 합니다.

<div id="request-options">
  ### 요청 옵션
</div>

애널리틱스 요청은 광고 계정 단위로 한정되며, 따라서 리소스 경로에 계정 ID가 필요합니다. 아래에 나열된 요청 옵션은 쿼리 매개변수로 지정됩니다. 다음 유형의 값이 필요합니다.

* 엔티티: 애널리틱스를 조회하려는 엔티티 유형과 최대 20개의 엔티티 ID
* 시간 범위: ISO 8601 형식으로 표현된 시작 및 종료 시간
  * **주의:** 반드시 정시(시간 단위)로 표현해야 합니다
* 메트릭 그룹: 하나 이상의 관련 메트릭 집합(각 메트릭 그룹에 포함된 메트릭 목록은 Metrics and Segmentation을 참조)
* 집계 단위(Granularity): 메트릭이 어떤 집계 수준으로 반환될지 지정
* 게재 위치(Placement): 메트릭을 X 온사이트 또는 오프사이트에서 집행된 광고 중 어디에서 집계할지 결정
  * **주의:** 요청당 하나의 placement 값만 지정할 수 있습니다

`start_time` 및 `end_time` 요청 매개변수를 사용하여 시간 범위를 지정합니다. 이 값들은 다음과 같이 지정된 집계 단위와 일치해야 합니다.

1. `TOTAL`: 엔드포인트의 제한 내에서 임의의 시간 범위를 지정
2. `DAY`: 시작 시간과 종료 시간 값 모두 계정의 시간대 기준 자정에 맞춰야 함
3. `HOUR`: 엔드포인트의 제한 내에서 임의의 시간 범위를 지정

종료 시간은 포함되지 않습니다. 예를 들어, `start_time=2019-01-01T00:00:00Z` 및 `end_time=2019-01-02T00:00:00Z`로 요청하면 이 시간 범위가 24시간만 포함하므로 하루치 애널리틱스 메트릭만(이틀치가 아니라) 반환됩니다.

**세그멘테이션**

비동기 애널리틱스 엔드포인트에서만 제공되는 세그멘테이션은 파트너와 광고주가 특정 타게팅 값별로 분리된 메트릭을 조회할 수 있도록 해줍니다. 세그멘테이션된 메트릭을 요청하려면 `segmentation_type` 요청 매개변수를 사용하십시오. 세그멘테이션 옵션에 대한 자세한 내용은 [Metrics and Segmentation](/ko/x-ads-api/analytics#metrics-and-segmentation)을 참조하십시오.

<div id="faqs">
  ## 자주 묻는 질문(FAQs)
</div>

왜 Ads API 숫자가 X Ads UI에 표시되는 값과 일치하지 않나요?

* `ALL_ON_TWITTER`, `PUBLISHER_NETWORK`, `SPOTLIGHT`, `TREND` 모든 게재 위치에 대한 데이터를 요청했는지 확인하세요.
* Ads API에서 종료 시간은 제외(end-exclusive)되지만, Ads UI에서는 포함(end-inclusive)된다는 점을 기억하세요.

왜 데이터를 요청하는 시점에 따라 숫자가 달라지나요?

* 리포팅 메트릭이 사용 가능해지는 즉시 해당 값을 조회할 수 있으며, 거의 실시간에 가깝게 제공됩니다. 다만 이러한 초기 결과는 추정치이기 때문에 변경될 수 있습니다. 지출(spend) 데이터를 제외한 메트릭은 24시간 후에 확정됩니다.
* 지출 메트릭은 일반적으로 이벤트 발생 후 3일 이내에 확정됩니다. 다만 스팸 필터링 등의 이유로, 청구 데이터는 이벤트 발생일로부터 최대 14일 동안 처리될 수 있습니다.

특정 기간에 대해 어떤 엔티티 ID를 요청해야 하는지 어떻게 알 수 있나요?

* [Active Entities endpoint](/ko/x-ads-api/analytics#active-entities-2)를 사용하세요.

왜 analytics 응답의 모든 값이 `null`인가요?

* 요청한 기간 동안 캠페인이 게재되지 않았을 가능성이 높습니다.
* 어떤 엔티티에 대해 어떤 기간의 analytics를 조회해야 하는지 확인하려면 [Active Entities endpoint](/ko/x-ads-api/analytics#active-entities-2)를 사용하세요.

왜 API에서는 `null` 값을 보여주는데 UI에서는 0으로 표시되나요?

* UI에서는 이러한 값을 0으로 표시하도록 선택했을 뿐이며, 두 값은 동등합니다.

X 타임라인처럼 세부적인 게재 위치와 관련된 메트릭은 어떻게 요청할 수 있나요?

* analytics에서는 다음과 같은 placement 값을 지원합니다: `ALL_ON_TWITTER`, `PUBLISHER_NETWORK`, `SPOTLIGHT`, `TREND` (즉, [X Audience Platform](/ko/x-ads-api/audiences)).

삭제되었거나 일시 중지된 엔티티에 대한 메트릭도 조회할 수 있나요?

* 예. 엔티티의 상태는 analytics 메트릭의 가용성에 영향을 미치지 않습니다.

세분화된 값이 비세분화 값과 일치하지 않는 이유는 무엇인가요?

* 이 정보가 산출되는 방식 때문에, 세분화된 데이터가 비세분화 데이터와 100% 합산(roll-up)될 것으로 기대해서는 *안* 됩니다.

여러 차원으로 세분화된 데이터를 요청할 수 있나요?

* 다중 세분화는 지원하지 않습니다.

<div id="best-practices">
  ## 모범 사례
</div>

Ads API에서 [analytics](/ko/x-ads-api/analytics) 데이터를 수집할 때 따라야 할 모범 사례는 다음과 같습니다.

<div id="rate-limiting-and-retries">
  ### Rate Limiting 및 재시도
</div>

* rate limit이 적용된 쿼리(`HTTP 429` 상태 코드를 반환하는 경우)의 경우, `x-rate-limit-reset` 헤더를 확인하고 명시된 시점에 도달했을 때 또는 그 이후에만 재시도해야 합니다.
* HTTP 503 Service Unavailable 상태 코드를 반환하는 쿼리의 경우, `retry-after` 헤더를 확인하고 명시된 시간이 지난 뒤에만 재시도해야 합니다.
* 재시도에 대해 명시된 시간을 준수하지 않는 애플리케이션은 사전 고지 없이 Ads API에 대한 액세스가 취소되거나 제한(throttling)될 수 있습니다.

<div id="analytics-metrics-in-a-nutshell">
  ### 애널리틱스 메트릭 한눈에 보기
</div>

* `billed_charge_local_micro`를 제외한 모든 애널리틱스 메트릭은 24시간이 지나면 고정되며 더 이상 변경되지 않습니다.
* `billed_charge_local_micro` 메트릭은 데이터가 반환된 후 최대 3일 동안은 추정값입니다.
* 24시간이 지난 후에는 초과 지출(지정된 `end_time` 이후에 집행된 광고) 및 정크로 판정된 청구 가능 이벤트에 대한 크레딧으로 인해 이 메트릭이 감소할 수 있습니다. 이 메트릭은 24시간 이후에는 거의 변동이 없습니다.
* 자세한 내용은 [Analytics](/ko/x-ads-api/analytics)를 참조하세요.

<div id="fetching-real-time-non-segmented-data">
  ### 실시간, 비세그먼트 데이터 가져오기
</div>

* 항상 `start_time`과 `end_time`을 모두 제공하십시오.
* 7일보다 오래된 엔터티에 대한 데이터는 가져오지 마십시오.
* 가능한 경우 `HOUR` 단위로 데이터를 요청하십시오. 이후 언제든 메트릭을 집계하여 `DAY` 및 `TOTAL` 단위로 변환할 수 있습니다.
* 가능한 경우 `line_items` 및 `promoted_tweets` 레벨에서 데이터를 요청하십시오. 이후 언제든 이러한 메트릭을 집계하여 전체 광고 엔터티 계층(예: 캠페인, funding instrument 또는 계정 레벨)에 대한 전체 합계를 계산할 수 있습니다.
* 분석 메트릭 값을 귀사 측(로컬)에 저장해 두십시오.
* 30일보다 오래된 데이터에 대해서는 반복해서 쿼리하지 마십시오. 이 데이터는 변경되지 않으며 로컬에 저장해야 합니다.
* 모든 비세그먼트 데이터는 실시간이며, 이벤트가 발생한 후 수 초 이내에 데이터를 사용할 수 있어야 합니다.
* 전환 메트릭과 비전환 메트릭을 별도의 요청으로 그룹화하십시오.

<div id="fetching-segmented-data">
  ### 세그먼트 데이터 가져오기
</div>

* 위의 “실시간 비세그먼트 데이터 가져오기” 가이드를 참조하십시오. 추가 권장 사항은 아래를 참고하세요.
* 대부분의 세그먼트 데이터 유형의 경우, 상황에 따라 최대 1시간 동안 데이터가 완전하지 않을 수 있습니다. `INTERESTS`로 세분화된 데이터는 최대 12시간까지 지연될 수 있습니다.
* 이 정보가 산출되는 방식으로 인해, 세그먼트 데이터가 비세그먼트 데이터와 100%까지 합산되어 정확히 일치할 것으로 기대해서는 안 됩니다.

<div id="fetching-historical-data">
  ### 과거 데이터 가져오기
</div>

* 데이터 백필(예: 새 광고주 계정 추가) 시, 더 작은 `start_time` 및 `end_time` 구간으로 나누어 여러 번 요청해야 할 수 있습니다.
* 가져오는 범위를 최대 30일 단위의 날짜 구간으로 제한하세요.
* 이러한 요청의 속도를 조절하고 시간을 분산해, 이 가져오기 작업을 위한 요청 한도를 소진하지 않도록 하세요.

<div id="metrics-by-objective">
  ## 목표별 메트릭
</div>

엔티티에 어떤 메트릭이 적용되는지는 [캠페인 목표](/ko/x-ads-api/campaign-management)에 따라 달라집니다. 이 가이드를 참고하여 각 목표 유형에 대해 조회해야 할 관련 메트릭 그룹을 파악하고, 추가 파생 메트릭을 계산하는 방법을 확인하세요.

<div id="engagements">
  ### `ENGAGEMENTS`
</div>

**관련 메트릭 그룹:** `ENGAGEMENT` 및 `BILLING`. 크리에이티브에 미디어가 사용된 경우 `MEDIA`도 적용됩니다.

|     |     |
| :--- | :--- |

<div id="active-entities">
  ### 활성 엔티티
</div>

<div id="introduction">
  #### 소개
</div>

[Active Entities 엔드포인트](/ko/x-ads-api/analytics#get-stats-accounts-account-id-active-entities)는 어떤 캠페인에 대해 애널리틱스를 요청해야 하는지에 대한 정보를 제공하므로, [동기식](/ko/x-ads-api/analytics#get-stats-accounts-account-id) 및 [비동기식](/ko/x-ads-api/analytics#asynchronous-analytics) 애널리틱스 엔드포인트와 함께 사용하도록 설계되었습니다. 이 엔드포인트는 광고 엔티티에 대한 세부 정보와 해당 메트릭이 언제 변경되었는지를 반환함으로써 이를 구현합니다. 이 엔드포인트를 사용하면 코드와 애널리틱스 조회 로직을 크게 단순화할 수 있습니다.

이 가이드에서는 이 엔드포인트와 그 데이터 소스에 대한 정보와 배경 설명을 제공합니다. 또한 [사용 가이드라인](#usage)과 일련의 [예제 요청](#example)을 통해 Active Entities를 애널리틱스 엔드포인트와 함께 사용하는 방법을 보여줍니다. [요약 섹션](#summary)에서는 권장 접근 방식에 대한 개괄적인 설명을 제공합니다.

<div id="data">
  #### 데이터
</div>

Ads 엔티티의 메트릭이 변경될 때마다 해당 변경에 대한 정보를 기록합니다. 이러한 변경 이벤트는 시간 단위 버킷에 저장되며, 엔티티에 대한 세부 정보와 함께 해당 변경이 적용되는 시점에 대한 정보도 포함합니다. 변경 이벤트가 항상 기록된 시점과 일치하지는 않기 때문에 후자의 정보가 필요합니다. 청구 조정이 그 대표적인 이유이지만, 이 외의 경우도 있습니다.

<div id="endpoint">
  #### 엔드포인트
</div>

<div id="request">
  ### Request
</div>

Active Entities 요청은 Ads 계정을 기준으로 범위가 지정되며, `entity`, `start_time`, `end_time` 세 가지 쿼리 매개변수가 모두 필수입니다.

`twurl -H ads-api.x.com "/11/stats/accounts/18ce54d4x5t/active_entities?entity=PROMOTED_TWEET&start_time=2019-03-05T00:00:00Z&end_time=2019-03-06T00:00:00Z"`

지원되는 `entity` 값은 다음과 같습니다: `CAMPAIGN`, `FUNDING_INSTRUMENT`, `LINE_ITEM`, `MEDIA_CREATIVE`, `PROMOTED_ACCOUNT`, `PROMOTED_TWEET`. 이는 당사 애널리틱스 엔드포인트에서 지원하는 엔티티 유형을 반영합니다.

`start_time` 및 `end_time` 값은 [ISO 8601](https://en.wikipedia.org/wiki/ISO_8601) 형식으로 표현되어야 하며, 조회할 시간별 버킷을 지정합니다. 이 값은 반드시 정시 단위로만 표현해야 합니다.

이 엔드포인트는 결과를 필터링하는 데 사용할 수 있는 선택적 매개변수 세 가지도 지원합니다: `funding_instrument_ids`, `campaign_ids`, `line_item_ids`. 이는 Ads 계층 구조의 모든 수준에서, 지정된 모든 `entity` 유형과 함께 사용할 수 있습니다.

<div id="response">
  ### 응답
</div>

위 요청에 대한 Active Entities 응답은 다음과 같습니다.

```json
    {
      "request": {
        "params": {
          "account_id": "18ce54d4x5t",
          "entity": "PROMOTED_TWEET",
          "start_time": "2019-03-05T00:00:00Z",
          "end_time": "2019-03-06T00:00:00Z"
        }
      },
      "data": [
        {
          "entity_id": "2r0wxw",
          "activity_start_time": "2019-03-04T20:55:20Z",
          "activity_end_time": "2019-03-05T03:43:56Z",
          "placements": [
            "ALL_ON_TWITTER"
          ]
        },
        {
          "entity_id": "2r30fn",
          "activity_start_time": "2019-03-05T08:11:08Z",
          "activity_end_time": "2019-03-05T14:40:59Z",
          "placements": [
            "ALL_ON_TWITTER",
            "PUBLISHER_NETWORK"
          ]
        }
      ]
    }
```

`data` 배열에는 이후 애널리틱스 요청에 포함되어야 하는 각 엔티티에 대한 객체가 포함됩니다. 이 집합에 없는 ID에 대해 애널리틱스를 요청해서는 안 됩니다.

각 객체에는 네 개의 필드가 포함됩니다: `entity_id`, `activity_start_time`, `activity_end_time`, `placements`. 활동 시작 시간과 종료 시간은 해당 엔티티의 변경 이벤트가 적용되는 시간 범위를 나타내며, 따라서 이후 애널리틱스 요청에서 지정해야 하는 날짜를 결정합니다. `placements` 배열에는 `ALL_ON_TWITTER`, `PUBLISHER_NETWORK`, `SPOTLIGHT`, `TREND` 값을 포함할 수 있습니다. 이는 주어진 엔티티 ID에 대해 어떤 게재 위치를 요청해야 하는지를 나타냅니다.

<div id="usage">
  #### 사용 방법
</div>

Active Entities 엔드포인트는 분석 요청 방식을 결정하는 기준이 되어야 합니다. 아래 사용 지침은 분석 동기화를 지원하기 위해 작성되었으며, 이를 통해 파트너가 자신의 데이터 저장소를 Twitter와 동기화 상태로 유지할 수 있습니다. 다시 말해, 정기적인 백그라운드 동기화 작업을 수행하는 방법을 설명합니다.

개발자는 두 가지 결정을 내려야 합니다.

1. 활성 엔터티 정보를 얼마나 자주 요청할지, 그리고 그에 따라 분석 데이터를 얼마나 자주 가져올지.
2. 활동 시작 및 종료 시간을 어떻게 사용하여 분석 요청의 `start_time` 및 `end_time` 값을 결정할지.

이 두 가지 사항은 아래의 요약 이후 이어지는 두 개의 하위 섹션에서 더 자세히 설명합니다.

<div id="summary">
  ### 요약
</div>

Active Entities 엔드포인트를 다음과 같은 방식으로 사용해 분석 요청 방식을 정의합니다. 활성 엔티티 정보를 얼마나 자주 요청할지, 그리고 그에 따라 분석을 얼마나 자주 수집할지 결정한 후 이 절차를 따르십시오.

1. Active Entities 요청을 수행합니다.
2. 응답을 placement별로 분리합니다. `ALL_ON_TWITTER`, `PUBLISHER_NETWORK`, `SPOTLIGHT`, `TREND` 각각에 대해 하나의 그룹을 만듭니다.
3. 각 placement 그룹에 대해 다음을 수행합니다.
   1. 엔티티 ID를 추출합니다.
   2. 분석용 `start_time` 및 `end_time` 값을 결정합니다.
      * 최소 `activity_start_time`을 찾고, 이 값을 내림 처리합니다.
      * 최대 `activity_end_time`을 찾고, 이 값을 올림 처리합니다.
   3. 분석 요청을 보냅니다.
      * 엔티티 ID를 20개씩 배치로 묶습니다.
      * #3b에서 얻은 `start_time` 및 `end_time` 값을 사용합니다.
      * 적절한 `placement` 값을 지정합니다.
   4. 데이터 저장소에 기록합니다.

Python SDK를 사용하는 예제로 [active&#95;entities.py](https://github.com/xdevplatform/twitter-python-ads-sdk/blob/master/examples/active_entities.py)를 참고하십시오.

<div id="frequency">
  ### Frequency
</div>

첫 번째 질문에 대한 답변에 따라 Active Entities 요청에 사용해야 하는 시간 범위가 결정됩니다. 예를 들어, 매시간 Active Entities 정보를 요청하는 경우 시간 범위는 1시간이어야 합니다. 하루에 한 번 Active Entities 정보를 요청하는 경우 시간 범위는 하루여야 합니다. 즉, 현재 요청의 `start_time`이 이전 요청의 `end_time`과 같도록 시간 범위를 선택해야 합니다.

<Note>
  **참고**: 하나의 시간 구간은 한 번만 요청해야 합니다. 하나의 시간 구간을 여러 번 요청하면 불필요한 분석(analytics) 요청이 발생합니다. (아래 예외 참조.)
</Note>

파트너가 *현재* 시간에 대해 한 시간 동안 여러 번 analytics를 요청하려는 경우에도 동일한 패턴이 적용됩니다. 즉, 요청 빈도가 시간 범위를 결정합니다. 아래 표는 이 시나리오에서의 예시 Active Entities 시작 및 종료 타임스탬프를 보여줍니다.

|     |     |     |
| :--- | :--- | :--- |
| **Request time** | **`start_time` timestamp** | **`end_time` timestamp** |
| 00:15:00 | 00:00:00 | 00:15:00 |
| 00:30:00 | 00:15:00 | 00:30:00 |
| 00:45:00 | 00:30:00 | 00:45:00 |
| 01:00:00 | 00:45:00 | 01:00:00 |

change event가 저장되는 방식 때문에, 위의 네 개 Active Entities 요청은 모두 동일한 시간별 버킷을 조회하며, 이는 이 사용 사례에 필요합니다. 그러나 해당 현재 시간이 지난 후에는 이 시간별 버킷을 더 이상 조회해서는 안 됩니다.

<div id="activity-times">
  ### 활동 시간
</div>

활동 시작 및 종료 시간을 다룰 때는 다음과 같은 방식을 권장합니다. Active Entities 응답에 포함된 모든 객체에서 최소 `activity_start_time` 과 최대 `activity_end_time` 을 찾습니다. 그런 다음 최소 활동 시작 시간은 내림(이전 시각으로 반올림)하고 최대 활동 종료 시간은 올림(다음 시각으로 반올림)하여 값을 조정합니다. 구체적으로, 두 타임스탬프 모두 시간, 분, 초를 0으로 설정하고 종료 시간에는 하루를 추가합니다. 아래 표에서 그 예를 확인할 수 있습니다. 이렇게 계산된 시작 및 종료 시간을 이후 analytics 요청에서 지정해야 합니다.

|     |     |
| :--- | :--- |
| **Min, max activity times** | **Derived times** |
| 2019-03-04T20:55:20Z  <br />  <br />2019-03-05T14:40:59Z | 2019-03-04T00:00:00Z  <br />  <br />2019-03-06T00:00:00Z |

**참고**: 시간, 분, 초가 0으로 설정된 타임스탬프를 반드시 포함하는 것이 중요합니다. 그렇지 않고 날짜만 전달되면, Ads 계정의 표준 시간대 기준으로 자정에 시작해서 자정에 끝나는 analytics 를 요청하는 것으로 간주하게 되는데, 이는 원하지 않는 결과일 수 있습니다. 예를 들어 최소 활동 시작 시간이 2019-02-28T01:30:07Z 인데, 오프셋이 -08:00:00 인 Ads 계정에 대해 타임스탬프를 생략하면, analytics 요청에서는 01:30과 08:00 사이에 발생한 변경 사항을 놓치게 됩니다.

또는, 전체 하루 단위로 확장하지 않고 반환된 활동 시간 구간에 대해서만 analytics 를 요청하고자 하는 경우에도 그렇게 할 수 있습니다. 이 방식을 사용할 경우, 파생된 시작 및 종료 시간은 각각 2019-03-04T20:00:00Z 와 2019-03-05T15:00:00Z 가 됩니다. (analytics 요청에서 `DAY` granularity 를 지정한 경우에는 이와 같은 범위는 허용되지 않는다는 점에 유의하십시오.)

<div id="asynchronous-analytics">
  ### 비동기 분석

<div id="introduction">
  #### 소개
</div>

비동기 분석 엔드포인트를 사용하면 파트너와 광고주가 생성 요청을 제출해 서버가 비동기적으로 처리하도록 메트릭을 요청할 수 있습니다. (이를 비동기 분석 &quot;작업(job)&quot;이라고 합니다.) 이 방식에서는 요청이 완료될 때까지 클라이언트 연결을 계속 유지할 필요가 없습니다.

이 엔드포인트는 동기식 엔드포인트와 마찬가지로, 파트너와 광고주가 캠페인 성과에 대한 상세 통계를 요청할 수 있도록 합니다. 계정, 펀딩 수단(funding instrument), 캠페인, 라인 아이템, 프로모션 트윗, 미디어 크리에이티브에 대한 데이터 요청을 지원합니다. 동기식 엔드포인트와의 차이점은, 비동기 분석 엔드포인트가 최대 90일까지 더 긴 날짜 범위와 세분화를 지원한다는 점입니다. 두 방식 간 차이에 대한 추가 세부 정보는 [Analytics 개요](/ko/x-ads-api/analytics) 페이지에서 확인할 수 있습니다.

동기식 엔드포인트와 달리, 요청 제한(rate limit)은 특정 계정에 대해 동시에 실행 중인 작업 개수를 기준으로 합니다. 다시 말해, 특정 시점에 처리 상태에 있을 수 있는 작업 개수에 따라 결정됩니다. 이 개수는 광고 계정 수준에서 계산합니다.

<div id="usage">
  #### 사용 방법
</div>

비동기 분석 엔드포인트를 사용해 캠페인 메트릭을 조회하는 과정은 여러 단계로 이루어집니다. 작업을 생성하고, 작업 처리가 완료되었는지 확인한 뒤, 마지막으로 데이터를 다운로드해야 합니다. 데이터 파일은 압축을 해제해야 합니다. 네 가지 구체적인 단계는 아래와 같습니다.

1. [POST stats/jobs/accounts/:account&#95;id](/ko/x-ads-api/analytics#asynchronous-analytics) 엔드포인트를 사용해 작업을 생성합니다.
2. 작업 처리가 완료되었는지 확인하기 위해 [GET stats/jobs/accounts/:account&#95;id](/ko/x-ads-api/analytics#asynchronous-analytics) 엔드포인트에 일정 간격으로 요청을 보냅니다.
3. 작업 처리가 완료되면 데이터 파일을 다운로드합니다.
4. 데이터 파일의 압축을 해제합니다.

데이터 파일에 포함되어 반환되는 응답 객체는 동기식 분석 엔드포인트의 응답과 동일한 JSON 스키마를 가집니다.

세분화된 캠페인 메트릭은 비동기 분석 엔드포인트에서만 조회할 수 있습니다. 캠페인 메트릭은 지역, 성별, 관심사, 키워드 등으로 세분화할 수 있습니다. 사용 가능한 전체 옵션 목록은 [Metrics and Segmentation](/ko/x-ads-api/analytics#metrics-and-segmentation) 페이지를 참고하십시오. 세분화된 메트릭을 요청하려면 작업을 생성할 때 `segmentation_type` 요청 매개변수를 사용하십시오.

지정한 기간 동안 어떤 엔티티의 분석 메트릭이 변경되었는지에 대한 세부 정보를 반환합니다.

이 엔드포인트는 분석 엔드포인트와 함께 사용해야 합니다. 이 엔드포인트의 응답은 어떤 광고 엔티티에 대해 분석을 요청해야 하는지를 나타냅니다. 사용 방법에 대한 지침은 [Active Entities Guide](/ko/x-ads-api/analytics#active-entities)를 참고하세요.

변경 이벤트는 한 시간 단위 버킷으로 제공됩니다.

* `start_time` 및 `end_time` 값은 조회할 시간 단위 버킷을 지정합니다.
* 반환되는 `data` 배열에는 이후 분석 요청에 포함되어야 하는 각 엔티티에 대한 객체가 포함됩니다.
* **중요**: 이후 분석 요청에 지정해야 하는 날짜는 `activity_start_time` 및 `activity_end_time` 값을 기준으로 결정해야 합니다.
  * 이 값들은 저장된 변경 이벤트가 *적용되는* 시간 범위를 나타냅니다. 이는 엔티티별로 반환됩니다.

**참고**: 허용되는 최대 시간 범위(`end_time` - `start_time`)는 90일입니다.
