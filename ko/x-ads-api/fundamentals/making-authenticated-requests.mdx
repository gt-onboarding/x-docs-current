X Ads API 엔드포인트에 액세스하려면 애플리케이션이 TLS를 사용해 https://ads-api.x.com 으로 인증된 웹 요청을 안전하게 전송해야 합니다.

다음 섹션에서는 인증된 API 요청을 수행하는 방법에 대한 개요, API와 상호 작용하기 위한 [Twurl](https://github.com/twitter/twurl#getting-started)을 설정하는 방법, 그리고 애플리케이션을 확장해 OAuth 1.0a를 지원하고 Ads 계정으로 요청을 보내는 방법을 설명합니다.

<div id="requirements">
  ## 요구 사항
</div>

X Ads API에 인증된 요청을 보내기 전에 다음이 필요합니다:

* [승인된 개발자 계정](/ko/resources/fundamentals/developer-portal)
* [Ads API](/ko/x-ads-api/introduction) 액세스가 승인된 앱
* [앱 관리 UI](/ko/resources/fundamentals/developer-apps)를 통해 발급받은 API 키와 시크릿, 그리고
* X Ads 계정에 액세스 권한이 있는 사용자의 [액세스 토큰](/ko/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow)

<div id="using-the-api">
  ## API 사용
</div>

Advertising API는 https://ads-api.x.com 에서 이용할 수 있습니다. [표준 REST API](https://developer.x.com/en/docs/x-api/v1/tweets/post-and-engage/overview)와 Advertising API는 동일한 클라이언트 앱에서 함께 사용할 수 있습니다. Advertising API는 HTTPS를 강제하므로, HTTP로 엔드포인트에 접근하려는 모든 시도는 오류 메시지를 반환합니다.

Ads API는 JSON을 출력합니다. 모든 식별자는 문자열이며, 모든 문자열은 UTF-8 인코딩을 사용합니다. Advertising API는 [버전 관리](/ko/x-ads-api/fundamentals/versioning)가 적용되어 있으며, 버전은 모든 리소스 URL의 첫 번째 경로 요소로 지정됩니다.

`https://ads-api.x.com/<version>/accounts`

<div id="http-verbs-typical-response-codes">
  ## HTTP 동사 &amp; 일반적인 응답 코드
</div>

Ads API에서는 네 가지 HTTP 동사를 사용합니다:

* **GET**은 데이터를 조회합니다 
* **POST**는 캠페인과 같은 새 데이터를 생성합니다
* **PUT**은 라인 아이템과 같은 기존 데이터를 업데이트합니다
* **DELETE**는 데이터를 삭제합니다.

삭제는 영구적이지만, 리소스를 요청할 때 `with_deleted=true` 파라미터를 명시적으로 포함하면 대부분의 GET 기반 메서드에서 삭제된 데이터도 계속 조회할 수 있습니다. 그렇지 않으면 삭제된 레코드는 HTTP 404를 반환합니다.

성공적인 요청은 리소스를 생성, 삭제 또는 업데이트할 때 객체를 나타내는 JSON 응답과 함께 HTTP 200번대 응답을 반환합니다.

HTTP PUT으로 데이터를 업데이트할 때는 지정한 필드만 업데이트됩니다. 파라미터에 빈 문자열을 지정하여 선택적 값을 비울 수 있습니다. 예를 들어, 다음과 같은 파라미터 그룹은 이미 지정된 `end_time` 값을 제거합니다: `&end_time=&paused=false`.

에러 응답에 대한 자세한 내용은 [Error Codes &amp; Responses](/ko/x-ads-api/fundamentals/error-codes-and-responses)를 참조하세요.

<div id="in-line-parameters">
  ## 인라인 매개변수
</div>

대부분의 리소스 URL에는 하나 이상의 인라인 매개변수가 포함됩니다. 많은 URL은 쿼리 문자열의 명시적으로 선언된 매개변수를 사용하거나, POST 또는 PUT 요청의 경우 본문에 매개변수를 포함합니다.

인라인 매개변수는 각 리소스의 **Resource Path** 섹션에서 앞에 콜론(”:”)을 붙여 표시합니다. 예를 들어, 작업 중인 계정의 식별자가 `"abc1"`이고 [계정과 연결된 캠페인 가져오기](/ko/x-ads-api/campaign-management#campaigns)를 수행한다고 할 때, `https://ads-api.x.com/6/accounts/abc1/campaigns` URL을 사용하여 해당 목록에 접근합니다. 리소스 URL(`https://ads-api.x.com/6/accounts/:account_id/campaigns`)에 설명된 인라인 `account_id` 매개변수를 지정함으로써, 요청 대상을 그 계정과 연결된 객체로만 한정하게 됩니다.

<div id="using-access-tokens">
  ## 액세스 토큰 사용
</div>

X Ads API는 애플리케이션의 신원을 검증하고, 애플리케이션이 대신하여 API 요청을 수행하는 최종 사용자가 부여한 권한을 확인하기 위해 서명된 HTTPS 요청을 사용합니다. 이 권한은 사용자의 액세스 토큰으로 표현됩니다. Ads API에 대한 모든 HTTP 호출은 HTTPS 프로토콜을 통해 Authorization 요청 헤더(OAuth 1.0a 사용)를 포함해야 합니다.

X Ads API와 통합하려면 애플리케이션에 OAuth 1.0a Authorization 요청 헤더를 생성하는 기능을 추가해야 합니다. 다만 서명된 요청을 생성하는 과정이 복잡하므로, X API를 지원하거나 OAuth 1.0a 요청 처리를 구현한 기존 라이브러리를 사용하는 것을 파트너에게 강력히 권장합니다. [권장 OAuth 라이브러리](/ko/resources/fundamentals/authentication#oauth-1-0a-2)와 [인증 코드 샘플](/ko/resources/fundamentals/authentication#oauth-1-0a-2) 목록은 해당 링크에서 확인할 수 있습니다.

잘 알려진 라이브러리를 사용하는 중에 인증 오류가 발생한 경우에는 지원해 드릴 수 있지만, 직접 구현한 사용자 정의 OAuth는 지원하지 않습니다.

<div id="http-oauth">
  ## HTTP &amp; OAuth
</div>

X REST API v1.1과 마찬가지로 Advertising API에서는 [OAuth 1.0A](/ko/resources/fundamentals/authentication)와 HTTPS를 모두 사용해야 합니다. API 키는 [app management console](/ko/resources/fundamentals/developer-apps)을 통해 발급받을 수 있습니다. 액세스 토큰은 “현재 사용자”를 나타내는 데도 반드시 사용되어야 합니다. 여기서 현재 사용자는 광고 기능을 보유한 X 계정을 의미합니다.

파트너는 직접 구현하기보다 OAuth 라이브러리를 사용하는 것이 강력히 권장됩니다. 잘 알려진 라이브러리를 사용하는 경우에는 디버깅을 지원할 수 있지만, 자체적으로 OAuth를 구현한 경우에는 지원이 불가능합니다. 사용할 수 있는 [libraries](/ko/resources/fundamentals/authentication#oauth-1-0a-2)를 참고하세요.

이 API는 HTTP/1.1 및 OAuth 규격을 엄격히 따릅니다. OAuth signature base string을 준비하기 전에 URL과 POST body 내에서 [예약 문자 인코딩](https://tools.ietf.org/html/rfc3986#section-2.2)이 올바르게 적용되었는지 확인하세요. 특히 Advertising API에서는 시간을 지정할 때 &quot;:&quot; 문자를, 여러 옵션을 제공할 때 &quot;,&quot; 문자를 사용합니다. 이 두 문자는 모두 다음의 예약 문자 집합에 속합니다.

| Symbol | URL Encoded |
| :--- | :--- |
| !   | %21 |
| #   | %23 |
| $   | %24 |
| &amp;   | %26 |
| &#39;   | %27 |
| (   | %28 |
| )   | %29 |
| *   | %2A |
| +   | %2B |
| ,   | %2C |
| /   | %2F |
| :   | %3A |
| ;   | %3B |
| =   | %3D |
| ?   | %3F |
| @   | %40 |
| [   | %5B |
| ]   | %5D |

<div id="making-your-first-api-request-with-twurl">
  ## Twurl로 첫 API 요청 보내기
</div>

X는 OAuth 1.0a 인증 헤더를 지원하는 명령줄 도구 [Twurl](https://developer.x.com/en/docs/tutorials/using-twurl)을 관리하고 있으며, 이는 [cURL](https://en.wikipedia.org/wiki/CURL)의 대안으로 사용할 수 있습니다. Twurl은 인증된 API 요청을 보내고, 애플리케이션에 인증을 추가하기 전에 Ads API를 살펴볼 수 있는 간단한 방법을 제공합니다.

[Twurl을 설치하고 승인한](https://github.com/twitter/twurl#getting-started) 후에는 액세스 토큰을 빠르게 생성하고 Ads API에 인증된 요청을 보낼 수 있습니다.

```bash
twurl -H "ads-api.x.com" "/5/accounts/"
```

Twurl과 API에 익숙해질 수 있도록, API를 통해 캠페인을 생성하는 방법을 단계별로 설명한 이 [튜토리얼](/ko/x-ads-api/campaign-management#creating-a-campaign-step-by-step)을 따라 해 보세요.

<div id="extending-your-application-to-make-authenticated-requests">
  ### 애플리케이션을 확장하여 인증된 요청 보내기
</div>

Twurl을 사용해 Ads API에 요청을 보내는 방법에 익숙해졌다면, 이제 애플리케이션에 OAuth 1.0a 인증 헤더를 생성하는 기능을 추가할 차례입니다.

[OAuth 1.0a](/ko/resources/fundamentals/authentication) 인증 헤더에는 애플리케이션과 사용자의 신원을 검증하고, 요청 위·변조를 방지하는 데 필요한 정보가 포함됩니다. 애플리케이션은 각 API 요청마다 새로운 Authorization 헤더를 생성해야 합니다. 많은 프로그래밍 언어에는 X에 API 요청을 보내기 위한 이 Authorization 헤더 생성을 지원하는 오픈 소스 라이브러리가 있습니다.

다음은 C#, PHP, Ruby, Python을 사용한 예시입니다 - [코드 샘플](/ko/resources/fundamentals/authentication#oauth-1-0a-2).

<div id="custom-implementation">
  ## 커스텀 구현
</div>

오픈 소스 라이브러리의 지원 없이 OAuth 1.0a 인증을 구현해야 하는 경우가 있습니다. [요청 승인하기](/ko/resources/fundamentals/authentication#authorizing-a-request)는 Authorization 헤더를 생성하는 기능을 구현하는 방법에 대한 상세한 지침을 제공합니다. 커뮤니티에서 지원하는 라이브러리를 사용할 것을 강력히 권장합니다.

개요:

1. 헤더에 사용할 oauth&#95;로 시작하는 key/value(키/값) 쌍 7개를 수집합니다.
2. 해당 key/value 쌍을 사용하여 [OAuth 1.0a HMAC-SHA1 서명(signature)](/ko/resources/fundamentals/authentication#creating-a-signature)을 생성합니다.
3. 위의 값들을 사용하여 [Authorization 헤더](/ko/resources/fundamentals/authentication#authorizing-a-request)를 구성합니다.
