<div id="typical-response-structure">
  ## 일반적인 응답 구조
</div>

성공적인 응답은 200번대 HTTP 코드와 함께, 요청된 객체가 반환되거나 생성, 수정, 삭제된 결과와 서버가 요청을 어떻게 해석했는지를 나타내는 표현이 포함된 JSON 기반 페이로드로 제공됩니다.

성공적인 요청을 보낸 경우, 응답의 일부로 요청 내용을 그대로 반영한 `request` 노드를 받게 됩니다.

*예시:* [GET accounts/abcdefg/campaigns?with&#95;deleted=true](/ko/x-ads-api/campaign-management#campaigns)

```json
{
  /* 응답 데이터... */,
  "request": {
    "params": {
      "account_id": "abcdefg",
      "with_deleted": "true"
    }
  }
}
```

JSON 응답의 `data` 필드에는 사용 중인 리소스와 연결된 특정 객체들이 포함됩니다. 응답에 하나 이상의 결과가 포함될 수 있는 경우 `data` 노드의 형식은 JSON 배열이 됩니다. 응답에서 하나의 결과만 가능한 경우에는 JSON 해시로 반환됩니다. 드물게, 일반적으로 컬렉션이 포함되는 응답에서 대신 해시맵이 포함된 것을 볼 수도 있습니다. 이 경우, 단일 해시맵은 `type` 필드에 지정된 것과 동일한 타입의 객체라고 가정하면 됩니다.

<div id="error-response-structure">
  ## 오류 응답 구조
</div>

오류 응답은 200대가 아닌 HTTP 상태 코드와 함께 제공됩니다. 보통 JSON 응답이 포함되지만, 일부 오류는 다른 형태의 본문으로 응답될 수 있습니다. 이처럼 응답 구조를 파싱할 수 없는 상황에서는 해당 HTTP 코드의 핵심 의미를 우선시해야 합니다. 예를 들어, 때때로 HTTP 404와 함께 HTML 응답이 반환될 수 있습니다. 이 경우, 콘텐츠를 찾을 수 없다고 안전하게 가정할 수 있습니다(HTTP 404는 “Not Found”를 의미합니다).

일반적인 오류 응답은 성공 응답과 유사한 구조를 따릅니다. 오류의 성격은 응답의 `errors` 노드에서 전달됩니다. `errors/code` 노드는 프로그램에서 활용하여 해결 방법을 결정할 수 있는 CAPS&#95;CASE 상수 오류 코드를 나타냅니다. `errors/message` 노드는 (일반적으로) 오류에 대한 사람이 읽고 이해할 수 있는 영어 설명을 나타냅니다. 오류에 대한 더 세밀한 정보를 제공하기 위해 추가 필드가 포함될 수 있습니다.

```json
{
  "errors": [
    {
      "parameter": "start_time",
      "details": "invalid date",
      "code": "INVALID_PARAMETER",
      "value": "",
      "message": "Expected time, got \"\" for start_time"
    }
  ],
  "request": {
    "params": {
      "account_id": "hkk5"
    }
  }
}
```

위 예제에서는 `start_time` 파라미터에 유효하지 않은 값을 사용하여 analytics 엔드포인트에 요청을 보냈습니다. 잘못된 파라미터가 포함된 요청의 `errors/code` 값은 `INVALID_PARAMETER`입니다.
