<div id="user-accounts-vs-ad-accounts">
  # 사용자 계정 vs 광고 계정
</div>

Ads API를 사용할 때는 두 가지 유형의 계정이 사용됩니다. 바로 광고 계정과 X 사용자 계정입니다. Ads API 문서 전반에서 “account”라는 용어는 보통 광고 계정을 의미합니다.

* 광고 계정은 **business.x.com**에서 등록되며, API에서는 account&#95;id로 식별됩니다. 광고 계정은 결제 수단과 직접 연결되며, 하나 이상의 X 사용자 계정에서 가져온 콘텐츠를 ‘프로모션 대상 사용자(promotable users)’로 활용합니다. 각 광고 계정은 하나 이상의 X 사용자 계정에 권한을 부여할 수 있습니다. 광고 계정, 즉 “current account”는 대부분의 URL에서 인라인 :account&#95;id 파라미터로 나타납니다.
* X 사용자 계정(예: @AdsAPI)은 Ads API에서 user&#95;id로 식별됩니다. 이러한 계정 하나 이상이 하나의 광고 계정과 연결될 수 있습니다. API에 요청을 보내는 인증된 X 사용자 계정은 ‘current user’라고 합니다. current user가 접근 권한을 가진 광고 계정 목록은 [GET accounts](/ko/x-ads-api/campaign-management#accounts)로 조회할 수 있습니다. ‘Promotable users’는 특정 광고 계정이 프로모션할 수 있는 X 핸들(handle)입니다. 이에 대한 자세한 내용은 [Obtaining Ads Account Access](/ko/x-ads-api/introduction)를 참조하세요.

<div id="methods-for-ad-account-access">
  ## 광고 계정 액세스 방법
</div>

광고주의 계정에 대해 Ads API 요청을 수행하는 방법은 두 가지가 있습니다.

1. 광고주를 대신하여 요청하기 (**권장**)
2. 여러 계정을 지원하는 [Agency](https://business.x.com/en/blog/agencies-manage-multiple-Twitter-Ads-accounts.html)처럼, 광고주의 계정에 대한 액세스 권한을 부여받은 본인 계정을 사용해 요청하기

이 문서는 이러한 옵션 간 차이에 대한 간단한 개요이며, [다중 사용자 로그인 FAQ](https://business.x.com/en/help/troubleshooting/multi-user-login-faq.html)와 같은 기타 자료와 함께 활용해야 합니다. 

[요청 승인](/ko/resources/fundamentals/authentication#authorizing-a-request#authorizing-a-request)에 설명된 것처럼, Ads API에 대한 모든 요청에는 [3-legged](/ko/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow)[OAuth](/ko/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) 플로를 통해 획득한 액세스 토큰을 사용하는 [OAuth 1.0a](/ko/resources/fundamentals/authentication) 기반 Authorization 헤더가 필요합니다. 애플리케이션은 [액세스 토큰을 얻기](/ko/resources/fundamentals/authentication#oauth-1-0a-2) 위한 웹 기반 OAuth 플로를 구현해야 합니다. Ads API 개발자는 X 광고주에게 로그인 자격 증명을 공유해 달라고 요청해서는 안 됩니다.

기본적으로 각 [X 개발자 앱](/ko/resources/fundamentals/developer-apps)에는 앱 소유 계정에 대해 Ads API 요청을 수행하는 데 사용할 수 있는 고정된 **액세스 토큰**이 포함되어 있습니다. 이러한 자격 증명은 3-legged 또는 [PIN 기반 OAuth](/ko/resources/fundamentals/authentication#pin-based-authorization) 플로 없이 단일 계정 사용 사례에 적합합니다. 다른 X 광고 계정에 액세스하지 않는 경우, 다음 단계 대신 이러한 [단일 사용자](/ko/resources/fundamentals/authentication#oauth-1-0a-2) 자격 증명을 사용하십시오.

<div id="levels-of-access">
  ## 접근 권한 수준
</div>

<div id="app-level-permissions">
  ### 앱 수준 권한
</div>

각 사용자는 [Ads API 신청](/ko/x-ads-api/getting-started/step-by-step-guide) 시 요청한 내용에 따라 액세스 수준이 부여됩니다.

* **전환 전용(Conversion Only)**: 사용자는 [모바일](/ko/x-ads-api/measurement/mobile-conversions) 및 [웹 전환](/ko/x-ads-api/measurement/web-conversions) 엔드포인트에 대해 [읽기 및 쓰기](/ko/resources/fundamentals/developer-apps) 액세스 권한을 가집니다.
* **표준 액세스(Standard Access)**: 사용자는 [Analytics](/ko/x-ads-api/analytics), [Campaign Management](https://developer.x.com/en/docs/twitter-ads-api/campaign-management/overview), [Creatives](https://developer.x.com/en/docs/twitter-ads-api/creatives/overview), [Custom Audiences](https://developer.x.com/en/docs/twitter-ads-api/audiences/overview), [Conversion](/ko/x-ads-api/measurement/web-conversions) 엔드포인트에 대해 [읽기 및 쓰기](/ko/resources/fundamentals/developer-apps) 액세스 권한을 가집니다.

**참고:** 2023년 7월 이전에 액세스를 요청한 Ads API 개발자는 서로 다른 수준의 액세스 권한을 가질 수 있으며, OAuth 토큰이 최대 5개로 제한될 수 있습니다. 추가 엔드포인트에 대한 액세스를 받거나 기존 앱의 토큰 제한을 완화하려면 [액세스 확대](/ko/x-ads-api/getting-started/increasing-access) 가이드를 참조하세요.

<div id="ad-account-level-permissions">
  ### 광고 계정 수준 권한
</div>

Ads 계정에 액세스 권한이 있는 각 사용자는 **Account administrator**, **Ad manager**, **Campaign analyst**, **Organic analyst**, **Creative Manager** 중 하나의 계정 수준 권한을 갖습니다. 계정 수준 권한에 대한 최신 문서는 **business.x.com**을 참고하세요. 애플리케이션은 현재 인증된 사용자의 권한을 [Authenticated User Access](/ko/x-ads-api/campaign-management#authenticated-user-access) API 엔드포인트를 통해 조회하여, 해당 사용자가 어떤 API 엔드포인트와 Ads 기능에 액세스할 수 있는지 확인해야 합니다.

**참고:** [Conversion API](/ko/x-ads-api/measurement/web-conversions)에 사용되는 모든 사용자 토큰은 **Account administrator** 또는 **Ad manager** 계정 수준 권한을 가진 사용자의 토큰이어야 합니다.                                                                                                                                              

<div id="methods-of-obtaining-access-tokens">
  ## 액세스 토큰을 발급받는 방법
</div>

<div id="1-obtain-an-advertisers-user-access-token">
  ### 1. 광고주(사용자) 액세스 토큰 얻기
</div>

광고주의 **액세스 토큰**을 얻는 방법은 두 가지가 있습니다. 가장 일반적인 방법은 웹 UI 내에서 직접 [3-legged OAuth 플로우](/ko/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow)를 사용하는 것입니다. 광고주가 접근할 수 있는 공개 UI가 없는 애플리케이션은 [PIN 기반](/ko/resources/fundamentals/authentication#pin-based-authorization) OAuth 프로세스를 구현할 수 있습니다. 사용자가 3-legged 플로우를 완료하면, 애플리케이션은 API를 통해 해당 사용자의 Ads 계정에 요청을 보낼 수 있는 자격 증명을 갖게 됩니다.

OAuth 플로우를 통해 사용자 자격 증명을 얻는 방법은 대부분의 Ads API 개발자가 광고주 계정에 대한 액세스 권한을 얻기 위해 **강력히 권장되는** 방식입니다. 이 방법을 사용하면 사용자를 대신하여 API를 호출하고, 해당 사용자로서 작업을 수행할 수 있습니다. 이 토큰은 만료되지 않지만, 사용자가 언제든지 취소(철회)할 수 있습니다.

<div id="2-obtain-your-developer-access-token">
  ### 2. (Developer) 액세스 토큰 발급받기
</div>

이 옵션을 사용하려면 광고주가 business.x.com의 X UI를 통해 자신의 X Ads 계정에 대한 액세스 권한을 [귀하의 @username](https://business.x.com/en/help/troubleshooting/multi-user-login-faq.html) (또는 여러 @username)에게 부여해야 합니다. 귀하의 계정에 대해 3-legged OAuth 플로우를 통해 얻은 **액세스 토큰**은 광고주의 X Ads 계정에 접근할 수 있습니다.

이를 통해 광고주의 OAuth 토큰이 아니라, 귀하 자신의 @username에 대한 OAuth 토큰을 사용하여 API를 호출할 수 있습니다. 이 옵션에서 핵심적인 차이점은, 귀하의 @username에 게시물 위임/작성 권한이 부여된 경우에만 Promoted-Only 게시물을 생성할 수 있다는 점입니다.

계정의 `FULL` 프로모션 가능 사용자를 대신하여 Promoted-Only 게시물을 생성할 수 있는 권한을 얻으려면, 이 플로우에서 게시물을 생성할 수 있는 권한도 추가로 부여받아야 합니다. 이렇게 하면 [GET accounts/:account&#95;id/authenticated&#95;user&#95;access](/ko/x-ads-api/campaign-management#accounts) 엔드포인트의 `TWEET_COMPOSER` 권한을 통해 액세스할 수 있습니다.

<div id="differences-between-these-methods">
  ## 이러한 방법 간의 차이점
</div>

|     |     |     |
| :--- | :--- | :--- |
|     | 광고주(사용자) OAuth 토큰 | (Developer) OAuth 토큰  <br /> (@username이 별도의 계정에 추가됨) |
| Ads 계정에 액세스 | ✔   | ✔   |
| 사용자를 대신해 게시물 생성 | ✔   | ✔*  |
| 캠페인 관리 | ✔   | ✔   |
| 애널리틱스에 액세스 | ✔   | ✔   |
| 사용자를 대신해 카드 생성 | ✔   | ✔   |
| 개발자가  <br />X Ads UI를 통해 액세스 |     | ✔   |
| [요청 한도](/ko/x-api/enterprise-gnip-2.0/fundamentals/rate-limits) | 광고주별로 별도 적용 | 광고주 계정별로 별도 적용 |

<div id="advertisers-access-token-via-oauth-3-legged-web-flow">
  ### 광고주의 액세스 토큰 (OAuth 3-legged 웹 플로우)
</div>

표준 플로우는 웹 기반이며 [3-legged authorization](/ko/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) OAuth 플로우를 사용합니다. 여기에서 설명하는 스크린샷은 [https://github.com/xdevplatform/twauth-web](https://github.com/xdevplatform/twauth-web)에서 소스를 확인할 수 있는 샘플의 일부입니다.

애플리케이션 플로우 중 어느 시점에서, 애플리케이션을 승인받기 위해 X로 리디렉션해야 합니다.
