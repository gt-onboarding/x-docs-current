  ## 개요
</div>

<div id="introduction">
  ### 소개
</div>

A/B 테스트를 통해 광고주는 X에서 도달하고 있는 사용자를 여러 그룹으로 분할하여, 캠페인 성과를 최적화하는 가장 효과적인 방법을 파악하고 마케팅 전략 수립에 필요한 인사이트를 얻을 수 있습니다.

이러한 세그먼트는 사용자 그룹 분할(user group split)이라고 하며, 무작위로 구성되고 서로 배타적입니다. 무작위 배분을 통해 결과에 영향을 미치는 요인이 각 그룹에 고르게 분포됩니다. 다시 말해, 그룹 간 또는 이들의 예상 행동 간에는 본질적인 차이가 없습니다. 그렇기 때문에 특정 사용자 그룹에만 하나의 실험 버전(variation)을 적용하고 다른 그룹에는 적용하지 않았을 때, 캠페인 성과의 차이는 해당 실험 버전에 기인한 것으로 볼 수 있습니다.

한 번에 여러 실험 버전을 테스트하는 것도 가능하지만, 한 번에 하나의 실험 버전만 테스트할 것을 강력히 권장합니다. 이렇게 하면 관측된 캠페인 성과 차이를 유발한 인과 요인을 명확히 분리해낼 수 있습니다.

실험 버전은 캠페인 단위에서 설정됩니다. 예를 들어, 광고주가 새로운 크리에이티브의 효과를 테스트하고자 하는 경우, 크리에이티브만 다르고 나머지는 동일한 두 개의 캠페인을 생성해야 합니다. 향후에는 라인 아이템 단위의 실험 버전도 지원할 계획입니다.

<div id="use-cases">
  ### 사용 사례
</div>

A/B 테스트는 주로 (1) X에서 어떤 요소가 가장 효과적인지 파악해 투자 효율을 극대화하려는 성과형 광고주의 최적화 목적과, (2) 실험에서 얻은 인사이트를 마케팅 전략 수립에 활용하려는 브랜드 광고주의 학습 목적을 지원하는 데 사용됩니다. 

API는 다음을 포함한 모든 캠페인 변수에 대해 A/B 테스트를 지원합니다:

* 크리에이티브 

* 타게팅 

* 입찰 유형

* 입찰 단위

<div id="ab-testing">
  ## A/B 테스트
</div>

A/B 테스트를 사용하면 광고주가 X에서 도달하는 사용자를 세분화하여 캠페인 성과를 어떻게 최적으로 개선할 수 있는지 파악하고, 마케팅 전략 수립에 참고할 인사이트를 수집할 수 있습니다.

이러한 세분화된 그룹은 사용자 그룹 분할(user group split)이라고 하며, 무작위로 구성되고 서로 배타적입니다. 무작위화를 통해 결과에 영향을 미치는 요인이 각 그룹에 균등하게 분배됩니다. 다시 말해, 그룹 간이나 이들의 예상 행동 간에는 본질적인 차이가 없습니다. 따라서 하나의 실험 버전만 특정 사용자 그룹에 적용하고 다른 그룹에는 적용하지 않았을 때, 캠페인 성과의 차이는 그 실험 버전에 기인한 것으로 볼 수 있습니다.

한 번에 여러 실험 버전을 테스트하는 것도 가능하지만, 한 번에 하나의 실험 버전만 테스트할 것을 강력히 권장합니다. 이렇게 하면 관찰된 캠페인 성과 차이에 영향을 준 인과 요인을 분리해낼 수 있습니다.

실험 버전은 캠페인 수준 또는 광고 그룹(ad group) 수준에서 설정됩니다. 광고 그룹은 Ads API의 [line item](https://developer.x.com/en/docs/twitter-ads-api/campaign-management/api-reference/line-items)을 통해 설정합니다. 광고 그룹 수준 실험 버전의 예로, 광고주가 새로운 크리에이티브의 효과를 테스트하고자 하는 경우, 크리에이티브만 서로 다르고 나머지 설정은 동일한 광고 그룹 2개를 포함하는 하나의 캠페인을 생성해야 합니다.

<div id="use-cases">
  ### 사용 사례
</div>

A/B 테스트는 주로 (1) X에서 무엇이 가장 효과적인지 파악해 투자를 최적화하려는 퍼포먼스 광고주의 최적화 사용 사례와 (2) 학습 결과를 바탕으로 마케팅 전략을 수립하려는 브랜드 광고주의 학습 사용 사례를 지원하는 데 활용됩니다. 

API는 다음을 포함한 모든 캠페인 변수를 대상으로 A/B 테스트를 지원합니다:

* 크리에이티브

* 타기팅

* 입찰 방식

* 입찰 단위

<div id="attributes">
  ### 속성
</div>

A/B 테스트는 중첩된 구조로 표현됩니다. A/B 테스트 자체에 대한 최상위 필드와, 각각을 설명하는 필드 집합을 가진 사용자 그룹 객체 배열로 구성됩니다.

전체적으로, 모든 A/B 테스트에는 다음 정보가 포함되어야 합니다.

* 테스트 기간(start&#95;time 및 end&#95;time 필드로 표현됨)

* 실험 분할이 이루어지는 수준(entity&#95;type 필드로 표현됨)

* 최소 두 개(최대 30개)의 사용자 그룹으로, 각 그룹은 user&#95;groups 배열의 객체로 표현됨

각 사용자 그룹에는 *반드시* 다음 정보가 포함되어야 합니다.

* 해당 사용자 그룹에 할당되어야 하는 사용자 비율(size 필드로 표현됨)

* 해당 사용자 그룹의 사용자 풀을 구성해야 하는 campaign ID/line item ID(entity&#95;ids 배열로 표현됨)

선택적으로, A/B 테스트 및 사용자 그룹에 대해 name 및 description 값을 설정할 수 있습니다. 검증 규칙 및 기타 제약 조건에 대한 정보는 아래에서 확인할 수 있습니다.

ID나 생성 시각과 같은 기타 메타데이터도 포함되지만, 이는 X에 의해 자동으로 설정됩니다.

<div id="creating">
  #### 생성
</div>

[POST accounts/:account&#95;id/ab&#95;tests](https://developer.x.com/en/docs/x-ads-api/measurement/api-reference/ab-tests) 엔드포인트를 사용해 A/B 테스트를 생성합니다. 이 엔드포인트는 JSON POST 본문만 허용합니다. Content-Type은 application/json으로 설정해야 합니다.

광고주가 두 개 이상의 캠페인을 설정한 후 A/B 테스트를 생성할 수 있습니다. 앞에서 설명했듯이, A/B 테스트에는 반드시 테스트 기간, 분할 수준(split level), 그리고 최소 두 개의 사용자 그룹이 포함되어야 합니다. 각 사용자 그룹은 자신에게 할당되어야 하는 사용자 비율과, 해당 사용자 풀을 구성할 캠페인 ID들을 선언해야 합니다. 각 항목에 대해서는 아래에서 더 자세히 설명합니다.

테스트 기간:

* start&#95;time 및 end&#95;time 값은 다음 조건을 만족해야 합니다.
  * (A/B 테스트가 생성되는 시점 기준으로) 미래 시점이어야 합니다.
  * 캠페인/라인 아이템 플라이트 기간과 겹쳐야 합니다.
* 앱 기반이 아닌 캠페인의 경우 테스트는 최소 하루 이상이어야 하며, 앱 기반 캠페인의 경우 최소 5일 이상이어야 합니다.

분할 수준:

* entity&#95;type은 CAMPAIGN 또는 LINE&#95;ITEM으로 설정할 수 있습니다.

사용자 그룹:

* 각 사용자 그룹은 user&#95;groups 배열 내의 객체로 표현됩니다.
  * 최소 두 개의 사용자 그룹이 필요합니다.
  * 최대 30개의 사용자 그룹까지 허용됩니다.
* 각 사용자 그룹의 크기는 1.00 이상 99.00 이하의 숫자 값을 문자열로 표현하여 설정합니다.
  * **참고**: 모든 객체의 size 값 합은 반드시 100.00이 되어야 합니다.
* 캠페인 ID는 각 사용자 그룹의 entity&#95;ids 배열에 지정해야 합니다.

필요에 따라 A/B 테스트 또는 하나 이상의 사용자 그룹에 대해 name과 description을 설정할 수 있습니다.

다음 요청은 캠페인 수준에서 4일 동안 진행되는 A/B 테스트를 생성하며, 두 개의 사용자 그룹을 두고 각 그룹에 50%의 사용자를 배정합니다. 첫 번째 사용자 그룹은 f2qcw 및 f2tht 캠페인을 기반으로 하고, 두 번째 사용자 그룹은 f2rqi 및 f2tws 캠페인을 기반으로 합니다. 또한 이 요청은 엔티티의 일부 구성요소에 name과 description을 추가합니다.

**라인 아이템 수준 A/B 테스트용**

캠페인 수준과 라인 아이템 수준에서 A/B 테스트를 할 때의 가장 큰 차이점은 `entity_type` 값입니다. 라인 아이템 수준에서 A/B 테스트를 수행하려면 `entity_type`을 `LINE_ITEM`으로 설정해야 합니다. 이 설정은 아래에 나오는, 이미 생성된 A/B 테스트에 대해 수행하는 모든 액션에 적용됩니다. 

요구 사항:

1. A/B 테스트 캠페인의 모든 라인 아이템은 분할 테스트에 포함되어야 합니다.
2. 라인 아이템 수준에서는 균등 분할만 허용됩니다.
3. 하나의 분할 테스트에 포함될 수 있는 사용자 그룹 라인 아이템 수는 최대 5개입니다. 
4. 사용자 그룹당 하나의 라인 아이템만 허용됩니다.

<div id="updating">
  ### 업데이트
</div>

[PUT accounts/:account&#95;id/ab&#95;tests/:ab&#95;test&#95;id](https://developer.x.com/en/docs/x-ads-api/measurement/api-reference/ab-tests) 엔드포인트를 사용해 A/B Test를 업데이트합니다. 이 엔드포인트는 요청에 JSON blob을 포함해 전송해야 합니다. Content-Type은 application/json으로 설정해야 합니다.

다른 업데이트 엔드포인트와 마찬가지로, [PUT accounts/:account&#95;id/ab&#95;tests/:ab&#95;test&#95;id](/ko/x-ads-api/measurement/ab-testing#get-accounts-account-id-ab-tests) 엔드포인트는 URL에 A/B Test ID를 포함할 것을 요구합니다. 일반적으로 A/B Test는 상태가 SCHEDULED일 때만 업데이트할 수 있습니다. 한 가지 예외로, A/B Test가 LIVE 상태일 때도 end&#95;time은 업데이트할 수 있습니다.

이 엔드포인트는 객체 ID를 포함한 partial JSON을 지원합니다. 다음 원칙이 적용됩니다.

* 객체나 요소를 추가하거나 제거하려면 전체 배열(및 그 하위 구조 전체)을 전달해야 하며, 이는 **치환(replacement)** 작업입니다.

* 그 외의 경우, 키 이름이나 ID를 참조해 기존 *fields*를 수정(변경, 추가, 제거)합니다.

  * 필드를 제거하려면 해당 값을 null로 설정합니다.

  * 전달되지 않은 필드는 수정되지 않습니다.

예를 들어, 앞서 생성한 A/B Test에 세 번째 user group을 추가하려면, 기존의 두 user group 객체와 새로 추가하려는 객체를 모두 포함한 user&#95;groups 배열을 전송해야 합니다. 이를 user&#95;groups 배열을 *다시 생성하는* 것처럼 생각하면 됩니다. 처음부터 이렇게 생성한다고 가정하고 데이터를 전달해야 하며(개별 user group 객체 ID만 전달하지 마십시오), 업데이트 요청에 포함되는 user&#95;groups 배열은 다음과 같이 표현될 수 있습니다.

```json
[
  {
    "entity_ids": [
      "f2qcw",
      "f2tht"
    ],
    "size": "30.0",
    "name": "first group"
  },
  {
    "entity_ids": [
      "f2rqi",
      "f2tws"
    ],
    "size": "30.0",
    "name": "second group",
    "description": "second AB test group"
  },
  {
    "entity_ids": [
      "i1vwr",
      "i1xre"
    ],
    "size": "40.0"
  }
]
```

각 객체의 size 값 합이 여전히 100.00이라는 점에 유의하세요. 만약 첫 두 객체(이전에 각각 50.00으로 설정됨)의 값을 업데이트하지 않았다면, 요청은 실패했을 것입니다.

이와 달리, 첫 번째 사용자 그룹에 설명만 추가하려는 경우, 업데이트 요청의 user&#95;groups 배열은 다음과 같이 표현됩니다.

```json
[
  {
    "id": "p1bcx",
    "description": "PUT 요청을 사용하여 업데이트됨"
  }
]
```

id를 사용해 사용자 그룹 객체를 지정하고, 변경하려는 필드만 포함합니다.
