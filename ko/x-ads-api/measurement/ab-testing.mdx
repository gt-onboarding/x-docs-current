  ## 개요

<div id="introduction">
  ### 소개
</div>

A/B 테스트를 사용하면 광고주가 X에서 도달하는 사용자를 여러 세그먼트로 나누어, 캠페인 성과를 가장 효과적으로 최적화하는 방법을 파악하고 마케팅 전략 수립에 참고할 인사이트를 수집할 수 있습니다.

이러한 세그먼트(사용자 그룹 분할이라고도 함)는 무작위로 구성되며 서로 배타적입니다. 무작위화를 통해 결과에 영향을 미치는 요인이 각 그룹에 고르게 분포됩니다. 즉, 그룹 간 또는 각 그룹의 예상 행동 간에 본질적인 차이가 없습니다. 따라서 하나의 사용자 그룹에만 특정 실험 버전을 적용하고 다른 그룹에는 적용하지 않았을 때, 캠페인 성과의 차이는 그 실험 버전에 기인한다고 볼 수 있습니다.

한 번에 여러 실험 버전을 테스트하는 것도 가능하지만, 한 번에 하나의 버전만 테스트할 것을 강력히 권장합니다. 이렇게 하면 관측된 캠페인 성과 차이를 발생시킨 인과 요인을 분리할 수 있습니다.

실험 버전은 캠페인 단위에서 설정됩니다. 예를 들어 광고주가 새로운 크리에이티브의 효율성을 테스트하고자 하는 경우, 크리에이티브를 제외한 모든 요소가 동일한 두 개의 캠페인을 생성해야 합니다. 향후에는 라인 아이템 단위의 실험 버전도 지원할 예정입니다.

<div id="use-cases">
  ### 활용 사례
</div>

A/B 테스트는 주로 (1) X에서 어떤 요소가 가장 효과적인지 파악해 성과 캠페인 광고주의 투자를 최적화하려는 최적화 목적과, (2) 얻은 인사이트를 마케팅 전략 수립에 활용하려는 브랜드 광고주의 학습 목적을 지원하는 데 사용됩니다. 

API는 다음을 포함한 모든 캠페인 변수를 대상으로 A/B 테스트를 지원합니다:

* 크리에이티브 

* 타기팅 

* 입찰 유형

* 입찰 단위

<div id="ab-testing">
  ## A/B Testing
</div>

A/B Testing은 광고주가 X에서 도달하고 있는 사용자를 세분화하여, 캠페인 성과를 가장 효과적으로 최적화하는 방법을 파악하고 마케팅 전략 수립에 참고할 인사이트를 확보할 수 있도록 해 줍니다.

이러한 세그먼트는 user group split이라고 하며, 무작위로 배정되고 서로 겹치지 않습니다(상호 배타적입니다). 무작위 배정을 통해 결과에 영향을 미치는 요인이 각 그룹에 고르게 분포하게 됩니다. 다시 말해, 그룹 간이거나 각 그룹의 예상 행동 간에 본질적인 차이가 없습니다. 따라서 하나의 variation만 특정 사용자 그룹에 적용하고 다른 그룹에는 적용하지 않았을 때, 캠페인 성과의 차이는 해당 variation에 기인한 것으로 볼 수 있습니다.

한 번에 여러 variation을 테스트하는 것도 가능하지만, 한 번에 하나의 variation만 테스트할 것을 강력히 권장합니다. 이렇게 해야 관측된 캠페인 성과 차이를 유발한 인과 요인을 명확히 분리할 수 있습니다.

Variation은 캠페인 레벨 또는 광고 그룹(ad group) 레벨에서 설정됩니다. 광고 그룹은 Ads API의 [line item](https://developer.x.com/en/docs/twitter-ads-api/campaign-management/api-reference/line-items)을 통해 설정합니다. 광고 그룹 레벨 variation의 예로, 광고주가 새로운 크리에이티브의 효과를 테스트하고자 하는 경우, 하나의 캠페인 안에 크리에이티브만 다른 동일한 광고 그룹 2개를 만들어야 합니다.

### 사용 사례

A/B 테스트는 주로 (1) X에서 어떤 방식이 가장 효과적인지 파악해 투자 효율을 극대화하려는 퍼포먼스 광고주를 위한 최적화 사용 사례와, (2) 마케팅 전략 수립에 활용할 인사이트를 얻으려는 브랜드 광고주를 위한 학습 목적의 사용 사례를 지원하는 데 사용됩니다. 

API는 다음을 포함한 모든 캠페인 변수에 대해 A/B 테스트를 지원합니다:

* 크리에이티브

* 타게팅

* 입찰 유형

* 입찰 단위

<div id="attributes">
  ### 속성
</div>

A/B 테스트는 중첩 구조로 표현됩니다. A/B 테스트 자체에 대한 최상위 필드와, 각각을 설명하는 필드 세트를 가진 사용자 그룹 객체 배열로 구성됩니다.

개괄적으로, 모든 A/B 테스트에는 다음 정보가 포함되어야 합니다.

* 테스트 기간: start&#95;time 및 end&#95;time 필드로 표현됩니다.

* 분할이 발생하는 수준: entity&#95;type 필드로 표현됩니다.

* 최소 두 개(최대 30개)의 사용자 그룹: 각각 user&#95;groups 배열의 객체로 표현됩니다.

각 사용자 그룹에는 다음 정보가 *반드시* 포함되어야 합니다.

* 해당 사용자 그룹에 할당되어야 하는 사용자 비율: size 필드로 표현됩니다.

* 해당 사용자 그룹의 사용자 풀을 구성해야 하는 캠페인 ID/라인 아이템 ID: entity&#95;ids 배열로 표현됩니다.

선택적으로, A/B 테스트와 사용자 그룹에 대해 name 및 description 값을 설정할 수 있습니다. 검증 규칙 및 기타 제약 조건에 대한 정보는 아래에서 확인할 수 있습니다.

ID나 생성 시각과 같은 기타 메타데이터도 포함되지만, 이는 X에 의해 자동으로 설정됩니다.

아래 섹션에서는 A/B 테스트를 생성하고 업데이트하는 방법을 설명합니다. 조회 및 삭제 작업은 다른 Ads API 엔드포인트와 동일하게 동작합니다.

<div id="creating">
  #### 생성
</div>

[POST accounts/:account&#95;id/ab&#95;tests](https://developer.x.com/en/docs/x-ads-api/measurement/api-reference/ab-tests) 엔드포인트를 사용해 A/B 테스트를 생성합니다. 이 엔드포인트는 JSON 형식의 POST 본문만 허용합니다. `Content-Type`은 `application/json`으로 설정해야 합니다.

광고주가 두 개 이상의 캠페인을 설정한 후에 A/B 테스트를 생성할 수 있습니다. 위에서 언급했듯이, A/B 테스트에는 반드시 테스트 기간, 분할 수준(split level), 그리고 최소 두 개의 사용자 그룹이 포함되어야 합니다. 각 사용자 그룹은 해당 그룹에 할당할 사용자 비율과, 해당 그룹의 사용자 풀을 구성하는 캠페인 ID들을 선언해야 합니다. 각각에 대해서는 아래에서 더 자세히 설명합니다.

테스트 기간:

* `start_time` 및 `end_time` 값은 다음을 만족해야 합니다.

  * (A/B 테스트가 생성되는 시점 기준으로) 미래 시점이어야 합니다.

  * 캠페인/라인 아이템의 집행 기간과 겹쳐야 합니다.

* 테스트는 앱 기반이 아닌 캠페인의 경우 최소 1일, 앱 기반 캠페인의 경우 최소 5일 동안 진행되어야 합니다.

분할 수준(split level):

* `entity_type`은 `CAMPAIGN` 또는 `LINE_ITEM`으로 설정할 수 있습니다.

사용자 그룹:

* 각 사용자 그룹은 `user_groups` 배열 내의 객체로 표현됩니다.

  * 최소 두 개의 사용자 그룹이 필요합니다.

  * 최대 30개의 사용자 그룹까지 허용됩니다.

* 각 사용자 그룹의 크기는 1.00에서 99.00 사이의 숫자 값을 문자열로 표현하여 설정합니다.

  * **참고**: 모든 객체의 크기(size) 값 합계는 반드시 100.00이 되어야 합니다.

* 캠페인 ID는 각 사용자 그룹의 `entity_ids` 배열에 지정해야 합니다.

선택적으로, A/B 테스트 자체 또는 하나 이상의 사용자 그룹에 대해 `name`과 `description`을 설정할 수 있습니다.

**라인 아이템 수준 A/B 테스트용**

캠페인 수준과 라인 아이템 수준에서의 A/B 테스트의 주요 차이점은 `entity_type` 값입니다. 라인 아이템 수준에서 A/B 테스트를 진행하려면 `entity_type`을 `LINE_ITEM`으로 설정해야 합니다. 아래에 나와 있는, 이미 생성된 A/B 테스트에 대한 모든 작업에 이 설정이 적용됩니다. 

요구 사항:

1. A/B 테스트 캠페인에 속한 모든 라인 아이템이 분할 테스트에 포함되어야 합니다.
2. 라인 아이템 수준에서는 동일 비율로만 분할할 수 있습니다.
3. 하나의 분할 테스트에 포함될 수 있는 사용자 그룹용 라인 아이템 수는 최대 5개입니다. 
4. 사용자 그룹당 라인 아이템은 1개만 허용됩니다.

<div id="updating">
  ### 업데이트
</div>

[PUT accounts/:account&#95;id/ab&#95;tests/:ab&#95;test&#95;id](https://developer.x.com/en/docs/x-ads-api/measurement/api-reference/ab-tests) 엔드포인트를 사용해 A/B 테스트를 업데이트합니다. 이 엔드포인트를 호출할 때는 요청에 JSON blob 데이터를 포함해야 합니다. Content-Type은 application/json으로 설정해야 합니다.

다른 업데이트 엔드포인트와 마찬가지로 [PUT accounts/:account&#95;id/ab&#95;tests/:ab&#95;test&#95;id](/ko/x-ads-api/measurement/ab-testing#get-accounts-account-id-ab-tests) 엔드포인트는 URL에 A/B 테스트 ID를 포함할 것을 요구합니다. 일반적으로 A/B 테스트는 상태가 SCHEDULED일 때만 업데이트할 수 있습니다. 한 가지 예외로, LIVE 상태일 때 end&#95;time을 업데이트하는 것은 가능합니다.

이 엔드포인트는 객체 ID를 사용하는 부분 JSON을 지원합니다. 다음 원칙이 적용됩니다.

* 객체나 요소를 추가하거나 제거하려면 전체 배열(및 그 하위 구조)을 전달해야 하며, 이는 **대체** 작업입니다.

* 그 외의 경우에는 키 이름이나 ID를 참조하여 기존 *필드*를 수정(변경, 추가, 제거)합니다.

  * 필드를 제거하려면 해당 값을 null로 설정합니다.

  * 전달되지 않은 필드는 수정되지 않습니다.

예를 들어, 앞에서 생성한 A/B 테스트에 세 번째 사용자 그룹을 추가하려면, 기존 두 개의 사용자 그룹 객체와 추가하려는 새 사용자 그룹 객체가 모두 포함된 user&#95;groups 배열을 전송해야 합니다. 이를 user&#95;groups 배열을 *재생성*한다고 생각하면 됩니다. 애초에 이렇게 생성하는 것처럼 데이터를 전달해야 하며(사용자 그룹 객체 ID는 전달하지 마십시오), 업데이트 요청의 user&#95;groups 배열은 다음과 같이 표현될 수 있습니다.

```json
[
  {
    "entity_ids": [
      "f2qcw",
      "f2tht"
    ],
    "size": "30.0",
    "name": "first group"
  },
  {
    "entity_ids": [
      "f2rqi",
      "f2tws"
    ],
    "size": "30.0",
    "name": "second group",
    "description": "second AB test group"
  },
  {
    "entity_ids": [
      "i1vwr",
      "i1xre"
    ],
    "size": "40.0"
  }
]
```

객체 전체의 size 값 합이 여전히 100.00이 된다는 점에 유의하세요. 앞의 두 객체(이전에 각각 50.00으로 설정되어 있었던 값)를 업데이트하지 않았다면, 요청은 실패했을 것입니다.

반대로, 첫 번째 사용자 그룹에 설명만 추가하려는 경우라면, 업데이트 요청의 user&#95;groups 배열은 다음과 같이 표현됩니다.

```json
[
  {
    "id": "p1bcx",
    "description": "PUT 요청을 사용하여 업데이트됨"
  }
]
```

user group 객체를 id로 참조하고, 수정하려는 필드만 포함합니다.
