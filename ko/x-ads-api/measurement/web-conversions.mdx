<div id="ads-api-access-new-applications">
  #### **Ads API Access - 신규 애플리케이션**
</div>

* Conversion API의 기본 요건은 [개발자 계정](https://developer.x.com/en/portal/dashboard)과 [Ads API Access](https://developer.x.com/en/docs/twitter-ads-api/getting-started)를 보유하는 것입니다. 전체 절차는 [Ads API 시작하기](https://developer.x.com/en/docs/twitter-ads-api/getting-started) 가이드에 설명되어 있습니다. 아래 사항을 참고하십시오:

**Step 1: 개발자 계정**

* [개발자 계정](https://developer.x.com/en/portal/dashboard)을 신청할 때는, 즉시 승인을 받기 위해 구독 [플랜](https://developer.x.com/en/portal/products/) 중 하나에 신청하십시오. 
* **참고**: 모범 사례로서, 공식 회사 X 핸들을 사용해 개발자 계정을 생성하고 Ads API 액세스를 신청할 것을 강력히 권장합니다. 개발자 계정이 개발자 개인 핸들과 연결된 경우, 필요 시 해당 자격 증명을 이전할 방법이 없습니다. 지속적인 관리를 위해 회사 계정으로 관리하고, 필요에 따라 [Multi-user login](https://business.x.com/en/help/troubleshooting/multi-user-login-faq.html)을 활용하는 것이 가장 좋습니다. 그렇지 않은 경우 최소한 계정은 기본값이 아닌 설정(헤더 이미지, 아바타, 프로필 설명, 프로필 URL)으로 구성하고, 2단계 인증을 사용해야 합니다.

**Step 2: Ads API 애플리케이션**

* Ads API 애플리케이션에 사용할 올바른 **App** **ID**를 준비했는지 확인하십시오. App ID는 [Developer Portal](https://developer.x.com/en/portal/dashboard)의 Projects &amp; Apps에서 확인할 수 있습니다. 예: 16489123
* X 담당자에게 연락하여 Ads API 액세스를 요청하십시오.

<div id="ads-api-access-existing-applications">
  #### **Ads API 액세스 - 기존 앱**
</div>

* 이미 현재 사용 중인 Ads API 앱이 있다면, 해당 앱과 기존 액세스 토큰을 그대로 Conversion API에 사용할 수 있습니다.

<div id="access-tokens">
  #### **Access Tokens**
</div>

* Ads API 애플리케이션을 소유한 사용자 핸들에 대한 User Access Token은 개발자 포털에서 직접 생성하고 조회할 수 있습니다. 이는 본인의 X 핸들에 사용하도록 설계되었기 때문에 “personal access token”이라고 부릅니다. 인증 및 개발자 포털에 대한 전반적인 정보는 [여기](/ko/resources/fundamentals/authentication/overview)에서 확인할 수 있습니다.
* Ads API 애플리케이션을 소유한 핸들이 아닌 다른 사용자 핤들의 User Access Tokens는 3-legged OAuth 플로우로 생성해야 합니다. 3-legged OAuth를 사용하여 Access Token을 생성하는 옵션은 다음과 같습니다.
  * [twurl 유틸리티를 사용한 웹 기반 인가 방식의 명령줄](https://github.com/twitter/twurl#getting-started)
  * [PIN 기반 인가 방식의 명령줄](https://developer.x.com/resources/fundamentals/authentication/pin-based-oauth)
  * [3-legged OAuth 패턴을 구현한 커스텀 웹 플로우](https://developer.x.com/resources/fundamentals/authentication/obtaining-user-access-tokens)
* Conversion API와 함께 사용하는 모든 사용자 토큰은 AD&#95;MANAGER 또는 ACCOUNT&#95;ADMIN 액세스 레벨을 가진 사용자에 대한 것이어야 하며, 이는 [authenticated&#95;user&#95;access](https://developer.x.com/en/docs/twitter-ads-api/campaign-management/api-reference/authenticated-user-access) 엔드포인트를 통해 확인할 수 있습니다.
* **참고:** 위와 같이 생성된 토큰 자체는 AD&#95;MANAGER 또는 ACCOUNT&#95;ADMIN 액세스 레벨이 없는 사용자에게도 사용 목적으로 공유할 수 있습니다.

<div id="creating-the-conversion-api-event">
  ### Conversion API 이벤트 생성
</div>

Conversion API를 사용하려면 Ads Manager에서 새로운 전환 이벤트를 생성하거나, 이미 생성되어 X Pixel과 함께 사용 중인 기존 이벤트를 사용해야 합니다. Pixel 이벤트와 Conversion API 이벤트 간에 이벤트 중복 제거(deduplication)를 하려는 경우, Pixel용으로 생성한 기존 이벤트를 그대로 사용해야 합니다. 

<div id="option-1-using-an-existing-conversion-event-in-ads-manager">
  ##### 옵션 1: Ads Manager에서 기존 전환 이벤트 사용
</div>

이미 X 픽셀과 함께 사용 중인 기존 이벤트를 사용하려면, 해당 이벤트의 Event ID를 가져와 사용하면 됩니다. 동일한 이벤트에 대해 픽셀과 Conversion API를 모두 사용하는 경우, 같은 이벤트에 대해 픽셀과 Conversion API 간 이벤트를 중복 제거할 수 있도록 픽셀 코드 스니펫과 Conversion API 요청 모두에서 중복 제거 키(`conversion_id`)를 동일하게 사용해야 합니다. 자세한 내용은 d. Testing Events and Deduplication 섹션을 참고하세요. 

<div id="option-2-creating-a-new-conversion-event-in-ads-manager">
  ##### 옵션 2: Ads Manager에서 새 전환 이벤트 생성하기:
</div>

이벤트를 생성하기 전에 Events Manager에서 Event Source를 먼저 만들어 두는 것이 중요합니다. 계정에 Event Source(X Pixel)가 추가되어 있는지 확인하려면 Events Manager로 이동한 후 왼쪽 메뉴에 X Pixel이 있는지 확인하세요. 

아직 Event Source가 추가되어 있지 않다면, 아래 단계를 따라 Event Source를 생성하세요.

1. ads.x.com으로 이동합니다.
2. 왼쪽 상단의 **Tools** 섹션으로 이동한 후 **Events Manager**를 클릭합니다.
3. 왼쪽 사이드바에 아직 X Pixel Event Source가 없다면, 오른쪽 상단에서 **Add event source**를 선택해 Event Source를 추가합니다.
   1. X Pixel Event Source의 ID가 바로 **Pixel ID**입니다.

이제 Event Source와 Pixel ID가 준비되었습니다. 추적하려는 전환 이벤트를 위해 해당 Event Source 내에 이벤트를 생성해야 합니다:

1. X Pixel Event Source 안에서, 오른쪽에 있는 **Add events**를 선택합니다.
2. **Conversion API로 설치(Install with Conversion API)**를 선택합니다.
3. API에서 사용될 이 이벤트의 **Pixel ID**와 **Event ID**가 표시됩니다.
   1. 이 이벤트의 ID가 Event ID입니다.
4. **Save**를 클릭하면 전환 이벤트가 생성되며 바로 사용할 수 있습니다.

<div id="preparing-identifiers-for-conversion-events">
  ### 전환 이벤트를 위한 식별자 준비
</div>

현재 Click ID(`twclid`), 이메일 주소, 전화번호 중 최소 하나의 식별자를 전달해야 합니다. IP 주소나 User Agent를 사용하는 경우, 정확한 전환 매칭을 위해 두 번째 식별자도 함께 전송해야 합니다.

더 많은 식별자를 전달할수록 전환 매칭률이 높아집니다.

|     |     |     |
| :--- | :--- | :--- |
| 고객 매칭 필드 | 형식 | 해시 처리 필요 여부 |
| X Click ID | X에서 생성됨 ([자세히 알아보기](https://business.x.com/en/help/campaign-measurement-and-analytics/conversion-tracking-for-websites.html#faq)) | 아니요  |
| 이메일 주소 | 앞뒤 공백 제거 | 필요 (SHA256) |
| 전화번호 | [E164 표준](https://en.wikipedia.org/wiki/E.164) | 필요 (SHA256) |

|     |     |     |
| :--- | :--- | :--- |
| IP 주소 | 앞뒤 공백 제거 | 아니요  |
| User Agent | 앞뒤 공백 제거 | 아니요  |

<div id="1-prepare-x-click-id-identifier">
  #### 1. X Click ID 식별자 준비
</div>

전환 요청에 항상 Click ID를 포함하는 것을 권장합니다.

사용자가 목적지 웹사이트로 이동한 후, 가능한 경우 쿼리 문자열 매개변수 `twclid` 에서 Click ID를 추출해야 합니다. 

기본 JavaScript 코드 예시:

```js
var queryString = document.location.search;
if (queryString.has('twclid') {
  twitterClickID = getParam(queryString, 'twclid');
  // 권장 다음 단계: 로깅, 로컬 스토리지에 삽입
}
```

다음을 권장합니다. 

1. URL 쿼리 매개변수에 `twclid` 값이 포함되어 있으면 항상 이를 파싱합니다.

2. 해당 데이터를 관련 폼 필드나 전환 이벤트 정보와 함께 저장합니다.

Click ID를 전환 이벤트 및 워크플로 정보와 연계하면 배치 처리, 여러 웹사이트 내비게이션 흐름을 기반으로 전환 이벤트를 분석·생성하는 알고리즘, 대량 업로드와 같은 시나리오를 구현할 수 있습니다.

Event Source URL은 [URL 인코딩](https://en.wikipedia.org/wiki/Percent-encoding)된 값이어야 하며, 이벤트를 트리거한 웹페이지를 나타냅니다.

<div id="2-prepare-email-identifier">
  #### 2. 이메일 식별자 준비
</div>

지원되는 고객 매칭 필드는 전송할 수 있지만, 전송 전에 정규화하고, 필요한 경우 개인정보 보호를 위해 해시 처리해야 합니다.

이 정보는 salt를 사용하지 않고 SHA256으로 해시해야 합니다. 

예를 들어, 이메일 주소 test@x.com의 경우, 다음과 같은 해시 값 형식으로 전송해야 합니다: d360d510a224510f373931ce2d6215a799f5a9c1cef221b0149b6b6b50cced62.

<div id="3-prepare-phone-identifier">
  #### 3. 전화번호 식별자 준비
</div>

전화번호는 [E.164 표준](https://en.wikipedia.org/wiki/E.164) 형식으로 전달해야 하며, 이 정보는 솔트 없이 SHA-256으로 해시해야 합니다. 

예를 들어 미국 전화번호 +11234567890의 경우, 다음과 같이 해시된 값으로 보내야 합니다: 1fa6b8d986d9b9cd01bf36951815158bbde9f520c0567c835dfe34783d0a4231.

<div id="4-prepare-ip-address-identifier">
  #### 4. IP 주소 식별자 준비
</div>

IP 주소는 다른 식별자(twclid, 이메일 주소, 전화번호 또는 user agent)와 함께 전송해야 합니다. 이 식별자에는 해싱이 필요하지 않습니다.

이 값은 점으로 구분된 10진 표기법으로 표현되며, 마침표로 구분된 네 개의 숫자로 구성됩니다. 예를 들어, 사용자의 IP 주소는 8.25.197.25일 수 있습니다.

<div id="5-prepare-user-agent-identifier">
  #### 5. User Agent 식별자 준비
</div>

User Agent는 다른 식별자(twclid, 이메일 주소, 전화번호 또는 IP 주소)와 함께 전달되어야 합니다. 이 식별자에 대해서는 해싱이 필요하지 않습니다.

이 식별자를 사용하면 서버가 요청을 보내는 User Agent의 애플리케이션, 운영 체제, 공급업체 및/또는 버전을 식별할 수 있습니다. 예를 들어, 이 값은 Mozilla/5.0 (Macintosh; Intel Mac OS X 10&#95;15&#95;7) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/127.0.0.0 Safari/537.36처럼 전달할 수 있습니다.

<div id="constructing-the-conversion-event-request">
  ### 전환 이벤트 요청 구성
</div>

`POST: version/measurement/conversions/:pixel_id`

특정 광고 계정에 대한 전환 이벤트를 전송합니다. 응답 코드가 성공(HTTP 200 OK)인지 확인해야 합니다. 오류 코드가 반환될 경우를 대비해 재시도 메커니즘과 기본 로깅을 구현할 것을 권장합니다.

엔드포인트의 URL과 POST 요청 본문 매개변수에 대한 자세한 내용은 [API Reference](https://developer.x.com/en/docs/twitter-ads-api/measurement/api-reference/conversions) 섹션을 참고하세요.

<div id="rate-limit">
  ### Rate Limit
</div>

요청 한도는 15분마다 계정당 60,000개의 이벤트입니다.

다음과 같이 이 요청 호출 외부에서 서버 코드에 별도의 로직을 구현해야 할 수도 있다는 점에 유의하세요.

1. 각 이벤트마다 올바른 전환 데이터를 전송할 수 있도록 사용자 행동을 계측(로그 기록)하는 기능 구현

2. 관련 개인정보 보호 선택권을 행사한 사용자의 전환 이벤트를 필터링하기 위한 필요한 로직 구현 - 예를 들어, 광고주 웹사이트에서 추적이나 개인정보 판매를 거부(옵트아웃)한 경우

3. 이벤트를 수집하고 전환을 전송할 수 있도록 이벤트 트리거 및 페이지와의 연동

  ### 이벤트 테스트와 중복 방지
</div>

<div id="testing-events">
  ### 이벤트 테스트
</div>

이벤트가 전환 이벤트를 성공적으로 수신하면, 12~24시간 이내에 Ads Manager의 전환 추적 페이지에서 ‘Single event web tag’의 상태가 TRACKING으로 표시되어야 합니다. Conversion API를 통해 전환을 전송하더라도 진행 중인 캠페인에는 영향을 주지 않습니다. 

다음 방법으로 태그 ID별 전환 이벤트의 분석 결과를 확인할 수도 있습니다.

* Ads Manager에서 데이터 내보내기 ([웹사이트 전환 추적 분석 도움말 페이지](https://business.x.com/en/help/campaign-measurement-and-analytics/conversion-tracking-for-websites.html))

* Ads API를 통해 데이터 내보내기 (segmentation&#95;type=CONVERSION&#95;TAGS) 

<div id="duplication-between-pixel-and-conversion-api">
  ### Pixel과 Conversion API 간 중복 처리
</div>

Pixel과 Conversion API 요청 간 전환을 중복 제거하려면 `conversion_id`를 중복 제거 키로 사용할 수 있습니다. 중복 제거는 이벤트 단위에서만 수행됩니다. 즉, Pixel과 CAPI 요청 간에 중복 제거를 하려면 광고주는 동일한 `conversion_id`를 사용하는 것뿐 아니라 Pixel 요청과 CAPI 요청 모두에서 **동일한 이벤트**를 사용해야 합니다. 중복 제거는 48시간 이내에 수신된 이벤트에만 가능합니다.

<div id="conversion-tracking-overview">
  ## 전환 추적 (개요)
</div>

<div id="summary">
  ### 요약
</div>

전환 추적을 사용하면 X에서 광고를 보고 상호작용한 후 원하는 행동을 수행한 X 사용자 수를 측정할 수 있습니다. 이를 통해 어떤 캠페인이 사이트 방문, 가입, 구매와 같은 행동을 유도하는지 측정할 수 있습니다. 이는 광고주가 X 외부에서 다이렉트 리스폰스 광고의 성과를 측정해, 비용 효율적으로 고객을 확보할 수 있도록 해 줍니다.

전환 태그를 사용하면 광고주는 사용자의 전환을 추적하고, 이를 X상의 광고 캠페인과 연결할 수 있습니다. 이를 통해 캠페인을 최적화하여 목표로 하는 고객 획득당 비용(CPA) 목표를 달성할 수 있는 가시성을 확보할 수 있습니다.

광고주는 전환 추적을 통해 다양한 웹사이트 내 행동을 측정할 수 있습니다. 광고 캠페인을 통해 유도하려는 행동에 따라 하나 이상을 선택할 수 있습니다.

* 사이트 방문: 사용자가 광고주의 사이트 내 랜딩 페이지를 방문하는 경우
* 구매: 사용자가 광고주의 사이트에서 제품 또는 서비스를 구매 완료하는 경우
* 다운로드: 사용자가 광고주의 사이트에서 백서나 소프트웨어 패키지와 같은 파일을 다운로드하는 경우
* 가입: 사용자가 광고주의 서비스, 뉴스레터 또는 이메일 수신에 가입하는 경우
* 사용자 정의: 위 카테고리에 포함되지 않는 사용자 정의 행동을 위한 포괄 카테고리

X의 전환 추적은 광고주에게 전환 기여를 전체적으로 파악할 수 있는 시야를 제공합니다. 고유 클릭 URL과 서드파티 트래킹 태그 조합과 같이, X 고유의 전환 추적 기능 대신 서드파티 측정 시스템을 사용해 왔을 수 있는 광고주가 있는 경우와 비교하면, X의 전환 태그는 트윗 펼쳐보기, 리트윗, 좋아요, 답글, 팔로우뿐 아니라 노출(impression)과 같은 퍼널 상·중단계에서의 참여에 기인한 전환까지 광고주가 추적할 수 있도록 해 줍니다.

    캠페인 세분화가 세밀할수록 전환 리포트 결과를 더 유의미하게 활용할 수 있습니다. 예를 들어, 50개의 키워드를 사용하는 캠페인보다 4개의 키워드를 사용하는 캠페인이 훨씬 최적화하기 쉽습니다.

<div id="401-unauthorized">
  ### 401 Unauthorized
</div>

**이유**: 인증 자격 증명이 없거나 올바르지 않음 

**해결 방법**: 다음 3가지 인증 방법 중 하나를 사용하여 [Set Up 문서](https://developer.x.com/en/docs/twitter-ads-api/measurement/guides/conversion-api)에 나와 있는 인증 단계를 따르십시오:

Ads API 앱을 소유한 핸들이 아닌 다른 사용자 핸들에 대한 User Access Tokens(사용자 액세스 토큰)는 3-legged OAuth 플로우로 생성해야 합니다. 3-legged OAuth를 사용하여 액세스 토큰을 생성하는 옵션에는 다음이 포함됩니다.

* [twurl 유틸리티를 사용한 웹 기반 승인 방식의 명령줄](https://github.com/twitter/twurl#getting-started)

* [PIN 기반 승인을 사용하는 명령줄](https://developer.x.com/resources/fundamentals/authentication/pin-based-oauth)

* [3-legged OAuth 패턴을 구현하는 커스텀 웹 플로우](https://developer.x.com/resources/fundamentals/authentication/obtaining-user-access-tokens)

Conversion API에 사용되는 모든 사용자 토큰은 AD&#95;MANAGER 또는 ACCOUNT&#95;ADMIN 액세스 수준*을 가진 사용자에 대한 것이어야 하며, 이는 [authenticated&#95;user&#95;access](https://developer.x.com/en/docs/twitter-ads-api/campaign-management/api-reference/authenticated-user-access) 엔드포인트를 통해 확인할 수 있습니다.

`POST version/measurement/conversions/:pixel_id`

단일 Event Tag ID에 대한 웹사이트 전환 이벤트를 전송합니다.

응답 코드가 성공인지(HTTP 200 OK) 확인해야 합니다. 오류 코드가 반환되는 경우를 대비해 재시도 로직과 기본 로깅을 구현할 것을 권장합니다.

요청 한도는 계정당 15분마다 100,000개의 요청입니다(각 요청에는 최대 500개의 이벤트를 포함할 수 있습니다).
