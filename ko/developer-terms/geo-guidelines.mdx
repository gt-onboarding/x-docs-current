마지막 업데이트: 2014년 10월 22일.

<div id="introduction">
  ## 소개
</div>

Posting With Location은 X API의 위치 태그(geotagging) 기능입니다. 이 기능은 게시물에 위치 맥락을 더해, 사용자에게 더 의미 있는 경험을 제공하는 데 도움이 됩니다. 예를 들어 “Fireworks!”라는 문구는 위치와 함께할 때 훨씬 더 의미가 분명해집니다. 예: “Fireworks!” — 미국 캘리포니아주 샌프란시스코의 SoMa Neighborhood에서. 이 기능을 중심으로 더 뛰어난 사용자 경험을 만들기 위해 개발자 생태계와 함께 협력하길 기대합니다!

<div id="user-privacy-and-user-control-of-their-location-information">
  ## 사용자 프라이버시 및 위치 정보에 대한 사용자 제어
</div>

안전하고 좋은 사용 경험을 제공하기 위해, 개발자는 사용자의 위치 정보를 다룰 때 사용자 프라이버시를 특히 유의해야 하며, “위치와 함께 게시하기(Posting with Location)” 환경에서 기본적인 일관성을 유지해야 합니다. 다음은 사용자 프라이버시를 보호하고 사용자가 자신의 행동이 미치는 영향을 이해할 수 있도록 하기 위해 유념해야 할 몇 가지 핵심 사항입니다:

<div id="general-guidelines">
  ## 일반 지침
</div>

* 사용자는 위치와 함께 게시하기(Posting With Location) 기능을 사용하려면 옵트인(위치 “켜기”)해야 합니다.
* 사용자의 게시물에 정확한 위치 정보가 표시되려면, 사용자가 이에 명시적으로 동의해야 합니다. 클라이언트의 위치 기능이 “sticky”(즉, 한 번 위치를 켜면 이후 게시물에 항상 위치가 함께 표시되는 방식)인 경우, 장소 이름(place&#95;ids)만 표시하도록 하고, 게시물 단위 또는 별도의 설정을 통해 사용자가 위도·경도와 같은 정확한 위치 공개에 다시 한 번 옵트인할 수 있도록 하는 방식을 고려해야 합니다.
* 사용자의 위치 데이터를 이용해 게시물을 올릴 때, 사용자가 명시적으로 허용하지 않는 한, 해당 정보를 게시물과 별도로 저장해서는 안 됩니다.
* 사용자의 게시물에 위치 정보가 표시되는 경우, 어떤 수준의 위치 정보가 표시되는지 사용자에게 명확히 전달되어야 합니다. 이를 구현하는 한 가지 방법은, 게시물 발행 전에 해당 정보가 어떻게 표시되는지 보여주는 지도를 노출하는 것입니다. 클라이언트에서 사용자가 발행하는 정보에 정확한 위도와 경도가 포함되거나, 그 정보만 포함되는 경우, 사용자의 정확한 좌표가 게시된다는 사실이 사용자에게 분명하게 전달되어야 합니다.
* 사용자는 게시물을 작성할 때마다 위치를 켜거나 끌 수 있어야 합니다.
* X의 [Developer Policy](/ko/developer-terms/policy.html)를 준수해야 합니다. 여기에는 게시물의 일부인 경우를 제외하고, X API로부터 획득한 위치 데이터 및 기타 지리 정보를 집계, 캐시 또는 저장해서는 안 되며, 이러한 위치 데이터나 지리 정보를 단독으로 사용해서도 안 된다는 요구 사항이 포함됩니다.

<div id="specific-examples">
  ## 구체적인 예시
</div>

예시 1: *위도와 경도를 명시적으로 추가하지 않는 경우(현재 X Web Client UI)*

1. 사용자가 게시물을 작성할 때 “위치 추가” 링크를 표시합니다.
2. 백그라운드에서 X [reverse&#95;geocode](https://developer.x.com/en/docs/geo/places-near-location/api-reference/get-geo-reverse_geocode.html) API 호출을 사용하여 위도, 경도, 위치 정확도, 사용자가 선호하는 세부 수준(기본값은 “neighborhood”)을 전달합니다.
3. 기본 장소의 이름을 표시합니다(X는 X 클라이언트가 별도 수정 없이 목록을 그대로 표시할 수 있도록, 사용자에게 자연스러운 순서로 정렬하려고 시도합니다).
4. 사용자가 리스트에서 다른 위치를 선택할 수 있도록 합니다(예: “SoMa, San Francisco”에서 “South Beach, San Francisco”로 변경).
5. 중요: 정확한 위치 정보를 표시하는 경우 사용자에게 이를 투명하게 알리십시오. 정확한 위치 정보가 표시되는지 사용자가 분명히 알 수 있도록 하십시오.
6. 사용자가 게시물을 작성할 때마다 위치를 켜거나 끌 수 있는 선택권을 제공하십시오.

예시 2: *기존 지오태깅 구현*

여러 X 클라이언트 앱에는 원래의 지오태깅 기능(place&#95;ID를 제공하지 않고 위도/경도만 전송하는 방식)이 이미 구현되어 있습니다. API의 기본 동작은 이러한 동작을 그대로 따릅니다. 즉, 개발자로서 현재의 동작을 유지하기 위해 기존 앱을 변경할 필요가 없습니다. 실제로 X에 위도/경도 좌표만 전달하더라도, X는 해당 위도와 경도를 자동으로 역지오코딩하여, 데이터가 있는 영역에 대해서는 위도/경도와 함께 place&#95;ID도 표시합니다.

사용자의 정확한 좌표가 X에서 사용되고 표시된다는 점을 사용자에게 투명하게 알리십시오. 기존 앱에 대한 모범 사례는 사용자에게 게시물 단위로 위치를 켜거나 끌 수 있는 옵션을 제공하는 것입니다. 또한 기본 “고정(sticky)” 설정으로, 정확한 좌표 대신 place&#95;ID만 공유하도록 사용자가 선택할 수 있도록 할 수도 있습니다.

<div id="x-storage-of-user-location-data">
  ## X의 사용자 위치 데이터 저장
</div>

X는 클라이언트에서 전달된 데이터를 다음과 같이 저장합니다.

* X는 사용자가 자신의 팔로워에게 공개하기로 선택한 모든 데이터를 저장합니다. 이는 X가 게시물이 작성된 시점을 나타내는 타임스탬프를 저장하는 방식과 유사합니다. 즉, 사용자가 게시물을 올릴 때 자신의 정확한 좌표를 포함하면, 게시물이 존재하는 동안 X는 해당 게시물과 함께 이 정확한 위치 정보를 저장합니다. 사용자는 [여기](https://support.twitter.com/articles/122236-how-to-tweet-with-your-location)에 설명된 대로 설정 페이지에서 ‘내 위치 기록 지우기’ 버튼을 클릭하여 게시물의 위치 데이터를 삭제할 수 있습니다.

<div id="summary">
  ## 요약
</div>

게시물에 위치 정보를 추가하면 사용자 관점에서 상당한 맥락과 가치를 부여할 수 있습니다. 다만, 클라이언트에 위치 기능을 구현할 때에는 위치 중심의 제품을 설계하는 과정에서 개인 정보 보호와 사용자에 대한 투명성을 핵심적인 고려 사항으로 삼아야 합니다.

개인 정보 보호, 위치, 기타 위치 기반 기능에 관해 질문이나 의견이 있다면 [문의해 주세요.](https://support.twitter.com/forms/platform)
