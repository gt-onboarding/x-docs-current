<div id="sensitive-information">
  ## 민감한 정보
</div>

X 사용자의 민감할 수 있는 특성을 도출하거나 추론하기 위해 X 데이터를 사용할 때에는 주의해야 합니다. 다음과 같은 X 사용자의 정보를 도출·추론하거나, 그렇게 도출·추론된 정보를 저장해서는 안 됩니다.

* 건강 상태(임신 포함)
* 부정적인 재정 상태 또는 재정적 상황
* 정치적 성향 또는 신념
* 인종 또는 민족적 출신
* 종교적 또는 철학적 성향 또는 신념
* 성생활 또는 성적 지향
* 노동조합 가입 여부
* 범죄 혐의 또는 실제 범죄 행위

개인 데이터(예: 사용자 id, 사용자 이름 및 기타 식별자)를 저장하지 않는 X 콘텐츠에 대한 집계 분석은, 해당 분석이 관련 법령과 Developer Agreement 및 Policy의 모든 조항을 준수하는 경우에 한해 허용됩니다.

<div id="off-x-matching">
  ## X 외부 매칭
</div>

X 외부 매칭은 X 콘텐츠(X 사용자 이름 또는 사용자 ID 포함)를 사람, 가구, 기기, 브라우저 또는 기타 X 외부 식별자와 연관시키는 것을 의미합니다. 한 가지 예로, X 사용자 이름을 비즈니스의 고객 레코드와 연관시키는 경우가 있습니다(예: 고객 레코드의 “John Doe”가 X의 @johndoe와 매칭되는 경우).

우리는 사람들이 X에서 별도의 신원을 만들고, 원한다면 가명 신원까지도 편안하게 사용할 수 있기를 바랍니다. X 사용자에 관한 어떤 정보를 X 외부 식별자와 연관시키려는 경우, 그 연관을 만들기 전에 해당 사용자로부터 명시적 옵트인 동의를 받아야 합니다. 예를 들어, 사용자가 귀하의 서비스 가입 절차의 일환으로 자신의 X 핸들을 직접 제공하는 경우, 이러한 동의를 얻을 수 있습니다.

사용자의 X 신원을 X 외부 식별자와 연결하는 것에 대해 사용자의 명시적 옵트인 동의가 없는 상황에서는, 귀하가 형성하는 모든 연결은 사용자 입장에서 그 목적을 위해 사용될 것이라고 합리적으로 예상할 수 있는 정보에만 기반해야 합니다. 사용자가, 귀하가 자신이 제공한 정보를 사용해 자신의 X 계정을 X 외부의 신원과 연결하고 있다는 사실을 알게 되면 놀랄 만한 경우라면, 그렇게 해서는 안 됩니다. 또한 개인의 명시적 옵트인 동의가 없는 경우, 다음에 근거해서만 누군가에 대한 귀하의 기록을 X 신원과 매칭하려고 시도할 수 있습니다.

* **사용자가 귀하에게 직접 제공한 정보.** 귀하와 사전 관계가 전혀 없는 개인에 대한 기록(예: 제3자로부터 획득한 개인 데이터)은 이 기준을 충족하지 않습니다. 및/또는
* **공개 데이터.** 이 문맥에서 “공개 데이터”는 다음을 의미합니다.
  * 귀하가 공적이고 일반적으로 접근 가능한 자원(예: 직업 협회의 회원 명부)에서 획득한 사용자의 정보
  * X 상에서 사용자에 대해 공개적으로 이용 가능한 정보로, 예를 들면 다음이 포함됩니다.
    * 게시물
    * 사용자의 바이오 및 공개적으로 명시한 위치를 포함한 프로필 정보
    * 표시 이름 및 사용자 이름

<div id="redistribution-of-x-content">
  ## X 콘텐츠 재배포
</div>

X API를 통해 획득한 X 콘텐츠를 다른 당사자와 공유해야 하는 경우, 가장 좋은 방법은 게시물 ID, 다이렉트 메시지 ID 및/또는 사용자 ID를 공유하는 것입니다. 그러면 콘텐츠 최종 사용자가 X API를 사용하여 이를 재조회(즉, 전체 게시물, 사용자 또는 다이렉트 메시지 콘텐츠를 다시 요청)할 수 있습니다. 이렇게 하면 X 콘텐츠의 최종 사용자가 항상 X로부터 최신 정보를 직접 받을 수 있습니다.

X는 비자동화된 방식을 통한 재조회된 X 콘텐츠의 제한적인 재배포를 허용합니다. 이 방식으로 재조회된 X 콘텐츠를 다른 당사자와 공유하는 경우, 수신자 1인당 1일 기준 최대 50,000개의 재조회된 공개 게시물 오브젝트 및/또는 사용자 오브젝트만 공유할 수 있으며, 이 데이터를 블로그 게시물의 첨부 파일이나 공개 GitHub 리포지토리와 같이 공중에게 공개적으로 제공해서는 안 됩니다.

X 콘텐츠를 재배포할 때 유의해야 할 몇 가지 추가 사항은 다음과 같습니다.

* X로부터 사전에 명시적인 서면 허가를 받지 않은 한, 30일 기간 동안 단일 단체에 배포할 수 있는 게시물 ID는 총 1,500,000개로 제한됩니다.
* 비상업적 연구만을 목적으로 학술 기관을 대신하여 게시물 ID 및/또는 사용자 ID를 재배포하는 개인은 게시물 ID 및/또는 사용자 ID를 무제한으로 재배포할 수 있습니다.
* 위에 설명된 바와 같이 X 콘텐츠 공유에 대한 허가를 요청하려면 API Policy 지원 양식을 사용하십시오.

X 콘텐츠를 제3자에게 배포할 수 있는 범위 내에서, 해당 콘텐츠는 계속해서 Developer Agreement 및 Policy의 적용을 받으며, 이러한 제3자는 X [이용 약관](https://twitter.com/en/tos), [개인정보 처리방침](https://twitter.com/en/privacy), [Developer Agreement](/ko/developer-terms/agreement) 및 [Developer Policy](/ko/developer-terms/policy)에 동의해야만 X 콘텐츠를 수령할 수 있습니다.

<div id="multiple-applications">
  ## 여러 애플리케이션
</div>

하나의 사용 사례 또는 실질적으로 유사하거나 중복되는 사용 사례에 대해 여러 애플리케이션을 등록하는 것은 허용되지 않습니다. 이러한 정책에 대해 더 알아보려면 [여기](https://twittercommunity.com/t/policy-clarification-multiple-applications-for-the-same-use-case/92080)를 참고하세요.

이 문맥에서 “사용 사례(use case)”는 애플리케이션을 통해 수행되는 일관된 분석, 표시 또는 작업의 집합으로 정의합니다. 동일한 서비스나 애플리케이션을 서로 다른 최종 사용자에게 제공하는 것은 단일 사용 사례로 간주됩니다. 다만 도구나 서비스의 &quot;화이트 라벨(white label)&quot; 버전은 여기에 포함되지 않습니다. 애플리케이션의 &quot;화이트 라벨&quot; 버전을 제공할 계획이라면 반드시 당사에 통지하고 별도의 승인을 받아야 합니다. 이러한 규칙은 여러분이 등록하는 애플리케이션뿐만 아니라, 여러분의 도구나 서비스를 사용하는 최종 사용자가 등록하는 애플리케이션에도 적용됩니다. 여러분의 도구나 서비스를 사용하기 위한 목적으로 최종 사용자에게 애플리케이션 등록을 요구하는 경우, 여러분과 여러분의 애플리케이션, 고객, 그리고/또는 여러분의 도구나 서비스의 최종 사용자에 대해 제재 조치가 내려질 수 있습니다.

이 규칙의 유일한 예외는 동일한 서비스에 대해 개발(“dev”), 스테이징(“staging”), 프로덕션(“prod”) 인스턴스를 생성하는 경우입니다. 이러한 애플리케이션은 (예를 들어 애플리케이션 이름이나 설명에서) 명확하게 구분되도록 라벨링해야 하며, 개발 또는 스테이징 애플리케이션을 프로덕션 용도로 사용해서는 안 됩니다.

<div id="automation-spam-and-auto-responses">
  ## 자동화, 스팸, 자동 응답
</div>

스팸을 생성하거나 스팸성 행위를 하는 데 X의 API 및 개발자 제품을 사용하는 것은 금지됩니다. 스팸 관련 [X 규칙](https://t.co/rules)을 검토하고, 귀하의 앱이 당사의 정책을 위반하지 않으며 사용자 역시 위반하지 못하도록 해야 합니다.

앱이 X 서비스에서 게시물 작성, 계정 팔로우, 다이렉트 메시지 전송 등 쓰기 작업을 수행하는 데 사용될 경우, 서비스가 당사 지침을 준수하도록 [자동화 규칙](https://support.twitter.com/articles/76915)을 면밀히 검토해야 합니다. 특히 다음 사항을 준수해야 합니다.

* [자동화된 답글 또는 메시지](https://twittercommunity.com/t/policy-clarification-automated-replies-and-mentions/94444)를 보내기 전에 항상 사용자로부터 명시적 동의를 받아야 합니다.
* 사용자로부터 귀하에게 연락받기를 원하지 않는다는 요청을 받으면 즉시 이에 따라야 합니다.
* [대량 팔로우](https://twittercommunity.com/t/policy-clarification-aggressive-following-and-inorganic-following-behavior/92769)를 포함해 대량, 공격적 또는 스팸성 행위를 절대 수행해서는 안 됩니다.
* [여러 계정](https://blog.twitter.com/developer/en_us/topics/tips/2018/automation-and-the-use-of-multiple-accounts.html)에서 동일하거나 실질적으로 유사한 콘텐츠를 게시해서는 안 됩니다.

<div id="measuring-the-x-service">
  ## X 서비스 측정
</div>

벤치마킹이나 경쟁 목적을 위해 X의 가용성, 성능, 기능 또는 사용량을 측정하는 데 X API를 사용하지 마십시오. 예를 들어, 다음과 같은 목적으로 X API를 사용해서는 안 됩니다:

* 활성 사용자 수 또는 계정 수 등 X 사용자 메트릭의 총합을 계산하는 것
* 하루 게시물 총개수, 사용자 참여 수, 계정 참여 수 등 X 게시물 메트릭의 총합을 계산하는 것
* [X Rules](https://help.twitter.com/en/rules-and-policies/twitter-rules)에서 허용한 경우를 제외하고 X에서의 스팸이나 보안을 측정하거나 분석하는 것

<div id="surveillance-privacy-and-user-protection">
  ## 감시, 프라이버시, 그리고 사용자 보호
</div>

X에서는 사용자 프라이버시를 보호하고 수호하는 것이 회사의 핵심 DNA에 내재되어 있으며, 당사의 개발자 및 데이터 제품은 이러한 약속을 반영합니다. 우리는 X 데이터가 전 세계적으로 긍정적인 변화를 이끄는 강력한 힘이 될 수 있다고 믿습니다. 예를 들어 자카르타 홍수 당시 인명 구조를 돕거나, USGS가 지진을 추적하도록 지원하거나, UN과 함께 지속가능발전목표(SDGs)를 달성하는 데 기여하는 등의 사용 사례가 있습니다. 그러나 우리는 어떤 주체든 감시 목적이나, 그 밖에 어떤 방식으로든 사용자의 합리적인 프라이버시 기대에 부합하지 않는 방식으로 X 데이터 및 X API를 사용하는 것을 금지합니다. 예외는 없습니다.

데이터 및 개발자 제품의 금지된 사용 방식은 [Developer Agreement](/ko/developer-terms/agreement)에 설명되어 있으며, 여기에는 X 사용자나 그 콘텐츠를 조사하거나 추적하는 행위뿐만 아니라, 시위, 집회 또는 커뮤니티 조직 회의와 같은 민감한 이벤트를 추적·경고·모니터링하는 행위에 대한 금지 조항이 포함됩니다.

이 약관에 따라 금지되는 활동의 다른 범주에는 (이에 한정되지 않지만) 다음이 포함됩니다.

* 노동조합이나 활동가 단체와 같은 민감한 집단 및 조직을 조사하거나 추적하는 행위
* 신원 조회 또는 그 밖의 형태의 극단적인 심사(vetting)
* 신용 또는 보험 위험 분석
* 개인 프로파일링 또는 심리그래픽(심리·성향 기반) 세분화
* 안면 인식

이 정책은 당사 API의 모든 사용자에게 적용됩니다. 이러한 목적으로 X API를 오용하는 경우, 접근의 일시 중지 또는 종료를 포함한 제재 조치의 대상이 됩니다.

X 계정 관련 정보를 요청하는 법 집행 기관을 위한 추가 정보는 [https://t.co/le](https://t.co/le)를 방문하십시오.

또한, 현재 X는 [Grok](https://help.x.com/en/using-x/about-grok)을 예외로 하고, 기반(foundation) 또는 프런티어(frontier) 모델을 미세 조정(fine-tune)하거나 학습시키기 위한 어떠한 X API 및/또는 X 콘텐츠의 사용도 금지합니다.
