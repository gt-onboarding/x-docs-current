<div id="standard-v11-compared-to-x-api-v2">
  ### Standard v1.1과 X API v2 비교
</div>

v1.1 [statuses/filter](https://developer.x.com/en/docs/x-api/v1/tweets/filter-realtime/api-reference/post-statuses-filter) 엔드포인트를 사용해 오셨다면, 이 가이드는 Standard v1.1과 X API v2 필터링 스트림 엔드포인트 간의 유사점과 차이점을 이해하는 데 도움이 됩니다.

* **유사점**
  * 요청 파라미터와 연산자
  * 게시물 편집 내역 및 메타데이터 지원
* **차이점**
  * 엔드포인트 URL
  * 앱 및 프로젝트 요구 사항
  * 인증 방식
  * 규칙 개수와 지속 스트림
  * 응답 데이터 형식
  * 요청 파라미터
  * 복구 및 이중화 기능의 제공 여부
  * 쿼리 연산자

<div id="similarities">
  #### 유사점
</div>

**요청 파라미터와 연산자**

standard v1.1 `statuses/filter` endpoint에는 스트림을 필터링하기 위해 요청과 함께 전달할 수 있는 몇 가지 파라미터가 있습니다. v2 filtered stream에서는 대신 부울 논리로 서로 연결할 수 있는 [operators](/ko/x-api/posts/filtered-stream#building-rules-for-filtered-stream)를 사용해 원하는 게시물을 필터링합니다. 사용 가능한 연산자 중 일부는 기존 standard v1.1 파라미터를 직접 대체합니다. 

다음 standard v1.1 요청 파라미터는 X API v2에서 이에 상응하는 연산자가 있습니다:
 

| **Standard** | **X API v2** |
| :--- | :--- |
| follow - 쉼표로 구분된 user ID 목록으로, 스트림에 게시물이 전달되어야 하는 사용자를 나타냅니다. | 특정 사용자와 관련된 게시물을 찾는 데 도움이 되는 여러 연산자:<br /><br />* @<br />* from:<br />* to:<br />* 등 |
| track - 스트림에 어떤 게시물이 전달될지를 결정하는 데 사용되는, 쉼표로 구분된 문구 목록입니다. | 특정 키워드와 관련된 게시물을 찾는 데 도움이 되는 여러 연산자:<br /><br />* keyword<br />* &quot;exact phrase match&quot;<br />* #<br />* 등 |

**게시물 수정 이력 및 메타데이터 지원**

두 버전 모두 수정 이력을 설명하는 메타데이터를 제공합니다. 자세한 내용은 [filtered stream API reference](/ko/x-api/posts/filtered-stream#api-reference-index)와 [Post edits fundamentals 페이지](/ko/x-api/fundamentals/edit-posts)를 참고하세요. 

<div id="differences">
  #### 차이점
</div>

**엔드포인트 URL**

* 표준 v1.1 엔드포인트:
  * https://stream.x.com/1.1/statuses/filter.json
* X API v2 엔드포인트:
  * https://api.x.com/2/tweets/search/stream
  * https://api.x.com/2/tweets/search/stream/rule

**앱 및 프로젝트 요건**

X API v2 엔드포인트는 요청을 인증할 때 [developer App](/ko/resources/fundamentals/developer-apps)과 연결된 [Project](/ko/resources/fundamentals/projects)의 자격 증명을 사용해야 합니다. 모든 X API v1.1 엔드포인트는 단독 App 또는 Project와 연결된 App의 자격 증명을 사용할 수 있습니다.

**인증 방식**

표준 엔드포인트는 [OAuth 1.0a User Context](/ko/resources/fundamentals/authentication)를 지원하는 반면, X API v2 필터링 스트림 엔드포인트는 [OAuth 2.0 App-Only](/ko/resources/fundamentals/authentication#oauth-2-0) (Application-only authentication이라고도 함)을 지원합니다. X API v2 버전에 요청을 보내려면 App Access Token(앱 액세스 토큰)을 사용해 요청을 인증해야 합니다.

developer portal에서 Project와 앱을 생성할 때 발급되었던 App Access Token이 더 이상 없다면, developer portal에서 앱의 “Keys and tokens” 페이지로 이동해 새 토큰을 생성할 수 있습니다. App Access Token을 프로그래밍 방식으로 생성하려면 이 [OAuth 2.0 App-Only 가이드](/ko/resources/fundamentals/authentication#app-only-authentication-and-oauth-2-0-bearer-token)를 참조하세요. 

**규칙 수량 및 지속 스트림**

표준 v1.1 엔드포인트는 스트리밍 연결을 필터링하기 위해 단일 규칙만 지원합니다. 규칙을 변경하려면 스트림 연결을 끊고, 수정된 필터링 규칙을 파라미터로 포함한 새 요청을 보내야 합니다. 

X API v2 필터링 스트림 엔드포인트는 단일 스트림에 여러 규칙을 적용할 수 있으며, 스트림 연결을 유지한 채로 스트림에 규칙을 추가·삭제할 수 있습니다. 

**응답 데이터 형식**

표준 v1.1과 X API v2 엔드포인트 간 가장 큰 차이점 중 하나는 페이로드에서 어떤 필드를 반환할지 선택하는 방식입니다.

표준 엔드포인트의 경우, 기본적으로 많은 응답 필드를 받고, 이후 파라미터를 사용하여 페이로드에 어떤 필드 또는 필드 집합을 반환할지 지정할 수 있습니다.

X API v2 버전은 기본적으로 게시물 id와 text 필드만 제공합니다. 추가 필드나 객체를 요청하려면 [fields](/ko/x-api/fundamentals/fields) 및 [expansions](/ko/x-api/fundamentals/expansions) 파라미터를 사용해야 합니다. 이 엔드포인트에서 요청한 모든 게시물 필드는 기본 게시물 객체에 포함되어 반환됩니다. 확장된 사용자, 미디어, 투표, 장소 객체 및 필드는 응답 내 includes 객체에 포함되어 반환됩니다. 그런 다음 게시물과 확장된 객체 모두에 있는 ID를 매칭하여 모든 확장 객체를 다시 게시물 객체와 연결할 수 있습니다. 

이 새 파라미터에 대해 더 자세히 알아보려면 각 파라미터에 대한 개별 가이드 또는 [fields 및 expansions 사용 방법](/ko/x-api/fundamentals/data-dictionary#how-to-use-fields-and-expansions) 가이드를 읽어 보시기 바랍니다. 

또한 표준 v1.1 필드를 최신 v2 필드에 매핑하는 데 도움이 되는 [데이터 형식 마이그레이션 가이드](/ko/x-api/migrate/data-format-migration#migrating-from-standard-v1-1s-data-format-to-v2)를 준비했습니다. 이 가이드는 특정 필드를 반환하기 위해 v2 요청과 함께 전달해야 하는 expansions 및 fields 파라미터도 구체적으로 제공합니다. 
 

특정 필드를 요청하는 방식의 변경 외에도, X API v2는 API가 반환하는 객체(게시물 객체와 [user](/ko/x-api/fundamentals/data-dictionary#user) 객체 포함)에 대해 새로운 JSON 설계를 도입합니다.

* JSON 루트 레벨에서, 표준 엔드포인트는 게시물 객체를 statuses 배열로 반환하는 반면, X API v2는 data 배열로 반환합니다. 
* Retweeted 및 Quoted &quot;statuses&quot;를 참조하는 대신, X API v2 JSON은 Retweeted 및 Quoted 트윗을 참조합니다. contributors 및 user.translator&#95;type과 같은 많은 레거시·사용 중단 필드는 제거됩니다. 
* 게시물 객체의 favorites 및 user 객체의 favourites를 모두 사용하는 대신, X API v2는 like라는 용어를 사용합니다. 
* X는 값이 없는 JSON 값(예: null)은 페이로드에 기록하지 않는 규칙을 채택하고 있습니다. 게시물 및 user 속성은 null이 아닌 값을 가질 때만 포함됩니다. 
   

또한 [Post object](/ko/x-api/fundamentals/data-dictionary#tweet)에 다음과 같은 새로운 필드 집합을 도입했습니다:

* [conversation&#95;id](/ko/x-api/fundamentals/conversation-id) 필드
* context와 entities를 포함하는 두 개의 새로운 [annotations](/ko/x-api/fundamentals/post-annotations) 필드
* 여러 개의 새로운 [metrics](/ko/x-api/fundamentals/metrics) 필드
* 특정 게시물에 누가 답글을 달 수 있는지 보여주는 새로운 reply&#95;setting 필드

**요청 파라미터**

다음은 X API v2에서는 지원되지 **않는** 표준 filtered stream 요청 파라미터입니다:

| Standard v1.1 parameter | Details |
| :--- | :--- |
| locations - 게시물을 필터링할 경계 상자 집합을 지정하는 경도, 위도 쌍의 쉼표로 구분된 목록입니다. | X API v2에서는 아직 위치 기반 operator를 제공하지 않습니다. |
| Delimited | v1.1 엔드포인트에서는 이 값을 문자열 길이로 설정하면 스트림에서 status가 구분되도록 하여, 클라이언트가 status 메시지의 끝까지 읽어야 하는 바이트 수를 알 수 있게 합니다.<br /><br />이 기능은 X API v2에서는 사용할 수 없습니다. |
| Stall&#95;warnings | v1.1 엔드포인트에서는 이 파라미터를 true로 설정하면 클라이언트의 연결이 끊길 위험이 있을 때 정기적인 메시지가 전송됩니다.<br /><br />X API v2에서는 개행이 일정 간격으로 전송되는 stall warning이 기본적으로 전송됩니다. |

**복구 및 중복 기능 사용 가능 여부**

X API v2 버전의 filtered stream은 스트리밍 가동 시간을 최대화하고, 5분 이하 동안의 연결 끊김으로 인해 누락되었을 수 있는 게시물을 복구하는 데 도움이 되는 복구 및 중복 기능을 도입합니다.

중복 연결을 사용하면 특정 스트림에 최대 두 번까지 연결할 수 있으며, 그중 하나의 연결이 실패하더라도 항상 스트림에 대한 연결을 유지하는 데 도움이 됩니다. 

backfill&#95;minutes 파라미터는 최대 5분 동안 누락된 데이터를 복구하는 데 사용할 수 있습니다. 

이 두 기능은 모두 [Academic Research access](/ko/x-api/getting-started/about-x-api)를 통해서만 사용할 수 있습니다. 이 기능에 대해 더 알아보려면 [recovery and redundancy features](/ko/x-api/posts/filtered-stream#filtered-stream-recovery-and-redundancy-features) 통합 가이드를 참고하세요. 

**새로운 쿼리 operator**

X API v2는 두 가지 새로운 기능을 지원하기 위해 새로운 operator를 도입합니다: 

* **[Conversation IDs](/ko/x-api/fundamentals/conversation-id)** - X에서 대화가 전개됨에 따라, 대화에 속한 게시물임을 표시하기 위해 conversation ID가 제공됩니다. 대화에 포함된 모든 게시물의 conversation&#95;id는 대화를 시작한 게시물의 Post ID로 설정됩니다. 
  * conversation&#95;id:
* **[X Annotations](/ko/x-api/fundamentals/post-annotations)** 는 게시물에 대한 맥락 정보를 제공하며, entity 및 context annotations를 포함합니다. entity는 사람, 장소, 제품, 조직 등으로 구성됩니다. context는 노출된 entity가 속한 domain 또는 주제입니다. 예를 들어, 게시물에 언급된 사람은 운동선수, 배우, 정치인인지 여부를 나타내는 context를 가질 수 있습니다.  
  * context: - 관심 있는 context로 annotation된 게시물과 매치합니다. 
  * entity: - 관심 있는 entity로 annotation된 게시물과 매치합니다.
