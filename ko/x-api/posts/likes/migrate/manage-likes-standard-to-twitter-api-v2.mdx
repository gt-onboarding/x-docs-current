<div id="manage-likes-standard-v11-compared-to-x-api-v2">
  ### 좋아요 관리: Standard v1.1과 X API v2 비교
</div>

Standard v1.1 [POST favorites/create](https://developer.x.com/en/docs/twitter-api/v1/tweets/post-and-engage/api-reference/post-favorites-create) 및 [POST favorites/destroy](https://developer.x.com/en/docs/twitter-api/v1/tweets/post-and-engage/api-reference/post-favorites-destroy) 엔드포인트를 사용해 오셨다면, 이 가이드는 Standard v1.1과 X API v2의 좋아요 관리 엔드포인트 간 유사점과 차이점을 이해하는 데 도움이 되도록 작성되었습니다.

* **유사점**
  * OAuth 1.0a 사용자 컨텍스트
* **차이점**
  * 엔드포인트 URL 및 HTTP 메서드
  * 앱 및 프로젝트 요구 사항
  * 요청 파라미터

<div id="similarities">
  #### 유사점
</div>

**OAuth 1.0a 사용자 컨텍스트 인증 방식**

두 엔드포인트 버전은 모두 [OAuth 1.0a 사용자 컨텍스트](/ko/resources/fundamentals/authentication#oauth-1-0a-2)를 지원합니다. 따라서 이전에 standard v1.1 manage favorites 엔드포인트 중 하나를 사용해 왔다면, X API v2로 마이그레이션하더라도 동일한 인증 방식을 계속 사용할 수 있습니다.

<div id="differences">
  #### 차이점
</div>

**엔드포인트 URL 및 HTTP 메서드**

* Standard v1.1 엔드포인트:
  * POST https://api.x.com/1.1/favorites/create.json\
    (게시물 좋아요)
  * POST https://api.x.com/1.1/favorites/destroy.json\
    (게시물 좋아요 취소)
* X API v2 엔드포인트:
  * POST https://api.x.com/2/tweets/:id/likes\
    (게시물 좋아요)
  * DELETE https://api.x.com/2/tweets/:id/likes/:tweet&#95;id\
    (게시물 좋아요 취소) 

**앱 및 프로젝트 요구 사항**

X API v2 엔드포인트를 사용하려면 요청을 인증할 때 [Project](/ko/resources/fundamentals/projects)에 연결된 [developer App](/ko/resources/fundamentals/developer-apps)의 자격 증명을 사용해야 합니다. 모든 X API v1.1 엔드포인트는 Project에 연결되지 않은 독립 실행형 앱이나 Project에 연결된 앱의 자격 증명을 모두 사용할 수 있습니다.

**요청 파라미터**

다음 Standard v1.1 요청 파라미터는 X API v2에서 이에 해당하는 항목이 있습니다:

| Standard v1.1 | X API v2 |
| :--- | :--- |
| id  | id  |
| includes&#95;entities | 해당 없음 |

Standard v1.1 파라미터는 쿼리 파라미터로 전달되고, X API v2 파라미터는 POST 엔드포인트의 경우 본문 파라미터로, DELETE 엔드포인트의 경우 경로 파라미터로 전달된다는 점에 유의하세요.

또한 Standard v1.1 엔드포인트를 사용할 때에는 게시물에 좋아요를 누르는 사용자의 id를 별도로 전달할 필요가 없습니다. [OAuth 1.0a 사용자 컨텍스트](/ko/resources/fundamentals/authentication)와 함께 전달되는 [Access Tokens](/ko/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow)를 통해 어떤 사용자가 좋아요/좋아요 취소를 수행하는지 유추할 수 있기 때문입니다.
