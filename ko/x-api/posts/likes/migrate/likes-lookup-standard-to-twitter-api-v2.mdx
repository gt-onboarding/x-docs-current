<div id="likes-lookup-standard-v11-compared-to-x-api-v2">
  ### 좋아요 조회: 표준 v1.1과 X API v2 비교
</div>

표준 v1.1의 [GET favorites/list](https://developer.x.com/en/docs/x-api/v1/tweets/post-and-engage/api-reference/get-favorites-list) 엔드포인트를 사용해 왔다면, 이 가이드는 표준 v1.1과 X API v2 좋아요 조회 엔드포인트 간의 공통점과 차이점을 이해하는 데 도움을 주기 위한 것입니다.

v2에서는 게시물을 좋아요한 사용자 정보를 가져올 수 있는 새로운 &#39;liked users&#39; 엔드포인트도 도입했습니다.

* **유사점**
  * 인증
  * 요청 한도
* **차이점**
  * 엔드포인트 URL

  * 요청 제한

  * 앱 및 프로젝트 요구 사항

  * 요청 매개변수

  * 새로운 JSON 형식

<div id="similarities">
  #### 유사점
</div>

**인증**

표준 v1.1과 X API v2의 Likes 조회 엔드포인트는 모두 [OAuth 1.0a 사용자 컨텍스트](/ko/resources/fundamentals/authentication#oauth-1-0a-2) 또는 [OAuth 2.0 Bearer Token](/ko/resources/fundamentals/authentication#bearer-token-also-known-as-app-only)을 사용합니다. 따라서 이전에 표준 v1.1의 [GET favorites/list 엔드포인트](https://developer.x.com/en/docs/x-api/v1/tweets/post-and-engage/api-reference/get-favorites-list)를 사용하고 있었다면, X API v2 버전으로 마이그레이션하더라도 원하는 경우 동일한 인증 방식을 계속 사용할 수 있습니다. 

사용하려는 인증 라이브러리/패키지에 따라 다를 수 있지만, Bearer Token 인증이 시작하기에 가장 쉬운 방법인 경우가 많으며 간단한 요청 헤더 설정만으로 사용할 수 있습니다. Bearer Token 생성 방법을 알아보려면 [이 OAuth 2.0 Bearer Token 가이드](/ko/resources/fundamentals/authentication#bearer-token-also-known-as-app-only)를 참조하세요. 
 

**요청 한도**

표준 v1.1 GET favorites/list 엔드포인트에는 사용자당 15분에 75회 요청 한도가 있습니다. v2의 해당 좋아요한 게시물 엔드포인트에도 동일한 요청 한도가 적용됩니다. 그러나 이 v2 엔드포인트에는 추가로 앱당 15분에 75회 요청 한도가 있습니다.

<div id="differences">
  #### 차이점
</div>

**엔드포인트 URL**

* 표준 v1.1 엔드포인트:
  * GET https://api.x.com/1.1/favorites/list.json\
    (지정된 사용자가 좋아요한 게시물 목록)
  * v2 liking users 엔드포인트에 해당하는 v1.1 엔드포인트는 없습니다.
* X API v2 엔드포인트:
  * GET https://api.x.com/2/users/:id/liked&#95;tweets\
    (지정된 사용자 ID가 좋아요한 게시물 목록)
  * GET https://api.x.com/2/tweets/:id/liking&#95;users\
    (지정된 게시물 ID에 좋아요를 누른 사용자 목록)

**요청 제한 사항**

v2 liked Posts 엔드포인트는 요청당 5~100개의 게시물을 조회할 수 있지만, 페이지네이션 토큰을 사용하여 특정 게시물에 대한 모든 좋아요를 요청할 수 있습니다. v1.1 GET favorites/list 엔드포인트 역시 게시물의 모든 좋아요를 가져올 수 있지만, 요청당 20~200개의 게시물을 조회할 수 있습니다.

v2 liking users 엔드포인트의 경우, 게시물당 최대 100명의 좋아요한 사용자로 제한됩니다. 
 

**앱 및 프로젝트 요구 사항**

X API v2 엔드포인트는 요청을 인증할 때 [개발자 앱](/ko/resources/fundamentals/developer-apps)이 [프로젝트](/ko/resources/fundamentals/developer-apps)와 연결되어 있어야 하며, 해당 앱의 자격 증명을 사용해야 합니다. 모든 X API v1.1 엔드포인트는 독립형 앱 또는 프로젝트와 연결된 앱의 자격 증명을 사용할 수 있습니다.

**요청 매개변수**

다음과 같은 표준 v1.1 엔드포인트는 두 개의 요청 쿼리 매개변수(user&#95;id 또는 screen&#95;name)를 허용했습니다. X API v2는 숫자 사용자 ID만 허용하며, 반드시 엔드포인트 경로의 일부로 전달해야 합니다.

표준 v1.1과 X API v2 엔드포인트 버전 간 가장 큰 차이점 중 하나는 페이로드에 어떤 필드가 포함되어 반환될지를 선택하는 방식입니다. 표준 엔드포인트에서는 여러 매개변수를 사용해 페이로드에 어떤 필드 또는 필드 집합을 반환할지 지정할 수 있었던 반면, X API v2 버전에서는 이러한 다양한 매개변수를 [fields](/ko/x-api/fundamentals/fields)와 [expansions](/ko/x-api/fundamentals/expansions)로 단순화했습니다. 
 

**새 JSON 형식**

X API v2는 API가 반환하는 객체(예: [Post](/ko/x-api/fundamentals/data-dictionary#tweet) 및 [user](/ko/x-api/fundamentals/data-dictionary#user) 객체)에 대해 새로운 JSON 설계를 도입합니다.

* JSON 루트 레벨에서, 표준 엔드포인트는 statuses 배열 안에 user 객체를 반환하는 반면, X API v2는 data 배열을 반환합니다. 
* 리트윗 및 인용을 &quot;statuses&quot;로 지칭하는 대신, X API v2 JSON은 리트윗 및 인용 트윗으로 지칭합니다. contributors, user.translator&#95;type과 같은 많은 레거시 및 사용 중단 필드는 제거됩니다. 
* Post 객체의 favorites와 user 객체의 favourites를 모두 사용하는 대신, X API v2는 like라는 용어를 사용합니다. 
* X는 값이 없는 JSON 값(예: null)은 페이로드에 기록하지 않는 규칙을 채택하고 있습니다. Post 및 user 속성은 null이 아닌 값을 가진 경우에만 포함됩니다.
   

새 JSON 형식에 적용한 변경 사항 외에도, Post 객체에 다음을 포함한 새로운 필드 세트를 도입했습니다:

* [conversation&#95;id](/ko/x-api/fundamentals/conversation-id)
* context 및 entities를 포함하는 두 개의 새로운 [Post annotations](/ko/x-api/fundamentals/post-annotations) 필드
* 여러 새로운 [metrics](/ko/x-api/fundamentals/metrics) 필드
