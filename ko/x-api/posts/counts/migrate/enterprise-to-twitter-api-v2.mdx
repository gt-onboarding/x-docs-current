<div id="enterprise-compared-to-x-api-v2">
  ### Enterprise와 X API v2 비교
</div>

**유사점**

* 세분화 수준
* 페이지네이션
* 시간대

**차이점**

* 엔드포인트 URL
* App 및 Project 필요 여부
* 사용 가능한 기간
* 응답 데이터 포맷
* HTTP 메서드
* 요청 시각 형식
* 요청 파라미터
* 필터링 연산자

<div id="similarities">
  #### 유사점
</div>

**세분성**

반환되는 데이터의 세분성을 선택하는 데 사용하는 파라미터는 서로 다르지만(Enterprise 버전에서는 `bucket`, v2 버전에서는 `granularity`), 해당 파라미터에 전달할 수 있는 값과 기본 동작은 동일합니다.

* `day`
* `hour` (기본값)
* `minute`

**페이지네이션**

v2에는 추가 페이지네이션 기능이 있습니다(`since_id`와 `until_id`를 사용해 게시물 ID 기준으로 이동할 수 있게 해 주는 새로운 페이지네이션 파라미터). 하지만 Enterprise와 v2 모두 시간 기준으로 페이지네이션을 할 수 있습니다(Enterprise에서는 `fromDate`와 `toDate`, v2에서는 `start_time`과 `end_time` 사용).

Enterprise 버전을 사용하는 경우, 페이지네이션을 위해 `next` 파라미터를 사용하며, 다음 토큰 필드는 `next`라고 불리고 응답의 루트에 위치합니다.

v2를 사용하는 경우, 페이지네이션을 위해 `next_token` 또는 `pagination_token` 파라미터를 사용할 수 있으며, 다음 토큰은 응답의 `meta.next_token`에 위치합니다.
 

**시간대**

페이지네이션 섹션에서 언급했듯이, Enterprise와 v2 모두에서 시간을 사용해 서로 다른 데이터 페이지로 이동할 수 있습니다. 두 경우 모두 이러한 파라미터를 사용할 때 UTC를 시간대로 사용합니다.

<div id="differences">
  #### 차이점
</div>

**엔드포인트 URL**

* Enterprise 엔드포인트:
  * 30일 - `http://gnip-api.x.com/search/30day/accounts/:account_name/:label/counts.json`
  * 전체 아카이브(Full-archive) - `http://gnip-api.x.com/search/fullarchive/accounts/:account_name/:label/counts.json`
* X API v2 엔드포인트
  * 최근 (7일) - `https://api.x.com/2/tweets/counts/recent`
  * 전체 아카이브(Full-archive) - `https://api.x.com/2/tweets/counts/all`

**앱 및 프로젝트 요구 사항**

X API v2 엔드포인트에서는 요청을 인증할 때 [developer App](/ko/resources/fundamentals/developer-apps)과 [Project](/ko/resources/fundamentals/developer-apps)에 연결된 자격 증명을 사용해야 합니다. X API v1.1 엔드포인트는 프로젝트에 연결되지 않은 독립 실행형 앱 또는 프로젝트에 연결된 앱의 자격 증명을 모두 사용할 수 있습니다.
 

**사용 가능한 기간**

Enterprise API와 X API v2 모두 전체 아카이브에 대해 게시물 볼륨 데이터를 조회할 수 있는 엔드포인트를 제공합니다.

그러나 X API v2는 Enterprise API와 같은 30일 기간 엔드포인트를 제공하지 않습니다. 대신 앞서 언급한 전체 아카이브 엔드포인트 또는 [v2 Search Posts 엔드포인트](/ko/x-api/posts/search/introduction)와 일치하는 7일 기간 엔드포인트를 제공합니다.
 

**응답 데이터 형식**

Enterprise와 X API v2를 통해 수신하는 데이터 형식에는 약간의 차이가 있습니다.

* Enterprise의 count 데이터는 `results` 객체 안에 위치하는 반면, v2 count 데이터는 `data` 객체 안에 위치합니다.
* Enterprise의 count 필드는 `timePeriod`(시작 시각)와 `count`인 반면, v2는 기간을 `start`와 `end` 필드로 분리하고(‘요청 시간 형식’에서 설명하는 Enterprise와는 다른 날짜/시간 형식을 사용) count 필드 이름을 `tweet_count`로 변경했습니다.
* Enterprise 메타데이터는 루트 레벨에서 `totalCount`, `next`, 그리고 `requestParameters` 객체를 포함합니다. 반면 v2는 `requestParameters` 객체를 포함하지 않고, 다음 항목을 루트 레벨에 존재하는 `meta` 객체로 이동 및 이름 변경합니다: `total_tweet_count` 및 `next_token`.
   

**HTTP 메서드**

Enterprise 버전 API에서는 JSON 본문을 포함하는 POST HTTP 메서드나 쿼리 문자열을 포함하는 GET HTTP 메서드 중 하나로 요청을 보낼 수 있습니다.

v2에서는 쿼리 문자열을 사용하는 GET HTTP 메서드만 사용할 수 있습니다.
 

**요청 시간 형식**

이 엔드포인트의 Enterprise 버전에서는 페이지네이션 파라미터와 `timePeriod` 응답 필드 모두에 다음 날짜/시간 형식을 사용합니다: `YYYYMMDDHHmm`

v2 엔드포인트에서는 페이지네이션 파라미터와 `start` 및 `end` 응답 필드 모두에 ISO 8601/RFC 3339 날짜/시간 형식을 사용합니다: `YYYY-MM-DDTHH:mm:ssZ`

**요청 파라미터**

다음은 Enterprise와 X API v2의 요청 파라미터 표입니다.

| Enterprise | Search Posts v2 |
| :--- | :--- |
| query | query |
| bucket | granularity |
| fromDate (YYMMDDHHmm) | start&#95;time (YYYY-MM-DDTHH:mm:ssZ) |
| toDate (YYMMDDHHmm) | end&#95;time (YYYY-MM-DDTHH:mm:ssZ) |
|     | since&#95;id |
|     | until&#95;id |
| next | next&#95;token 및 pagination&#95;token |

**필터링 연산자**

Enterprise와 X API v2 간의 연산자는 대부분 동일하지만, 연산자 가용성 측면에서 몇 가지 차이가 있고 X API v2 버전에만 도입된 새로운 연산자도 일부 있습니다.

X API v2, Enterprise, 그리고 Premium에서 사용 가능한 연산자 전체 표를 보려면 [Post counts 마이그레이션 랜딩 페이지](/ko/x-api/posts/counts/migrate/overview)를 방문하세요.
