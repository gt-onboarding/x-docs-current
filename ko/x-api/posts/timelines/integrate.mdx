  ### 유용한 도구

몇 가지 핵심 개념을 살펴보기 전에, 아래 도구나 코드 샘플 중 하나를 사용해 이 엔드포인트들의 기능을 먼저 테스트해 볼 것을 권장합니다.

**코드 샘플**

선호하는 프로그래밍 언어로 이 엔드포인트들을 연동해 보고 싶나요? 시작점으로 활용할 수 있는 다양한 코드 샘플을 [GitHub 페이지](https://github.com/xdevplatform/Twitter-API-v2-sample-code)에서 제공하고 있습니다.

**라이브러리**

커뮤니티에서 제공하는 [서드파티 라이브러리](/ko/x-api/tools-and-libraries/overview) 중 하나를 활용해 손쉽게 시작해 보세요. 올바른 버전 태그를 확인하면 v2 엔드포인트와 호환되는 라이브러리를 찾을 수 있습니다.

**Postman**

Postman은 이 엔드포인트들을 테스트하는 데 매우 유용한 도구입니다. 각 Postman 요청에는 모든 path 및 body 파라미터가 포함되어 있어, 어떤 기능을 사용할 수 있는지 빠르게 이해하는 데 도움이 됩니다. Postman 컬렉션에 대해 더 알아보려면 [Using Postman](/ko/tutorials/postman-getting-started) 페이지를 방문하세요.

<div id="key-concepts">
  ### Key concepts
</div>

**인증**

모든 X API v2 엔드포인트는 요청 시 키와 토큰이라고도 불리는 일련의 자격 증명을 사용해 [인증](/ko/resources/fundamentals/authentication)되어야 합니다. 이 엔드포인트 요청을 인증할 때는 OAuth 1.0a 사용자 컨텍스트(OAuth 1.0a User Context) 또는 OAuth 2.0 Authorization Code with PKCE 중 하나를 사용할 수 있습니다. 사용자 게시물 타임라인과 사용자 멘션 타임라인에는 OAuth 2.0 앱 전용(App-Only)을 사용할 수 있습니다.

[OAuth 1.0a 사용자 컨텍스트](/ko/resources/fundamentals/authentication#oauth-1-0a-2)는 요청에 함께 전달할 [Authorization 헤더를 생성](https://developer-staging.x.com/resources/fundamentals/authentication/authorizing-a-request)하기 위해 API 키, 사용자 액세스 토큰, 그리고 몇 가지 다른 파라미터를 사용해야 합니다. 액세스 토큰은 요청을 대신 보내려는 사용자의 계정과 연결되어 있어야 합니다. 다른 사용자를 위한 액세스 토큰을 생성하려면, 해당 사용자가 [3-legged OAuth 플로우](https://developer-staging.x.com/resources/fundamentals/authentication/obtaining-user-access-tokens)를 통해 귀하의 앱을 승인해야 합니다. 

OAuth 1.0a는 사용하기 까다로울 수 있다는 점에 유의하십시오. 이 인증 방식에 익숙하지 않다면, [라이브러리](/ko/x-api/tools-and-libraries/overview)를 사용하거나 Postman 같은 도구를 사용하거나, OAuth 2.0으로 요청을 인증하는 것을 권장합니다. 이 엔드포인트에서 게시물 또는 비공개 메트릭을 요청하려는 경우, 해당 콘텐츠를 소유한 사용자로부터 적절한 권한을 얻기 위해 반드시 OAuth 1.0a 사용자 컨텍스트 또는 OAuth 2.0 Authorization Code with PKCE를 사용해야 합니다.

[OAuth 2.0 앱 전용(App-Only)](/ko/resources/fundamentals/authentication#bearer-token-also-known-as-app-only)은 요청에 [OAuth 2.0 App Access Token](/ko/resources/fundamentals/authentication#bearer-token-also-known-as-app-only)을 포함하기만 하면 됩니다. 개발자 앱 내에서 바로 앱 액세스 토큰을 생성하거나, [POST oauth2/token](/ko/resources/fundamentals/authentication#post-oauth2-token) 엔드포인트를 사용해 생성할 수 있습니다. 이는 사용자 게시물 타임라인 또는 사용자 멘션 타임라인에 사용할 수 있습니다.

[OAuth 2.0 Authorization Code with PKCE](/ko/resources/fundamentals/authentication#oauth-2-0-authorization-code-flow-with-pkce-3)는 애플리케이션의 스코프와 여러 기기에 걸친 인증 플로우를 보다 세밀하게 제어할 수 있게 해 줍니다. OAuth 2.0을 사용하면 사용자 대신 특정 권한을 부여하는 세밀한 스코프를 선택할 수 있습니다. 

앱에서 OAuth 2.0을 사용하려면, developer 포털의 App settings 섹션에 있는 앱의 인증 설정에서 OAuth 2.0을 활성화해야 합니다.

  * `tweet.fields.non_public_metrics`
  * `tweet.fields.promoted_metrics`

  * `media.fields.organic_metrics`

**[](/ko/x-api/fundamentals/pagination)Developer portal, Projects, and developer Apps**

어떤 X API v2 엔드포인트를 사용하려면, 먼저 [개발자 계정에 가입](https://developer.x.com/en/portal/petition/essential/basic-info)하고 해당 계정 내에 [프로젝트](/ko/resources/fundamentals/projects)를 설정한 뒤, 그 프로젝트 내에 [개발자 앱](/ko/resources/fundamentals/developer-apps)을 생성해야 합니다. 해당 개발자 앱의 키와 토큰은 이 타임라인 엔드포인트들에 사용할 수 있습니다.

**요청 한도**

매일 수많은 개발자가 X API에 요청을 보냅니다. 이런 트래픽 규모를 관리하기 위해, 각 엔드포인트에는 앱을 대신해서 또는 인증된 사용자를 대신해서 각 개발자가 보낼 수 있는 요청 수를 제한하는 [요청 한도](/ko/x-api/fundamentals/rate-limits)가 설정되어 있습니다. 

이 엔드포인트들에는 사용 중인 인증 방식에 따라 서로 다른 요청 한도가 적용됩니다. 앱 수준 요청 한도는 OAuth 2.0 앱 전용(App-Only)을 사용해 요청을 보내는 앱에 적용되는 반면, 사용자 수준 요청 한도는 OAuth 1.0a 사용자 컨텍스트를 사용해 특정 승인 사용자를 대신하여 보내는 요청에 적용됩니다. 이 두 가지 요청 한도는 15분 단위 윈도우 내의 요청 빈도를 기준으로 합니다.

예를 들어, 이 두 타임라인 엔드포인트 모두에 대해 OAuth 2.0 앱 전용(App-Only) 인증을 사용하는 앱은 15분 동안 사용자 게시물 타임라인에 대해 페이지네이션 요청을 포함하여 1,500개의 요청을, 사용자 멘션 타임라인에 대해 페이지네이션 요청을 포함하여 450개의 요청을 보낼 수 있습니다. 동일한 앱이 동일한 15분 동안 두 명의 서로 다른 인가된 사용자(OAuth 1.0a 사용자 컨텍스트 사용)를 통해, 인증된 각 사용자마다 사용자 게시물 타임라인에는 페이지네이션 요청을 포함하여 900개의 요청을, 사용자 멘션 타임라인에는 페이지네이션 요청을 포함하여 180개의 요청을 보낼 수 있습니다. 

역순(최신순) 홈 타임라인은 사용자당 15분 윈도우당 180개의 요청 한도가 있습니다. 이 엔드포인트를 사용하면 지난 7일 동안 타임라인에 생성된 모든 게시물과, 생성 날짜와 관계없이 가장 최근의 800개 게시물을 반환할 수 있습니다.

**필드와 expansions**

X API v2에서는 [fields](/ko/x-api/fundamentals/fields)와 [expansions](/ko/x-api/fundamentals/expansions)를 사용해 API에서 반환받고 싶은 정확한 데이터를 선택할 수 있습니다. `expansions` 파라미터를 사용하면 페이로드에서 참조되는 객체를 확장할 수 있습니다. 예를 들어, 이 엔드포인트에서는 `expansions` 파라미터를 사용해 설문조사(poll), 장소(place), 미디어(media) 및 기타 객체를 요청할 수 있습니다.

`fields` 파라미터를 사용하면 서로 다른 데이터 객체 내에서 어떤 필드를 받을지 정확하게 선택할 수 있습니다. 기본적으로 이들 엔드포인트에서 반환되는 기본 게시물 객체에는 `id`와 `text` 필드가 포함됩니다. `author_id`나 `public_metrics`와 같은 추가 필드를 받으려면 `fields` 파라미터를 사용해 해당 필드를 명시적으로 요청해야 합니다. 통합 시 사용을 고려할 만한 중요한 필드로는 설문조사 데이터, [metrics](/ko/x-api/fundamentals/metrics), [Post annotations](/ko/x-api/fundamentals/post-annotations), [conversation ID](/ko/x-api/fundamentals/conversation-id) 필드 등이 있습니다.

[X API v2 데이터 사전](/ko/x-api/fundamentals/data-dictionary)에 [fields와 expansions를 함께 사용하는 방법](/ko/x-api/fundamentals/data-dictionary#how-to-use-fields-and-expansions)에 대한 가이드를 추가했습니다.

**게시물 메트릭**

X API v2 엔드포인트를 사용하면, 요청 시 올바른 필드를 전달하는 경우 반환된 게시물 객체에서 게시물 메트릭을 직접 요청할 수 있습니다.

게시물 메트릭에는 특히 사용자 프라이버시 및 다음 응답 필드와 관련된 몇 가지 제한 사항이 있습니다.

* `tweet.fields.non_public_metrics`
* `tweet.fields.promoted_metrics`
* `tweet.fields.organic_metrics`
* `media.fields.non_public_metrics`
* `media.fields.promoted_metrics`
* `media.fields.organic_metrics`

위의 필드에는 비공개 메트릭 데이터가 포함되므로, 사용자 게시물 타임라인 엔드포인트를 사용할 때 해당 게시물 게시자를 대신해 이 데이터를 조회하려면 반드시 게시물 게시자로부터 인가를 받아야 하며, 이를 위해서는 [OAuth 1.0a User Context](/ko/resources/fundamentals/authentication) 또는 [PKCE를 사용하는 OAuth 2.0 Authorization Code Flow](/ko/resources/fundamentals/authentication#oauth-2-0-authorization-code-flow-with-pkce-3)를 사용해야 합니다.

예를 들어, 사용자 ID 1234의 사용자 게시물 타임라인에 대해 `non_public_metrics`를 받으려면, 해당 사용자와 연결된 액세스 토큰을 요청에 포함해야 합니다. [3-legged OAuth 플로우](/ko/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow)를 사용해 사용자가 앱을 인가하도록 하고, 그 사용자와 연결된 액세스 토큰 세트를 받을 수 있습니다. 

사용자 멘션 타임라인을 사용하는 경우, 멘션한 작성자가 앱이 자신의 비공개 메트릭 데이터에 접근하도록 인가하지 않았거나, OAuth 1.0a 사용자 컨텍스트로 요청을 보낼 때 그 사용자의 액세스 토큰을 사용하지 않았다면 위의 필드를 사용할 수 없습니다.

모든 `non_public_metrics`, `organic_metrics`, `promoted_metrics`는 지난 30일 동안 생성된 게시물에만 사용할 수 있습니다. 따라서 위의 필드를 요청할 경우, 결과는 자동으로 최근 30일 이내의 게시물만 포함하도록 조정됩니다.

위의 필드를 요청하면, 인증된 사용자가 작성한 게시물만 반환되며, 그 외의 모든 게시물은 오류 메시지를 받게 됩니다.

**페이지네이션**

이들 엔드포인트는 페이지네이션을 사용하여 응답을 빠르게 반환합니다. 단일 응답으로 보낼 수 있는 결과 수(타임라인 엔드포인트의 경우 최대 100개 게시물)보다 더 많은 결과가 있는 경우 페이지네이션을 사용해야 합니다. 페이지당 반환할 결과 수를 지정하려면 `max_results` 파라미터를 사용하고, 다음 페이지의 결과를 반환받으려면 `pagination_token` 파라미터를 사용합니다. 자세한 내용은 [페이지네이션 가이드](/ko/x-api/fundamentals/pagination)를 참조하세요.

**결과 필터링**

이 엔드포인트들에는 결과를 필터링할 수 있는 여러 매개변수가 포함되어 있습니다. start&#95;date와 end&#95;date를 사용하면 결과를 특정 기간으로 좁힐 수 있습니다. 특정 게시물 집합을 선택하기 위해 게시물 ID를 사용하려면 since&#95;id와 until&#95;id를 사용할 수 있습니다. 사용자 게시물 타임라인에는 리트윗과 답글을 결과에서 제외할 수 있는 exclude 매개변수도 있습니다. 

**게시물 상한 및 반환되는 게시물 수**

사용자 게시물 타임라인 엔드포인트와 사용자 멘션 타임라인 엔드포인트는 한 달 동안 반환할 수 있는 게시물 개수에 제한이 있습니다. 역순 시간순 홈 타임라인 엔드포인트에는 이러한 제한이 적용되지 않습니다.

어느 타임라인 엔드포인트를 사용하든, 반환된 게시물은 프로젝트 수준 [Post caps](/ko/x-api/fundamentals/post-cap)에 포함됩니다. 사용량은 developer portal에 표시되며, &#39;month&#39;는 [developer portal dashboard](https://developer.x.com/en/portal/dashboard)에 표시된 구독 갱신일에 시작됩니다. 

사용자 게시물 타임라인 엔드포인트는 사용자의 타임라인에 게시된 가장 최근 3,200개의 게시물만 반환합니다. start&#95;time과 end&#95;time을 가장 최근 3,200개를 넘어서는 게시물이 포함되도록 설정하면, 응답 자체는 성공적으로 받지만 게시물은 반환되지 않습니다.

또한 사용자 게시물 타임라인 요청에 excludes=replies를 전달한 경우, 가장 최근 800개의 게시물만 반환된다는 점에 유의해야 합니다.

사용자 멘션 타임라인 엔드포인트는 가장 최근 800개의 게시물 멘션만 반환합니다.

역순 시간순 홈 타임라인 엔드포인트는 마지막 3,200개의 게시물을 반환합니다.

**게시물 수정**

수정 가능한 게시물은 원래 게시물이 게시된 후 30분 동안 최대 다섯 번까지 수정할 수 있습니다. 검색 엔드포인트는 항상 게시물의 최신 버전을 제공합니다. 게시된 지 30분 이상 지난 게시물만 요청하면, 항상 해당 게시물의 최종 버전만 받게 됩니다. 그러나 거의 실시간에 가까운 사용 사례로, 최근 30분 이내에 게시된 게시물을 조회하는 경우에는, 해당 게시물을 수신한 이후에 수정되었을 수 있습니다. 이러한 게시물은 search 또는 Post Lookup 엔드포인트로 다시 조회하여 최종 상태를 확인할 수 있습니다. 게시물 수정이 어떻게 동작하는지 더 알아보려면 [Edit Posts fundamentals](/ko/x-api/fundamentals/edit-posts) 페이지를 참조하세요.  

**예외 상황**

* 사용자 게시물 타임라인 엔드포인트에서 게시된 지 30일이 지난 게시물에 대해 비공개 메트릭을 요청하는 경우, 결과 개수가 0이면서 응답에 next&#95;token이 표시될 수 있습니다. 이 문제를 피하려면, non&#95;public&#95;metrics 매개변수로 요청하는 기간을 최근 30일 이내로 설정해야 합니다. 추가로, max&#95;results 최소 값은 10 이상이어야 합니다. 이러한 설정이 이 시나리오를 피하는 데 도움이 될 수 있지만, 여전히 발생할 수 있습니다.

* 프로모션된 적이 없는 게시물에 대해 promoted metrics를 요청하면, 게시물 데이터 대신 비어 있는 응답이 반환됩니다. 이 문제를 해결하기 위해 현재 작업을 진행 중입니다.

* 본문 텍스트 길이가 140자를 초과하는 게시물을 리트윗한 경우, 전체 게시물 텍스트를 반환하는 대신 text 필드가 잘립니다. 단기적인 우회 방법은 참조된 게시물을 확장(expand)하여, 확장에서 전체 텍스트를 가져오는 것입니다. 이는 향후 수정할 버그입니다.
