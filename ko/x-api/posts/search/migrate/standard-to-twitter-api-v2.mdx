<div id="standard-v11-compared-to-x-api-v2">
  ### 표준 v1.1과 X API v2 비교
</div>

v1.1 [search/posts](/ko/x-api/posts/search/introduction)를 사용해 왔다면, 이 가이드는 표준 검색 엔드포인트와 X API v2 검색 게시물 엔드포인트 간의 공통점과 차이점을 이해하는 데 도움이 되도록 작성되었습니다.

* **유사점**
  * OAuth 1.0a 사용자 컨텍스트와 OAuth 2.0 앱 전용(App-Only)
  * 게시물 편집 이력과 메타데이터 지원
* **차이점**
  * 엔드포인트 URL

  * 앱 및 프로젝트 요건

  * 응답 데이터 형식

  * 요청 매개변수

  * 새로운 쿼리 연산자

  * AND / OR 연산자 우선순위 

<div id="similarities">
  #### 유사점
</div>

**OAuth 1.0a 사용자 컨텍스트 및 OAuth 2.0 앱 전용(App-Only) 인증**

v1.1 search/posts 엔드포인트와 X API v2 recent search 엔드포인트는 모두 [OAuth 1.0a 사용자 컨텍스트](/ko/resources/fundamentals/authentication)와 [OAuth 2.0 앱 전용(App-Only)](/ko/resources/fundamentals/authentication#oauth-2-0)를 지원합니다. 

따라서 이전에 표준 v1.1 search 엔드포인트를 사용하고 있었다면, X API v2 버전으로 마이그레이션하더라도 동일한 인증 방식을 계속 사용할 수 있습니다. 

사용 중인 인증 라이브러리/패키지에 따라 다를 수 있지만, 앱 전용(App-Only) 인증이 시작하기에 가장 간단한 방법일 가능성이 높으며, 간단한 요청 헤더 설정만으로 사용할 수 있습니다. 앱 액세스 토큰을 생성하는 방법은 [OAuth 2.0 앱 전용(App-Only) 가이드](/ko/resources/fundamentals/authentication#bearer-token-also-known-as-app-only)를 참고하세요. 

X API v2 엔드포인트를 사용해 비공개 또는 광고용 메트릭을 조회하려면 OAuth 1.0a 사용자 컨텍스트를 사용해야 하며, 메트릭을 조회하려는 게시물을 작성한 사용자와 연결된 사용자 액세스 토큰을 전달해야 합니다. 

**게시물 수정 이력 및 메타데이터 지원**

두 버전 모두 수정 이력을 설명하는 메타데이터를 제공합니다. 자세한 내용은 [search API 레퍼런스](/ko/x-api/posts/recent-search)와 [게시물 수정 기본 사항 페이지](/ko/x-api/fundamentals/edit-posts)를 확인하세요. 

<div id="differences">
  #### 차이점
</div>

**엔드포인트 URL**

* Standard v1.1 엔드포인트:
  * https://api.x.com/1.1/search/tweets

* X API v2 엔드포인트:
  * https://api.x.com/2/tweets/search/recent
     

**앱 및 프로젝트 요구 사항**

X API v2 엔드포인트를 호출할 때는 요청을 인증하기 위해 [developer App](/ko/resources/fundamentals/developer-apps)과 [Project](/ko/resources/fundamentals/projects)에 연결된 자격 증명을 사용해야 합니다. 모든 X API v1.1 엔드포인트는 단독 App 또는 Project에 연결된 App의 자격 증명을 모두 사용할 수 있습니다. 

**응답 데이터 포맷**

Standard v1.1과 X API v2 엔드포인트 버전 간의 가장 큰 차이점 중 하나는 페이로드에서 어떤 필드를 반환할지 선택하는 방식입니다.

Standard 엔드포인트에서는 많은 응답 필드가 기본적으로 반환되고, 이후 파라미터를 사용하여 어떤 필드나 필드 집합을 페이로드에 포함할지 선택할 수 있습니다.

X API v2 버전은 기본적으로 게시물 id와 text 필드만 반환합니다. 추가적인 필드나 객체를 요청하려면 [fields](/ko/x-api/fundamentals/fields) 및 [expansions](/ko/x-api/fundamentals/expansions) 파라미터를 사용해야 합니다. 이 엔드포인트에서 요청한 모든 게시물 필드는 기본 Post 객체에 포함되어 반환됩니다. 확장된 사용자, 미디어, 투표(poll), 장소(place) 객체와 필드는 응답 내의 includes 객체에 포함되어 반환됩니다. 그런 다음 게시물과 확장된 객체 양쪽에 있는 ID를 맞춰서, 확장된 객체를 해당 게시물 객체와 연결할 수 있습니다. 

이 새로운 파라미터에 대해서는 각 가이드를 참고하시거나, [fields 및 expansions 사용 방법](/ko/x-api/fundamentals/data-dictionary#how-to-use-fields-and-expansions)에 관한 가이드를 읽어보시기를 권장합니다. 

또한 Standard v1.1 필드를 최신 v2 필드에 매핑하는 데 도움이 되는 [데이터 포맷 마이그레이션 가이드](/ko/x-api/migrate/data-format-migration#migrating-from-standard-v1-1s-data-format-to-v2)를 제공하고 있습니다. 이 가이드는 특정 필드를 반환하기 위해 v2 요청에 포함해야 하는 expansions 및 fields 파라미터도 구체적으로 제공합니다. 
 

특정 필드를 요청하는 방식의 변경 외에도, X API v2는 [Post](/ko/x-api/fundamentals/data-dictionary#tweet) 및 [user](/ko/x-api/fundamentals/data-dictionary#user) 객체를 포함하여 API가 반환하는 객체에 대해 새로운 JSON 설계를 도입하고 있습니다.

* JSON 루트 레벨에서 Standard 엔드포인트는 statuses 배열에 Post 객체를 반환하는 반면, X API v2는 data 배열에 반환합니다. 
* 리트윗(Retweeted) 및 인용(Quoted) &quot;statuses&quot;를 참조하는 대신, X API v2 JSON은 리트윗 및 인용 트윗을 참조합니다. contributors, user.translator&#95;type과 같은 많은 레거시 및 사용 중단 필드는 제거되고 있습니다. 
* Post 객체의 favorites와 user 객체의 favourites를 모두 사용하는 대신, X API v2는 like라는 용어를 사용합니다. 
* X는 값이 없는 JSON 값(예: null)은 페이로드에 기록하지 않는 규칙을 채택하고 있습니다. Post 및 user 속성은 null이 아닌 값을 가질 때만 포함됩니다. 
   

또한 [Post 객체](/ko/x-api/fundamentals/data-dictionary#tweet)에 다음과 같은 새로운 필드 세트를 추가했습니다.

* [conversation&#95;id](/ko/x-api/fundamentals/conversation-id) 필드
* context와 entities를 포함하는 두 개의 새로운 [annotations](/ko/x-api/fundamentals/post-annotations) 필드
* 여러 개의 새로운 [metrics](/ko/x-api/fundamentals/metrics) 필드 
* 특정 게시물에 누가 답장할 수 있는지를 보여주는 새로운 reply&#95;setting 필드

<div id="request-parameters">
  #### 요청 매개변수
</div>

다음 표준 v1.1 요청 매개변수는 X API v2에서 다음과 같이 대응됩니다:

| 표준 검색 v1.1 | Search Posts v2 |
| :--- | :--- |
| q   | query |
|     | start&#95;time (YYYY-MM-DDTHH:mm:ssZ) |
| until (YYYY-MM-DD) | end&#95;time (YYYY-MM-DDTHH:mm:ssZ) |
| since&#95;id | since&#95;id |
| max&#95;id | until&#95;id |
| count | max&#95;results |
| 응답에서 search&#95;metadata.next&#95;results 제공 | next&#95;token |

또한 X API v2에서는 다음과 같은 표준 검색용 Posts 요청 매개변수를 **지원하지 않습니다**:

| **표준 v1.1 매개변수** | **세부 정보** |
| :--- | :--- |
| geocode | Basic Access 레벨의 Search Posts는 geo 연산자를 지원하지 않습니다. |
| locale | 표준 검색에서는 이 값을 사용해 쿼리의 언어를 지정했지만, 완전히 구현되지는 않았습니다. |
| lang | Search Posts 엔드포인트는 관심 있는 언어에 대한 매칭을 위해 lang 쿼리 연산자를 제공합니다. |
| include&#95;entities | 게시물 엔티티는 항상 포함됩니다. |
| result&#95;type | Search Posts 엔드포인트는 참여 수준과 관계없이 일치하는 모든 게시물을 반환합니다. |
| extended | X API v2는 처음부터 최대 280자의 게시물을 지원하도록 설계되었습니다. v2에는 ‘확장(extended)’ 게시물이라는 개념이 없습니다. |

다음은 Standard 요청과 Search Posts 요청 간 차이를 보여 주는 예시 요청입니다:

|     |     |
| :--- | :--- |
| **표준 v1.1** | **X API v2** |
| https://api.x.com/1.1/search/tweets.json?q=snow&amp;count=50 | https://api.x.com/2/tweets/search/recent?query=snow&amp;max&#95;results=50 |

이 두 요청은 모두 키워드 snow를 포함하는 가장 최근 게시물 50개를 반환합니다. v2 요청은 일치하는 게시물의 기본 id 및 text 필드를 반환합니다. 다음은 JSON 페이로드에 포함할 추가 게시물 및 사용자 필드를 지정하는 예시입니다:

|     |
| :--- |
| **X API v2** |
| https://api.x.com/2/tweets/search/recent?query=snow&amp;max&#95;results=50&amp;tweet.fields=id,created&#95;at,author&#95;id,text,source,entities,attachments&amp;user.fields=id,name,username,description |

**새 쿼리 연산자**

Search Posts에서는 두 가지 X API v2의 새로운 기능을 지원하기 위해 새 연산자를 도입합니다: 

* **[Conversation IDs](/ko/x-api/fundamentals/conversation-id)** - X에서 대화가 전개되면, 대화에 속한 게시물을 표시하기 위한 conversation ID가 제공됩니다. 대화에 포함된 모든 게시물은 해당 대화를 시작한 게시물의 게시물 ID를 conversation&#95;id로 갖게 됩니다. 
  * `conversation_id:`
* **[X Annotations](/ko/x-api/fundamentals/post-annotations)** 는 게시물에 대한 문맥 정보를 제공하며, 엔티티 및 컨텍스트 애노테이션을 포함합니다. 엔티티는 사람, 장소, 제품, 조직으로 구성됩니다. 컨텍스트는 표면화된 엔티티가 속한 도메인 또는 주제를 의미합니다. 예를 들어, 게시물에 언급된 사람은 운동선수, 배우, 정치인인지 여부를 나타내는 컨텍스트를 가질 수 있습니다.  
  * context: 관심 있는 컨텍스트로 주석이 달린 게시물과 매칭합니다. 
  * entity: 관심 있는 엔티티로 주석이 달린 게시물과 매칭합니다. 

     

<div id="and-or-operator-precedence">
  #### AND / OR 연산자 우선순위
</div>

검색 쿼리를 구성하는 기본 요소는 OR와 AND 논리 연산자를 사용하는 것입니다. 표준 검색 API는 AND보다 OR을 먼저 적용하는 반면, Search Posts 엔드포인트(및 premium 및 Enterprise 버전)는 OR보다 AND를 먼저 적용합니다. 이 차이는 두 버전 간에 쿼리를 변환할 때 매우 중요합니다. 

예를 들어, 표준 검색에서 ‘storm’이라는 키워드를 포함하고, 단어 ‘colorado’ 또는 #COwx 해시태그가 언급된 게시물을 찾고 싶다면, 다음과 같은 검색 쿼리를 사용할 수 있습니다:

storm #COwx OR colorado

Search Posts의 연산자 우선순위에서는 AND가 OR보다 먼저 적용됩니다. 따라서 위 쿼리는 다음과 동일합니다: 

(storm #COwx) OR colorado

그러나 위 규칙은 게시물이 ‘storm’이나 #COwx 해시태그를 언급했는지와 관계없이, ‘Colorado’를 언급한 모든 게시물과도 일치하게 됩니다. 또한 키워드 ‘storm’과 #COwx 해시태그를 모두 언급한 게시물도 일치하게 됩니다. 

쿼리가 원래 의도대로 동작하도록 하려면 OR 절을 함께 묶어야 합니다. 원래 표준 쿼리를 Search Posts용으로 변환하면 다음과 같습니다:

storm (#COwx OR colorado)

이 두 규칙은 매우 다른 매칭 결과를 보입니다. 2019년 10월 한 달 동안, 원래 규칙은 1,175,000개가 넘는 게시물과 일치한 반면, 올바르게 변환된 규칙은 약 5,600개의 게시물과만 일치합니다. AND와 OR의 사용에 항상 유의하고, 필요한 곳에는 괄호를 사용하십시오. 

<div id="curl-requests">
  #### cURL 요청
</div>

**다음 섹션에서는 표준 v1.1 엔드포인트와 해당하는 v2 엔드포인트에 대한 cURL 요청 예시를 보여줍니다.**

요청은 [OAuth 2.0 App-Only]([https://developer.x.com(/resources/fundamentals/authentication#app-only-authentication-and-oauth-2-0-bearer-token)](https://developer.x.com\(/resources/fundamentals/authentication#app-only-authentication-and-oauth-2-0-bearer-token\))를 사용하여 수행됩니다. 다음 cURL 요청을 실행하려면 Authorization 헤더의 ACCESS&#95;TOKEN을 앱 액세스 토큰으로 바꿔야 합니다. v2 엔드포인트의 경우, 앱 액세스 토큰은 [Project](/ko/resources/fundamentals/projects/overview) 내의 [developer App](/ko/resources/fundamentals/developer-apps/overview)에 속해야 합니다. 

v1.1 엔드포인트가 반환하는 응답 페이로드는 v2 엔드포인트가 반환하는 응답 페이로드와 다릅니다. v2에서는 응답에 기본 필드뿐만 아니라 [fields](/ko/x-api/fundamentals/fields) 및 [expansions](/ko/x-api/fundamentals/expansions) 파라미터로 요청한 기타 필드도 포함됩니다. 이 파라미터들을 사용하여 어떤 필드를 반환할지 제어할 수 있습니다.

**표준 v1.1 [GET search/tweets](https://developer.x.com/en/docs/twitter-api/v1/tweets/search/api-reference/get-search-tweets) 및 v2 [GET tweets/search/recent](/ko/x-api/posts/recent-search) 엔드포인트**

```bash
      curl --request GET \
  --url 'https://api.x.com/1.1/search/tweets.json?q=from%3ATwitterDev%20-is%3Aretweet&count=100' \
  --header 'Authorization: Bearer $ACCESS_TOKEN'
```

```bash
      curl --request GET \
  --url 'https://api.x.com/2/tweets/search/recent?query=from%3ATwitterDev%20-is%3Aretweet&tweet.fields=created_at%2Cconversation_id%2Centities&max_results=100' \
  --header 'Authorization: Bearer $ACCESS_TOKEN'
```
