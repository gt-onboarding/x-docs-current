<div id="enterprise-compared-to-x-api-v2">
  ### Enterprise와 X API v2 비교
</div>

**유사점**

* 페이지네이션
* 시간대
* 게시물 편집 이력 및 메타데이터 지원

**차이점**

* 엔드포인트 URL
* 앱 및 프로젝트 요구 사항
* 지원되는 기간
* 응답 데이터 형식
* HTTP 메서드
* 요청 시간 형식
* 요청 매개변수
* 필터링 연산자

<div id="similarities">
  #### 유사점
</div>

**페이지네이션**

v2에는 추가적인 페이지네이션 기능(게시물 ID를 사용해 `since_id` 및 `until_id`로 탐색할 수 있게 해주는 새로운 페이지네이션 파라미터)이 있지만, Enterprise와 v2 모두 시간 기준(`fromDate` 및 `toDate`는 Enterprise, `start_time` 및 `end_time`은 v2)으로 페이지네이션을 할 수 있습니다.
 

**시간대**

페이지네이션 섹션에서 언급했듯이, Enterprise와 v2 모두 시간 기준으로 서로 다른 데이터 페이지를 탐색할 수 있습니다. 두 경우 모두 이러한 파라미터를 사용할 때는 UTC를 기준 시간대로 사용합니다.

**게시물 편집 내역 및 메타데이터 지원**

두 버전 모두 편집 내역을 설명하는 메타데이터를 제공합니다. 더 자세한 내용은 [search API 참고 문서](/ko/x-api/posts/recent-search) 및 [Post 편집 기본 사항 페이지](/ko/x-api/fundamentals/edit-posts)를 확인하세요. 

<div id="differences">
  #### 차이점
</div>

**엔드포인트 URL**

* Enterprise 엔드포인트:
  * 30일 - `http://gnip-api.x.com/search/30day/accounts/:account_name/:label.json`
  * 전체 아카이브 - `http://gnip-api.x.com/search/fullarchive/accounts/:account_name/:label.json`
* X API v2 엔드포인트
  * 최근 (7일) - `https://api.x.com/2/tweets/search/recent`
  * 전체 아카이브 - `https://api.x.com/2/tweets/search/all`

**앱 및 프로젝트 요구 사항**

X API v2 엔드포인트를 사용하려면 요청을 인증할 때 프로젝트의 자격 증명을 사용해야 합니다. 모든 X API v1.1 엔드포인트는 독립 실행형 앱 또는 프로젝트에 연결된 앱의 자격 증명을 사용할 수 있습니다.
 

**이용 가능한 기간**

Enterprise API와 X API v2 모두 전체 아카이브에 있는 게시물에 대해 필터링된 게시물 데이터를 검색할 수 있는 엔드포인트를 제공합니다.

그러나 X API v2는 Enterprise API처럼 30일 기간 엔드포인트를 제공하지는 않습니다. 대신 앞에서 언급한 전체 아카이브 또는 7일 기간을 제공하는데, 이는 Enterprise 데이터에서 v2로의 Native Enriched 및 [Activity Streams to v2](/ko/x-api/migrate/data-format-migration#migrating-from-activity-streams-data-format-to-v2)와 맞물려 있어 Enterprise 필드를 새로운 v2 필드에 매핑하는 데 도움이 됩니다. 이 가이드는 또한 v2 요청에 포함해 특정 필드를 반환받기 위해 전달해야 하는 구체적인 expansions 및 fields 파라미터도 제공합니다.
 

**응답 데이터 형식**

[Enterprise 응답 형식](https://developer.x.com/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary)과 [X API v2 형식](/ko/x-api/fundamentals/data-dictionary)의 가장 큰 차이점 중 하나는 페이로드에서 어떤 필드를 반환할지 선택하는 방식입니다.

Enterprise Search API의 경우, 응답 필드 대부분을 기본적으로 받게 되며, 그다음 파라미터를 사용해 페이로드에 반환해야 할 필드 또는 필드 집합을 지정할 수 있는 옵션이 있습니다.

X API v2 버전은 기본적으로 게시물 `id` 및 `text` 필드만 제공합니다. 추가 필드나 오브젝트를 요청하려면 [fields](/ko/x-api/fundamentals/fields) 및 [expansions](/ko/x-api/fundamentals/expansions) 파라미터를 사용해야 합니다. 이러한 엔드포인트에서 요청한 게시물 필드는 기본 게시물 오브젝트에 반환됩니다. 확장된 user, media, poll, place 오브젝트와 해당 필드는 응답 내 `includes` 오브젝트에 반환됩니다. 그런 다음 게시물과 확장된 오브젝트 양쪽에 있는 ID를 일치시켜 확장된 오브젝트를 게시물 오브젝트와 다시 연결할 수 있습니다.

이러한 새 파라미터에 대해서는 각 가이드를 더 읽어보시거나, [fields 및 expansions 사용 방법](/ko/x-api/fundamentals/data-dictionary#how-to-use-fields-and-expansions)에 대한 가이드를 참고하시기 바랍니다.

특정 필드를 요청하는 방식의 변경 외에도, X API v2는 API가 반환하는 오브젝트(예를 들어 [Post](/ko/x-api/fundamentals/data-dictionary#tweet) 및 [user](/ko/x-api/fundamentals/data-dictionary#user) 오브젝트)에 대해 새로운 JSON 설계를 도입하고 있습니다.

* JSON 루트 레벨에서, 표준 엔드포인트는 게시물 오브젝트를 `statuses` 배열로 반환하는 반면 X API v2는 `data` 배열로 반환합니다.
* 리트윗 및 인용된 &quot;statuses&quot;를 참조하는 대신, X API v2 JSON은 리트윗 및 인용 트윗을 참조합니다. `contributors`, `user.translator_type`과 같은 많은 레거시 및 사용 중단 필드는 제거되고 있습니다.
* 게시물 오브젝트의 `favorites`와 user 오브젝트의 `favourites`를 모두 사용하는 대신, X API v2는 like라는 용어를 사용합니다.
* X는 값이 없는 JSON 값(예: `null`)은 페이로드에 기록하지 않는 규칙을 채택하고 있습니다. 게시물 및 user 속성은 null이 아닌 값을 가진 경우에만 포함됩니다.
   

또한 게시물 오브젝트에도 다음과 같은 새로운 필드 집합이 도입되었습니다.

* [conversation&#95;id](/ko/x-api/fundamentals/conversation-id) 필드
* context와 entities를 포함하는 두 개의 새로운 [annotations](/ko/x-api/fundamentals/post-annotations) 필드
* 여러 개의 새로운 [metrics](/ko/x-api/fundamentals/metrics) 필드
* 특정 게시물에 누가 답글을 달 수 있는지를 보여주는 새로운 `reply_setting` 필드

마지막으로 한 가지 더. premium 응답에는 루트 레벨에 `requestParameters` 오브젝트가 포함되며, 여기에 요청에 포함한 파라미터가 들어 있습니다. 반면 v2 버전에는 루트 레벨에 `meta` 오브젝트가 포함되며, 추가 결과 페이지가 있는 경우 `newest_id`, `oldest_id`, `result_count`, `next_token`을 포함합니다.

**HTTP 메서드**

API의 Enterprise 버전에서는 JSON 본문을 포함하는 POST HTTP 메서드 또는 쿼리 문자열을 사용하는 GET HTTP 메서드로 요청을 보낼 수 있습니다.

v2에서는 쿼리 문자열을 사용하는 GET HTTP 메서드만 사용할 수 있습니다.
 

**요청 시간 형식**

이 엔드포인트의 Enterprise 버전은 페이지네이션 관련 매개변수와 `timePeriod` 응답 필드 모두에서 다음 날짜/시간 형식을 사용합니다: `YYYYMMDDHHmm`

v2 엔드포인트는 페이지네이션 관련 매개변수와 `start` 및 `end` 응답 필드 모두에서 ISO 8601/RFC 3339 날짜/시간 형식을 사용합니다: `YYYY-MM-DDTHH:mm:ssZ
 `

**요청 매개변수**

다음 표는 Enterprise와 X API v2의 요청 매개변수입니다:

| Enterprise | Search Posts v2 |
| :--- | :--- |
| query | query |
| maxResults | max&#95;results |
| fromDate (YYMMDDHHmm) | start&#95;time (YYYY-MM-DDTHH:mm:ssZ) |
| toDate (YYMMDDHHmm) | end&#95;time (YYYY-MM-DDTHH:mm:ssZ) |
|     | since&#95;id |
|     | until&#95;id |
| next | next&#95;token or pagination&#95;token |

**필터링 연산자**

Enterprise와 X API v2 간의 연산자는 대부분 동일하지만, 연산자 제공 여부 측면에서 몇 가지 차이가 있으며 X API v2 버전에만 새로 도입된 연산자도 있습니다.

X API v2, Enterprise, 그리고 premium 및 standard에서 사용 가능한 연산자의 전체 표를 보려면 [Search Posts migration landing page](/ko/x-api/posts/search/migrate/overview)를 방문하세요.
