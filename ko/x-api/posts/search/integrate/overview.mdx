<div id="authentication">
  #### 인증
</div>

모든 X API v2 엔드포인트는 키와 토큰(keys and tokens)이라고도 하는 자격 증명 세트를 사용해 요청을 인증해야 합니다. 최근 검색 엔드포인트에 대한 요청을 인증할 때는 OAuth 1.0a 사용자 컨텍스트(OAuth 1.0a User Context), OAuth 2.0 앱 전용(App-Only), 또는 OAuth 2.0 Authorization Code with PKCE 중 하나를 사용할 수 있습니다. 전체 아카이브 검색 엔드포인트를 사용할 때는 반드시 OAuth 2.0 앱 전용(App-Only)을 사용해야 합니다.

[OAuth 2.0 App-Only](/ko/resources/fundamentals/authentication#oauth-2-0-authorization-code-flow-with-pkce-3)는 요청에 [OAuth 2.0 App Access Token](/ko/resources/fundamentals/authentication#bearer-token-also-known-as-app-only)을 포함하기만 하면 됩니다. 개발자 앱 내에서 직접 App Access Token을 생성하거나, [POST oauth2/token](/ko/resources/fundamentals/authentication#post-oauth2-token) 엔드포인트를 사용하여 생성할 수 있습니다.

[OAuth 1.0a User Context](/ko/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow)를 사용하려면 API 키, 사용자 액세스 토큰, 그리고 몇 가지 추가 파라미터를 활용하여 [authorization 헤더를 생성](https://developer-staging.x.com/resources/fundamentals/authentication/authorizing-a-request)해야 하며, 이후 이 헤더를 요청에 포함해야 합니다. 액세스 토큰은 요청을 대신 수행하려는 해당 사용자와 연관되어 있어야 합니다. 다른 사용자를 위한 액세스 토큰 세트를 생성하려면, 해당 사용자가 [3-legged OAuth 플로우](https://developer-staging.x.com/resources/fundamentals/authentication/obtaining-user-access-tokens)를 통해 앱을 승인해야 합니다. 

OAuth 1.0a는 사용하기가 어려울 수 있다는 점에 유의하십시오. 이 인증 방식에 익숙하지 않다면, [라이브러리](/ko/x-api/tools-and-libraries/overview)를 사용하거나 Postman과 같은 도구를 사용하거나, OAuth 2.0을 사용해 요청을 인증할 것을 권장합니다. 이 엔드포인트들에서 게시물 또는 비공개 메트릭을 요청하려면 OAuth 1.0a User Context 또는 OAuth 2.0 Authorization Code with PKCE를 사용해야 하며, 이를 통해 해당 콘텐츠를 소유한 사용자의 적절한 권한을 보유하고 있음을 보장할 수 있습니다.

[OAuth 2.0 Authorization Code with PKCE](/ko/resources/fundamentals/authentication#oauth-2-0-authorization-code-flow-with-pkce-2)는 애플리케이션 범위(scope)에 대한 더 세밀한 제어와 여러 기기에 걸친 인증 플로우를 가능하게 합니다. OAuth 2.0을 사용하면 사용자 대신 수행할 수 있는 구체적인 권한을 부여하는 세분화된 스코프를 선택할 수 있습니다. 

앱에서 OAuth 2.0을 사용하려면, developer portal의 App 설정 섹션에 있는 해당 App의 인증 설정에서 이를 활성화해야 합니다.

<Note>
  **Please note**

  다음 [필드](/ko/x-api/fundamentals/fields)를 요청하는 경우, OAuth 1.0a User Context 또는 OAuth 2.0 Authorization Code가 필요합니다: 

  * tweet.fields.non&#95;public&#95;metrics
  * tweet.fields.promoted&#95;metrics
  * tweet.fields.organic&#95;metrics
  * media.fields.non&#95;public&#95;metrics
  * media.fields.promoted&#95;metrics
  * media.fields.organic&#95;metrics
</Note>

####

developer portal, 프로젝트, 그리고 개발자 앱

어떤 X API v2 엔드포인트를 사용하려면, 먼저 [개발자 계정에 가입](https://developer.x.com/en/portal/petition/essential/basic-info)한 뒤 해당 계정 내에 프로젝트를 설정하고, 그 프로젝트 안에 개발자 앱을 생성해야 합니다. 해당 개발자 앱의 키와 토큰은 이 검색 엔드포인트들에서 사용할 수 있습니다.

프로젝트의 키와 토큰은 [access level](/ko/x-api/getting-started/about-x-api)이 어떤 수준이든 최근 검색 엔드포인트에 요청을 보내는 데 사용할 수 있습니다. 그러나 전체 아카이브 검색 엔드포인트에 요청을 보내려면 Pro 또는 Enterprise access level이 있는 프로젝트를 사용해야 합니다. [Enterprise access](/ko/x-api/getting-started/getting-access)가 있으면, 추가 연산자 사용 가능 및 더 긴 쿼리 길이 등을 포함한 추가 기능에 접근할 수 있습니다.
 

<div id="rate-limits">
  #### 요청 한도
</div>

매일 수천 명이 넘는 개발자들이 X API에 요청을 보냅니다. 이 트래픽 규모를 관리하기 위해 각 엔드포인트에는 [요청 한도](/ko/x-api/fundamentals/rate-limits)가 설정되어 있으며, 이는 각 개발자가 앱 또는 인증된 사용자를 대신해 보낼 수 있는 요청 수를 제한합니다.

이러한 엔드포인트에는 사용 중인 인증 방식에 따라 서로 다른 요청 한도가 적용됩니다. 앱 수준 요청 한도는 OAuth 2.0 앱 전용(App-Only)을 사용해 요청을 보내는 앱에 적용되며, 사용자 수준 요청 한도는 OAuth 1.0a 사용자 컨텍스트 또는 OAuth 2.0 Authorization Code with PKCE를 사용해 특정 승인 사용자를 대신해 수행되는 요청에 적용됩니다. 이 두 가지 요청 한도는 15분 시간 창 내의 요청 빈도에 따라 결정됩니다.

예를 들어, OAuth 2.0 앱 전용(App-Only) 인증을 사용하는 앱이 최근 검색 엔드포인트에 요청을 보낼 때, 15분 동안 최대 450건의 요청(페이지네이션 요청 포함)을 보낼 수 있습니다. 동일한 앱이 같은 15분 시간 창에서 서로 다른 두 명의 인증된 사용자(각각 OAuth 1.0a 사용자 컨텍스트 또는 OAuth 2.0 Authorization Code with PKCE를 사용)를 사용하는 경우, 각 인증된 사용자마다 최근 검색 엔드포인트에 최대 180건의 요청(페이지네이션 요청 포함)을 보낼 수 있습니다.
 

<div id="fields-and-expansions">
  #### 필드와 expansions
</div>

X API v2를 사용하면 [fields](/ko/x-api/fundamentals/fields) 및 [expansions](/ko/x-api/fundamentals/expansions)를 통해 API에서 어떤 데이터가 반환될지 정확히 선택할 수 있습니다. expansions 파라미터를 사용하면 페이로드에서 참조된 객체를 확장할 수 있습니다. 예를 들어, 이 엔드포인트를 사용하면 expansions 파라미터를 통해 투표, 장소, 미디어 및 기타 객체를 요청할 수 있습니다.

fields 파라미터를 사용하면 각 데이터 객체 내에서 어떤 필드를 받을지 정확히 선택할 수 있습니다. 기본적으로 이 엔드포인트에서 반환되는 기본 게시물 객체에는 id 및 text 필드(해당 기능이 출시된 이후에 생성된 게시물의 경우 edit&#95;history&#95;tweet&#95;ids 필드도 포함)가 포함됩니다. author&#95;id 또는 public&#95;metrics와 같은 추가 필드를 받으려면 fields 파라미터를 사용해 해당 필드를 명시적으로 요청해야 합니다. 통합 시 사용을 고려할 만한 중요한 필드로는 투표 데이터, [metrics](/ko/x-api/fundamentals/metrics), [게시물 주석](/ko/x-api/fundamentals/post-annotations), 그리고 [conversation ID](/ko/x-api/fundamentals/conversation-id) 필드가 있습니다.

[X API v2 데이터 사전](/ko/x-api/fundamentals/data-dictionary)에 [fields와 expansions를 함께 사용하는 방법](/ko/x-api/fundamentals/data-dictionary#how-to-use-fields-and-expansions)에 대한 가이드를 제공하고 있습니다.
 

<div id="metrics">
  #### 메트릭
</div>

X API v2 엔드포인트를 사용하면 요청에 적절한 [fields](/ko/x-api/fundamentals/fields)를 함께 전달하는 경우 반환된 객체에서 메트릭을 직접 요청할 수 있습니다.

게시물 메트릭에는 알아 두어야 할 몇 가지 제한 사항이 있으며, 특히 사용자 프라이버시와 다음 응답 필드와 관련이 있습니다.

* tweet.fields.non&#95;public&#95;metrics
* tweet.fields.promoted&#95;metrics
* tweet.fields.organic&#95;metrics
* media.fields.non&#95;public&#95;metrics
* media.fields.promoted&#95;metrics
* media.fields.organic&#95;metrics

위에 언급된 필드에는 비공개 메트릭 데이터가 포함되어 있습니다. 이는 최근 검색 엔드포인트를 사용할 때 게시물 작성자가 대신 조회할 수 있도록 당신에게 권한을 부여해야 하며, 이를 위해 [OAuth 1.0a User Context](/ko/resources/fundamentals/authentication)를 사용해야 함을 의미합니다. 이 인증 방식은 최근 검색에서만 사용할 수 있기 때문에 전체 아카이브 검색 엔드포인트를 통해서는 이러한 메트릭을 조회할 수 없습니다.

예를 들어, 사용자 ID 1234의 게시물에 대한 `non_public_metrics`를 받으려면 해당 사용자와 연결된 액세스 토큰을 요청에 포함해야 합니다. [3-legged OAuth 플로우](/ko/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow)를 사용하면 사용자들이 앱을 승인하도록 하고, 그들과 연결된 액세스 토큰 세트를 발급받을 수 있습니다.

모든 `non_public_metrics`, `organic_metrics`, 및 `promoted_metrics`는 최근 30일 이내에 생성된 게시물에 대해서만 제공됩니다. 즉, 위에 언급된 필드를 요청하는 경우 결과는 자동으로 최근 30일 이내의 게시물만 포함하도록 조정됩니다.

이러한 필드가 요청되면 인증된 사용자가 작성한 게시물만 반환되며, 그 외 모든 게시물에는 오류가 반환됩니다.
 

<div id="building-search-queries">
  #### 검색 쿼리 구성
</div>

이 엔드포인트의 핵심 기능은 단일 검색 쿼리를 사용해 전달되는 게시물을 필터링하는 것입니다. 이러한 쿼리는 메시지 키워드, 해시태그, URL과 같은 게시물 및 사용자 속성과 일치하는 연산자로 구성됩니다. 연산자는 불리언 논리와 괄호를 사용해 쿼리로 결합할 수 있으며, 이를 통해 쿼리의 일치 방식을 더욱 세밀하게 조정할 수 있습니다.

[쿼리를 구성하는 방법](/ko/x-api/posts/search/integrate/build-a-query)에 대한 가이드를 참고해 더 자세히 알아볼 수 있습니다.

<div id="pagination">
  #### 페이지네이션
</div>

이 엔드포인트는 응답을 빠르게 반환하기 위해 페이지네이션을 사용합니다. 단일 응답으로 보낼 수 있는 결과 수(최신 검색의 경우 최대 100개의 게시물, 전체 아카이브 검색의 경우 최대 500개)를 초과하는 경우 페이지네이션을 사용해야 합니다. 페이지마다 반환할 결과 수를 지정하려면 max&#95;results 파라미터를 사용하고, 다음 페이지의 결과를 가져오려면 pagination&#95;token 파라미터를 사용하세요. 자세한 내용은 [페이지네이션 가이드](/ko/x-api/fundamentals/pagination)를 참고하세요.

<div id="post-caps">
  #### 게시물 상한
</div>

Search Posts 엔드포인트는 페이지네이션 여부와 관계없이, 한 달 동안 반환할 수 있는 게시물 개수에 제한이 있습니다.

어떤 검색 엔드포인트를 사용하더라도, 반환된 게시물은 프로젝트 단위 [게시물 상한](/ko/x-api/fundamentals/post-cap)에 포함됩니다. 사용량은 developer portal에 표시되며, &#39;월&#39; 단위는 [developer portal dashboard](https://developer.x.com/en/portal/dashboard)에 표시되는 구독 갱신 날짜를 기준으로 시작됩니다.

**게시물 수정**

수정이 허용되는 게시물은 원본 게시물이 게시된 후 30분 동안 최대 다섯 번까지 수정할 수 있습니다. 검색 엔드포인트는 항상 해당 게시물의 최신 버전을 제공합니다. 게시된 지 30분 이상 지난 게시물만 요청하는 경우, 항상 최종 버전의 게시물을 받게 됩니다. 그러나 거의 실시간에 가까운 사용 사례에서 게시된 지 30분 이내의 게시물을 쿼리하는 경우, 해당 게시물이 여러분이 수신한 이후에 수정되었을 수 있습니다. 이러한 게시물은 검색 또는 Post Lookup 엔드포인트를 사용해 다시 조회(rehydrate)하여 최종 상태를 확인할 수 있습니다. 게시물 수정 동작 방식에 대한 자세한 내용은 [Post edits fundamentals](/ko/x-api/fundamentals/edit-posts) 페이지를 참고하세요.
