<div id="authentication">
  #### Authentication
</div>

모든 X API v2 엔드포인트를 호출할 때에는 키와 토큰으로 구성된 자격 증명 세트를 사용해 요청을 인증해야 합니다. 최근 검색 엔드포인트에 대한 요청을 인증할 때에는 OAuth 1.0a 사용자 컨텍스트(OAuth 1.0a User Context), OAuth 2.0 앱 전용(App-Only), 또는 OAuth 2.0 Authorization Code with PKCE 중 하나를 사용할 수 있습니다. 전체 아카이브 검색 엔드포인트를 사용할 때에는 반드시 OAuth 2.0 앱 전용(App-Only)을 사용해야 합니다.

[OAuth 2.0 App-Only](/ko/resources/fundamentals/authentication#oauth-2-0-authorization-code-flow-with-pkce-3)를 사용하면, 요청에 [OAuth 2.0 App Access Token](/ko/resources/fundamentals/authentication#bearer-token-also-known-as-app-only)만 함께 전달하면 됩니다. App Access Token은 개발자 앱에서 직접 생성하거나, [POST oauth2/token](/ko/resources/fundamentals/authentication#post-oauth2-token) 엔드포인트를 사용해 생성할 수 있습니다.

[OAuth 1.0a User Context](/ko/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow)를 사용하려면 API 키, 사용자 액세스 토큰, 그리고 몇 가지 다른 매개변수를 활용해 [Authorization 헤더를 생성](https://developer-staging.x.com/resources/fundamentals/authentication/authorizing-a-request)한 다음, 이를 요청과 함께 전달해야 합니다. 액세스 토큰은 요청을 대신 수행하려는 사용자의 계정과 연결되어 있어야 합니다. 다른 사용자를 위한 액세스 토큰 세트를 생성하려면, 해당 사용자가 [3-legged OAuth 플로우](https://developer-staging.x.com/resources/fundamentals/authentication/obtaining-user-access-tokens)를 사용해 귀하의 앱을 승인해야 합니다. 

OAuth 1.0a는 사용하기 까다로울 수 있다는 점에 유의하세요. 이 인증 방식에 익숙하지 않다면, [라이브러리](/ko/x-api/tools-and-libraries/overview)를 사용하거나 Postman 같은 도구를 사용하거나, OAuth 2.0을 사용해 요청을 인증할 것을 권장합니다. 이 엔드포인트들에서 게시물이나 비공개 메트릭을 요청하려면, OAuth 1.0a 사용자 컨텍스트 또는 OAuth 2.0 Authorization Code with PKCE 중 하나를 사용해야 하며, 이를 통해 해당 콘텐츠를 소유한 사용자의 적절한 권한을 부여받았는지 확인할 수 있습니다.

[OAuth 2.0 Authorization Code with PKCE](/ko/resources/fundamentals/authentication#oauth-2-0-authorization-code-flow-with-pkce-2)는 애플리케이션의 스코프와 여러 기기에 걸친 인가 플로우를 보다 세밀하게 제어할 수 있게 해 줍니다. OAuth 2.0을 사용하면 사용자 대신 수행할 수 있는 권한을 세밀하게 구분한 스코프를 선택할 수 있습니다. 

앱에서 OAuth 2.0을 활성화하려면, developer portal의 App settings 섹션에 있는 앱 인증 설정에서 OAuth 2.0을 활성화해야 합니다.

<Note>
  **참고하세요**

  아래 [fields](/ko/x-api/fundamentals/fields)를 요청하는 경우, OAuth 1.0a User Context 또는 OAuth 2.0 Authorization Code가 필요합니다: 

  * tweet.fields.non&#95;public&#95;metrics
  * tweet.fields.promoted&#95;metrics
  * tweet.fields.organic&#95;metrics
  * media.fields.non&#95;public&#95;metrics
  * media.fields.promoted&#95;metrics
  * media.fields.organic&#95;metrics
</Note>

####

Developer portal, Projects, and developer Apps

어떤 X API v2 endpoint를 사용하려면 먼저 [개발자 계정에 가입](https://developer.x.com/en/portal/petition/essential/basic-info)하고, 해당 계정 내에 프로젝트(Project)를 설정한 뒤, 그 프로젝트 내에 개발자 앱(developer App)을 생성해야 합니다. 해당 개발자 앱에 있는 키와 토큰은 이 검색 endpoint들에서 사용할 수 있습니다.

어떤 [access level](/ko/x-api/getting-started/about-x-api)이든, 해당 프로젝트(Project)의 키와 토큰을 사용해 최근 검색(recent search) endpoint에 요청을 보낼 수 있습니다. 하지만 전체 아카이브 검색(full archive search) endpoint에 요청을 보내려면 Pro 또는 Enterprise access level을 가진 프로젝트(Project)를 사용해야 합니다. [Enterprise access](/ko/x-api/getting-started/getting-access)가 있는 경우, 추가 operator 사용 및 더 긴 쿼리 길이 등 추가 기능을 사용할 수 있습니다.
 

<div id="rate-limits">
  #### 요청 한도
</div>

매일 수천 명이 넘는 개발자들이 X API에 요청을 보냅니다. 요청량을 관리하기 위해 각 엔드포인트에는 [요청 한도](/ko/x-api/fundamentals/rate-limits)가 설정되어 있으며, 이를 통해 모든 개발자가 앱을 대신하거나 인증된 사용자를 대신해 보낼 수 있는 요청 수가 제한됩니다.

이 엔드포인트들에는 사용 중인 인증 방식에 따라 서로 다른 요청 한도가 적용됩니다. 앱 수준 요청 한도는 OAuth 2.0 앱 전용(App-Only)을 사용해 요청을 보내는 앱에 적용되고, 사용자 수준 요청 한도는 OAuth 1.0a 사용자 컨텍스트 또는 OAuth 2.0 Authorization Code with PKCE를 사용해 특정 승인 사용자를 대신하여 이루어지는 요청에 적용됩니다. 이 두 종류의 요청 한도는 15분 단위 시간 구간(time window) 내의 요청 빈도에 기반합니다.

예를 들어, OAuth 2.0 앱 전용(App-Only) 인증을 사용하는 앱은 최근 검색 엔드포인트에 대해 15분 동안 450개의 요청(페이지네이션 요청 포함)을 보낼 수 있습니다. 같은 앱이 동일한 15분 동안 두 명의 서로 다른 인증된 사용자(각각 OAuth 1.0a 사용자 컨텍스트 또는 OAuth 2.0 Authorization Code with PKCE 중 하나를 사용)를 통해 요청을 보낼 경우, 각 인증된 사용자별로 최근 검색 엔드포인트에 최대 180개의 요청(페이지네이션 요청 포함)을 보낼 수 있습니다.
 

<div id="fields-and-expansions">
  #### 필드와 expansions
</div>

X API v2를 사용하면 [fields](/ko/x-api/fundamentals/fields)와 [expansions](/ko/x-api/fundamentals/expansions)를 통해 API에서 어떤 데이터를 반환받을지 정확하게 선택할 수 있습니다. `expansions` 파라미터를 사용하면 페이로드에서 참조되는 객체를 확장할 수 있습니다. 예를 들어, 이 엔드포인트에서는 `expansions` 파라미터를 사용해 투표, 장소, 미디어 및 기타 객체를 요청할 수 있습니다.

`fields` 파라미터를 사용하면 여러 데이터 객체에서 어떤 필드를 받을지 정확하게 선택할 수 있습니다. 기본적으로, 이 엔드포인트들이 반환하는 주요 게시물 객체에는 `id` 및 `text` 필드가 포함됩니다(해당 기능이 출시된 이후에 생성된 게시물의 경우 `edit_history_tweet_ids`도 추가로 포함). `author_id` 또는 `public_metrics`와 같은 추가 필드를 받으려면 `fields` 파라미터를 사용해 명시적으로 요청해야 합니다. 통합 시 활용을 고려할 만한 중요한 필드로는 투표 데이터, [metrics](/ko/x-api/fundamentals/metrics), [Post annotations](/ko/x-api/fundamentals/post-annotations), [conversation ID](/ko/x-api/fundamentals/conversation-id) 필드 등이 있습니다.

[X API v2 data dictionary](/ko/x-api/fundamentals/data-dictionary)에 [fields와 expansions를 함께 사용하는 방법](/ko/x-api/fundamentals/data-dictionary#how-to-use-fields-and-expansions)에 대한 가이드를 추가했습니다.
 

<div id="metrics">
  #### 메트릭
</div>

X API v2 엔드포인트를 사용하면, 요청에 적절한 [fields](/ko/x-api/fundamentals/fields)를 포함하는 경우 반환된 객체에서 메트릭을 직접 요청할 수 있습니다.

게시물 메트릭에는 알아두어야 할 몇 가지 제한 사항이 있으며, 특히 사용자 프라이버시와 다음 응답 필드와 관련이 있습니다.

* tweet.fields.non&#95;public&#95;metrics
* tweet.fields.promoted&#95;metrics
* tweet.fields.organic&#95;metrics
* media.fields.non&#95;public&#95;metrics
* media.fields.promoted&#95;metrics
* media.fields.organic&#95;metrics

위에 언급된 필드에는 비공개 메트릭 데이터가 포함되어 있습니다. 이는 최근 검색 엔드포인트를 사용할 때 게시물 작성자의 권한을 받아 그들을 대신해 이 데이터를 가져와야 하며, 이를 위해 [OAuth 1.0a User Context](/ko/resources/fundamentals/authentication)를 사용해야 한다는 의미입니다. 이 인증 방식은 최근 검색에서만 사용할 수 있으므로, 전체 아카이브 검색 엔드포인트를 통해서는 이러한 메트릭을 가져올 수 없습니다.

예를 들어, 사용자 ID 1234의 게시물에 대한 `non_public_metrics`를 받으려면 해당 사용자와 연결된 액세스 토큰을 요청에 포함해야 합니다. [3-legged OAuth flow](/ko/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow)를 사용하면, 사용자에게 앱을 승인받고 그 사용자와 연결된 액세스 토큰 세트를 받을 수 있습니다.

모든 `non_public_metrics`, `organic_metrics`, 그리고 `promoted_metrics`는 최근 30일 이내에 생성된 게시물에만 사용할 수 있습니다. 따라서 이 필드들을 요청하면, 결과는 자동으로 조정되어 최근 30일 이내의 게시물만 포함하게 됩니다.

이러한 필드가 요청되면 인증된 사용자가 작성한 게시물만 반환되며, 그 외 게시물에는 오류가 반환됩니다.
 

<div id="building-search-queries">
  #### 검색 쿼리 구성
</div>

이 엔드포인트의 핵심 기능은 단일 검색 쿼리를 사용해 전달할 게시물을 필터링하는 것입니다. 이러한 쿼리는 메시지 키워드, 해시태그, URL과 같은 게시물 및 사용자 속성과 일치하는 연산자로 구성됩니다. 연산자는 불리언 논리와 괄호를 사용해 쿼리로 결합할 수 있으며, 이를 통해 쿼리의 일치 방식을 세밀하게 조정할 수 있습니다.

자세한 내용은 [쿼리를 구성하는 방법](/ko/x-api/posts/search/integrate/build-a-query)에 대한 가이드를 참고하세요.

<div id="pagination">
  #### 페이지네이션
</div>

이 엔드포인트는 응답을 빠르게 반환하기 위해 페이지네이션을 사용합니다. 하나의 응답으로 보낼 수 있는 결과 수(최근 검색의 경우 게시물 최대 100개, 전체 아카이브 검색의 경우 최대 500개)를 초과하는 결과가 있는 경우 페이지네이션을 사용해야 합니다. 한 페이지에 반환할 결과 수를 지정하려면 max&#95;results 파라미터를 사용하고, 다음 페이지의 결과를 받으려면 pagination&#95;token 파라미터를 사용하세요. 자세한 내용은 [페이지네이션 가이드](/ko/x-api/fundamentals/pagination)를 참고하세요.

<div id="post-caps">
  #### 게시물 상한
</div>

Search Posts 엔드포인트는 페이지네이션과 관계없이, 한 달 동안 반환할 수 있는 게시물의 개수가 제한됩니다.

어떤 검색 엔드포인트를 사용하든, 반환된 게시물은 프로젝트 수준 [Post caps](/ko/x-api/fundamentals/post-cap)에 포함됩니다. 사용량은 developer portal에 표시되며, 여기서의 ‘month’는 [developer portal dashboard](https://developer.x.com/en/portal/dashboard)에 표시된 구독 갱신일을 기준으로 시작됩니다.

**게시물 편집**

편집이 가능한 게시물은 원본 게시물이 게시된 시점부터 30분 동안 최대 다섯 번까지 편집할 수 있습니다. 검색 엔드포인트는 항상 해당 게시물의 최신 버전을 제공합니다. 게시된 지 30분 이상 지난 게시물만 요청하는 경우, 항상 해당 게시물의 최종 버전을 받게 됩니다. 그러나 사실상 실시간에 가까운 사용 사례가 있어 최근 30분 이내에 게시된 게시물을 조회하는 경우, 이 게시물들은 수신한 뒤에 편집되었을 수 있습니다. 이러한 게시물은 검색 또는 Post Lookup 엔드포인트를 사용해 다시 리하이드레이션(rehydrate)하여 최종 상태를 확인할 수 있습니다. 게시물 편집 방식에 대한 자세한 내용은 [Post edits fundamentals](/ko/x-api/fundamentals/edit-posts) 페이지를 참조하세요.
