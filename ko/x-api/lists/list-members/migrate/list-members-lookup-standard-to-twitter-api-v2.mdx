<div id="list-members-lookup-standard-v11-compared-to-x-api-v2">
  ### List 멤버 조회: Standard v1.1과 X API v2 비교
</div>

Standard v1.1 [GET lists/members](https://developer.x.com/en/docs/twitter-api/v1/accounts-and-users/create-manage-lists/api-reference/get-lists-members) 및 [GET lists/memberships](https://developer.x.com/en/docs/twitter-api/v1/accounts-and-users/create-manage-lists/api-reference/get-lists-memberships) 엔드포인트를 사용해 왔다면, 이 가이드는 Standard v1.1과 X API v2 List 멤버 엔드포인트 간의 유사점과 차이점을 이해하는 데 도움이 되도록 작성되었습니다.

* **유사점**
  * 인증 방식
* **차이점**
  * 엔드포인트 URL
  * 요청 한도
  * 앱 및 프로젝트 요구 사항
  * 요청당 데이터 객체 개수 한도
  * 응답 데이터 형식
  * 요청 매개변수

<div id="similarities">
  #### 유사점
</div>

**인증**

두 버전의 엔드포인트 모두 [OAuth 1.0a 사용자 컨텍스트](/ko/resources/fundamentals/authentication#oauth-1-0a-2)와 [앱 전용](/ko/resources/fundamentals/authentication#oauth-2-0) 인증을 지원합니다. 따라서 이전에 표준 v1.1 List 멤버 엔드포인트 중 하나를 사용하고 있었다면, X API v2 버전으로 마이그레이션하더라도 동일한 인증 방식을 계속 사용할 수 있습니다.

사용하는 인증 라이브러리/패키지에 따라 앱 전용 인증이 가장 간편한 시작 방법인 경우가 많으며, 단순히 요청 헤더만 설정해서 사용할 수 있습니다. 앱 전용 액세스 토큰을 생성하는 방법을 알아보려면 [이 앱 전용 가이드](/ko/resources/fundamentals/authentication#bearer-token-also-known-as-app-only)를 참고하세요.

<div id="differences">
  #### 차이점
</div>

**엔드포인트 URL**

* 표준 v1.1 엔드포인트:
  * GET https://api.x.com/1.1/lists/members.json
    (지정된 리스트의 멤버 조회)
  * GET https://api.x.com/1.1/lists/memberships.json
    (사용자가 멤버로 포함된 리스트 조회)
* X API v2 엔드포인트:
  * GET https://api.x.com/2/lists/:id/members
    (지정된 리스트의 멤버 조회)

  * GET https://api.x.com/2/users/:id/list&#95;memberships
    (사용자가 멤버로 포함된 리스트 조회)

**요청 한도**

|     |     |
| :--- | :--- |
| **Standard v1.1** | **X API v2** |
| /1.1/lists/members.json<br /><br />OAuth 1.0a 사용자 컨텍스트로 15분당 900회 요청<br /><br />앱 전용(App only) 인증으로 15분당 15회 요청 | /2/lists/:id/members<br /><br />OAuth 1.0a 사용자 컨텍스트로 15분당 900회 요청<br /><br />OAuth 2.0 Authorization Code with PKCE로 15분당 900회 요청<br /><br />앱 전용(App only) 인증으로 15분당 900회 요청 |
| /1.1/lists/memberships.json<br /><br />OAuth 1.0a 사용자 컨텍스트로 15분당 15회 요청<br /><br />앱 전용(App only) 인증으로 15분당 15회 요청 | /2/users/:id/list&#95;memberships<br /><br />OAuth 1.0a 사용자 컨텍스트로 15분당 15회 요청<br /><br />OAuth 2.0 Authorization Code with PKCE로 15분당 15회 요청<br /><br />앱 전용(App only) 인증으로 15분당 15회 요청 |

**앱 및 프로젝트 요구 사항**

X API v2 엔드포인트를 사용하려면 요청을 인증할 때 [프로젝트](/ko/resources/fundamentals/projects)에 연결된 [개발자 앱](/ko/resources/fundamentals/developer-apps)의 자격 증명을 사용해야 합니다. 모든 X API v1.1 엔드포인트는 독립 실행형 앱 또는 프로젝트에 연결된 앱의 자격 증명을 사용할 수 있습니다.

**요청당 데이터 객체 한도**

표준 v1.1 /1.1/lists/members 엔드포인트는 요청당 최대 5000명의 사용자를 반환할 수 있습니다. 새로운 v2 엔드포인트는 요청당 최대 100명의 사용자를 반환할 수 있습니다. 기본적으로 100개의 사용자 객체가 반환되며, 결과 개수를 변경하려면 1–100 사이의 숫자와 함께 쿼리 매개변수 max&#95;results=를 전달해야 합니다. 그런 다음 응답 페이로드에 반환된 next&#95;token 값을 다음 요청에서 pagination&#95;token 쿼리 매개변수로 전달하여 페이지네이션할 수 있습니다.

추가로, /1.1/lists/memberships 엔드포인트는 요청당 최대 1000개의 리스트를 반환할 수 있습니다. v2 교체 엔드포인트에서는 요청당 최대 100개의 리스트를 허용합니다. 기본적으로 100개의 리스트 객체가 반환되며, /1.1/lists/members와 동일한 방식으로 쿼리 매개변수 max&#95;results= 및 pagination&#95;token을 사용하여 결과 개수를 변경합니다.

**응답 데이터 형식**

표준 v1.1 엔드포인트와 X API v2 엔드포인트 버전 간의 가장 큰 차이점 중 하나는 페이로드에 어떤 필드를 반환할지 선택하는 방식입니다.

표준 엔드포인트에서는 기본적으로 많은 응답 필드가 반환되고, 이후 매개변수를 사용해 페이로드에 추가로 반환되어야 하는 필드 또는 필드 집합을 지정할 수 있습니다.

X API v2 버전 /users/:id/list&#95;memberships는 기본적으로 List id 및 name 필드를 제공합니다. 추가 필드나 객체를 요청하려면 [fields](/ko/x-api/fundamentals/fields) 및 [expansions](/ko/x-api/fundamentals/expansions) 매개변수를 사용해야 합니다. 이 엔드포인트에서 요청한 모든 List 필드는 기본 List 객체에 반환됩니다. 확장된 객체와 필드는 응답 내 includes 객체에 반환됩니다. 그런 다음 기본 객체와 확장된 객체 모두에 있는 ID를 매칭하여, 확장된 객체를 기본 List 객체와 다시 연결할 수 있습니다.

다음은 가능한 List 필드 및 expansions 예시입니다:

* created&#95;at

* follower&#95;count

* member&#95;count

* owner&#95;id

* description

* private

|     |     |
| :--- | :--- |
| **엔드포인트** | **확장(Expansion)** |
| /2/lists/:id/members | pinned&#95;tweet&#95;id |
| /2/users/:id/list&#95;memberships | owner&#95;id |

이 새로운 매개변수에 대해 더 자세히 알아보려면, 각 가이드를 참고하거나 [fields 및 expansions 사용 방법](/ko/x-api/fundamentals/data-dictionary#how-to-use-fields-and-expansions)에 대한 가이드를 읽어보시기 바랍니다. 

또한 표준 v1.1 필드를 새로운 v2 필드에 매핑하는 데 도움이 되는 [데이터 형식 마이그레이션 가이드](/ko/x-api/migrate/data-format-migration)를 제공하고 있습니다. 이 가이드는 특정 필드를 반환하기 위해 v2 요청에 함께 전달해야 하는 expansion 및 field 파라미터도 구체적으로 안내합니다. 

특정 필드를 요청하는 방식의 변경에 더해, X API v2는 [게시물](/ko/x-api/fundamentals/data-dictionary#tweet) 및 [사용자](/ko/x-api/fundamentals/data-dictionary#user) 객체를 포함하여 API가 반환하는 객체에 대해 새로운 JSON 설계도 도입하고 있습니다.

* JSON 루트 수준에서, 표준 엔드포인트는 statuses 배열로 Post 객체를 반환하는 반면, X API v2는 data 배열을 반환합니다. 

* 리트윗된 및 인용된 항목을 &quot;statuses&quot;로 참조하는 대신, X API v2 JSON은 리트윗된 및 인용된 트윗으로 참조합니다. contributors 및 user.translator&#95;type과 같은 많은 레거시 및 사용 중단 필드는 제거됩니다. 

* Post 객체에서 favorites, user 객체에서 favourites를 모두 사용하는 대신, X API v2는 like라는 용어를 사용합니다. 

* X는 값이 없는 JSON 값(예: null)은 페이로드에 기록하지 않는 관례를 채택하고 있습니다. Post 및 user 속성은 null이 아닌 값을 가진 경우에만 포함됩니다.

**요청 파라미터**

다음 표준 v1.1 요청 파라미터는 X API v2에서 대응하는 항목이 있습니다:

**리스트 멤버 조회**

|     |     |
| :--- | :--- |
| **Standard v1.1** | **X API v2** |

| owner&#95;screen&#95;name | 해당 사항 없음 |
| owner&#95;id | 해당 사항 없음 |
| count | max&#95;results |
| cursor | pagination&#95;token |
