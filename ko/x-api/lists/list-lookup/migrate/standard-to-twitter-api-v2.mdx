<div id="list-lookup-standard-v11-compared-to-x-api-v2">
  ### 리스트 조회: 표준 v1.1과 X API v2 비교
</div>

표준 v1.1의 [GET lists/show](https://developer.x.com/en/docs/twitter-api/v1/accounts-and-users/create-manage-lists/api-reference/get-lists-show) 및 [GET lists/ownerships](https://developer.x.com/en/docs/twitter-api/v1/accounts-and-users/create-manage-lists/api-reference/get-lists-ownerships) 엔드포인트를 사용해 왔다면, 이 가이드는 표준 v1.1과 X API v2 리스트 조회 엔드포인트 간의 유사점과 차이점을 이해하는 데 도움이 되도록 작성되었습니다.

* **유사점**
  * 인증 방식
  * 요청 한도
* **차이점**
  * 엔드포인트 URL
  * 앱 및 프로젝트 요구 사항
  * 요청당 데이터 객체 개수 제한
  * 응답 데이터 형식
  * 요청 매개변수

<div id="similarities">
  #### 유사점
</div>

**인증**

두 엔드포인트 버전은 모두 [OAuth 1.0a 사용자 컨텍스트](/ko/resources/fundamentals/authentication#oauth-1-0a-2)와 [App 전용](/ko/resources/fundamentals/authentication#app-only-authentication-and-oauth-2-0-bearer-token)을 지원합니다. 따라서 이전에 기본 v1.1 리스트 조회 엔드포인트 중 하나를 사용하고 있었다면, X API v2 버전으로 마이그레이션하더라도 동일한 인증 방식을 계속 사용할 수 있습니다.

사용하는 인증 라이브러리/패키지에 따라 다를 수 있지만, App 전용 인증 방식이 시작하기에 가장 간단한 방법일 가능성이 높으며, 간단한 요청 헤더로 설정할 수 있습니다. App 전용 액세스 토큰을 생성하는 방법을 알아보려면 [이 App 전용 가이드](/ko/resources/fundamentals/authentication#using-and-generating-an-app-only-bearer-token)를 참고하세요.

**요청 한도**

|     |     |
| :--- | :--- |
| **Standard v1.1** | **X API v2** |
| /1.1/lists/show.json<br /><br />OAuth 1.0a 사용자 컨텍스트로 15분당 75회 요청<br /><br />App 전용으로 15분당 75회 요청 | /2/lists/:id<br /><br />OAuth 1.0a 사용자 컨텍스트로 15분당 75회 요청<br /><br />OAuth 2.0 Authorization Code with PKCE로 15분당 75회 요청 |
| /1.1/lists/ownerships.json<br /><br />OAuth 1.0a 사용자 컨텍스트로 15분당 15회 요청<br /><br />App 전용으로 15분당 15회 요청 | /2/users/:id/owned&#95;lists<br /><br />OAuth 1.0a 사용자 컨텍스트로 15분당 15회 요청<br /><br />OAuth 2.0 Authorization Code with PKCE로 15분당 15회 요청<br /><br />App 전용으로 15분당 15회 요청 |

<div id="differences">
  #### 차이점
</div>

**엔드포인트 URL**

* 표준 v1.1 엔드포인트:
  * GET https://api.x.com/1.1/lists/show.json
    (지정한 리스트 조회)
  * GET https://api.x.com/1.1/lists/ownerships.json
    (지정한 사용자가 소유한 리스트 조회)
* X API v2 엔드포인트:
  * GET https://api.x.com/2/lists/:id
    (지정한 리스트 조회)

  * GET https://api.x.com/2/users/:id/owned&#95;lists
    (지정한 사용자가 소유한 리스트 조회)

**앱 및 프로젝트 요구 사항**

X API v2 엔드포인트를 사용하려면 요청을 인증할 때 [Project](/ko/resources/fundamentals/projects)에 연결된 [developer App](/ko/resources/fundamentals/developer-apps)의 인증 정보를 사용해야 합니다. 모든 X API v1.1 엔드포인트는 독립 실행형 앱 또는 프로젝트에 연결된 앱의 인증 정보를 사용할 수 있습니다.

**요청당 데이터 객체 제한**

표준 v1.1 /lists/ownerships 엔드포인트는 요청당 최대 1000개의 리스트를 반환할 수 있습니다. 새로운 v2 엔드포인트는 요청당 최대 100개의 리스트를 반환할 수 있습니다. 기본적으로 100개의 사용자 객체가 반환되며, 결과 개수를 변경하려면 1~100 사이의 숫자와 함께 max&#95;results= 쿼리 매개변수를 전달해야 합니다. 그런 다음 응답 페이로드에 반환되는 next&#95;token 값을 다음 요청에서 pagination&#95;token 쿼리 매개변수에 전달할 수 있습니다.

**응답 데이터 형식**

표준 v1.1과 X API v2 엔드포인트 버전 간의 가장 큰 차이점 중 하나는 페이로드에 어떤 필드를 반환할지 선택하는 방식입니다.

표준 엔드포인트의 경우 응답 필드 다수가 기본적으로 반환되며, 그 후 추가로 어떤 필드 또는 필드 세트를 페이로드에 반환할지 매개변수를 사용해 선택할 수 있습니다.

X API v2 버전은 기본적으로 리스트 id 및 name 필드만 제공합니다. 추가 필드나 객체를 요청하려면 [fields](/ko/x-api/fundamentals/fields) 및 [expansions](/ko/x-api/fundamentals/expansions) 매개변수를 사용해야 합니다. 이 엔드포인트에서 요청한 모든 리스트 필드는 기본 리스트 객체에 포함되어 반환됩니다. 확장된 게시물 또는 사용자 객체 및 필드는 응답 내의 includes 객체에 포함되어 반환됩니다. 그런 다음 사용자와 확장된 게시물 객체에 있는 ID를 매칭해, 확장된 객체를 리스트 객체와 다시 연결할 수 있습니다. 

아래는 가능한 리스트 필드 및 expansions 예시입니다:

* created&#95;at

* follower&#95;count

* member&#95;count

* owner&#95;id

* description

* private

|     |     |
| :--- | :--- |
| **엔드포인트** | **Expansion** |
| /2/lists/:id | owner&#95;id |
| /2/users/:id/owned&#95;lists | owner&#95;id |

이 새로운 매개변수에 대해서는 각 가이드와, [fields 및 expansions 사용 방법](/ko/x-api/fundamentals/data-dictionary#how-to-use-fields-and-expansions)에 대한 가이드를 추가로 읽어보시기 바랍니다. 

또한 표준 v1.1 필드를 최신 v2 필드에 매핑하는 데 도움이 되는 [데이터 형식 마이그레이션 가이드](/ko/x-api/migrate/data-format-migration)도 제공하고 있습니다. 이 가이드는 특정 필드를 반환하기 위해 v2 요청에 함께 전달해야 하는 expansions 및 fields 매개변수를 구체적으로 안내합니다. 

특정 필드를 요청하는 방식의 변경뿐 아니라, X API v2는 API가 반환하는 객체(예: [Post](/ko/x-api/fundamentals/data-dictionary#tweet) 및 [user](/ko/x-api/fundamentals/data-dictionary#user) 객체)에 대해 새로운 JSON 설계를 도입하고 있습니다.

* JSON 루트 수준에서, 표준 엔드포인트는 statuses 배열에 게시물 객체를 반환하는 반면, X API v2는 data 배열을 반환합니다. 

* 리트윗 및 인용 &quot;statuses&quot;를 참조하는 대신, X API v2 JSON은 리트윗 및 인용 트윗을 참조합니다. contributors 및 user.translator&#95;type과 같은 많은 레거시 및 사용 중단 예정 필드는 제거되고 있습니다. 

* Post 객체의 favorites 및 user 객체의 favourites 두 필드를 사용하는 대신, X API v2는 like라는 용어를 사용합니다. 

* X는 값이 없는 JSON 값(예: null)을 페이로드에 기록하지 않는 관례를 채택하고 있습니다. 게시물 및 사용자 속성은 null이 아닌 값을 가진 경우에만 포함됩니다.

**요청 매개변수**

다음 표준 v1.1 요청 매개변수는 X API v2에서 다음과 같이 대응됩니다:

**ID로 리스트 조회**

|     |     |
| :--- | :--- |
| **표준 v1.1** | **X API v2** |
| list&#95;id | id  |
| slug | 해당 없음 |
| owner&#95;screen&#95;name | 해당 없음 |
| owner&#95;id | expansions/fields 매개변수로 요청 |

**사용자 소유 리스트 조회**

|     |     |
| :--- | :--- |
| **Standard v1.1** | **X API v2** |
| user&#95;id | id  |
| screen&#95;name | 해당 없음 |
| count | max&#95;results |
| cursor | pagination&#95;token |
