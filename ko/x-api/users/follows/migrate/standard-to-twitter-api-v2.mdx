<div id="manage-follows-standard-v11-compared-to-x-api-v2">
  #### 팔로우 관리: 스탠다드 v1.1과 X API v2 비교
</div>

스탠다드 v1.1 [POST friendships/create](https://developer.x.com/en/docs/twitter-api/v1/accounts-and-users/follow-search-get-users/api-reference/post-friendships-create) 및 [POST friendships/destroy](https://developer.x.com/en/docs/twitter-api/v1/accounts-and-users/follow-search-get-users/api-reference/post-friendships-destroy) 엔드포인트를 사용해 왔다면, 이 가이드는 스탠다드와 X API v2 팔로우 관리 엔드포인트 간의 공통점과 차이점을 이해하는 데 도움을 드리기 위한 것입니다.

* **공통점**
  * OAuth 1.0a 사용자 컨텍스트
* **차이점**
  * 엔드포인트 URL
  * 앱 및 프로젝트 요구 사항
  * HTTP 메서드
  * 요청 파라미터

     

<div id="similarities">
  #### 유사점
</div>

**OAuth 1.0a 사용자 컨텍스트 인증 방식**

두 엔드포인트 버전 모두 [OAuth 1.0a 사용자 컨텍스트](/ko/resources/fundamentals/authentication#oauth-1-0a-2)를 지원합니다. 따라서 이전에 표준 v1.1 manage follows 엔드포인트 중 하나를 사용해 오셨다면, X API v2 버전으로 마이그레이션하더라도 동일한 인증 방식을 계속 사용할 수 있습니다. 

<div id="differences">
  #### 차이점
</div>

**엔드포인트 URL**

* 표준 v1.1 엔드포인트:
  * POST https://api.x.com/1.1/friendships/create.json
    (사용자 팔로우)
  * POST https://api.x.com/1.1/friendships/destroy.json
    (사용자 팔로우 해제)
* X API v2 엔드포인트:
  * POST https://api.x.com/2/users/:id/following
    (사용자 팔로우)
  * DELETE https://api.x.com/2/users/:source&#95;user&#95;id/following/:target&#95;user&#95;id
    (사용자 팔로우 해제) 

**앱 및 프로젝트 요구 사항**

X API v2 엔드포인트를 호출할 때는 요청을 인증할 때 [developer App](/ko/resources/fundamentals/developer-apps) 및 [Project](/ko/resources/fundamentals/projects)에 연결된 자격 증명을 사용해야 합니다. 모든 X API v1.1 엔드포인트는 프로젝트에 연결되지 않은 독립형 앱이나 프로젝트에 연결된 앱의 자격 증명을 모두 사용할 수 있습니다.

**요청 파라미터**

다음 표준 v1.1 요청 파라미터는 X API v2에서 다음과 같이 대응됩니다.

| 표준 v1.1 | X API v2 |
| :--- | :--- |
| 해당 없음 | id (POST), source&#95;user&#95;id (DELETE) |
| user&#95;id | target&#95;user&#95;id |
| screen&#95;name | 해당 없음 |

표준 v1.1 파라미터는 쿼리 파라미터로 전달되고, X API v2 파라미터는 본문(body) 파라미터(POST 엔드포인트의 경우) 또는 경로(path) 파라미터(DELETE 엔드포인트의 경우)로 전달된다는 점에 유의하세요.

또한 v2의 id 및 source&#95;user&#95;id는 표준 v1.1 엔드포인트를 사용할 때는 필수 항목이 아니었습니다. 이는 OAuth 1.0a 사용자 컨텍스트와 함께 전달된 액세스 토큰을 통해 어떤 사용자가 팔로우/언팔로우를 시작했는지 추론했기 때문입니다.
