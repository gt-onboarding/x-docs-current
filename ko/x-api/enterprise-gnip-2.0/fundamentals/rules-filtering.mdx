<div id="getting-started-with-enterprise-rules-and-queries">
  ## Enterprise 규칙 및 쿼리 시작하기
</div>

Enterprise 연산자를 사용하는 제품은 여러분이 설정한 필터링 규칙에 따라 소셜 데이터를 제공합니다. 규칙은 하나 이상의 ‘절(clause)’로 구성되며, 각 절은 키워드, 정확한 구(phrase), 또는 여러 Enterprise 연산자 중 하나입니다. Enterprise 연산자로 규칙을 만들기 전에, 아래에 설명된 구문을 검토하고, 사용 가능한 연산자 목록을 살펴보며, 규칙 작성과 관련된 제한 사항을 이해해야 합니다. 또한 &quot;[연산 순서](#orderofoperations)&quot; 섹션에서 규칙이 논리적으로 어떻게 평가되는지에 대한 세부적인 동작 방식을 반드시 이해해야 합니다.

여러 절은 &quot;and&quot; 및 &quot;or&quot; 논리를 모두 사용하여 결합할 수 있습니다.

**참고:** &quot;and&quot; 논리는 절 사이에 공백을 넣어 지정하며, &quot;or&quot; 논리는 대문자 `OR`로 지정합니다. 

각 규칙은 최대 2,048자까지 작성할 수 있으며, 양의 절(매칭하거나 필터링하려는 항목)과 음의 절(제외하거나 매칭하지 않으려는 항목)의 개수에는 제한이 없습니다.
 

<div id="building-rules-and-queries">
  ### 규칙 및 쿼리 구축
</div>

**키워드 일치**

키워드 일치는 검색 인터페이스의 쿼리와 비슷합니다. 예를 들어, 다음 Enterprise 연산자 규칙은 텍스트 본문에 &quot;social&quot;이라는 용어가 포함된 활동과 일치합니다.

`social`

**공백으로 용어 AND 결합하기**

다른 키워드를 추가하는 것은 일치 조건을 하나 더 추가하는 것과 같습니다. 예를 들어, 이 규칙은 텍스트에 &quot;social&quot;과 &quot;media&quot;가 둘 다 존재할 때만, 순서와 관계없이 활동과 일치합니다. *용어 사이에 공백을 두는 것은 불리언 AND 로직으로 동작합니다.* 규칙에 명시적인 AND를 포함하면 해당 규칙은 rules 엔드포인트에 의해 거부됩니다.

`social media`

**대문자 OR로 용어 OR 결합하기**

하지만 많은 상황에서는 실제로 불리언 OR 로직이 필요합니다. 이것도 쉽게 구현할 수 있습니다. *OR 연산자는 반드시 대문자여야 하며, 소문자 ‘or’은 일반 키워드로 처리됩니다.*

`social OR data`

**용어 부정하기**

또 다른 시나리오에서는 특정 키워드가 포함된 결과를 제외해야 할 수도 있습니다(불리언 NOT 로직). 예를 들어, 텍스트에 ‘happy’는 있지만 ‘birthday’가 포함된 것은 제외하는 활동을 찾는 경우입니다.

`social -personality`

**괄호로 그룹화하기**

이러한 유형의 로직은 괄호를 사용한 그룹화를 통해 결합할 수 있고, 훨씬 더 복잡한 쿼리로 확장할 수 있습니다.

`(social OR data) (academic OR research) -personality -information -university`

하지만 이는 시작에 불過합니다. 위 예제들은 키워드에 대해 토큰화된 매칭만 사용하는 반면, Enterprise 제품은 텍스트에 대해 다양한 유형의 매칭을 수행할 수 있는 연산자도 제공합니다.

**정확 일치**

`"social media research"`

**부분 문자열 일치**

`contains:info`

**근접 일치**

`"social media research"~3`

또한, 다른 연산자를 사용하면 텍스트뿐 아니라 소셜 데이터의 고유한 특성을 기준으로 필터링할 수 있습니다. 

**게시물을 작성하는 사용자**

`from:XDeveloeprs`

**미국 콜로라도주 볼더 Pearl St.에서 반경 10마일 내의 지오태그된 트윗**

`point_radius:[-105.27346517 40.01924738 10.0mi]`

**모두 결합하기**

이 연산자들은 위에서 설명한 것과 동일한 유형의 로직을 사용하여 텍스트 필터와 결합할 수 있습니다.

`(social OR data) (academic OR research OR "social media research") point_radius:[-105.27346517 40.01924738 10.0mi] lang:en -personality -information -university`

  ### Boolean Syntax
</div>

|     |     |     |
| :--- | :--- | :--- |
| **Logic type** | **Operator syntax** | Description |
| AND | social data | 두 연산자 사이에 공백을 두면 그 사이에는 AND 로직이 적용됩니다.  <br />  <br />두 키워드(&quot;social&quot;, &quot;data&quot;)를 모두 포함하는 activity와 일치합니다.  <br />  <br />**규칙에서 AND를 명시적으로 사용하지 마세요. 공백만 사용해야 합니다. 명시적인 AND는 일반 키워드처럼 처리됩니다.** |
| OR  | social OR data | 두 연산자를 OR로 결합하려면, 두 연산자 사이에 앞뒤로 공백을 둔 대문자 OR을 삽입합니다.  <br />  <br />두 키워드(&quot;social&quot; 또는 &quot;data&quot;) 중 하나라도 포함된 activity와 일치합니다.  <br />  <br />단일 규칙에서 OR과 AND 기능을 함께 사용하는 경우, &quot;[Order of operations](#OrderOfOperations)&quot; 섹션에서 설명한 연산자 우선순위를 이해해야 하며, 규칙이 의도한 대로 동작하도록 아래에 설명된 대로 괄호를 사용해 부정되지 않은 연산자들을 함께 그룹화하는 것을 고려해야 합니다.  <br />  <br />규칙에서는 반드시 대문자 &quot;OR&quot;을 사용해야 합니다. 소문자 &#39;or&#39;는 일반 키워드로 처리됩니다. |

**주의:** 연산자는 긍정 또는 부정일 수 있습니다.

**Positive Operators**는 결과에 **포함**하고자 하는 항목을 정의합니다. 예를 들어 `has:hashtags` 연산자는 “해시태그를 포함하는 activity가 필요하다”는 의미입니다.

**Negative Operators**는 결과에서 **제외**하고자 하는 항목을 정의하며, 위에서 설명한 Boolean NOT 로직을 사용해 생성됩니다. 예를 들어 `-has:hashtags`는 “내 규칙과 일치하더라도, 해시태그를 포함하는 activity는 모두 제외하라”는 의미입니다.

Premium operator 제품은 최대 길이 2,048자 이내라는 제한만 있을 뿐, 긍정 및 부정 절의 개수에는 제한이 없습니다.
 

<div id="order-of-operations">
  #### 연산 순서
</div>

하나의 규칙에서 AND와 OR 기능을 함께 사용할 때, 아래 연산 순서에 따라 해당 규칙이 평가됩니다.

1. AND 논리로 연결된 연산자가 먼저 결합됩니다.
2. 그다음 OR 논리로 연결된 연산자가 적용됩니다.

**예시:**

* `apple OR iphone ipad` 는 `apple OR (iphone ipad)` 로 평가됩니다.
* `ipad iphone OR android` 는 `(iphone ipad) OR android` 로 평가됩니다.

불확실성을 줄이고 규칙이 의도한 대로 평가되도록 하려면, 필요한 경우 괄호를 사용해 용어를 함께 묶으세요. 예를 들어:

* `(apple OR iphone) ipad`
* `iphone (ipad OR android)`

<div id="punctuation-diacritics-and-case-sensitivity">
  #### 문장 부호, 발음 구별 기호, 대소문자 민감도
</div>

Enterprise operator에 대해 발음 구별 기호나 악센트가 있는 문자로 키워드 또는 해시태그 규칙을 지정하면, 해당 발음 구별 기호를 그대로 반영하여 게시물 텍스트(해시태그 또는 키워드)와 매칭됩니다. 예를 들어 키워드 `Diacr**í**tica` 또는 해시태그 `#cumplea**ñ**os` 규칙은 &quot;Diacr**í**tica&quot; 또는 &quot;#cumplea**ñ**os&quot;에는 매칭되지만, 물결표가 없는 &quot;Diacritica&quot; 또는 &quot;#cumpleanos&quot;, 즉 **í** 또는 **에녜(eñe)** 가 없는 형태에는 매칭되지 않습니다.

악센트나 발음 구별 기호가 있는 문자는 일반 문자와 동일하게 처리되며, 단어 경계로 취급되지 않습니다. 예를 들어 규칙이 cumplea**ñ**os인 경우, cumpleaños라는 단어를 포함하는 활동에만 매칭되며, cumplea, cumplean 또는 os를 포함하는 활동에는 매칭되지 않습니다.

모든 operator는 대소문자를 구분하지 않고(case-insensitive) 평가됩니다. 예를 들어 규칙 `Cat`은 &quot;cat&quot;, &quot;CAT&quot;, &quot;Cat&quot; 모두에 매칭됩니다.

<div id="powertrack-rule-tags">
  #### PowerTrack 규칙 태그
</div>

&quot;[Matching rules](/ko/x-api/enterprise-gnip-2.0/fundamentals/data-enrichments#matching-rules)&quot; 페이지에서 설명했듯이, 각 규칙은 태그와 함께 생성할 수 있습니다. 이러한 태그는 필터링에는 아무런 영향을 미치지 않지만, 앱 내에서 규칙을 논리적으로 그룹화하는 데 사용할 수 있습니다. 각 규칙에는 하나의 태그만 지정할 수 있으며, 태그의 최대 길이는 255자입니다. 태그는 &quot;Matching rules&quot; 페이지에 설명된 대로 API를 통해 규칙을 생성할 때 JSON 형식의 규칙에 함께 포함됩니다.

<div id="putting-rules-in-json-format">
  #### 규칙을 JSON 형식으로 작성하기
</div>

API를 통해 스트림에 규칙을 추가하거나 삭제하려면 규칙을 JSON 형식으로 작성해야 합니다. 즉, 각 규칙을 다음 구조에 넣어야 합니다:

`{"value":"insert_rule_here"}`

**큰따옴표가 포함된 규칙**

규칙에 정확 일치 또는 기타 연산자에 사용되는 큰따옴표 문자 (`“`)가 포함되어 있는 경우, JSON 형식의 구조와 구분할 수 있도록 백슬래시를 사용해 이스케이프해야 합니다.

`"social data" @XDevelopers`

JSON 형식의 규칙은 다음과 같습니다:

`{"value":"\"social data\" @XDevelopers"}`

**큰따옴표 문자열 리터럴이 포함된 규칙**

정확 일치 검색 내에서 큰따옴표 문자를 문자열 리터럴로 포함하려면, 두 번 이스케이프해야 합니다. 예를 들어, &quot;Toys “R” Us&quot;라는 정확한 문구(“R” 주변의 큰따옴표 포함)를 매칭하는 규칙의 일반 텍스트 표현은 다음과 같습니다:

`"Toys \"R\" Us"`

이를 JSON 형식으로 변환할 때는 다음 구조를 사용해야 합니다:

`{"value":"\"Toys \\\"R\\\" Us\""}`

**태그가 포함된 규칙**

위에서 설명한 것처럼 규칙에 선택적 태그를 포함하려면, 규칙 값과 함께 추가 `tag` 필드를 포함하면 됩니다.

`{"value":"\"social data\" @XDevelopers","tag":"RULE-TAG-01"}`

**API 요청을 위한 포맷**

API를 통해 스트림에서 규칙을 추가하거나 삭제할 때, 여러 개의 JSON 형식 규칙은 콤마로 구분하고, 아래와 같이 JSON `rules` 배열로 감싸야 합니다:

`{"rules":[{"value":"from:XDevelopers"},{"value":"\social data\" @XDevelopers","tag":"RULE-TAG-01"}]}`

<div id="operators-that-match-quote-tweets">
  #### 인용 트윗과 일치하는 연산자
</div>

[PowerTrack API](/ko/x-api/enterprise-gnip-2.0/powertrack-api) 및 Historical PowerTrack API를 사용할 때 아래 연산자는 인용된 원본 게시물과 새 인용 트윗 둘 모두의 콘텐츠와 일치합니다.

그러나 [Search API](/ko/x-api/enterprise-gnip-2.0/fundamentals/search-api)를 사용하는 경우 이러한 연산자는 인용 트윗의 콘텐츠에만 일치하며, 인용된 원본 게시물의 어떤 콘텐츠와도 일치하지 않습니다.

* `Keywords`
* `Phrases`
* `Proximity`
* `#hashtags`
* `@mentions`
* `$cashtags`
* `url:`
* `url_contains:`
* `has:links`
* `has:mentions`
* `has:hashtags`
* `has:media`
* `has:symbols`
* `is:quote`
* `is:reply`
