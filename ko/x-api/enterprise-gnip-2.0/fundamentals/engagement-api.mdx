<div id="overview">
  ## 개요
</div>

[`Enterprise`](https://developer.x.com/en/products/x-api/enterprise)

*이 API는 관리형 액세스 레벨에서만 사용할 수 있는 Enterprise API입니다. 이 API를 사용하려면 먼저 Enterprise 영업팀을 통해 계정을 개설해야 합니다. [자세히 알아보기](/ko/x-api/enterprise-gnip-2.0/enterprise-gnip)*

Engagement API는 게시물 노출 수와 참여 메트릭에 대한 액세스를 제공합니다. 대부분의 메트릭과 엔드포인트는 [OAuth 1.0a 사용자 컨텍스트](/ko/resources/fundamentals/authentication)를 사용해 인증해야 하지만, 공개 Favorite, Retweet, Reply, Video Views 메트릭은 [OAuth 2.0 Bearer Token](/ko/resources/fundamentals/authentication#oauth-2-0)과 /totals 엔드포인트를 사용해 액세스할 수 있습니다.  

**참고:** 일부 X 웹 대시보드에 보고되는 데이터와 Engagement API에 보고되는 데이터 사이에 차이가 있을 수 있습니다. 이러한 차이는 일반적으로 웹 대시보드가 선택한 기간 내에 발생한 참여 및/또는 노출만 표시하기 때문에 발생합니다. 예를 들어, 웹 대시보드는 한 달의 기간 동안 발생한 게시물의 참여를 표시할 수 있지만, Engagement API는 요청한 시간 범위 내에 있는 경우 해당 월의 범위를 넘어서는 참여도 표시할 수 있습니다. 이러한 경우에는 Engagement API를 신뢰할 수 있는 데이터 원본으로 간주해야 합니다.
 

<div id="request-endpoints">
  ### 요청 엔드포인트
</div>

Engagement API에는 세 개의 엔드포인트가 있습니다.

<div id="current-totals-totals">
  #### Current Totals: [/totals]
</div>

* 요청 시 원하는 게시물에 대해 Impressions 총합 메트릭과 Engagements 총합 메트릭을 반환합니다
* 다음 메트릭으로 제한됩니다: Impressions, Engagements, Favorites, Replies, Retweets, Quote Tweets, Video Views
* OAuth 1.0a 사용자 컨텍스트를 사용하여 **최근 90일 이내**에 생성된 게시물에 대한 **Impressions** 및 **Engagements** 메트릭 조회를 지원합니다
* OAuth 2.0 Bearer Token을 사용하여 **모든 게시물**에 대한 **Favorites, Retweets, Quote Tweets, Replies** 및 **Video Views** 메트릭 조회를 지원합니다
* 결과는 요청 시점의 최신 Impressions 및 Engagements 총합을 기준으로 합니다
* 대시보드 보고서를 구성하고 다양한 @handle 전반의 참여율을 계산하는 데 이상적입니다
* 요청당 최대 250개의 게시물에 대한 메트릭 요청을 지원합니다
   

<div id="last-28-hours-28hr">
  #### 지난 28시간: [/28hr]
</div>

* 요청은 노출 수에 대한 총 메트릭, 참여 수에 대한 총 메트릭, 그리고 지난 28시간 동안 발생한 개별 참여 메트릭의 세부 내역을 반환할 수 있습니다
* 데이터는 게시물 ID별 또는 집계된 시계열(일별, 시간별)로 그룹화할 수 있습니다
* 최근에 생성된 콘텐츠의 성과를 추적하는 데 적합합니다
* 사용 가능한 모든 메트릭을 지원합니다
* 요청 1회당 최대 25개 게시물에 대한 메트릭 요청을 지원합니다
   

<div id="historical-historical">
  #### Historical: [/historical]
</div>

* 요청은 최근 1년간의 노출 수, 참여 수, 그리고 개별 참여 메트릭의 세부 내역을 Post 생성 시간이 아닌 참여 발생 시간을 기준으로 반환할 수 있습니다.
* 요청은 시작 날짜와 종료 날짜 파라미터를 지원하여, 최대 4주 길이의 특정 기간으로 범위를 좁힐 수 있는 유연성을 제공합니다.
* 게시물 참여 데이터는 과거 365일까지만 제공됩니다.
* 데이터는 게시물 ID별로, 그리고 시계열 집계(일 단위 또는 시간 단위)로 그룹화할 수 있습니다.
* 최근 성과를 과거 기준과 비교 평가하거나 특정 @handle의 성과에 대한 역사적 추이를 파악하는 데 적합합니다.
* 사용 가능한 모든 메트릭을 지원합니다.
* 요청당 최대 25개의 게시물에 대한 메트릭 요청을 지원합니다.

<div id="engagement-groupings">
  ### 참여 그룹화
</div>

그룹화를 통해 반환된 참여 메트릭을 원하는 방식으로 구성할 수 있습니다. 요청 한 번당 최대 3개의 그룹화를 포함할 수 있습니다. 메트릭을 다음 값 중 하나 이상으로 그룹화하도록 선택할 수 있습니다:

*모든 세 엔드포인트에서 지원:*

* tweet.id
* engagement.type
   

*`/28hr` 및 `/historical` 엔드포인트는 시계열 메트릭을 제공할 수 있으므로, 다음도 지원합니다:*

* engagement.day
* engagement.hour

그룹화에 대해 더 자세히 알아보려면 Guides 섹션에 있는 [Engagement API Grouping](/ko/x-api/enterprise-gnip-2.0/fundamentals/engagement-api#engagement-api-groupings) 페이지를 방문하세요.

<div id="introduction">
  #### 소개
</div>

이 문서의 목적은 개발자를 위한 Engagement API 통합 입문서를 제공하는 것입니다. 먼저 왜 통합해야 하는지에 대해 살펴본 뒤, 기술적으로 어떻게 통합하는지에 대한 세부 내용을 다루겠습니다.

<div id="what-does-the-engagement-api-provide">
  ##### **Engagement API는 무엇을 제공하나요?**
</div>

* Engagement API는, 해당 계정이 [3-legged OAuth](/ko/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow)를 사용해 귀하의 앱이 자신들을 대신해 메트릭을 요청하도록 승인한 경우, 지난 90일 동안 특정 X 계정이 소유한 게시물에 대한 노출 수(impression) 및 참여(engagement) 데이터를 제공합니다. 이 강력하면서도 구현이 쉬운 솔루션을 통해 노출 수뿐 아니라 URL 클릭, #해시태그 클릭 등 다양한 유형의 심층 참여 데이터에 즉시 액세스할 수 있습니다.
* Engagement API는 어떤 게시물이든 즐겨찾기(좋아요), 리트윗, 인용 트윗, 답글, 동영상 조회수에 대한 전체 집계 메트릭을 제공합니다. 이는 개별 게시물 또는 게시물 집합에 대한 기본적인 참여 데이터를 얻는 강력한 방법으로 활용할 수 있습니다.
* Engagement API는 15개 이상의 성과 메트릭을 사용해 콘텐츠 성과를 효과적으로 측정함으로써, 소셜 리스닝, 마케팅, 퍼블리싱 플랫폼에 새로운 가치를 제공합니다. 이를 통해 고객은 X 상의 ROI를 측정할 수 있습니다.
* Engagement API는 요청/응답 방식의 API로, 앱 개발자가 게시물 ID, 필요한 메트릭, 기간을 포함해 요청을 보내면, API가 해당 데이터를 즉시 반환합니다.
   

<div id="why-integrate-example-use-cases">
  ##### **왜 통합해야 하나요? 사용 예시**
</div>

* 내 콘텐츠의 전체 도달 범위를 파악해 얼마나 많은 사람이 조회하는지 이해할 수 있습니다. 얼마나 많은 사람이 동영상을 시청하고, 링크를 클릭하고, 해시태그를 클릭하며, 내 앱을 설치하는지 확인합니다.
* 총합 및 시계열 참여 메트릭을 모두 생성합니다.
* 모든 공개 게시물에 대한 기본 참여 메트릭(즐겨찾기, 리트윗, 인용 트윗, 답글)을 파악합니다.
* 이러한 메트릭을 사용해 어떤 유형의 게시물이 효과적인지 알아내고, 그런 게시물을 더 자주 게시하여 내 콘텐츠의 노출 수와 참여를 더 많이 확보합니다.
* 내 게시물 중 하나가 좋아요 100개(또는 다른 임계값)에 도달할 때마다, 예를 들어 내가 소유한 다른 계정의 콘텐츠를 리트윗하는 등의 마케팅 작업을 자동화합니다.
* A/B 테스트 도구로 활용하기 위해 캠페인 간 성과를 벤치마킹하고 비교합니다.
* 고객 서비스 부서에서 어떤 유형의 콘텐츠에 고객이 공감하는지 분석해, 언제 어떻게 응답할지 결정합니다.
* 내 플랫폼에서 게시된 콘텐츠에 대한 분석 정보를 표시합니다.
   

[Engagement API는 2016년에 출시되었으며](https://blog.x.com/official/en_us/a/2016/gnip-s-engagement-api-is-now-generally-available.html), 이러한 심층적인 참여 메트릭을 대규모로 제공한 최초의 X API였습니다. Engagement API는 사용하기 쉬우며, 고객이 이 프로세스를 자동화할 수 있도록 해 줍니다. 다음은 예시 통합을 설명하는 사례 연구입니다.

* [Measuring campaign success with the Red Cross](https://blog.x.com/developer/en_us/topics/spotlight/2016/measuring-campaign-success-with-the-red-cross.html)[](https://simplymeasured.com/blog/true-twitter-impressions-and-url-clicks-new-from-simply-measured/#sm.0007werel134td8zqf02m2mduumr6)

이제 Engagement API를 사용하는 이유를 살펴보았으니, 기술적인 세부 정보를 알아보겠습니다.

<div id="integrating-the-engagement-api">
  #### Engagement API 통합하기
</div>

<div id="introduction-to-api">
  ##### **API 소개**
</div>

Engagement API는 JSON으로 인코딩된 요청을 받고 JSON으로 인코딩된 참여 메트릭으로 응답하는 간단한 RESTful API입니다. 요청은 세 가지 주요 부분으로 구성됩니다(자세한 내용은 링크를 참고하세요).

* ***Post IDs*** 배열
* 관심 있는 [메트릭 유형](https://developer.x.com/en/docs/x-api/v1/metrics/get-tweet-engagement/overview#EngagementTypes)을 지정하는 배열. 유형에는 ‘impressions’, ‘retweets’, ‘hashtag&#95;clicks’, ‘user&#95;follows’ 등이 포함됩니다.
* API 응답에서 참여 데이터를 어떻게 구성할지 나타내는 JSON 구조인 [Engagement groupings](/ko/x-api/enterprise-gnip-2.0/fundamentals/engagement-api#engagement-api-groupings)

대부분의 경우 Engagement Types와 Groupings은 요청 간에 비교적 일정하게 유지되고, Post IDs만 변경됩니다.

Engagement API는 [세 가지 엔드포인트](https://developer.x.com/en/docs/x-api/v1/metrics/get-tweet-engagement/overview#RequestEndpoints)를 제공합니다.

* **Totals** - 게시물에 대한 전체 참여 합계를 제공합니다. 일부 메트릭은 모든 게시물에 대해 사용할 수 있지만, 일부는 최근 90일에 대해서만 사용할 수 있습니다.
* **28 hour** - 최근 28시간 동안의 시계열 참여 메트릭을 제공합니다.
* **Historical** - 2014년 9월 1일 이후 게시된 게시물에 대해, 최대 4주 연속 기간의 시계열 참여 메트릭을 제공합니다.

**/totals** 엔드포인트는 한 번의 요청으로 **최대 250개의 게시물에 대한 메트릭**을 요청할 수 있습니다. **/28hour** 및 **/historical** 엔드포인트는 **요청당 25개**를 지원합니다.

Engagement API 접근 권한을 확보하는 방법을 살펴본 다음, API 요청을 만드는 방법을 단계별로 안내하고, OAuth 개요를 제공하며, 기타 기술 리소스에 대한 링크를 제공합니다.

<div id="getting-api-access">
  ##### **API 액세스 권한 얻기**
</div>

이 문서를 읽고 있다면, 대개는 이미 Engagement API 액세스 권한을 보유하고 있을 것입니다. 그렇지 않은 경우, Enterprise 계정 관리자에게 문의하거나 [여기](/ko/x-api/enterprise-gnip-2.0/enterprise-gnip)에서 Enterprise 액세스 권한을 신청하세요.

첫 번째 단계는 승인된 개발자 계정을 사용하여 [developer portal](/ko/resources/fundamentals/developer-portal)을 통해 [X 앱](/ko/resources/fundamentals/developer-apps)을 생성하는 것입니다. 계정 관리자는 액세스를 제공하기 위해 이 애플리케이션과 연결된 숫자로 된 App ID가 필요합니다. 개발자 계정을 신청해야 하는 경우 [여기](/ko/resources/fundamentals/developer-apps)에서 신청할 수 있습니다.

첫 번째 단계는 JSON 형식으로 API 요청을 구성하는 것입니다. 여기에는 배열에 담긴 두 개의 게시물 ID, 관심 있는 참여 유형을 담은 배열, 그리고 응답에서 메트릭을 어떻게 배열할지 지정하는, 사용자 정의 이름을 가진 &quot;groupings&quot; JSON 객체가 포함됩니다. 요청은 다음과 같습니다:

```json
{
  "tweet_ids": [
    1260294888811347969, 850006245121695744
  ],
  "engagement_types": [
    "retweets", "quote_tweets", "favorites", "replies"
  ],
  "groupings": {
    "engagement-types-by-id": {
      "group_by": [
        "Tweet.id", "engagement.type"
      ]
    }
  }
}
```

이 총 메트릭을 조회하기 위해 이 JSON 요청을 [https://data-api.x.com/insights/engagement/totals](https://data-api.x.com/insights/engagement/totals) 엔드포인트에 POST합니다.

요청이 JSON 형식으로 인코딩되어 있고 Gzip으로 압축되어 있음을 나타내기 위해 다음 헤더를 포함합니다(요청 본문 크기가 커질 수 있습니다):

* Content-Type: application/json
* Accept-Encoding: gzip
   

요청을 보낼 때는 OAuth를 사용해 인증하며, 이에 대해서는 다음 섹션에서 더 자세히 설명합니다.

API는 다음과 같은 페이로드를 반환합니다:

```json
{
  "grouping name": {
    "1260294888811347969": {
      "favorites": "17111",
      "quote_tweets": "3254",
      "replies": "1828",
      "retweets": "5218"
    },
    "850006245121695744": {
      "favorites": "492",
      "quote_tweets": "66",
      "replies": "42",
      "retweets": "324"
    }
  }
}
```

응답에는 우리가 요청한 메트릭이 “groupings” 정의에서 설명한 구조로 포함되어 있으며, 첫 번째 수준에서는 게시물 ID별로, 그다음 수준에서는 참여 유형별로 메트릭이 나열된다는 점에 유의하세요.

꽤 간단했죠. OAuth 인증이 처음이라면 다음 섹션을 살펴보세요.

<div id="authenticating-with-oauth">
  #### OAuth로 인증하기
</div>

OAuth는 기술 업계에서 매우 널리 사용되는 인증 표준입니다. 이미 OAuth(예를 들어 다른 X API와 함께)를 사용 중이라면, 아마도 언어별 OAuth 패키지를 사용해 복잡한 세부 사항을 모두 추상화하고 있을 것입니다. OAuth가 처음이라면, [X API로 OAuth 사용하기](/ko/resources/fundamentals/authentication) 페이지를 방문하거나 바로 [https://oauth.net](https://oauth.net/)으로 이동해 자세한 내용을 확인해 보세요. 그런 다음, 통합에 사용할 프로그래밍 언어에 맞는 OAuth 패키지를 찾아 그 패키지부터 시작할 것을 권장합니다. 이러한 패키지를 사용하면 보통 키와 토큰을 구성하고, HTTP 클라이언트 객체를 생성한 뒤, 그 객체로 요청을 보내는 방식으로 인증 과정을 진행하게 됩니다.

예를 들어, Ruby 생태계에서는 다음 의사 코드가 Ruby gem ‘oauth’를 사용해 OAuth 지원 앱을 만들고 POST 요청을 보내는 방법을 보여주는 예시입니다.

```js
require 'oauth'

#JSON 요청을 조립합니다(위와 같이).
request = make_json_request()

#내 앱 컨슈머 키와 시크릿으로 앱 컨슈머 객체를 빌드합니다.
consumer = OAuth::Consumer.new(keys['consumer_key'], keys['consumer_secret'], {:site => ‘https://data-api.x.com’})
#권한을 제공하는 계정에서 제공한 토큰으로 사용자 토큰을 빌드합니다. 앱 전용 #요청을 수행하는 경우(권한이 필요하지 않은 게시물을 /totals 엔드포인트로 확인), 이 단계는 #건너뛸 수 있습니다.
token = {:oauth_token => keys['access_token'], :oauth_token_secret => keys['access_token_secret']}

#OAuth가 활성화된 앱 객체를 생성합니다.
app = OAuth::AccessToken.from_hash(consumer, token)
#POST 요청을 수행합니다.
result = app.post(“/insights/engagement/totals", request, {"content-type" => "application/json", "Accept-Encoding" => "gzip"})
```

Engagement API는 애플리케이션 전용 인증과 사용자 컨텍스트 인증을 모두 지원합니다. /totals 엔드포인트로 소유하지 않은 공개 게시물의 참여 메트릭을 수집하는 경우에는 사용자 권한이 필요하지 않으며, 애플리케이션 전용 인증을 사용할 수 있습니다. 이 경우에는 앱 키와 시크릿만으로 인증을 수행합니다.

OAuth를 사용하면 앱이 사용자와 연관된 토큰을 사용하여 “다른 사용자를 대신하여(on behalf of another user)” API 요청을 수행할 수도 있습니다. 소유한 게시물(즉, 사용자 토큰을 보유한 사용자가 게시한 게시물)에 대한 참여 메트릭을 생성하는 경우에는 사용자 컨텍스트로 요청을 수행하게 되며, 이는 앱 키와 사용자별 액세스 토큰을 모두 사용해 인증한다는 의미입니다. 이러한 사용자 액세스 토큰은 일반적으로 &#39;[X로 로그인하기](/ko/resources/fundamentals/authentication#log-in-with-x)&#39; 프로세스를 통해 제공되거나 사용자로부터 직접 획득합니다(사용자로부터 토큰을 직접 획득하는 경우 반드시 [twurl](https://github.com/xdevplatform/xurl)을 사용해야 합니다). 사용자가 토큰을 제공하면 해당 토큰은 만료되지 않으며, 사용자가 토큰을 재설정하거나 비밀번호를 변경하지 않는 한 Engagement API에서 사용자를 대신해 요청을 수행하는 데 계속 사용할 수 있습니다. 이러한 경우에는 사용자가 새로운 토큰을 다시 제공해야 합니다.

어떤 메트릭에 어떤 인증이 필요한지는 [이 표](/ko/x-api/enterprise-gnip-2.0/fundamentals/engagement-api#engagement-api-groupings)에서 확인할 수 있습니다.

<div id="selecting-an-engagement-api-endpoint">
  ### Engagement API 엔드포인트 선택
</div>

Engagement API는 서로 다른 세 가지 엔드포인트를 제공합니다:

* **[Totals](/ko/x-api/enterprise-gnip-2.0/fundamentals/engagement-api#post-insights-engagement)** - 소유된(‘owned’) 또는 비소유된(‘unowned’) 게시물에서 선택된 메트릭의 전체 합계를 제공합니다. 일부 메트릭은 모든 게시물에 대해 제공되지만, 다른 메트릭은 최근 90일 내에 게시된 게시물에 대해서만 제공됩니다. 요청당 최대 250개의 게시물을 지원합니다.
* **[28 hour](/ko/x-api/enterprise-gnip-2.0/fundamentals/engagement-api#post-insights-engagement)** - 최근 28시간 동안의 ‘owned’ 게시물에 대한 시계열 Engagement 메트릭을 제공합니다. 요청당 최대 25개의 게시물을 지원합니다.
* **[Historical](/ko/x-api/enterprise-gnip-2.0/fundamentals/engagement-api#post-insights-engagement-historical)** - 2014년 9월 1일 이후에 게시된 ‘owned’ 게시물에 대해 최대 4주 연속 기간의 시계열 Engagement 메트릭을 제공합니다. 요청당 최대 25개의 게시물을 지원합니다.
   

각 엔드포인트는 고유한 특성을 갖고 있습니다. 세 가지를 모두 사용할 계획이든, 아니면 어떤 엔드포인트가 사용자의 사용 사례에 가장 잘 맞는지 결정하려는 경우든, 이들 간의 차이를 이해하는 것이 중요합니다.

아래에서는 몇 가지 핵심 개념을 소개한 뒤, 세 엔드포인트를 더 자세히 살펴보고, 일반적으로 특정 엔드포인트에 매핑되는 예시 사용 사례를 제시합니다. 이 정보가 세 가지를 모두 더 효율적으로 통합하거나, 단일 엔드포인트 중 어떤 것이 사용자의 목적에 가장 잘 부합하는지 결정하는 데 도움이 되기를 바랍니다.
 

<div id="key-concepts">
  #### 핵심 개념
</div>

세 가지 Engagement API 엔드포인트가 제공하는 서로 다른 기능과 데이터에 대한 이해를 돕는 몇 가지 핵심 개념이 있습니다.
 

<div id="impressions-and-engagement-metrics">
  ##### 노출 수 및 참여 메트릭
</div>

노출 수는 특정 게시물이 X 플랫폼에서 자연 노출(organic) 환경에서 조회된 횟수를 나타냅니다. 프로모션(Promoted) 또는 유료(Paid) 맥락에서 노출된 게시물로부터 발생한 노출 수는 포함되지 않습니다. Engagement API 도입 이전에는 게시물 노출 수가 잠재적인 조회 수의 지표만을 의미했습니다. 이는 작성자의 계정과 해당 콘텐츠를 리트윗한 계정의 팔로워 수를 집계하는 방식에 기반했으며, 특정 사용자가 실제로 해당 게시물을 보지 못하는 일반적인 상황은 고려하지 않았습니다.

Engagement API가 생성하는 노출 메트릭은 게시물이 실제로 표시되도록 렌더링된 횟수를 측정한 실제 지표입니다. 귀하의 계정을 팔로우하는 사용자가 게시물을 놓친 경우, 이는 노출로 집계되지 않습니다.

Engagement API는 사용자가 게시물을 접했을 때 취할 수 있는 서로 다른 행동 각각을 나타내는 14가지 고유한 참여 [메트릭 유형](/ko/x-api/enterprise-gnip-2.0/fundamentals/engagement-api#available-metrics)에 대한 메트릭을 제공합니다. 여기에는 리트윗, 좋아요, 답글, #해시태그·링크·미디어와 같은 엔티티 클릭, 작성자 팔로우, 작성자 프로필 보기 등이 포함됩니다. 이러한 개별 행동은 모두 하나의 Engagements 메트릭으로 집계됩니다.

<div id="owned-and-unowned-x-content">
  ##### 소유한 X 콘텐츠와 소유하지 않은 X 콘텐츠
</div>

Engagement API는 소유한 게시물과 소유하지 않은 게시물을 명확하게 구분합니다. 소유한 게시물은 귀하의 계정에서 게시된 게시물이거나, Engagement 데이터 요청에 대한 권한을 획득한 게시물입니다. 다른 X API와 마찬가지로, 다른 X 사용자/계정이 귀하를 대신해 API 요청을 보낼 수 있도록 해주는 액세스 토큰을 공유하도록 함으로써 이 권한을 얻습니다. 이러한 토큰을 얻는 일반적인 방법은 [‘Sign in with X’ 프로세스](/ko/resources/fundamentals/authentication#log-in-with-x)를 사용하는 것입니다.

`/totals` 엔드포인트는 소유한 게시물과 소유하지 않은 게시물 모두에 대한 참여 데이터를 제공합니다. 소유하지 않은 게시물의 경우, 게시물 화면에 공개적으로 표시되는 Favorite, Retweet, Reply와 같은 참여 메트릭을 요청할 수 있습니다. 이들 메트릭과 관련해 Engagement API가 제공하는 가치는, 이러한 메트릭을 대규모로 자동화된 방식으로 조회할 수 있게 해준다는 점입니다. 소유한 게시물의 경우, `/totals` 엔드포인트는 Impression 및 총 Engagement 메트릭도 제공합니다.

`/28hr` 및 `/historical` 엔드포인트는 소유한 게시물에 대한 메트릭만 제공합니다. 이는 이들 엔드포인트에 요청을 보낼 때 사용자 컨텍스트를 함께 전달해야 함을 의미합니다.

<div id="total-and-time-series-engagement-data">
  ##### 총합 및 시계열 참여 데이터
</div>

`/totals` 엔드포인트는 이름에서 알 수 있듯이 참여 유형별 총합만 제공합니다. 이 숫자들은 해당 게시물이 게시된 이후부터 현재까지의 최신 누적 합계를 나타냅니다. 게시물이 방금 게시된 후 그 메트릭을 반복해서 요청하면, 이러한 합계는 일반적으로 요청할 때마다 변경됩니다.

`/28hr` 및 `/historical` 엔드포인트는 총합과 시계열 데이터 모두를 제공할 수 있습니다. 시계열 데이터를 요청할 때는 참여 메트릭을 일별 또는 시간별 데이터로 집계할 수 있습니다.   

`/28hr` 및 `/historical` 엔드포인트로 시계열 데이터를 요청하는 방법은 [engagement groupings](/ko/x-api/enterprise-gnip-2.0/fundamentals/engagement-api#engagement-api-groupings) 관련 문서를 참고하세요.

<div id="endpoints-and-example-use-cases">
  #### 엔드포인트 및 예시 사용 사례
</div>

위에서 논의한 특성과 차이점을 고려했을 때, 각 개별 엔드포인트는 일반적으로 서로 다른 유형의 사용 사례에 매핑됩니다. 특정 사용 사례에 가장 적합한 엔드포인트를 선택할 수 있도록, 아래에는 예시 사용자 요청과 이를 가장 잘 충족하는 엔드포인트를 함께 제시합니다.

<div id="totals">
  ##### **/totals**
</div>

* 일부 지표 유형(노출 수, 참여 수, 즐겨찾기, 리포스트, 인용 리포스트, 답글, 동영상 조회 수)만 필요합니다.
* 내가 소유한 게시물뿐만 아니라, 어떤 게시물이든 기본 수준의 참여 데이터에 액세스할 수 있어야 합니다.
* 경쟁사와 성과를 비교하고 싶습니다.
* 내가 소유하지 않은 게시물을 포함하는 해시태그나 캠페인의 기본 참여 통계를 추적하고 싶습니다.
* 일별이나 시간별로 분리된 데이터는 필요 없고, 요청 시점의 현재 합계만 필요합니다.
* 보고서나 대시보드에 보여줄 단일 지표만 필요하고, 데이터를 저장하고 싶지 않습니다.
* 페이지 로드 시점에 데이터를 표시하고 싶으며, 요청만 보내서 응답만 받으면 됩니다.
* 하루에 수십만 개에서 수백만 개까지의 게시물에 대한 데이터에 액세스할 수 있어야 합니다.
   

<div id="28hr">
  ##### **/28hr**
</div>

* 17가지 메트릭 유형 모두에 액세스해야 합니다.
* 지난 28시간 동안 게시된 최신 게시물에 대한 데이터를 보고 싶습니다.
* 하루에 한 번 실행되는 작업이 있어서 필요한 데이터만, 즉 마지막 하루치 데이터만 가져오면 됩니다.
* 메트릭을 일 단위 또는 시간 단위로 나누어 확인할 수 있어야 합니다.
* 대시보드에서 시간별 활동을 시계열로 나누어 보여주고 싶습니다.
* 하루에 수십만 개의 게시물에 대해 높은 수준의 액세스 권한이 필요합니다.
* 저장 용량이 있어서 하루에 한 번 데이터를 새로 고치고 누적 합계를 유지할 수 있습니다.
   

<div id="historical">
  ##### **/historical**
</div>

* 17가지 모든 메트릭 유형에 액세스해야 합니다.
* 2014년 9월까지 거슬러 올라가는 게시물에 대한 과거 데이터를 가져와야 합니다.
* 캠페인을 비교하는 상세한 과거 분석을 보여주고 싶습니다.
* 메트릭이 일별 또는 시간별로 분리되어 제공되어야 합니다.
* Engagement API에 대한 높은 수준의 액세스는 필요 없고, 하루에 수백 개에서 수천 개 정도의 게시물에 대한 데이터만 가져오면 됩니다.

<div id="engagement-api-key-characteristics">
  ### Engagement API 주요 특징
</div>

* JSON 데이터를 제공하고 JSON 본문을 포함한 POST 요청을 지원하는 **RESTful API**입니다.
* **요청 유형:** 클라이언트 앱은 다음 유형의 요청을 보낼 수 있습니다.
  * **총 참여 수(Total engagements)** -- /totals 엔드포인트로의 HTTP POST 요청
  * **최근 28시간 참여 수(Last 28-hour engagements)** -- /28hr 엔드포인트로의 HTTP POST 요청
  * **과거 참여 수(Historical engagements)** -- /historical 엔드포인트로의 HTTP POST 요청
* **OAuth 인증:**
  * [OAuth 1.0 User Context](/ko/resources/fundamentals/authentication): [3-legged OAuth](https://developer.x.com/resources/fundamentals/authentication/obtaining-user-access-tokens)를 사용해 귀하의 앱을 승인한 사용자가 소유한 게시물에 대해서는, 사용 가능한 모든 메트릭을 조회할 수 있습니다. 요청을 보낼 때는 해당 사용자의 액세스 토큰을 사용해야 합니다.[](/ko/resources/fundamentals/authentication)
  * [OAuth 2.0 Bearer Token](/ko/resources/fundamentals/authentication#oauth-2-0): 특정 메트릭(리트윗, 인용 트윗, 답글, 즐겨찾기, 동영상 조회수)은 모든 공개 게시물에 대해 사용할 수 있습니다. 
* **요청 메타데이터 및 구조**: 요청 데이터는 게시물 ID 배열, 참여 유형 배열, 참여 그룹화 구조로 구성된 JSON 객체입니다.
* **요청당 게시물 수:**
  * /totals 엔드포인트: 게시물 ID 250개
  * /28hr 엔드포인트: 게시물 ID 25개
  * /historical 엔드포인트: 게시물 ID 25개
* **참여 메트릭 제공 가능 여부:**
  * **/totals** -- 게시물이 게시된 시점부터의 메트릭 합계입니다. 노출수(Impressions)와 참여수(Engagements)는 최근 90일 이내에 게시된 게시물에 대해 제공되며, 리트윗, 인용 트윗, 답글, 즐겨찾기, 동영상 조회수는 모든 게시물에 대해 제공됩니다.
  * **/28hr** -- 요청 시점으로부터 최근 28시간입니다.
  * **/historical** -- 2014년 9월 1일부터 시작하는 임의의 28일 기간입니다.
* **메트릭 유형**: 각 요청에는 [Metric Types](http://support.gnip.com/apis/engagement_api/overview.html#EngagementTypes) 배열이 포함됩니다. 이 메트릭의 제공 가능 여부는 엔드포인트에 따라 다르며, /totals 엔드포인트에서 요청하는 경우에는 게시물이 사용자 권한이 필요한지 여부에 따라 달라집니다.
  * /totals 엔드포인트:
    * 모든 게시물: 즐겨찾기, 리트윗, 인용 트윗, 답글, 동영상 조회수
    * OAuth 1.0a 사용자 컨텍스트 필요: 노출수(Impressions), 참여수(Engagements), 즐겨찾기, 답글, 리트윗
  * /28hr 및 /historical 엔드포인트 (게시물 소유자의 액세스 토큰을 사용하는 OAuth 1.0a 사용자 컨텍스트 필요): 노출수(Impressions), 참여수(Engagements), 즐겨찾기, 답글, 리트윗, URL 클릭수, 해시태그 클릭수, 상세 보기 클릭수, 퍼머링크 클릭수, 미디어 클릭수, 앱 설치 시도, 앱 실행(App Opens), 게시물 이메일(Post Emails), 동영상 조회수, 미디어 조회수
* **참여 그룹화(Engagement groupings):** 각 요청에는 [Engagement Groupings](/ko/x-api/enterprise-gnip-2.0/fundamentals/engagement-api#engagement-api-groupings) 배열이 포함됩니다. 이러한 그룹화를 사용하여 반환되는 메트릭의 구성 방식을 사용자 정의할 수 있습니다. 각 요청에는 최대 세 개의 그룹화를 포함할 수 있습니다. 메트릭은 다음 값으로 구성할 수 있습니다.
  * 모든 엔드포인트: 게시물 ID, 참여 유형
  * /28hr 및 /historical 엔드포인트: 다음 추가 그룹화를 지정하면 이 엔드포인트들은 시계열 데이터를 제공합니다: 참여 일(Engagement Day), 참여 시(Engagement Hour)
* **통합 관련 요구 사항(Integration Expectations):** 귀하의 팀은 다음 사항에 대해 책임을 집니다.
  * 요청에 포함된 게시물 ID에 대한 참여 메트릭을 반환하는 Engagement API로 HTTP 요청을 전송할 수 있는 클라이언트 앱을 생성하고 유지 관리합니다.
* **제한 사항**
* 동영상 조회수는 게시 후 1,800일 이하인 게시물에 대해서만 제공됩니다.

<div id="authenticating-with-the-engagement-api">
  ### Engagement API 인증
</div>

> **주의**: 
>
> API를 사용하기 전에 X에서 먼저 개발자 앱에 대해 Engagement API 액세스를 활성화해야 합니다. 이를 위해 인증에 사용할 App ID를 계정 관리자(account manager)나 기술 지원 팀에 반드시 전달해야 합니다.

Engagement API에는 두 가지 인증 방법을 사용할 수 있습니다: [OAuth 1.0a](/ko/resources/fundamentals/authentication#oauth-1-0a-2)와 [OAuth 2.0 Bearer Token](/ko/resources/fundamentals/authentication#bearer-token-also-known-as-app-only). 

**OAuth 2.0 Bearer Token**(“application-only”라고도 함)을 사용하면 공개된 참여 메트릭에 접근할 수 있습니다. 이 인증 방법은 [/totals endpoint](/ko/x-api/enterprise-gnip-2.0/fundamentals/engagement-api#post-insights-engagement-totals)에 요청을 보낼 때, 공개된 모든 게시물에 대해 Favorites(또는 Likes), Retweets, Quote Tweets, Replies, 그리고 동영상 조회수의 총 개수를 가져오는 데 사용할 수 있습니다. 

**OAuth 1.0a**(“user context”라고도 함)는 사용자를 대신하여 요청을 보내고, 해당 사용자에게 속한 비공개 참여 메트릭에 접근할 수 있게 해줍니다. 

이 인증 방법은 다음에 필수입니다:

* [/28hr endpoint](/ko/x-api/enterprise-gnip-2.0/fundamentals/engagement-api#post-insights-engagement-28hr) 및 [/historical endpoint](/ko/x-api/enterprise-gnip-2.0/fundamentals/engagement-api#post-insights-engagement-historical)에 전송되는 모든 요청
* 다음 비공개 메트릭에 접근하는 경우: Impressions, Engagements, Media Views, Media Engagements, URL Clicks, Hashtag Clicks, Detail Expands, Permalink Clicks, App Install Attempts, App Opens, Email Post, User Follows, User Profile Clicks

OAuth 1.0a로 요청을 보낼 때는, 관심 대상인 게시물 또는 보호된 리소스를 소유한 사용자의 액세스 토큰(Access Token 및 Secret)을 포함해야 합니다. 보호된 사용자 데이터를 요청할 때 올바른 사용자 액세스 토큰을 제공하지 않으면, Engagement API는 `403 Forbidden` 오류를 반환합니다.

Engagement API는, 해당 게시물을 소유한 사용자를 대신해 인증하더라도, [protected Posts](https://help.x.com/en/safety-and-security/public-and-protected-tweets)에 대한 참여 데이터를 가져오는 것을 허용하지 않습니다. 이와 같은 요청을 시도하면 `"Tweet ID(s) are unavailable"` 메시지와 함께 `400 Bad Request` 오류가 반환됩니다.

본인의 X 계정(즉, 개발자 앱을 소유한 계정)을 대신해 요청을 보내는 경우, [developer portal](https://developer.x.com/en/portal/projects-and-apps)의 해당 개발자 앱 “Keys and tokens” 탭에서 필요한 액세스 토큰을 직접 생성할 수 있습니다.

다른 사용자를 대신해 요청을 보내는 경우, 필요한 액세스 토큰을 얻으려면 3-legged OAuth 플로우를 사용해야 합니다. 이에 대한 자세한 내용은 다음 문서를 참고하십시오: [OAuth 1.0a: how to obtain a user’s access tokens](/ko/resources/fundamentals/authentication#oauth-1-0a-2).

추가 예제(특히 OAuth 1.0a를 사용해 인증하는 방법 포함)는 [XDevelopers sample Python code for the Engagement API](https://github.com/xdevplatform/enterprise-scripts-python/tree/master/Engagement-API)를 참고하십시오.

<div id="recent-changes-to-the-engagement-api">
  ### Engagement API의 최근 변경 사항
</div>

Engagement API는 X에서의 활동 성과를 모니터링할 수 있도록, 노출 및 참여 메트릭을 제공하는 매우 유용한 도구입니다. X 전반의 메트릭 간 일관성을 높여 당사의 데이터에 기반한 마케팅 의사결정을 지원하기 위한 지속적인 노력의 일환으로, Engagement API에 최근 적용된 변경 사항을 공유합니다.  

최근 우리는 Engagement API를 현대화하여, X 분석 대시보드(analytics.x.com)에서 사용 중인 것과 동일한 메트릭 집계 방법론을 사용하도록 변경했습니다. 새로운 메트릭을 도입하는 과정에서 기존과 호환되지 않는 API 변경을 최소화하기 위해 신중하게 접근했으며, 첫 번째 변경 세트는 2017년 10월 9일에 배포했습니다. 이러한 변경 사항은 여러분 또는 여러분의 고객이 X에서 성과를 모니터링하는 모든 지점 간의 일관성을 개선합니다. 자세한 변경 사항은 아래 내용을 참조하세요:

|     |     |

<div id="interpreting-the-metrics">
  ### 메트릭 해석하기
</div>

**참고:** 일부 X 웹 대시보드에 표시된 데이터와 Engagement API에서 보고되는 데이터 사이에 차이가 있을 수 있습니다. 이러한 차이는 웹 대시보드가 일반적으로 선택한 기간 내에 발생한 참여 및/또는 노출만 표시하기 때문에 발생합니다. 예를 들어, 웹 대시보드는 특정 달의 달력 범위 내에서 게시물에 대한 참여만 보여줄 수 있지만, Engagement API는 요청한 기간 내에만 포함된다면 해당 달의 범위를 넘어서는 참여까지도 포함할 수 있습니다. 이러한 경우에는 Engagement API를 신뢰할 수 있는 기준으로 봐야 합니다.
 

<div id="impressions-and-engagement-data">
  #### 노출 수 및 참여 데이터
</div>

Engagement API는 **유기적(organic)** 노출 수와 참여 데이터를 제공합니다.

노출 수(Impressions)는 특정 게시물이 X 플랫폼에서 유기적(organic) 문맥에서 조회된 횟수를 나타냅니다. 프로모션 또는 유료(Paid) 문맥에서 게시물이 노출되어 발생한 노출 수는 **포함되지 않습니다**.

참여(Engagements)는 특정 게시물이 **유기적(organic)** 및 **프로모션(promoted)** 문맥 모두에서 사용자에 의해 상호작용된 횟수를 나타냅니다. 참여에는 리트윗, 즐겨찾기, 답글, URL 클릭, 해시태그 클릭, 멘션 클릭, 미디어 조회 등이 포함되지만, 이에 한정되지는 않습니다. 포함되는 전체 참여 액션 목록은 Engagement Data 섹션을 참조하십시오. 

기본적인 참여율을 계산하려면, 분석하려는 기간 동안 특정 게시물에 대해 총 참여 수를 해당 게시물의 총 노출 수로 나눈 값을 사용하십시오.

노출 수 및 참여 데이터는 귀하가 소유한 @handle의 게시물이거나, 귀하의 앱이 해당 게시물의 세부 정보를 열람할 수 있도록 권한을 부여한 @handle의 게시물에 대해서만 조회할 수 있습니다. 내부적으로 Engagement API는 계약된 @handle 한도에 대해 요청된 고유 @handle 수를 추적합니다. 월간 단위로 클라이언트 측에서도 @handle 요청 사용량을 함께 추적하는 것을 권장합니다.  

<div id="video-metrics">
  #### 동영상 메트릭
</div>

X 내에서 미디어 노출을 나타내는 메트릭에는 몇 가지 서로 다른 메트릭이 있습니다. 첫 번째는 동영상 조회수 메트릭으로, 최소 2초 동안 동영상의 50% 이상이 화면에 노출되었는지에 대한 MRC 표준에 기반합니다. 두 번째는 Media Views로, 동영상, vine, gif, 이미지를 통틀어 자동 재생 및 클릭으로 발생한 모든 미디어 조회수를 포함합니다.

동영상 조회수 메트릭은 /28hour 및 /historical 엔드포인트를 통해 소유한 게시물에 대해 제공되며, /totals 엔드포인트를 통해 소유하지 않은 모든 게시물에 대해서도 제공됩니다. 

X 사용자 인터페이스의 동영상 조회수 메트릭도 동일한 MRC 표준을 사용하지만, X가 소유·운영하는 플랫폼(모바일 앱 및 웹사이트)에 표시되는 동영상 조회수 메트릭과 Engagement API의 여러 엔드포인트를 통해 수신하는 값 사이에 차이가 있을 수 있다는 점에 유의하세요.

* /totals 엔드포인트와 X 사용자 인터페이스에서 제공되는 동영상 조회수는 해당 동영상이 게시된 모든 게시물에 걸쳐 집계된 동영상 조회수를 표시합니다. 이는 /totals를 통해 제공되고 X UI에 표시되는 메트릭이, 해당 동영상이 리트윗되었거나 별도의 게시물에서 재게시된 모든 인스턴스의 조회수를 합산하여 포함한다는 의미입니다.
* /28hour 및 /historical Engagement API 엔드포인트에서 제공되는 동영상 조회수는 메트릭을 조회 중인 특정 게시물이 생성한 조회수만 포함합니다. 
   

게시 후 1800일이 지난 게시물에 대해서는 동영상 조회수를 제공하지 않는다는 점에 유의하세요. 대신, 1800일이 지난 게시물 목록을 포함하는 객체를 제공합니다. 요청한 게시물에 대한 다른 모든 메트릭은 별도의 객체로 계속 수신하게 됩니다. 아래는 응답 예시입니다:

```json
{
  "unsupported_for_video_views_tweet_ids": [
    "479311209565413376",
    "477139122520219648"
  ],
  "grouping name": {
    "479311209565413376": {
      "favorites": "69",
      "impressions": "1692",
      "retweets": "142",
      "video_views": "0"
    },
    "477139122520219648": {
      "favorites": "10",
      "impressions": "1023",
      "retweets": "6",
      "video_views": "0"
    },
    "1397568983931392004": {

```

Media Views 지표는 /28hour 및 /historical 엔드포인트에서 소유 게시물에 대해서만 제공됩니다.

<div id="engagement-api-groupings">
  ### Engagement API 그룹화
</div>

그룹화를 사용하면 반환된 참여 메트릭을 원하는 방식으로 구성할 수 있습니다. 요청당 최대 3개의 그룹화를 포함할 수 있습니다. 메트릭을 다음 값 중 하나 이상으로 그룹화할 수 있습니다:

*세 엔드포인트 모두에서 지원:*

* tweet.id
* engagement.type
   

*`/28hr` 및 `/historical` 엔드포인트는 시계열 메트릭을 제공할 수 있으므로 다음을 지원합니다:*

* engagement.day
* engagement.hour

그룹화는 순차적으로 적용되므로, `group_by` 값의 순서를 변경하여 원하는 결과 형식을 바꿀 수 있습니다. 네 개의 `group_by` 값을 포함하는 그룹화는 다음 두 가지 형식 중 하나로만 지원됩니다:

```bash
"group_by": [
    "tweet.id",
    "engagement.type",
    "engagement.day",
    "engagement.hour"
  ]
```

또는

```bash
"group_by": [
    "engagement.type",
    "tweet.id",
    "engagement.day",
    "engagement.hour"
]
```

예를 들어, 메트릭 유형의 총계를 생성하려면 요청의 일부로 다음과 같은 “groupings” 사양을 포함하세요(요청을 구성하는 방법에 대한 자세한 내용은 [API Reference](/ko/x-api/enterprise-gnip-2.0/fundamentals/engagement-api#post-insights-engagement) 페이지를 참고하세요):

```json
{
  "engagement_types": [
    "favorites",
    "replies",
    "retweets"
  ],
  "groupings": {
    "Grand Totals": {
      "group_by": [
        "engagement.type"
      ]
    }
  }
}
```

이러한 그룹화를 사용하면 Engagement API의 JSON 응답 루트 레벨에 `"Grand Totals"` 속성이 추가되며, 이 속성에는 메트릭 유형별 총합이 포함됩니다:

```json
{
  "총계": {
    "favorites": "12",
    "replies": "2",
    "retweets": "2"
  }
}
```

단일 게시물에 대해 게시물 ID 기준으로 그룹화된 4시간 동안의 시계열 메트릭을 생성하려면, 요청에 포함되는 Grouping 사양은 다음과 같습니다:

```json
{
  "start": "2016-02-10T17:00:00Z",
  "end": "2016-02-10T21:00:00Z",
  "tweet_ids": [
    "697506383516729344"
  ],
  "engagement_types": [
    "impressions",
    "engagements"
  ],
  "groupings": {
    "Tweets_MetricType_TimeSeries": {
      "group_by": [
        "tweet.id",
        "engagement.type",
        "engagement.hour"
      ]
    }
  }
}
```

이 그룹화를 사용하면 Engagement API의 JSON 응답 루트에 `"Tweets_MetricType_TimeSeries"` 속성이 포함되며, 여기에는 게시물 id별, 메트릭 유형별로 세분화된 메트릭과 이에 대한 시간별 시계열 데이터가 들어 있습니다.

```json
{
  "Tweets_MetricType_TimeSeries": {
    "697506383516729344": {
      "impressions": {
        "2016-02-10": {
          "17": "551",
          "18": "412",
          "19": "371",
          "20": "280"
        }
      },
      "engagements": {
        "2016-02-10": {
          "17": "8",
          "18": "6",
          "19": "3",
          "20": "0"
        }
      }
    }
  }
}
```

    계약에 Engagement API와 함께 사용할 수 있는 고유 핸들 수에 대한 제한이 포함된 경우, 내부 X 시스템은 Engagement API로 쿼리된 게시물을 소유한 인증된 사용자 수를 추적합니다. 고객도 클라이언트 측에서 이 고유한 수치를 함께 추적해야 합니다. 현재 Engagement API용 @handle 사용량을 확인할 수 있는 사용량 API나 UI는 없습니다. 계약된 것보다 더 많은 @handle이 요청되더라도 시스템에서 초과 사용을 차단하지는 않습니다. 청구 월 말에 쿼리된 고유 @handle 수가 계약된 수와 비교되며, 계약 조건에 따라 초과 사용분이 청구됩니다.
  </Accordion>

  <Accordion title="Engagement API에 대한 @handle 사용량을 확인할 수 있나요?">
    현재 Engagement API용 @handle 사용량을 확인할 수 있는 사용량 API나 UI는 없습니다. 계약된 것보다 더 많은 @handle이 요청되더라도 시스템에서 초과 사용을 차단하지는 않습니다. 청구 월 말에 쿼리된 고유 @handle 수가 계약된 수와 비교되며, 계약 조건에 따라 초과 사용분이 청구됩니다.

    **페이로드에 반환되는 `engagements` 메타데이터 필드 값이 다양한 참여 메트릭의 합과 일치하지 않습니다. 왜 그런가요?**

    이는 정상적인 동작입니다. `engagements` 메타데이터 필드는 API에서 반환되는 개별 참여 메트릭의 합과 항상 일치하지 않을 수 있습니다. 이는 `engagements` 메타데이터 필드에 페이로드에서 개별 메트릭으로 분리되어 제공되지 않는 추가 참여가 포함될 수 있기 때문입니다. 다시 말해, 여러 참여 메트릭의 합계를 모두 더하더라도 페이로드에 반환되는 `engagements` 메트릭 필드에 표시되는 값과 일치하지 않을 수 있습니다.

    `engagements` 메타데이터 필드는 해당 게시물에서 발생한 모든 클릭 또는 상호작용으로 이해하시면 됩니다.
     

    **페이로드 응답의 `url_clicks` 필드가 숫자를 반환하는데, 실제로 해당 게시물에는 URL이 없습니다. 어떻게 가능한가요?**

    이는 해시태그처럼 다른 페이지로 연결되는 링크를 생성하는 요소를 포함한 게시물이 사용자가 클릭했을 때 URL 클릭으로 계산될 수 있기 때문입니다.
     
  </Accordion>

  <Accordion title="왜 특정 게시물에 대한 참여 데이터를 받을 수 없나요?">
    특정 게시물에 대한 참여 데이터를 검색할 수 없는 이유는 다음과 같이 여러 가지가 있을 수 있습니다.

    * 요청한 게시물 ID(또는 여러 ID)가 서드파티를 대신해 데이터를 검색하기 위해 사용 중인 인증 토큰으로는 사용할 수 없는 경우
    * /totals 엔드포인트에 대해 요청한 게시물 ID(또는 여러 ID)가 90일 이내의 것이 아니어서 노출수(impressions)나 참여 메트릭을 반환할 수 없는 경우
    * 요청한 게시물 ID(또는 여러 ID)가 더 이상 사용 가능하지 않은 경우로, 일반적으로 삭제되었거나 다른 이유로 더 이상 공개적으로 제공되지 않음을 의미합니다.

    위와 같은 경우에 수신하게 될 수 있는 다양한 [오류 메시지](/ko/x-api/enterprise-gnip-2.0/fundamentals/engagement-api#error-messages)를 검토해 주세요.
  </Accordion>

  <Accordion title="Engagement API에서 레이트 리밋을 어떻게 처리할 수 있나요?">
    Engagement API에 요청을 보낼 때 응답 헤더에 반환되는 `x-per-minute-limit` 및 `x-per-minute-remaining` 정보를 사용하여 사용량을 모니터링할 수 있습니다.

    `x-per-minute-limit`은 허용량을, `x-per-minute-remaining`은 남은 호출 수를 나타냅니다.
  </Accordion>
</AccordionGroup>

<div id="authentication">
  #### **Authentication**
</div>

Engagement API는 HTTPS 사용을 요구하며 User Context와 Application-Only OAuth 두 가지 방식을 모두 지원합니다. 대부분의 Engagement API 요청은 3-Legged OAuth(특정 버전의 User Context) 사용이 필요합니다. 이는 Twitter 계정 담당자에 의해 Engagement API 액세스로 등록 및 승인된 앱의 consumer key와 secret, 그리고 트윗 소유자의 access token 및 access token secret을 사용하여 엔드포인트를 호출해야 함을 의미합니다. 다음 요청에는 이 유형의 OAuth가 필요합니다.

* 소유한 트윗에만 제한되는 Impressions 및 Engagements 메트릭 유형을 얻기 위한 /totals에 대한 모든 요청
* /28hr에 대한 모든 요청
* /historical에 대한 모든 요청

일부 Engagement API 요청은 Application-Only OAuth를 사용하여 수행할 수 있으며, 이 경우 consumer key와 secret 또는 bearer token만 제공하면 됩니다. 다음 요청은 이 유형의 OAuth로 수행할 수 있습니다.

* 모든 트윗에 대해 조회할 수 있는 Favorites, Replies, Retweets 또는 Video Views 메트릭 유형을 얻기 위한 /totals에 대한 모든 요청

어떤 요청이든, [developer.x.com](https://developer.x.com/en/portal/dashboard)에 있는 앱 관리 콘솔에서 Twitter 앱과 해당 API 키를 먼저 설정해야 합니다.

참고: developer.x.com에서 Twitter 계정으로 로그인한 상태라면 기존 [Twitter apps](/ko/resources/fundamentals/developer-apps)를 [Twitter app dashboard](https://developer.x.com/en/apps)를 통해 조회 및 편집할 수 있습니다.

앱을 설정한 후에는 Engagement API에 요청을 보낼 수 있도록 앱 ID가 계정 담당자의 승인을 받아야 합니다. 액세스 토큰은 &quot;current user&quot;(현재 사용자)를 나타내는 데 사용되어야 하며, 별도의 사용자를 대신하여 수행되는 요청은 유효한 토큰으로 서명되어야 합니다. OAuth signature base string을 준비하기 전에 URL과 POST 본문 내에서 [예약 문자 인코딩](https://tools.ietf.org/html/rfc3986)이 적절히 이루어지고 있는지 확인하십시오.

OAuth 시작 방법에 대한 자세한 내용은 다음 링크를 참조하십시오.

* [OAuth 개요](/ko/resources/fundamentals/authentication)
* [User Context라고도 하는 3-Legged OAuth 사용](/ko/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow)
* [Application-Only OAuth 사용](/ko/resources/fundamentals/authentication#bearer-token-also-known-as-app-only)

  #### **Error Messages**
</div>

여러 가지 상황에서 Engagement API는 상황별 오류 메시지를 반환하며, 이에 대해 애플리케이션이 적절히 대응할 수 있어야 합니다. 아래 표에는 이러한 오류 메시지의 일반적인 예시와, 이를 어떻게 해석해야 하는지에 대한 설명이 포함되어 있습니다. 많은 경우 Engagement API는 보다 자세한 정보를 제공하는 200 OK 응답의 일부로, 사용 가능한 데이터에 대한 일부 결과와 함께 특정 오류 메시지를 함께 반환한다는 점에 유의하세요.
