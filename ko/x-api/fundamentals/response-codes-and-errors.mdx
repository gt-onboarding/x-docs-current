X API는 상황을 바로 이해하고 디버깅할 수 있도록 다음과 같은 응답 및 오류 코드를 제공합니다. 아래의 디버깅 가이드와 오류 인덱스를 추가적인 참고 자료로 활용하십시오.

성공적인 응답은 200번대 HTTP 상태 코드와 함께 JSON 기반 페이로드로 반환되며, 여기에는 요청한 객체(조회, 생성, 수정, 삭제된 객체)와 서버가 해당 요청을 어떻게 해석했는지가 포함됩니다.

오류 응답은 200번대가 아닌 HTTP 상태 코드와 함께 반환됩니다. 서로 다른 오류 코드는 오류가 발생한 서로 다른 원인을 나타냅니다. X API는 모든 요청에 대해 적절한 [HTTP 상태 코드](https://en.wikipedia.org/wiki/List_of_HTTP_status_codes)를 반환하도록 노력합니다.

<div id="x-api-v2-http-status-codes">
  ## X API v2 HTTP 상태 코드
</div>

| Code | Text                  | Description                                                                                                                                                        | Troubleshooting tips                                                                                                                                   |
| :--- | :-------------------- | :----------------------------------------------------------------------------------------------------------------------------------------------------------------- | :----------------------------------------------------------------------------------------------------------------------------------------------------- |
| 200  | OK                    | 요청이 성공했습니다.                                                                                                                                                        |                                                                                                                                                        |
| 400  | Bad Request           | 요청이 유효하지 않거나 처리할 수 없습니다. 함께 반환되는 오류 메시지가 추가 설명을 제공합니다. 인증이 없거나 쿼리 매개변수가 잘못된 요청은 유효하지 않은 것으로 간주되며 이 응답이 반환됩니다.                                                      | JSON 쿼리의 형식을 다시 확인하세요. 예를 들어, 규칙에 정확히 일치하는 연산자나 기타 연산자와 함께 큰따옴표 문자가 포함되어 있다면, JSON 형식의 구조와 구분하기 위해 역슬래시를 사용해 이를 이스케이프해야 할 수 있습니다.                      |
| 401  | Unauthorized          | 요청을 인증하는 데 문제가 있었습니다. 누락되었거나 잘못된 인증 자격 증명 때문에 발생할 수 있습니다. 정의되지 않은 다른 상황에서도 이 코드가 반환될 수 있습니다.                                                                       | 올바른 [인증 방식](https://docs.x.com/resources/fundamentals/authentication/overview)을 사용하고 있는지, 자격 증명이 정확한지 확인하세요.                                           |
| 403  | Forbidden             | 요청은 이해되었지만 거부되었거나 접근이 허용되지 않습니다. 함께 반환되는 오류 메시지가 이유를 설명합니다.                                                                                                        | 사용 중인 엔드포인트에 접근할 수 있도록 개발자 계정에 필요한 권한이 포함되어 있는지 확인하세요. App이 허용 목록(allowlist)에 포함되어야 할 수도 있습니다(예: Engagement API 또는 Ads API). 또는 액세스 권한에 가입해야 할 수 있습니다. |
| 404  | Not Found             | 요청된 URI가 유효하지 않거나, 사용자와 같은 요청된 리소스가 존재하지 않습니다.                                                                                                                     | 사용 중인 엔드포인트에 대해 유효한 매개변수와 올바른 URI를 사용하고 있는지 확인하세요.                                                                                                     |
| 409  | Connection Exception  | 규칙이 하나도 설정되지 않은 filtered stream에 연결을 시도할 때 반환됩니다.                                                                                                                  | 연결하려는 스트림에 최소 하나의 규칙을 생성했는지 확인하세요. filtered stream은 활성 규칙과 일치하는 게시물만 반환합니다. 규칙이 없으면 스트림은 어떤 게시물도 반환하지 않습니다.                                            |
| 429  | Too Many Requests     | App의 [요청 한도](/ko/resources/fundamentals/rate-limits) 또는 [Post cap](/ko/x-api/fundamentals/post-cap)이 소진되었을 때 반환됩니다. [요청 한도](/ko/resources/fundamentals/rate-limits)를 참조하세요. | 사용 중인 엔드포인트에서 허용되는 단위 시간당 요청 개수를 확인하세요. 해당 시간이 다시 초기화될 때까지 기다리세요. 요청 간 간격을 두어 요청 한도에 도달하지 않도록 하거나, 더 높은 데이터 플랜으로 업그레이드하는 것을 고려하세요.                     |
| 500  | Internal Server Error | 문제가 발생했습니다. 일반적으로 이는 높은 부하 상황이거나 특정 엔드포인트에 일시적인 문제가 있을 때 발생하는 일시적인 오류입니다.                                                                                          | 다른 사용자도 유사한 문제를 겪고 있는지 확인하기 위해 X API [상태 페이지](/ko/status)나 [개발자 커뮤니티 포럼](https://devcommunity.x.com/)을 확인하거나, 잠시 기다린 후 다시 시도하세요.                          |
| 501  | Unimplemented         | X API가 이 엔드포인트를 지원하지 않으며 요청을 처리할 수 없습니다.                                                                                                                           |                                                                                                                                                        |
| 503  | Service Unavailable   | X 서버는 동작 중이지만 요청이 과도하게 몰린 상태입니다. 나중에 다시 시도하세요.                                                                                                                     | 다른 사용자도 유사한 문제를 겪고 있는지 확인하기 위해 X API [상태 페이지](/ko/status)나 [개발자 커뮤니티 포럼](https://devcommunity.x.com/)을 확인하거나, 잠시 기다린 후 다시 시도하세요.                          |

요청 중 오류가 발생하면, 문제 진단을 돕기 위해 응답 본문에 오류에 대한 상세 정보가 반환됩니다. 문제의 유형을 나타내는 URI 형식의 type 필드가 제공되며, 추가 필드는 문제에 대한 세부 정보를 제공합니다. 이러한 응답 본문에는 항상 type, title, detail 필드가 포함됩니다(아래 표 참조). 아래 예시와 같이 추가 필드는 오류 유형에 따라 달라집니다.

```json
{
  "client_id": "101010101",
  "required_enrollment": "Standard Basic",
  "registration_url": "https://developer.twitter.com/en/account",
  "title": "Client Forbidden",
  "detail": "이 요청은 요청된 엔드포인트에 등록된 승인된 개발자 계정을 사용하여 수행해야 합니다. 자세한 내용은 문서를 참조하세요.",
  "reason": "client-not-enrolled",
  "type": "https://api.x.com/2/problems/client-forbidden"
}
```

<div id="partial-errors">
  ## 부분 오류
</div>

경우에 따라 200 상태 코드를 반환한 응답에서 위에 설명된 오류들을 볼 수 있습니다. 이런 경우 엔드포인트는 반환할 수 있는 데이터는 최대한 반환하는 동시에, 반환할 수 없었던 내용에 대해서는 자세한 오류 정보를 제공합니다.

예를 들어, [Posts 조회 엔드포인트](/ko/x-api/posts/lookup/introduction)는 X [developer App](/ko/resources/fundamentals/developer-apps#overview)이 둘 이상의 ID를 요청할 수 있도록 합니다. 이들 Posts 중 일부는 조회할 수 있지만 그중 하나가 삭제된 경우, 조회 가능한 Posts는 응답의 `data` 필드에 반환됩니다. 추가로 `errors` 필드가 페이로드에 포함되어, 어떤 요청된 게시물을 반환할 수 없었는지 표시합니다. 문제를 더 쉽게 진단할 수 있도록, 전체 요청이 실패했을 때와 동일한 오류 형식을 사용합니다.

<div id="error-information">
  ### 오류 정보
</div>

각 문제 유형은 발생한 문제의 성격을 나타냅니다. 발생할 수 있는 문제의 전체 목록은 [API 사양](https://api.x.com/2/openapi.json)에서도 확인할 수 있습니다.

X API는 모든 요청에 대해 적절한 [HTTP 상태 코드](https://en.wikipedia.org/wiki/List_of_HTTP_status_codes)를 반환하려고 합니다.

| Title | Type | Description |
| :--- | :--- | :--- |

| Invalid Request Problem | `https://api.X.com/2/problems/invalid-request` | 이 요청이 잘못되었음을 나타내는 문제입니다. 요청이 POST 본문을 사용하는 경우, 내용이 유효한 JSON이며 [OpenAPI 사양](https://api.x.com/2/openapi.json)과 일치하는지 확인하십시오. |

<div id="troubleshooting-tips">
  ## 문제 해결 팁
</div>

<Accordion title="코드 디버깅">
  애플리케이션을 빌드할 때 때때로 오류나 예기치 않은 문제가 발생하는 것은 자연스러운 일입니다. 아래는 코드를 디버깅하는 데 도움이 될 몇 가지 팁입니다.

  먼저 문제를 더 작은 부분으로 나누어 어디에서 문제가 발생하는지 파악하세요. 예를 들어 REST 또는 스트리밍 엔드포인트를 통합하는 경우, 문제는 다음 항목들 중 어느 곳에서든 발생할 수 있습니다.

  * 엔드포인트에는 올바른 [인증](/ko/resources/fundamentals/authentication/overview)이 필요합니다.
  * 엔드포인트에는 유효한 파라미터와 헤더가 필요합니다. 모든 필터링 규칙은 올바른 연산자와 적절한 [구문](/ko/x-api/enterprise-gnip-2.0/fundamentals/rules-filtering#building-rules-and-queries)을 사용해 작성되어야 합니다.
  * 엔드포인트의 URI가 올바르고, REST 엔드포인트의 경우 올바른 HTTP 메서드가 사용되고 있어야 합니다.
  * 액세스하려는 데이터 또는 리소스를 사용할 권한이 없을 수 있습니다(예: 비공개 데이터는 인증된 사용자만 사용할 수 있습니다).
  * 현재 데이터 패키지에서는 특정 엔드포인트와 특정 요청 한도에 대해서만 액세스를 제공할 수 있습니다. 자세한 내용은 [developer portal](https://developer.x.com/en/portal/products)을 확인하세요.
  * 코드에서 엔드포인트를 통합하기 위해 서드파티 라이브러리를 사용하고 있습니다.
  * 코드가 엔드포인트 응답을 올바르게 파싱해야 합니다.

  함께 제공되는 오류 메시지를 읽어 보세요. 이는 문제가 무엇인지에 대한 좋은 단서를 제공합니다. 각 오류 코드에 대한 문제 해결 팁은 [오류 및 응답 코드 섹션](#x-api-http-status-codes)의 표를 참고하세요.

  REST 엔드포인트의 경우, [Postman](https://www.postman.com/) 또는 [Insomnia](https://insomnia.rest/)와 같은 REST 클라이언트를 사용해 위의 (1)부터 (5)까지 단계를 검증할 수 있습니다( [&quot;Getting started with Postman&quot;](/ko/tutorials/postman-getting-started) 가이드를 참고하세요). REST 클라이언트로 보낸 요청이 200 성공 상태 코드를 반환한다면, 문제는 엔드포인트 요청 자체가 아니라 사용 중인 코드나 라이브러리에 있다고 볼 수 있습니다.

  스트리밍 엔드포인트의 경우, cURL(URL 문법을 사용해 요청을 보내거나 받는 명령줄 도구)을 사용해 문제가 엔드포인트에 대한 요청(위의 (1)부터 (5)까지)에서 발생하는지, 아니면 코드 자체(위의 (6)부터 (7)까지)에서 발생하는지 확인할 수 있습니다.
</Accordion>

<Accordion title="인증 문제">
  * 엔드포인트에서 요구하는 올바른 [인증 방식](/ko/resources/fundamentals/authentication/overview)을 사용하고 있는지 확인하세요. 이는 엔드포인트의 API 레퍼런스 페이지에서 확인할 수 있습니다.
  * 인증 자격 증명이 올바른지 확인하세요. 개발자 대시보드의 [Apps 섹션](https://developer.x.com/en/portal/projects-and-apps)에서(“Details” 아래) 앱의 키와 토큰을 확인하거나 재생성할 수 있습니다.
  * 요청에 대해 `oauth_nonce`, `oauth_signature`, `oauth_timestamp`를 사용하여 [OAuth 1.0a 요청을 올바르게 인가](/ko/resources/fundamentals/authentication/oauth-1-0a/authorizing-a-request)했는지 확인하세요.
  * 문제가 계속될 경우, OAuth 라이브러리, [Postman](https://www.postman.com/) 또는 [Insomnia](https://insomnia.rest/)와 같은 REST 클라이언트, 혹은 [xurl](https://github.com/xdevplatform/xurl) 사용을 고려하세요.

  위 모든 내용에 대한 추가 정보는 [인증 안내서](/ko/resources/fundamentals/authentication/overview)를 참고하세요.
</Accordion>

<Accordion title="요청 한도 및 게시물 상한(Post cap) 관련 문제">
  429 오류는 특정 엔드포인트의 요청 한도에 도달했거나, 게시물 상한(Post cap)에 도달했을 때 발생할 수 있습니다.

  &#39;Too many requests&#39;라는 구체적인 오류가 반환되었다면, 해당 엔드포인트의 요청 한도에 도달했다는 의미입니다. 즉, 지정된 시간 동안 해당 엔드포인트에 허용된 최대 요청 개수를 초과한 것입니다.

  요청 한도는 앱 단위와 사용자 단위로 설정되어 있다는 점에 유의하십시오.

  * **X App 수준**은 OAuth 2.0 앱 전용(App-Only)을 사용할 때 허용되는 요청 개수를 의미하며, 이 경우 요청 한도는 전체 App에 대해 전역적으로 결정됩니다. 예를 들어, 어떤 메서드가 한 요청 한도 윈도우당 15개의 요청을 허용한다면, 사용 중인 X App 전체에 대해 윈도우당 15개의 요청을 보낼 수 있음을 의미합니다. 이 한도는 사용자 수준 한도와는 완전히 별개로 간주됩니다. 자세한 내용은 [OAuth 2.0 App-Only](/ko/resources/fundamentals/authentication/oauth-2-0/application-only) 가이드를 참고하십시오.
  * **사용자 컨텍스트 수준**은 OAuth 1.0a 사용자 컨텍스트 또는 OAuth 2.0 Authorization Code with PKCE를 사용할 때, 사용자별 액세스 토큰당 허용되는 요청 개수를 의미합니다. 예를 들어, 어떤 메서드가 한 요청 한도 윈도우당 15개의 요청을 허용한다면, 윈도우당 그리고 사용자 액세스 토큰당 15개의 요청을 허용한다는 의미입니다. 사용자 액세스 토큰을 얻는 방법에 대해서는 [OAuth 1.0a](/ko/resources/fundamentals/authentication/oauth-1-0a/obtaining-user-access-tokens) 및 [OAuth 2.0](/ko/resources/fundamentals/authentication/oauth-2-0/user-access-token) 가이드를 참고하십시오.

  먼저, 사용 중인 엔드포인트의 요청 한도를 확인하십시오. 이 정보는 해당 엔드포인트의 API 레퍼런스 페이지와 새로운 Developer Portal 대시보드에서 확인할 수 있습니다.

  요청 한도에 대한 추가 정보(특정 요청 한도에 대해 App의 사용량을 HTTP 헤더로 추적하는 방법, 요청 한도 429 오류 코드에서 복구하는 방법, 애초에 요청 한도에 걸리지 않도록 하는 팁 등)는 다음 문서를 참고하십시오.

  * [v2 엔드포인트의 요청 한도](/ko/x-api/fundamentals/rate-limits)

  &quot;Usage cap exceeded: Monthly product cap&quot; 오류를 받은 경우, 현재 액세스 레벨에 대한 게시물 상한(Post cap)에 도달했다는 의미입니다. 이러한 [Post caps](/ko/x-api/fundamentals/post-cap)가 무엇인지에 대해서는 문서 페이지에서 자세히 확인하실 수 있습니다.
</Accordion>

<Accordion title="게시물 누락 관련 문제">
  엔드포인트에서 반환될 것으로 예상한 게시물이 있었는데 반환되지 않은 경우, 아래 단계를 따라 주십시오.

  * 올바른 [연산자](/ko/x-api/enterprise-gnip-2.0/fundamentals/rules-filtering#enterprise-operators)와 [구문](/ko/x-api/enterprise-gnip-2.0/fundamentals/rules-filtering#building-rules-and-queries)을 사용하고 있는지 규칙을 확인하십시오. 올바른 구문을 사용하고 있는지 확인하기 위해 규칙을 더 작은 절로 나누어 보십시오.
  * 게시물을 보낸 계정이 게시물이 생성될 당시 [보호됨 상태](https://help.twitter.com/en/safety-and-security/public-and-protected-tweets)였다면, 요청 시점에 해당 계정이 공개 상태라고 하더라도 게시물은 반환되지 않습니다. 일반적으로 이는 [X 고급 검색](https://x.com/search-advanced)을 사용해 확인할 수 있습니다. X 고급 검색 기능으로 특정 게시물이 검색되지 않는다면, 해당 게시물은 엔드포인트에서도 반환되지 않을 것이라고 가정해야 합니다.

  다음 단계는 스트리밍 엔드포인트에만 적용됩니다.

  * 게시물이 전송될 때 스트림에 연결된 상태였습니까? 게시물 객체에 포함된 타임스탬프는 UTC 시간임을 기억하십시오. 게시물이 전송될 당시 연결이 끊겼다면, 누락된 데이터를 보완하기 위해 제공되는 [복구 및 중복 기능](/ko/x-api/posts/filtered-stream/integrate/recovery-and-redundancy-features#recovery-and-redundancy)을 검토하십시오.
  * 게시물이 전송될 때 규칙이 이미 설정되어 있었습니까? 게시물 객체에 포함된 타임스탬프는 UTC 시간임을 기억하십시오.
</Accordion>

<Accordion title="버퍼 가득 참으로 인한 연결 해제 문제">
  스트림이 X에서 데이터를 전달하는 속도를 따라가지 못하고, 앱이 스트림에서 데이터를 충분히 빠르게 소비하지 못할 때 다음 오류 중 하나가 발생할 수 있습니다:

  ```
  This stream has been disconnected because your client was unable to keep up with us.
  ```

  ```
  This stream has been disconnected for operational reasons.
  ```

  일정 기간 동안은 전송 지연을 허용하고, X 측에서는 각 스트림마다 임시 스테이징 버퍼 용량을 보유합니다. 그러나 해당 기간 안에 따라잡지 못하면, 현재 시점에서 다시 연결할 수 있도록 연결을 해제합니다. 전체 버퍼로 인한 연결 해제 시점에 버퍼 내에 있던 데이터는 손실될 수 있다는 점에 유의하세요.

  이러한 문제는 데이터가 급증하는 시점에 발생할 수 있습니다. 일반적으로는 처리 프로세스와 분리된, 데이터를 빠르게 소비하기 위한 별도의 버퍼링 프로세스를 사용할 것을 권장합니다.

  v1.1 엔드포인트를 사용하는 Enterprise 고객이라면, 이와 같은 연결 해제를 방지하기 위해 앱을 최적화하는 방법에 대해 [connection](/ko/x-api/enterprise-gnip-2.0/powertrack-api#disconnections-explained) 관련 문서와 스트리밍 데이터 소비에 관한 [여기](https://developer.x.com/en/docs/tutorials/consuming-streaming-data) 및 [여기](/ko/x-api/enterprise-gnip-2.0/powertrack-api#planning-for-high-volume-social-data-events) 문서에서 자세한 내용을 확인할 수 있습니다.

  연결 해제로 인해 누락된 게시물을 복구하기 위한 다양한 도구가 있으며, 아래에 나열된 도구들도 포함됩니다. 아래 도구들은 Enterprise 수준 접근 권한이 있는 v1.1 엔드포인트에서만 사용할 수 있습니다.

  * [**Redundant Connections**](/ko/x-api/enterprise-gnip-2.0/powertrack-api#recovery-and-redundancy-features) - 여러 연결을 통해 여러 서버에서 스트림을 소비하여, 한 연결이 끊어지더라도 데이터 누락을 방지합니다.
  * [**Replay**](/ko/x-api/enterprise-gnip-2.0/powertrack-api#recovery-and-redundancy-features) - 별도의 스트림을 사용해 최근 5일 이내의 데이터를 복구합니다.
  * [**Backfill**](/ko/x-api/enterprise-gnip-2.0/powertrack-api#recovery-and-redundancy-features) - 5분 이내에 다시 연결하여 중단된 지점부터 다시 시작합니다.
  * **Historical PowerTrack** - 전체 X 아카이브에서 데이터를 복구합니다.
</Accordion>
