---

매일 수많은 개발자가 X API에 요청을 보냅니다. 이러한 요청의 방대한 양을 관리하기 위해 보낼 수 있는 요청 수에 제한을 둡니다. 이러한 제한 덕분에 개발자 커뮤니티가 의존하는 안정적이고 확장 가능한 API를 제공할 수 있습니다.

허용되는 최대 요청 수는 일반적으로 특정 기간, 즉 시간 윈도우를 기준으로 정해집니다. 가장 일반적인 간격은 15분입니다. 예를 들어, 15분당 900회의 요청 한도가 있는 엔드포인트는 임의의 15분 구간 동안 최대 900회의 요청을 허용합니다.

요청 한도는 인증 방식에 따라 달라집니다. 예를 들어 [OAuth 1.0a 사용자 컨텍스트](/ko/resources/fundamentals/authentication)를 사용하는 경우, 각 사용자의 [액세스 토큰](/ko/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow)에는 기간별로 자체 요청 한도가 있습니다. 반대로 [OAuth 2.0 Bearer Token](/ko/resources/fundamentals/authentication#oauth-2-0)을 사용하는 경우, 앱마다 기간별로 별도의 한도가 부여됩니다. 이러한 한도를 초과하면 오류가 반환됩니다.

<div id="rate-limits-and-authentication-method">
  ## 요청 한도 및 인증 방법
</div>

요청 한도는 개발자 앱 수준과 사용자 액세스 토큰 수준 모두에 설정됩니다.

* **[OAuth 2.0 Bearer Token](/ko/resources/fundamentals/authentication/oauth-2-0/application-only): 앱 요청 한도**
  이 방법을 사용하면 개발자 앱을 대신하여 일정 수의 요청을 보낼 수 있습니다. 이 인증 방법을 사용할 때 요청 한도는 해당 Bearer Token으로 수행한 요청 수를 기준으로 적용됩니다.
  * 예: 15분 간격당 450건의 요청 한도가 있는 경우, 해당 시간 동안 앱을 대신하여 450건의 요청을 보낼 수 있습니다.

* **[OAuth 1.0a User Context](/ko/resources/fundamentals/authentication/oauth-1-0a/obtaining-user-access-tokens): 사용자 요청 한도**
  이 방법을 사용하면 사용자 액세스 토큰으로 식별되는 X 사용자를 대신하여 요청을 보낼 수 있습니다. 예를 들어 게시물의 비공개 메트릭을 조회하는 경우, 해당 사용자에 대해 [3-legged OAuth flow](/ko/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow)를 사용해 생성한 사용자 액세스 토큰으로 인증해야 합니다.
  * 예: 사용자당 15분당 900건의 요청 한도가 있는 경우, 해당 시간 동안 사용자당 최대 900건의 요청을 보낼 수 있습니다.

<div id="http-headers-and-response-codes">
  ## HTTP 헤더와 응답 코드
</div>

가장 최근에 보낸 요청을 기준으로, 현재 rate limit 상황에서 애플리케이션이 어느 상태에 있는지 파악하기 위해 HTTP 헤더를 사용합니다.

* `x-rate-limit-limit`: 해당 엔드포인트의 rate limit 상한
* `x-rate-limit-remaining`: 15분 시간 구간에서 남은 요청 수
* `x-rate-limit-reset`: rate limit이 재설정되기까지 남은 시간(UTC epoch 초)

<div id="error-responses">
  ### 오류 응답
</div>

애플리케이션이 특정 엔드포인트의 레이트 리밋을 초과하면 API는 응답 본문에 다음 오류 메시지와 함께 [HTTP 429 “Too Many Requests”](http://tools.ietf.org/html/rfc6585) 응답을 반환합니다.

```json
{ "errors": [ { "code": 88, "message": "Rate limit exceeded" } ] }
```

<div id="recovering-from-a-rate-limit">
  ## 요청 한도에서 복구하기
</div>

이러한 요청 한도를 초과하면 엔드포인트에서 429 &#39;Too many requests&#39; 오류가 반환됩니다. 아래에서 설명하듯이, 요청 한도 오류가 발생했을 때의 모범 사례는 한도가 언제 재설정되는지 나타내는 HTTP 헤더를 확인하고, 그때까지 요청을 일시 중지하는 것입니다.<br />&quot;too many requests&quot; 또는 요청 한도 관련(rate-limiting) 오류가 발생하면, 요청을 보내는 빈도를 늦출 필요가 있습니다. 요청 한도 오류가 발생했을 때는 `x-rate-limit-reset:` HTTP 헤더를 확인해 요청 한도가 언제 재설정되는지 알 수 있습니다.<br /><br />또 다른 일반적인 패턴은 지수 백오프(exponential backoff)에 기반한 것으로, 요청 간 시간이 처음에는 짧게(예: 몇 초) 시작한 뒤 재시도할 때마다 두 배로 늘리는 방식입니다. 이는 요청이 성공하거나, 요청 간 최대 허용 시간(예: 몇 분)에 도달할 때까지 계속됩니다.<br /><br />이상적으로는 클라이언트 측에서 현재 적용된 요청 한도를 인지하고, 현재 초과한 시간 창(window)이 만료될 때까지 요청을 일시 중지할 수 있어야 합니다. 15분 한도를 초과한 경우, 재시도 전에 1~2분 정도 기다리는 것이 합리적입니다.<br /><br />요청 수에 대한 이러한 한도 외에도, Standard Basic 액세스 레벨은 최근 검색 및 filtered stream 엔드포인트에서 월 최대 500,000개의 게시물을 제공합니다. 게시물 수에 대한 월 한도를 초과했다면, 앱이 알림을 발생시키고, 매월 등록일을 인지한 뒤 해당 날짜까지 요청을 보류하는 편이 더 합리적입니다.

<div id="tips-to-avoid-being-rate-limited">
  ### rate limit에 걸리는 것을 피하기 위한 팁
</div>

아래 팁은 방어적으로 코드를 작성하고 rate limit에 걸릴 가능성을 줄이는 데 도움이 됩니다. 제공하려는 애플리케이션 기능 중 일부는, 특히 결과의 최신성과 관련해서는, rate limit 때문에 애초에 구현이 불가능할 수도 있습니다. 애플리케이션의 목표가 실시간 정보 제공이라면, filtered stream 및 sampled stream endpoint 사용을 검토해 보십시오.

<div id="caching">
  #### 캐싱
</div>

자주 사용될 것으로 예상되는 경우 API 응답을 저장하세요. 페이지를 불러올 때마다 API를 호출하는 대신, 응답을 로컬에 캐시하세요.

<div id="prioritize-active-users">
  #### 활성 사용자를 우선시하세요
</div>

사이트에서 많은 X 사용자를 추적하고 있다면(예: 현재 상태를 가져오거나 X 사용 통계를 가져오는 경우), 최근에 사이트에 로그인한 사용자에 대해서만 데이터를 요청하는 것을 고려하세요.

<div id="adapt-to-the-search-results">
  #### 검색 결과에 맞게 조정하기
</div>

애플리케이션에서 많은 검색어를 모니터링하는 경우, 결과가 없는 검색어에 대해서는 결과가 있는 검색어보다 요청 빈도를 더 낮게 설정하는 것이 좋습니다. 백오프 전략을 사용하면 자주 결과가 나오는 인기 쿼리는 최신 상태로 유지하면서, 거의 변하지 않는 쿼리를 계속 요청하느라 리소스를 낭비하지 않을 수 있습니다. 또는 filtered stream 엔드포인트를 사용해 검색 쿼리로 필터링하는 방법을 고려해 보세요.

<div id="denylist">
  #### 차단 목록
</div>

애플리케이션이 요청 한도를 남용한 것으로 판단되면 차단됩니다. 차단된 앱은 X API로부터 응답을 받을 수 없습니다. 본인이나 애플리케이션이 차단되었는데 이것이 실수라고 생각되는 경우, [Platform Support forms](https://support.x.com/forms/platform)를 통해 지원을 요청할 수 있습니다. 다음 정보를 포함해 주십시오:

1. 애플리케이션이 왜 차단되었다고 생각하는지 설명해 주십시오.
2. 현재 더 이상 요청 한도에 걸리지 않는 경우, 문제를 어떻게 해결했는지 자세히 설명해 주십시오.
