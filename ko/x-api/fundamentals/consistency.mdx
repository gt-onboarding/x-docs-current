<div id="consistency-across-the-x-api-v2-endpoints">
  ## X API v2 엔드포인트 전반에 걸친 일관성
</div>

새로운 X API v2의 주요 특징 중 하나는 엔드포인트 전반에 걸쳐 일관성이 유지된다는 점입니다. 이는 반환되는 객체, 기능, 그리고 API의 동작 방식이 유사한 엔드포인트 간에 통일된 형태를 가진다는 것을 의미합니다.

모든 엔드포인트에서 다음 요소들이 동일하다고 기대할 수 있습니다:

<div id="path-naming">
  ### 경로 이름 지정
</div>

경로 이름에는 항상 엔드포인트 [버전](/ko/x-api/fundamentals/versioning)이 포함되며, 그 뒤에 **리소스**가 옵니다. 그 다음으로, 앞의 리소스와 관련된 **ID**, 어떤 데이터를 반환할지 결정하는 데 도움이 되는 **선택 동사**(예: `search` 또는 `sample`), 데이터가 어떤 방식으로 전달될지 결정하는 데 도움이 되는 **전달 동사**(예: `stream`), 또는 기본 리소스와 **관계**를 가지는 다른 리소스(예: `/user/12/tweets`)가 경로에 포함될 수 있습니다. 마지막으로, 엔드포인트가 쿼리 매개변수를 지원한다면 끝에 **쿼리 매개변수**를 추가할 수 있습니다.

이러한 경로 및 쿼리 항목을 어떻게 구성할 수 있는지에 대한 몇 가지 예시는 다음과 같습니다:

`/version/resource/id?param1=value&param2=value /version/resource/delivery/selection?param1=value&param2=value`

실제 요청 예시는 다음과 같습니다:

`/2/tweets/1067094924124872705?expansions=attachments.media_keys&tweet.fields=author_id /2/users/2244994945?user.fields=created_at,description /2/tweets/search/stream /2/tweets/search/recent?query=snow`

<div id="json-schema">
  ### JSON 스키마
</div>

요청에 대한 응답은 [JSON Schema](http://json-schema.org/)를 사용해 정의됩니다. 이는 요청의 응답이 항상 객체 집합을 배열 형태로 반환되며, 각 요소는 id를 가진다는 것을 의미합니다. 요청은 id를 키로 사용하는 맵 형태로는 반환되지 않습니다.

<div id="response-object-and-parameters">
  ### 응답 객체 및 매개변수
</div>

기본적으로 반환되는 객체는 해당 객체 유형의 모든 엔드포인트에서 동일합니다.

* `id` 객체는 항상 문자열입니다.
* 매개변수와 응답 필드는 일관되게 미국식 영어 철자를 사용합니다.
* 값이 없으면 필드는 비어 있거나 아예 반환되지 않습니다.
* `entities` 객체에는 게시물 텍스트에서 추출된 엔티티만 포함됩니다. 여기에는 `urls`, `hashtags`, `mentions`, `cashtags`가 포함됩니다.
* `media_keys` 및 `poll_ids` 필드를 포함한 모든 카드 관련 정보는 `attachments` 객체로 반환됩니다.

<div id="authentication">
  ### 인증
</div>

모든 X API v2 엔드포인트는 인증이 필요합니다. 이들 중 많은 엔드포인트는 [OAuth 2.0 Bearer Token](/ko/resources/fundamentals/authentication#oauth-2-0) 방식을 지원하며, 요청을 성공적으로 수행하려면 해당 엔드포인트로의 요청에 Bearer Token을 포함해야 합니다.

다른 사용자를 대신하여 데이터를 생성, 조작 또는 조회할 수 있는 권한이 필요한 엔드포인트의 경우 [OAuth 1.0a 사용자 컨텍스트](/ko/resources/fundamentals/authentication)를 사용하세요. 이는 [개발자 앱](/ko/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow)의 [API 키와 토큰](/ko/resources/fundamentals/authentication#api-key-and-secret)과 함께, 요청을 수행하려는 대상 사용자를 위해 생성한 사용자 [액세스 토큰](/ko/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) 세트를 제공해야 한다는 뜻입니다. [3-legged OAuth 플로우](/ko/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow)를 사용하면 액세스 토큰을 발급받을 수 있습니다. 이 Authorization 헤더를 자동으로 만들어 주는 [도구나 라이브러리](/ko/x-api/tools-and-libraries/overview)를 사용하세요.

인증에 대한 자세한 내용은 [authentication](/ko/resources/fundamentals/authentication) 관련 문서를 참조하세요.

<div id="fields">
  ### 필드
</div>

각 endpoint에서 반환되는 객체는 기본적으로 축약된 형태입니다. 개발자가 API로부터 받는 응답을 더 세밀하게 맞춤 구성할 수 있도록, 원하는 필드를 요청할 때 [`fields`](/ko/x-api/fundamentals/fields) 파라미터를 사용합니다. 필드는 endpoint 전반에서 일관되게 유지됩니다. 게시물(Post) 객체는 게시물 객체가 반환되는 모든 endpoint에서 동일한 필드를 반환합니다. 동일한 필드 집합은 유사한 endpoint들에서도 쿼리할 수 있습니다.

예를 들어, 동일한 게시물 필드는 [Posts lookup](/ko/x-api/posts/lookup/introduction)과 [Users lookup](/ko/x-api/users/lookup/introduction)에서 확장된 고정 게시물(pinned Post)에 대해 쿼리할 수 있습니다.

***

<div id="expansions">
  ### Expansions
</div>

적절한 경우, [expansions](/ko/x-api/fundamentals/expansions)는 모든 중첩된 id 필드(예: `author_id`와 같이 이름이 `*_id`인 모든 필드)에 대해 제공됩니다. 또한 expansions는 현재 객체의 최상위 식별자가 아닌 id를 가지는 모든 필드에도 사용할 수 있습니다. 예를 들어, [Posts 조회](/ko/x-api/posts/lookup/introduction)에서 게시물(Post)은 필드 `id`를 최상위 식별자로 갖는 현재 객체입니다. `author_id` 또는 `referenced_tweets.id` 필드는 이러한 값을 쉼표로 구분해 `expansions` 파라미터에 추가하면, 완전한 사용자(user) 또는 Post 객체로 확장할 수 있습니다.

이러한 필드와 관련하여 발견하신 [일관성 문제를 보고해 주세요](https://twitterdevfeedback.uservoice.com/).
