Log in with X(일명 Sign in with X)를 사용하면 사이트나 애플리케이션에 버튼을 추가하여 X 사용자가 한 번의 클릭만으로 등록된 사용자 계정과 동일한 혜택을 누릴 수 있습니다. 이 기능은 웹사이트, iOS, 모바일 및 데스크톱 애플리케이션에서 모두 사용할 수 있습니다.

<div id="features">
  ## 기능
</div>

* 사용이 간편함 - 사이트를 처음 방문한 사용자는 로그인하는 데 버튼 두 개만 클릭하면 됩니다.
* X와의 통합 - Log in with X 플로우를 통해 사용자를 대신하여 X API를 사용할 수 있는 권한을 부여할 수 있습니다.
* OAuth 기반 - 다양한 클라이언트 라이브러리와 예제 코드가 Log in with X API와 호환됩니다.

<div id="available-for">
  ## 지원 환경
</div>

* 브라우저 - 사용자가 브라우저에 접속할 수 있다면 Log in with X를 연동할 수 있습니다. 브라우저 로그인 흐름에 대해 알아보세요.
* 모바일 기기 - 인터넷에 연결된 모든 모바일 기기는 Log in with X를 사용할 수 있습니다. 모바일 로그인 흐름에 대해 알아보세요.

<div id="implementing-log-in-with-x">
  ## X로 로그인 구현하기
</div>

브라우저 및 모바일 웹에서의 Log in with X 구현은 OAuth를 기반으로 합니다. 이 페이지에서는 로그인 플로우를 위해 액세스 토큰을 얻는 데 필요한 요청을 보여줍니다.

“Log in with X&quot; 플로우를 사용하려면, [X 앱 설정](/ko/resources/fundamentals/developer-apps)으로 이동해 *&quot;Allow this app to be used to Sign in with X?*” 옵션이 활성화되어 있는지 확인하세요.

이 페이지는 독자가 OAuth 1.0a 프로토콜을 사용해 요청에 서명하는 방법을 알고 있다고 가정합니다. 요청 서명 방법을 알고 싶다면 [Authorizing a request](/ko/resources/fundamentals/authentication/oauth-1-0a/authorizing-a-request) 페이지를 참조하세요.

이 페이지에 있는 요청들의 서명을 검증하고 싶다면, 사용된 consumer secret은 다음과 같습니다: L8qq9PZyRg6ieKGEKhZolGC0vJWLw8iEJ88DRdyOg. 이 값은 테스트 목적으로만 제공되며 실제 요청에는 사용할 수 없습니다.

요청 토큰을 획득하고, 사용자를 리다이렉트하며, 요청 토큰을 액세스 토큰으로 전환하는 세 단계로 Log in with X를 구현하는 방법은 아래에 나와 있습니다.

<Tabs>
  <Tab title="1단계">
    ### 1단계: request token 가져오기

    로그인 플로우를 시작하려면, [X 앱](/ko/resources/fundamentals/developer-apps)이 서명된 메시지를 [POST oauth/request&#95;token](/ko/resources/fundamentals/authentication/api-reference#post-oauth-request-token) 엔드포인트로 전송하여 request token을 가져와야 합니다. 이 요청에서 고유한 매개변수는 oauth&#95;callback 하나뿐이며, 사용자가 2단계를 완료했을 때 리디렉션되기를 원하는 URL을 URL 인코딩한 버전이어야 합니다. 나머지 매개변수는 OAuth 서명 과정에서 추가됩니다.

    <Note>
      **참고:** [POST oauth/request&#95;token](/ko/resources/fundamentals/authentication/api-reference#post-oauth-request-token) 엔드포인트에서 사용하는 모든 [callback URL](/ko/resources/fundamentals/developer-apps#callback-urls)은 [developer portal](/ko/resources/fundamentals/developer-portal)의 [X 앱 설정](/ko/resources/fundamentals/developer-apps)에 등록되어 있어야 합니다.
    </Note>

    <Frame>
      <img src="/images/auth-4.png.twimg.1920.png" alt="" />
    </Frame>

    **요청 예시 (Authorization 헤더는 줄바꿈 처리됨):**

    ```
    POST /oauth/request_token HTTP/1.1
    User-Agent: themattharris' HTTP Client
    Host: api.x.com
    Accept: */*
    Authorization:
            OAuth oauth_callback="http%3A%2F%2Flocalhost%2Fsign-in-with-twitter%2F",
                  oauth_consumer_key="cChZNFj6T5R0TigYB9yd1w",
                  oauth_nonce="ea9ec8429b68d6b77cd5600adbbb0456",
                  oauth_signature="F1Li3tvehgcraF8DMJ7OyxO4w9Y%3D",
                  oauth_signature_method="HMAC-SHA1",
                  oauth_timestamp="1318467427",
                  oauth_version="1.0"
    ```

    앱은 응답의 HTTP 상태 코드를 확인해야 합니다. 200이 아닌 값은 모두 실패를 의미합니다. 응답 본문에는 oauth&#95;token, oauth&#95;token&#95;secret, oauth&#95;callback&#95;confirmed 매개변수가 포함됩니다. 앱은 oauth&#95;callback&#95;confirmed 값이 true인지 확인하고, 나머지 두 값을 다음 단계에서 사용할 수 있도록 저장해야 합니다.

    **예시 응답(응답 본문은 줄바꿈하여 표기함):**

    ```
    HTTP/1.1 200 OK
    Date: Thu, 13 Oct 2011 00:57:06 GMT
    Status: 200 OK
    Content-Type: text/html; charset=utf-8
    Content-Length: 146
    Pragma: no-cache
    Expires: Tue, 31 Mar 1981 05:00:00 GMT
    Cache-Control: no-cache, no-store, must-revalidate, pre-check=0, post-check=0
    Vary: Accept-Encoding
    Server: tfe

    oauth_token=NPcudxy0yU5T3tBzho7iCotZ3cnetKwcTIRlX0iwRl0&
    oauth_token_secret=veNRnAWe6inFuo8o2u8SLLZLjolYDmDP7SzL0YfYI&
    oauth_callback_confirmed=true

    ```

  <Tab title="2단계">
    ### 2단계: 사용자 리디렉션

    다음 단계는 아래의 브라우저 로그인 플로우에 설명된 대로, 사용자가 적절한 플로우를 완료할 수 있도록 X로 이동시키는 것입니다. 사용자를 [GET oauth/authenticate](/ko/resources/fundamentals/authentication/api-reference#get-oauth-authenticate)로 리디렉션하고, 1단계에서 획득한 요청 토큰을 oauth&#95;token 파라미터로 전달해야 합니다.

    웹사이트에서 이를 구현하는 가장 자연스러운 방법은 원래 “로그인” 요청에 대한 응답으로 HTTP 302 리디렉트를 보내는 것입니다. 모바일 및 데스크톱 앱은 새 브라우저 창을 열거나 임베디드 웹뷰를 통해 해당 URL로 이동해야 합니다.

    **리디렉션할 URL 예시:**

    [https://api.x.com/oauth/authenticate?oauth&#95;token=NPcudxy0yU5T3tBzho7iCotZ3cnetKwcTIRlX0iwRl0](https://api.x.com/oauth/authenticate?oauth_token=NPcudxy0yU5T3tBzho7iCotZ3cnetKwcTIRlX0iwRl0)

    로그인 엔드포인트는 사용자의 상태에 따라 다음 세 가지 방식 중 하나로 동작합니다:

    1. **로그인 및 승인 완료**: 사용자가 x.com에 로그인되어 있고 이미 호출 애플리케이션을 승인한 경우, 즉시 인증이 완료되고 유효한 OAuth 요청 토큰과 함께 콜백 URL로 반환됩니다. x.com으로의 리디렉션은 사용자에게 눈에 띄지 않습니다.
    2. **로그인했지만 승인하지 않음**: 사용자가 x.com에 로그인해 있지만 호출 애플리케이션을 승인하지 않은 경우, 호출 애플리케이션과 액세스를 공유하라는 요청이 표시됩니다. 권한 부여 요청을 수락하면, 사용자는 유효한 OAuth 요청 토큰과 함께 콜백 URL로 리디렉션됩니다.
    3. **로그인하지 않음**: 사용자가 x.com에 로그인하지 않은 경우, 같은 화면에서 애플리케이션이 사용자의 정보에 액세스할 수 있도록 자격 증명을 입력하고 액세스를 허용하라는 프롬프트가 표시됩니다. 로그인에 성공하면, 사용자는 유효한 OAuth 요청 토큰과 함께 콜백 URL로 반환됩니다.

    인증이 성공하면, callback&#95;url은 oauth&#95;token 및 oauth&#95;verifier 파라미터를 포함한 요청을 받게 됩니다. 애플리케이션은 해당 토큰이 1단계에서 받은 요청 토큰과 일치하는지 확인해야 합니다.

    **클라이언트 리디렉트로부터의 요청(쿼리 스트링 파라미터를 줄바꿈으로 구분):**

    ```
    GET /sign-in-with-twitter/?
            oauth_token=NPcudxy0yU5T3tBzho7iCotZ3cnetKwcTIRlX0iwRl0&
            oauth_verifier=uw7NjWHT6OJ1MpJOXsHfNxoAhPKpgI8BlYDhxEjIBY HTTP/1.1
    Host: localhost
    User-Agent: Mozilla/5.0 (Macintosh; Intel Mac OS X 10_6_8) AppleWebKit/535.5 (KHTML, like Gecko) Chrome/16.0.891.1 Safari/535.5
    Accept: text/html,application/xhtml+xml,application/xml;q=0.9,*/*;q=0.8
    Referer: http://localhost/sign-in-with-twitter/
    Accept-Encoding: gzip,deflate,sdch
    Accept-Language: en-US,en;q=0.8
    Accept-Charset: ISO-8859-1,utf-8;q=0.7,*;q=0.3
    ```

  <Tab title="3단계">
    ### 3단계: 요청 토큰을 액세스 토큰으로 변환하기

    요청 토큰을 사용 가능한 액세스 토큰으로 변환하려면, 앱에서 2단계에서 얻은 oauth&#95;verifier 값을 포함해 [POST oauth/access&#95;token](/ko/resources/fundamentals/authentication/api-reference#post-oauth-access-token) 엔드포인트로 요청을 보내야 합니다. 요청 토큰은 헤더의 oauth&#95;token 부분에도 포함되지만, 이는 서명 과정에서 이미 추가되어 있습니다.

    **요청 예시 (Authorization 헤더 줄바꿈 표시):**

    ```
    POST /oauth/access_token HTTP/1.1
    User-Agent: themattharris' HTTP Client
    Host: api.x.com
    Accept: */*
    Authorization: OAuth oauth_consumer_key="cChZNFj6T5R0TigYB9yd1w",
                          oauth_nonce="a9900fe68e2573b27a37f10fbad6a755",
                          oauth_signature="39cipBtIOHEEnybAR4sATQTpl2I%3D",
                          oauth_signature_method="HMAC-SHA1",
                          oauth_timestamp="1318467427",
                          oauth_token="NPcudxy0yU5T3tBzho7iCotZ3cnetKwcTIRlX0iwRl0",
                          oauth_version="1.0"
    Content-Length: 57
    Content-Type: application/x-www-form-urlencoded

    oauth_verifier=uw7NjWHT6OJ1MpJOXsHfNxoAhPKpgI8BlYDhxEjIBY
    ```

    성공한 응답에는 oauth&#95;token, oauth&#95;token&#95;secret 매개변수가 포함됩니다. 토큰과 토큰 시크릿은 저장해 두었다가 이후 X API에 인증된 요청을 보낼 때 사용해야 합니다. 사용자의 신원을 확인하려면 [GET account/verify&#95;credentials](https://dev.x.com/rest/reference/get/account/verify_credentials)를 사용하세요.

    **예시 응답(응답 본문은 줄바꿈 처리됨):**

    ```
    HTTP/1.1 200 OK
    Date: Thu, 13 Oct 2011 00:57:08 GMT
    Status: 200 OK
    Content-Type: text/html; charset=utf-8
    Content-Length: 157
    Pragma: no-cache
    Expires: Tue, 31 Mar 1981 05:00:00 GMT
    Cache-Control: no-cache, no-store, must-revalidate, pre-check=0, post-check=0
    Vary: Accept-Encoding
    Server: tfe

    oauth_token=7588892-kagSNqWge8gB1WwE3plnFsJHAZVfxWD7Vb57p0b4&
    oauth_token_secret=PbKfYqSryyeKDWz4ebtY3o5ogNLG11WJuZBc9fQrQo
    ```

    ### Log in with X 리소스

    #### 클라이언트 라이브러리

    [X libraries](/ko/resources/tools-and-libraries)에 나열된 클라이언트 라이브러리는 Log in with X를 구현하는 데 도움이 됩니다. 앞선 단계에서 설명한 대로 `/oauth/authenticate` 엔드포인트를 사용하세요.

    #### 브랜드 툴킷

    일관된 브랜딩을 위해 X는 애플리케이션에서 공식 [X Brand Toolkit](https://about.x.com/en/who-we-are/brand-toolkit)을 사용할 것을 권장합니다. 이 애셋들을 저장해 두었다가 ‘Log in with X’ 버튼을 만들 때 사용하세요.
  </Tab>

  <Tab title="브라우저 로그인 플로우">
    브라우저 로그인 플로우는 웹 브라우저를 열거나 임베드할 수 있는 웹사이트와 애플리케이션에 적합합니다. 개략적인 흐름은 다음과 같습니다.

    * 애플리케이션이 “Sign in with X” 링크 또는 버튼을 렌더링합니다.
    * 사용자가 로그인 버튼을 클릭합니다.
    * 현재 웹 브라우저가 X로 리다이렉트되거나, 새 브라우저가 열려 X로 이동합니다.
    * 필요할 경우, 사용자가 X에서 로그인 및 권한 부여 단계를 완료합니다.
    * X는 애플리케이션이 제어하는 URL로 다시 리다이렉트하면서, 사용자에 대한 인가 정보를 전달합니다.

    X는 권한 부여 상태를 추적하므로, 이미 X.com에 로그인했고 애플리케이션을 승인한 사용자의 경우 UI는 표시되지 않고, 대신 자동으로 애플리케이션으로 리다이렉트됩니다.
