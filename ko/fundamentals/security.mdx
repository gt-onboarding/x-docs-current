우리는 개인정보 보호는 특권이 아니라 권리이며, 우리 회사의 토대에 내재되어 있다고 믿습니다. X 개발자 플랫폼을 사용하고 개발자 정책을 준수함으로써, 여러분은 X에서 플랫폼이 공적 대화를 지원하고 우리의 개인정보 보호에 대한 약속을 지킬 수 있도록 하는 데 중요한 역할을 합니다.

여러분 자신의 데이터와 앱 사용자들의 데이터를 모두 보호하기 위해서는 보안을 염두에 두고 개발하는 일이 중요하다는 점을 상기하고자 합니다. 보안 침해 위협으로부터 보호하는 것은 여러분의 책임이며, X를 사용하는 사람들을 보호하는 것은 우리 모두가 공유하는 책임입니다. 이 페이지에서는 보안성이 뛰어난 애플리케이션을 구축하고 데이터와 액세스를 가능한 한 안전하게 유지하는 것과 관련된 기대 사항을 설명합니다.

<Note>X Developer Platform에서 사용할 수 있는 [인증](/ko/resources/fundamentals/authentication), TLS, [앱 권한](/ko/resources/fundamentals/developer-apps#app-permissions) 등의 보안 기술과, X 사용자의 관점에서 [서드파티 애플리케이션 및 세션 사용](https://help.x.com/en/managing-your-account/connect-or-revoke-access-to-third-party-apps)에 관한 내용을 반드시 숙지하세요.</Note>

<div id="reporting-security-issues">
  ## 보안 문제 신고
</div>

X Developer Platform 사용자는 보안 사고가 발생했다고 최초로 의심한 시점으로부터 48시간 이내에 [X의 취약점 신고 프로그램](https://hackerone.com/twitter)을 통해 X에 이를 통지해야 합니다.

<div id="security-best-practices">
  ## 보안 모범 사례
</div>

X 개발자 플랫폼에서 서비스를 구축할 때뿐만 아니라 인터넷 전반에서 항상 이 보안 모범 사례를 염두에 두고 적용해 주시기 바랍니다.

<div id="security-by-design">
  ### 설계 단계부터의 보안
</div>

위협 모델 평가 및/또는 침투 테스트를 수행하기 위해 보안 전문가를 고용하는 방안을 고려하십시오. 우수한 보안 업체는 문제를 찾아내기 위해 깊이 있게 분석합니다. X가 이러한 사고방식을 어떻게 받아들였는지는 [이 블로그 게시물](https://blog.x.com/en_us/topics/company/2019/privacy_data_protection.html)에서 자세히 확인할 수 있습니다.

또한 X는 모든 파트너가 다음 사항에 책임을 지도록 요구합니다.

* 코드를 보안이 보장된 저장소에서 관리할 것.
* 시스템 개발 생명주기(SDLC) 전체에 걸쳐 위험 분석을 수행할 것.
* SDLC 전체에 걸쳐 보안 문제가 식별되고 완화되도록 보장할 것.
* SDLC 프로세스 전반에 걸쳐 환경이 분리되도록 보장할 것.
* 모든 테스트 결함이 수정되고, 재검증되며, 최종적으로 종료되도록 보장할 것.

<div id="monitor-and-get-alerted">
  ### 모니터링 및 알림 설정
</div>

웹 앱에 문제가 있다고 의심된다면, 어떻게 확실히 알 수 있을까요? 꼼꼼하게 로그를 남기고, 심각한 예외와 오류에 대해 알림을 받도록 설정해야 합니다. 한눈에 이상 여부를 확인할 수 있도록, 핵심 지표로 구성된 대시보드를 마련하는 것도 좋습니다.

<div id="create-a-reporting-channel">
  ### 신고 채널 만들기
</div>

사용자가 앱에서 겪은 잠재적인 보안 문제를 쉽게 신고하고 연락할 수 있도록 하세요. X 사용자와 데이터에 영향을 미치는 문제가 발견된 경우, [해당 문제를 X에 보고](#report)하는 것도 여러분의 책임입니다. 보안 사고가 발생했을 때 영향을 받는 사용자에게 알리기 위한 대응 계획과 절차를 미리 준비해 두세요.

<div id="adequate-testing">
  ### 충분한 테스트
</div>

엔드투엔드 테스트가 충분히 철저하며, 무단 접근과 같은 보안 시나리오에서 예상되는 실패 사례를 포함하도록 최신 상태로 유지되었는지 확인하세요. 공격자의 관점에서 생각하고, 공격자가 X 데이터나 권한이 부여된 기능에 무단으로 접근하려 할 때 이를 차단하도록 설계된 시스템 테스트를 구성하세요.

<div id="securing-api-keys-and-tokens">
  ### API 키와 토큰 보호
</div>

X 플랫폼의 개발자인 경우(해당 사용자가 개발자 앱을 승인했다는 전제하에) X에 저장된 본인의 데이터와 사용자 데이터 모두에 프로그램을 통해 접근할 수 있습니다. 모든 API 요청은 개발자 앱의 key와 secret, 그리고 경우에 따라 권한을 부여한 사용자의 [액세스 토큰](/ko/resources/fundamentals/authentication)을 사용해 OAuth로 [인증](/ko/resources/fundamentals/authentication)해야 합니다. 자격 증명을 안전하게 보호하는 것은 전적으로 개발자의 책임입니다.

권장 모범 사례는 다음과 같습니다:

* 비밀번호/토큰 갱신 로테이션을 구성합니다.
* 필요한 경우에는 항상 중요 데이터를 암호화하고, 데이터는 가능한 한 뒤 단계에서 복호화합니다.
* 사용자의 액세스 토큰을 암호화된 저장소에 보관합니다.
* 키와 토큰이 유출되었다고 판단되는 경우 [재생성](/ko/resources/fundamentals/authentication#regenerate-api-keys-and-tokens)하거나 [무효화](/ko/resources/fundamentals/authentication#post-oauth2-invalidate-token)합니다.

X에서 OAuth를 사용해 디버깅하고 애플리케이션을 구축하는 방법에 대한 더 많은 논의는 커뮤니티 포럼의 [보안(Security) 카테고리](https://devcommunity.x.com/c/oauth/12)를 방문해 확인하십시오.

<div id="input-validation">
  ### 입력 검증
</div>

사용자가 항상 유효하고 신뢰할 수 있는 데이터를 제공한다고 가정하지 마십시오. X API 요청에 포함될 수 있는 사용자 입력 데이터는 모두 정제(sanitize)하십시오. 애플리케이션에서 허용하는 입력 유형만 허용 목록(allowlist)으로 정의하고, 이 목록에 없는 것은 모두 폐기하십시오.

<div id="encrypted-communication">
  ### 암호화된 통신
</div>

X는 모든 API 요청이 TLS를 통해 이루어지도록 요구합니다. 가능한 모든 경우에, 사용자 소유 서버로의 통신도 암호화해야 합니다.

<div id="exposed-debugging-information">
  ### 노출된 디버깅 정보
</div>

디버깅 화면/로그를 통해 민감한 X 데이터나 자격 증명이 노출되지 않도록 하십시오. 일부 웹 프레임워크는 애플리케이션이 올바르게 구성되지 않은 경우 디버깅 정보에 쉽게 접근할 수 있도록 합니다. 데스크톱 및 모바일 앱을 개발하는 경우, 디버깅 플래그나 심볼이 활성화된 빌드를 실수로 배포하기 쉽습니다. 이러한 설정을 점검하는 단계를 배포/빌드 프로세스에 포함하십시오. 또한, 보고를 위해 스택 트레이스나 크래시 덤프를 공유하는 경우, 비공개 X 사용자의 데이터가 반드시 삭제되거나 비식별 처리되었는지 확인하십시오.

<div id="unfiltered-input-unescaped-output">
  ### 필터링되지 않은 입력, 이스케이프되지 않은 출력
</div>

입력 검증에 대해 기억하기 쉬운 한 가지 접근 방식은 FIEO(Filter Input, Escape Output)입니다.

애플리케이션 외부에서 들어오는 모든 것을 필터링하세요. 여기에는 X API 데이터, 쿠키 데이터, 사용자가 입력한 폼 데이터, URL 매개변수, 데이터베이스에서 가져오는 데이터 등이 포함됩니다. 애플리케이션에서 내보내는 모든 출력을 이스케이프 처리하세요. 여기에는 데이터베이스 서버로 전송하는 SQL, 사용자의 브라우저로 보내는 HTML, 다른 시스템으로 보내는 JSON 출력, 셸 프로그램으로 보내는 명령 등이 포함됩니다.

<div id="cross-site-scripting-xss">
  ### 크로스 사이트 스크립팅 (XSS)
</div>

[XSS](https://owasp.org/) 공격은 전반적으로 웹에서 가장 흔하게 발생하는 보안 문제 유형입니다. 공격자가 자신의 JavaScript 코드를 애플리케이션에 주입할 수 있다면, 다양한 악의적인 행위를 수행할 수 있습니다. 신뢰할 수 없는 사용자가 제공한 데이터를 저장하거나 화면에 표시하는 모든 위치에서는 반드시 데이터를 검사하고 정제(sanitize)한 뒤, HTML 이스케이프 처리를 해야 합니다. 이를 올바르게 구현하는 것은 쉽지 않습니다. 해커들이 [XSS 공격을 수행하는 방법이 매우 다양하기 때문](https://www.owasp.org/index.php/Types_of_Cross-Site_Scripting)입니다. 사용하는 언어나 웹 개발 프레임워크에는 크로스 사이트 스크립팅을 방어하기 위한 널리 사용되고 검증된 메커니즘이 있을 가능성이 높으므로, 반드시 이를 활용해야 합니다.

<div id="sql-injection">
  ### SQL injection
</div>

애플리케이션에서 데이터베이스를 사용한다면 [SQL injection](http://en.wikipedia.org/wiki/SQL_injection)에 대해 알고 있어야 합니다. 입력을 받는 모든 지점은 공격자가 자신의 입력 영역을 벗어나 쿼리를 조작하고 데이터베이스에 침투하려고 시도할 수 있는 잠재적인 표적입니다. SQL injection을 체계적으로 방어하도록 설계된 데이터베이스 라이브러리를 사용하십시오. 이 방식을 벗어나 직접 SQL을 작성해야 한다면, 이러한 형태의 공격에 노출되지 않았는지 확인할 수 있도록 철저한 테스트를 작성해야 합니다. SQL injection에 대응하는 두 가지 주요 방법은 SQL 문을 구성하기 전에 이스케이프 처리를 수행하는 것과, 파라미터화된 입력을 사용해 문을 생성하는 것입니다. 후자가 프로그래머의 실수에 덜 취약하므로 더 권장되는 방식입니다.

<div id="cross-site-request-forgery-csrf">
  ### 크로스 사이트 요청 위조(CSRF)
</div>

애플리케이션으로 들어오는 요청이 정말로 애플리케이션에서 발생한 것이라고 확신할 수 있습니까? [CSRF](http://en.wikipedia.org/wiki/Cross-site_request_forgery) 공격은, 사이트에 로그인해 있는 사용자가 특정 동작을 수행하는 URL을 사용자 모르게 열도록 강제함으로써 이러한 불확실성을 악용합니다. 개발자 앱의 경우, 이는 공격자가 앱을 이용해 사용자가 원치 않는 트윗을 게시하게 하거나 스팸 계정을 팔로우하도록 강제하는 것을 의미할 수 있습니다.

CSRF를 가장 철저하게 방지하는 방법은, 신뢰할 수 있는 곳에 저장된 무작위 토큰을 모든 폼에 포함하는 것입니다. 폼에 올바른 토큰이 없으면 오류를 발생시키면 됩니다. 최신 웹 프레임워크는 이를 체계적으로 처리하는 방법을 제공하며, 운이 좋다면 기본값으로 이미 처리하고 있을 수도 있습니다. 단순한 예방 단계(그러나 결코 이것만으로 충분하다는 뜻은 아님)로는, 데이터 생성, 수정, 삭제 작업이 반드시 POST 요청을 요구하도록 만드는 것이 있습니다.

<div id="lack-of-rate-limiting">
  ### 요청 속도 제한 미적용
</div>

잠재적인 스패머와 공격자의 활동을 지연시키기 위해, 적절한 경우 CAPTCHA를 사용하세요.
