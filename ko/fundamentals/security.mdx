우리는 프라이버시가 특권이 아닌 권리이며, 우리 회사 운영의 근간에 자리 잡고 있다고 믿습니다. X 개발자 플랫폼을 사용하고 개발자 정책을 준수함으로써, 여러분은 X에서 공적 대화를 가능하게 하고 우리의 프라이버시 보호 약속을 지키는 데 중요한 역할을 수행하게 됩니다.

여러분이 자신의 데이터와 여러분의 앱 사용자 데이터 모두를 보호하기 위해 보안을 염두에 두고 개발하는 것이 얼마나 중요한지 다시 한번 상기하고자 합니다. 보안 침해 위협에 대비하는 것은 여러분의 책임이며, X를 이용하는 사람들을 보호하는 것은 우리가 함께 져야 할 공동 책임입니다. 이 페이지에서는 보안성이 높은 애플리케이션을 구축하고 데이터와 접근 권한을 가능한 한 안전하게 유지하는 데 대한 기대 사항을 설명합니다.

<Note>X Developer Platform에서 사용할 수 있는 [인증](/ko/resources/fundamentals/authentication), TLS, [앱 권한](/ko/resources/fundamentals/developer-apps#app-permissions) 등의 보안 기술과, X 사용자 관점에서의 [서드파티 애플리케이션 및 세션 사용](https://help.x.com/en/managing-your-account/connect-or-revoke-access-to-third-party-apps)에 관한 내용을 반드시 숙지해 주시기 바랍니다.</Note>

  ## 보안 문제 보고
</div>

X Developer Platform 사용자는 보안 사고 발생이 의심되는 최초 시점으로부터 48시간 이내에 [X 취약점 신고 프로그램](https://hackerone.com/twitter)을 통해 X에 이를 신고해야 합니다.

<div id="security-best-practices">
  ## 보안 모범 사례
</div>

X 개발자 플랫폼과 인터넷 전반에서 무언가를 구축할 때에는, 다음 사항을 항상 염두에 두시기 바랍니다.

<div id="security-by-design">
  ### 설계 단계부터의 보안
</div>

위협 모델링 평가 및/또는 침투 테스트를 수행할 보안 전문가를 고용하는 것을 고려하십시오. 훌륭한 보안 전문 업체는 깊이 있게 분석해 문제를 찾아냅니다. X가 이 사고방식을 어떻게 도입했는지에 대해서는 [블로그 게시물](https://blog.x.com/en_us/topics/company/2019/privacy_data_protection.html)을 참고하십시오.

또한 X는 모든 파트너에게 다음 사항을 요구합니다.

* 코드를 안전한 리포지토리에서 유지·관리해야 합니다.
* 시스템 개발 생명주기(SDLC) 전반에 걸쳐 위험 분석을 수행해야 합니다.
* SDLC 전 과정에서 보안 문제가 식별되고 완화되도록 보장해야 합니다.
* SDLC 과정 전반에 걸쳐 환경이 분리되도록 보장해야 합니다.
* 모든 테스트 결함을 수정하고, 재테스트하여, 완전히 종료되도록 보장해야 합니다.

<div id="monitor-and-get-alerted">
  ### 모니터링 및 알림 설정
</div>

웹 앱에 문제가 있다고 생각될 때 어떻게 확실히 알 수 있을까요? 충분한 로그를 수집하고, 중요한 예외와 오류에 대한 알림을 받도록 설정해야 합니다. 문제가 발생하고 있는지를 한눈에 파악할 수 있도록 주요 지표로 구성된 대시보드를 구축하는 것이 좋습니다.

<div id="create-a-reporting-channel">
  ### 신고 채널 만들기
</div>

사용자가 앱에서 겪은 잠재적인 보안 문제를 여러분에게 쉽게 신고할 수 있도록 하세요. X 사용자와 데이터에 영향을 미치는 문제가 발견된 경우, [해당 문제를 X에 신고](#report)하는 것 또한 여러분의 책임입니다. 보안 사고가 발생했을 때 영향을 받는 사용자에게 통지하기 위한 대응 계획/프로세스를 미리 준비해 두세요.

<div id="adequate-testing">
  ### 충분한 테스트
</div>

엔드투엔드 테스트가 충분히 철저하고, 무단 액세스와 같은 보안 시나리오에서 예상되는 실패 사례를 포함하도록 최신 상태로 유지해야 합니다. 공격자의 관점에서 생각해 보고, 공격자가 X 데이터나 권한이 부여된 기능에 무단으로 접근하려 할 때 이를 차단하는 것이 기대되는 시스템 테스트를 구성하십시오.

<div id="securing-api-keys-and-tokens">
  ### API 키와 토큰 보안
</div>

X 플랫폼의 개발자로서, (사용자가 개발자 앱을 승인했다는 전제하에) X가 보관하고 있는 본인의 데이터와 사용자의 데이터에 프로그램적으로 접근할 수 있습니다. 모든 API 요청은 개발자 앱의 키와 시크릿, 그리고 일부 경우에는 권한을 부여한 사용자의 [액세스 토큰](/ko/resources/fundamentals/authentication)을 사용해 OAuth로 [인증](/ko/resources/fundamentals/authentication)해야 합니다. 자격 증명을 안전하게 보호하는 것은 전적으로 귀하의 책임입니다.

다음은 권장되는 모범 사례입니다.

* 비밀번호/토큰 갱신 로테이션을 구성하십시오.
* 필요한 경우 항상 민감한 데이터를 암호화하고, 지나치게 이른 단계에서 데이터를 복호화하지 마십시오.
* 사용자의 액세스 토큰을 암호화된 저장소에 보관하십시오.
* 키와 토큰이 손상되었다고 판단되는 경우, [재생성](/ko/resources/fundamentals/authentication#regenerate-api-keys-and-tokens)하거나 [무효화](/ko/resources/fundamentals/authentication#post-oauth2-invalidate-token)하십시오.

X용 OAuth 디버깅 및 개발에 대한 더 자세한 논의는 커뮤니티 포럼의 [보안 카테고리](https://devcommunity.x.com/c/oauth/12)를 방문하십시오.

<div id="input-validation">
  ### 입력 검증
</div>

사용자가 항상 유효하고 신뢰할 수 있는 데이터를 제공한다고 가정하지 마세요. X API 요청에 사용될 수 있는 사용자 입력 데이터는 모두 정제(sanitize)하세요. 애플리케이션에서 허용할 입력 유형만 허용 목록(allowlist)에 포함하고, 허용 목록에 없는 것은 모두 버리세요.

<div id="encrypted-communication">
  ### 암호화된 통신
</div>

X는 모든 API 요청이 TLS를 사용해 전송되도록 요구합니다. 가능하면 자체 서버로의 통신도 암호화해야 합니다.

<div id="exposed-debugging-information">
  ### 노출된 디버깅 정보
</div>

디버깅 화면/로그를 통해 민감한 X 데이터나 자격 증명이 노출되지 않도록 하십시오. 일부 웹 프레임워크는 애플리케이션이 올바르게 설정되지 않은 경우 디버깅 정보에 쉽게 접근할 수 있도록 합니다. 데스크톱 및 모바일 개발의 경우, 디버깅 플래그나 심볼이 활성화된 상태로 빌드를 실수로 배포하기 쉽습니다. 이러한 설정에 대한 검사를 배포/빌드 프로세스에 포함하십시오. 또한, 보고 목적으로 스택 트레이스나 크래시 덤프를 공유할 때는 비공개 X 사용자의 데이터가 편집(삭제)되어 있는지 확인하십시오.

<div id="unfiltered-input-unescaped-output">
  ### 필터링되지 않은 입력, 이스케이프되지 않은 출력
</div>

입력 검증을 쉽게 기억하는 방법으로 FIEO라는 원칙이 있습니다: Filter Input, Escape Output

X API 데이터, 쿠키 데이터, 사용자가 입력한 폼 값, URL 매개변수, 데이터베이스에서 가져온 데이터 등 애플리케이션 외부에서 들어오는 모든 것은 모두 필터링하세요. 애플리케이션이 내보내는 모든 출력을 이스케이프하세요. 여기에는 데이터베이스 서버로 전송되는 SQL, 사용자 브라우저로 보내는 HTML, 다른 시스템으로 전송하는 JSON 출력, 셸 프로그램으로 전송하는 명령 등이 포함됩니다.

<div id="cross-site-scripting-xss">
  ### 크로스사이트 스크립팅 (XSS)
</div>

[XSS](https://owasp.org/) 공격은 웹 보안 문제 가운데 가장 흔한 유형입니다. 공격자가 자신의 JavaScript 코드를 애플리케이션에 주입할 수 있다면, 악의적인 행위를 수행할 수 있습니다. 사용자가 제공한 신뢰할 수 없는 데이터를 저장하고 표시하는 모든 부분은 반드시 검사하고 정제(sanitize)하며 HTML 이스케이프 처리를 해야 합니다. 이를 제대로 구현하는 것은 어렵습니다. 해커들이 [XSS 공격을 시도할 수 있는 방법](https://www.owasp.org/index.php/Types_of_Cross-Site_Scripting)이 매우 다양하기 때문입니다. 사용하는 프로그래밍 언어나 웹 개발 프레임워크에는 대개 크로스사이트 스크립팅을 방어하기 위한 널리 사용되고 잘 검증된 메커니즘이 있으므로 반드시 이를 활용해야 합니다.

<div id="sql-injection">
  ### SQL 인젝션
</div>

애플리케이션에서 데이터베이스를 사용하는 경우 [SQL injection](http://en.wikipedia.org/wiki/SQL_injection)에 대해 알고 있어야 합니다. 입력을 받는 모든 지점은 공격자가 입력 필드 범위를 벗어나 데이터베이스에 침투하려고 시도할 수 있는 잠재적인 대상입니다. SQL 인젝션으로부터 체계적으로 보호해 주는 데이터베이스 라이브러리를 사용하십시오. 이러한 라이브러리의 보호 범위를 벗어나 직접 SQL을 작성하는 경우, 이 형태의 공격에 노출되지 않았는지 확인할 수 있도록 철저한 테스트를 작성해야 합니다. SQL 인젝션에 대응하는 두 가지 주요 방법은 SQL 문을 구성하기 전에 이스케이프 처리를 하는 것과, 매개변수화된 입력(파라미터 바인딩)을 사용해 문을 생성하는 것입니다. 후자의 방법이 프로그래머 실수가 발생할 가능성이 더 적기 때문에 권장됩니다.

<div id="cross-site-request-forgery-csrf">
  ### 교차 사이트 요청 위조(CSRF)
</div>

애플리케이션으로 들어오는 요청이 정말로 애플리케이션에서 시작된 것이라고 확신할 수 있나요? [CSRF](http://en.wikipedia.org/wiki/Cross-site_request_forgery) 공격은 로그인한 사용자가 눈치채지 못하는 사이에, 특정 동작을 수행하는 URL을 열도록 강제함으로써 이러한 불확실성을 악용합니다. 개발자 앱의 경우, 이는 공격자가 앱을 이용해 사용자가 원치 않는 트윗을 게시하게 하거나 스팸 계정을 팔로우하도록 강제하고 있을 수 있음을 의미합니다.

CSRF를 가장 철저하게 방지하는 방법은 모든 폼에 무작위 토큰을 포함하고, 이 토큰을 신뢰할 수 있는 위치에 저장하는 것입니다. 폼에 올바른 토큰이 없으면 오류를 발생시키면 됩니다. 최신 웹 프레임워크는 이를 체계적으로 처리하는 기능을 제공하며, 운이 좋다면 기본값으로 이미 활성화되어 있을 수도 있습니다. 간단한 예방 조치(그러나 결코 취해야 할 유일한 조치는 아님)는 데이터 생성, 수정, 삭제 작업이 모두 POST 요청을 필요로 하도록 만드는 것입니다.

<div id="lack-of-rate-limiting">
  ### 요청 속도 제한 미적용
</div>

잠재적인 스패머와 공격자의 속도를 늦추기 위해, 필요한 경우 CAPTCHA를 사용하세요.
