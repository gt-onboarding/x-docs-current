<div id="overview">
  ## 개요
</div>

기존에 생성해 둔 앱이 있다면, [developer portal의 App 페이지](https://developer.x.com/en/portal/projects-and-apps)에서 해당 앱을 조회, 수정 또는 삭제할 수 있습니다.

X API 및 X Ads API를 사용하려면 각 요청에 함께 전달해야 하는 [인증](/ko/resources/fundamentals/authentication) 자격 증명(키와 토큰)이 필요합니다. 이 자격 증명은 사용 중인 특정 엔드포인트에서 요구하는 인증 유형에 따라 서로 다른 형태일 수 있습니다.

앱에서 생성할 수 있는 다양한 자격 증명과 그 사용 방법은 다음과 같습니다.

* **[API Key and Secret](/ko/resources/fundamentals/authentication#api-key-and-secret):** 기본적으로 앱의 사용자 이름과 비밀번호에 해당합니다. 이는 [OAuth 1.0a 사용자 컨텍스트](/ko/resources/fundamentals/authentication#oauth-1-0a-2)가 필요한 요청을 인증하거나, 사용자 액세스 토큰 또는 App Access Token과 같은 다른 토큰을 생성하는 데 사용합니다.

* **[Access Token and Secret](/ko/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow):** 일반적으로 액세스 토큰은 요청을 대신 수행하는 사용자를 나타냅니다. developer portal을 통해 생성되는 액세스 토큰은 앱을 소유한 사용자(계정)를 나타냅니다. 이는 [OAuth 1.0a 사용자 컨텍스트](/ko/resources/fundamentals/authentication#oauth-1-0a-2)가 필요한 요청을 인증하는 데 사용됩니다. 다른 사용자를 대신해 요청을 보내려면, 해당 사용자가 3-legged OAuth 플로우를 통해 귀하의 앱을 승인하도록 해야 합니다.

* **[Client ID and Client Secret](/ko/resources/fundamentals/authentication#oauth-2-0):** 이 자격 증명은 OAuth 2.0 인증을 사용해 사용자 액세스 토큰을 얻는 데 사용됩니다. OAuth 1.0a와 마찬가지로, 사용자 액세스 토큰은 비공개 사용자 계정 정보를 제공하거나 다른 계정을 대신해 작업을 수행하는 요청을 인증하는 데 사용되지만, 클라이언트 애플리케이션이 사용자에 대해 어떤 범위로 액세스할 수 있는지 더 세밀하게 제어할 수 있는 정교한 범위(scope)를 제공합니다.

* **[App only Access Token](/ko/resources/fundamentals/authentication#bearer-token-also-known-as-app-only):** X에서 공개적으로 제공되는 정보로 응답하는 엔드포인트에 요청을 보낼 때 이 토큰을 사용합니다.

X API 요청에 필요한 키와 토큰을 생성하는 것 외에도, [권한](/ko/resources/fundamentals/developer-apps#app-permissions)을 설정하고, 앱의 사용 사례 또는 목적을 문서화하며, [callback URL](/ko/resources/fundamentals/developer-apps#callback-urls)을 정의하고, [management dashboard](https://developer.x.com/en/portal/projects-and-apps) 내에서 앱 개발 환경과 관련된 기타 설정을 수정할 수 있습니다.

<div id="apps-and-projects">
  ### 앱과 프로젝트
</div>

사용 사례별로 X Developer Platform에서의 작업을 더 잘 구성하기 위해 앱과 [프로젝트](/ko/resources/fundamentals/projects)를 사용할 수 있습니다. 각 프로젝트에는 Free X API 플랜에서는 앱 1개, Basic 플랜에서는 최대 앱 2개, Pro 플랜에서는 최대 앱 3개를 포함할 수 있습니다.

새로운 [X API v2](/ko/x-api/introduction) 엔드포인트에 액세스하려면 프로젝트에 연결된 앱의 키와 토큰을 반드시 사용해야 합니다.

프로젝트 기능을 출시하기 전에 생성한 앱이 있는 경우 해당 앱은 &quot;Standalone Apps&quot;라는 섹션에 표시됩니다. Standalone Apps는 프로젝트 구조 밖에 있는 앱을 의미합니다. Standalone App을 프로젝트에 연결하면, 해당 앱은 v2 엔드포인트로 요청을 보낼 수 있게 됩니다.

<div id="developer-portal-dashboard">
  ### Developer portal 대시보드
</div>

[대시보드](https://developer.x.com/content/developer-twitter/en/apps)에 방문하여 계정과 연결된 앱을 관리할 수 있습니다. 더 자세한 내용은 [developer portal](/ko/resources/fundamentals/developer-portal)에 대한 문서 페이지를 참고하세요. 대시보드를 사용하면 개발자는 다음 작업을 빠르고 쉽게 수행할 수 있습니다.

* 기존 Standalone 앱과 각 앱의 App ID를 확인합니다.
* 새 프로젝트, 앱 또는 Standalone 앱을 생성합니다.
* 사용하지 않는 프로젝트나 앱을 삭제합니다.
* 이름, 설명, 웹사이트, callback URL 및 [permissions](/ko/resources/fundamentals/developer-apps#app-permissions) 등을 포함한 특정 앱의 설정을 검토하거나 업데이트합니다.
* API Key &amp; Secret, App 액세스 토큰, 앱 소유자의 사용자 액세스 토큰과 같은 앱별 자격 증명을 다시 생성합니다.

<div id="signing-up-for-access">
  ### 액세스 신청하기
</div>

기존 X 앱이 있는 경우 developer.x.com에서 X 계정으로 로그인되어 있다면 X [App dashboard](https://developer.x.com/content/developer-twitter/en/portal/projects-and-apps)에서 앱을 확인하고 수정할 수 있습니다. 이전에 developer.x.com에서 생성된 모든 앱을 관리하기 위해 새 계정을 만들 필요는 **없다는** 점에 유의하세요.

새로운 X 앱을 생성해야 하는 경우 [개발자 계정에 가입](https://developer.x.com/en/portal/petition/essential/basic-info)되어 있어야 합니다. [team account](/ko/resources/fundamentals/developer-portal#team-management)의 구성원이라면, 새 App을 만들 수 있도록 admin 역할이 할당되어 있어야 합니다.

<div id="automated-account-labeling-for-bots">
  ### 봇을 위한 자동화 계정 레이블
</div>

X 사용자들이 봇이 자동화된 계정임을 알 수 있도록, 봇 계정에 Automated Account 레이블을 추가할 수 있습니다. 이 봇들은 X API를 통해 미리 프로그래밍된 동작을 수행합니다. 봇에 Automated Account 레이블을 추가하면, 오디언스와의 신뢰를 쌓고 계정의 신뢰성을 높이며 스팸성 봇과 차별화할 수 있습니다. 이를 통해 X 이용자들이 봇과 상호작용할 때 계정의 목적을 더 잘 이해하는 데 도움이 됩니다.

봇 계정에 레이블을 연결하려면 다음 단계를 따르세요:

1. 계정 설정으로 이동합니다.
2. &quot;Your account&quot;를 선택합니다.
3. &quot;Automation&quot;을 선택합니다.
4. &quot;Managing account&quot;를 선택합니다.
5. 다음으로, 봇 계정을 운영하는 X 계정을 선택합니다.
6. 비밀번호를 입력해 로그인합니다.
7. 계정에 레이블이 적용되었다는 확인 메시지가 표시되는 것을 확인합니다.

<div id="app-management">
  ## 앱 관리
</div>

<div id="introduction">
  ### 소개
</div>

[X App 대시보드](https://developer.x.com/en/portal/projects-and-apps)를 사용하면 개발자가 다음 작업을 빠르고 쉽게 수행할 수 있습니다.

* 기존 App과 [프로젝트](/ko/resources/fundamentals/projects), 그리고 이들과 연결된 App ID를 확인합니다.
* 새 프로젝트 또는 단독 App을 생성합니다.
* 프로젝트, App 또는 단독 App을 삭제합니다.
* 특정 App을 클릭해 해당 App의 설정 페이지를 엽니다. 설정에서 App 세부 정보, 키와 토큰, 권한을 확인할 수 있습니다.
* App의 사용자 인증 설정을 OAuth 1.0a 또는 OAuth 2.0을 사용하도록 업데이트합니다.

<Note>
  **참고:**

  모든 App 키와 토큰은 더 이상 Developer Portal 내에서 확인할 수 없으며, 생성된 후에는 반드시 안전하게 저장해야 합니다. 키와 토큰은 비밀번호 관리자를 사용해 보관할 것을 권장합니다.

  API 키 힌트를 표시하여 자격 증명 정보를 해당 App과 쉽게 매칭할 수 있습니다.
</Note>

<div id="app-settings">
  ### 앱 설정
</div>

<div id="app-details">
  #### 앱 세부 정보
</div>

여기에서 앱 아이콘, 앱 이름, 앱 설명, 웹사이트 URL, [callback URLs/redirect URIs](/ko/resources/fundamentals/developer-apps#callback-urls), 서비스 약관 URL, 개인정보 처리방침 URL, 조직 이름, 조직 URL, 그리고 앱의 목적 또는 사용 사례를 수정할 수 있습니다.

OAuth 2.0 및 OAuth 1.0a는 사용자가 X를 통해 여러분의 앱에 로그인할 수 있도록 해주는 인증 방식입니다. 또한 인증된 사용자를 대신해 앱이 특정 요청을 수행할 수 있도록 허용합니다. 앱에서 이들 중 하나 또는 둘 다를 사용할 수 있습니다.

이 정보는 항상 최신 상태로 유지하는 것이 중요합니다. 앱 이름과 웹사이트 URL은 여러분의 애플리케이션이 프로그래밍 방식으로 생성한 트윗의 메타데이터 내 출처 정보로 표시됩니다. X 앱의 사용 사례가 변경되는 경우, [Developer Terms](https://developer.x.com/content/developer-twitter/en/developer-terms/agreement-and-policy)를 준수하고 있는지 확인하기 위해 이 설정에서 사용 사례를 반드시 업데이트해야 합니다.

애플리케이션에 &#39;suspended&#39; 태그가 표시되는 경우, 이는 여러분의 앱이 [automation rules](https://help.x.com/en/rules-and-policies/twitter-automation)와 같은 X의 하나 이상의 [developer terms](https://developer.x.com/en/developer-terms.html)를 위반했기 때문입니다. Developer Platform 정책 팀은 앱 소유자의 X 계정에 설정된 이메일 주소를 통해 개발자와 소통합니다. 이 이메일 주소를 확인하려면 [X 계정 설정](https://x.com/settings/your_twitter_data/account)을 확인하십시오. 앱 정지에 대한 알림 이메일은 &quot;Application suspension notice&quot;와 유사한 제목으로 이 이메일 주소로 전송되며, 이후에 수행해야 할 작업에 대한 구체적인 정보를 포함합니다. X 팀과 함께 앱 정지를 해결하려면, 이메일에 있는 구체적인 안내를 확인하거나 [platform help form](https://help.x.com/forms/platform)을 이용해 주시기 바랍니다.

<div id="keys-and-tokens">
  #### 키와 토큰
</div>

프로젝트 내의 앱 또는 독립 실행형 앱에서 다음 토큰들을 조회, 재생성(regenerate)하거나 폐기(revoke)할 수 있습니다:

* [API Key (Consumer Key) and API Secret (Consumer Secret)](/ko/resources/fundamentals/authentication#api-key-and-secret)
* [App Access Token](/ko/resources/fundamentals/authentication#bearer-token-also-known-as-app-only)
* [User Access Token and Access Token Secret](/ko/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) -
  developer portal 내에서 확인할 수 있는 Access Token 및 Secret은 앱을 소유한 사용자에 해당합니다.

**참고:** 다른 사용자(즉, 앱을 소유한 사용자가 아닌 사용자)를 대신해 요청을 보내고 싶다면, [OAuth 1.0a 3-legged OAuth flow](https://developer.x.com/resources/fundamentals/authentication/obtaining-user-access-tokens) 또는 OAuth 2.0 [Authorization Code with PKCE flow](/ko/resources/fundamentals/authentication#oauth-2-0-authorization-code-flow-with-pkce-2)를 사용해 해당 사용자의 Access Token 일련을 생성해야 합니다. 그런 다음 이러한 사용자 전용 토큰을 API 요청에 사용해야 합니다.

<div id="user-authentication-settings">
  #### 사용자 인증 설정
</div>

앱의 인증 설정을 OAuth 1.0a 또는 OAuth 2.0 중에서 선택할 수 있습니다. OAuth 2.0은 X API v2에서만 사용할 수 있습니다. OAuth 2.0을 사용하면 세밀한 단위의 scope를 선택하여, 사용자를 대신해 특정 권한을 부여할 수 있습니다. OAuth 1.0a는 X API v1.1 및 v2에서 사용할 수 있으며, 보다 포괄적인 scope를 사용하는 광범위한 권한 부여 방식입니다.

<div id="oauth-10a-application-user-permissions">
  #### OAuth 1.0a 애플리케이션-사용자 권한
</div>

앱 소유자인 경우 앱의 권한(읽기 전용, 읽기 및 쓰기, 또는 읽기, 쓰기 및 다이렉트 메시지)을 조정할 수 있습니다. 이는 X API를 통해 어떤 리소스와 이벤트에 액세스할 수 있는지를 제어합니다(참고: 일부 리소스는 X의 추가 승인이 필요합니다).

이 페이지에서 앱이 사용자 이메일 주소를 요청할 수 있는지 여부를 켜거나 끌 수도 있습니다(이를 위해서는 &quot;App details&quot; 페이지에 서비스 약관 URL과 개인정보 처리방침 URL이 있어야 합니다).

<div id="oauth-20-type-of-app">
  #### OAuth 2.0 앱 유형
</div>

사용자 인증 방식으로 OAuth 2.0을 선택하면, 생성하려는 앱 유형을 선택해야 합니다. 선택 가능한 옵션은 Native App, Single page App, Web App, Automated App 또는 봇(bot)입니다. Native App 및 Single page App은 공개 클라이언트(public client)이고, Web App 및 Automated App 또는 봇은 비공개 클라이언트(confidential client)입니다.

Confidential client는 인가 서버(authorization server)와 보안 방식으로 인증을 수행합니다. 이들은 client secret을 안전하게 보호합니다. Public client는 일반적으로 브라우저나 모바일 기기에서 실행되는 애플리케이션으로, client secret을 사용할 수 없습니다. Confidential client에 해당하는 앱 유형을 선택하면, client secret이 발급됩니다.

<div id="app-permissions">
  ## 앱 권한
</div>

<div id="oauth-10a-app-permissions">
  ### OAuth 1.0a 앱 권한
</div>

앱 권한은 OAuth 1.0a 애플리케이션과 사용자 간 인증에 대해 부여되는 접근 수준을 의미합니다. 앱 권한은 [X App](/ko/resources/fundamentals/developer-apps)의 각 앱 설정에서 구성합니다.

사용 가능한 권한 수준은 다음 세 가지입니다:

1. 읽기 전용
2. 읽기 및 쓰기
3. 읽기, 쓰기 및 다이렉트 메시지 접근

사용자의 이메일 주소를 열람할 수 있도록 요청하는 추가 권한이 있으며, 이는 위에 나열된 세 가지 수준 중 어느 것과도 조합해 사용할 수 있습니다.

권한 수준이 변경되면, 해당 X 앱에 이미 발급된 모든 사용자 토큰은 폐기해야 하며, 토큰에 변경된 권한이 반영되도록 사용자가 앱을 다시 승인해야 합니다.

권장되는 모범 사례는 애플리케이션이나 서비스에 필요한 사용자의 계정 데이터에 대해 *필요한 최소한의 접근 수준만* 요청하는 것입니다.

<div id="read-only">
  ### 읽기 전용
</div>

이 권한 수준은 사용자의 트윗, 홈 타임라인, 프로필 정보 등과 같은 X 리소스에 대한 읽기 액세스만 허용합니다. 사용자의 다이렉트 메시지는 읽을 수 없으며, 어떤 요소나 객체도 변경할 수 없습니다.

<div id="read-and-write">
  ### 읽기 및 쓰기
</div>

이 권한 수준에서는 X 리소스에 대한 읽기 및 쓰기 권한이 부여됩니다. 읽기 권한뿐 아니라 트윗을 게시하고, 사용자를 팔로우하며, 사용자의 프로필 정보 일부를 업데이트할 수 있습니다. 또한 인증된 사용자를 대신하여 답글을 숨길 수 있습니다. 이 권한 수준은 다이렉트 메시지(Direct Messages)에 대한 읽기, 쓰기, 삭제 등 어떠한 접근도 허용하지 않습니다.

<div id="read-write-and-access-direct-messages">
  ### 읽기, 쓰기 및 다이렉트 메시지에 대한 액세스
</div>

이 권한 수준에는 위에 나열된 모든 권한이 포함되며, 여기에 사용자를 대신하여 다이렉트 메시지를 읽고, 보내고, 삭제할 수 있는 권한이 추가됩니다.

* [POST /2/dm&#95;conversations/:dm&#95;conversation&#95;id/messages](/ko/x-api/direct-messages/send-a-new-message-to-a-dm-conversation)
* [POST /2/dm&#95;conversations/](/ko/x-api/direct-messages/create-a-new-dm-conversation)
* [POST /2/dm&#95;conversations/with/:participant&#95;id/messages](/ko/x-api/direct-messages/send-a-new-message-to-a-user)
* [GET /2/dm&#95;conversations/with/:user&#95;id/dm&#95;events](/ko/x-api/direct-messages/get-dm-events-for-a-dm-conversation)
* [GET /2/dm&#95;conversations/:dm&#95;conversation&#95;id/dm&#95;events](/ko/x-api/direct-messages/get-dm-events-for-a-dm-conversation-1)
* [GET /2/dm&#95;events](/ko/x-api/direct-messages/get-recent-dm-events)

<div id="determining-permissions">
  ### 권한 수준 확인
</div>

모든 인증된 API 요청은 HTTP 응답에 `x-access-level` 헤더를 반환합니다. 이 헤더의 값은 현재 사용 중인 권한 수준을 나타냅니다. 가능한 값은 read, read-write, read-write-directmessages입니다.

<div id="callback-urls">
  ## 콜백 URL
</div>

OAuth 1.0a 사용자 컨텍스트와 OAuth 2.0 Authorization Code with PKCE 인증 방식은 특정 로그인 플로우를 완료한 여러 X 사용자를 대신해 요청을 보낼 수 있도록 개발자가 사용할 수 있게 합니다. 현재 사용자에게 애플리케이션에 대한 권한 부여를 요청할 때 사용할 수 있는 플로우는 다음 두 가지입니다:

* [PKCE를 사용하는 OAuth 2.0 authorization code 플로우](/ko/resources/fundamentals/authentication#oauth-2-0-authorization-code-flow-with-pkce-2)
* [OAuth 1.0a 3-legged OAuth 플로우](/ko/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) (그리고 별도로, [Sign in with X 플로우](/ko/resources/fundamentals/authentication#log-in-with-x))

사용자가 이러한 플로우를 진행하는 동안, 성공적으로 로그인하고 개발자의 앱에 권한을 부여한 이후에 이동하게 될 웹 페이지 또는 위치가 필요합니다. 이 이후의 웹페이지나 위치를 콜백 URL이라고 부릅니다.

잠재 사용자들이 이러한 플로우를 수행할 수 있도록 설정할 때, 개발자는 앞에서 언급한 플로우를 구성하는 인증 엔드포인트로 보내는 요청에 콜백 URL을 포함해야 합니다. 예를 들어, OAuth 1.0a 사용자 컨텍스트를 사용하는 개발자는 [GET oauth/request&#95;token](/ko/resources/fundamentals/authentication#post-oauth-request-token) 엔드포인트에 요청을 보낼 때 `callback_url` 파라미터를 전달해야 합니다. 마찬가지로, OAuth 2.0 Authorization Code with PKCE를 사용하는 개발자는 GET oauth2/authorize 엔드포인트로의 요청에 `redirect_uri` 파라미터를 포함해야 합니다.

이러한 파라미터를 사용하는 것 외에도, 개발자는 콜백 URL이 [앱의](/ko/resources/fundamentals/developer-apps) 콜백 URL allowlist에도 추가되어 있는지 확인해야 하며, 이는 [developer portal](/ko/resources/fundamentals/developer-portal)의 앱 설정 페이지에서 확인할 수 있습니다.

설정이 올바르게 구성되어 있다면, 이러한 플로우의 일부로 X에 성공적으로 로그인한 후 사용자는 콜백 URL로 리디렉션됩니다.

<div id="things-to-keep-in-mind">
  ### 유의해야 할 사항
</div>

콜백 URL을 설정할 때는 다음과 같은 사항을 유의해야 합니다:

**쿼리 매개변수 HTTP 인코딩**

`callback_url` 또는 `redirect_uri` 매개변수로 콜백 URL을 쿼리 매개변수에 포함해 전달하므로, 반드시 URL을 HTTP 인코딩해야 합니다.

**콜백 URL 제한**

X Apps 대시보드에서는 콜백 URL을 최대 10개까지만 등록할 수 있습니다. 콜백 URL은 Apps 대시보드의 허용 목록에 추가한 콜백 URL과 인가 흐름에서 매개변수로 추가하는 콜백 URL이 항상 정확히 일치해야 합니다.

콜백 URL에 요청별 데이터를 포함하고 싶다면, 사용자가 리디렉션된 후 포함될 데이터를 저장하기 위해 `state` 매개변수를 사용할 수 있습니다. `state` 매개변수 자체에 데이터를 인코딩하거나, 이를 세션 ID로 사용해 서버에 상태를 저장하는 방식으로 사용할 수 있습니다.

**콜백 URL로 localhost를 사용하지 마세요**

localhost 대신 로컬 환경에서는 커스텀 호스트를 사용하거나 http(s)://127.0.0.1 을 사용하세요.

**커스텀 프로토콜 URL**

모바일 딥링킹을 활용하려면, twitter://callback/path 와 같이 path와 도메인 부분을 포함하는 커스텀 프로토콜 URL을 사용할 수 있습니다. 다만, 피해야 하는 허용되지 않는 프로토콜 목록이 있습니다. 아래의 허용되지 않는 프로토콜 목록을 확인하세요.

<div id="error-example">
  ### 오류 예시
</div>

developer portal의 앱 설정에 콜백 URL을 올바르게 추가하지 않은 상태에서 이를 사용하면, 다음과 같은 오류 메시지가 표시됩니다:

**OAuth 1.0a**

```json HTTP 403 - Forbidden

{
  "errors": [
    {
      "code":415,"message":"이 클라이언트 애플리케이션에 대해 콜백 URL이 승인되지 않았습니다. 승인된 콜백 URL은 애플리케이션 설정에서 조정할 수 있습니다."
    }
  ]
}
```

또는

```xml
<?xml version="1.0" encoding="UTF-8"?>
<hash>
<error>Callback URL not approved for this client application. Approved callback URLs can be adjusted in your application settings</error>
<request>/oauth/request_token</request>
</hash>
```

**OAuth 2.0**

```json HTTP 400

{
  "error": "invalid_request",
  "error_description": "redirect uri에 전달된 값이 authorization code의 uri와 일치하지 않습니다."
}
```

**문제 해결**

오류가 발생한 경우, 인증 요청에 사용하는 callback URL이 HTTP 인코딩되어 있는지, 그리고 그 URL이 현재 요청에 사용 중인 App의 키와 토큰에 대해 allowlist에 추가된 callback URL과 정확히 일치하는지 확인하세요.
