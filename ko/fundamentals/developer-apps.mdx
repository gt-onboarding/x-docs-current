<div id="overview">
  ## 개요
</div>

기존 앱이 있는 경우 [developer portal의 App 페이지](https://developer.x.com/en/portal/projects-and-apps)에서 확인, 수정 또는 삭제할 수 있습니다.

X API와 X Ads API에 액세스하려면 각 요청에 함께 보내야 하는 [인증](/ko/resources/fundamentals/authentication) 자격 증명 세트(키와 토큰이라고도 함)가 필요합니다. 이러한 자격 증명은 사용 중인 특정 엔드포인트에서 요구하는 인증 유형에 따라 서로 다른 형태를 가질 수 있습니다.

다음은 앱에서 생성할 수 있는 다양한 자격 증명과 그 사용 방법입니다:

* **[API Key and Secret](/ko/resources/fundamentals/authentication#api-key-and-secret):** 기본적으로 앱의 사용자 이름과 비밀번호에 해당합니다. 이는 [OAuth 1.0a 사용자 컨텍스트](/ko/resources/fundamentals/authentication#oauth-1-0a-2)가 필요한 요청을 인증하거나, 사용자 액세스 토큰 또는 App Access Token과 같은 다른 토큰을 생성하는 데 사용합니다.

* **[Access Token and Secret](/ko/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow):** 일반적으로 액세스 토큰은 요청을 대신 수행하려는 사용자를 나타냅니다. developer portal을 통해 생성할 수 있는 액세스 토큰은 앱 소유자를 나타냅니다. 이는 [OAuth 1.0a 사용자 컨텍스트](/ko/resources/fundamentals/authentication#oauth-1-0a-2)가 필요한 요청을 인증하는 데 사용합니다. 다른 사용자를 대신해 요청을 보내려면, 해당 사용자가 여러분의 앱을 승인할 수 있도록 3-legged OAuth 플로우를 사용해야 합니다.

* **[Client ID and Client Secret](/ko/resources/fundamentals/authentication#oauth-2-0):** 이 자격 증명은 OAuth 2.0 인증을 사용하여 사용자 액세스 토큰을 얻는 데 사용됩니다. OAuth 1.0a와 유사하게, 사용자 액세스 토큰은 비공개 사용자 계정 정보를 제공하거나 다른 계정을 대신해 작업을 수행하는 요청을 인증하는 데 사용되지만, 클라이언트 애플리케이션이 사용자에 대해 가질 수 있는 접근 권한을 보다 세밀하게 제어할 수 있도록 세분화된 스코프를 제공합니다.

* **[App only Access Token](/ko/resources/fundamentals/authentication#bearer-token-also-known-as-app-only):** X에서 공개적으로 제공되는 정보를 반환하는 엔드포인트에 요청을 보낼 때 이 토큰을 사용합니다.

X API 요청에 필요한 키와 토큰을 생성하는 것 외에도, [권한](/ko/resources/fundamentals/developer-apps#app-permissions)을 설정하고, 앱의 사용 사례나 목적을 문서화하고, [callback URL](/ko/resources/fundamentals/developer-apps#callback-urls)을 정의하며, [관리 대시보드](https://developer.x.com/en/portal/projects-and-apps) 내에서 앱의 개발자 환경과 관련된 기타 설정을 수정할 수 있습니다.

<div id="apps-and-projects">
  ### 앱과 프로젝트
</div>

X Developer Platform에서의 작업을 사용 사례별로 더 잘 구성하기 위해 앱과 [프로젝트](/ko/resources/fundamentals/projects)를 사용할 수 있습니다. 각 프로젝트에는 Free X API 플랜에서는 앱 1개, Basic 플랜에서는 최대 앱 2개, Pro 플랜에서는 최대 앱 3개를 포함할 수 있습니다.

새로운 [X API v2](/ko/x-api/introduction) 엔드포인트에 액세스하려면 프로젝트에 연결된 앱의 키와 토큰을 사용해야 합니다.

프로젝트가 도입되기 전에 생성된 앱이 있는 경우, 해당 앱은 &quot;Standalone Apps&quot;라는 섹션에 표시됩니다. Standalone Apps는 프로젝트 구조 밖에 있는 앱입니다. Standalone App을 프로젝트에 연결하면 해당 앱을 사용해 v2 엔드포인트에 요청을 보낼 수 있게 됩니다.

<div id="developer-portal-dashboard">
  ### Developer portal 대시보드
</div>

계정과 연결된 앱을 관리하려면 [대시보드](https://developer.x.com/content/developer-twitter/en/apps)에 방문하세요. 자세한 내용은 [developer portal](/ko/resources/fundamentals/developer-portal) 관련 문서 페이지를 참조하세요. 대시보드에서는 개발자가 다음 작업을 빠르고 쉽게 수행할 수 있습니다:

* 기존 Standalone 앱과 해당 App ID를 확인합니다.
* 새 프로젝트, 앱 또는 Standalone 앱을 생성합니다.
* 사용하지 않는 프로젝트나 앱을 삭제합니다.
* 이름, 설명, 웹사이트, callback URL, 그리고 [permissions](/ko/resources/fundamentals/developer-apps#app-permissions)을 포함한 특정 앱의 설정을 검토하거나 업데이트합니다.
* API Key &amp; Secret, App 액세스 토큰, 앱 소유자의 사용자 액세스 토큰과 같은 앱 전용 자격 증명을 다시 생성합니다.

<div id="signing-up-for-access">
  ### 액세스 신청
</div>

기존 X 앱이 있는 경우 developer.x.com에서 X 계정으로 로그인한 상태라면 X [App dashboard](https://developer.x.com/content/developer-twitter/en/portal/projects-and-apps)에서 앱을 확인하고 편집할 수 있습니다. 이전에 developer.x.com에서 생성한 모든 앱을 관리하기 위해 **새로 계정을 가입할 필요는 없다는 점**에 유의하십시오.

새로운 X 앱을 생성해야 하는 경우 먼저 [개발자 계정에 가입](https://developer.x.com/en/portal/petition/essential/basic-info)해야 합니다. [팀 계정](/ko/resources/fundamentals/developer-portal#team-management)의 구성원인 경우 새 앱을 생성하려면 관리자 역할이 할당되어 있어야 합니다.

<div id="automated-account-labeling-for-bots">
  ### 봇에 대한 자동 계정 라벨링
</div>

X 사용자에게 봇 계정이 자동 계정임을 알리기 위해, 봇 계정에 Automated Account 라벨을 추가할 수 있습니다. 이러한 봇은 X API를 통해 미리 설정된 작업을 수행합니다. 봇에 Automated Account 라벨을 추가하면, 사용자와의 신뢰를 쌓고 계정의 신뢰도를 높이며, 스팸성 봇과 차별화할 수 있습니다. 이를 통해 사용자는 X에서 봇과 상호작용할 때 계정의 목적을 더 잘 이해할 수 있습니다.

봇 계정에 라벨을 연결하려면 다음 단계를 따르세요.

1. 계정 설정으로 이동합니다.
2. &quot;Your account&quot;을 선택합니다.
3. &quot;Automation&quot;을 선택합니다.
4. &quot;Managing account&quot;을 선택합니다.
5. 다음으로, 봇 계정을 운영하는 X 계정을 선택합니다.
6. 비밀번호를 입력해 로그인합니다.
7. 마지막으로, 라벨이 계정에 적용되었다는 확인을 볼 수 있습니다.

<div id="app-management">
  ## 앱 관리
</div>

<div id="introduction">
  ### 소개
</div>

[X App dashboard](https://developer.x.com/en/portal/projects-and-apps)를 사용하면 개발자는 다음 작업을 빠르고 쉽게 수행할 수 있습니다.

* 기존 앱과 [프로젝트](/ko/resources/fundamentals/projects), 그리고 각 앱에 연결된 App ID를 확인합니다.
* 새 프로젝트 또는 독립 실행형 앱을 생성합니다.
* 프로젝트, 앱 또는 독립 실행형 앱을 삭제합니다.
* 특정 앱의 설정으로 들어가 해당 앱의 설정을 엽니다. 설정에서 앱 세부 정보, 키와 토큰, 권한을 확인할 수 있습니다.
* OAuth 1.0a 또는 OAuth 2.0 중 하나를 사용하도록 앱의 사용자 인증 설정을 업데이트합니다.

<Note>
  **참고:**

  모든 앱 키와 토큰은 더 이상 developer portal 내에서 확인할 수 없으며, 생성 시 반드시 안전하게 저장해야 합니다. 키와 토큰은 비밀번호 관리자를 사용해 보관할 것을 권장합니다.

  해당 자격 증명이 어떤 앱에 해당하는지 식별하는 데 도움이 되도록 API Key 힌트를 표시할 수 있습니다.
</Note>

<div id="app-settings">
  ### 앱 설정
</div>

<div id="app-details">
  #### 앱 세부 정보
</div>

여기에서 앱 아이콘, 앱 이름, 앱 설명, 웹사이트 URL, [callback URLs/redirect URIs](/ko/resources/fundamentals/developer-apps#callback-urls), 서비스 약관 URL, 개인정보 처리방침 URL, 조직 이름, 조직 URL, 그리고 앱의 목적 또는 사용 사례를 수정할 수 있습니다.

OAuth 2.0 및 OAuth 1.0a는 사용자가 X 계정을 사용해 앱에 로그인할 수 있게 해 주는 인증 방식입니다. 또한 인증된 사용자를 대신해 앱이 특정 요청을 보낼 수 있도록 허용합니다. 앱에 대해 이들 방식 중 하나 또는 둘 다를 활성화할 수 있습니다.

이 정보는 항상 최신 상태로 유지하는 것이 중요합니다. 앱 이름과 웹사이트 URL은 애플리케이션에 의해 프로그래밍 방식으로 생성된 트윗의 메타데이터 내에서 출처로 표시됩니다. X 앱의 사용 사례를 변경하는 경우, [Developer Terms](https://developer.x.com/content/developer-twitter/en/developer-terms/agreement-and-policy)을 준수하고 있는지 확인하기 위해 이 설정에서 사용 사례 정보를 반드시 업데이트해야 합니다.

애플리케이션에 &#39;suspended&#39; 태그가 표시되는 경우, 이는 앱이 [automation rules](https://help.x.com/en/rules-and-policies/twitter-automation) 등 X의 [developer terms](https://developer.x.com/en/developer-terms.html) 중 하나 이상을 위반했기 때문입니다. 개발자 플랫폼 정책 팀은 앱 소유자의 X 계정에 설정된 이메일 주소를 통해 개발자와 소통합니다. 이 이메일 주소를 확인하려면 [X 계정 설정](https://x.com/settings/your_twitter_data/account)을 확인해 주세요. 정지 관련 알림 이메일은 &quot;Application suspension notice&quot;와 비슷한 제목으로 이 이메일 주소로 전송되며, 이후에 무엇을 해야 하는지에 대한 구체적인 정보가 포함됩니다. X 팀과 함께 정지 문제를 해결하려면, 구체적인 안내를 위해 이메일을 확인하거나 [platform help form](https://help.x.com/forms/platform)을 사용해 주세요.

<div id="keys-and-tokens">
  #### 키와 토큰
</div>

프로젝트 내의 앱 또는 단독 앱에서 다음 토큰을 확인, 재발급 또는 취소할 수 있습니다:

* [API Key (Consumer Key) and API Secret (Consumer Secret)](/ko/resources/fundamentals/authentication#api-key-and-secret)
* [App Access Token](/ko/resources/fundamentals/authentication#bearer-token-also-known-as-app-only)
* [User Access Token and Access Token Secret](/ko/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) -
  developer portal 내에서 제공되는 Access Token 및 Secret은 해당 앱을 소유한 사용자에 해당합니다.

**참고** - 다른 사용자를 대신하여 요청을 보내려면(즉, 앱 소유자가 아닌 사용자), [OAuth 1.0a 3-legged OAuth 플로우](https://developer.x.com/resources/fundamentals/authentication/obtaining-user-access-tokens) 또는 OAuth 2.0 [Authorization Code with PKCE 플로우](/ko/resources/fundamentals/authentication#oauth-2-0-authorization-code-flow-with-pkce-2)를 사용하여 해당 사용자의 Access Token 한 세트를 생성해야 합니다. 그런 다음 API에 요청할 때 이 사용자 전용 토큰을 사용하면 됩니다.

<div id="user-authentication-settings">
  #### 사용자 인증 설정
</div>

앱의 인증 설정을 OAuth 1.0a 또는 OAuth 2.0으로 선택할 수 있습니다. OAuth 2.0은 X API v2에서만 사용할 수 있습니다. OAuth 2.0은 세분화된 scope를 선택할 수 있게 해 주며, 이를 통해 사용자를 대신해 특정 권한을 부여할 수 있습니다. OAuth 1.0a는 X API v1.1 및 v2에서 사용할 수 있으며, 보다 넓은 범위의 권한을 제공하는 포괄적인 scope 기반 인가 방식을 사용합니다.

<div id="oauth-10a-application-user-permissions">
  #### OAuth 1.0a 앱-사용자 권한
</div>

앱 소유자인 경우, 앱의 권한 수준(읽기 전용, 읽기 및 쓰기, 또는 읽기·쓰기 및 다이렉트 메시지)을 조정할 수 있습니다. 이는 X API를 통해 어떤 리소스와 이벤트에 액세스할 수 있는지를 제어합니다(참고: 일부 리소스는 X의 별도 승인이 추가로 필요합니다).

또한 이 페이지에서 앱이 사용자 이메일 주소를 요청할 수 있는지 여부를 켜거나 끌 수 있습니다(이를 위해서는 &quot;App details&quot; 페이지에 이용약관 URL과 개인정보 처리방침 URL이 있어야 합니다).

<div id="oauth-20-type-of-app">
  #### OAuth 2.0 앱 유형
</div>

OAuth 2.0을 사용자 인증 방식으로 선택하는 경우 생성하려는 앱의 유형을 선택해야 합니다. 선택 가능한 옵션은 Native App, Single page App, Web App, Automated App 또는 bot입니다. Native App과 Single page App은 public client이고, Web App과 Automated App 또는 bot은 confidential client입니다.

Confidential client는 authorization server(인가 서버)와 안전하게 상호 인증을 수행하며, client secret을 안전하게 보호합니다. Public client는 보통 브라우저나 모바일 기기에서 실행되는 애플리케이션으로, client secret을 사용할 수 없습니다. 앱 유형으로 confidential client에 해당하는 항목을 선택하면 client secret이 제공됩니다.

<div id="app-permissions">
  ## 앱 권한
</div>

<div id="oauth-10a-app-permissions">
  ### OAuth 1.0a 앱 권한
</div>

앱 권한은 OAuth 1.0a 앱-사용자 인증에서의 접근 수준을 의미합니다. 앱 권한은 [X App](/ko/resources/fundamentals/developer-apps) 설정에서 앱별로 구성합니다.

사용 가능한 권한 수준은 세 가지입니다:

1. 읽기 전용
2. 읽기 및 쓰기
3. 읽기, 쓰기 및 다이렉트 메시지 접근

사용자의 이메일 주소 열람을 요청하기 위한 추가 권한이 있으며, 이는 위에 나열된 세 가지 수준 중 어느 것과도 함께 사용할 수 있습니다.

권한 수준이 변경되면, 해당 X 앱에 이미 발급된 모든 사용자 토큰은 폐기해야 하며, 토큰에 변경된 권한이 반영되도록 사용자가 앱을 다시 승인해야 합니다.

일반적으로 권장되는 모범 사례는 애플리케이션이나 서비스에 필요한 사용자 계정 데이터에 대해 *최소한의 접근 수준만* 요청하는 것입니다.

<div id="read-only">
  ### 읽기 전용
</div>

이 권한 수준에서는 사용자의 트윗, 홈 타임라인, 프로필 정보 등 X 리소스에 대해 읽기 전용으로만 접근할 수 있습니다. 사용자의 다이렉트 메시지를 읽을 수 없으며, 어떤 요소나 객체도 변경하거나 업데이트할 수 없습니다.

<div id="read-and-write">
  ### 읽기 및 쓰기
</div>

이 권한 수준은 X 리소스에 대한 읽기 및 쓰기 액세스를 허용합니다. 읽기 권한뿐만 아니라 트윗을 게시하고, 사용자를 팔로우하거나, 사용자의 프로필 정보 요소를 업데이트할 수도 있습니다. 또한 인증된 사용자를 대신해 답글을 숨길 수 있습니다. 이 권한 수준은 Direct Message(다이렉트 메시지)에 대한 어떠한 액세스(읽기, 쓰기, 삭제 포함)도 허용하지 않습니다.

<div id="read-write-and-access-direct-messages">
  ### 다이렉트 메시지 읽기, 쓰기 및 접근
</div>

<div id="determining-permissions">
  ### 권한 확인
</div>

인증된 모든 API 요청은 HTTP 응답에 `x-access-level` 헤더를 포함해 반환합니다. 이 헤더의 값은 현재 사용 중인 권한 수준을 나타냅니다. 가능한 값은 read, read-write, read-write-directmessages입니다.

<div id="callback-urls">
  ## Callback URLs
</div>

OAuth 1.0a 사용자 컨텍스트와 OAuth 2.0 Authorization Code with PKCE 인증 방식은 특정 로그인 플로우를 거친 서로 다른 X 사용자를 대신해 요청을 보낼 수 있도록 해 줍니다. 현재 사용자가 여러분의 애플리케이션을 승인할 수 있게 해주는 플로우는 다음 두 가지입니다:

* [OAuth 2.0 authorization code flow with PKCE](/ko/resources/fundamentals/authentication#oauth-2-0-authorization-code-flow-with-pkce-2)
* [OAuth 1.0a 3-legged OAuth flow](/ko/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) (그리고 별도로, [Sign in with X flow](/ko/resources/fundamentals/authentication#log-in-with-x))

사용자들이 이러한 플로우를 진행하는 동안, 로그인에 성공하고 개발자의 앱에 권한을 부여한 뒤 이동할 웹 페이지나 위치가 필요합니다. 이 후속 웹 페이지나 위치를 callback URL이라고 합니다.

잠재 사용자들이 이러한 플로우를 진행할 수 있도록 설정할 때, 개발자는 앞에서 언급한 플로우를 구성하는 인증 엔드포인트로 요청을 보낼 때 callback URL을 함께 전달해야 합니다. 예를 들어, OAuth 1.0a 사용자 컨텍스트를 사용하는 개발자는 [GET oauth/request&#95;token](/ko/resources/fundamentals/authentication#post-oauth-request-token) 엔드포인트에 요청을 보낼 때 `callback_url` 파라미터를 전달해야 합니다. 마찬가지로, OAuth 2.0 Authorization Code with PKCE를 사용하는 개발자는 GET oauth2/authorize 엔드포인트로 요청을 보낼 때 `redirect_uri` 파라미터를 전달해야 합니다.

이러한 파라미터를 사용하는 것 외에도, 개발자는 callback URL이 [App의](/ko/resources/fundamentals/developer-apps) callback URL allowlist에도 추가되어 있는지 확인해야 합니다. 이 설정은 [developer portal의](/ko/resources/fundamentals/developer-portal) App 설정 페이지에서 확인할 수 있습니다.

구성이 올바르게 되어 있다면, 사용자는 이러한 플로우의 일부로 X 로그인을 성공적으로 완료한 뒤 callback URL로 리디렉션됩니다.

<div id="things-to-keep-in-mind">
  ### 기억해야 할 사항
</div>

callback URL을 설정할 때 염두에 두어야 할 몇 가지 사항이 있습니다:

**쿼리 매개변수를 HTTP 인코딩하기**

`callback_url` 또는 `redirect_uri` 매개변수로 callback URL을 쿼리 매개변수로 전달하므로, 해당 URL을 반드시 HTTP 인코딩해야 합니다.

**Callback URL 제한**

X Apps 대시보드에서 callback URL은 최대 10개까지 등록할 수 있는 엄격한 제한이 있습니다. Apps 대시보드에 허용 목록으로 추가한 callback URL과 인증 플로우에 추가하는 매개변수 값은 항상 정확히 일치해야 합니다.

callback URL에 요청별 데이터를 포함하려는 경우, 사용자가 리디렉트된 후 포함될 데이터를 저장하기 위해 `state` 매개변수를 사용할 수 있습니다. `state` 매개변수 자체에 데이터를 인코딩하거나, 이 매개변수를 세션 ID로 사용해 서버에 상태를 저장할 수 있습니다.

**localhost를 callback URL로 사용하지 마세요**
localhost 대신 로컬 환경에서는 커스텀 호스트나 http(s)://127.0.0.1 을 사용하세요.

**커스텀 프로토콜 URL**
모바일 딥 링크 기능을 활용하려면, twitter://callback/path 와 같이 경로와 도메인 부분을 포함하는 커스텀 프로토콜 URL을 사용할 수 있습니다. 다만 피해야 하는 비허용 프로토콜 목록이 있습니다. 아래의 비허용 프로토콜 목록을 참고하세요.

<div id="error-example">
  ### 오류 예시
</div>

developer portal의 앱 설정에 콜백 URL이 올바르게 추가되지 않은 상태에서 해당 URL을 사용하면 다음과 같은 오류 메시지가 표시됩니다:

**OAuth 1.0a**

```json HTTP 403 - Forbidden

{
  "errors": [
    {
      "code":415,"message":"이 클라이언트 애플리케이션에 대해 콜백 URL이 승인되지 않았습니다. 승인된 콜백 URL은 애플리케이션 설정에서 조정할 수 있습니다."
    }
  ]
}
```

또는

```xml
<?xml version="1.0" encoding="UTF-8"?>
<hash>
<error>Callback URL not approved for this client application. Approved callback URLs can be adjusted in your application settings</error>
<request>/oauth/request_token</request>
</hash>
```

**OAuth 2.0**

```json HTTP 400

{
  "error": "invalid_request",
  "error_description": "redirect uri에 전달된 값이 authorization code의 uri와 일치하지 않습니다."
}
```

**문제 해결**

오류가 발생하면 인증 요청에서 사용하는 callback URL이 HTTP로 인코딩되어 있고, 해당 callback URL이 요청에 사용하는 App의 키와 토큰에 대해 허용 목록(allowlist)에 추가된 callback URL과 일치하는지 확인하세요.
