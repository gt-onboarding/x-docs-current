이 페이지에서는 트윗을 작성할 때와 X API 전반에서 문자가 어떻게 계산·처리되는지 설명합니다. 구현 방식에 대한 자세한 내용은 X에서 제공하는 오픈 소스 라이브러리인 [twitter-text](http://github.com/twitter/twitter-text)를 참고하세요. 이 라이브러리는 [GitHub](https://github.com/twitter)에서 확인할 수 있습니다.

<div id="background">
  ## 배경
</div>

X는 SMS 문자 메시지 기반 서비스로 시작되었습니다. 이 때문에 초기 트윗 길이는 140자로 제한되었는데, 이는 SMS의 160자 제한 중 20자를 명령어와 사용자 이름에 할당했기 때문입니다. 시간이 지나 X가 발전함에 따라 최대 트윗 길이는 280자로 늘어났습니다. 여전히 짧고 간결한 길이이지만, 보다 풍부한 표현이 가능해졌습니다.

<div id="definition-of-a-character">
  ## 문자의 정의
</div>

대부분의 경우 트윗의 텍스트 콘텐츠에는 최대 280개의 문자 또는 [Unicode](https://unicode.org/) 글리프를 포함할 수 있습니다. 일부 글리프는 문자 두 개 이상으로 계산됩니다.

글리프가 한 개 또는 그 이상의 문자로 계산되는지를 그 글리프의 가중치(weight)라고 부릅니다. 어떤 문자가 한 문자보다 큰 가중치를 갖는지에 대한 정확한 정의는 [twitter-text 트윗 파싱 라이브러리](https://github.com/twitter/twitter-text)의 [구성 파일(configuration file)](https://github.com/twitter/twitter-text/tree/master/config)에 나와 있습니다.

현재 버전의 구성 파일은 기본적으로 두 문자 가중치를 정의하고, 가중치가 다르게 적용되는 네 개의 [Unicode 코드 포인트](https://unicode.org/charts/About.html) 범위를 정의합니다. 현재 이 범위에 속하는 코드 포인트는 모두 한 문자로 계산됩니다.

* 첫 번째 범위는 Latin-1 코드 페이지 전반의 문자를 포함합니다(U+0000 - U+10FF).
* 두 번째 범위는 Zero Width Joiner(이모지 및 다른 글리프를 결합하는 데 사용됨)를 포함한 일반 문장 부호입니다(U+2000-U+200D).
* 세 번째 범위는 Unicode 방향 표식인 U+200E 및 U+200F를 제외한 일반 문장 부호입니다(U+2010-U+201F).
* 마지막 범위는 따옴표 문자 범위입니다(U+2032-U+2037).

트윗 텍스트와 twitter-text 라이브러리가 계산한 길이 예시는 라이브러리의 [validate.yml](https://github.com/twitter/twitter-text/blob/master/conformance/validate.yml) 테스트 구성 파일에서 확인할 수 있습니다.

**예시**

| 화면에 표시되는 문자 | 길이 | 설명 | Unicode 시퀀스 |
|:---------------------|:--------|:-------------|:------------------|
| a | 1 | 라틴 소문자 a | U+0061 |
| á | 1 | 라틴 소문자 a(예리 악센트 포함) | U+00E1 |
| ӑ | 1 | 키릴 소문자 a(브레브 포함) | U+04D1 |
| Ồ | 1 | 라틴 대문자 O(곡절 부호와 예리 악센트 포함) | U+1ED2 |

<div id="emojis">
  ## 이모지
</div>

[twemoji](https://twemoji.x.com/)에서 지원하는 이모지는 결합용 수정자와 관계없이 항상 두 문자로 계산됩니다. 여기에는 [Fitzpatrick 피부 톤](https://emojipedia.org/modifiers/)이나 [성별 수정자](https://blog.emojipedia.org/unicode-and-the-emoji-gender-gap/)로 변경된 이모지도 포함되며, 이러한 이모지가 훨씬 더 많은 Unicode 코드 포인트로 구성되어 있더라도 마찬가지입니다. 이모지 길이(가중치)는 표준 이모지와 하나 이상의 Unicode Zero Width Joiner(U+200D)가 결합된 시퀀스를 찾는 twitter-text의 정규식으로 정의됩니다.

**예시**

| 표시된 이모지 | 길이 | 설명 | Unicode 시퀀스 |
|:-----------------|:--------|:-------------|:------------------|
| 👾 | 2 | 알려진 이모지의 기본 길이 | -- |
| 🙋🏽 | 2 | 피부 톤 수정자가 있는 이모지 | 🙋 U+1F64B, 🏽 U+1F3FD |
| 👨‍🎤 | 2 | 결합 글리프(제로 폭 조인자)를 사용하는 이모지 시퀀스 | [👨 U+1F468](https://emojipedia.org/emoji/%F0%9F%91%A8/), [U+200D](https://emojipedia.org/emoji/%E2%80%8D/), [🎤 U+1F3A4](https://emojipedia.org/emoji/%F0%9F%8E%A4/) |
| 👨‍👩‍👧‍👦 | 2 | 여러 개의 결합 글리프(제로 폭 조인자)를 사용하는 이모지 시퀀스 | [👨 U+1F468](https://emojipedia.org/emoji/%F0%9F%91%A8/), [U+200D](https://emojipedia.org/emoji/%E2%80%8D/), [👩 U+1F469](https://emojipedia.org/emoji/%F0%9F%91%A9/), [U+200D](https://emojipedia.org/emoji/%E2%80%8D/), [👧 U+1F467](https://emojipedia.org/emoji/%F0%9F%91%A7/), [U+200D](https://emojipedia.org/emoji/%E2%80%8D/), [👦 U+1F466](https://emojipedia.org/emoji/%F0%9F%91%A6/) |

<div id="chinese-japanese-korean-glyphs">
  ## 중국어 / 일본어 / 한국어 글자
</div>

CJK(중국어 / 일본어 / 한국어) 언어에서 사용되는 글자는 두 글자로 계산됩니다. 따라서 CJK 텍스트로만 구성된 트윗은 최대 140개의 이러한 글자를 포함할 수 있습니다.

<div id="entity-objects">
  ## 엔터티 객체
</div>

트윗에는 [엔터티 객체](/ko/x-api/fundamentals/data-dictionary)가 포함될 수 있으며, 이 중 일부는 트윗의 길이에 영향을 줍니다.

URL: [모든 URL은 t.co 링크로 변환됩니다](https://developer.x.com/content/developer-twitter/en/docs/basics/tco). 이는 URL의 길이가 [twitter-text 설정 파일](https://github.com/twitter/twitter-text/tree/master/config)에 있는 `transformedURLLength` 매개변수로 정의된다는 의미입니다. 현재 트윗 내 URL의 길이는, 원래 URL 길이가 더 짧더라도 23자입니다.

답글: 답글 트윗의 시작 부분에 자동으로 채워지는 @이름은 문자 수 제한에 포함되지 않습니다. 새로 작성하는 답글이 아닌 트윗이 @멘션으로 시작하는 경우에는 문자 수에 포함되며, 사용자가 트윗 본문에 직접 추가한 @멘션도 문자 수에 포함됩니다.

미디어: 공식 클라이언트에서 게시된 경우 pic.x.com URL로 표현되는, 트윗에 첨부된 미디어는 문자 수에 포함되지 않습니다.

엔터티 객체에 대한 자세한 내용은 [개발자 문서](/ko/x-api/fundamentals/data-dictionary)를 참고하세요.

<div id="x-character-encoding">
  ## X 문자 인코딩
</div>

X API 엔드포인트는 UTF-8로 인코딩된 텍스트만 허용합니다. 그 외의 모든 인코딩은 텍스트를 API에 보내기 전에 UTF-8로 변환해야 합니다.

X는 트윗의 길이를 텍스트의 Normalization Form C(NFC) 버전을 사용해 계산합니다.

예를 들어 단어 &quot;café&quot;를 살펴보겠습니다. 겉보기와 발음은 같지만, 서로 다른 개수의 바이트를 사용하는 두 개의 바이트 시퀀스가 있습니다:

|     | | |
|:-----|:---|:---|
| café | 0x63 0x61 0x66 0xC3 0xA9 | &quot;é&quot; 문자, 즉 &quot;조합 문자(composed character)&quot;를 사용하는 경우. |
| café | 0x63 0x61 0x66 0x65 0xCC 0x81 | &quot;e&quot; 위에 겹쳐지는 결합 분음 부호(combining diacritical)를 사용하는 경우 |

Normalization Form C는 긴 형식(0x65 0xCC 0x81)보다 완전히 결합된 문자(위의 café 예시에서 0xC3 0xA9)를 사용하는 것을 선호합니다.

X는 UTF-8 바이트 수가 아니라 텍스트의 코드 포인트 개수를 기준으로 길이를 계산합니다. café 예시의 0xC3 0xA9는 UTF-8에서 두 바이트로 인코딩된 하나의 코드 포인트(U+00E9)이지만, 0x65 0xCC 0x81은 세 바이트로 인코딩된 두 개의 코드 포인트입니다.
