<div id="typical-response-structure">
  ## 一般的なレスポンス構造
</div>

成功したレスポンスは、200 番台の HTTP ステータスコードと、要求されたオブジェクトの取得・作成・更新・削除の結果に加え、サーバーがリクエストをどのように解釈したかを表す情報を含む、JSON ベースのペイロードによって示されます。

リクエストが成功した場合、レスポンスの一部として、リクエスト内容をそのまま返す `request` ノードが含まれます。

*例:* [GET accounts/abcdefg/campaigns?with&#95;deleted=true](/ja/x-ads-api/campaign-management#campaigns)

```json
{
  /* レスポンスのデータ... */,
  "request": {
    "params": {
      "account_id": "abcdefg",
      "with_deleted": "true"
    }
  }
}
```

JSON レスポンス内の `data` フィールドには、対象のリソースに関連付けられた特定のオブジェクトが含まれます。`data` ノードの形式は、レスポンスに 1 件以上の結果が含まれる可能性がある場合は JSON 配列になります。レスポンスとして 1 件の結果しか返されない場合は、JSON ハッシュとして返されます。まれなケースとして、通常はコレクションが含まれるはずのレスポンスで、代わりにハッシュマップが返されることがあります。この場合、その単一のハッシュマップは、`type` フィールドで指定されているものと同じ型のオブジェクトであると想定してください。

<div id="error-response-structure">
  ## エラーレスポンスの構造
</div>

エラーレスポンスは、200 番台以外の HTTP コードとともに返されます。通常は JSON レスポンスが付随しますが、一部のエラーでは別種のレスポンスボディが返される場合があります。レスポンス構造をパースできない状況では、HTTP コード本来の意味を優先して解釈してください。たとえば、HTTP 404 と共に HTML レスポンスが返されることがあります。この場合、コンテンツが見つからない（HTTP 404 は「Not Found」を意味します）と判断して問題ありません。

典型的なエラーレスポンスは、成功レスポンスと類似した構造になります。エラーの内容はレスポンスの `errors` ノードで伝達されます。`errors/code` ノードには、プログラムから解決方法を判断するために利用できる CAPS&#95;CASE の定数エラーコードが示されます。`errors/message` ノードには、エラーの内容を説明する（通常は）人間が読みやすい英語のメッセージが含まれます。さらに詳細なエラー情報を示すために、追加のフィールドが付与される場合があります。

```json
{
  "errors": [
    {
      "parameter": "start_time",
      "details": "invalid date",
      "code": "INVALID_PARAMETER",
      "value": "",
      "message": "Expected time, got \"\" for start_time"
    }
  ],
  "request": {
    "params": {
      "account_id": "hkk5"
    }
  }
}
```

上記の例では、analytics エンドポイントへのリクエストで、`start_time` パラメータに無効な値が指定されました。無効なパラメータを含むリクエストの `errors/code` は `INVALID_PARAMETER` です。
