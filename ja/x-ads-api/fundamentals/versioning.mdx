<div id="overview">
  ## 概要
</div>

毎月、私たちは X Ads API に変更を加え、複数の新機能をロールアウトしています。これらの変更はほとんど常に後方互換性を保っていますが、それでも毎年、少数ながら互換性を破る変更が発生しがちです。Ads API における高速な変更サイクルが、新機能の実装、非推奨対応、変更のテストといった開発サイクルに与える影響について、開発者の皆様からフィードバックを受けてきました。私たちは Ads プラットフォームを利用する際の開発者体験を向上させたいと考えており、そのためにエンドポイントのバージョニングという概念を導入しました。

以下に、本ドキュメントで取り上げるいくつかの概念を定義します。

**Version**: Ads API リクエストの URL パス内に含まれるバージョン番号を指します（例: GET /{version_number}/accounts）。この形式のバージョニングは URI バージョニングとして知られています。

**Breaking Changes**: 互換性を破る変更とは、既存の機能を維持するために開発者側で対応が必要となるあらゆる変更を指します。ここには、必要な変更内容の調査、非推奨とすべき機能／エンドポイントの判断、そしてそれらの変更を最終的に実装する作業に要するリソースが含まれます。互換性を破る変更の例としては、次のようなものがあります。

* API のリクエスト／レスポンスからパラメータを削除する

* いずれかのパラメータまたはエンドポイントの名前を変更する

* 値の表現方法を変更する（preview&#95;url → card&#95;uri）

* エンドポイントの動作を変更する（例: 非同期 vs 同期の統計）

* 任意／必須パラメータを追加・変更する（例: リクエスト内で name を必須フィールドにする）

**Deprecation**: 非推奨となったバージョンまたはプロダクトはサポート対象外となり、開発者はこれらの API の利用を中止することが推奨されます。

**Sunset**: プロダクトまたは API がサンセットされると、対応するエンドポイント群には API 経由でアクセスできなくなります。

<div id="versioning-strategy">
  ## バージョン管理戦略
</div>

この戦略の主な原則は次のとおりです。

1. すべての後方互換性を損なう変更は、新しいバージョンとしてまとめて提供されます

2. 新しいバージョンがアナウンスされるたびに、既存バージョンの非推奨期間は 6 か月となります

3. 任意の時点で、API は 2 つのバージョンからのリクエストを同時に許可しますが、そのうち古い方のバージョンはサポート対象外となります

4. 新しいプロダクトをより迅速に採用できるようにするため、これらはバージョン管理のリリースサイクルとは別に、継続的にリリースされます

5. すべての API レスポンスには、API の現在のバージョンを示す `x-current-api-version` が含まれ、さらに、非推奨となった API エンドポイントを呼び出した場合には `x-api-warn` ヘッダーも含まれます。

API に後方互換性を損なう変更を要するような、根本的なプロダクト要件の変更（例: 複数の年齢バケットターゲティングの非推奨化）が発生した場合、この破壊的変更をアナウンスする 90 日前予告の通知を送信し、通知の発出から少なくとも 90 日後に、その破壊的変更をデプロイします。

</div>

本日、2021年3月3日付で、X Ads API バージョン 9（v9）が利用可能になりました。このリリースは、機能差を解消し、キャンペーン作成を簡素化するとともに、Cards エンドポイントおよび Mobile App Promotion エンドポイントに対する重要なアップデートを導入することを目的としています。

<div id="new">
  #### 新機能
</div>

**どのエンティティがアクティブかを判定する**

[Active Entities](/ja/x-ads-api/analytics#active-entities-2) エンドポイントは、広告エンティティのアナリティクス・メトリクスに変更があったかどうかを示します。アナリティクス用エンドポイントと組み合わせて使用することを想定して設計されており、エンティティ種別と日付範囲（最大 90 日間）を指定することで、プラットフォームがアナリティクスを取得すべきエンティティ ID の配列を返します。返却された ID 以外は、その後のアナリティクスリクエストで指定しないでください。

このエンドポイントは、次のエンティティ種別をサポートします：`CAMPAIGN`、`FUNDING_INSTRUMENT`、`LINE_ITEM`、`MEDIA_CREATIVE`、`PROMOTED_TWEET`。

**`MEDIA_CREATIVE` の統計**

Ads API のアナリティクスエンドポイントで、Media Creative エンティティ向けのメトリクスが新たに提供されるようになりました。Media Creative は、インストリーム広告や X Audience Platform 上の画像広告をプロモーションするためのエンティティです。X Ads UI では、Media Creative のメトリクスは「In-stream videos」と「Display creatives」タブに表示されます。[同期](/ja/x-ads-api/analytics#get-stats-accounts-account-id)および[非同期](/ja/x-ads-api/analytics#asynchronous-analytics)のいずれのアナリティクスエンドポイントも、`MEDIA_CREATIVE` エンティティ enum をサポートするようになりました。

**複数のカードを取得**

カード URI 値で単一のカードを取得するために設計されたエンドポイントの [v3 リリース](https://devcommunity.x.com/t/ads-api-version-3/100732) からの改善として、[GET accounts/:account&#95;id/cards/all](/ja/x-ads-api/creatives#cards-fetch) エンドポイントを使用して *複数* のカードを取得できるようになりました。これにより、カードごとにリクエストを送るのではなく、1 回のリクエストで最大 200 件のカードを取得できます。

注意点は次の 2 点です。

1. URL パスが `accounts/:account_id/cards/all` になりました（以前のパスは利用できません）。これは、ID でカードを取得するために設計された[エンドポイント](/ja/x-ads-api/creatives#cards-fetch)との一貫性を保つためです。
2. 必須のリクエストパラメータ名は card&#95;uris（複数形）になりました。

**取得の柔軟性向上**

[GET accounts/:account&#95;id/targeting&#95;criteria](/ja/x-ads-api/campaign-management#get-accounts-account-id-targeting-criteria) エンドポイントは、複数の Line Item ID をサポートするようになりました。最大 200 個の ID を受け付ける line&#95;item&#95;ids パラメータが必須です。以前は単一の Line Item しか受け付けなかったため、同期が難しい状況がありました。この変更により、より多くのターゲティングを短時間で取得できるようになります。

次のエンドポイントも複数の Line Item ID をサポートするようになりましたが、これらについては line&#95;item&#95;ids パラメータは任意です。

* [GET accounts/:account&#95;id/line&#95;item&#95;apps](https://developer.x.com/en/docs/x-ads-api/campaign-management/api-reference/line-item-apps#get-accounts-account-id-line-item-apps)
* [GET accounts/:account&#95;id/media&#95;creatives](/ja/x-ads-api/campaign-management#get-accounts-account-id-media-creatives)
* [GET accounts/:account&#95;id/promoted&#95;accounts](/ja/x-ads-api/campaign-management#get-accounts-account-id-promoted-accounts)
* [GET accounts/:account&#95;id/preroll&#95;call&#95;to&#95;actions](/ja/x-ads-api/creatives#preroll-call-to-actions)

<div id="changed">
  #### 変更点
</div>

**ドラフトキャンペーンとラインアイテムの取得**

ドラフトキャンペーンとラインアイテムの取得方法が更新されました。現在では、`with_draft(boolean)` パラメーターを `true` に設定すると、ドラフトと非ドラフトの両方のエンティティが返されます。これは削除済みエンティティの取得方法（`with_deleted` を使用）と一貫しています。以前は、ドラフトと非ドラフトの両方のエンティティを取得するには少なくとも 2 回のリクエストが必要でしたが、現在は 1 回の API 呼び出しで実行できるようになりました。

| **v4** | **v5** |
| :--- | :--- | :--- |
| `draft_only` | `with_draft` |     |

**Network activation duration ターゲティング**

Ads API では、Network Activation Duration ターゲティングを追加した後、レスポンス内のターゲティングタイプに `_IN_SEC` サフィックスが含まれてしまう表示上の問題を解消しました。Network Activation Duration は常に月単位で表現されるため、秒を示す表記があると紛らわしい状況でした。この修正により表現が一貫し、混乱が軽減されました。

| **v4** | **v5** |
| :--- | :--- | :--- |
| `NETWORK_ACTIVATION_DURATION_IN_SEC` | `NETWORK_ACTIVATION_DURATION` |     |

**合計件数とカーソル**

v5 では、`with_total_count` と `cursor` は排他的です。リクエストで両方を指定すると、`EXCLUSIVE_PARAMETERS` エラーコードが返されます。v5 より前は、`cursor` が指定されている場合、`with_total_count` は無視されていました。この変更により、両者の関係が明示的になりました。

<div id="removed">
  #### 削除
</div>

3 つのフィールドが Ads API のレスポンスから削除されます: preview&#95;url、account&#95;id、parent&#95;ids です。これら 3 つに対する開発工数は最小限です。

* v4 では、カードに対する preview&#95;url レスポンスパラメータは常に null であると案内していました。この移行の最終ステップとして、すべてのカードレスポンスから preview&#95;url を削除します。
* 次のリソースについては、ads アカウント ID がすでに URL と request.params の両方に含まれているため、account&#95;id レスポンス属性が削除されます。（Funding Instrument をこのリストから意図的に除外しているのは、可能な場合には parent ID がレスポンスオブジェクトに存在すべきであり、アカウント ID は Funding Instrument の親エンティティであるためです。）
  * アカウントメディア
  * アプリイベントプロバイダー
  * アプリイベントタグ
  * キャンペーン
  * カード
  * ラインアイテム
  * プロモーション対象ユーザー
  * ターゲティング条件
* [GET accounts/:account&#95;id/targeting&#95;criteria](/ja/x-ads-api/campaign-management#get-accounts-account-id-targeting-criteria) リクエストでは、parent&#95;ids フィールドは常に空配列であったため、今後はレスポンスとして返されません。

**非メディア App カード**

v5 では、非メディア（メディアを含まない）App カードはサポートされなくなります。[以前](https://devcommunity.x.com/t/deprecation-announcement-app-download-cards/87807) のアナウンスで、非メディア App カードを作成または編集する機能はすでに削除済みでした。今回、このリソース用に残っているエンドポイントが非推奨となります。

* 注: これは画像および動画の App ダウンロードカードには影響しません。

**アカウントメディアの作成**

POST accounts/:account&#95;id/account&#95;media エンドポイントは v5 では利用できなくなります。このリソース向けの他のエンドポイントには影響は *ありません*。この変更の理由は、[Media Library](/ja/x-ads-api/creatives#media-library) にメディアを追加する際に、これらのアセットが *自動的に* アカウントメディアエンティティとして追加されるケースがあり、すでに存在するアセットをアカウントメディアリソースに再度追加しようとするとエラーになるためです。これは次のケースで発生します。

* Media Library に追加された `AMPLIFY_VIDEO` アセットは、自動的に `PREROLL` クリエイティブタイプのアカウントメディアアセットとして追加されます。
* 特定の寸法の画像を Media Library に追加すると、自動的にアカウントメディアアセットとして追加されます。クリエイティブタイプ（例: `INTERSTITIAL`）は画像の寸法に依存します。（寸法については、[Enumerations](/ja/x-ads-api/introduction) ページを参照してください。）

<div id="v4">
  ### v4
</div>

Ads API のバージョン 4 は 2018 年 8 月 28 日にリリースされました。

このリリースには、より堅牢なオーディエンス処理バックエンドを基盤とした新しい API インターフェースを含む、[Audiences](/ja/x-ads-api/audiences) プロダクトの改善が含まれます。バージョン 4 には、ユーザー、アカウント、税務設定を管理するための一連のエンドポイントも含まれています。加えて、accounts/:account&#95;id/videos エンドポイントは非推奨となります。このリリースには、いくつかの軽微なパラメータおよびレスポンス名の変更も含まれます。

バージョン 3 と同様に、6 か月の移行期間を設けています。**2019-02-28** をもって、Ads API のバージョン 3 は利用できなくなります。サービス中断を回避するため、すべてのパートナーができるだけ早く最新バージョンの API に移行することを推奨します。バージョン管理方針の詳細については、[Versions](/ja/x-ads-api/introduction) ページを参照してください。

#### New

***Audience API***

新しい Audiences API は、堅牢性と信頼性が向上した新しいオーディエンス処理バックエンド上に構築されています。この新しいエンドポイントにより、パートナーは 1 人のユーザーに対して複数のユーザー識別子の種類を提供できるようになり、これにより照合に追加のシグナルを使用できるようになります。新しい Audience エンドポイントのリファレンスドキュメントは[こちら](/ja/x-ads-api/audiences)にあります。今年残りの期間を通じて、このプロダクトに対するアップデートと改善を継続してリリースしていく予定です。

以下のエンドポイントは、機能が重複しているため v4 では利用できなくなります（v3 では引き続き動作し、v3 が利用できなくなった時点で完全に**廃止**されます）。

* TON Upload:
  * GET accounts/:account&#95;id/tailored&#95;audience&#95;changes
  * GET accounts/:account&#95;id/tailored&#95;audience&#95;changes/:tailored&#95;audience&#95;change&#95;id
  * POST accounts/:account&#95;id/tailored&#95;audience&#95;changes
  * PUT accounts/:accounti&#95;d/tailored&#95;audiences/global&#95;opt&#95;out
* Real Time Audiences:
  * POST tailored&#95;audience&#95;memberships

最後に、バージョン 4 では、`list_type` パラメータは *すべての* [Tailored Audiences エンドポイント](/ja/x-ads-api/audiences) におけるリクエストおよびレスポンスから削除されます。

***Settings Endpoints***

アカウント管理者は、ユーザー設定、アカウント設定、および税設定を構成および更新できるようになりました。[User settings](/ja/x-ads-api/campaign-management#get-accounts-account-id-user-settings-user-id) は、特定の広告アカウントに対するユーザー固有の連絡先設定に対応します。[PUT accounts/:account&#95;id](/ja/x-ads-api/campaign-management#put-accounts-account-id) エンドポイントを使用すると、広告主はアカウント名と業種を更新できるようになりました。最後に、[tax settings](/ja/x-ads-api/campaign-management#get-accounts-account-id-tax-settings) エンドポイントにより、付加価値税（VAT）が課税される国の広告主は、会社名、住所、VAT ID、アカウントが広告主本人に帰属するか、広告主に代わって出稿する代理店に帰属するかといった情報を更新できます。

  ### v3
</div>

Ads API のバージョン 3 は、2018 年 2 月 1 日に[リリースされました](https://devcommunity.x.com/t/ads-api-version-3/100732)。Ads API のバージョン 2 は、2018 年 8 月 1 日にサポート終了となります。

このリリースには、新しい Audience Intelligence プロダクト、Media Library へのアクセス、カードワークフローの改善が含まれます。また、[PUT accounts/:account&#95;id/targeting&#95;criteria](/ja/x-ads-api/campaign-management#get-accounts-account-id-targeting-criteria) エンドポイントの非推奨化も発表しました。最後に、バージョン 3 では、いくつかのパラメータおよびレスポンスの軽微な変更と、バッチサイズ上限の引き下げが行われています。

[バージョン 2](https://devcommunity.x.com/t/ads-api-version-2/90360) と同様に、パートナーの皆様には移行のために **6 か月** の期間を提供します。**2018-08-01** に Ads API の v2 は停止されます。すべてのパートナーおよび開発者の皆様には、できるだけ早く v3 へ移行することをお勧めします。

**Audience Intelligence**

Audience Intelligence は、特定の X オーディエンスに最も関連性の高い上位のハッシュタグ、@handle、イベントに関するリアルタイムなインサイトを提供します。たとえば、「US 在住の 18〜34 歳男性」と入力すると、このオーディエンスにおいて #nintendoswitch、#cardinal、@ricegum がトレンドになっていることがわかります。

Audience Intelligence の [エンドポイント](/ja/x-ads-api/audiences) では、次の機能を提供します。

* 入力オーディエンスを指定して、関連性の高い上位のハッシュタグ、@handle、イベントを取得します。
* 入力オーディエンスを指定して、年齢、性別、世帯収入などの主要なデモグラフィック情報を取得します。
* キーワードを指定して、ポスト数の時系列データを取得します。

**Media Library**

[Media Library](/ja/x-ads-api/creatives#get-accounts-account-id-media-library) は、広告アカウント向けに画像、GIF、動画を管理する機能を提供します。これらのメディアオブジェクトは、ポストやカードの作成に利用できます。また、同じアセットを複数回アップロードする必要がないよう、複数のクリエイティブで再利用することも可能です。

Media Library 内のオブジェクトは media&#95;key によって識別されます。media&#95;key は、たとえば 13&#95;875943225764098048 のような形式の文字列です。Ads API では、すべてのメディアに対して media&#95;key を使用する方向に移行しています。

**カードワークフローの改善**

すべてのカードエンドポイントで media&#95;key がサポートされるようになりました。これにより、Media Library 内のオブジェクトを使用してカードを作成または更新できるようになります。

さらに、新たに 2 つのエンドポイントを導入し、[カード詳細の取得](/ja/x-ads-api/creatives#fetching-cards) を可能にしました。これらのエンドポイントは、たとえば card&#95;uri または id を指定することで、ポストや予約済みポストで使用されているカードを検索するために使用できます。これは従来は不可能でした。

  #### その他の変更点
</div>

これらの新機能に加えて、バージョン 3 には以下の変更が含まれます。

**新規**

* [GET insights/keywords/search](/ja/x-ads-api/audiences) エンドポイントのレスポンスに、入力キーワードに関連する 30 個のキーワードを含む `related_keywords` 属性が追加されました。

**変更**

* ターゲティング条件のバッチの最大サイズが 500 になりました。
* `card_uri` と `preview_url` レスポンス属性は、相互に排他的になりました。カードに `card_uri` がある場合、`preview_url` は `null` になります。カードに `card_uri` がない場合は、`preview_url` のみが返されます。
  * 2018-01-29 以降に作成されたすべてのカードには `card_uri` が付きます。
  * バージョン 4 までに、既存のすべてのカードに `card_uri` が付与されます。
* 5:2 画像を使用したカードを新規に *作成* することはできなくなりました。既存の 5:2 画像ベースのカードは引き続き動作しますが、パートナーの皆様には、（サポートされている場合は）より高いパフォーマンスを発揮する 1.91:1 または 1:1 のアスペクト比の利用に切り替えることを推奨します。

**削除**

* [PUT accounts/:account&#95;id/targeting&#95;criteria](/ja/x-ads-api/campaign-management#get-accounts-account-id-targeting-criteria) エンドポイントは利用できなくなりました。この変更を行ったのは、このエンドポイントのリプレース動作が広告主に混乱を招き、1 回に 1 つのリソースを更新する他の PUT エンドポイントと一貫していなかったためです。代わりに、[POST batch/accounts/:account&#95;id/targeting&#95;criteria](/ja/x-ads-api/campaign-management#post-batch-accounts-account-id-targeting-criteria) エンドポイントを使用してください。このエンドポイントは、1 回のリクエストでターゲティングの追加と削除の両方が可能になるなど、より高い柔軟性を提供します。
* `paused` レスポンス属性は、ファンディングインストルメントについては返されなくなりました。代わりに、そのファンディングインストルメントが一時停止されているかどうかを判断するには、`entity_status` レスポンス属性を参照してください。さらに、`paused` と `cancelled` は同じ値に対応するため、`cancelled` もレスポンスでは返されなくなりました。
* [GET accounts/:account&#95;id/tweet/preview](/ja/x-ads-api/creatives#get-accounts-account-id-tweets) エンドポイントから `card_id` パラメータを削除しました。
* 削除済みの予約済みポストを取得することはできないため、`with_deleted` パラメータはサポートされなくなりました。
* 次のエンドポイントから `draft_only` パラメータを削除しました。これらのエンティティはドラフト状態になることがないためです。
  * [GET accounts/:account&#95;id/targeting&#95;criteria](/ja/x-ads-api/campaign-management#get-accounts-account-id-targeting-criteria)
  * [GET accounts/:account&#95;id/promoted&#95;tweets](/ja/x-ads-api/campaign-management#get-accounts-account-id-promoted-tweets)
  * [GET accounts/:account&#95;id/promoted&#95;accounts](/ja/x-ads-api/campaign-management#get-accounts-account-id-promoted-accounts)
  * [GET accounts/:account&#95;id/media&#95;creatives](/ja/x-ads-api/campaign-management#get-accounts-account-id-media-creatives)
  * [GET accounts/:account&#95;id/preroll&#95;call&#95;to&#95;actions](/ja/x-ads-api/creatives#get-accounts-account-id-preroll-call-to-actions)

**注意**

Video Website Card と予約済みポストは、どちらもベータ版ではなくなりました。ローンチ以降に予約済みポストに対して行った変更については、[このスレッド](https://devcommunity.x.com/t/announcement-update-to-scheduled-tweets/94869) を参照してください。これには、予約済みポスト向けに HTML の[プレビュー](https://devcommunity.x.com/t/announcement-tweet-preview-improvements/96577) を生成できる機能も含まれます。

  * `estimated_daily_spend_local_micro`
* Tailored Audiences のデータ型変更
* Tailored Audiences の `data_type` は、当社のすべてのレスポンスにおいて `tailored_audiences` から `tailored_audience` に変更されました。
* 共有 Tailored Audiences が、*API 限定*のベータ版として利用可能になりました。共有 Tailored Audiences を使用すると、1 つのオーディエンスを複数の広告アカウントで共通利用できます。複数の広告アカウント間で共有したい Tailored Audience の権限を管理するには、[POST accounts/:account&#95;id/tailored&#95;audiences/:tailored&#95;audience&#95;id/permissions](/ja/x-ads-api/audiences) エンドポイント（および関連エンドポイント）を使用してください。
* 広告主アカウントのパフォーマンス[アナリティクス](/ja/x-ads-api/analytics)の収集方法が大幅に改善されました。
* [X Ads API ベストプラクティス](/ja/x-ads-api/analytics#best-practices)に沿うため、今後は[同期型の統計情報エンドポイント](/ja/x-ads-api/analytics#get-stats-accounts-account-id)で取得できるデータは最大**7日間**分までに制限されます。
* メトリクスの取得を簡素化するために、`metrics` パラメータを新しい `metric_groups` パラメータに置き換えました。開発者は、特定のリクエストに対してどのメトリクスグループを返してほしいかを必ず指定してリクエストする必要があります。
  * 対象のエンティティに適用できないメトリクスをリクエストした場合、それらはレスポンスから除外され、`null` 値として表されます。*これらのメトリクスはアナリティクスのコスト上限にはカウントされません。*
* レスポンスは**大幅に単純化**され、UI におけるメトリクスの表示方法とこれまで以上に近い形で整合するようになりました。
  * 以前は、配置ロケーションごとに個別のメトリクスを提供していました（検索でのプロモーションポスト、タイムラインでのプロモーションポスト、プロフィールとポスト詳細でのプロモーションポスト、X Audience Platform 内でのプロモーションポスト）。今後は、それぞれについて標準化されたメトリクスのセットを返します（`promoted_tweet_timeline_impressions`、`promoted_tweet_search_impressions`、`promoted_tweets_profile_impressions`、`promoted_tweets_tpn_impressions` を返す代わりに）。これらは、リクエスト時に以下のいずれかのカテゴリで、単一のメトリクス `impressions` として返されます（これはすべてのメトリクスに適用されます）。
  * `ALL_ON_TWITTER`
  * `PUBLISHER_NETWORK`
  * リクエストを行うと、UI 上の値と一致させやすくするため、単一の `impressions` メトリクスのみが返されます。
  * `ALL_ON_TWITTER` と `PUBLISHER_NETWORK` の両方のデータを取得するには、2 回クエリを実行する必要があります。これらは組み合わせることはできません。
* [非同期統計情報エンドポイント](/ja/x-ads-api/analytics#asynchronous-analytics) が、開発者の皆さまからいただいたフィードバックをもとにご利用いただけるようになりました。
  * 統計情報を*非同期で*リクエストするための新しい一連のエンドポイント。すぐには不要なデータや履歴データの取得に使用できます。
  * 新しい単一のエンドポイントを使って統計ジョブをキューに登録します。リソースが許す限り、リクエストされたデータを取得します。
  * データが利用可能かどうかを判定するために、ジョブステータスエンドポイントにクエリを投げることができます。
  * データが利用可能になると、同期エンドポイントからのレスポンスと同じ構造の JSON レスポンスをダウンロードするための取得用 ID を提供します。
  * 1 つのジョブで、最大 **90 日間**、最大 **20 個のエンティティ** のデータをクエリできます。
* analytics v1 への移行ガイドをご覧ください。そこには v0 メトリクスと v1 メトリクスの対応関係（マッピング）が記載されています。
* Sandbox の改善点 * Sandbox 環境で[複数のテスト用広告アカウントを作成](/ja/x-ads-api/campaign-management#post-accounts)できるようになりました。 * Sandbox 環境内のテスト広告アカウント向けに、複数のファンディングインストゥルメントを作成できるようになりました。これにより、すべてのファンディングインストゥルメントタイプでテストできます。以前はテストに利用できるファンディングインストゥルメントは `CREDIT_CARD` のみでした。 * ベータ機能をテストしたい場合は、Sandbox 環境でアカウント単位に機能をオン／オフできるようになり、ニーズに応じたテストが可能になりました。
