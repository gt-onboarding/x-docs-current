<div id="overview">
  ## 概要
</div>

クリエイティブとは、キャンペーンでプロモーションできるあらゆるエンティティを指します。ポストにはテキスト、画像、GIF、動画、またはカードを含めることができます。カードには画像または動画を含めることができます。

画像、GIF、または動画のクリエイティブは、[POST media/upload](/ja/x-api/media/upload-media#post-media-upload)（画像のみをサポートするシンプルなアップロードエンドポイント）または POST media/upload (chunked) エンドポイントのいずれかを使用してアップロードします。これらは次のオブジェクトに追加できます。

カード:

* POST accounts/:account&#95;id/cards
  ポスト:
* POST accounts/:account&#95;id/tweets - ポストにカードを追加するには、card&#95;uri パラメータを使用します。
  予約済みポスト:
* POST accounts/:account&#95;id/scheduled&#95;tweets

カードの詳細については、Cards ページを参照してください。Promoted Video ページでは、動画をカードまたはポストに関連付ける方法の詳細を説明しています。

<div id="cards">
  ### カード
</div>

Ads API は、ポストで利用でき、その後キャンペーンでプロモーションできる複数のカードタイプをサポートしています。**注意**: 一度ポストを公開すると、カードの詳細は公開情報となります。これには、そのカードを所有しているユーザーに関する情報が含まれる場合があります。

<div id="image">
  #### 画像
</div>

[Cards](/ja/x-ads-api/creatives#cards-2) で使用されるアセットには、以下の画像仕様が適用されます。画像は 3MB 以下で、幅が少なくとも 800px である必要があります。また、サポートされる縦横比（幅:高さ）は次のとおりです。

* Website: 1:1 および 1.91:1
* Image App Download: 1:1 および 1.91:1
* Poll: 1.91:1
* Image Conversation: 1.91:1
* Image Direct Message: 1.91:1

サポートされる画像フォーマットは .bmp、.jpeg、.png です。

<div id="video">
  #### 動画
</div>

以下の動画仕様は、[Cards](/ja/x-ads-api/creatives#cards-2) で使用されるアセットに適用されます。サポートしているアスペクト比（横:縦）は次のとおりです。

* Video Website: 16:9 および 1:1
* Video App Download: 16:9 および 1:1
* Poll: 16:9
* Video Conversation: 16:9
* Video Direct Message: 16:9

<div id="promoted-video">
  ### プロモート動画
</div>

このドキュメントでは、Ads API を使用して動画をアップロードし、プロモーションを行う手順の概要を簡潔に説明します。

Ads API は、[Tweets](/ja/x-ads-api/creatives#tweets-2) 内のプロモート動画と、次のカードでのプロモート動画をサポートしています。

* [Video Website](https://devcommunity.x.com/t/ads-api-version-11/168814)
* [Video App Download](https://devcommunity.x.com/t/ads-api-version-11/168814)
* [Video Conversation](/ja/x-ads-api/creatives#video-conversation-cards)

まず、[POST media/upload (chunked)](https://developer.x.com/en/docs/media/upload-media/api-reference/post-media-upload-init.html) エンドポイントを使用して動画をアップロードします。`media_id` を使用して、POST accounts/:account&#95;id/videos エンドポイントにより、その動画を広告アカウントに関連付けます。動画の `id` は `media_key` と呼ばれることもあり、後続のリクエストで使用されます。これは、整数で始まり、その後にアンダースコアが続き、末尾が long 型の値になる文字列です。例としては、`13_875943225764098048` のような形式になります。

<div id="promoted-video-in-tweets">
  #### ポストでのプロモ動画
</div>

ポストを作成するには、動画の `id` とあわせて [POST accounts/:account&#95;id/tweet](/ja/x-ads-api/creatives#get-accounts-account-id-tweets) エンドポイントを使用します。このステップでは、動画のタイトル、説明文、コールトゥアクション (CTA) も指定できます。これらの値はユーザーに表示される内容です。

<div id="promoted-video-in-cards">
  #### カードにおけるプロモーション動画
</div>

Video App Download カードおよび Video Conversation カードでは、ポスター画像を追加できます。これらのカードで使用する画像は、[POST media/upload](https://developer.x.com/en/docs/media/upload-media/api-reference/post-media-upload.html) エンドポイントを使ってアップロードします。

次のいずれかのエンドポイントを使用してカードを作成します。

* [POST accounts/:account&#95;id/cards/video&#95;website](https://devcommunity.x.com/t/ads-api-version-11/168814)
* [POST accounts/:account&#95;id/cards/video&#95;app&#95;download](https://devcommunity.x.com/t/ads-api-version-11/168814)
* [POST accounts/:account&#95;id/cards/video&#95;conversation](https://devcommunity.x.com/t/ads-api-version-11/168814)

動画の `id` と、必要に応じて画像の `media_id`（ポスター画像用）を指定します。

最後に、POST accounts/:account&#95;id/tweet エンドポイントを使用してポストを作成します。カードは `card_uri` パラメータを使用してポストに添付されます。

<div id="general-information">
  #### 一般情報
</div>

API を通じた動画アップロードに関する詳細なガイドについては、[動画アップロードガイド](https://developer.x.com/en/docs/media/upload-media/uploading-media.html) を参照してください。

動画はプレロールアセットとしてプロモーション配信することもできます。詳しくは [Video Views Pre-roll Objective ガイド](/ja/x-ads-api/campaign-management#video-views-preroll-objective) を参照してください。

* （2015-10-22 時点）プロモーションコンテンツで使用する動画をアップロードする場合、[POST media/upload (chunked)](https://developer.x.com/en/docs/media/upload-media/api-reference/post-media-upload-init.html) エンドポイントへのすべての `INIT` コマンドリクエストで、`media_category` パラメータに `amplify_video` を設定する必要があります。このパラメータを使用すると、動画が非同期で事前処理され、プロモーションコンテンツで使用できるように準備されます。動画アップロード後の非同期処理の完了を確認するには、`STATUS` コマンドを使用できます。
* 現在許可されているプロモーション動画の最大長は 10 分で、ファイルサイズは 500MB 以下です。
* アップロードする動画は mp4 または mov 形式である必要があります。
* アップロードされた動画の処理は一般的に速く行われますが、処理時間は動画の長さやファイルサイズによって変動します。
* アップロードするポスター画像は png または jpg 形式である必要があります。アスペクト比やサイズの要件はありませんが、ポスター画像は動画プレーヤーに合わせて調整されます。

<div id="scheduled-tweets">
  ### 予約済みポスト
</div>

<div id="introduction">
  #### はじめに
</div>

予約ポストを使用すると、広告主やユーザーは、後で公開されるようにスケジュールされたポストを作成できます。これらのポストを作成および管理できるだけでなく、API を使用して、ポストが公開されたタイミングでプロモーションできるように、これらのポストをラインアイテムに関連付けることもできます。これにより、広告主はネイティブポストをあらかじめ作成し、主要な施策に先立ってキャンペーンクリエイティブを計画できます。たとえば、新製品の発表と同時に公開されるようにポストクリエイティブを事前に用意しておくことが可能です。

Scheduled Tweets API エンドポイントによって提供される機能は、次のとおりです。

* 新規の予約ポストの作成・変更・表示
* 予約ポストをラインアイテムに関連付け
* 既存の予約ポストの検索および管理
* 予約ポストが公開された後、そのライブポストの `id` を取得

  #### 予約ポストの表示
</div>

* [GET accounts/:account&#95;id/scheduled&#95;tweets/preview/:scheduled&#95;tweet&#95;id](/ja/x-ads-api/creatives#scheduled-tweets-2)（既存の予約ポストを表示）

予約ポストは「ライブ」ポストとは別個のエンティティである性質上、これらのポストを新規作成または編集する際には 2 種類のバリデーションが実行されます。最初のバリデーションルールのセットは、予約ポストの作成ステップの際に、具体的には以下のとおり実行されます。

<div id="scheduled-tweet-create">
  #### 予約ポストの作成:
</div>

* 認証済みユーザーが、指定された @handle に対してオーガニックポストを作成する権限を持っていることを確認してください。プロモーション専用ポストの作成権限を得るには、認証済みユーザーが [Tweet composer permissions](https://developer.x.com/content/developer-twitter/en/docs/ads/campaign-management/api-reference/authenticated-user-access#get-accounts-account-id-authenticated-user-access) を持つアカウントユーザーである必要があります。
* `scheduled_at` の時刻を基準とした 15 分間の時間枠内で、作成が予約されているポストが 30 件を超えていないことを確認してください。`SCHEDULED_TWEET_LIMIT_EXCEEDED` エラーメッセージは、将来の同一 15 分間の時間枠内に予約された予約ポストが多すぎることを示します。広告主は既存の予約ポストを削除するか、`scheduled_at` の時間を前後に調整する必要があります。

<div id="scheduled-tweet-goes-live">
  #### スケジュール済みポストが「公開」されるタイミング:
</div>

* これらのバリデーションルールは `scheduled_at` 時刻に実行され、API で通常のポストを作成する際に適用されるルールと同一です。たとえば、スケジュール済みポストに画像と GIF が両方含まれている場合、そのスケジュール済みポストは公開されず、`scheduled_status` が `FAILED` に設定されます。

<div id="workflow">
  #### ワークフロー
</div>

**新しい予約ポストを作成する**

新しい予約ポストは、[POST accounts/:account&#95;id/scheduled&#95;tweets](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/scheduled-tweets#post-accounts-account-id-scheduled-tweets) エンドポイントを使用して作成できます。このエンドポイントは、必須パラメータとして `scheduled_at` 時刻と、ポスト内にメディアエンティティが含まれない場合のポストの `text` を受け取ります。さらに、このエンドポイントでは、`as_user_id` パラメータを使用して別の @handle を代行して予約ポストを作成したり、カード（`card_uri`）やメディア（`media_ids`）を追加したりできる、いくつかの追加オプションを提供します。なお、1つのポストには同じタイプのエンティティのみを含めることができます。つまり、Video、GIF、Image のいずれか一種類のみです。`nullcast` パラメータは、そのポストが「Promoted-Only」ポストかどうかを制御します。新しく作成されたすべての予約ポストは、デフォルトで「Promoted-Only」（`nullcast=true`）です。`nullcast=false` の場合は、オーガニックの予約ポストが作成されます。

予約ポストが正常に作成されると、レスポンスには、その予約ポスト自体の一意の識別子を示す `id` フィールドが含まれます。このフィールドに加えて、`tweet_id` という別のフィールドも返されます。このフィールドは最初は `null` ですが、ポストが公開されると、「ライブ」ポストの ID が設定されます。

```
twurl -H 'ads-api.x.com' -X POST "/6/accounts/:account_id/scheduled_tweets" -d 'scheduled_at=2017-12-24T23:59:00Z&text=Happy Holidays!'
```

これにより、次のような予約ポストが作成されます。

```json
{
  "request": {
    "params": {
      "text": "Happy Holidays!",
      "scheduled_at": "2017-12-24T23:59:00Z"
    }
  },
  "data": {
    "completed_at": null,
    "id_str": "917507899668099072",
    "text": "Happy Holidays!",
    "user_id": "3271358660",
    "scheduled_status": "SCHEDULED",
    "id": 917507899668099100,
    "nullcast": true,
    "created_at": "2017-10-09T21:51:44Z",
    "scheduled_at": "2017-12-24T23:59:00Z",
    "card_uri": null,
    "updated_at": "2017-10-09T21:51:44Z",
    "tweet_id": null,
    "media_keys": []
  }
}
```

この予約ポストが公開されると、`tweet_id` フィールドには実際に公開されたポストの ID が設定されます。

**スケジュール済みポストをラインアイテムに関連付ける**

スケジュール済みポストはオーガニックポストの作成に利用できますが、パートナーは「プロモーション専用」（`nullcast=true`）ポストを作成することもでき、どちらの場合もラインアイテムに関連付けることができます。この操作を行いやすくするために、[POST accounts/:account&#95;id/scheduled&#95;promoted&#95;tweets](https://developer.x.com/content/developer-twitter/en/docs/ads/campaign-management/api-reference/scheduled-promoted-tweets#post-accounts-account-id-scheduled-promoted-tweets) エンドポイントも提供しています。このエンドポイントでは、1 回の API コールにつき 1 つのスケジュール済みプロモーションポストのみをラインアイテムに関連付けることができます。同一のラインアイテムに複数のスケジュール済みポストを関連付けるには、複数回の API コールが必要です。

既存のスケジュール済みプロモーションポストを変更することはできない点に注意してください。

```
twurl -H 'ads-api.x.com' -X POST "/6/accounts/:account_id/scheduled_promoted_tweets" -d 'line_item_id=a44qc&scheduled_tweet_id=917507899668099072'
```

```json
      "line_item_id": "a44qc",
      "scheduled_tweet_id": 917507899668099100,
      "account_id": "aaaaa"
    }
  }
}
```

このエンドポイントは、指定された Scheduled Tweet と line item の間に関連付けを作成するだけです。campaign/line item のフライト期間の日付が有効になると、その line item は対応する「ライブ」の Tweet の配信を自動的に開始します。このステップでは、Scheduled Tweet が `SCHEDULED` 状態であること、および指定された Scheduled Tweet が指定された objective に対して有効であることは検証しますが、それ以外の検証は行われません。line item と Scheduled Tweet に適用される残りの検証ルールは、Tweet が「ライブ」になるタイミングで実行されます。

campaign の配信に問題が発生しないようにするため、Scheduled Tweet の `scheduled_at` は campaign/line item のフライト期間より前の時刻に設定することを推奨します。

たとえば、Scheduled Tweet が campaign の開始日より後にライブになるよう設定されている（かつ、1 つの line item に 1 つの Tweet だけが関連付けられている）とします。この場合、campaign は `ACTIVE` になりますが、Scheduled Tweet はまだライブではないため、配信可能なクリエイティブが存在しないことになります。

**Scheduled Tweet Management**

残りのエンドポイント群により、API 利用者はすべての Scheduled Tweet および Scheduled Promoted Tweet を管理できます。これらの API を使用して、すべての Scheduled Tweet の一覧を、任意で指定した state でフィルタリングして取得したり、指定した Scheduled Tweet をその `id` によって取得したりできます。

<div id="what-happens-when-a-scheduled-tweet-goes-live">
  #### 予約ポストが公開されるとどうなりますか？
</div>

ある予約ポストが公開されるタイミング、言い換えると `scheduled_at` の時刻になると、次の更新が行われます。

* 「ライブ」ポストが作成されますが、最大 1 秒程度の遅延が発生する場合があります
* `tweet_id` が次のエンティティに追加されます:
  * 予約ポスト
  * プロモーション予約ポスト
  * 新しいプロモーションポストのエンティティが作成されます

<div id="best-practices">
  #### ベストプラクティス
</div>

予約ポストを作成またはプロモーションする際は、次のベストプラクティスに従うことを推奨します。

* 予約ポストを作成する際に、ポストが有効な内容になっていることを確認してください（例：1つのポストに添付できるメディアは画像・動画・GIF のいずれか1種類のみで、複数種類を同時に添付することはできません）
* キャンペーンのフライト期間（`start_time` と `end_time`）が、予約ポストの `scheduled_at` 時刻と一致していることを確認してください
* 予約ポストは、現在から1年先（365 日）を超える日時には設定しないでください
* 予約ポストのプレビューは現在サポートされていません（作成前に予約ポストをプレビューすることはできません）

<div id="request-parameters">

</div>

すべての Media Library の POST リクエストには、メディア識別子が必要です。この値はアップロード手順の際に返されます。上記の例のように `media_id` を使用する場合は、`media_category` も指定する必要があります。指定できるカテゴリ値は 4 つあり、`AMPLIFY_VIDEO`、`TWEET_GIF`、`TWEET_IMAGE`、`TWEET_VIDEO` です。

<div id="attributes">
  #### 属性
</div>

Media Library では、media&#95;key という概念を正式に導入しています。これはライブラリ内のオブジェクトに対する一意の識別子です。media&#95;key は、13&#95;875943225764098048 のような形式の文字列値です。これらはすべてのカード系エンドポイントで完全にサポートされています。

さらに、Media Library のレスポンスには、文字列として表現された media&#95;id も含まれます。これは、現在まだ media&#95;key を受け付けないリソース向けに含まれており、[ポスト](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/tweets#post-accounts-account-id-tweet)*、[ポストプレビュー](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/tweets#get-accounts-account-id-tweet-preview)*、[予約ポスト](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/scheduled-tweets#post-accounts-account-id-scheduled-tweets) が該当します。現在、すべての場所で media&#95;key をサポートできるよう取り組んでいます。

aspect&#95;ratio 属性は GIF および動画について返されます。これは、特定のアスペクト比のみを受け付けるカードで使用するメディアをフィルタリングするために利用できます。

*これらのエンドポイントは video&#95;id パラメータをサポートしており、これは media&#95;key です。

<div id="identifying-cards">
  ### カードの識別方法
</div>

<div id="introduction">
  #### はじめに
</div>

Card は、メディアを使用するカスタマイズ可能な広告フォーマットで、Web サイト、アプリ、またはダイレクトメッセージの開始など特定のユーザーエンゲージメントを促すコールトゥアクションと関連付けることができます。Card は、ポスト、予約済みポスト、または下書きポストに付加できます。

Card は、ポストオブジェクト内で `card_uri` または `preview_url` の 2 通りの方法で参照される場合があります。各値の例を以下に示します。

| card&#95;uri | preview&#95;url |
| :--- | :--- |
| card://1043282691834048513 | https://cards.x.com/cards/18ce54d4x5t/68w3s |

**注**: Ads API バージョン 3 以降、新しく作成された Card については、cards レスポンスで生成され返されるのは `card_uri` のみです。

**注**: Ads API バージョン 5 以降、cards レスポンスでは `preview_url` は返されなくなりました。

ポストオブジェクトのレスポンス内でどのタイプの参照が使用されるかは、そのポストがどのように作成されたかによって決まります。言い換えると、`card_uri` リクエストパラメーターを使用してポストが作成された場合、レスポンスには Card URI の値が含まれます。一方で、`preview_url` がポスト本文の一部として含まれていた場合、レスポンスにはプレビュー URL が含まれます。

<div id="identifying-media">
  ### メディアの特定
</div>

<div id="introduction">
  #### はじめに
</div>

メディア（画像、GIF、動画）はポストやカードに追加できます。さらに、動画はプレロールアセットとして利用でき、画像は [X Audience Platform](https://developer.x.com/en/docs/ads/measurement/overview/twitter-audience-platform) 上でプロモーションすることができます。本セクションでは、これらのエンティティ間でメディア参照を特定する方法について説明します。

メディア識別子には、ID とキーの 2 種類があります。それぞれの例を以下に示します。

| **Media ID** | **Media key** |
| :--- | :--- |
| 1029825579531807971 | 13&#95;1029825579531807971 |

Media key は、ID の前に数値のプレフィックスとアンダースコアを付けたものです。

<div id="image">
  #### 画像
</div>

次の表は、各画像関連リソースのレスポンスで現在利用可能な識別子の種類と、それに対応する属性名を示しています。

| **Resource** | **Identifier** | **Attribute(s)** |
| :--- | :--- | :--- |
| Image cards | None |  |
| ポスト | Both | `entities["media"]["id_str"]`  `entities["media"]["media_key"]` |
| Scheduled Tweet | Both | `media_ids` and `media_keys` |
| Draft Tweet | Both | `media_ids` and `media_keys` |
| Account Media | None |  |
| Media Library | Both | `media_id` and `media_key` |

Image cards と Account Media の画像には、メディア識別子への参照は含まれません。ポストにはメディア ID のみが含まれます。Scheduled および Draft ポストには、メディア ID とメディアキーの両方が含まれます。Media Library も同様に両方を返します。

ポストの場合、entities[&quot;media&quot;] 配列内のオブジェクトにある id と id&#95;str フィールドがメディア ID に対応します。1 つのポストに複数の画像が含まれている場合、それぞれのメディアエンティティへの参照は extended&#95;entities[&quot;media&quot;] 内でのみ取得できます。

識別子への参照に加えて、画像の URL へアクセスできることも重要な場合がよくあります。

| **Resource** | **Attribute(s)** | **Format** |
| :--- | :--- | :--- |
| Image cards | `image` | .jpg |
| ポスト* | `entities["media"][0]["media_url"]` または `extended_entities["media"][i]["media_url"]` | .jpg |
| Scheduled Tweet | None |  |
| Draft Tweet | None |  |
| Account Media | `media_url` | .jpg |
| Media Library | `media_url` | .jpg |

* この URL の場所は、ポストに 1 枚の画像が含まれるか複数の画像が含まれるかによって異なります。

すべての Image card には、X 画像の URL を含む image レスポンス属性があります（image アプリダウンロードカードの場合、この名前は wide&#95;app&#95;image です）。

ポストの場合、メディア URL の場所は、メディアの種類と使用しているエンドポイントの両方によって異なります。1 枚の画像を含むポストでは、URL は entities[&quot;media&quot;][0][&quot;media&#95;url&quot;] にあります。これは Ads API と Standard API の両方で同じです。一方、複数の画像を含むポストの場合、URL は extended&#95;entities[&quot;media&quot;][i][&quot;media&#95;url&quot;] でのみ見つかります。これは Standard API でのみ利用可能です。

<div id="video">
  #### 動画
</div>

次の表は、各動画関連リソースのレスポンスで現在利用可能な識別子の種類と、それに対応する属性名を示しています。

| **Resource** | **Identifier** | **Attribute(s)** |
| :--- | :--- | :--- |
| Video cards | どちらか一方 | `video_content_id` |
| Video poll cards | なし |  |
| ポスト | 両方 | `entities["media"]["id_str"]`  `entities["media"]["media_key"]` |
| 予約済みポスト | 両方 | `media_ids` と `media_keys` |
| 下書きポスト | 両方 | `media_ids` と `media_keys` |
| Account Media | メディアキー | `video_id` |
| Media Library | 両方 | `media_id` と `media_key` |

Video cards（動画付き投票カードを除く）には `video_content_id` レスポンス属性が含まれますが、返される値の種類に一貫性がありません。メディア ID が返される場合もあれば、メディアキーが返される場合もあります。

動画の URL へのアクセス方法については、以下の表を参照してください。

| **Resource** | **Attribute(s)** | **Format** |
| :--- | :--- | :--- |
| Video cards | `video_url` と `video_hls_url` | .vmap  .m3u8 |
| 動画付きポスト | `extended_entities["media"][i]["video_info"]["variants"][j]["url"]` | .mp4 |
| 予約済みポスト | なし |  |
| 下書きポスト | なし |  |
| Account Media | なし |  |
| Media Library | `media_url` | .mp4 |

Video cards には、`.vmap` と `.m3u8` の URL を持つ `video_url` と `video_hls_url` のレスポンス属性が含まれます。

<div id="media-library">
  #### メディアライブラリ
</div>

メディアアセットに関する追加情報を取得する必要がある場合があります。たとえば、ビデオカードで vmap の URL ではなく mp4 の URL を取得したいケースです。これはメディアライブラリから取得できます。利用可能な情報の詳細については、[Media Library Guide](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/guides/media-library) を参照してください。広告アカウントの FULL promotable user に属するほとんどのアセットはライブラリ内で確認できますが、いくつかの例外があります。

**メディアの取得**

前述のとおり、イメージカードには media ID や media key への参照が含まれていません。その結果、メディアライブラリ経由でそれらのアセットを取得することはできません。これは [Account Media](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/account-media#account-media) の画像についても同様です。

ビデオカードを作成する前に、そのビデオアセットはメディアライブラリ（またはそれ以前の Videos リソース）の一部になっている必要があります。その結果、これらのアセットは常にメディアライブラリから取得可能です。これは Account Media の PREROLL アセットについても同様です。

なお、ポスト内のメディアは必ずメディアライブラリ内に存在することが保証されています。

以下の表は、リソースレスポンスに検索に使用する識別子が含まれているかどうかを考慮しつつ、どのアセットがメディアライブラリから取得可能かをまとめたものです。

| **Resource** | **In the Media Library** |
| :--- | :--- |
| Image cards | No |
| Video cards | Yes* |
| Tweets (any media)** | Yes |
| Scheduled Tweets | Yes |
| Draft Tweets | Yes |
| Account media images | No |
| Account media videos (`PREROLL`) | Yes |

* `video_content_id` がメディアキーであるカードの場合。値がメディア ID の場合でもアセットはメディアライブラリ内に存在しますが、取得するにはその値に数値のプレフィックスとアンダースコアを付加する必要があります。\
** ポストはメディア ID のみを返します。アセットはメディアライブラリ内に存在することが保証されていますが、取得するにはその値に数値のプレフィックスとアンダースコアを付加する必要があります。

**Account Media との連携**

ライブラリに追加されたメディアアセットが自動的に Account Media リソースにも追加されるケースが 2 つあります。

* AMPLIFY&#95;VIDEO アセットがメディアライブラリに追加されると、自動的に Account Media アセットとして、クリエイティブタイプ PREROLL で追加されます。
* 特定の寸法を持つ画像（[enumerations ページ](https://developer.x.com/content/developer-twitter/en/docs/ads/general/overview/enums) の「Creative Types」を参照）がメディアライブラリに追加されると、自動的に Account Media アセットとして追加されます。クリエイティブタイプ（例: INTERSTITIAL）は画像の寸法によって決まります。

<div id="tweets">
  ### ポスト
</div>

<div id="introduction">
  #### はじめに
</div>

X Ads API は、公開済み、予約済み、下書きの 3 つのタイプのポストをサポートしています。

<div id="nullcasted-tweets">
  #### Nullcastされたポスト
</div>

ポストは、nullcast（「Promoted-only」とも呼ばれます）かオーガニックかのいずれかです。Nullcastされたポストは公開されてもユーザーの公開タイムラインには表示されませんが、公開状態であることに変わりはありません。一方、オーガニックポストはユーザーのフォロワーに配信され、ユーザーの公開タイムラインにも表示されます。

**ポストの作成**

3つのポスト作成エンドポイントそれぞれは、nullcastされたポストかオーガニックポストかを選択できる Boolean 型の `nullcast` パラメータをサポートしています。Nullcastされたポストは、そのユーザー自身、またはユーザーに代わってポストを作成する権限を持つ任意のユーザーによって作成できます。オーガニックポストは、[フルプロモーション可能ユーザー](https://developer.x.com/content/developer-twitter/en/docs/tutorials/ads-api-hierarchy-terminology#promotable-users) のみが作成できます。

**ポストの更新**

予約済みポストおよび下書きポストについては、`nullcast` プロパティを更新できます。予約済みポストについては、そのポストの `scheduled_at` 時刻までは編集が可能です。下書きポストは無期限に編集できます。ただし、一度公開されたポストについては、nullcast からオーガニック、またはその逆へ変更することはできません。

<div id="promoting-tweets">
  #### ポストのプロモーション
</div>

プロモーションできるのは、公開済みポストと予約済みポストのみです。これらは nullcast（ナルキャスト）でもオーガニックでも問題ありません。特に制限はありません。広告主は、自身のポストだけでなく、許可を得ている場合は他のユーザーのポストもプロモーションの対象にできます。（詳しくは「[Promoting another user&#39;s Tweets](https://developer.x.com/content/developer-twitter/en/docs/tutorials/promoting-another-users-content)」を参照してください。）

1 つのキャンペーンで複数のポストをプロモーションできます。同様に、1 つのポストを 1 つ以上のキャンペーンでプロモーションすることも可能です。

公開済みポストをプロモーションするには、[POST accounts/:account&#95;id/promoted&#95;tweets エンドポイント](https://developer.x.com/content/developer-twitter/en/docs/ads/campaign-management/api-reference/promoted-tweets#post-accounts-account-id-promoted-tweets)を使用します。これにより、公開済みポストがラインアイテムに関連付けられます。予約済みポストをプロモーションするには、[POST accounts/:account&#95;id/scheduled&#95;promoted&#95;tweets](https://developer.x.com/content/developer-twitter/en/docs/ads/campaign-management/api-reference/scheduled-promoted-tweets#post-accounts-account-id-scheduled-promoted-tweets) エンドポイントを使用します。

<div id="tweet-ids">
  #### ポストID
</div>

公開済み、予約済み、および下書きのポストIDは[数値](https://developer.x.com/content/developer-twitter/en/docs/basics/twitter-ids)で、64ビットの符号なし整数です。たとえば、次の公開済みポストのIDは 1166476031668015104 です。

公開済みまたは予約済みポストをプロモーションすると、対応するプロモーション用のポストエンティティが作成されます。これらのエンティティにはそれぞれ独自のIDがあり、英数字で構成され、base-36 でエンコードされた値として表されます。たとえば、上記の公開済みポストをプロモーションする、つまりそれをラインアイテム 6c62d に関連付けると、次のような API レスポンスが返されます。

<div id="carousels">
  ### カルーセル
</div>

<div id="introduction">
  #### はじめに
</div>

X Ads API では、動画カルーセルおよび画像カルーセルの作成と取得をサポートしています。カルーセルは、2〜6 個のメディアアセットを含めることができるカードタイプです。カルーセルカードは、ユーザーをウェブサイトへ誘導したり、モバイルアプリのインストールを促したりできます。カルーセルの概要、利点、ベストプラクティス、FAQ については、[Carousel Ads on X](https://business.x.com/en/advertising/carousels.html) のページを参照してください。

カルーセルは、他のカードタイプと同様にポスト内で利用でき、そのポストをプロモーションに利用できます。ワークフローは、すでに慣れているものと同じです。

1. メディアをアップロードする
2. カードを作成する
3. ポストを作成する
4. ポストをプロモーションする

唯一の違いは、カードの作成方法です。他のカード作成リクエストがクエリパラメーターを受け付けるのに対して、カルーセルカードの作成リクエストは JSON 形式の POST ボディのみを受け付けます。

<div id="endpoints">
  #### エンドポイント
</div>

Ads API ではカルーセルの作成および取得が可能です。

任意の種類のカルーセルを作成するには、[POST accounts/:account&#95;id/cards](https://developer.x.com/en/docs/twitter-ads-api/creatives/api-reference/cards#post-accounts-account-id-cards) エンドポイントを使用します。カルーセルを取得するには、[GET accounts/:account&#95;id/cards](https://developer.x.com/en/docs/twitter-ads-api/creatives/api-reference/cards#get-accounts-account-id-cards) エンドポイントを使用します。

<div id="json-post-body">
  #### JSON POST Body
</div>

カルーセルは 2 つのコンポーネントを組み合わせて作成します。1 つ目は使用するメディアアセットを指定するコンポーネントです。2 つ目はウェブサイトまたはアプリのいずれかに関する情報を指定するコンポーネントです。

具体的には、カルーセルカードは次のコンポーネントをこの順番で使用して作成します。

* メディアキーの配列を受け取る `SWIPEABLE_MEDIA` コンポーネントを 1 つ
* 次の *いずれか 1 つ*：
  * ウェブサイト情報を指定するための `DETAILS` コンポーネント
  * アプリ情報を指定するための `BUTTON` コンポーネント

`SWIPEABLE_MEDIA` コンポーネントには `media_keys` 配列を含める必要があり、その中で 2～6 個の画像または動画を指定できます。渡されたメディアキーの順番によって、表示される順序が決まります。

```
{
  "type": "SWIPEABLE_MEDIA",
  "media_keys": [
    "13_1089771253848666112",
    "13_1191948012077092867"
  ]
}
```

念のため、[GET accounts/:account&#95;id/media&#95;library](https://developer.x.com/en/docs/twitter-ads-api/creatives/api-reference/media-library#get-accounts-account-id-media-library) エンドポイントにリクエストを送信すると、メディアキーを取得できます。

2 つ目のコンポーネントオブジェクトの構成は、ユーザーをウェブサイトに誘導したいのか、アプリのインストールを促したいのかによって異なります。以下の表は、これら 2 つのオプションをまとめたものです。（**注**: 記載されているキーはすべて必須です。）

|               | **Website**                                                        | **App**                                           |
| :------------ | :----------------------------------------------------------------- | :------------------------------------------------ |
| コンポーネントタイプの指定 | `"type": "DETAILS"`                                                | `"type": "BUTTON"`                                |
| タイトル / ラベル    | `"title": "X"`                                                     | `"label": { "type": "ENUM", "value": "INSTALL" }` |
| 遷移先           | `"destination": { "type": "WEBSITE", "url": "https://www.x.com" }` | `"destination": { "type": "APP", ... }`           |

これらを組み合わせると、ウェブサイトカルーセル用の JSON POST ボディの例は次のとおりです。

```
{
  "name": "website carousel",
  "components": [
    {
      "type": "SWIPEABLE_MEDIA",
      "media_keys": [
        "13_1089771253848666112",
        "13_1191948012077092867"
      ]
    },
    {
      "type": "DETAILS",
      "title": "X",
      "destination": {
        "type": "WEBSITE",
        "url": "https://www.x.com"
      }
    }
  ]
}
```

`BUTTON` コンポーネント内の App destination オブジェクトには、国コードと少なくとも 1 つのアプリ識別子が必要です。オプションでディープリンクを指定できます。これらのフィールドの説明については、[リファレンスドキュメント](https://developer.x.com/en/docs/twitter-ads-api/creatives/api-reference/cards#app-destination)を参照してください。

これらを組み合わせると、以下のようなアプリカルーセル用 JSON POST リクエストボディの例になります。

```
{
  "name": "app carousel",
  "components": [
    {
      "type": "SWIPEABLE_MEDIA",
      "media_keys": [
        "13_1089771253848666112",
        "13_1191948012077092867"
      ]
    },
    {
      "type": "BUTTON",
      "label": {
        "type": "ENUM",
        "value": "INSTALL"
      },
      "destination": {
        "type": "APP",
        "country_code": "US",
        "googleplay_app_id": "com.twitter.android",
        "iphone_app_id": "333903271"
      }
    }
  ]
}
```

<div id="creative-metadata-tagging">
  ### クリエイティブ・メタデータのタグ付け
</div>

<div id="introduction">
  #### はじめに
</div>

このガイドは、クリエイティブパートナー、代理店、およびクリエイティブ開発者が、X キャンペーン内で使用されるアセットにタグ付けを行い、個々のアセットの価値とパフォーマンスをより正確に把握できるようにするためのものです。

**注:** メディアアセットにタグ付けできるのは、そのメディアアセットを作成したパートナーまたは開発者のみです。メディアアセットの利用者がそのメディアアセットを作成していない場合は、メタデータのタグ付けは行わないでください。

Creative Metadata Tagging は、アセットがどこから X にアップロードされるか、またそのアセットを誰がアップロードしているかにかかわらず、Creative Partners によって作成された画像および動画に対してアトリビューションを提供します。クリエイティブアセットとクリエイティブパートナーを関連付けるために、[XMP](https://exiftool.org/TagNames/XMP.html) 標準が使用されます。

<div id="tagging-creative-assets">
  #### クリエイティブアセットへのタグ付け
</div>

次の表は、各画像関連リソースのレスポンスで現在利用可能な識別子の種類と、それに対応する属性名を示しています。クリエイティブアセットにタグを付けるには、タグ付けツールが必要です。[ExifTool](https://exiftool.org/) は、プラットフォームに依存しない Perl ライブラリと、メタ情報の読み取り・書き込み・編集を行うための[コマンドラインアプリケーション](https://exiftool.org/exiftool_pod.html)であり、推奨されています。サポートされているすべての[ファイルタイプ](https://exiftool.org/#supported)を参照してください。

提供されている[ExifTool のインストール手順](https://exiftool.org/install.html)に従ってください。[Homebrew](https://formulae.brew.sh/) から提供されているソフトウェアパッケージもあり、macOS と Linux 向けの [exiftool インストールコマンド](https://formulae.brew.sh/formula/exiftool) を利用することで、インストールをさらに簡略化できます。コマンドラインで `exiftool -ver` を入力し、ツールのバージョン番号が表示されることを確認して、正しくインストールされていることを確認してください。ExifTool のコマンドパラメータについて詳しくは、[ExifTool ドキュメント](https://exiftool.org/)を参照してください。

クリエイティブパートナーは、新規または既存のクリエイティブアセットに対して、X の app&#95;id をメタデータの contributor XMP タグおよび date タグに設定できます。クリエイティブアセットは、[メディアのアップロード](https://developer.x.com/en/docs/twitter-api/v1/media/upload-media/overview)時の既存のサイズ制限に従います。 

**注:** X による contributor XMP タグの使用により、メタデータには X 上のキャンペーン専用の値のみが記録されます。

`exiftool -contributor="<YOUR APP ID>" -creative_file.jpg`

`exiftool -date="<date>" -creative_file.jpg`

app&#95;id は、[Developer Portal](https://developer.x.com/en/portal/dashboard) の Projects &amp; Apps セクションで確認できます。例: 16489123

次の例では、画像に対して app&#95;id を contributor タグとして、date を date タグとして追加しています。

```
app_id:858382169
```

```
date:2022-03-13
```

```
creative_file: eiffel_tower.jpg
```

```
exiftool -contributor=858382169 eiffel_tower.jpg
```

```
   1 image files updated
```

```
exiftool -date=2022-03-13 eiffel_tower.jpg
```

```
  1 image files updated
```

画像に正しくタグが付与されていることを確認します: 

`exiftool -xmp:all -G1 <filename>`

例:

`exiftool -xmp:all -G1 eiffel_tower.jpg`

```
[XMP]        XMP ツールキット            : Image::ExifTool 12.30
```

```
[XMP]        Contributor            : 858382169
```

```
[XMP]        Date                   : 2022:03:13
```

  #### POST accounts/:account&#95;id/cards
</div>

指定されたアカウントに関連付けられた新しいカードを作成します。

カード作成リクエストでは、JSON 形式の POST ボディのみを受け付けます。`Content-Type` は `application/json` に設定する必要があります。

詳細な使用例については、[Carousels Guide](https://developer.x.com/en/docs/twitter-ads-api/creatives/guides/carousels) を参照してください。

<div id="resource-url">
  #### リソース URL
</div>

`https://ads-api.x.com/12/accounts/:account_id/cards`

#### パラメーター

JSON 形式の POST 本文には、カードの `name` と `components` の配列を含める必要があります。`components` はオブジェクトとして表され、カードの広告主向け属性を定義します。

次の例は、ペイロードの一般的な構造を示したものであり、実際には動作しない情報も含まれています。

```
{
  "name": "some name",
  "components": [
    {
      "type": "TYPE_ENUM",
      "key": "value"
    }
  ]
}
```

Additional information on components below.

| Name                              | Description                                                                                                                                                                                                                                              |
| :-------------------------------- | :------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| account&#95;id   *required*       | レバレッジ対象アカウントの識別子です。リソースのパス内に含まれ、[GET accounts](https://developer.x.com/en/docs/ads/campaign-management/api-reference/accounts#get-accounts) を除くすべての Advertiser API リクエストで通常は必須パラメータです。指定されたアカウントは認証済みユーザーに関連付けられている必要があります。  型: string  例: `18ce54d4x5t` |
| name   *required*                 | カードの名前です。最大長: 80 文字。  型: string  例: `component based card`                                                                                                                                                                                               |
| components   *sometimes required* | カードを作成するために使用するコンポーネントを指定します。詳細は後述します。`slides` と同時には指定できません。 **注**: コンポーネントの順序は重要です。  型: array of objects                                                                                                                                                |
| slides   *sometimes required*     | Multi-Destination Carousels を作成するためにこの配列の配列を使用します。各カードをコンポーネントのグループとして表します。各スライドはカードを完全に表現したものである必要があります。`components` と同時には指定できません。 **注**: 各スライドの順序は重要です。  型: array of array                                                                             |

<div id="components">
  #### コンポーネント
</div>

すべてのコンポーネントには、そのオブジェクトのスキーマを決定する `type` フィールドを含める必要があります。Ads API では、コンポーネントタイプをメディアベースと説明ベースのコンポーネントに分類し、次のタイプをサポートしています。

* メディア:
* `MEDIA`: 単一の動画または画像
* `SWIPEABLE_MEDIA`: 2～6 個の動画または画像
* 説明:
* `DETAILS`
* `BUTTON`

各コンポーネントには、（`type` キーに加えて）必須フィールドのセットがあります。これらは次の表に示します。

| Component `type`  | Field                   | Value type       |
| :---------------- | :---------------------- | :--------------- |
| `MEDIA`           | `media_key`             | string           |
| `SWIPEABLE_MEDIA` | `media_keys`            | array of strings |
| `DETAILS`         | `title`   `destination` | string   object  |
| `BUTTON`          | `label`   `destination` | object   object  |

以下は、`components` 配列内における `BUTTON` コンポーネントの例です（`name` キーは意図的に省略しています）。省略記号（…）は、さらに情報を指定する必要がある箇所を示しています。

```
{
  "components": [
    {
      "type": "BUTTON",
      "label": {
        ...
      },
      "destination": {
        ...
      }
    }
  ]
}
```

コンポーネントオブジェクトを指定する順序によって、上から下へのレンダリング順が決まります。カードは、1 つのメディアベースのコンポーネントと、`DETAILS` または `BUTTON` コンポーネントのいずれかを組み合わせて作成する必要があります。説明ベースのコンポーネントはメディアの下に表示され、URL またはモバイルアプリへの遷移先が関連付けられます。

**Label**

Label はボタンに表示されるテキストを定義するため、`BUTTON` コンポーネントにのみ適用されます。Label オブジェクトには必須キーが 2 つあります: `type` と `value` です。`type` は `ENUM` に設定する必要があり、`value` には `BOOK`、`CONNECT`、`INSTALL`、`OPEN`、`ORDER`、`PLAY`、`SHOP` のいずれかを指定できます。

前の例をもとに、次に示すのは `BUTTON` コンポーネント内の `label` オブジェクトです。

```
{
  "components": [
    {
      "type": "BUTTON",
      "label": {
        "type": "ENUM",
        "value": "INSTALL"
      },
      "destination": {
        ...
      }
    }
  ]
}
```

**宛先**

宛先は、広告主がユーザーを誘導したい場所です。`DETAILS` または `BUTTON` コンポーネント内で常に指定が必須です。宛先タイプには `WEBSITE` と `APP` の 2 種類があります。

**注記**: Website 宛先は `DETAILS` コンポーネントでのみ使用でき、APP 宛先は `BUTTON` コンポーネントでのみ使用できます。

**Website 宛先**

| Name              | Description                                                                                       |
| :---------------- | :------------------------------------------------------------------------------------------------ |
| type   *required* | 宛先のタイプで、そのスキーマを決定します。  Type: enum  Possible values: `WEBSITE`                                     |
| url   *required*  | ユーザーをリダイレクトする Web サイトの URL。  Type: string  Example: `https://devcommunity.x.com/c/advertiser-api` |

**App 宛先**

| Name                                                        | Description                                                                                                                                                 |
| :---------------------------------------------------------- | :---------------------------------------------------------------------------------------------------------------------------------------------------------- |
| type   *required*                                           | 宛先のタイプで、そのスキーマを決定します。  Type: enum  Possible values: `APP`                                                                                                   |
| country&#95;code   *required*                               | アプリが販売されている国を示す、[ISO 3166-1 alpha-2](https://en.wikipedia.org/wiki/ISO_3166-1_alpha-2) の 2 文字の国コード。  Type: string  Example: `US`                            |
| googleplay&#95;app&#95;id   *sometimes required*            | Google Play のアプリケーションパッケージ名。  **注記**: 次のうち少なくとも 1 つが必須です: `ios_app_store_identifier` または `googleplay_app_id`。  Type: string  Example: `com.twitter.android` |
| ios&#95;app&#95;store&#95;identifier   *sometimes required* | iOS App Store の識別子。  **注記**: 次のうち少なくとも 1 つが必須です: `ios_app_store_identifier` または `googleplay_app_id`。  Type: string  Example: `333903271`                    |
| googleplay&#95;deep&#95;link   *optional*                   | プロモーション対象の Android アプリへのディープリンク。  **注記**: `googleplay_app_id` が指定されている場合にのみ使用できます。  Type: string                                                            |
| ios&#95;deep&#95;link   *optional*                          | プロモーション対象の iOS アプリへのディープリンク。  **注記**: `ios_app_store_identifier` が指定されている場合にのみ使用できます。  Type: string                                                         |

<div id="post-accountsaccount_idtweet">

</div>

アカウントのプロモーション可能ユーザー（`FULL`、デフォルト）または `as_user_id` パラメータで指定したユーザーとしてポストを作成します。ヌルキャスト（デフォルト）ポストとオーガニックポストの両方の作成をサポートします。ヌルキャストポストは公開タイムラインには表示されず、フォロワーにも配信されません。どちらのタイプもキャンペーンで使用できます。

認証ユーザーがこのアカウントの `FULL` プロモーション可能ユーザーではない場合、[GET accounts/:account&#95;id/authenticated&#95;user&#95;access](https://developer.x.com/en/docs/ads/campaign-management/api-reference/authenticated-user-access#get-accounts-account-id-authenticated-user-access) エンドポイントにリクエストを送信し、このユーザーの代理でポストする権限があるかどうかを確認してください。`TWEET_COMPOSER` 権限がある場合、このエンドポイントを使用して、`FULL` [プロモーション可能ユーザー](https://developer.x.com/en/docs/ads/campaign-management/api-reference/promotable-users#get-accounts-account-id-promotable-users) の代理でヌルキャストポストを作成できます。

メディアのアップロードに [upload.x.com エンドポイント](https://developer.x.com/rest/reference/post/media/upload) を使用する場合、このエンドポイントに渡す `as_user_id` と同じ `user_id` の値を `additional_owners` パラメータとして渡してください。
