<div id="overview">
  ## 概要
</div>

クリエイティブとは、キャンペーンでプロモーションできるあらゆるエンティティを指します。ポストにはテキスト、画像、GIF、動画、またはカードを含めることができます。カードには画像または動画を含めることができます。

画像、GIF、または動画のクリエイティブは、[POST media/upload](/ja/x-api/media/upload-media#post-media-upload)（画像のみをサポートするシンプルなアップロードエンドポイント）または POST media/upload (chunked) エンドポイントのいずれかを使用してアップロードします。これらのクリエイティブは、次のオブジェクトに追加できます。

Cards:

* POST accounts/:account&#95;id/cards\
  Tweets:
* POST accounts/:account&#95;id/tweets - カードをツイート（ポスト）に追加するには、card&#95;uri パラメータを使用します。\
  Scheduled Tweets:
* POST accounts/:account&#95;id/scheduled&#95;tweets

カードの詳細については、Cards ページを参照してください。動画をカードやツイート（ポスト）に関連付ける方法の詳細については、「Promoted Video」ページを参照してください。

<div id="cards">
  ### カード
</div>

Ads API は、ポストで利用でき、その後キャンペーンでプロモーション可能な複数のカードタイプをサポートしています。**注記**: 一度ポストされると、カードの詳細は公開されます。これには、そのカードの所有者であるユーザーに関する情報が含まれる場合があります。

<div id="image">
  #### 画像
</div>

以下の画像仕様は、[Cards](/ja/x-ads-api/creatives#cards-2) で使用されるアセットに適用されます。画像は 3MB 以下で、幅が 800px 以上である必要があります。また、以下の幅:高さのアスペクト比をサポートします。

* Website: 1:1 および 1.91:1
* Image App Download: 1:1 および 1.91:1
* Poll: 1.91:1
* Image Conversation: 1.91:1
* Image Direct Message: 1.91:1

サポートしている画像形式は .bmp、.jpeg、.png です。

<div id="video">
  #### 動画
</div>

以下の動画仕様は、[Cards](/ja/x-ads-api/creatives#cards-2) で使用するアセットに適用されます。サポートしている縦横比は次のとおりです。

* Video Website: 16:9 および 1:1
* Video App Download: 16:9 および 1:1
* Poll: 16:9
* Video Conversation: 16:9
* Video Direct Message: 16:9

<div id="promoted-video">
  ### プロモート動画
</div>

このドキュメントでは、Ads API を通じて動画をアップロードしてプロモートする手順の概要を説明します。

Ads API は、[ポスト内のプロモート動画](/ja/x-ads-api/creatives#tweets-2)および次のカードでのプロモート動画をサポートしています。

* [Video Website](https://devcommunity.x.com/t/ads-api-version-11/168814)
* [Video App Download](https://devcommunity.x.com/t/ads-api-version-11/168814)
* [Video Conversation](/ja/x-ads-api/creatives#video-conversation-cards)

まず、[POST media/upload (chunked)](https://developer.x.com/en/docs/media/upload-media/api-reference/post-media-upload-init.html) エンドポイントを使用して動画をアップロードします。次に、`media_id` を使用して、POST accounts/:account&#95;id/videos エンドポイントを呼び出し、動画を広告アカウントに関連付けます。動画の `id` は `media_key` と呼ばれることもあり、後続のリクエストで使用されます。これは、先頭が整数で、その後にアンダースコアが続き、最後が long 型の値で終わる形式の文字列です。例: `13_875943225764098048`。

<div id="promoted-video-in-tweets">
  #### ポスト内のプロモーション動画
</div>

ポストを作成するには、動画の `id` と併せて [POST accounts/:account&#95;id/tweet](/ja/x-ads-api/creatives#get-accounts-account-id-tweets) エンドポイントを使用します。このステップでは、動画タイトル、説明文、コールトゥアクション（CTA）も指定できます。これらの値はユーザー向けに表示されます。

<div id="promoted-video-in-cards">
  #### カードでのプロモ動画
</div>

Video App Download カードと Video Conversation カードでは、ポスター画像を追加できます。これらのカードで使用する画像は、[POST media/upload](https://developer.x.com/en/docs/media/upload-media/api-reference/post-media-upload.html) エンドポイントを使用してアップロードします。

次のいずれかのエンドポイントを使用してカードを作成します。

* [POST accounts/:account&#95;id/cards/video&#95;website](https://devcommunity.x.com/t/ads-api-version-11/168814)
* [POST accounts/:account&#95;id/cards/video&#95;app&#95;download](https://devcommunity.x.com/t/ads-api-version-11/168814)
* [POST accounts/:account&#95;id/cards/video&#95;conversation](https://devcommunity.x.com/t/ads-api-version-11/168814)

動画の `id` と、必要に応じて画像の `media_id`（ポスター画像用）を指定します。

最後に、POST accounts/:account&#95;id/tweet エンドポイントを使用してポストを作成します。カードは、`card_uri` パラメーターを使用してポストに添付されます。

<div id="general-information">
  #### 一般情報
</div>

API を通じた動画アップロードの詳細については、[Video Upload Guide](https://developer.x.com/en/docs/media/upload-media/uploading-media.html) を参照してください。

動画はプレロールアセットとしてプロモーションすることもできます。詳しくは [Video Views Pre-roll Objective Guide](/ja/x-ads-api/campaign-management#video-views-preroll-objective) を参照してください。

* （2015-10-22 時点）プロモーション用コンテンツで使用する動画をアップロードする場合、[POST media/upload (chunked)](https://developer.x.com/en/docs/media/upload-media/api-reference/post-media-upload-init.html) エンドポイントへのすべての `INIT` コマンドリクエストで、`media_category` パラメータに `amplify_video` を設定する必要があります。このパラメータを使用すると、動画が非同期で事前処理され、プロモーション用コンテンツで使用できるように準備されます。動画アップロード後の非同期処理の完了状況は、`STATUS` コマンドで確認できます。
* 現在許可されているプロモーション動画の最大再生時間は 10 分で、ファイルサイズは 500MB 以下です。
* アップロードする動画は mp4 または mov のいずれかである必要があります。
* アップロードされた動画は一般的にすばやく処理されますが、処理時間は動画の長さやファイルサイズによって変動します。
* アップロードするポスター画像は png か jpg 形式である必要があります。アスペクト比やサイズの要件はありませんが、ポスター画像は動画プレーヤーに合わせて調整されます。

### 予約済みポスト

#### はじめに

予約ポストを使用すると、広告主やユーザーは、後で公開されるようにスケジュールされたポストを作成できます。これらのポストを作成および管理できるだけでなく、API を使用して、それらをラインアイテムに関連付け、ポストが公開された際にプロモーションできるようにすることもできます。これにより、広告主はネイティブポストをあらかじめ作成・用意し、重要な施策に先立ってキャンペーンのクリエイティブを計画できます。たとえば、新製品発表と同時に公開されるようにポストのクリエイティブを事前に用意しておく、といった使い方が可能です。

Scheduled Tweets API エンドポイントで提供される機能は、以下のとおりです。

* 新規の予約ポストを作成、変更、および表示
* 予約ポストをラインアイテムに関連付ける
* 既存の予約ポストを検索および管理する
* 予約ポストが公開されたら、その公開ポストの `id` を取得する

予約ポストは「ライブ」のポストとは別個のエンティティであるという性質上、これらのポストを作成または編集する際には、2 種類の異なる検証が実行されます。最初の検証ルールセットは、予約ポストの作成ステップの際に、特に次のように実行されます。

<div id="scheduled-tweet-create">
  #### Scheduled Tweet Create:
</div>

* 認証済みユーザーが、指定された @handle に対してオーガニックポストを作成する権限を持っていることを検証します。Promoted-Only ポストの作成権限を利用するには、認証済みユーザーが [Tweet composer permissions](https://developer.x.com/content/developer-twitter/en/docs/ads/campaign-management/api-reference/authenticated-user-access#get-accounts-account-id-authenticated-user-access) を持つアカウントユーザーである必要があります。
* `scheduled_at` で指定した時刻の前後 15 分以内に作成が予定されているポストが 30 件を超えていないことを検証します。`SCHEDULED_TWEET_LIMIT_EXCEEDED` エラーメッセージは、同じ将来の 15 分間の時間枠内にスケジュールされたポストが多すぎることを示します。広告主は既存のスケジュール済みポストを削除するか、`scheduled_at` 時刻を前後いずれかに移動する必要があります。

<div id="scheduled-tweet-goes-live">
  #### 予約ポストが「公開」されるとき:
</div>

* これらのバリデーションルールは `scheduled_at` で指定した時刻に実行され、API で通常のポストを作成する際に適用されるルールと同一です。たとえば、予約ポストに画像と GIF の両方が含まれている場合、その予約ポストは公開されず、`scheduled_status` は `FAILED` に設定されます。

<div id="workflow">
  #### Workflow
</div>

**新しい予約ポストの作成**

新しい予約ポストは、[POST accounts/:account&#95;id/scheduled&#95;tweets](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/scheduled-tweets#post-accounts-account-id-scheduled-tweets) エンドポイントを使用して作成できます。このエンドポイントには、必須パラメータとして `scheduled_at` の時刻と、ポスト内にメディアエンティティが含まれていない場合のポストの `text` が必要です。加えて、このエンドポイントには、`as_user_id` パラメータを使用して別の @handle に代わって予約ポストを作成したり、カード（`card_uri`）やメディア（`media_ids`）を追加したりできる、いくつかの追加オプションがあります。なお、1つのポストには同じ種類のエンティティのみを含めることができます。つまり、Video、Gif、Image のいずれか一種類のみです。`nullcast` パラメータは、そのポストが「プロモーション専用」ポストかどうかを制御します。新たに作成されるすべての予約ポストは、デフォルトでは &quot;Promoted-Only&quot;（`nullcast=true`）です。`nullcast=false` の場合は、オーガニックの予約ポストが作成されます。

予約ポストが正常に作成されると、レスポンスには、その予約ポスト自体の一意の識別子を指す `id` フィールドが含まれます。このフィールドに加えて、`tweet_id` という別のフィールドも返されます。このフィールドは最初は `null` ですが、ポストが公開されると、このフィールドには「ライブ」ポストの ID が設定されます。

```
twurl -H 'ads-api.x.com' -X POST "/6/accounts/:account_id/scheduled_tweets" -d 'scheduled_at=2017-12-24T23:59:00Z&text=Happy Holidays!'
```

この操作により、次のような予約ポストが作成されます。

```json
{
  "request": {
    "params": {
      "text": "Happy Holidays!",
      "scheduled_at": "2017-12-24T23:59:00Z"
    }
  },
  "data": {
    "completed_at": null,
    "id_str": "917507899668099072",
    "text": "Happy Holidays!",
    "user_id": "3271358660",
    "scheduled_status": "SCHEDULED",
    "id": 917507899668099100,
    "nullcast": true,
    "created_at": "2017-10-09T21:51:44Z",
    "scheduled_at": "2017-12-24T23:59:00Z",
    "card_uri": null,
    "updated_at": "2017-10-09T21:51:44Z",
    "tweet_id": null,
    "media_keys": []
  }
}
```

この予約ポストが公開されると、`tweet_id` フィールドには「ライブ」ポストの ID が格納されます。

**予約ポストをラインアイテムに関連付ける**

予約ポストはオーガニックポストの作成に使用できますが、パートナーは「プロモーション専用」 (`nullcast=true`) ポストを作成することもでき、いずれもラインアイテムに関連付けることができます。これを可能にするために、[POST accounts/:account&#95;id/scheduled&#95;promoted&#95;tweets](https://developer.x.com/content/developer-twitter/en/docs/ads/campaign-management/api-reference/scheduled-promoted-tweets#post-accounts-account-id-scheduled-promoted-tweets) エンドポイントも提供しています。このエンドポイントでは、1 回の API 呼び出しにつき、1 つのプロモーション用予約ポストのみをラインアイテムに関連付けることができます。同じラインアイテムに複数の予約ポストを関連付けるには、複数回の API 呼び出しが必要です。

既存のプロモーション用予約ポストを変更することはできない点に注意してください。

```
twurl -H 'ads-api.x.com' -X POST "/6/accounts/:account_id/scheduled_promoted_tweets" -d 'line_item_id=a44qc&scheduled_tweet_id=917507899668099072'
```

```json
{
  "data": {
    "line_item_id": "a44qc",
    "id": "26576",
    "created_at": "2017-10-09T22:24:16Z",
    "updated_at": "2017-10-09T22:24:16Z",
    "scheduled_tweet_id": "917507899668099072",
    "tweet_id": null,
    "deleted": false
  },
  "request": {
    "params": {
      "line_item_id": "a44qc",
      "scheduled_tweet_id": 917507899668099100,
      "account_id": "aaaaa"
    }
  }
}
```

This endpoint only creates an association between a given Scheduled Tweet and a line item. Once the campaign/line item flight dates are current, the line item with automatically start serving the corresponding “live” Tweet. While we do validate during this step that the Scheduled Tweet is in the `SCHEDULED` state, and that the given Scheduled Tweet is valid for the given objective, no other validations are run. Any remaining validation rules that apply to the line item and Scheduled Tweet are run when the Tweet goes “live”

このエンドポイントは、指定された予約済みポストとラインアイテムの間に関連付けを作成するだけです。キャンペーン／ラインアイテムのフライト期間が開始すると、そのラインアイテムは対応する「公開」ポストの配信を自動的に開始します。このステップでは、予約済みポストが `SCHEDULED` 状態であること、および指定された予約済みポストが指定のオブジェクティブに対して有効であることは検証しますが、それ以外のバリデーションは行いません。ラインアイテムおよび予約済みポストに適用される残りのバリデーションルールは、ポストが「公開」されるタイミングで実行されます。

In order to ensure that there are no issues with campaign serving it is recommend that the Scheduled Tweet be `scheduled_at` a time prior to the campaign/line item flight dates.

キャンペーン配信に問題が発生しないようにするため、予約済みポストの `scheduled_at` は、キャンペーン／ラインアイテムのフライト日程よりも前の時刻に設定することを推奨します。

For example, let&#39;s say the Scheduled Tweet is set to go live after the campaign start date (and that there is only a single Tweet associated with a single line item), then the campaign will be `ACTIVE`, however given that the Scheduled Tweet is not live yet, there will be no creatives available for serving.

たとえば、予約済みポストがキャンペーン開始日より後に公開されるように設定されており（かつ、1 つのラインアイテムに 1 つのポストのみが紐づいている）とします。この場合、キャンペーン自体は `ACTIVE` になりますが、予約済みポストはまだ公開されていないため、配信可能なクリエイティブが存在しない状態になります。

**Scheduled Tweet Management**

**予約済みポストの管理**

The remaining sets of endpoints allow API consumers to manage all their Scheduled Tweets and Scheduled Promoted Tweets. These APIs can be used to either return a list of all Scheduled Tweets optionally filtered by a given state as well as lookup a given Scheduled Tweet by its `id`.

残りの一連のエンドポイントにより、API 利用者は自身のすべての予約済みポストおよび予約済みプロモーションポストを管理できます。これらの API を使用すると、任意の状態でフィルタしたすべての予約済みポストのリストを取得したり、特定の予約済みポストをその `id` で取得したりできます。

<div id="what-happens-when-a-scheduled-tweet-goes-live">
  #### スケジュール済みポストが公開されたときに何が起こりますか？
</div>

特定のスケジュール済みポストが公開される直前、つまり `scheduled_at` の時刻になると、次の更新が行われます。

* 実際に公開される「ライブ」ポストが作成されますが、最大 1 秒程度の遅延が発生する場合があります
* `tweet_id` が次のエンティティに追加されます：
  * スケジュール済みポスト
  * プロモーション用スケジュール済みポスト
* 新しいプロモーション用ポストのエンティティが作成されます

<div id="best-practices">
  #### ベストプラクティス
</div>

予約ポストを作成またはプロモーションする際は、次のベストプラクティスに従うことを推奨します。

* 予約ポストを作成する際は、ポストが有効な内容になっていることを確認してください（たとえば、1 つのポストには画像・動画・GIF のいずれか 1 種類のみを含めることができ、3 種類を組み合わせて含めることはできません）
* キャンペーンのフライト期間（`start_time` と `end_time`）が、予約ポストの `scheduled_at` の日時と一致していることを確認してください
* 予約ポストは、現在から 1 年（365 日）を超える未来の日時にはスケジュールしないでください
* 予約ポストについては、現在ポストのプレビュー機能はサポートされていません（作成前に予約ポストをプレビューできる機能を指します）

<div id="request-parameters">

</div>

すべての Media Library の POST リクエストには、メディア識別子が必要です。この値はアップロード処理の際に返されます。上記の例のように `media_id` を使用する場合は、`media_category` も指定する必要があります。使用可能なカテゴリ値は 4 種類あります: AMPLIFY&#95;VIDEO、TWEET&#95;GIF、TWEET&#95;IMAGE、TWEET&#95;VIDEO です。

<div id="attributes">
  #### 属性
</div>

Media Library では、`media_key` という概念が正式に導入されています。これはライブラリ内のオブジェクトを一意に識別する識別子です。`media_key` は `13_875943225764098048` のような形式の文字列です。これらはすべてのカード関連エンドポイントで完全にサポートされています。

加えて、Media Library のレスポンスには、文字列として表現された `media_id` も含まれます。これは、現在 media key を受け付けない次のリソース向けに返されます：[ポスト](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/tweets#post-accounts-account-id-tweet)*、[ポストプレビュー](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/tweets#get-accounts-account-id-tweet-preview)*、および [予約ポスト](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/scheduled-tweets#post-accounts-account-id-scheduled-tweets)。すべてのエンドポイントで media key をサポートできるよう対応を進めています。

`aspect_ratio` 属性は GIF と動画に対して返されます。これは、特定のアスペクト比のみを受け付けるカードで使用するメディアをフィルタリングするために利用できます。

*これらのエンドポイントはいずれも `video_id` パラメータをサポートしており、これは media key です。

<div id="identifying-cards">
  ### カードの識別
</div>

<div id="introduction">
  #### Introduction
</div>

カードは、メディアを使用し、ウェブサイトやアプリ、またはダイレクトメッセージの開始など特定のユーザーエンゲージメントを促す行動喚起（call to action）と関連付けることができる、カスタマイズ可能な広告フォーマットです。カードは、ポスト、予約ポスト、下書きポストに追加できます。

カードは、ポストオブジェクト内で `card_uri` か `preview_url` のいずれか一方の方法で参照されます。それぞれの値の例を以下に示します。

| card&#95;uri | preview&#95;url |
| :--- | :--- |
| card://1043282691834048513 | https://cards.x.com/cards/18ce54d4x5t/68w3s |

**注記**: Ads API バージョン 3 以降、新しく作成されたカードに対しては、cards レスポンスで生成・返されるのは `card_uri` のみです。

**注記**: Ads API バージョン 5 以降、cards レスポンスでは `preview_url` は返されません。

ポストオブジェクトのレスポンス内でどちらの参照形式が使われるかは、そのポストがどのように作成されたかに依存します。言い換えると、ポストが `card_uri` リクエストパラメーターを使用して作成された場合、そのレスポンスには card URI の値が含まれます。一方で、`preview_url` がポスト本文の一部として含まれていた場合は、そのレスポンスにはプレビュー URL が含まれます。

| **Media ID** | **Media key** |

| **Resource** | **Identifier** | **Attribute(s)** |
| :--- | :--- | :--- |
| Image cards | None |  |
| Tweet | Both | `entities["media"]["id_str"]`  `entities["media"]["media_key"]` |
| Scheduled Tweet | Both | `media_ids` and `media_keys` |
| Draft Tweet | Both | `media_ids` and `media_keys` |
| Account Media | None |  |
| Media Library | Both | `media_id` and `media_key` |

Image cards と Account Media の画像には、いかなるメディア識別子への参照も含まれません。ポストにはメディア ID のみが含まれます。Scheduled ポストおよび Draft ポストには、メディア ID とメディアキーの両方が含まれます。Media Library も両方を返します。

ポストの場合、entities[&quot;media&quot;] 配列内オブジェクトの id および id&#95;str フィールドがメディア ID に対応します。1 つのポストに複数の画像が含まれている場合は、それぞれのメディアエンティティへの参照は extended&#95;entities[&quot;media&quot;] 内でしか見つかりません。

識別子への参照に加えて、画像の URL にアクセスできることも重要になる場合がよくあります。

| **Resource** | **Attribute(s)** | **Format** |
| :--- | :--- | :--- |
| Image cards | `image` | .jpg |
| Tweet* | `entities["media"][0]["media_url"]` or `extended_entities["media"][i]["media_url"]` | .jpg |
| Scheduled Tweet | None |  |
| Draft Tweet | None |  |
| Account Media | `media_url` | .jpg |
| Media Library | `media_url` | .jpg |

* この URL の位置は、ポストに 1 枚の画像が含まれているか複数含まれているかによって異なります。

すべての Image cards には、X の画像 URL を含む image というレスポンス属性があります（image app download cards の場合、この名前は wide&#95;app&#95;image です）。

ポストの場合、メディア URL の位置は、メディアの種類と使用しているエンドポイントの両方に依存します。単一画像のポストでは、URL は entities[&quot;media&quot;][0][&quot;media&#95;url&quot;] にあります。これは Ads API と Standard API の両方で同様です。ただし、複数画像を含むポストの場合、URL は extended&#95;entities[&quot;media&quot;][i][&quot;media&#95;url&quot;] にしか存在しません。これは Standard API でのみ利用できます。

<div id="video">
  #### 動画
</div>

次の表は、各動画関連リソースのレスポンスで現在利用可能な識別子の種類と、それに対応する属性名を示しています。

| **Resource** | **Identifier** | **Attribute(s)** |
| :--- | :--- | :--- |
| Video cards | どちらか一方 | `video_content_id` |
| Video poll cards | なし |  |
| ポスト | 両方 | `entities["media"]["id_str"]`  `entities["media"]["media_key"]` |
| Scheduled ポスト | 両方 | `media_ids` and `media_keys` |
| Draft ポスト | 両方 | `media_ids` and `media_keys` |
| Account Media | Media key | `video_id` |
| Media Library | 両方 | `media_id` and `media_key` |

Video cards（動画付きの投票カードを除く）には `video_content_id` レスポンス属性が含まれますが、返される値の種類に一貫性がありません。メディア ID が返される場合もあれば、メディアキーが返される場合もあります。

動画の URL の取得方法は、以下のとおりです。

| **Resource** | **Attribute(s)** | **Format** |
| :--- | :--- | :--- |
| Video cards | `video_url` and `video_hls_url` | .vmap  .m3u8 |
| 動画付きポスト | `extended_entities["media"][i]["video_info"]["variants"][j]["url"]` | .mp4 |
| Scheduled ポスト | なし |  |
| Draft ポスト | なし |  |
| Account Media | なし |  |
| Media Library | `media_url` | .mp4 |

Video cards には、.vmap と .m3u8 の URL をそれぞれ持つ `video_url` および `video_hls_url` レスポンス属性が含まれます。

<div id="media-library">
  #### Media Library
</div>

メディアアセットについて、追加情報を取得する必要が生じる場合があります。たとえば Video Card では、vmap の URL ではなく mp4 の URL を取得したいケースがあります。これは Media Library から取得できます。利用可能な情報の詳細については、[Media Library Guide](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/guides/media-library) を参照してください。広告アカウントの FULL promotable user に属するアセットのほとんどはライブラリ内にありますが、いくつかの例外があります。

**メディアの取得**

前述のとおり、Image Card には media ID や media key への参照が含まれていません。そのため、Media Library を通じてそれらのアセットを取得することはできません。これは [Account Media](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/account-media#account-media) の画像についても同様です。

Video Card を作成するには、その Video アセットを事前に Media Library（またはその前身である Videos リソース）に含めておく必要があります。そのため、これらのアセットは常に Media Library から取得できます。これは Account Media の PREROLL アセットについても同様です。

最後に、ポスト内のメディアは、常に Media Library に存在することが保証されています。

次の表は、ルックアップに使用できる識別子がリソースレスポンスに含まれているかどうかも踏まえつつ、Media Library から取得可能なアセットをまとめたものです。

| **Resource** | **In the Media Library** |
| :--- | :--- |
| Image cards | No |
| Video cards | Yes* |
| ポスト（任意のメディア）** | Yes |
| 予約済みポスト | Yes |
| 下書きポスト | Yes |
| Account media images | No |
| Account media videos (`PREROLL`) | Yes |

* `video_content_id` が media key である Card の場合。値が media ID の場合もアセット自体は Media Library に存在しますが、取得するには先頭に数値とアンダースコアを付与する必要があります。\
** ポストは media ID のみを返します。アセットが Media Library に存在することは保証されていますが、取得するには先頭に数値とアンダースコアを付与する必要があります。

**Account Media との相互作用**

Media Library に追加されたメディアアセットが、自動的に Account Media リソースにも追加されるケースが 2 つあります。

* AMPLIFY&#95;VIDEO アセットが Media Library に追加されると、自動的に PREROLL というクリエイティブタイプの Account Media アセットとして追加されます。
* 特定のサイズ（[enumerations ページ](https://developer.x.com/content/developer-twitter/en/docs/ads/general/overview/enums) の「Creative Types」を参照）の画像が Media Library に追加されると、自動的に Account Media アセットとして追加されます。クリエイティブタイプ（例: INTERSTITIAL）は、画像のサイズによって決まります。

<div id="tweets">
  ### ポスト
</div>

#### はじめに

X Ads API は、公開済みポスト、予約ポスト、下書きポストの 3 種類をサポートしています。

<div id="nullcasted-tweets">
  #### ヌルキャストされたポスト
</div>

ポストは、ヌルキャスト（「Promoted-only」とも呼ばれます）かオーガニックのいずれかです。ヌルキャストされたポストは公開されてもユーザーの公開タイムラインには表示されませんが、公開ステータスであることに変わりはありません。一方、オーガニックポストはユーザーのフォロワーに配信され、ユーザーの公開タイムラインにも表示されます。

**ポストの作成**

3つあるポスト作成エンドポイントはいずれも、ヌルキャストされたポストにするかオーガニックポストにするかを選択できる Boolean 型の `nullcast` パラメータをサポートしています。ヌルキャストされたポストは、そのユーザー本人、またはユーザーに代わってポストを作成する権限を持つユーザーであれば誰でも作成できます。オーガニックポストを作成できるのは、[フルプロモータブルユーザー](https://developer.x.com/content/developer-twitter/en/docs/tutorials/ads-api-hierarchy-terminology#promotable-users)のみです。

**ポストの更新**

予約投稿および下書きポストについては、`nullcast` プロパティを更新することができます。予約投稿の場合、ポストの `scheduled_at` 時刻まで編集が可能です。下書きポストは無期限に編集できます。ただし、一度公開されると、そのポストをヌルキャストからオーガニック、またはその逆に変更することはできません。

<div id="promoting-tweets">
  #### プロモーションするポスト
</div>

プロモーションできるのは、公開済みポストと予約済みポストのみです。これらは nullcast（ノルキャスト）でもオーガニックでも構いません。制限はありません。広告主は、自分自身のポストでも、必要な許可を得ている限り他のユーザーのポストでもプロモーションすることができます（詳細は「[Promoting another user&#39;s Tweets](https://developer.x.com/content/developer-twitter/en/docs/tutorials/promoting-another-users-content)」を参照してください）。

1 つのキャンペーンで複数のポストをプロモーションすることができます。同様に、1 つのポストを 1 つ以上のキャンペーンでプロモーションすることもできます。

公開済みポストをプロモーションするには、[POST accounts/:account&#95;id/promoted&#95;tweets エンドポイント](https://developer.x.com/content/developer-twitter/en/docs/ads/campaign-management/api-reference/promoted-tweets#post-accounts-account-id-promoted-tweets)を使用します。これにより、公開済みポストがラインアイテムに関連付けられます。予約済みポストをプロモーションするには、[POST accounts/:account&#95;id/scheduled&#95;promoted&#95;tweets エンドポイント](https://developer.x.com/content/developer-twitter/en/docs/ads/campaign-management/api-reference/scheduled-promoted-tweets#post-accounts-account-id-scheduled-promoted-tweets)を使用します。

<div id="tweet-ids">
  #### ポストID
</div>

公開済み、予約済み、および下書きのポスト ID は[数値](https://developer.x.com/content/developer-twitter/en/docs/basics/twitter-ids)であり、64 ビット符号なし整数です。たとえば、次の公開済みポストの ID は 1166476031668015104 です。

公開済みまたは予約済みのポストをプロモーションとして配信すると、対応するプロモーション用ポストエンティティが作成されます。これらのエンティティにはそれぞれ ID があり、英数字から成る 36 進数エンコード値として表現されます。たとえば、上記の公開済みポストをプロモーションし、ラインアイテム 6c62d に関連付けると、次のような API レスポンスが返されます。

<div id="carousels">
  ### カルーセル
</div>

#### はじめに

X Ads API は、動画カルーセルと画像カルーセルの作成および取得をサポートしています。カルーセルは、2 ～ 6 個のメディアアセットを含めることができるカードタイプの一種です。カルーセルカードは、ユーザーをウェブサイトに誘導したり、モバイルアプリのインストールを促したりすることができます。カルーセルの概要、その利点やベストプラクティス、FAQ については、[Carousel Ads on X](https://business.x.com/en/advertising/carousels.html) ページを参照してください。

カルーセルは、他のカードタイプと同様にポストで使用でき、そのポストをプロモーションとして配信できます。ワークフローは、すでに慣れているものと同じです。

1. メディアをアップロードする
2. カードを作成する
3. ポストを作成する
4. ポストをプロモーションとして配信する

唯一の違いは、カードの作成方法です。他のカード作成リクエストではクエリパラメータを指定できますが、カルーセルカードの作成リクエストでは JSON POST ボディのみを受け付けます。

<div id="endpoints">
  #### エンドポイント
</div>

Ads API はカルーセルの作成および取得をサポートしています。

あらゆる種類のカルーセルを作成するには、[POST accounts/:account&#95;id/cards](https://developer.x.com/en/docs/twitter-ads-api/creatives/api-reference/cards#post-accounts-account-id-cards) エンドポイントを使用します。カルーセルを取得するには、[GET accounts/:account&#95;id/cards](https://developer.x.com/en/docs/twitter-ads-api/creatives/api-reference/cards#get-accounts-account-id-cards) エンドポイントを使用します。

<div id="json-post-body">
  #### JSON POST Body
</div>

カルーセルは 2 つのコンポーネントを使用して作成されます。1 つ目は使用するメディアアセットを指定し、2 つ目はウェブサイトまたはアプリのいずれかに関する情報を指定します。

具体的には、カルーセルカードは次のコンポーネントをこの順序で使用して作成します。

* メディアキーの配列を受け取る `SWIPEABLE_MEDIA` コンポーネントを 1 つ
* 次のうち *いずれか 1 つ*：
* ウェブサイト情報を指定するための `DETAILS` コンポーネント
* アプリ情報を指定するための `BUTTON` コンポーネント

`SWIPEABLE_MEDIA` コンポーネントには、2～6 個の画像または動画を指定できる `media_keys` 配列を含める必要があります。メディアキーを渡す順序によって、レンダリングされる順序が決まります。

```
{
  "type": "SWIPEABLE_MEDIA",
  "media_keys": [
    "13_1089771253848666112",
    "13_1191948012077092867"
  ]
}
```

念のため、[GET accounts/:account&#95;id/media&#95;library](https://developer.x.com/en/docs/twitter-ads-api/creatives/api-reference/media-library#get-accounts-account-id-media-library) エンドポイントにリクエストを送信することで、メディアキーを取得できます。

2つ目のコンポーネントオブジェクトの構成は、ユーザーをウェブサイトに誘導したいのか、アプリのインストールを促したいのかによって異なります。下の表は、この2つのオプションをまとめたものです。（**注**: 記載されているキーはすべて必須です。）

|               | **ウェブサイト**                                                         | **アプリ**                                           |
| :------------ | :----------------------------------------------------------------- | :------------------------------------------------ |
| コンポーネントタイプの指定 | `"type": "DETAILS"`                                                | `"type": "BUTTON"`                                |
| タイトル/ラベル      | `"title": "X"`                                                     | `"label": { "type": "ENUM", "value": "INSTALL" }` |
| 遷移先           | `"destination": { "type": "WEBSITE", "url": "https://www.x.com" }` | `"destination": { "type": "APP", ... }`           |

これらを組み合わせると、ウェブサイト向けカルーセルの JSON POST リクエストボディの例は次のとおりです。

```
{
  "name": "website carousel",
  "components": [
    {
      "type": "SWIPEABLE_MEDIA",
      "media_keys": [
        "13_1089771253848666112",
        "13_1191948012077092867"
      ]
    },
    {
      "type": "DETAILS",
      "title": "X",
      "destination": {
        "type": "WEBSITE",
        "url": "https://www.x.com"
      }
    }
  ]
}
```

`BUTTON` コンポーネント内の App destination オブジェクトには、国コードと少なくとも 1 つのアプリ識別子が必要です。オプションでディープリンクを指定することもできます。これらのフィールドの説明については、[リファレンスドキュメント](https://developer.x.com/en/docs/twitter-ads-api/creatives/api-reference/cards#app-destination)を参照してください。

これらを踏まえると、以下のような app carousel の JSON POST リクエストボディの例になります。

```
{
  "name": "app carousel",
  "components": [
    {
      "type": "SWIPEABLE_MEDIA",
      "media_keys": [
        "13_1089771253848666112",
        "13_1191948012077092867"
      ]
    },
    {
      "type": "BUTTON",
      "label": {
        "type": "ENUM",
        "value": "INSTALL"
      },
      "destination": {
        "type": "APP",
        "country_code": "US",
        "googleplay_app_id": "com.twitter.android",
        "iphone_app_id": "333903271"
      }
    }
  ]
}
```

<div id="creative-metadata-tagging">
  ### クリエイティブメタデータのタグ付け
</div>

#### はじめに

このガイドは、クリエイティブパートナー、代理店、およびクリエイティブ開発者が、X キャンペーン内で使用されるアセットにタグ付けを行い、個々のアセットの価値とパフォーマンスをより正確に把握できるようにするためのものです。

**注:** メディアアセットにタグ付けできるのは、そのメディアアセットを作成したパートナーまたは開発者のみです。メディアアセットの利用者が、そのメディアアセットの作成者ではない場合は、メタデータタグ付けを実装しないでください。

Creative Metadata Tagging を使用すると、アセットが X のどこにアップロードされるか、またそのアセットをアップロードする組織や個人が誰であるかにかかわらず、クリエイティブパートナーによって作成された画像や動画に対する帰属情報を付与できます。クリエイティブアセットとクリエイティブパートナーを関連付けるために、[XMP](https://exiftool.org/TagNames/XMP.html) 標準が使用されます。

<div id="tagging-creative-assets">
  #### クリエイティブアセットへのタグ付け
</div>

次の表は、各画像関連リソースのレスポンスで現在利用可能な識別子の種類と、それに対応する属性名を示しています。クリエイティブアセットにタグを付けるには、タグ付けツールが必要です。[ExifTool](https://exiftool.org/) は、プラットフォームに依存しない Perl ライブラリと、メタ情報の読み取り・書き込み・編集を行うための [コマンドラインアプリケーション](https://exiftool.org/exiftool_pod.html) で構成されており、推奨ツールです。サポートされているすべての [ファイルタイプ](https://exiftool.org/#supported) を参照してください。

[ExifTool インストール手順](https://exiftool.org/install.html) に従ってインストールしてください。[Homebrew](https://formulae.brew.sh/) が提供するソフトウェアパッケージを利用すると、macOS および Linux 向けの [exiftool インストールコマンド](https://formulae.brew.sh/formula/exiftool) により、インストールをさらに簡略化できます。コマンドラインで exiftool -ver と入力し、ツールのバージョン番号が返されることを確認して、インストールが正しく行われたことを検証してください。ExifTool のコマンドパラメータの詳細は [ExifTool ドキュメント](https://exiftool.org/) を参照してください。

クリエイティブパートナーは、新規または既存のクリエイティブアセットに対して、X の app&#95;id を contributor XMP タグおよび date タグとしてメタデータに付与できます。クリエイティブアセットは、[メディアのアップロード](https://developer.x.com/en/docs/twitter-api/v1/media/upload-media/overview) 時に既存のサイズ制限に従います。 

**Note:** X による contributor XMP タグの使用により、メタデータには X 上のキャンペーン専用の値のみが記録されます。

`exiftool -contributor="<YOUR APP ID>" -creative_file.jpg`

`exiftool -date="<date>" -creative_file.jpg`

app&#95;id は、[Developer Portal](https://developer.x.com/en/portal/dashboard) の Projects &amp; Apps セクションで確認できます。例: 16489123

次の例では、画像に対して、contributor タグとして app&#95;id を、date タグとして date を追加しています。

```
app_id:858382169
```

```
date:2022-03-13
```

```
creative_file: eiffel_tower.jpg
```

```
exiftool -contributor=858382169 eiffel_tower.jpg
```

```
   1 image files updated
```

```
exiftool -date=2022-03-13 eiffel_tower.jpg
```

```
  1 image files updated
```

画像に正しくタグが付与されていることを確認してください：

`exiftool -xmp:all -G1 <filename>`

例：

`exiftool -xmp:all -G1 eiffel_tower.jpg`

```
[XMP]        XMP ツールキット            : Image::ExifTool 12.30
```

```
[XMP]        Contributor            : 858382169
```

```
[XMP]        Date                   : 2022:03:13
```

<div id="post-accountsaccount_idcards">
  #### POST accounts/:account&#95;id/cards
</div>

指定されたアカウントに関連付けられた新しいカードを作成します。

カード作成リクエストは JSON の POST ボディのみを受け付けます。`Content-Type` は `application/json` に設定する必要があります。

詳細な利用例については、[Carousels Guide](https://developer.x.com/en/docs/twitter-ads-api/creatives/guides/carousels) を参照してください。

#### リソースURL

`https://ads-api.x.com/12/accounts/:account_id/cards`

#### パラメーター

JSON の POST ボディには、カードの `name` と `components` の配列を必ず含める必要があります。コンポーネントはオブジェクトとして表現され、カードの広告主向け属性を定義します。

次の例は、ペイロードの一般的な構造を示したもので（実際には動作しない情報も含まれます）、参考用です。

```
{
  "name": "some name",
  "components": [
    {
      "type": "TYPE_ENUM",
      "key": "value"
    }
  ]
}
```

以下のコンポーネントに関する追加情報です。

| Name                              | Description                                                                                                                                                                                                                                                   |
| :-------------------------------- | :------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| account&#95;id   *required*       | 対象アカウントの識別子です。リソースのパス内に含まれ、[GET accounts](https://developer.x.com/en/docs/ads/campaign-management/api-reference/accounts#get-accounts) を除くすべての Advertiser API リクエストで、通常必須のパラメータです。指定したアカウントは、認証済みユーザーに関連付けられている必要があります。  Type: string  Example: `18ce54d4x5t` |
| name   *required*                 | カードの名前です。最大 80 文字です。  Type: string  Example: `component based card`                                                                                                                                                                                           |
| components   *sometimes required* | カードを作成するために使用するコンポーネントを定義します。詳細は以下を参照してください。`slides` と同時には指定できません。**注記**: コンポーネントの順序は重要です。  Type: array of objects                                                                                                                                            |
| slides   *sometimes required*     | Multi-Destination Carousel を作成するために、この配列の配列を使用します。各カードをコンポーネントのグループとして記述します。各スライドはカードを完全に表現している必要があります。`components` と同時には指定できません。**注記**: スライドの順序は重要です。  Type: array of array                                                                                  |

<div id="components">
  #### コンポーネント
</div>

すべてのコンポーネントには、そのオブジェクトのスキーマを決定する `type` フィールドを含める必要があります。Ads API では、メディアベースおよび説明ベースのコンポーネントに分類された、次のコンポーネントタイプをサポートしています。

* メディア:
* `MEDIA`: 単一の動画または画像
* `SWIPEABLE_MEDIA`: 2〜6 個の動画または画像
* 説明:
* `DETAILS`
* `BUTTON`

各コンポーネントには（`type` キーに加えて）、必須フィールドのセットが定義されています。これらは次の表に示します。

| コンポーネント `type`    | フィールド                   | 値の型              |
| :---------------- | :---------------------- | :--------------- |
| `MEDIA`           | `media_key`             | string           |
| `SWIPEABLE_MEDIA` | `media_keys`            | array of strings |
| `DETAILS`         | `title`   `destination` | string   object  |
| `BUTTON`          | `label`   `destination` | object   object  |

以下は、`components` 配列内における `BUTTON` コンポーネントの例です（意図的に `name` キーを省略しています）。省略記号は、追加情報を指定する必要がある箇所を示しています。

```
{
  "components": [
    {
      "type": "BUTTON",
      "label": {
        ...
      },
      "destination": {
        ...
      }
    }
  ]
}
```

コンポーネントオブジェクトを指定する順序が、それらが上から下へレンダリングされる順序を決定します。カードは 1 つのメディアベースのコンポーネントと、`DETAILS` または `BUTTON` コンポーネントのいずれかを組み合わせて作成する必要があります。説明ベースのコンポーネントはメディアの下にレンダリングされ、URL またはモバイルアプリをリンク先として持ちます。

**Label**

Label はボタンに表示されるテキストを定義するため、`BUTTON` コンポーネントにのみ適用されます。Label オブジェクトには、必須キーが 2 つあります: `type` と `value` です。`type` は `ENUM` に設定する必要があり、`value` には `BOOK`、`CONNECT`、`INSTALL`、`OPEN`、`ORDER`、`PLAY`、`SHOP` のいずれかを指定できます。

前の例に基づき、以下に `BUTTON` コンポーネント内にある `label` オブジェクトを示します。

```
{
  "components": [
    {
      "type": "BUTTON",
      "label": {
        "type": "ENUM",
        "value": "INSTALL"
      },
      "destination": {
        ...
      }
    }
  ]
}
```

**遷移先**

遷移先は、広告主がユーザーを誘導しようとする場所です。`DETAILS` コンポーネントまたは `BUTTON` コンポーネント内で常に必須です。遷移先には 2 種類があります: `WEBSITE` と `APP` です。

**注記**: `WEBSITE` の遷移先は `DETAILS` コンポーネントでのみ使用でき、`APP` の遷移先は `BUTTON` コンポーネントでのみ使用できます。

**Website 遷移先**

| Name              | Description                                                                                       |
| :---------------- | :------------------------------------------------------------------------------------------------ |
| type   *required* | 遷移先の種類であり、それによってスキーマが決定されます。  Type: enum  Possible values: `WEBSITE`                              |
| url   *required*  | ユーザーをリダイレクトする Web サイトの URL。  Type: string  Example: `https://devcommunity.x.com/c/advertiser-api` |

**App 遷移先**

| Name                                                        | Description                                                                                                                                           |
| :---------------------------------------------------------- | :---------------------------------------------------------------------------------------------------------------------------------------------------- |
| type   *required*                                           | 遷移先の種類であり、それによってスキーマが決定されます。  Type: enum  Possible values: `APP`                                                                                      |
| country&#95;code   *required*                               | アプリが販売されている国を示す、[ISO 3166-1 alpha-2](https://en.wikipedia.org/wiki/ISO_3166-1_alpha-2) の 2 文字の国コード。  Type: string  Example: `US`                      |
| googleplay&#95;app&#95;id   *sometimes required*            | Google Play アプリケーションのパッケージ名。  **注記**: `ios_app_store_identifier` または `googleplay_app_id` の少なくとも一方が必須です。  Type: string  Example: `com.twitter.android` |
| ios&#95;app&#95;store&#95;identifier   *sometimes required* | iOS App Store の識別子。  **注記**: `ios_app_store_identifier` または `googleplay_app_id` の少なくとも一方が必須です。  Type: string  Example: `333903271`                    |
| googleplay&#95;deep&#95;link   *optional*                   | プロモーション対象の Android アプリへのディープリンク。  **注記**: `googleplay_app_id` が指定されている場合にのみ使用できます。  Type: string                                                      |
| ios&#95;deep&#95;link   *optional*                          | プロモーション対象の iOS アプリへのディープリンク。  **注記**: `ios_app_store_identifier` が指定されている場合にのみ使用できます。  Type: string                                                   |

アカウントの完全プロモート可能ユーザー（デフォルト）または `as_user_id` パラメータで指定したユーザーのポストを作成します。nullcast（デフォルト）およびオーガニックポストの両方の作成に対応しています。nullcast ポストは公開タイムラインに表示されず、フォロワーにも配信されません。どちらのタイプもキャンペーンで使用できます。

認証済みユーザーがこのアカウントの `FULL` 完全プロモート可能ユーザーではない場合、[GET accounts/:account&#95;id/authenticated&#95;user&#95;access](https://developer.x.com/en/docs/ads/campaign-management/api-reference/authenticated-user-access#get-accounts-account-id-authenticated-user-access) エンドポイントにリクエストを行い、このユーザーの代理でポストを作成する権限があるかどうかを確認してください。`TWEET_COMPOSER` のパーミッションが付与されている場合、そのユーザーはこのエンドポイントを使用して `FULL` [完全プロモート可能ユーザー](https://developer.x.com/en/docs/ads/campaign-management/api-reference/promotable-users#get-accounts-account-id-promotable-users) の代理として nullcast ポストを作成できます。

[upload.x.com エンドポイント](https://developer.x.com/rest/reference/post/media/upload) でメディアを使用する場合、このエンドポイントに渡す `as_user_id` の値と同じ `user_id` の値を `additional_owners` パラメータに指定してください。
