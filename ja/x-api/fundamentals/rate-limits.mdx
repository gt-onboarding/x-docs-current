---

毎日、何千もの開発者が X API にリクエストを送信しています。この膨大な数のリクエストを管理するため、送信できるリクエスト数には制限が設けられています。これらの制限により、開発者コミュニティが依存している、信頼性が高くスケーラブルな API を提供できます。

許可されるリクエスト数の上限は、時間間隔に基づいて決まります。通常は、特定の期間または時間ウィンドウに対して設定されます。最も一般的な間隔は 15 分です。たとえば、15 分あたり 900 件のリクエストという制限を持つエンドポイントでは、任意の 15 分間に最大 900 件のリクエストを送信できます。

レート制限は認証方式によって異なります。たとえば、[OAuth 1.0a User Context](/ja/resources/fundamentals/authentication) を使用する場合、各ユーザーの[アクセストークン](/ja/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) セットごとに、一定期間あたりの個別のレート制限が適用されます。別の例として、[OAuth 2.0 Bearer Token](/ja/resources/fundamentals/authentication#oauth-2-0) を使用する場合は、アプリごとに一定期間あたりの個別の制限が適用されます。これらの制限を超えると、エラーが返されます。

<div id="rate-limits-and-authentication-method">
  ## レート制限と認証方法
</div>

レート制限は、開発者アプリおよびユーザーアクセストークンの両方のレベルで設定されています。

* **[OAuth 2.0 Bearer Token](/ja/resources/fundamentals/authentication/oauth-2-0/application-only): アプリのレート制限**
  この方法では、開発者アプリとして一定回数のリクエストを送信できます。この認証方法を使用する場合、制限は Bearer Token を用いて行われるリクエストに基づいて決まります。
  * 例: 15 分あたり 450 リクエストという制限がある場合、その 15 分間に開発者アプリとして 450 件のリクエストを送信できます。

* **[OAuth 1.0a User Context](/ja/resources/fundamentals/authentication/oauth-1-0a/obtaining-user-access-tokens): ユーザー単位のレート制限**
  この方法では、ユーザーアクセストークンによって識別される X ユーザーに代わってリクエストを送信できます。たとえば、ポストの非公開メトリクスを取得する場合は、そのユーザーのために [3-legged OAuth フロー](/ja/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) を使用して生成されたユーザーアクセストークンで認証します。
  * 例: ユーザーごとに 15 分あたり 900 リクエストという制限がある場合、その時間枠内でユーザーごとに最大 900 件のリクエストを送信できます。

<div id="http-headers-and-response-codes">
  ## HTTP ヘッダーとレスポンスコード
</div>

直近に行ったリクエストに基づいて、指定されたレート制限の中でアプリケーションがどの状態にあるかを把握するために HTTP ヘッダーを使用します。

* `x-rate-limit-limit`: エンドポイントに対するレート制限の上限
* `x-rate-limit-remaining`: 15 分間のウィンドウで残っているリクエスト数
* `x-rate-limit-reset`: レート制限がリセットされる時刻（UTC エポック秒）

<div id="error-responses">
  ### エラー応答
</div>

アプリケーションがそのエンドポイントのレート制限を超過した場合、API はレスポンス本文に次のエラーメッセージを含む [HTTP 429「Too Many Requests」](http://tools.ietf.org/html/rfc6585) 応答を返します。

```json
{ "errors": [ { "code": 88, "message": "Rate limit exceeded" } ] }
```

<div id="recovering-from-a-rate-limit">
  ## レート制限からの復旧
</div>

これらのレート制限を超過すると、エンドポイントから 429「Too many requests」エラーが返されます。以下で説明するように、レート制限エラーが発生した場合のベストプラクティスは、制限がいつリセットされるかを示す HTTP ヘッダーを確認し、それまではリクエストを一時停止することです。<br />&quot;too many requests&quot;（リクエスト過多）やレート制限エラーが発生した場合は、リクエストの送信頻度を下げる必要があります。レート制限エラーが発生したときは、`x-rate-limit-reset:` HTTP ヘッダーを確認することで、レート制限がいつリセットされるかを把握できます。<br /><br />もう 1 つの一般的なパターンは、指数バックオフに基づくものです。最初はリクエスト間隔を短く（例えば数秒）し、その後リトライのたびにその間隔を 2 倍にしていきます。これは、リクエストが成功するか、リクエスト間隔の妥当な最大値（例えば数分）に達するまで続けます。<br /><br />理想的には、クライアント側が既存のレート制限を把握しており、現在超過しているウィンドウが期限切れになるまでリクエストを一時停止できることが望ましいです。15 分単位の制限を超えた場合は、1～2 分待ってからリトライするのが妥当です。<br /><br />これらのリクエスト数に関する制限に加えて、Standard Basic レベルのアクセスでは、recent search および filtered stream エンドポイントから、月あたり最大 500,000 件のポストを利用できます。ポスト数の月次上限を超えている場合は、アプリ側で通知を出し、その月の利用開始日を把握して、その日まではリクエストを控える方が理にかなっています。

<div id="tips-to-avoid-being-rate-limited">
  ### レート制限を回避するためのヒント
</div>

以下のヒントは、防御的な実装を行い、レート制限を受ける可能性を減らすためのものです。提供したいと考えている一部のアプリケーション機能は、特に結果の鮮度を保つという観点では、レート制限の存在により実現がそもそも不可能な場合があります。アプリケーションの目的がリアルタイム性の高い情報の提供である場合は、filtered stream エンドポイントや sampled stream エンドポイントの利用を検討してください。

<div id="caching">
  #### キャッシュ
</div>

頻繁に利用することが想定される場合は、API レスポンスを保存しておきましょう。ページの読み込みごとに API を呼び出すのではなく、レスポンスをローカルにキャッシュします。

<div id="prioritize-active-users">
  #### アクティブなユーザーを優先する
</div>

サイトで多数の X ユーザーを管理している場合（たとえば、現在のステータスや X の利用状況に関する統計情報を取得している場合）、最近サイトにサインインしたユーザーに対してのみデータをリクエストすることを検討してください。

<div id="adapt-to-the-search-results">
  #### 検索結果に合わせて調整する
</div>

アプリケーションで大量の検索キーワードを監視している場合、結果が返ってこない検索については、結果が返ってくる検索よりもクエリの頻度を下げてください。バックオフを利用することで、人気のあるクエリは最新の状態を保ちつつ、ほとんど変化しないクエリに対して無駄にリクエストを送らずに済みます。別の方法として、filtered stream エンドポイントを使用し、検索クエリでフィルタリングすることも検討してください。

<div id="denylist">
  #### 拒否リスト
</div>

アプリケーションがレート制限を乱用した場合、そのアプリケーションは拒否リストに追加されます。拒否されたアプリは X API からレスポンスを受け取ることができません。あなた、またはあなたのアプリケーションが拒否されており、誤りだと思われる場合は、[Platform Support フォーム](https://support.x.com/forms/platform)からサポートを依頼できます。次の情報を必ず含めてください。

1. アプリケーションが拒否されたと考える理由を説明してください。
2. 現在はレート制限を受けていない場合は、どのように問題を解決したのかを詳しく説明してください。
