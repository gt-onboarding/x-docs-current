keywords: ["レート制限", "API のレート制限", "リクエスト制限", "レートリミット処理", "スロットリング", "15 分ウィンドウ", "レート制限エラー"]
---

毎日、何千人もの開発者が X API にリクエストを送信しています。これらのリクエストの膨大な量を管理するため、送信できるリクエスト数には制限が設けられています。こうした制限により、開発者コミュニティが利用している信頼性が高くスケーラブルな API を提供できています。

許可されるリクエストの最大数は、時間間隔に基づいています。通常は、特定の期間、つまり一定時間の「ウィンドウ」を基準とします。最も一般的な間隔は 15 分です。たとえば、15 分あたり 900 リクエストの制限があるエンドポイントでは、任意の 15 分間に最大 900 回のリクエストを送信できます。

レート制限は認証方法によって異なります。たとえば、[OAuth 1.0a User Context](/ja/resources/fundamentals/authentication) を使用している場合、各ユーザーの [アクセストークン](/ja/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) セットごとに、一定期間ごとのレート制限が個別に適用されます。一方、[OAuth 2.0 Bearer Token](/ja/resources/fundamentals/authentication#oauth-2-0) を使用している場合は、アプリごとに独立した期間あたりの制限が適用されます。これらの制限を超過すると、エラーが返されます。

<div id="rate-limits-and-authentication-method">
  ## レート制限と認証方法
</div>

レート制限は、開発者アプリ レベルとユーザーのアクセストークン レベルの両方に設定されています。

* **[OAuth 2.0 Bearer Token](/ja/resources/fundamentals/authentication/oauth-2-0/application-only): アプリのレート制限**\
  この方法では、開発者アプリを代表して一定数のリクエストを行うことができます。この認証方法を使用する場合、制限は Bearer Token を用いて行われるリクエストによって決まります。
  * 例: 15 分ごとの上限が 450 リクエストの場合、その 15 分間にアプリを代表して 450 件のリクエストを行うことができます。

* **[OAuth 1.0a User Context](/ja/resources/fundamentals/authentication/oauth-1-0a/obtaining-user-access-tokens): ユーザーのレート制限**\
  この方法では、ユーザーのアクセストークンによって識別される X ユーザーを代表してリクエストを行うことができます。たとえば、ポストから非公開メトリクスを取得する場合は、そのユーザー向けに [3-legged OAuth フロー](/ja/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) を使用して生成されたユーザーアクセストークンで認証します。
  * 例: ユーザーごとに 15 分あたり 900 リクエストの上限がある場合、その 15 分間にユーザーごとに最大 900 件のリクエストを行うことができます。

<div id="http-headers-and-response-codes">
  ## HTTP ヘッダーとレスポンスコード
</div>

最新のリクエストに基づいて、指定されたレート制限の中でアプリケーションの現在の状況を把握するために HTTP ヘッダーを使用します。

* `x-rate-limit-limit`: エンドポイントのレート制限の上限
* `x-rate-limit-remaining`: 15 分間の時間枠で残っているリクエスト数
* `x-rate-limit-reset`: レート制限がリセットされるまでの残り時間（UTC エポック秒）

<div id="error-responses">
  ### エラー応答
</div>

アプリケーションがエンドポイントのレート制限を超えた場合、API はレスポンス本文内に次のエラーメッセージを含む [HTTP 429 “Too Many Requests”](http://tools.ietf.org/html/rfc6585) レスポンスを返します。

```json
{ "errors": [ { "code": 88, "message": "Rate limit exceeded" } ] }
```

<div id="recovering-from-a-rate-limit">
  ## レート制限からの復旧
</div>

これらのレート制限を超過すると、エンドポイントから 429「Too Many Requests」エラーが返されます。後述のとおり、レート制限エラーが発生した場合のベストプラクティスは、制限がいつリセットされるかを示す HTTP ヘッダーを確認し、それまでリクエストを一時停止することです。<br />「Too Many Requests」またはレート制限エラーが発生した場合は、リクエストの送信頻度を下げる必要があります。レート制限エラーに達したら、レート制限がいつリセットされるかを把握するために `x-rate-limit-reset:` HTTP ヘッダーを確認できます。<br /><br />もう 1 つの一般的なパターンは指数バックオフに基づくもので、リクエスト間の時間を小さく（数秒など）始め、再試行のたびに 2 倍にしていく方法です。これは、リクエストが成功するか、あるいはリクエスト間の時間が妥当な最大値（数分など）に達するまで続けます。<br /><br />理想的には、クライアント側が既存のレート制限を把握しており、現在超過しているウィンドウが期限切れになるまでリクエストを一時停止できる状態であることが望まれます。15 分の制限を超えた場合は、1～2 分待ってから再試行するのが妥当です。<br /><br />リクエスト数に対するこれらの制限に加えて、Standard Basic アクセスレベルでは、recent search および filtered stream エンドポイントから、月あたり最大 500,000 件のポストを取得できます。ポスト数の月次上限を超過した場合は、アプリ側で通知を出し、利用開始日（課金サイクル開始日）を把握したうえで、その日までリクエストを控えるほうが理にかなっています。

<div id="tips-to-avoid-being-rate-limited">
  ### レート制限を回避するためのヒント
</div>

以下のヒントは、防御的なコードを書くことでレート制限を受ける可能性を減らすのに役立ちます。レート制限が存在することを踏まえると、特に結果の鮮度に関しては、提供することが事実上不可能なアプリケーション機能もあります。アプリケーションの目的がリアルタイムな情報の提供である場合は、filtered stream エンドポイントや sampled stream エンドポイントの利用を検討してください。

<div id="caching">
  #### キャッシュ
</div>

頻繁に利用することが想定される場合は、API レスポンスを保存して再利用してください。ページ読み込みのたびに API を呼び出すのではなく、レスポンスをローカルにキャッシュしましょう。

<div id="prioritize-active-users">
  #### アクティブなユーザーを優先する
</div>

サイトで多数の X ユーザーを管理している場合（たとえば、現在のステータスや X の利用状況に関する統計情報を取得している場合）、最近そのサイトにサインインしたユーザーに対してのみデータをリクエストすることを検討してください。

<div id="adapt-to-the-search-results">
  #### 検索結果に応じて調整する
</div>

アプリケーションで多数の検索語句を監視している場合、結果が返ってこない検索については、結果がある検索よりも問い合わせ頻度を下げてください。バックオフを行うことで、人気のある検索クエリについては最新の状態を保ちつつ、めったに変化しないクエリに対して不要なリクエストを行わずに済みます。あるいは、filtered stream エンドポイントを使用し、検索クエリでフィルタリングすることも検討してください。

<div id="denylist">
  #### 拒否リスト
</div>

アプリケーションがレート制限を乱用した場合、そのアプリは拒否リストに追加されます。拒否リストに入ったアプリは、X API からレスポンスを受け取ることができません。あなた、またはあなたのアプリケーションが拒否リストに入っており、誤りだと思われる場合は、[Platform Support forms](https://support.x.com/forms/platform) からサポートを依頼できます。次の情報を含めてください。

1. あなたのアプリケーションが拒否リストに入ったと考える理由を説明してください。
2. すでにレート制限を受けていない場合は、どのように問題を解決したかを詳細に記述してください。
