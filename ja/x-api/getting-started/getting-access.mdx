<div id="step-one-sign-up-for-a-developer-account">
  ### ステップ1：開発者アカウントに登録する
</div>

開発者アカウントの登録は簡単で、すぐに完了します。以下のボタンをクリックしていくつかの質問に回答すると、[Basic access](https://developer.x.com/en/portal/products/basic) を使って X API v2 を探索・構築し始めることができます。

次に、オンボーディングプロセスの中でプロジェクトとそれに関連付けられた開発者アプリを作成します。ここで、API へのすべてのリクエストを認証するために使用する一連の認証情報が発行されます。

[Sign up](https://developer.x.com/en/portal/products/basic)

<div id="step-two-save-your-apps-key-and-tokens-and-keep-them-secure">
  ### ステップ 2：アプリのキーとトークンを保存し、安全に保管する
</div>

アクセス権を取得し、プロジェクトとアプリを作成すると、開発者アプリ内で次の認証情報を確認または生成できるようになります。

* **[API Key and Secret](https://developer.x.com/resources/fundamentals/authentication/api-key-and-secret)：** アプリ用のユーザー名とパスワードのようなものです。[OAuth 1.0a User Context](/ja/resources/fundamentals/authentication) が必要なリクエストの認証や、ユーザーのアクセストークンや App Access Token など他のトークンを生成する際に使用します。

* **[Access Token and Secret](https://developer.x.com/resources/fundamentals/authentication/obtaining-user-access-tokens)：** 一般的に、アクセストークンは、あなたがどのユーザーを代表してリクエストを行っているかを表します。developer portal から生成できるものは、アプリの所有者であるユーザーを表します。これらは [OAuth 1.0a User Context](/ja/resources/fundamentals/authentication) が必要なリクエストの認証に使用します。別のユーザーに代わってリクエストを行いたい場合は、そのユーザーに承認してもらうために 3-legged OAuth フローを利用する必要があります。

* **[Client ID and Client Secret](/ja/resources/fundamentals/authentication#how-to-connect-to-endpoints-using-oauth-2-0-authorization-code-flow-with-pkce)：** これらの認証情報は、OAuth 2.0 認証を使用してユーザーのアクセストークンを取得する際に使用します。OAuth 1.0a と同様に、ユーザーのアクセストークンは、非公開のユーザーアカウント情報を取得するリクエストや、別のアカウントに代わってアクションを実行するリクエストの認証に使用されますが、クライアントアプリケーションがユーザーに対して持つアクセス権を、スコープによってより細かく制御できるようになっています。

* **[App only Access Token](/ja/resources/fundamentals/authentication#bearer-token-also-known-as-app-only)：** X 上で公開されている情報を返すエンドポイントにリクエストを行う際に、このトークンを使用します。

これらのキーとトークンは再生成しない限り有効期限がないため、認証情報を受け取ったら、パスワードマネージャーなどの安全な場所に保存しておくことをお勧めします。

> **ご注意：キーとトークンは [developer portal](/ja/resources/fundamentals/developer-portal) 上で一度しか表示されません。そのため、生成後すぐにパスワード管理システムへ保存することが重要です。**
>
> キーとトークンを紛失または失念した場合は再生成する必要があり、その際は新しい認証情報が作成され、古いものは無効になります。これは、以前の認証情報で設定していたすべての連携を更新しなければならないことを意味します。
>
> [認証におけるベストプラクティス](/ja/resources/fundamentals/authentication#authentication-best-practices) について、さらに詳しく確認してください。
