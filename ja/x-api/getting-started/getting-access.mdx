<div id="step-one-sign-up-for-a-developer-account">
  ### ステップ 1：開発者アカウントに登録する
</div>

開発者アカウントの登録は簡単で、すぐに完了します。以下のボタンをクリックし、いくつかの質問に答えるだけで、[Basic access](https://developer.x.com/en/portal/products/basic) を使って X API v2 を探索したり、アプリケーションの開発を始めたりできます。

次に、オンボーディングプロセスの中でプロジェクトと、それに関連付けられた開発者アプリを作成します。ここで、API へのすべてのリクエストを認証する際に使用する認証情報一式が付与されます。

[登録](https://developer.x.com/en/portal/products/basic)

<div id="step-two-save-your-apps-key-and-tokens-and-keep-them-secure">
  ### ステップ2：アプリのキーとトークンを保存し、安全に保管する
</div>

アクセス権を取得し、プロジェクトとアプリを作成すると、開発者アプリ内で次の認証情報を確認または生成できるようになります。

* **[API Key and Secret](https://developer.x.com/resources/fundamentals/authentication/api-key-and-secret)：** アプリのユーザー名とパスワードに相当します。これらは、[OAuth 1.0a User Context](/ja/resources/fundamentals/authentication) を必要とするリクエストの認証や、ユーザーのアクセストークンや App Access Token など、他のトークンを生成する際に使用します。

* **[Access Token and Secret](https://developer.x.com/resources/fundamentals/authentication/obtaining-user-access-tokens)：** 一般的に、アクセストークンは、あなたがリクエストを送信する際に、そのリクエストの対象となるユーザーを表します。developer portal から生成できるものは、そのアプリの所有者であるユーザーを表します。これらは、[OAuth 1.0a User Context](/ja/resources/fundamentals/authentication) を必要とするリクエストの認証に使用します。別のユーザーの代わりにリクエストを送信したい場合は、そのユーザーに許可してもらうため、3-legged OAuth フローを使用する必要があります。

* **[Client ID and Client Secret](/ja/resources/fundamentals/authentication#how-to-connect-to-endpoints-using-oauth-2-0-authorization-code-flow-with-pkce)：** これらの認証情報は、OAuth 2.0 認証を用いてユーザーのアクセストークンを取得する際に使用されます。OAuth 1.0a と同様に、ユーザーのアクセストークンは、非公開のユーザーアカウント情報を取得するリクエストや、他のアカウントの代わりにアクションを実行するリクエストを認証するために使用されますが、クライアントアプリケーションがユーザーに対して持つ権限を、より細かいスコープで制御できるようになっています。

* **[App only Access Token](/ja/resources/fundamentals/authentication#bearer-token-also-known-as-app-only)：** X 上で一般公開されている情報を返すエンドポイントに対してリクエストを行う際に、このトークンを使用します。

これらのキーとトークンは再生成しない限り有効期限がないため、認証情報を受け取ったら、パスワードマネージャーなどの安全な場所に保存しておくことを推奨します。

> **ご注意：キーとトークンは [developer portal](/ja/resources/fundamentals/developer-portal) 上で一度しか表示されません。そのため、生成後すぐにパスワード管理システムに保存することが重要です。**
>
> キーとトークンを紛失または失念した場合は、それらを再生成する必要があります。再生成すると新しい認証情報が作成され、古い認証情報は無効になります。これは、以前の認証情報を使用して設定していたあらゆる連携を更新しなければならないことを意味します。
>
> [認証におけるベストプラクティス](/ja/resources/fundamentals/authentication#authentication-best-practices) について、さらに詳しくご覧いただけます。
