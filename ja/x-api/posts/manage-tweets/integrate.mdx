  ## 便利なツール

このエンドポイントを統合する際に役立つ重要な概念を説明する前に、まず次のツールに慣れておくことをおすすめします。

<div id="postman">
  ### Postman
</div>

Postman は、エンドポイントを試す際に利用できる便利なツールです。各 Postman リクエストには、何が利用できるかをすばやく把握できるよう、すべてのパスパラメータおよびボディパラメータが含まれています。Postman コレクションの詳細については、[&quot;Using Postman&quot;](/ja/tutorials/postman-getting-started) のページをご覧ください。

<div id="authentication">
  ### 認証
</div>

すべての X API v2 エンドポイントでは、キーとトークンとも呼ばれる一連の認証情報を使用してリクエストを認証する必要があります。 

これらの特定のエンドポイントでは [OAuth 1.0a User Context](/ja/resources/fundamentals/authentication) を使用する必要があります。つまり、リクエストを正常に送信するには、一連の API キーとユーザーのアクセストークンを使用する必要があります。アクセストークンは、あなたが代理でリクエストを行うユーザーに関連付けられている必要があります。別のユーザー向けのアクセストークンを生成したい場合は、そのユーザーが [3-legged OAuth フロー](/ja/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) を使用してあなたのアプリを承認または認証する必要があります。

OAuth 1.0a の利用は難しい場合があることに注意してください。この認証方法に慣れていない場合は、[ライブラリ](/ja/x-api/tools-and-libraries/overview) を使用するか、Postman のようなツールを利用するか、あるいは OAuth 2.0 を使用してリクエストを認証することをおすすめします。

[OAuth 2.0 Authorization Code with PKCE](/ja/resources/fundamentals/authentication#oauth-2-0-authorization-code-flow-with-pkce-2) を使用すると、アプリケーションのスコープや複数デバイス間にまたがる認可フローを、より細かく制御できます。OAuth 2.0 では、ユーザーに代わって特定の権限を付与するための、粒度の細かいスコープを個別に選択できます。 

アプリで OAuth 2.0 を有効にするには、developer portal の App 設定セクションにあるアプリの認証設定で OAuth 2.0 を有効にする必要があります。

<div id="developer-portal-projects-and-developer-apps">
  ### Developer portal、Projects、および developer Apps
</div>

X API v2 エンドポイントで使用できる一連の認証情報を取得するには、[開発者アカウント](/ja/resources/fundamentals/developer-portal) を保有し、そのアカウント内に [プロジェクト](/ja/resources/fundamentals/projects) を設定し、そのプロジェクト内に [開発者アプリ](/ja/resources/fundamentals/developer-apps) を作成している必要があります。これらのキーとトークンは、開発者アプリ内で確認できます。 
 

<div id="rate-limits">
  ### レート制限
</div>

毎日、何万もの開発者が X API にリクエストを送信しています。これらの膨大なリクエスト量を管理するために、各エンドポイントには [レート制限](https://developer.x.com/x-api/fundamentals/rate-limits) が設けられており、アプリまたは認証済みユーザーに代わって送信できるリクエスト数が制限されています。

これらのエンドポイントにはユーザーレベルのレート制限が適用されます。つまり、あなたがリクエストを送信している認証済みユーザーは、どの開発者アプリからであっても、そのエンドポイントを呼び出せる回数が一定数に制限されます。POST メソッドには 15 分あたり 200 リクエストのユーザーレート制限があります。DELETE メソッドには 15 分あたり 50 リクエストのレート制限があります。さらに、[3 時間あたり 300 リクエスト](https://blog.x.com/developer/en_us/topics/tools/2018/new-developer-requirements-to-protect-our-platform)という上限があり、manage Posts または manage Retweets 権限で作成されたポストも含まれます。
 

<div id="source-labels">
  ### ソースラベル
</div>

あなたのアプリ名とウェブサイトの URL は、アプリケーションによってプログラム経由で作成されたポストのメタデータ内で [ソースラベル](https://help.x.com/en/using-twitter/how-to-tweet#source-labels) として表示されます。X アプリのユースケースを変更する場合は、[Developer Terms](https://developer.x.com/content/developer-twitter/en/developer-terms/agreement-and-policy) に準拠し続けるために、必ずこれらの設定でユースケースも更新してください。
 

<div id="profile-settings">
  ### プロフィール設定
</div>

プロフィール設定で位置情報（ジオ）機能を有効にしている場合にのみ、ポストに位置情報を追加できます。位置情報機能を有効にしていない場合でも、リクエストボディに location パラメータを追加すること自体はできますが、その位置情報がポストに紐付くことはありません。画像へのユーザーのタグ付けについても同様です。タグ付けしようとしているユーザーが写真へのタグ付け機能を有効にしていない場合、そのポストが正常に作成されても、タグ付けされたユーザー一覧にそのユーザー名は表示されません。 
 

<div id="adding-media-to-a-post">
  ### ポストにメディアを追加する
</div>

現在のところ、X API v2 だけでメディアを完全にアップロードする方法はありません。ただし、事前にアップロード済みのメディアをポストに添付することはできます。すでに [v1 の media エンドポイントを使用してアップロードされているメディア](https://developer.x.com/en/docs/twitter-api/v1/media/upload-media/api-reference/post-media-upload)や、[X Media Studio](https://media.x.com/en/articles/products/2018/media-studio) を使用してアップロードされたメディアの media id を利用できます。これらの media id は、自分自身、もしくは認証済みユーザーのものである必要があります。
