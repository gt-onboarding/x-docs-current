<div id="introduction">
  ## はじめに
</div>

Posting With Location は、X API におけるジオタグ機能です。この機能により、ポストに文脈情報を付与できるため、ユーザーにとってより意味のある体験を提供できます。たとえば、「Fireworks!」というポストは、場所情報と結び付けることで、カリフォルニア州サンフランシスコの SoMa 地区からの「Fireworks!」のように、ぐっと意味合いが増します。私たちは、この機能を活用して開発者エコシステムの皆さんと協力し、優れた体験を創出していけることを楽しみにしています。

<div id="user-privacy-and-user-control-of-their-location-information">
  ## ユーザーのプライバシーと位置情報のユーザーによる管理
</div>

良質で安全な利用体験を提供するために、開発者はユーザーの位置情報を扱う際にユーザーのプライバシーに十分配慮するとともに、「位置情報を付けてポストする」エクスペリエンスにおいて基本的な一貫性を確保する必要があります。以下は、ユーザーのプライバシーを保護し、ユーザーが自らの行動の影響を理解できるようにするために留意すべき主なポイントです。

<div id="general-guidelines">
  ## 一般的なガイドライン
</div>

* ユーザーは、「位置情報付きポスト」機能を利用するために、自らオプトイン（位置情報を「オン」にする）しなければなりません。
* ユーザーは、自分のポストに正確な位置情報を表示することについて明示的に許可する必要があります。クライアントの位置情報機能が「スティッキー」、つまりいったん位置情報をオンにすると常にポストに付随する仕様である場合は、場所名（place&#95;ids）のみを表示し、ユーザーがポストごと、または特定の代替設定として、正確な緯度・経度の表示にさらにオプトインできるようにすることを検討してください。
* ユーザーによるポストの投稿に際してユーザーの位置情報データにアクセスする場合、その情報は、ユーザーの明示的な許可がない限り、そのポストとは切り離して保存してはなりません。
* ユーザーのポストに関連して、どのレベルの位置情報が（もしあれば）表示されるのかを、ユーザーが明確に理解できるようにしなければなりません。これを実現する一つの方法として、ポストを公開する前に、その情報がどのように表示されるかを示す地図を表示することが考えられます。クライアント上で、ユーザーが公開する情報の一部（または唯一の情報）として正確な緯度・経度が含まれる場合、ユーザーに対して、自分の正確な座標が公開されることが明確に分かるようにする必要があります。
* ユーザーは、ポストを作成するたびに、自分の位置情報をオン・オフできるようにする必要があります。
* X の [Developer Policy](/ja/developer-terms/policy.html) の条項 — 特に、ポストの一部としての場合を除き、X API から取得した位置情報データやその他の地理情報を集約、キャッシュ、保存してはならず、そのような位置情報データや地理情報を単独で利用してはならない、という要件 — に従わなければなりません。

<div id="specific-examples">
  ## 具体的な例
</div>

例 1: *緯度・経度を明示的に追加しない場合（現在の X Web クライアントの UI）*

1. ユーザーがポストを作成するときに「位置情報を追加」のリンクを表示する。
2. バックグラウンドで X の [reverse&#95;geocode](https://developer.x.com/en/docs/geo/places-near-location/api-reference/get-geo-reverse_geocode.html) API を呼び出し、緯度、経度、測位の精度、およびユーザーの希望する粒度（デフォルトは「neighborhood」）を渡す。
3. デフォルトの場所名を表示する（X は、X クライアントがそのリストを変更せずに表示できるよう、意味の通る順序になるよう並べ替えを試みる）。
4. ユーザーがリストから別の場所を選択できるようにする（例: 「SoMa, San Francisco」から「South Beach, San Francisco」に変更する）。
5. 重要: 正確な位置情報を表示する場合は、ユーザーに対してその点を明確に示すこと。正確な位置情報が表示されることを、ユーザーが認識できるようにすること。
6. ユーザーがポストを作成するたびに、位置情報をオンまたはオフに切り替えられるようにする。

例 2: *既存のジオタグ実装*

多くの X クライアントアプリには、place&#95;ID を提供せずに緯度・経度のみをブロードキャストする、元々のジオタグ機能が実装されている。API のデフォルトの動作は、この挙動を模倣している。つまり、開発者としては、現在の挙動を維持するために既存のアプリを変更する必要はない。実際、単に緯度・経度の座標を X に渡すだけで、X は自動的にその緯度と経度を逆ジオコーディングし、データが利用可能な地域については、緯度・経度に加えて place&#95;ID を表示する。

ユーザーの正確な座標が X 上で使用・表示されていることが、ユーザーから見て明確になるようにすること。既存のアプリにおけるベストプラクティスとしては、ユーザーがポストごとに位置情報をオンまたはオフにできるようにすることが挙げられる。また、デフォルトの「スティッキー」設定として、ユーザーが正確な座標ではなく place&#95;ID のみを共有するように設定できるようにすることも可能である。

<div id="x-storage-of-user-location-data">
  ## X によるユーザー位置情報データの保存
</div>

X はクライアントから渡されたデータを次のように保存します。

* X は、ユーザーが自分のフォロワーに対して公開することを選択したすべてのデータを保存します。これは、ポストが行われた時刻を示すタイムスタンプを X が保存する方法と同様であり、ユーザーが正確な座標付きでポストした場合、ポストが存在する限り、X はそのポストとともにこの正確な位置情報を保存します。ユーザーは、[こちら](https://support.twitter.com/articles/122236-how-to-tweet-with-your-location) で説明されているように、設定ページの「Clear my location history」ボタンをクリックすることで、自分のポストから位置情報データを削除できます。

<div id="summary">
  ## まとめ
</div>

ポストに位置情報を付与すると、ユーザーにとってのコンテキストと価値が大きく高まります。ただし、クライアントで位置情報を実装する際には、位置情報に特化したプロダクトを設計するうえで、プライバシーとユーザーへの透明性を最優先に、慎重に取り扱う必要があります。
