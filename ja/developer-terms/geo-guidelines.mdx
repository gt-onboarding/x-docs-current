<div id="introduction">
  ## はじめに
</div>

Posting With Location は、X API におけるジオタグ（位置情報）機能です。この機能により、ポストに文脈を与えることで、ユーザーにとってより意味のある体験を提供できます。例えば、「Fireworks!（花火！）」というポストも、場所と結び付いていれば、はるかに関連性が高くなります。たとえば「Fireworks!」 ― アメリカ・カリフォルニア州サンフランシスコのソーマ地区からのポスト、といった具合です。私たちは、この機能を活用してすばらしい体験を創出するために、開発者エコシステムのみなさんと協力できることを楽しみにしています。

<div id="user-privacy-and-user-control-of-their-location-information">
  ## ユーザーのプライバシーと位置情報のユーザーによる管理
</div>

安全で快適なエクスペリエンスを提供するために、開発者はユーザーの位置情報を取り扱う際にユーザーのプライバシーに配慮するとともに、「位置情報付きでポストする」エクスペリエンスにおける一定の一貫性を確保する必要があります。以下は、ユーザーのプライバシーを保護し、ユーザーが自らの行動の影響を理解できるようにするために留意すべき重要なポイントです。

<div id="general-guidelines">
  ## 一般的なガイドライン
</div>

* ユーザーは、位置情報付きポスト機能（位置情報を「オン」にする）を利用するためにオプトインする必要があります。
* ユーザーは、自分のポストに正確な位置情報を表示することについて明示的に許可しなければなりません。クライアントの位置情報機能が「スティッキー」— つまり一度位置情報がオンになると常にポストに付随する — 場合は、場所名（place&#95;ids）のみを表示し、ユーザーがポストごと、または特定の代替設定として、正確な緯度・経度の表示についてさらにオプトインできるようにすることを検討してください。
* ポストを投稿するためにユーザーの位置情報データにアクセスする場合、その情報は、ユーザーの明示的な許可がない限り、ポスト本体とは別に保存してはなりません。
* ユーザーのポストに、どの程度の位置情報が（もしあれば）関連付けられて表示されるのかが、ユーザーにはっきり分かるようにしなければなりません。その一つの方法として、ポストの公開前に、この情報がどのように表示されるかを示す地図を表示することが挙げられます。クライアント上でユーザーによって公開される情報の一部（または唯一の情報）が正確な緯度・経度である場合は、正確な座標が公開されることをユーザーが明確に理解できるようにする必要があります。
* ユーザーは、ポストを作成するたびに、自分の位置情報をオン／オフできるようにする必要があります。
* X の [Developer Policy](/ja/developer-terms/policy.html) に従う必要があります。これには、ポストの一部である場合を除き、X API から取得した位置情報データやその他の地理情報を集約、キャッシュ、保存してはならず、そのような位置情報データや地理情報を単独で利用してはならないという要件が含まれます。

<div id="specific-examples">
  ## 具体的な例
</div>

例 1: *緯度・経度を明示的に追加しない場合（現在の X Web クライアントの UI）*

1. ユーザーがポストを作成する際に「場所を追加」というリンクを表示する。
2. バックグラウンドで X の [reverse&#95;geocode](https://developer.x.com/en/docs/geo/places-near-location/api-reference/get-geo-reverse_geocode.html) API を呼び出し、緯度、経度、位置情報の精度、およびユーザーが希望する粒度（デフォルトは「近隣地域」）を渡す。
3. デフォルトの場所名を表示する（X は、X クライアント側でリストをそのまま表示できるよう、意味のある順序になるよう並び替えを試みる）。
4. ユーザーがリストから別の場所を選択できるようにする（例: 「SoMa, San Francisco」から「South Beach, San Francisco」に変更する）。
5. 重要: 正確な位置情報を表示する場合は、ユーザーに対して透明性を保つこと。正確な位置情報が表示されるかどうかを、ユーザーが必ず認識できるようにすること。
6. ユーザーがポストを作成するたびに、位置情報をオンまたはオフにできるようにする。

例 2: *既存のジオタグ実装*

多くの X クライアント アプリは、place&#95;ID を付与せずに緯度・経度のみを送信する元のジオタグ機能を実装している。API のデフォルトの挙動はこの動作を模倣しており、開発者としては、既存のアプリの挙動を現状どおり維持するために変更を加える必要はない。実際、単に緯度・経度の座標を X に渡せば、X はその緯度・経度を自動的にリバースジオコーディングし、データが利用可能なエリアについては、緯度・経度に加えて place&#95;ID も表示する。

ユーザーの正確な座標が X 上で使用・表示されていることを、ユーザーに対して明確に示すようにすること。既存アプリにおけるベストプラクティスとしては、ユーザーがポストごとに位置情報をオンまたはオフにできるようにすることが挙げられる。また、デフォルトの「保持される」設定として、正確な座標ではなく place&#95;ID のみを共有するようにしておくことも可能である。

<div id="x-storage-of-user-location-data">
  ## X におけるユーザー位置情報データの保存
</div>

X はクライアントから渡されたデータを次のように保存します。

* X は、ユーザーがフォロワーに対して公開表示することを選択したすべてのデータを保存します。これは、ポストの投稿時刻を示すタイムスタンプを X が保存する仕組みと同様であり、ユーザーが正確な座標情報を付けてポストした場合、そのポストが存在する限り、X はその正確な位置情報をポストとともに保存します。ユーザーは、[こちら](https://support.twitter.com/articles/122236-how-to-tweet-with-your-location)) で説明されているように、設定ページの「位置情報履歴を消去」ボタンをクリックすることで、自分のポストから位置情報データを消去できます。

<div id="summary">
  ## まとめ
</div>

ポストに位置情報を追加すると、ユーザーの視点から見て非常に多くの文脈や価値が付与されます。しかし、クライアントアプリケーションで位置情報を実装する際は、位置情報を重視したプロダクトを設計するうえで、プライバシーとユーザーへの透明性を最優先で考慮し、慎重に実装する必要があります。
