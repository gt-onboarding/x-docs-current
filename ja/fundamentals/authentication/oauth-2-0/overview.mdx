<div id="bearer-token-also-known-as-app-only">
  ### Bearer Token（app-only とも呼ばれます）
</div>

OAuth 2.0 Bearer Token は、[開発者アプリ](/ja/resources/fundamentals/developer-apps)を代表してリクエストを認証します。この方式はアプリ固有であるため、ユーザーは関与しません。通常、公開情報への読み取り専用アクセスが必要な開発者向けに利用されます。 

この認証方式では、リクエストに Bearer Token を含める必要があります。Bearer Token は、開発者アプリの「キーとトークン」セクション内で生成できます。以下は、ダミーの Bearer Token を用いたリクエスト例です。

```json
  curl "https://api.x.com/2/tweets?ids=1261326399320715264,1278347468690915330" \
  -H "Authorization: Bearer AAAAAAAAAAAAAAAAAAAAAFnz2wAAAAAAxTmQbp%2BIHDtAhTBbyNJon%2BA72K4%3DeIaigY0QBrv6Rp8KZQQLOTpo9ubw5Jt?WRE8avbi"
```

アプリ専用認証を使用した API 呼び出しは、エンドポイントごとにアプリレベルで[レート制限](/ja/resources/fundamentals/rate-limits)の対象となります。

この方法を使用するには Bearer Token が必要です。Bearer Token は、API Key と Secret を [POST oauth2/token](/ja/resources/fundamentals/authentication/api-reference#post-oauth2-token) エンドポイントに渡して生成するか、[developer portal](/ja/resources/fundamentals/developer-portal) のアプリ設定内にある「keys and tokens」セクションで生成できます。

Bearer Token を失効させたい場合は、[POST oauth2/invalidate&#95;token](/ja/resources/fundamentals/authentication/api-reference#post-oauth2-invalidate-token) エンドポイントを使用するか、アプリ設定内の「keys and tokens」セクションで Bearer Token の横に表示されている「revoke」をクリックします。

<div id="oauth-20-authorization-code-flow-with-pkce">
  ### OAuth 2.0 Authorization Code Flow with PKCE
</div>

OAuth 2.0 Authorization Code Flow with PKCE を使用すると、別のユーザーに代わって認証を行う際にアプリケーションのスコープをより細かく制御でき、複数デバイス間での認可フローを改善できます。言い換えると、X を利用する人向けにアプリケーションを構築する開発者は、自身のアプリがユーザーから要求する情報をより厳密にコントロールできるようになり、エンドユーザーには本当に必要なデータと情報だけを要求すればよくなります。

この最新の認可プロトコルによって、エンドユーザーに対し、アプリの認可に関する同意フローをよりシンプルな形で提示できるようになります。このフローでは、要求した特定のスコープだけが表示されます。これにより、取り扱うデータ量を削減できるだけでなく、エンドユーザーからの信頼向上にもつながる可能性があります。
