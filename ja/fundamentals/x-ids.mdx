X 上の各オブジェクト ― ポスト、ダイレクトメッセージ、ユーザー、リストなど ― には固有の ID が付与されています。

プラットフォームのごく初期には、これらの ID は連番で生成できる程度に十分小さな数値でした。時間の経過とともに成長に対応するため、ID は 32 ビットから 64 ビットへと拡張されました。現在、X の ID は一意な 64 ビット符号なし整数であり、連番ではなく時刻に基づいています。ID 全体はタイムスタンプ、ワーカー番号、シーケンス番号で構成されています。X は、これらの ID を安定して生成するために「Snowflake」と呼ばれる内部サービスを開発しました（詳細は [X ブログの記事](https://blog.x.com/engineering/en_us/a/2010/announcing-snowflake.html) を参照してください）。

64 ビットという大きな数値は、整数を 64 ビット未満で表現するプログラミング言語において問題を引き起こす可能性があります。その一例が JavaScript であり、整数サイズが 53 ビットに制限されています。これに対する回避策を提供するため、X API（v1/1.1）の当初の設計では、ID の値は整数と文字列の 2 つの形式で返されていました。

```json
{"id": 10765432100123456789, "id_str": "10765432100123456789"}
```

ブラウザの JavaScript コンソールで `(10765432100123456789).toString()` コマンドを実行すると、結果は `"10765432100123458000"` になります。64 ビット整数は、この変換処理の過程で精度が失われます（これは「munging」（マングリング）と呼ばれることもあり、データが破壊的に変更されることを意味します）。

バージョン 1.1 までの X API では、精度の損失を避けるため、常に数値の文字列表現を使用する必要があります。

より新しいバージョンの API では、すべての大きな整数値はデフォルトで文字列として表現されます。
