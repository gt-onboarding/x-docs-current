X 上の各オブジェクト ― ポスト、ダイレクトメッセージ、ユーザー、リストなど ― には一意の ID が割り当てられています。

プラットフォーム初期には、これらの ID は連番で生成できるほど十分に小さい数値でした。時間の経過とともに成長に対応するため、ID は 32-bit から 64-bit へと移行しました。現在、X の ID は、連番ではなく時間に基づく一意の 64-bit 符号なし整数です。ID 全体は、タイムスタンプ、ワーカー番号、シーケンス番号で構成されています。X は、これらの ID を安定して生成するために「Snowflake」と呼ばれる内部サービスを開発しました（[詳細は X ブログをご覧ください](https://blog.x.com/engineering/en_us/a/2010/announcing-snowflake.html)）。

64-bit のような大きな数値は、64-bit 未満で整数を表現するプログラミング言語では問題を引き起こす可能性があります。その一例が JavaScript で、整数は 53-bit の大きさに制限されています。この問題の回避策として、X API（v1/1.1）の当初の設計では、ID 値は整数と文字列の 2 種類の形式で返されていました。

```json
{"id": 10765432100123456789, "id_str": "10765432100123456789"}
```

ブラウザの JavaScript コンソールで `(10765432100123456789).toString()` コマンドを実行すると、結果は `"10765432100123458000"` になります。64 ビット整数は、この変換の結果として精度が失われます（これは「munging」と呼ばれることがあり、データが破壊的に変更されることを指します）。

X API v1.1 以前のバージョンでは、精度の低下を避けるために、常に数値の文字列表現を使用する必要があります。

新しいバージョンの API では、すべての大きな整数値はデフォルトで文字列として表現されます。
