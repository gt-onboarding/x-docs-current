<div id="overview">
  ## Descripción general
</div>

Los elementos creativos son cualquier entidad que se pueda promocionar en una campaña. Las Publicaciones pueden incluir texto, imágenes, GIF, vídeos o tarjetas. Las tarjetas pueden incluir imágenes o vídeos.

Los elementos creativos de imagen, GIF o vídeo se cargan mediante [POST media/upload](/es/x-api/media/upload-media#post-media-upload), un endpoint de carga sencilla que solo admite imágenes, o mediante los endpoints POST media/upload (chunked). Estos se pueden agregar a

tarjetas:

* POST accounts/:account&#95;id/cards
  Publicaciones:
* POST accounts/:account&#95;id/tweets - Para agregar tarjetas a las Publicaciones, usa el parámetro card&#95;uri.
  Publicaciones programadas:
* POST accounts/:account&#95;id/scheduled&#95;tweets

Para obtener más detalles sobre las tarjetas, consulta la página Cards. La página Promoted Video proporciona detalles sobre cómo asociar vídeos con tarjetas o Publicaciones.

<div id="cards">
  ### Tarjetas
</div>

La Ads API admite varios tipos de tarjetas que se pueden usar en publicaciones, y que luego se pueden promocionar en campañas. **Nota**: una vez publicada la publicación, los detalles de la tarjeta son visibles públicamente. Esto puede incluir información sobre el usuario propietario de la tarjeta.

<div id="image">
  #### Imagen
</div>

Las siguientes especificaciones de imagen se aplican a los elementos creativos usados en [Cards](/es/x-ads-api/creatives#cards-2). Las imágenes deben tener un tamaño de 3 MB o menos y un ancho de al menos 800px. Además, son compatibles las siguientes relaciones de aspecto ancho:alto.

* Website: 1:1 y 1.91:1
* Image App Download: 1:1 y 1.91:1
* Poll: 1.91:1
* Image Conversation: 1.91:1
* Image Direct Message: 1.91:1

Son compatibles los siguientes formatos de imagen: .bmp, .jpeg y .png.

<div id="video">
  #### Video
</div>

Las siguientes especificaciones de video se aplican a los recursos usados en [Cards](/es/x-ads-api/creatives#cards-2). Se admiten las siguientes relaciones de aspecto (ancho:alto).

* Video Website: 16:9 y 1:1
* Video App Download: 16:9 y 1:1
* Poll: 16:9
* Video Conversation: 16:9
* Video Direct Message: 16:9

<div id="promoted-video">
  ### Video promocionado
</div>

Este documento ofrece una breve descripción general del proceso para subir y promocionar video mediante la Ads API.

La Ads API admite Video promocionado en [Publicaciones](/es/x-ads-api/creatives#tweets-2) y en las siguientes tarjetas:

* [Video Website](https://devcommunity.x.com/t/ads-api-version-11/168814)
* [Video App Download](https://devcommunity.x.com/t/ads-api-version-11/168814)
* [Video Conversation](/es/x-ads-api/creatives#video-conversation-cards)

Primero, sube el video usando el endpoint [POST media/upload (chunked)](https://developer.x.com/en/docs/media/upload-media/api-reference/post-media-upload-init.html). Usando el `media_id`, asocia el video con una cuenta de anuncios usando el endpoint POST accounts/:account&#95;id/videos. El `id` del video, a veces denominado `media_key`, se usará en solicitudes posteriores. Es una cadena de texto que comienza con un número entero, seguida de un guion bajo y que termina con un valor de tipo long. Como ejemplo, consulta: `13_875943225764098048`.

<div id="promoted-video-in-tweets">
  #### Video promocionado en Publicaciones
</div>

Para crear una Publicación, usa el endpoint [POST accounts/:account&#95;id/tweet](/es/x-ads-api/creatives#get-accounts-account-id-tweets) junto con el `id` del video. En este paso, también puedes proporcionar un título del video, una descripción y una llamada a la acción (CTA). Estos valores son visibles para los usuarios.

<div id="promoted-video-in-cards">
  #### Video promocionado en tarjetas
</div>

Las tarjetas de descarga de aplicaciones de video (Video App Download) y las tarjetas de conversación en video (Video Conversation) permiten añadir imágenes de portada. Sube una imagen para usarla en estas tarjetas mediante el endpoint [POST media/upload](https://developer.x.com/en/docs/media/upload-media/api-reference/post-media-upload.html).

Crea la tarjeta utilizando uno de los siguientes endpoints:

* [POST accounts/:account&#95;id/cards/video&#95;website](https://devcommunity.x.com/t/ads-api-version-11/168814)
* [POST accounts/:account&#95;id/cards/video&#95;app&#95;download](https://devcommunity.x.com/t/ads-api-version-11/168814)
* [POST accounts/:account&#95;id/cards/video&#95;conversation](https://devcommunity.x.com/t/ads-api-version-11/168814)

utilizando el `id` del video y, opcionalmente, el `media_id` de la imagen (para la imagen de portada).

Por último, crea la Publicación usando el endpoint POST accounts/:account&#95;id/tweet. Las tarjetas se adjuntan a las Publicaciones mediante el parámetro `card_uri`.

<div id="general-information">
  #### Información general
</div>

Para obtener instrucciones detalladas sobre la carga de vídeo a través de la API, consulta la [Guía de carga de vídeo](https://developer.x.com/en/docs/media/upload-media/uploading-media.html).

Los vídeos también se pueden promocionar como recursos pre-roll. Consulta la [Guía del objetivo de vistas de vídeo pre-roll](/es/x-ads-api/campaign-management#video-views-preroll-objective) para una explicación detallada.

* (A partir de 2015-10-22) Al cargar vídeos que se usarán en contenido promocionado, el parámetro `media_category` debe configurarse con el valor `amplify_video` para todas las solicitudes de comando `INIT` al endpoint [POST media/upload (chunked)](https://developer.x.com/en/docs/media/upload-media/api-reference/post-media-upload-init.html). El uso de este nuevo parámetro garantiza que el vídeo se procese de forma asíncrona y se prepare para su uso en contenido promocionado. El comando `STATUS` se puede usar para comprobar la finalización del procesamiento asíncrono después de la carga del vídeo.
* La duración máxima de vídeo promocionado permitida actualmente es de 10 minutos con un tamaño de archivo de 500 MB o menos.
* El vídeo cargado debe ser mp4 o mov.
* El vídeo cargado generalmente se procesa rápidamente, pero los tiempos de procesamiento pueden variar según la duración del vídeo y el tamaño del archivo.
* Las imágenes de portada cargadas deben estar en formato png o jpg. No hay requisitos de relación de aspecto o tamaño, pero la imagen de portada se ajustará para adaptarse al reproductor de vídeo.

<div id="scheduled-tweets">
  ### Publicaciones programadas
</div>

<div id="introduction">
  #### Introducción
</div>

Las Publicaciones programadas permiten que un anunciante o usuario cree una Publicación y la programe para que se publique en una fecha posterior. Además de poder crear y administrar estas Publicaciones, la API permite asociarlas con un line item para que se promocionen una vez que la Publicación esté activa. Esto permite a los anunciantes preparar Publicaciones nativas y planificar sus creatividades de campaña con antelación a cualquier iniciativa clave. Por ejemplo, preparar una creatividad de Publicación para que se publique inmediatamente tras el anuncio de un nuevo producto.

El conjunto completo de funciones que ofrecen los endpoints de la API de Publicaciones programadas se detalla a continuación:

* Crear, modificar y ver nuevas Publicaciones programadas
* Asociar una Publicación programada con un line item
* Consultar y administrar Publicaciones programadas existentes
* Una vez que una Publicación programada se haya publicado, recuperar el `id` de la Publicación activa

<div id="api-endpoints">
  #### Endpoints de la API
</div>

A continuación se presenta el conjunto completo de endpoints relacionados con la funcionalidad anterior:

<div id="scheduled-tweet-management">
  #### Gestión de publicaciones programadas
</div>

* [GET accounts/:account&#95;id/scheduled&#95;tweets](/es/x-ads-api/creatives#get-accounts-account-id-scheduled-tweets) (obtener una lista de todas las publicaciones programadas)
* [GET accounts/:account&#95;id/scheduled&#95;tweets/:scheduled&#95;tweet&#95;id](/es/x-ads-api/creatives#get-accounts-account-id-scheduled-tweets) (consultar una publicación programada específica usando su `id`)
* [POST accounts/:account&#95;id/scheduled&#95;tweets](/es/x-ads-api/creatives#post-accounts-account-id-scheduled-tweets) (crear una nueva publicación programada)
* [PUT accounts/:account&#95;id/scheduled&#95;tweets/:scheduled&#95;tweet&#95;id](/es/x-ads-api/creatives#example-request-39) (modificar una publicación programada existente)
* [DELETE accounts/:account&#95;id/scheduled&#95;tweets/:scheduled&#95;tweet&#95;id](/es/x-ads-api/creatives#example-request-40) (eliminar una publicación programada usando su `id`)
* [GET accounts/:account&#95;id/scheduled&#95;tweets/preview/:scheduled&#95;tweet&#95;id](/es/x-ads-api/creatives#scheduled-tweets-2) (mostrar una vista previa de una publicación programada existente)

<div id="scheduled-promoted-tweets">
  #### Publicaciones promocionadas programadas
</div>

* [GET accounts/:account&#95;id/scheduled&#95;promoted&#95;tweets](/es/x-ads-api/campaign-management#get-accounts-account-id-scheduled-promoted-tweets) (obtener una lista de todas las publicaciones promocionadas programadas)
* [GET accounts/:account&#95;id/scheduled&#95;promoted&#95;tweets/:scheduled&#95;promoted&#95;tweet&#95;id](/es/x-ads-api/campaign-management#get-accounts-account-id-scheduled-promoted-tweets-scheduled-promoted-tweet-id) (consultar una publicación promocionada programada mediante su `id`)
* [POST accounts/:account&#95;id/scheduled&#95;promoted&#95;tweets](/es/x-ads-api/campaign-management#post-accounts-account-id-scheduled-promoted-tweets) (crear una nueva publicación promocionada programada)
* [DELETE accounts/:account&#95;id/scheduled&#95;promoted&#95;tweets/:scheduled&#95;promoted&#95;tweet&#95;id](/es/x-ads-api/campaign-management#delete-accounts-account-id-scheduled-promoted-tweets-scheduled-promoted-tweet-id) (eliminar una publicación promocionada programada existente mediante su `id`)

<div id="scheduled-tweet-view">
  #### Ver Publicación Programada
</div>

* [GET accounts/:account&#95;id/scheduled&#95;tweets/preview/:scheduled&#95;tweet&#95;id](/es/x-ads-api/creatives#scheduled-tweets-2) (ver una Publicación Programada existente)

Debido a que las Publicaciones Programadas son entidades independientes de las Publicaciones “en vivo”, se ejecutan dos conjuntos diferentes de validaciones al crear o editar estas Publicaciones. El primer conjunto de reglas de validación se ejecuta durante el proceso de creación de la Publicación Programada, específicamente:

<div id="scheduled-tweet-create">
  #### Creación de Publicaciones programadas:
</div>

* Verifica que el usuario autenticado tenga acceso para crear Publicaciones orgánicas para un @handle determinado. Los privilegios para crear Publicaciones solo promocionadas requieren que el usuario autenticado sea un usuario de la cuenta con [permisos de Tweet composer](https://developer.x.com/content/developer-twitter/en/docs/ads/campaign-management/api-reference/authenticated-user-access#get-accounts-account-id-authenticated-user-access)
* Verifica que no haya más de 30 Publicaciones programadas para crearse dentro de una ventana de 15 minutos con respecto a la hora `scheduled_at`. Un mensaje de error `SCHEDULED_TWEET_LIMIT_EXCEEDED` indica que se han programado demasiadas Publicaciones programadas dentro del mismo intervalo futuro de 15 minutos. Los anunciantes tendrán que eliminar una Publicación programada existente o mover la hora `scheduled_at` a un momento anterior o posterior.

<div id="scheduled-tweet-goes-live">
  #### La Publicación programada se publica:
</div>

* Estas reglas de validación se ejecutan en la hora indicada en `scheduled_at` y son idénticas a las que se aplican al crear una Publicación normal en la API. Por ejemplo, una Publicación programada no se publicará y `scheduled_status` se establecerá en `FAILED` si la Publicación programada contiene tanto una imagen como un GIF.

<div id="workflow">
  #### Flujo de trabajo
</div>

**Crear una nueva Publicación programada**

Se puede crear una nueva Publicación programada utilizando el endpoint [POST accounts/:account&#95;id/scheduled&#95;tweets](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/scheduled-tweets#post-accounts-account-id-scheduled-tweets). Este endpoint tiene los siguientes parámetros obligatorios: la hora `scheduled_at` junto con el `text` de la Publicación si no se incluyen entidades de contenido multimedia en la Publicación. Además, este endpoint ofrece algunas opciones adicionales que te permiten crear una Publicación programada en nombre de otro @handle mediante el parámetro `as_user_id`, junto con la posibilidad de agregar una card (`card_uri`) y cualquier contenido multimedia (`media_ids`). Ten en cuenta que una Publicación solo puede contener entidades de un mismo tipo, es decir, ya sea video, GIF o imagen. El parámetro `nullcast` controla si la Publicación es una Publicación “solo promocionada” o no. Todas las Publicaciones programadas recién creadas son “solo promocionadas” (`nullcast=true`) de forma predeterminada. Si `nullcast=false`, entonces se crea una Publicación programada orgánica.

Una vez que se crea correctamente una Publicación programada, la respuesta contendrá un campo `id`, que hace referencia al identificador único de la propia Publicación programada. Además de este campo, también se devuelve otro campo llamado `tweet_id`. Este campo es `null` inicialmente; sin embargo, una vez que la Publicación se publica, este campo se completa con el id de la Publicación “activa”.

```
twurl -H 'ads-api.x.com' -X POST "/6/accounts/:account_id/scheduled_tweets" -d 'scheduled_at=2017-12-24T23:59:00Z&text=Happy Holidays!'
```

Esto creará la siguiente publicación programada:

```json
{
  "request": {
    "params": {
      "text": "Happy Holidays!",
      "scheduled_at": "2017-12-24T23:59:00Z"
    }
  },
  "data": {
    "completed_at": null,
    "id_str": "917507899668099072",
    "text": "Happy Holidays!",
    "user_id": "3271358660",
    "scheduled_status": "SCHEDULED",
    "id": 917507899668099100,
    "nullcast": true,
    "created_at": "2017-10-09T21:51:44Z",
    "scheduled_at": "2017-12-24T23:59:00Z",
    "card_uri": null,
    "updated_at": "2017-10-09T21:51:44Z",
    "tweet_id": null,
    "media_keys": []
  }
}
```

Una vez que esta Publicación programada se publique, el campo `tweet_id` se completará con el ID de la Publicación &quot;en vivo&quot;.

La respuesta de la API contendrá una URL de iframe lista para usarse para renderizar una vista previa de la Publicación programada. El CSS correspondiente y las imágenes se servirán directamente desde X.

**Asociar una Publicación programada con un line item**

Si bien las Publicaciones programadas se pueden usar para crear Publicaciones orgánicas, también permitimos que los socios creen una Publicación “solo promocionada” (`nullcast=true`), y cualquiera de ellas se puede asociar con un line item. Para facilitar esto, también proporcionamos el endpoint [POST accounts/:account&#95;id/scheduled&#95;promoted&#95;tweets](https://developer.x.com/content/developer-twitter/en/docs/ads/campaign-management/api-reference/scheduled-promoted-tweets#post-accounts-account-id-scheduled-promoted-tweets). Este endpoint solo permite asociar una única Publicación programada promocionada con un line item en una única llamada a la API. Para asociar varias Publicaciones programadas con el mismo line item, es necesario realizar varias llamadas a la API.

Ten en cuenta que no es posible modificar una Publicación programada promocionada existente.

```
twurl -H 'ads-api.x.com' -X POST "/6/accounts/:account_id/scheduled_promoted_tweets" -d 'line_item_id=a44qc&scheduled_tweet_id=917507899668099072'
```

```json
{
  "data": {
    "line_item_id": "a44qc",
    "id": "26576",
    "created_at": "2017-10-09T22:24:16Z",
    "updated_at": "2017-10-09T22:24:16Z",
    "scheduled_tweet_id": "917507899668099072",
    "tweet_id": null,
    "deleted": false
  },
  "request": {
    "params": {
      "line_item_id": "a44qc",
      "scheduled_tweet_id": 917507899668099100,
      "account_id": "aaaaa"
    }
  }
}
```

Este endpoint solo crea una asociación entre una Publicación programada determinada y un line item. Una vez que las fechas de vuelo de la campaña/line item sean actuales, el line item comenzará automáticamente a difundir la Publicación “activa” correspondiente. Aunque durante este paso sí validamos que la Publicación programada esté en el estado `SCHEDULED` y que la Publicación programada indicada sea válida para el objetivo dado, no se ejecutan otras validaciones. Cualquier regla de validación restante que aplique al line item y a la Publicación programada se ejecuta cuando la Publicación pasa a estar “activa”.

Para asegurarse de que no haya problemas con la entrega de la campaña, se recomienda que a la Publicación programada se le asigne un `scheduled_at` en un momento anterior a las fechas de vuelo de la campaña/line item.

Por ejemplo, supongamos que la Publicación programada está configurada para activarse después de la fecha de inicio de la campaña (y que solo hay una única Publicación asociada a un único line item). En ese caso, la campaña estará `ACTIVE`; sin embargo, dado que la Publicación programada aún no está activa, no habrá creativos disponibles para su entrega.

**Gestión de Publicaciones programadas**

Los conjuntos restantes de endpoints permiten a los consumidores de la API gestionar todas sus Publicaciones programadas y Publicaciones promocionadas programadas. Estas API se pueden usar para devolver una lista de todas las Publicaciones programadas, opcionalmente filtradas por un estado determinado, así como para buscar una Publicación programada específica por su `id`.

<div id="what-happens-when-a-scheduled-tweet-goes-live">
  #### ¿Qué ocurre cuando una Publicación programada se publica?
</div>

Una vez que una Publicación programada está a punto de publicarse, o dicho de otro modo, en el momento `scheduled_at`, se realizan las siguientes actualizaciones:

* Se crea la Publicación “en vivo”; sin embargo, esto puede tener una latencia de hasta 1 segundo
* El `tweet_id` se añade a las siguientes entidades:
* Publicación programada
* Publicación programada promocionada
* Se crea una nueva entidad de Publicación promocionada

<div id="best-practices">
  #### Mejores prácticas
</div>

Se recomiendan las siguientes mejores prácticas al crear o promocionar Publicaciones programadas:

* Asegúrate de que la Publicación sea válida al crear la Publicación programada (por ejemplo, una Publicación solo puede tener una imagen, video o GIF, y no una combinación de los tres)
* Asegúrate de que las fechas de vigencia de la campaña (es decir, `start_time` y `end_time`) sean coherentes con la hora `scheduled_at` de la Publicación programada
* Las Publicaciones programadas no deben programarse para más de un año en el futuro (365 días)
* Actualmente no está disponible la vista previa para Publicaciones programadas (es decir, la posibilidad de previsualizar Publicaciones programadas antes de crearlas)

### Biblioteca multimedia

<div id="introduction">
  #### Introducción
</div>

Los endpoints de Media Library permiten administrar imágenes, GIFs y vídeos para cuentas de X Ads. Los recursos de contenido multimedia en la biblioteca se pueden usar en Publicaciones y para crear tarjetas. También se pueden reutilizar en múltiples [creativos](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/overview), lo que elimina la necesidad de subir el mismo recurso varias veces.

#### Extremos de la API

* [POST media/upload](https://developer.x.com/content/developer-twitter/en/docs/media/upload-media/api-reference/post-media-upload#post-media-upload) (subir contenido multimedia) o [POST media/upload (chunked)](https://developer.x.com/content/developer-twitter/en/docs/media/upload-media/api-reference/post-media-upload-init#post-media-upload-init) (subir contenido multimedia en fragmentos)

<div id="request-parameters">
  #### Parámetros de la solicitud
</div>

Todas las solicitudes POST de Media Library requieren un identificador de contenido multimedia. Este valor se devuelve durante el paso de carga. Al usar el media&#95;id, como en el ejemplo anterior, también se debe especificar un media&#95;category. Hay cuatro posibles valores de categoría: AMPLIFY&#95;VIDEO, TWEET&#95;GIF, TWEET&#95;IMAGE y TWEET&#95;VIDEO.

Opcionalmente, se pueden establecer valores de name y file&#95;name para los objetos de la Media Library. Estos atributos ayudan a los usuarios a distinguir entre variantes de contenido multimedia en la biblioteca.

Para videos, también es posible establecer un título y una descripción. Estos valores están pensados para enviarse como los parámetros de solicitud video&#95;title y video&#95;description con el endpoint [POST accounts/:account&#95;id/tweet](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/tweets#post-accounts-account-id-tweet). En la Publicación, este texto aparece debajo del video.

<div id="attributes">
  #### Atributos
</div>

La Media Library introduce formalmente el concepto de media&#95;key. Este es el identificador único de los objetos en la biblioteca. Las media keys son cadenas de texto con el siguiente formato: 13&#95;875943225764098048. Estas están totalmente admitidas en todos nuestros endpoints de Cards.

Además, la respuesta de la Media Library incluye el media&#95;id, representado como una cadena de texto. Esto se incluye para los recursos que actualmente no aceptan una media key: [Publicaciones](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/tweets#post-accounts-account-id-tweet)*, [Vista previa de la Publicación](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/tweets#get-accounts-account-id-tweet-preview)* y [Publicaciones programadas](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/scheduled-tweets#post-accounts-account-id-scheduled-tweets). Estamos trabajando para admitir media keys en todos los casos.

El atributo aspect&#95;ratio se devuelve para GIFs y videos. Se puede usar para filtrar contenido multimedia que se usará en Cards que solo aceptan determinadas relaciones de aspecto.

*Estos endpoints admiten el parámetro video&#95;id, que es una media key.

<div id="introduction">
  #### Introducción
</div>

Las Cards son formatos de anuncio personalizables que usan contenido multimedia y que se pueden asociar con un sitio web, una aplicación o con llamadas a la acción para incentivar ciertos tipos de interacción de los usuarios, como iniciar un mensaje directo. Se pueden adjuntar a publicaciones, publicaciones programadas o publicaciones en borrador.

Las Cards pueden aparecer referenciadas en los objetos de publicación de dos maneras: mediante el card&#95;uri de la Card o mediante su preview&#95;url. A continuación se presentan valores de ejemplo para cada uno.

| card&#95;uri | preview&#95;url |
| :--- | :--- |
| card://1043282691834048513 | https://cards.x.com/cards/18ce54d4x5t/68w3s |

**Nota**: A partir de la versión 3 de la Ads API, solo se genera y devuelve el card&#95;uri en la respuesta de cards para las Cards creadas recientemente.

**Nota**: A partir de la versión 5 de la Ads API, el preview&#95;url en la respuesta de cards ya no se devuelve.

El tipo de referencia en la respuesta del objeto de la publicación dependerá de la forma en que se creó la publicación. En otras palabras, si la publicación se creó usando el parámetro de solicitud card&#95;uri, el valor del URI de la Card aparecerá en la respuesta. Si, por el contrario, el preview&#95;url se incluyó como parte del texto de la publicación, la URL de vista previa aparecerá en la respuesta.

<div id="identifying-media">
  ### Identificar contenido multimedia
</div>

#### Introduction

El contenido multimedia (imágenes, GIF y videos) se puede añadir a las Publicaciones y a las tarjetas. Además, los videos se pueden usar como recursos de pre-roll y las imágenes se pueden promocionar en la [X Audience Platform](https://developer.x.com/en/docs/ads/measurement/overview/twitter-audience-platform). Esta sección describe cómo encontrar referencias de contenido multimedia en estas entidades.

Hay dos tipos de identificadores de contenido multimedia: IDs y claves. A continuación se muestran valores de ejemplo para cada uno.

| **Media ID** | **Media key** |
| :--- | :--- |
| 1029825579531807971 | 13&#95;1029825579531807971 |

La clave de contenido multimedia es la ID más un prefijo numérico y un guion bajo.

<div id="images">
  #### Imágenes
</div>

La siguiente tabla muestra los tipos de identificadores disponibles actualmente en la respuesta de cada recurso relacionado con imágenes, así como el nombre de los atributos correspondientes.

| **Resource** | **Identifier** | **Attribute(s)** |
| :--- | :--- | :--- |
| Image cards | None |  |
| Tweet | Both | `entities["media"]["id_str"]`  `entities["media"]["media_key"]` |
| Scheduled Tweet | Both | `media_ids` and `media_keys` |
| Draft Tweet | Both | `media_ids` and `media_keys` |
| Account Media | None |  |
| Media Library | Both | `media_id` and `media_key` |

Las Image cards y las imágenes de Account Media no incluyen ninguna referencia a identificadores de contenido multimedia. Los Tweets solo incluyen media IDs. Los Scheduled y Draft Tweets incluyen tanto el media ID como la media key. La Media Library también devuelve ambos.

En el caso de los Tweets, los campos id e id&#95;str en el objeto dentro del array entities[&quot;media&quot;] corresponden al media ID. En los casos en que un Tweet incluye varias imágenes, las referencias a cada media entity solo se pueden encontrar en extended&#95;entities[&quot;media&quot;].

Además de las referencias a identificadores, a menudo es importante tener acceso a la URL de la imagen.

| **Resource** | **Attribute(s)** | **Format** |
| :--- | :--- | :--- |
| Image cards | `image` | .jpg |
| Tweet* | `entities["media"][0]["media_url"]` or `extended_entities["media"][i]["media_url"]` | .jpg |
| Scheduled Tweet | None |  |
| Draft Tweet | None |  |
| Account Media | `media_url` | .jpg |
| Media Library | `media_url` | .jpg |

* La ubicación de esta URL depende de si el Tweet contiene una sola imagen o varias imágenes.

Todas las Image cards incluyen un atributo de respuesta de imagen que contiene la URL de la imagen de X. (Para las image app download cards, el nombre es wide&#95;app&#95;image).

En el caso de los Tweets, la ubicación de la media URL depende tanto del tipo de contenido multimedia como del endpoint que se esté utilizando. Para Tweets con una sola imagen, la URL se puede encontrar en entities[&quot;media&quot;][0][&quot;media&#95;url&quot;]. Esto es cierto tanto para la Ads API como para la Standard API. Sin embargo, cuando los Tweets contienen varias imágenes, las URLs solo se pueden encontrar en extended&#95;entities[&quot;media&quot;][i][&quot;media&#95;url&quot;]. Esto solo está disponible en la Standard API.

<div id="media-library">
  #### Biblioteca de contenido multimedia
</div>

En ocasiones es necesario obtener información adicional sobre un activo de contenido multimedia. Un caso de uso, en tarjetas de vídeo, es recuperar la URL mp4 en lugar de la vmap. Esto está disponible en la Biblioteca de contenido multimedia. Para obtener detalles sobre la información disponible, consulta nuestra [guía de la Biblioteca de contenido multimedia](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/guides/media-library). La mayoría de los activos que pertenecen al usuario promocionable de nivel FULL de la cuenta de anuncios se pueden encontrar en la biblioteca. Sin embargo, hay algunas excepciones.

**Obtención de contenido multimedia**

Como se indicó anteriormente, las tarjetas de imagen no contienen referencias ni a media IDs ni a claves de contenido multimedia. Como resultado, no es posible obtener sus activos a través de la Biblioteca de contenido multimedia. Esto también se aplica a las imágenes de [Account Media](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/account-media#account-media).

Las tarjetas de vídeo requieren que el activo de vídeo forme parte de la Biblioteca de contenido multimedia (o del recurso Videos antes de esto) antes de crear la tarjeta. Como resultado, estos activos siempre se podrán obtener en la Biblioteca de contenido multimedia. Esto también se aplica a los activos PREROLL de Account Media.

Por último, se garantiza que el contenido multimedia en las Publicaciones siempre esté en la Biblioteca de contenido multimedia.

La siguiente tabla resume qué activos se pueden obtener en la Biblioteca de contenido multimedia, teniendo en cuenta si la respuesta del recurso incluye un identificador que se pueda usar en la búsqueda.

| **Recurso** | **En la Biblioteca de contenido multimedia** |
| :--- | :--- |
| Tarjetas de imagen | No |
| Tarjetas de vídeo | Sí* |
| Publicaciones (cualquier contenido multimedia)** | Sí |
| Publicaciones programadas | Sí |
| Publicaciones en borrador | Sí |
| Imágenes de Account Media | No |
| Vídeos de Account Media (`PREROLL`) | Sí |

* Para tarjetas en las que el `video_content_id` es una clave de contenido multimedia. Cuando el valor es un media ID, el activo aún existe en la Biblioteca de contenido multimedia, pero obtenerlo implica anteponerle un prefijo numérico y un guion bajo.
** Las Publicaciones solo devuelven media IDs. Aunque se garantiza que el activo exista en la Biblioteca de contenido multimedia, obtenerlo implica anteponerle un prefijo numérico y un guion bajo.

**Interacciones con Account Media**

Hay dos casos en los que los activos de contenido multimedia agregados a la biblioteca se añaden automáticamente al recurso Account Media.

* Cuando se agrega un activo AMPLIFY&#95;VIDEO a la Biblioteca de contenido multimedia, se agrega automáticamente como un activo de Account Media con el tipo de creativo PREROLL.
* Cuando se agregan a la Biblioteca de contenido multimedia imágenes que tienen dimensiones específicas (consulta &quot;Creative Types&quot; en nuestra [página de enumeraciones](https://developer.x.com/content/developer-twitter/en/docs/ads/general/overview/enums)), se agregan automáticamente como activos de Account Media. El tipo de creativo (por ejemplo, INTERSTITIAL) depende de las dimensiones de la imagen.

<div id="tweets">
  ### Publicaciones
</div>

<div id="introduction">
  #### Introducción
</div>

La API de anuncios de X admite tres tipos de publicaciones: publicadas, programadas y en borrador.

<div id="nullcasted-tweets">
  #### Publicaciones nullcasted
</div>

Las publicaciones pueden ser nullcasted (también conocidas como «solo promocionadas») u orgánicas. Las publicaciones nullcasted, una vez publicadas, no aparecen en la cronología pública del usuario, aunque son públicas. Las publicaciones orgánicas, por otro lado, se muestran a los seguidores del usuario y sí aparecen en la cronología pública del usuario.

**Creación de publicaciones**

Cada uno de los tres endpoints de creación de publicaciones admite un parámetro booleano `nullcast` que ofrece al usuario de la API la opción de crear publicaciones nullcasted u orgánicas. Las publicaciones nullcasted pueden ser creadas por el usuario o por cualquier persona que tenga permiso para crear publicaciones en nombre del usuario. Las publicaciones orgánicas solo pueden ser creadas por el [usuario totalmente promocionable](https://developer.x.com/content/developer-twitter/en/docs/tutorials/ads-api-hierarchy-terminology#promotable-users).

**Actualización de publicaciones**

Es posible actualizar la propiedad `nullcast` para publicaciones programadas y borradores. En el caso de las publicaciones programadas, las ediciones se pueden realizar hasta la hora `scheduled_at` de la publicación. Los borradores de publicaciones se pueden editar indefinidamente. Sin embargo, una vez publicadas, no es posible cambiar una publicación de nullcasted a orgánica o viceversa.

<div id="promoting-tweets">
  #### Promocionar Publicaciones
</div>

Solo las Publicaciones publicadas y programadas se pueden promocionar. Estas pueden ser nullcasted u orgánicas; no hay ninguna restricción. Un anunciante puede promocionar sus propias Publicaciones o las de otro usuario, siempre que haya obtenido permiso para hacerlo. (Consulta [Promoting another user&#39;s Tweets](https://developer.x.com/content/developer-twitter/en/docs/tutorials/promoting-another-users-content) para obtener más información).

Se pueden promocionar múltiples Publicaciones en una sola campaña. Del mismo modo, una sola Publicación se puede promocionar en una o más campañas.

Para promocionar Publicaciones publicadas, usa el endpoint [POST accounts/:account&#95;id/promoted&#95;tweets](https://developer.x.com/content/developer-twitter/en/docs/ads/campaign-management/api-reference/promoted-tweets#post-accounts-account-id-promoted-tweets). Este endpoint asocia las Publicaciones publicadas con un line item. Para promocionar Publicaciones programadas, usa el endpoint [POST accounts/:account&#95;id/scheduled&#95;promoted&#95;tweets](https://developer.x.com/content/developer-twitter/en/docs/ads/campaign-management/api-reference/scheduled-promoted-tweets#post-accounts-account-id-scheduled-promoted-tweets).

<div id="carousels">
  ### Carruseles
</div>

<div id="introduction">
  #### Introducción
</div>

La X Ads API permite crear y obtener carruseles de video y carruseles de imagen. El carrusel es un tipo de tarjeta que puede contener entre 2 y 6 elementos de contenido multimedia. La tarjeta de carrusel puede dirigir a un usuario a un sitio web o animarlo a instalar una aplicación móvil. Para obtener más información sobre los carruseles, sus beneficios, buenas prácticas y preguntas frecuentes, consulta nuestra página [Carousel Ads on X](https://business.x.com/en/advertising/carousels.html).

Un carrusel, como cualquier otro tipo de tarjeta, puede usarse en Publicaciones y esas Publicaciones luego pueden promocionarse. El flujo de trabajo es el mismo al que ya estás acostumbrado:

1. Subir contenido multimedia
2. Crear la tarjeta
3. Crear la Publicación
4. Promocionar la Publicación

La única diferencia está en cómo se crea la tarjeta. Mientras que otras solicitudes de creación de tarjetas aceptan parámetros de consulta, las solicitudes de creación de tarjetas de carrusel solo aceptan cuerpos de solicitudes POST en JSON.

<div id="endpoints">
  #### Endpoints
</div>

La API de Ads permite crear y obtener carruseles.

Para crear cualquier tipo de carrusel, usa el endpoint [POST accounts/:account&#95;id/cards](https://developer.x.com/en/docs/twitter-ads-api/creatives/api-reference/cards#post-accounts-account-id-cards). Para obtener carruseles, usa el endpoint [GET accounts/:account&#95;id/cards](https://developer.x.com/en/docs/twitter-ads-api/creatives/api-reference/cards#get-accounts-account-id-cards).

<div id="json-post-body">
  #### Cuerpo de la solicitud POST en JSON
</div>

Los carruseles se crean utilizando dos componentes. El primero especifica los recursos multimedia que se usarán. El segundo especifica información sobre el sitio web o la aplicación.

En concreto, una tarjeta de carrusel se crea con los siguientes componentes, en este orden:

* Un componente `SWIPEABLE_MEDIA`, que acepta un arreglo de claves de contenido multimedia
* *Uno* de los siguientes:
* Un componente `DETAILS` para especificar la información del sitio web
* Un componente `BUTTON` para especificar la información de la aplicación

El componente `SWIPEABLE_MEDIA` debe incluir un arreglo `media_keys` donde puedes especificar entre 2 y 6 imágenes o videos. El orden en el que se pasan las claves de contenido multimedia determina el orden en el que se mostrarán.

```
{
  "type": "SWIPEABLE_MEDIA",
  "media_keys": [
    "13_1089771253848666112",
    "13_1191948012077092867"
  ]
}
```

Como recordatorio, puedes obtener claves de contenido multimedia haciendo una solicitud al endpoint [GET accounts/:account&#95;id/media&#95;library](https://developer.x.com/en/docs/twitter-ads-api/creatives/api-reference/media-library#get-accounts-account-id-media-library).

La composición del segundo objeto de componente depende de si deseas dirigir a un usuario a un sitio web o animarlo a instalar una aplicación. La siguiente tabla resume las dos opciones. (**Nota**: todas las claves enumeradas son obligatorias).

|                                  | **Sitio web**                                                      | **Aplicación**                                    |
| :------------------------------- | :----------------------------------------------------------------- | :------------------------------------------------ |
| Especifica el tipo de componente | `"type": "DETAILS"`                                                | `"type": "BUTTON"`                                |
| Título/Etiqueta                  | `"title": "X"`                                                     | `"label": { "type": "ENUM", "value": "INSTALL" }` |
| Destino                          | `"destination": { "type": "WEBSITE", "url": "https://www.x.com" }` | `"destination": { "type": "APP", ... }`           |

Con todo esto, a continuación se muestra un ejemplo de cuerpo de la solicitud POST en JSON para un carrusel de sitio web.

```
{
  "name": "website carousel",
  "components": [
    {
      "type": "SWIPEABLE_MEDIA",
      "media_keys": [
        "13_1089771253848666112",
        "13_1191948012077092867"
      ]
    },
    {
      "type": "DETAILS",
      "title": "X",
      "destination": {
        "type": "WEBSITE",
        "url": "https://www.x.com"
      }
    }
  ]
}
```

Los objetos de destino de aplicación dentro de los componentes `BUTTON` requieren un código de país y al menos un identificador de aplicación. Opcionalmente aceptan deep links. Para obtener una descripción de estos campos, consulta la [documentación de referencia](https://developer.x.com/en/docs/twitter-ads-api/creatives/api-reference/cards#app-destination).

Con todo esto, a continuación se muestra un ejemplo de cuerpo JSON de POST para un carrusel de aplicaciones.

```
{
  "name": "app carousel",
  "components": [
    {
      "type": "SWIPEABLE_MEDIA",
      "media_keys": [
        "13_1089771253848666112",
        "13_1191948012077092867"
      ]
    },
    {
      "type": "BUTTON",
      "label": {
        "type": "ENUM",
        "value": "INSTALL"
      },
      "destination": {
        "type": "APP",
        "country_code": "US",
        "googleplay_app_id": "com.twitter.android",
        "iphone_app_id": "333903271"
      }
    }
  ]
}
```

<div id="creative-metadata-tagging">
  ### etiquetado-de-metadatos-de-creatividades
</div>

<div id="introduction">
  #### Introducción
</div>

Esta guía está dirigida a socios creativos, agencias y desarrolladores creativos para etiquetar recursos utilizados en campañas de X y así comprender mejor el valor y el rendimiento de cada recurso individual.

**Nota:** Los recursos de contenido multimedia solo deben ser etiquetados por el socio o desarrollador que crea el recurso de contenido multimedia. Si el usuario del recurso de contenido multimedia no creó dicho recurso, no se debe implementar el etiquetado de metadatos.

Creative Metadata Tagging proporciona atribución de imágenes y videos creados por socios creativos, sin importar dónde se cargue el recurso en X ni qué entidad cargue el recurso. Para crear la conexión entre el recurso creativo y el socio creativo, se utiliza el estándar [XMP](https://exiftool.org/TagNames/XMP.html).

<div id="tagging-creative-assets">
  #### Etiquetado de recursos creativos
</div>

La siguiente tabla muestra los tipos de identificadores disponibles actualmente en la respuesta de cada recurso relacionado con imágenes, así como los nombres de atributos correspondientes. Necesitas una herramienta de etiquetado para etiquetar los recursos creativos. Se recomienda [ExifTool](https://exiftool.org/), una biblioteca de Perl multiplataforma junto con una [aplicación de línea de comandos](https://exiftool.org/exiftool_pod.html) para leer, escribir y editar metadatos. Consulta todos los [tipos de archivo](https://exiftool.org/#supported) compatibles.

Sigue las [instrucciones para instalar ExifTool](https://exiftool.org/install.html) proporcionadas. También hay paquetes de software ofrecidos por [Homebrew](https://formulae.brew.sh/) para simplificar aún más la instalación, proporcionando el [comando de instalación de exiftool](https://formulae.brew.sh/formula/exiftool) para macOS y Linux. Confirma que tu herramienta está correctamente instalada introduciendo exiftool -ver en la línea de comandos para obtener el número de versión de la herramienta. Obtén más información sobre los parámetros de comando de ExifTool en la [documentación de ExifTool](https://exiftool.org/).

Los socios creativos pueden asignar etiquetas de metadatos a recursos creativos nuevos o existentes con su X app&#95;id a la etiqueta XMP contributor y a la etiqueta date. Los recursos creativos seguirán las restricciones de tamaño existentes al [subir contenido multimedia](https://developer.x.com/en/docs/twitter-api/v1/media/upload-media/overview). 

**Nota:** El uso que hace X de la etiqueta XMP contributor garantiza que los metadatos capturen valores únicamente para campañas en X.

`exiftool -contributor="<YOUR APP ID>" -creative_file.jpg`

`exiftool -date="<date>" -creative_file.jpg`

<div id="components">
  #### Componentes
</div>

Cada componente debe incluir un campo `type` que determina el esquema del objeto. La Ads API admite los siguientes tipos de componentes, agrupados en componentes basados en contenido multimedia y en descripción.

* Contenido multimedia:
* `MEDIA`: un único video o una única imagen
* `SWIPEABLE_MEDIA`: entre 2 y 6 videos o imágenes
* Descripción:
* `DETAILS`
* `BUTTON`

Cada componente tiene un conjunto de campos obligatorios (además de la clave `type`). Estos se enumeran en la siguiente tabla.

| Component `type`  | Field                   | Value type       |
| :---------------- | :---------------------- | :--------------- |
| `MEDIA`           | `media_key`             | string           |
| `SWIPEABLE_MEDIA` | `media_keys`            | array of strings |
| `DETAILS`         | `title`   `destination` | string   object  |
| `BUTTON`          | `label`   `destination` | object   object  |

A continuación se muestra un ejemplo de un componente `BUTTON` en el contexto del array `components` (omitiendo intencionalmente la clave `name`). (Los puntos suspensivos indican los lugares donde sería necesario especificar más información.)

```
{
  "components": [
    {
      "type": "BUTTON",
      "label": {
        ...
      },
      "destination": {
        ...
      }
    }
  ]
}
```

El orden en el que se especifican los objetos de componentes define el orden de arriba hacia abajo en el que se renderizarán. Las Cards deben crearse con un componente basado en contenido multimedia y un componente `DETAILS` o `BUTTON`. Los componentes basados en descripciones se renderizan debajo del contenido multimedia y tienen destinos asociados, ya sean URLs o aplicaciones móviles.

**Label**

Las etiquetas definen el texto que se muestra en los botones y, por lo tanto, solo se aplican al componente `BUTTON`. Los objetos de tipo label tienen dos claves obligatorias: `type` y `value`. El `type` debe establecerse en `ENUM` y el `value` puede ser uno de los siguientes: `BOOK`, `CONNECT`, `INSTALL`, `OPEN`, `ORDER`, `PLAY` o `SHOP`.

Siguiendo con el ejemplo anterior, a continuación se muestra el objeto `label` dentro del componente `BUTTON`.

```
{
  "components": [
    {
      "type": "BUTTON",
      "label": {
        "type": "ENUM",
        "value": "INSTALL"
      },
      "destination": {
        ...
      }
    }
  ]
}
```

**Destino**

Los destinos son el lugar adonde los anunciantes desean llevar a los usuarios. Siempre son obligatorios dentro de los componentes `DETAILS` o `BUTTON`. Hay dos tipos de destino: `WEBSITE` o `APP`.

**Nota**: Los destinos de sitio web solo se pueden usar con componentes `DETAILS` y los destinos de aplicación solo se pueden usar con componentes `BUTTON`.

  #### POST accounts/:account&#95;id/tweet
</div>

Crea una Publicación para el usuario promocionable `FULL` de la cuenta (valor predeterminado) o para el usuario especificado en el parámetro `as_user_id`. Se admite tanto la creación de Publicaciones nullcast (predeterminada) como de Publicaciones orgánicas. Las Publicaciones nullcast no aparecen en la cronología pública ni se muestran a los seguidores. Cualquiera de los dos tipos puede usarse en campañas.

Si el usuario autenticado no es el usuario promocionable `FULL` de esta cuenta, determina si tiene permiso para publicar en nombre de este usuario realizando una solicitud al endpoint [GET accounts/:account&#95;id/authenticated&#95;user&#95;access](https://developer.x.com/en/docs/ads/campaign-management/api-reference/authenticated-user-access#get-accounts-account-id-authenticated-user-access). Un permiso de `TWEET_COMPOSER` indica que el usuario puede usar este endpoint para crear Publicaciones nullcast en nombre del [usuario promocionable](https://developer.x.com/en/docs/ads/campaign-management/api-reference/promotable-users#get-accounts-account-id-promotable-users) `FULL`.

Cuando uses el [endpoint upload.x.com](https://developer.x.com/rest/reference/post/media/upload) para contenido multimedia, pasa el mismo valor de `user_id` para el parámetro `additional_owners` que el valor de `as_user_id` que envías a este endpoint.
