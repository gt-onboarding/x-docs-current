<div id="overview">
  ## Descripción general
</div>

Los creativos son entidades que se pueden promocionar en una campaña. Las Publicaciones pueden incluir texto, imágenes, GIFs, videos o cards. Las cards pueden incluir imágenes o videos.

Los creativos de imagen, GIF o video se cargan mediante [POST media/upload](/es/x-api/media/upload-media#post-media-upload), un endpoint de carga sencilla que solo admite imágenes, o mediante los endpoints POST media/upload (chunked). Estos se pueden agregar a

cards:

* POST accounts/:account&#95;id/cards
  Publicaciones:
* POST accounts/:account&#95;id/tweets - Para agregar cards a las Publicaciones, usa el parámetro card&#95;uri.
  Publicaciones programadas:
* POST accounts/:account&#95;id/scheduled&#95;tweets

Para obtener información adicional sobre las cards, consulta la página de Cards. La página de Promoted Video proporciona detalles sobre cómo asociar videos con cards o Publicaciones.

### Cards

La Ads API admite varios tipos de tarjetas que se pueden usar en Publicaciones, que luego pueden promocionarse en campañas. **Nota**: una vez publicada, la información de la tarjeta es visible públicamente. Esto puede incluir información sobre el usuario que es propietario de la tarjeta.

<div id="image">
  #### Imagen
</div>

Las siguientes especificaciones de imagen se aplican a los activos usados en las [Cards](/es/x-ads-api/creatives#cards-2). Las imágenes deben pesar 3 MB o menos y tener un ancho mínimo de 800 px. Además, admitimos las siguientes relaciones de aspecto (ancho:alto).

* Sitio web: 1:1 y 1.91:1
* Descarga de aplicación con imagen: 1:1 y 1.91:1
* Encuesta: 1.91:1
* Conversación con imagen: 1.91:1
* Mensaje directo con imagen: 1.91:1

Admitimos los siguientes formatos de imagen: .bmp, .jpeg y .png.

<div id="video">
  #### Video
</div>

Las siguientes especificaciones de video se aplican a los recursos utilizados en [Cards](/es/x-ads-api/creatives#cards-2). Se admiten las siguientes relaciones de aspecto (ancho:alto).

* Video Website: 16:9 y 1:1
* Video App Download: 16:9 y 1:1
* Poll: 16:9
* Video Conversation: 16:9
* Video Direct Message: 16:9

<div id="promoted-video">
  ### Video promocionado
</div>

Este documento ofrece una breve descripción general del proceso para subir y promocionar video mediante la Ads API.

La Ads API admite video promocionado en [Publicaciones](/es/x-ads-api/creatives#tweets-2) y en las siguientes tarjetas:

* [Video Website](https://devcommunity.x.com/t/ads-api-version-11/168814)
* [Video App Download](https://devcommunity.x.com/t/ads-api-version-11/168814)
* [Video Conversation](/es/x-ads-api/creatives#video-conversation-cards)

Primero, sube el video usando el endpoint [POST media/upload (chunked)](https://developer.x.com/en/docs/media/upload-media/api-reference/post-media-upload-init.html). Con el `media_id`, asocia el video a una cuenta de anuncios usando el endpoint POST accounts/:account&#95;id/videos. El `id` del video, a veces denominado `media_key`, se usará en solicitudes posteriores. Es una cadena que comienza con un entero (`int`), seguida de un guion bajo y que termina con un valor de tipo `long`. Por ejemplo: `13_875943225764098048`.

<div id="promoted-video-in-tweets">
  #### Video promocionado en Publicaciones
</div>

Para crear una Publicación, usa el endpoint [POST accounts/:account&#95;id/tweet](/es/x-ads-api/creatives#get-accounts-account-id-tweets) junto con el `id` del video. En este paso, también puedes proporcionar un título del video, una descripción y un llamado a la acción (CTA). Estos valores son visibles para los usuarios.

<div id="promoted-video-in-cards">
  #### Video promocionado en Cards
</div>

Las cards de Video App Download y Video Conversation admiten agregar imágenes de portada. Sube una imagen para usarla en estas cards mediante el endpoint [POST media/upload](https://developer.x.com/en/docs/media/upload-media/api-reference/post-media-upload.html).

Crea la card usando uno de los siguientes endpoints:

* [POST accounts/:account&#95;id/cards/video&#95;website](https://devcommunity.x.com/t/ads-api-version-11/168814)
* [POST accounts/:account&#95;id/cards/video&#95;app&#95;download](https://devcommunity.x.com/t/ads-api-version-11/168814)
* [POST accounts/:account&#95;id/cards/video&#95;conversation](https://devcommunity.x.com/t/ads-api-version-11/168814)

usando el `id` del video y, opcionalmente, el `media_id` de la imagen (para la imagen de portada).

Por último, crea la Publicación usando el endpoint POST accounts/:account&#95;id/tweet. Las cards se adjuntan a las Publicaciones mediante el parámetro `card_uri`.

<div id="general-information">
  #### Información general
</div>

Para obtener instrucciones detalladas sobre la carga de videos a través de la API, consulta la [Guía de carga de video](https://developer.x.com/en/docs/media/upload-media/uploading-media.html).

Los videos también se pueden promocionar como recursos de pre-roll. Consulta la [Guía del objetivo de vistas de video pre-roll](/es/x-ads-api/campaign-management#video-views-preroll-objective) para una explicación detallada.

* (A partir del 22-10-2015) Al cargar videos que se utilizarán en contenido promocionado, el parámetro `media_category` debe establecerse con el valor `amplify_video` para todas las solicitudes con el comando `INIT` al endpoint [POST media/upload (chunked)](https://developer.x.com/en/docs/media/upload-media/api-reference/post-media-upload-init.html). Usar este parámetro garantiza que el video se procese de forma asíncrona y se prepare para su uso en contenido promocionado. El comando `STATUS` se puede usar para comprobar la finalización del procesamiento asíncrono después de cargar el video.
* La duración máxima actualmente permitida para videos promocionados es de 10 minutos, con un tamaño de archivo de 500 MB o menos.
* El video cargado debe estar en formato mp4 o mov.
* El video cargado generalmente se procesa rápidamente, pero los tiempos de procesamiento pueden variar según la duración del video y el tamaño del archivo.
* Las imágenes de póster cargadas deben estar en formato png o jpg. No hay requisitos de proporción de aspecto ni de tamaño, pero la imagen de póster se ajustará para adaptarse al reproductor de video.

<div id="scheduled-tweets">
  ### Publicaciones programadas
</div>

<div id="introduction">
  #### Introducción
</div>

Las Publicaciones programadas permiten que un anunciante o usuario cree una Publicación que se pueda programar para activarse en una fecha posterior. Además de poder crear y administrar estas Publicaciones, la API permite asociarlas con un elemento de línea para que se promocionen una vez que la Publicación esté activa. Esto permite a los anunciantes preparar Publicaciones nativas y planificar los recursos creativos de sus campañas con anticipación a cualquier iniciativa clave. Por ejemplo, preparar una creatividad de Publicación para que se publique inmediatamente tras el anuncio de un nuevo producto.

El conjunto completo de funcionalidades que proporcionan los endpoints de la API de Publicaciones programadas se enumera a continuación:

* Crear, modificar y ver nuevas Publicaciones programadas
* Asociar una Publicación programada con un elemento de línea
* Consultar y administrar Publicaciones programadas existentes
* Una vez que una Publicación programada esté activa, recuperar el `id` de la Publicación activa

<div id="api-endpoints">
  #### Endpoints de la API
</div>

A continuación se muestra el conjunto completo de endpoints relacionados con la funcionalidad anterior:

<div id="scheduled-tweet-management">
  #### Administración de publicaciones programadas
</div>

* [GET accounts/:account&#95;id/scheduled&#95;tweets](/es/x-ads-api/creatives#get-accounts-account-id-scheduled-tweets) (obtener una lista de todas las publicaciones programadas)
* [GET accounts/:account&#95;id/scheduled&#95;tweets/:scheduled&#95;tweet&#95;id](/es/x-ads-api/creatives#get-accounts-account-id-scheduled-tweets) (buscar una publicación programada específica usando su `id`)
* [POST accounts/:account&#95;id/scheduled&#95;tweets](/es/x-ads-api/creatives#post-accounts-account-id-scheduled-tweets) (crear una nueva publicación programada)
* [PUT accounts/:account&#95;id/scheduled&#95;tweets/:scheduled&#95;tweet&#95;id](/es/x-ads-api/creatives#example-request-39) (modificar una publicación programada existente)
* [DELETE accounts/:account&#95;id/scheduled&#95;tweets/:scheduled&#95;tweet&#95;id](/es/x-ads-api/creatives#example-request-40) (eliminar una publicación programada usando su `id`)
* [GET accounts/:account&#95;id/scheduled&#95;tweets/preview/:scheduled&#95;tweet&#95;id](/es/x-ads-api/creatives#scheduled-tweets-2) (mostrar una vista previa de una publicación programada existente)

<div id="scheduled-promoted-tweets">
  #### Publicaciones promocionadas programadas
</div>

* [GET accounts/:account&#95;id/scheduled&#95;promoted&#95;tweets](/es/x-ads-api/campaign-management#get-accounts-account-id-scheduled-promoted-tweets) (obtener una lista de todas las publicaciones promocionadas programadas)
* [GET accounts/:account&#95;id/scheduled&#95;promoted&#95;tweets/:scheduled&#95;promoted&#95;tweet&#95;id](/es/x-ads-api/campaign-management#get-accounts-account-id-scheduled-promoted-tweets-scheduled-promoted-tweet-id) (consultar una publicación promocionada programada mediante su `id`)
* [POST accounts/:account&#95;id/scheduled&#95;promoted&#95;tweets](/es/x-ads-api/campaign-management#post-accounts-account-id-scheduled-promoted-tweets) (crear una nueva publicación promocionada programada)
* [DELETE accounts/:account&#95;id/scheduled&#95;promoted&#95;tweets/:scheduled&#95;promoted&#95;tweet&#95;id](/es/x-ads-api/campaign-management#delete-accounts-account-id-scheduled-promoted-tweets-scheduled-promoted-tweet-id) (eliminar una publicación promocionada programada existente mediante su `id`)

<div id="scheduled-tweet-view">
  #### Vista de Publicación programada
</div>

* [GET accounts/:account&#95;id/scheduled&#95;tweets/preview/:scheduled&#95;tweet&#95;id](/es/x-ads-api/creatives#scheduled-tweets-2) (ver una Publicación programada existente)

Debido a que las Publicaciones programadas son entidades independientes de las Publicaciones “en vivo”, se aplican dos conjuntos diferentes de validaciones a cualquier operación de creación o edición de estas Publicaciones. El primer conjunto de reglas de validación se aplica durante el paso de creación de la Publicación programada, específicamente:

<div id="scheduled-tweet-create">
  #### Creación de Publicación programada:
</div>

* Verifica que el usuario autenticado tenga acceso para crear Publicaciones orgánicas para un @handle determinado. Los privilegios para crear Publicaciones solo promocionadas requieren que el usuario autenticado sea un usuario de la cuenta con [permisos del editor de Publicaciones](https://developer.x.com/content/developer-twitter/en/docs/ads/campaign-management/api-reference/authenticated-user-access#get-accounts-account-id-authenticated-user-access)
* Verifica que no haya más de 30 Publicaciones que estén programadas para crearse dentro de una ventana de 15 minutos alrededor de la hora `scheduled_at`. Un mensaje de error SCHEDULED&#95;TWEET&#95;LIMIT&#95;EXCEEDED indica que se han programado demasiadas Publicaciones dentro del mismo intervalo futuro de 15 minutos. Los anunciantes tendrán que eliminar una Publicación programada existente o mover la hora `scheduled_at` a un momento anterior o posterior.

<div id="scheduled-tweet-goes-live">
  #### La publicación programada se publica:
</div>

* Estas reglas de validación se ejecutan en el momento indicado por scheduled&#95;at y son idénticas a las que se aplican en la creación de una publicación normal en la API. Por ejemplo, una publicación programada no se publicará y el scheduled&#95;status se establecerá en FAILED si la publicación programada contiene tanto una imagen como un GIF.

<div id="workflow">
  #### Flujo de trabajo
</div>

**Crear una nueva Publicación programada**

Se puede crear una nueva Publicación programada usando el endpoint [POST accounts/:account&#95;id/scheduled&#95;tweets](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/scheduled-tweets#post-accounts-account-id-scheduled-tweets). Este endpoint tiene los siguientes parámetros obligatorios: la hora `scheduled_at`, junto con el `text` de la Publicación si no se incluyen entidades de contenido multimedia en la Publicación. Además, este endpoint proporciona algunas opciones adicionales que te permiten crear una Publicación programada en nombre de otro @handle mediante el parámetro `as_user_id`, junto con la posibilidad de añadir una tarjeta (`card_uri`) y cualquier contenido multimedia (`media_ids`). Ten en cuenta que una Publicación solo puede contener entidades del mismo tipo, es decir, video, GIF o imagen. El parámetro `nullcast` controla si la Publicación es una Publicación “solo promocionada” o no. Todas las Publicaciones programadas recién creadas son &quot;solo promocionadas&quot; (`nullcast=true`) de forma predeterminada. Si `nullcast=false`, entonces se crea una Publicación programada orgánica.

Una vez que una Publicación programada se crea correctamente, la respuesta contendrá un campo `id`, que se refiere al identificador único de la propia Publicación programada. Además de este campo, también se devuelve otro campo llamado `tweet_id`. Este campo es `null` inicialmente; sin embargo, una vez que la Publicación se publique, este campo se rellena con el ID de la Publicación “en vivo”.

```
twurl -H 'ads-api.x.com' -X POST "/6/accounts/:account_id/scheduled_tweets" -d 'scheduled_at=2017-12-24T23:59:00Z&text=Happy Holidays!'
```

Esta acción creará la siguiente publicación programada:

```json
{
  "request": {
    "params": {
      "text": "Happy Holidays!",
      "scheduled_at": "2017-12-24T23:59:00Z"
    }
  },
  "data": {
    "completed_at": null,
    "id_str": "917507899668099072",
    "text": "Happy Holidays!",
    "user_id": "3271358660",
    "scheduled_status": "SCHEDULED",
    "id": 917507899668099100,
    "nullcast": true,
    "created_at": "2017-10-09T21:51:44Z",
    "scheduled_at": "2017-12-24T23:59:00Z",
    "card_uri": null,
    "updated_at": "2017-10-09T21:51:44Z",
    "tweet_id": null,
    "media_keys": []
  }
}
```

Una vez que esta Publicación programada se publique, el campo `tweet_id` se rellenará con el ID de la Publicación &quot;en vivo&quot;.

**Asociar una Publicación programada con un line item**

Si bien las Publicaciones programadas se pueden usar para crear Publicaciones orgánicas, también permitimos que los socios creen una Publicación «solo promocionada» (`nullcast=true`), cualquiera de las cuales se puede asociar con un line item. Para facilitar esto, también proporcionamos el endpoint [POST accounts/:account&#95;id/scheduled&#95;promoted&#95;tweets](https://developer.x.com/content/developer-twitter/en/docs/ads/campaign-management/api-reference/scheduled-promoted-tweets#post-accounts-account-id-scheduled-promoted-tweets). Este endpoint solo permite que una única Publicación programada promocionada se asocie con un line item en una sola llamada a la API. Para asociar varias Publicaciones programadas al mismo line item, es necesario realizar múltiples llamadas a la API.

Ten en cuenta que no es posible modificar una Publicación programada promocionada existente.

```
twurl -H 'ads-api.x.com' -X POST "/6/accounts/:account_id/scheduled_promoted_tweets" -d 'line_item_id=a44qc&scheduled_tweet_id=917507899668099072'
```

```json
{
  "data": {
    "line_item_id": "a44qc",
    "id": "26576",
    "created_at": "2017-10-09T22:24:16Z",
    "updated_at": "2017-10-09T22:24:16Z",
    "scheduled_tweet_id": "917507899668099072",
    "tweet_id": null,
    "deleted": false
  },
  "request": {
    "params": {
      "line_item_id": "a44qc",
      "scheduled_tweet_id": 917507899668099100,
      "account_id": "aaaaa"
    }
  }
}
```

Este endpoint solo crea una asociación entre una Publicación programada determinada y un elemento de línea. Una vez que las fechas de vuelo de la campaña/elemento de línea estén vigentes, el elemento de línea comenzará automáticamente a difundir la Publicación “en vivo” correspondiente. Aunque durante este paso validamos que la Publicación programada esté en el estado `SCHEDULED` y que la Publicación programada indicada sea válida para el objetivo dado, no se ejecutan otras validaciones. Cualquier regla de validación restante que se aplique al elemento de línea y a la Publicación programada se ejecuta cuando la Publicación pasa a estar “en vivo”.

Para garantizar que no haya problemas con la entrega de la campaña, se recomienda que la Publicación programada se haya configurado en `scheduled_at` para un momento anterior a las fechas de vuelo de la campaña/elemento de línea.

Por ejemplo, supongamos que la Publicación programada está configurada para pasar a estar en vivo después de la fecha de inicio de la campaña (y que solo haya una única Publicación asociada con un único elemento de línea), entonces la campaña estará `ACTIVE`; sin embargo, dado que la Publicación programada aún no está en vivo, no habrá creativos disponibles para la entrega.

**Gestión de publicaciones programadas**

Los conjuntos restantes de endpoints permiten a los consumidores de la API gestionar todas sus Publicaciones programadas y Publicaciones promocionadas programadas. Estas API se pueden usar para devolver una lista de todas las Publicaciones programadas, opcionalmente filtradas por un estado determinado, así como para buscar una Publicación programada concreta por su `id`.

<div id="what-happens-when-a-scheduled-tweet-goes-live">
  #### ¿Qué ocurre cuando una Publicación programada se publica?
</div>

Una vez que una Publicación programada está a punto de publicarse, o dicho de otro modo, en el momento indicado por `scheduled_at`, se realizan las siguientes actualizaciones:

* Se crea la Publicación “en vivo”; sin embargo, esto puede tener una latencia de hasta 1 segundo
* El `tweet_id` se añade a las siguientes entidades:
* Publicación programada
* Publicación programada promocionada
* Se crea una nueva entidad de Publicación promocionada

<div id="best-practices">
  #### Mejores prácticas
</div>

Se recomiendan las siguientes mejores prácticas al crear o promocionar Publicaciones programadas:

* Asegúrate de que la Publicación sea válida al crear la Publicación programada (por ejemplo, una Publicación solo puede tener una imagen, un video o un GIF, y no una combinación de los tres)
* Asegúrate de que las fechas de la campaña (es decir, `start_time` y `end_time`) se alineen con la hora `scheduled_at` de la Publicación programada
* Las Publicaciones programadas no deben programarse para más de un año en el futuro (365 días)
* Actualmente no se admite la vista previa de Publicaciones programadas (es decir, la posibilidad de previsualizar las Publicaciones programadas antes de crearlas)

<div id="media-library">
  ### Biblioteca multimedia
</div>

<div id="introduction">
  #### Introducción
</div>

Los endpoints de Media Library permiten gestionar imágenes, GIFs y vídeos para cuentas de X Ads. Los recursos multimedia de la biblioteca se pueden usar en Publicaciones y para crear tarjetas. También se pueden reutilizar en múltiples [creativos](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/overview), lo que elimina la necesidad de subir el mismo recurso varias veces.

<div id="api-endpoints">
  #### Endpoints de la API
</div>

* [POST media/upload](https://developer.x.com/content/developer-twitter/en/docs/media/upload-media/api-reference/post-media-upload#post-media-upload) (cargar contenido multimedia) o [POST media/upload (chunked)](https://developer.x.com/content/developer-twitter/en/docs/media/upload-media/api-reference/post-media-upload-init#post-media-upload-init) (cargar contenido multimedia por partes)

<div id="request-parameters">
  #### Parámetros de la solicitud
</div>

Todas las solicitudes POST de Media Library requieren un identificador de contenido multimedia. Este valor se devuelve durante la etapa de carga. Cuando se usa el media&#95;id, como en el ejemplo anterior, también se debe especificar una media&#95;category. Hay cuatro posibles valores de categoría: AMPLIFY&#95;VIDEO, TWEET&#95;GIF, TWEET&#95;IMAGE y TWEET&#95;VIDEO.

Opcionalmente, se pueden establecer valores de name y file&#95;name para los objetos en la Media Library. Estos atributos ayudan a los usuarios a distinguir entre variantes de contenido multimedia en la biblioteca.

En el caso de los vídeos, también es posible establecer un título y una descripción. Estos valores están pensados para pasarse como los parámetros de solicitud video&#95;title y video&#95;description con el endpoint [POST accounts/:account&#95;id/tweet](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/tweets#post-accounts-account-id-tweet). En la Publicación, este texto aparece debajo del vídeo.

<div id="attributes">
  #### Atributos
</div>

La Media Library introduce formalmente el concepto de `media_key`. Este es el identificador único de los objetos de la biblioteca. Las media keys son valores de tipo cadena con el siguiente formato: 13&#95;875943225764098048. Son totalmente compatibles con todos nuestros endpoints de tarjetas.

Además, la respuesta de Media Library incluye el `media_id`, representado como una cadena. Esto se incluye para los recursos que actualmente no aceptan una media key: [Tweets](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/tweets#post-accounts-account-id-tweet)*, [Tweet preview](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/tweets#get-accounts-account-id-tweet-preview)* y [Scheduled Tweets](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/scheduled-tweets#post-accounts-account-id-scheduled-tweets). Estamos trabajando para admitir media keys en todas partes.

El atributo `aspect_ratio` se devuelve para GIF y vídeos. Se puede usar para filtrar contenido multimedia para su uso en tarjetas que solo aceptan determinadas relaciones de aspecto.

*Estos endpoints admiten el parámetro `video_id`, que es una media key.

<div id="identifying-cards">
  ### Identificación de Cards
</div>

<div id="introduction">
  #### Introducción
</div>

Las Cards son formatos de anuncio personalizables que usan contenido multimedia y que se pueden asociar con un sitio web, una aplicación o con llamadas a la acción para fomentar determinados tipos de interacción de los usuarios, como iniciar un Mensaje Directo. Se pueden adjuntar a publicaciones, publicaciones programadas o borradores de publicaciones.

Las Cards se pueden hacer referencia en los objetos de publicación de dos formas: mediante la `card_uri` de la Card o mediante su `preview_url`. A continuación se muestran valores de ejemplo para cada una.

| card&#95;uri | preview&#95;url |
| :--- | :--- |
| card://1043282691834048513 | https://cards.x.com/cards/18ce54d4x5t/68w3s |

**Nota**: A partir de la versión 3 de Ads API, solo se genera y se devuelve la `card_uri` en la respuesta de cards para las Cards recién creadas.

**Nota**: A partir de la versión 5 de Ads API, el `preview_url` ya no se devuelve en la respuesta de cards.

El tipo de referencia en la respuesta del objeto de la publicación dependerá de la forma en que se creó la publicación. En otras palabras, si la publicación se creó usando el parámetro de solicitud `card_uri`, el valor de la `card_uri` de la Card aparecerá en la respuesta. Por otro lado, si el `preview_url` se incluyó como parte del texto de la publicación, la URL de vista previa aparecerá en la respuesta.

<div id="identifying-media">
  ### Identificar contenido multimedia
</div>

<div id="introduction">
  #### Introducción
</div>

El contenido multimedia (imágenes, GIFs y vídeos) se puede agregar a las Publicaciones y a las tarjetas. Además, los vídeos se pueden usar como recursos de pre-roll y las imágenes se pueden promocionar en la [X Audience Platform](https://developer.x.com/en/docs/ads/measurement/overview/twitter-audience-platform). Esta sección describe cómo encontrar referencias a contenido multimedia en estas entidades.

Hay dos tipos de identificadores de contenido multimedia: IDs y claves. A continuación se muestran valores de ejemplo para cada uno.

| **ID de contenido multimedia** | **Clave de contenido multimedia** |
| :--- | :--- |
| 1029825579531807971 | 13&#95;1029825579531807971 |

La clave de contenido multimedia es el ID más un prefijo numérico y un guion bajo.

<div id="images">
  #### Imágenes
</div>

La siguiente tabla muestra los tipos de identificadores disponibles actualmente en la respuesta de cada recurso relacionado con imágenes, así como los nombres de atributos correspondientes.

| **Recurso** | **Identificador** | **Atributo(s)** |
| :--- | :--- | :--- |
| Tarjetas de imagen | Ninguno |  |
| Publicación | Ambos | `entities["media"]["id_str"]`  `entities["media"]["media_key"]` |
| Publicación programada | Ambos | `media_ids` y `media_keys` |
| Borrador de Publicación | Ambos | `media_ids` y `media_keys` |
| Contenido multimedia de la cuenta | Ninguno |  |
| Media Library | Ambos | `media_id` y `media_key` |

Las tarjetas de imagen y las imágenes de Contenido multimedia de la cuenta no incluyen ninguna referencia a un identificador de contenido multimedia. Las Publicaciones solo incluyen ID de contenido multimedia. Las Publicaciones programadas y los borradores de Publicaciones incluyen tanto el ID de contenido multimedia como la media key. Media Library también devuelve ambos.

Para las Publicaciones, los campos id e id&#95;str en el objeto dentro del arreglo entities[&quot;media&quot;] corresponden al ID de contenido multimedia. En los casos en que una Publicación incluye varias imágenes, las referencias a cada entidad de contenido multimedia solo se pueden encontrar en extended&#95;entities[&quot;media&quot;].

Además de las referencias a identificadores, a menudo es importante tener acceso a la URL de la imagen.

| **Recurso** | **Atributo(s)** | **Formato** |
| :--- | :--- | :--- |
| Tarjetas de imagen | `image` | .jpg |
| Publicación* | `entities["media"][0]["media_url"]` o `extended_entities["media"][i]["media_url"]` | .jpg |
| Publicación programada | Ninguno |  |
| Borrador de Publicación | Ninguno |  |
| Contenido multimedia de la cuenta | `media_url` | .jpg |
| Media Library | `media_url` | .jpg |

* La ubicación de esta URL depende de si la Publicación contiene una sola imagen o varias imágenes.

Todas las tarjetas de imagen incluyen un atributo de respuesta de imagen que contiene la URL de imagen de X. (Para las tarjetas de descarga de aplicaciones con imagen, el nombre es wide&#95;app&#95;image).

Para las Publicaciones, la ubicación de la URL del contenido multimedia depende tanto del tipo de contenido multimedia como del endpoint que se esté utilizando. Para las Publicaciones con una sola imagen, la URL se puede encontrar en entities[&quot;media&quot;][0][&quot;media&#95;url&quot;]. Esto es válido tanto para la Ads API como para la Standard API. Sin embargo, cuando las Publicaciones contienen varias imágenes, las URL solo se pueden encontrar en extended&#95;entities[&quot;media&quot;][i][&quot;media&#95;url&quot;]. Esto solo está disponible en la Standard API.

#### Media Library

A veces es necesario recuperar información adicional sobre un recurso de contenido multimedia. Un caso de uso, en el caso de las tarjetas de video, es recuperar la URL mp4 en lugar de la vmap. Esto se puede hacer desde la Media Library. Para obtener detalles sobre la información disponible, consulta nuestra [Media Library Guide](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/guides/media-library). La mayoría de los recursos que pertenecen al usuario promocionable FULL de la cuenta de anuncios se pueden encontrar en la biblioteca. Sin embargo, hay algunas excepciones.

**Obtención de contenido multimedia**

Como se indicó anteriormente, las tarjetas de imagen no contienen referencias ni a media IDs ni a media keys. Como resultado, no es posible obtener sus recursos a través de la Media Library. Esto también es cierto para las imágenes de [Account Media](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/account-media#account-media).

Las tarjetas de video requieren que el recurso de video forme parte de la Media Library (o del recurso Videos antes de eso) antes de crearlas. Como resultado, estos recursos siempre se podrán recuperar en la Media Library. Esto también es cierto para los recursos PREROLL de Account Media.

Por último, se garantiza que el contenido multimedia en Publicaciones siempre estará en la Media Library.

La siguiente tabla resume qué recursos se pueden recuperar en la Media Library, teniendo en cuenta si la respuesta del recurso incluye un identificador que se pueda usar en la búsqueda.

| **Resource** | **In the Media Library** |
| :--- | :--- |
| Image cards | No |
| Video cards | Yes* |
| Tweets (any media)** | Yes |
| Scheduled Tweets | Yes |
| Draft Tweets | Yes |
| Account media images | No |
| Account media videos (`PREROLL`) | Yes |

* Para tarjetas donde `video_content_id` es una media key. Cuando el valor es un media ID, el recurso aún existe en la Media Library, pero recuperarlo implica agregarle un prefijo numérico y un guion bajo.
** Las Publicaciones solo devuelven media IDs. Aunque se garantiza que el recurso existe en la Media Library, obtenerlo implica agregarle un prefijo numérico y un guion bajo.

**Interacciones con Account Media**

Hay dos casos en los que los recursos de contenido multimedia agregados a la biblioteca se añaden automáticamente al recurso Account Media.

* Cuando se agrega un recurso AMPLIFY&#95;VIDEO a la Media Library, se agrega automáticamente como un recurso de Account Media con un tipo de creativo PREROLL.
* Cuando se agregan a la Media Library imágenes que tienen dimensiones específicas (consulta &quot;Creative Types&quot; en nuestra [página de enumeraciones](https://developer.x.com/content/developer-twitter/en/docs/ads/general/overview/enums)), se agregan automáticamente como recursos de Account Media. El tipo de creativo (por ejemplo, INTERSTITIAL) depende de las dimensiones de la imagen.

<div id="tweets">
  ### Publicaciones
</div>

<div id="introduction">
  #### Introducción
</div>

La Ads API de X admite tres tipos de publicaciones: publicadas, programadas y en borrador.

<div id="nullcasted-tweets">
  #### Publicaciones nullcast
</div>

Las Publicaciones pueden ser nullcast (también conocidas como &quot;solo promocionadas&quot;) u orgánicas. Las Publicaciones nullcast, una vez publicadas, no aparecen en la cronología pública del usuario, aunque sí son públicas. Las Publicaciones orgánicas, en cambio, se muestran a los seguidores del usuario y sí aparecen en la cronología pública del usuario.

**Creación de Publicaciones**

Cada uno de los tres endpoints de creación de Publicaciones admite un parámetro booleano nullcast que ofrece al usuario de la API la opción de crear Publicaciones nullcast u orgánicas. Las Publicaciones nullcast pueden ser creadas por el usuario o por cualquier persona que tenga permiso para crear Publicaciones en nombre del usuario. Las Publicaciones orgánicas solo pueden ser creadas por el [usuario totalmente promocionable](https://developer.x.com/content/developer-twitter/en/docs/tutorials/ads-api-hierarchy-terminology#promotable-users).

**Actualización de Publicaciones**

Es posible actualizar la propiedad nullcast para Publicaciones programadas y borradores. En el caso de las Publicaciones programadas, se pueden hacer modificaciones hasta la hora `scheduled_at` de la Publicación. Los borradores de Publicaciones pueden editarse indefinidamente. Sin embargo, una vez publicadas, no es posible cambiar una Publicación de nullcast a orgánica o viceversa.

<div id="promoting-tweets">
  #### Promocionar Publicaciones
</div>

Solo las Publicaciones publicadas y programadas pueden promocionarse. Estas pueden ser de tipo nullcast u orgánicas; no hay ninguna restricción al respecto. Un anunciante puede promocionar sus propias Publicaciones o las Publicaciones de otro usuario siempre que haya obtenido permiso para hacerlo. (Consulta [Promoting another user&#39;s Tweets](https://developer.x.com/content/developer-twitter/en/docs/tutorials/promoting-another-users-content) para obtener más información).

Pueden promocionarse varias Publicaciones en una sola campaña. De manera similar, una sola Publicación puede promocionarse en una o más campañas.

Para promocionar Publicaciones publicadas, usa el endpoint [POST accounts/:account&#95;id/promoted&#95;tweets](https://developer.x.com/content/developer-twitter/en/docs/ads/campaign-management/api-reference/promoted-tweets#post-accounts-account-id-promoted-tweets). Esto asocia Publicaciones publicadas con un line item. Para promocionar Publicaciones programadas, usa el endpoint [POST accounts/:account&#95;id/scheduled&#95;promoted&#95;tweets](https://developer.x.com/content/developer-twitter/en/docs/ads/campaign-management/api-reference/scheduled-promoted-tweets#post-accounts-account-id-scheduled-promoted-tweets).

<div id="tweet-ids">

</div>

Las IDs de las Publicaciones publicadas, programadas y en borrador son [numéricas](https://developer.x.com/content/developer-twitter/en/docs/basics/twitter-ids): son enteros sin signo de 64 bits. Por ejemplo, la ID de la siguiente Publicación publicada es 1166476031668015104.

<div id="carousels">
  ### Carruseles
</div>

<div id="introduction">
  #### Introducción
</div>

La X Ads API permite crear y obtener carruseles de video e imagen. El carrusel es un tipo de tarjeta que puede contener entre 2 y 6 elementos de contenido multimedia. La tarjeta de carrusel puede dirigir a un usuario a un sitio web o incentivar la instalación de una aplicación móvil. Para obtener más información sobre los carruseles, sus beneficios, prácticas recomendadas y preguntas frecuentes, consulta nuestra página [Carousel Ads on X](https://business.x.com/en/advertising/carousels.html).

Un carrusel, como cualquier otro tipo de tarjeta, se puede usar en Publicaciones y luego esas Publicaciones se pueden promocionar. El flujo de trabajo es el mismo que ya conoces:

1. Cargar contenido multimedia
2. Crear la tarjeta
3. Crear la Publicación
4. Promocionar la Publicación

La única diferencia está en cómo se crea la tarjeta. Mientras que otras solicitudes de creación de tarjetas aceptan parámetros de consulta, las solicitudes de creación de tarjetas de carrusel solo aceptan cuerpos POST en formato JSON.

<div id="endpoints">
  #### Endpoints
</div>

La Ads API permite crear y obtener carruseles.

Para crear un carrusel (de cualquier tipo), usa el endpoint [POST accounts/:account&#95;id/cards](https://developer.x.com/en/docs/twitter-ads-api/creatives/api-reference/cards#post-accounts-account-id-cards). Para obtener carruseles, usa el endpoint [GET accounts/:account&#95;id/cards](https://developer.x.com/en/docs/twitter-ads-api/creatives/api-reference/cards#get-accounts-account-id-cards).

<div id="json-post-body">
  #### Cuerpo de la solicitud POST en JSON
</div>

Los carruseles se crean mediante dos componentes. El primero especifica los recursos multimedia que se utilizarán. El segundo especifica información sobre el sitio web o la aplicación.

Específicamente, una tarjeta de carrusel se crea mediante los siguientes componentes, en este orden:

* Un componente `SWIPEABLE_MEDIA`, que acepta un array de claves de contenido multimedia
* *Uno* de los siguientes:
* Un componente `DETAILS` para especificar la información del sitio web
* Un componente `BUTTON` para especificar la información de la aplicación

El componente `SWIPEABLE_MEDIA` debe incluir un array `media_keys` donde se pueden especificar entre 2 y 6 imágenes o vídeos. El orden en el que se pasan las claves de contenido multimedia determina el orden en el que se mostrarán.

```
{
  "type": "SWIPEABLE_MEDIA",
  "media_keys": [
    "13_1089771253848666112",
    "13_1191948012077092867"
  ]
}
```

Como recordatorio, puedes obtener claves de contenido multimedia realizando una solicitud al endpoint [GET accounts/:account&#95;id/media&#95;library](https://developer.x.com/en/docs/twitter-ads-api/creatives/api-reference/media-library#get-accounts-account-id-media-library).

La composición del segundo objeto de componente depende de si deseas dirigir a un usuario a un sitio web o invitarle a instalar una aplicación. La siguiente tabla resume las dos opciones. (**Nota**: todas las claves enumeradas son obligatorias).

|                                   | **Sitio web**                                                      | **Aplicación**                                    |
| :-------------------------------- | :----------------------------------------------------------------- | :------------------------------------------------ |
| Especificar el tipo de componente | `"type": "DETAILS"`                                                | `"type": "BUTTON"`                                |
| Título/Etiqueta                   | `"title": "X"`                                                     | `"label": { "type": "ENUM", "value": "INSTALL" }` |
| Destino                           | `"destination": { "type": "WEBSITE", "url": "https://www.x.com" }` | `"destination": { "type": "APP", ... }`           |

Con todo esto, a continuación se muestra un ejemplo de cuerpo de solicitud POST en JSON para un carrusel de sitio web.

```
{
  "name": "website carousel",
  "components": [
    {
      "type": "SWIPEABLE_MEDIA",
      "media_keys": [
        "13_1089771253848666112",
        "13_1191948012077092867"
      ]
    },
    {
      "type": "DETAILS",
      "title": "X",
      "destination": {
        "type": "WEBSITE",
        "url": "https://www.x.com"
      }
    }
  ]
}
```

Los objetos de destino de aplicación dentro de componentes `BUTTON` requieren un código de país y al menos un identificador de aplicación. Opcionalmente aceptan deep links. Para una descripción de estos campos, consulta la [documentación de referencia](https://developer.x.com/en/docs/twitter-ads-api/creatives/api-reference/cards#app-destination).

Teniendo esto en cuenta, a continuación se muestra un ejemplo de cuerpo de la solicitud POST en JSON para un carrusel de aplicaciones.

```
{
  "name": "app carousel",
  "components": [
    {
      "type": "SWIPEABLE_MEDIA",
      "media_keys": [
        "13_1089771253848666112",
        "13_1191948012077092867"
      ]
    },
    {
      "type": "BUTTON",
      "label": {
        "type": "ENUM",
        "value": "INSTALL"
      },
      "destination": {
        "type": "APP",
        "country_code": "US",
        "googleplay_app_id": "com.twitter.android",
        "iphone_app_id": "333903271"
      }
    }
  ]
}
```

<div id="creative-metadata-tagging">
  ### etiquetado-de-metadatos-creativos
</div>

<div id="introduction">
  #### Introducción
</div>

Esta guía está dirigida a socios creativos, agencias y desarrolladores creativos para que etiqueten los recursos utilizados en campañas de X y así comprender mejor el valor y el rendimiento de cada recurso individual.

**Nota:** Los recursos de contenido multimedia solo deben ser etiquetados por el socio o desarrollador que los creó. Si el usuario de un recurso de contenido multimedia no lo creó, no se debe implementar el etiquetado de metadatos.

El etiquetado de metadatos creativos proporciona atribución de imágenes y videos creados por socios creativos, independientemente de dónde se cargue el recurso en X o de cuál sea la entidad que esté cargando el recurso. Para crear la conexión entre el recurso creativo y el socio creativo, se utiliza el estándar [XMP](https://exiftool.org/TagNames/XMP.html).

<div id="tagging-creative-assets">
  #### Etiquetado de recursos creativos
</div>

La siguiente tabla muestra los tipos de identificadores actualmente disponibles en la respuesta de cada recurso relacionado con imágenes, así como el(los) nombre(s) de atributo correspondiente(s). Se necesita una herramienta de etiquetado para etiquetar recursos creativos. Se recomienda [ExifTool](https://exiftool.org/), una biblioteca de Perl independiente de la plataforma junto con una [aplicación de línea de comandos](https://exiftool.org/exiftool_pod.html) para leer, escribir y editar metainformación. Consulta todos los [tipos de archivo](https://exiftool.org/#supported) compatibles.

Sigue las [instrucciones para instalar ExifTool](https://exiftool.org/install.html) proporcionadas. También hay paquetes de software ofrecidos por [Homebrew](https://formulae.brew.sh/) para simplificar aún más la instalación proporcionando el [comando de instalación de exiftool](https://formulae.brew.sh/formula/exiftool) para macOS y Linux. Confirma que tu herramienta esté instalada correctamente introduciendo exiftool -ver en la línea de comandos para que devuelva el número de versión de la herramienta. Obtén más información sobre los parámetros de comando de ExifTool en la [documentación de ExifTool](https://exiftool.org/).

Los socios creativos pueden asignar etiquetas de metadatos a recursos creativos nuevos o existentes con su X app&#95;id a la etiqueta XMP contributor y a la etiqueta date. Los recursos creativos seguirán las restricciones de tamaño existentes al [Subir contenido multimedia](https://developer.x.com/en/docs/twitter-api/v1/media/upload-media/overview).

<div id="components">
  #### Componentes
</div>

Cada componente debe incluir un campo `type`, que determina el esquema del objeto. La Ads API admite los siguientes tipos de componentes, agrupados en componentes basados en contenido multimedia y en componentes basados en descripción.

* Contenido multimedia:
* `MEDIA`: un solo video o una sola imagen
* `SWIPEABLE_MEDIA`: entre 2 y 6 videos o imágenes
* Descripción:
* `DETAILS`
* `BUTTON`

Cada componente tiene un conjunto de campos obligatorios (además de la clave `type`). Estos se enumeran en la siguiente tabla.

| Tipo de componente `type` | Campo                   | Tipo de valor    |
| :------------------------ | :---------------------- | :--------------- |
| `MEDIA`                   | `media_key`             | string           |
| `SWIPEABLE_MEDIA`         | `media_keys`            | array of strings |
| `DETAILS`                 | `title`   `destination` | string   object  |
| `BUTTON`                  | `label`   `destination` | object   object  |

A continuación se muestra un ejemplo de un componente `BUTTON` en el contexto del array `components` (omitiendo intencionalmente la clave `name`). (Los puntos suspensivos indican los lugares donde sería necesario especificar más información.)

```
{
  "components": [
    {
      "type": "BUTTON",
      "label": {
        ...
      },
      "destination": {
        ...
      }
    }
  ]
}
```

El orden en el que se especifican los objetos de componente define el orden de arriba a abajo en el que se mostrarán. Las Cards deben crearse usando un componente basado en contenido multimedia y un componente `DETAILS` o `BUTTON`. Los componentes basados en descripciones se muestran debajo del contenido multimedia y tienen destinos asociados, ya sean URL o aplicaciones móviles.

**Label**

Las etiquetas definen el texto que se muestra en los botones y, por lo tanto, solo se aplican al componente `BUTTON`. Los objetos de etiqueta tienen dos claves obligatorias: `type` y `value`. El `type` debe establecerse en `ENUM` y el `value` puede ser uno de los siguientes: `BOOK`, `CONNECT`, `INSTALL`, `OPEN`, `ORDER`, `PLAY` o `SHOP`.

Tomando como base el ejemplo anterior, a continuación se muestra el objeto `label` dentro del componente `BUTTON`.

```
{
  "components": [
    {
      "type": "BUTTON",
      "label": {
        "type": "ENUM",
        "value": "INSTALL"
      },
      "destination": {
        ...
      }
    }
  ]
}
```

**Destino**

Los destinos son los lugares a los que los anunciantes desean llevar a los usuarios. Siempre son obligatorios en los componentes `DETAILS` o `BUTTON`. Hay dos tipos de destino: `WEBSITE` o `APP`.

**Nota**: Los destinos de tipo sitio web solo se pueden usar con componentes `DETAILS` y los destinos de tipo aplicación solo se pueden usar con componentes `BUTTON`.
