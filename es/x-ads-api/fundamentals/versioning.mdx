<div id="overview">
  ## Descripción general
</div>

Cada mes realizamos cambios y lanzamos varias funciones nuevas en la X Ads API. Estos cambios casi siempre son compatibles con versiones anteriores; sin embargo, solemos tener un puñado de cambios incompatibles cada año. Hemos recibido comentarios de desarrolladores sobre los desafíos que nuestro rápido ritmo de cambios en la Ads API genera en sus ciclos de desarrollo a la hora de implementar nuevas funciones, gestionar deprecaciones y probar cambios. Queremos mejorar la experiencia de desarrollo al usar nuestra plataforma de Ads, y por eso introdujimos el concepto de versionar nuestros endpoints.

Algunas definiciones de conceptos sobre los que hablamos:

**Versión**: Se refiere al número de versión que se encuentra en la ruta de la URL de cualquier solicitud a la Ads API, por ejemplo: GET /{version_number}/accounts. Este estilo de versionado se conoce como versionado mediante URI.

**Cambios incompatibles**: Los cambios incompatibles son todos aquellos cambios que requieren recursos de desarrollo para mantener la funcionalidad existente. Esto incluye recursos utilizados para investigar qué cambios deben realizarse, determinar las funciones/endpoints que se van a desaprobar y la implementación final de todos estos cambios. Una lista de cambios incompatibles incluye cosas como:

* Eliminar un parámetro de la solicitud/respuesta de la API

* Modificar el nombre de cualquier parámetro o endpoint

* Cambio en la representación de valores (preview&#95;url → card&#95;uri)

* Cambio en el comportamiento de los endpoints (por ejemplo, estadísticas async vs sync)

* Agregar/modificar parámetros opcionales u obligatorios (por ejemplo, hacer que name sea un campo obligatorio en la solicitud)

**Deprecación**: Las versiones o productos en desuso dejarán de estar soportados y se recomienda que los desarrolladores dejen de usar estas API.

**Retiro (sunset)**: Una vez que un producto o una API se retira (sunset), el conjunto correspondiente de endpoints dejará de ser accesible a través de la API.

<div id="versioning-strategy">
  ## Estrategia de versionado
</div>

Los principios fundamentales de la estrategia son:

1. Todos los cambios incompatibles se agruparán en una nueva versión

2. El periodo de deprecación para las versiones existentes cuando se anuncie una nueva versión será de 6 meses

3. En todo momento, la API permitirá solicitudes desde dos versiones simultáneamente; sin embargo, la más antigua de las dos no contará con soporte

4. Con el fin de permitir una adopción más rápida de nuevos productos, estos se lanzarán de forma continua (fuera de la cadencia de versionado)

5. Todas las respuestas de la API contendrán un `x-current-api-version`, que se establecerá en la versión actual de la API, además de un encabezado `x-api-warn` al llamar a cualquier endpoint de API en desuso.

Si hubiera cambios fundamentales en los requisitos del producto que requirieran un cambio incompatible en la API (por ejemplo, la deprecación de la segmentación por múltiples tramos de edad), enviaremos un aviso con 90 días de antelación para anunciar este cambio incompatible y, después de al menos 90 días desde que se emita el aviso, se implementará el cambio incompatible.

<div id="v9">
  ### v9
</div>

Hoy, 3 de marzo de 2021, la versión 9 (v9) de la X Ads API ya está disponible. Esta versión está diseñada para aumentar la paridad de funcionalidades, simplificar la creación de campañas e introducir actualizaciones clave en nuestros endpoints de Cards y Mobile App Promotion.

Como en nuestras versiones anteriores, habrá un período de transición de 6 meses para migrar a v9. El 31 de agosto de 2021, la versión 8 (v8) de la Ads API dejará de estar disponible. Recomendamos a todos los desarrolladores que migren a la versión más reciente de la Ads API lo antes posible para evitar interrupciones del servicio.

</Note>

<div id="v5">
  ### v5
</div>

Hoy, 28 de febrero de 2019, X presenta Ads API v5, con actualizaciones centradas en mejorar la escalabilidad y la eficiencia.

Esta versión incluye un nuevo endpoint para determinar qué entidades estuvieron activas en un período de tiempo dado, estadísticas para Media Creatives (es decir, videos in-stream e imágenes en X Audience Platform), la posibilidad de obtener *múltiples* cards por URI de card y más flexibilidad al recuperar criterios de segmentación y otras entidades. Además, hemos corregido algunos errores y realizado actualizaciones en los nombres de parámetros y atributos de respuesta. Por último, las non-media app cards y el endpoint POST `accounts/:account_id/account_media` se han declarado obsoletos.

Al igual que con las versiones anteriores, habrá un período de transición de 6 meses para migrar a v5. El **2019-08-28**, la versión 4 de Ads API dejará de estar disponible. Recomendamos a todos los socios que migren a la versión más reciente de la API lo antes posible para evitar cualquier interrupción del servicio. *La versión 3 de Ads API ha llegado al final de su vida útil y ya no está disponible.*

#### Nuevo

**Determinar qué entidades estuvieron activas**

El endpoint de [Active Entities](/es/x-ads-api/analytics#active-entities-2) indica si las métricas de analytics para entidades de anuncios han cambiado. Diseñado para usarse junto con los endpoints de analytics, Active Entities funciona especificando un tipo de entidad y un rango de fechas —con un máximo de 90 días— y devuelve una lista de IDs de entidad para las que tu plataforma debería solicitar analytics. En las solicitudes de analytics posteriores no se deben consultar IDs distintos de los que se devuelven.

Este endpoint admite los siguientes tipos de entidad: `CAMPAIGN`, `FUNDING_INSTRUMENT`, `LINE_ITEM`, `MEDIA_CREATIVE` y `PROMOTED_TWEET`.

**Estadísticas de `MEDIA_CREATIVE`**

Los endpoints de analytics de la Ads API ahora proporcionan métricas para entidades Media Creative. Las Media Creatives son la forma en que se promocionan anuncios in-stream o imágenes en X Audience Platform. La interfaz de X Ads muestra las métricas de Media Creative en las pestañas “In-stream videos” y “Display creatives”. Tanto los endpoints de analytics [sincrónicos](/es/x-ads-api/analytics#get-stats-accounts-account-id) como [asincrónicos](/es/x-ads-api/analytics#asynchronous-analytics) ahora admiten la enumeración de entidad `MEDIA_CREATIVE`.

**Obtener varias cards**

Mejorando la [versión v3](https://devcommunity.x.com/t/ads-api-version-3/100732) del endpoint diseñado para obtener una sola card por su URI de card, ahora es posible obtener *varias* cards usando el endpoint [GET accounts/:account&#95;id/cards/all](/es/x-ads-api/creatives#cards-fetch). Ahora, en lugar de hacer una solicitud por cada card, puedes recuperar hasta 200 cards en una sola solicitud.

Dos cosas a tener en cuenta:

1. La ruta de la URL ahora es `accounts/:account_id/cards/all`. (La ruta anterior ya no está disponible). Esto es para ser consistentes con [el endpoint](/es/x-ads-api/creatives#cards-fetch) diseñado para obtener una card por su ID.
2. El parámetro de solicitud obligatorio ahora se llama card&#95;uris (plural).

**Flexibilidad en la recuperación**

El endpoint [GET accounts/:account&#95;id/targeting&#95;criteria](/es/x-ads-api/campaign-management#get-accounts-account-id-targeting-criteria) ahora admite múltiples IDs de line item. El parámetro line&#95;item&#95;ids, que acepta hasta 200 IDs, es obligatorio. Antes solo se aceptaba un único line item, lo que dificultaba la sincronización. Con este cambio, ahora es posible recuperar más segmentación en menos tiempo.

Los siguientes endpoints también admiten ahora múltiples IDs de line item, aunque el parámetro line&#95;item&#95;ids es opcional en estos casos.

* [GET accounts/:account&#95;id/line&#95;item&#95;apps](https://developer.x.com/en/docs/x-ads-api/campaign-management/api-reference/line-item-apps#get-accounts-account-id-line-item-apps)
* [GET accounts/:account&#95;id/media&#95;creatives](/es/x-ads-api/campaign-management#get-accounts-account-id-media-creatives)
* [GET accounts/:account&#95;id/promoted&#95;accounts](/es/x-ads-api/campaign-management#get-accounts-account-id-promoted-accounts)
* [GET accounts/:account&#95;id/preroll&#95;call&#95;to&#95;actions](/es/x-ads-api/creatives#preroll-call-to-actions)

<div id="changed">
  #### Cambiado
</div>

**Recuperación de campañas y líneas de pedido en borrador**

La forma en que se recuperan las campañas y las líneas de pedido en borrador se ha actualizado. Ahora, el parámetro with&#95;draft(boolean), cuando se establece en true, devuelve *tanto* entidades en borrador como entidades no en borrador. Esto es coherente con la forma en que se recuperan las entidades eliminadas (es decir, usando with&#95;deleted). Anteriormente, obtener tanto entidades en borrador como no en borrador requería al menos dos solicitudes. Ahora, esto se puede hacer en una única llamada a la API.

| **v4** | **v5** |
| :--- | :--- | :--- |
| `draft_only` | `with_draft` |     |

**Segmentación por duración de activación de red**

La Ads API ha resuelto un problema de visualización en el que, después de agregar la segmentación Network Activation Duration, el tipo de segmentación en la respuesta incluía el sufijo &#95;IN&#95;SEC. Hacer referencia a segundos resultaba confuso, ya que Network Activation Duration siempre se representa en meses. Esta corrección hace que la representación sea coherente y reduce la confusión.

| **v4** | **v5** |
| :--- | :--- | :--- |
| `NETWORK_ACTIVATION_DURATION_IN_SEC` | `NETWORK_ACTIVATION_DURATION` |     |

**Recuentos totales y cursores**

En v5, with&#95;total&#95;count y cursor son exclusivos. Especificar ambos en una solicitud devolverá el código de error EXCLUSIVE&#95;PARAMETERS. Antes de v5, with&#95;total&#95;count se ignoraba cuando cursor estaba especificado. Este cambio hace que la relación sea explícita.

<div id="removed">
  #### Eliminados
</div>

Se están eliminando tres campos de las respuestas de Ads API: preview&#95;url, account&#95;id y parent&#95;ids. El esfuerzo de ingeniería necesario para estos tres es mínimo.

* En v4, se anunció que el parámetro de respuesta preview&#95;url para las cards siempre era null. El paso final en esta migración es eliminar preview&#95;url de todas las respuestas de cards.
* El atributo de respuesta account&#95;id se está eliminando para los siguientes recursos, dado que el ID de la cuenta de anuncios ya está presente en la URL, así como en request.params. (Se excluyen intencionalmente los instrumentos de financiación de esta lista, ya que los parent IDs deberían estar presentes en los objetos de respuesta, cuando sea posible, y los account IDs son entidades padre de los instrumentos de financiación).
  * Account media
  * App event providers
  * App event tags
  * Campaigns
  * Cards
  * Line items
  * Promotable users
  * Targeting criteria
* Para las solicitudes [GET accounts/:account&#95;id/targeting&#95;criteria](/es/x-ads-api/campaign-management#get-accounts-account-id-targeting-criteria), ya no devolvemos el campo parent&#95;ids, ya que siempre era un array vacío.

**Cards de aplicaciones sin contenido multimedia**

En v5, las cards de aplicaciones sin contenido multimedia ya no se admiten. [Anteriormente](https://devcommunity.x.com/t/deprecation-announcement-app-download-cards/87807), se eliminó la capacidad de crear o editar cards de aplicaciones sin contenido multimedia. Ahora, los endpoints restantes para este recurso se están marcando como obsoletos.

* Nota: Esto **no** afecta a las cards de descarga de aplicaciones de imagen y video.

**Creación de Account Media**

El endpoint POST accounts/:account&#95;id/account&#95;media ya no está disponible en v5. Otros endpoints para este recurso *no* se ven afectados. La razón de este cambio es que, al agregar contenido multimedia a la [Media Library](/es/x-ads-api/creatives#media-library), hay casos en los que esos recursos se agregan *automáticamente* como entidades de Account Media y, al intentar agregar un recurso ya existente al recurso Account Media, se produce un error. Esto ocurre en los siguientes casos.

* Los recursos `AMPLIFY_VIDEO` agregados a la Media Library se agregan automáticamente como recursos de Account Media con el tipo creativo `PREROLL`.
* Las imágenes con dimensiones específicas agregadas a la Media Library se agregan automáticamente como recursos de Account Media. El tipo creativo (por ejemplo, `INTERSTITIAL`) depende de las dimensiones de la imagen. (Para las dimensiones, consulta nuestra página de [Enumerations](/es/x-ads-api/introduction).)

<div id="v4">
  ### v4
</div>

La versión 4 de la Ads API se publica hoy, 28 de agosto de 2018.

Esta versión incluye mejoras en nuestro producto de [Audiences](/es/x-ads-api/audiences), incluida una nueva interfaz de API respaldada por un backend de procesamiento de audiencias más sólido. La versión 4 también incluye un conjunto de endpoints para administrar la configuración de usuario, cuenta e impuestos. Además, los endpoints accounts/:account&#95;id/videos quedan obsoletos. Esta versión también incluye algunos cambios menores en los nombres de parámetros y respuestas.

Al igual que con la versión 3, proporcionamos un período de transición de 6 meses. El **2019-02-28**, la versión 3 de la Ads API dejará de estar disponible. Recomendamos a todos los socios migrar a la última versión de la API lo antes posible para evitar interrupciones del servicio. Consulta nuestra página de [Versiones](/es/x-ads-api/introduction) para obtener detalles sobre nuestra estrategia de control de versiones.

<div id="new">
  #### Novedades
</div>

***Audiences API***

La nueva Audiences API está basada en nuestro nuevo backend de procesamiento de audiencias, que proporciona mayor robustez y confiabilidad. Este nuevo endpoint permitirá a los socios proporcionar varios tipos de identificadores de usuario para un único usuario, lo que significa que podremos usar señales adicionales para la coincidencia. La documentación de referencia para el nuevo endpoint de Audiences se puede encontrar [aquí](/es/x-ads-api/audiences). Planeamos seguir lanzando actualizaciones y mejoras de este producto durante el resto del año.

Los siguientes endpoints ya no estarán disponibles en la v4 debido a que su funcionalidad es redundante (seguirán funcionando en la v3 y se **retirarán** por completo cuando la v3 ya no esté disponible):

* TON Upload:
  * GET accounts/:account&#95;id/tailored&#95;audience&#95;changes
  * GET accounts/:account&#95;id/tailored&#95;audience&#95;changes/:tailored&#95;audience&#95;change&#95;id
  * POST accounts/:account&#95;id/tailored&#95;audience&#95;changes
  * PUT accounts/:accounti&#95;d/tailored&#95;audiences/global&#95;opt&#95;out
* Real Time Audiences:
  * POST tailored&#95;audience&#95;memberships

Por último, el parámetro `list_type` se eliminará de la solicitud y de la respuesta en *todos* los [endpoints de Tailored Audiences](/es/x-ads-api/audiences) en la versión 4.

***Endpoints de configuración***

Ahora ofrecemos la posibilidad de que los administradores de cuentas configuren y actualicen la configuración de usuario, de cuenta y de impuestos. Los [user settings](/es/x-ads-api/campaign-management#get-accounts-account-id-user-settings-user-id) corresponden a las preferencias de contacto específicas del usuario para una cuenta de anuncios determinada. Usando el endpoint [PUT accounts/:account&#95;id](/es/x-ads-api/campaign-management#put-accounts-account-id), los anunciantes ahora pueden actualizar el nombre de su cuenta y el tipo de industria. Por último, los endpoints de [tax settings](/es/x-ads-api/campaign-management#get-accounts-account-id-tax-settings) permiten a los anunciantes en países donde se cobra un impuesto sobre el valor añadido (IVA) actualizar información como el nombre de la empresa, la dirección, el ID de IVA y si la cuenta es propiedad del anunciante o de una agencia que anuncia en nombre de un anunciante.

#### Cambios

***Cambios de nombre en Universal Lookalike***

Estamos actualizando los valores del enum para el parámetro `lookalike_expansion` en los endpoints [POST accounts/:account&#95;id/line&#95;items](/es/x-ads-api/campaign-management#post-accounts-account-id-line-items) y [PUT accounts/:accountit/line&#95;items/:line&#95;item&#95;id](/es/x-ads-api/campaign-management#put-accounts-account-id-line-items-line-item-id).

| **v3** | **v4** |
| :--- | :--- |
| `NARROW` | `DEFINED` |
| `BALANCED` | `EXPANDED` |

**Uso de `country_code` en todas partes**

Como parte de un esfuerzo más amplio relacionado con la [coherencia](https://devcommunity.x.com/t/feedback-ads-api-consistency/109145) en la Ads API, estamos cambiando el nombre de los parámetros en los siguientes endpoints de `app_country_code` a `country_code`.

* [POST accounts/:account&#95;id/cards/image&#95;app&#95;download](https://devcommunity.x.com/t/ads-api-version-11/168814)
* [PUT accounts/:account&#95;id/cards/image&#95;app&#95;download/:card&#95;id](https://developer.x.com/content/developer-twitterhttps://developer.x.com/en/docs/ads/creatives/api-reference/image-app-download#put-accounts-account-id-cards-image-app-download-card-id)
* [POST accounts/:account&#95;id/cards/video&#95;app&#95;download](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/video-app-download#post-accounts-account-id-cards-video-app-download)
* [PUT accounts/:account&#95;id/cards/video&#95;app&#95;download/:card&#95;id](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/video-app-download#put-accounts-account-id-cards-video-app-download-card-id)

Esto no afecta el comportamiento ni los valores aceptados para estos parámetros y es únicamente un cambio de nombre.

**`preview_url` siempre null**

Tal como se prometió en el anuncio de v3, todas las tarjetas existentes ahora tienen un `card_uri`. Como resultado, el valor de `preview_url` siempre será `null`.

Como recordatorio, asocia una tarjeta con una Publicación usando su valor `card_uri`. Consulta el siguiente ejemplo de solicitud.

$ twurl -X POST -H ads-api.x.com &quot;/4/accounts/18ce54d4x5t/tweet?text=Version 4&amp;card&#95;uri=card://958225772740714496&quot;

#### Eliminado

**Endpoints de vídeo**

Los endpoints accounts/:account&#95;id/videos ya no estarán disponibles en v4. Este endpoint ha quedado obsoleto con la introducción de los endpoints de [Media Library](/es/x-ads-api/creatives#get-accounts-account-id-media-library). Consulta la siguiente comparación de uso.

* endpoint de vídeos de v3: `twurl -H ads-api.x.com "/3/accounts/18ce54d4x5t/videos"`

* endpoint de Media Library de v4 para vídeos: `twurl -H ads-api.x.com "/4/accounts/18ce54d4x5t/media_library?media_type=VIDEO"`

Los endpoints de Media Library ofrecen plena paridad con los endpoints de vídeos y también admiten funcionalidad adicional, como la posibilidad de gestionar imágenes y GIF. Se solicita a los socios que utilicen exclusivamente Media Library para cualquier administración de contenido multimedia.

**`as_user_id` en vista de Publicación**

El parámetro `as_user_id` disponible en el endpoint [GET accounts/:account&#95;id/tweet/preview/:tweet&#95;id](/es/x-ads-api/creatives#get-accounts-account-id-tweets) ya no será aceptado. La vista previa siempre se mostrará como si fuera del autor de la Publicación.

<div id="v3">
  ### v3
</div>

La versión 3 de la Ads API [se lanzó](https://devcommunity.x.com/t/ads-api-version-3/100732) el 1 de febrero de 2018. La versión 2 de la Ads API llegará a su fin de vida útil el 1 de agosto de 2018.

Esta versión incluye nuestro nuevo producto Audience Intelligence, acceso a la Media Library y flujos de trabajo de tarjetas mejorados. También anunciamos la desaprobación (deprecación) del endpoint [PUT accounts/:account&#95;id/targeting&#95;criteria](/es/x-ads-api/campaign-management#get-accounts-account-id-targeting-criteria). Finalmente, la versión 3 incluye algunos cambios menores en parámetros y respuestas y un límite inferior para el tamaño de los lotes.

Como con la [versión 2](https://devcommunity.x.com/t/ads-api-version-2/90360), damos a los socios **6 meses** para hacer la transición. El **2018-08-01**, la v2 de la Ads API se desactivará. Recomendamos a todos los socios y desarrolladores que migren a v3 lo antes posible.

**Audience Intelligence**

Audience Intelligence ofrece información en tiempo real sobre los principales hashtags, @handles y eventos más relevantes para una audiencia determinada de X. Por ejemplo, introduce Hombre 18-34 en EE. UU. y verás #nintendoswitch, #cardinal y @ricegum en tendencia entre esta audiencia.

Los [endpoints](/es/x-ads-api/audiences) de Audience Intelligence proporcionarán la siguiente funcionalidad:

* Dada una audiencia de entrada, obtener los hashtags, @handles y eventos más relevantes.
* Dada una audiencia de entrada, obtener información demográfica clave (como edad, género e ingresos del hogar).
* Dada una palabra clave, obtener la serie temporal del volumen de Publicaciones.

**Media Library**

La [Media Library](/es/x-ads-api/creatives#get-accounts-account-id-media-library) permite gestionar imágenes, GIFs y videos para cuentas de anuncios. Estos objetos de contenido multimedia se pueden usar en Publicaciones y para crear tarjetas. También se pueden reutilizar en múltiples creatividades, lo que elimina la necesidad de cargar el mismo recurso varias veces.

Los objetos de la biblioteca se identifican mediante un media&#95;key. Las claves de contenido multimedia son valores de tipo cadena de texto con el siguiente formato: por ejemplo, 13&#95;875943225764098048. En la Ads API, avanzamos hacia el uso de claves de contenido multimedia para todo el contenido multimedia.

**Flujo de trabajo de tarjetas mejorado**

Todos nuestros endpoints de tarjetas ahora son compatibles con claves de contenido multimedia. Esto permite usar objetos de la Media Library para crear o actualizar tarjetas.

Además, estamos introduciendo dos nuevos endpoints para [recuperar detalles de tarjetas](/es/x-ads-api/creatives#fetching-cards). Estos endpoints se pueden usar para buscar tarjetas utilizadas en Publicaciones o Publicaciones programadas, por ejemplo, especificando el card&#95;uri o el id. Anteriormente, esto no era posible.

<div id="other-changes">
  #### Otros cambios
</div>

Además de estas nuevas funciones, incluimos los siguientes cambios en la versión 3.

**Nuevo**

* La respuesta del endpoint [GET insights/keywords/search](/es/x-ads-api/audiences) ahora incluye un atributo related&#95;keywords con 30 términos relacionados con las palabras clave de entrada.

**Cambiado**

* El tamaño máximo del lote de criterios de segmentación ahora es 500.
* Los atributos de respuesta card&#95;uri y preview&#95;url ahora son mutuamente excluyentes. Cuando una tarjeta tiene un card&#95;uri, el preview&#95;url será null. Cuando una tarjeta no tiene un card&#95;uri, solo se devolverá preview&#95;url.
  * Todas las tarjetas creadas a partir de 2018-01-29 tendrán un card&#95;uri.
  * Para la versión 4, todas las tarjetas existentes tendrán un card&#95;uri.
* Ya no es posible *crear* tarjetas con imágenes 5:2. Aunque las tarjetas existentes basadas en imágenes 5:2 seguirán funcionando, recomendamos a los partners cambiar a las relaciones de aspecto de mejor rendimiento 1.91:1 o 1:1 (cuando se admitan).

**Eliminado**

* El endpoint [PUT accounts/:account&#95;id/targeting&#95;criteria](/es/x-ads-api/campaign-management#get-accounts-account-id-targeting-criteria) ya no está disponible. Hemos decidido realizar este cambio porque el comportamiento de reemplazo con este endpoint causaba confusión a los anunciantes y no era coherente con nuestros otros endpoints PUT que actualizan un único recurso a la vez. En su lugar, los partners deben usar el endpoint [POST batch/accounts/:account&#95;id/targeting&#95;criteria](/es/x-ads-api/campaign-management#post-batch-accounts-account-id-targeting-criteria), que ofrece mayor flexibilidad, incluida la posibilidad de agregar y eliminar segmentación en una sola solicitud.
* El atributo de respuesta paused ya no se devuelve para instrumentos de financiación. En su lugar, consulta el atributo de respuesta entity&#95;status para determinar si un instrumento de financiación está pausado o no. Además, dado que paused y cancelled corresponden al mismo valor, cancelled tampoco se devuelve ya en la respuesta.
* Hemos eliminado el parámetro card&#95;id del endpoint [GET accounts/:account&#95;id/tweet/preview](/es/x-ads-api/creatives#get-accounts-account-id-tweets).
* Dado que no es posible recuperar publicaciones programadas eliminadas, el parámetro with&#95;deleted ya no se admite.
* El parámetro draft&#95;only se ha eliminado de los siguientes endpoints, ya que estas entidades nunca pueden estar en estado de borrador:
  * [GET accounts/:account&#95;id/targeting&#95;criteria](/es/x-ads-api/campaign-management#get-accounts-account-id-targeting-criteria)
  * [GET accounts/:account&#95;id/promoted&#95;tweets](/es/x-ads-api/campaign-management#get-accounts-account-id-promoted-tweets)
  * [GET accounts/:account&#95;id/promoted&#95;accounts](/es/x-ads-api/campaign-management#get-accounts-account-id-promoted-accounts)
  * [GET accounts/:account&#95;id/media&#95;creatives](/es/x-ads-api/campaign-management#get-accounts-account-id-media-creatives)
  * [GET accounts/:account&#95;id/preroll&#95;call&#95;to&#95;actions](/es/x-ads-api/creatives#get-accounts-account-id-preroll-call-to-actions)

**Nota**

Tanto las Video Website Cards como las publicaciones programadas han salido ahora de la fase beta. Consulta [este hilo](https://devcommunity.x.com/t/announcement-update-to-scheduled-tweets/94869) para conocer los cambios que hemos realizado en las publicaciones programadas desde su lanzamiento. **Cambios importantes/Deprecaciones[¶](#breaking-changes-deprecations "Enlace permanente a este encabezado")**

* `total_count` ahora es un atributo de respuesta opcional. Solo estará disponible si `with_total_count` se establece en `true`

La única excepción es que, en v2, `CITY` ahora es `METROS`, para reflejar correctamente el hecho de que el tipo de ubicación se refiere a Designated Market Areas (DMAs) o “metros”.

* `display_properties` en los endpoints PUT accounts/:account&#95;id/promoted&#95;tweets. Este valor ya no se incluirá como parte de la respuesta

* Como resultado del punto anterior, ya no es posible actualizar (PUT) entidades promoted&#95;tweets

* El parámetro `line_item_id` en el endpoint [GET accounts/:account&#95;id/promoted&#95;tweets](/es/x-ads-api/campaign-management#get-accounts-account-id-promoted-tweets) se ha eliminado

**Cambios en la versión 1:[¶](#changes-in-version-1 "Enlace permanente a este encabezado")**

* [Compatibilidad con versiones](https://blog.x.com/2016/versioning-is-coming-to-twitter-s-ads-apis)
* `CUSTOM` [objective](/es/x-ads-api/campaign-management#objective-based-campaigns) ya no se admite

* [Cambios en la estimación del alcance](https://developer.x.com/en/docs/ads/campaign-management/api-reference/reach-estimate.html#get-accounts-account-id-reach-estimate):
* Para ofrecer una mejor estimación del alcance, el endpoint ahora tiene en cuenta el presupuesto. Los siguientes parámetros ahora son obligatorios:
  * [nuevo] `campaign_daily_budget_amount_local_micro`
  * `currency`
  * `bid`
  * `objective`
* El objeto de respuesta ha cambiado y ahora devuelve rangos en los valores de la respuesta.
* `infinite_count` se ha renombrado a `infinite_bid_count` para evitar confusiones acerca de su propósito
* Además de `count` e `infinite_bid_count`, ahora se devolverán los siguientes nuevos datos:
  * `impressions`
  * `engagements`
  * `estimated_daily_spend_local_micro`
* Cambio en el tipo de datos de las audiencias personalizadas
* El `data_type` de Tailored Audiences se ha cambiado de `tailored_audiences` a `tailored_audience` en todas nuestras respuestas.
* Las Shared Tailored Audiences ahora están disponibles en beta *solo mediante API*. Las Shared Tailored Audiences permiten que una única audiencia se utilice en varias cuentas publicitarias. Usa el endpoint [POST accounts/:account&#95;id/tailored&#95;audiences/:tailored&#95;audience&#95;id/permissions](/es/x-ads-api/audiences) (y los relacionados) para administrar los permisos de una audiencia personalizada que quieras compartir entre varias cuentas publicitarias.
* Mejoras significativas en la forma de recopilar estadísticas de rendimiento [analytics](/es/x-ads-api/analytics) para cuentas de anunciantes:
* Para alinearnos con nuestras [mejores prácticas](/es/x-ads-api/analytics#best-practices), ahora solo permitiremos obtener datos correspondientes a un máximo de **7 días** para los [endpoints síncronos de estadísticas](/es/x-ads-api/analytics#get-stats-accounts-account-id).
* Para simplificar la obtención de métricas, hemos sustituido el parámetro `metrics` por un nuevo parámetro `metric_groups`. Los desarrolladores deben especificar qué grupos de métricas desean recibir en la respuesta de una solicitud determinada.
  * Cualquier solicitud de métricas que no sean apropiadas para una entidad determinada se excluirá de la respuesta y se representará con valores `null`. *Estas métricas no se tendrán en cuenta para tu límite de costos de analítica.*
* La respuesta se ha **simplificado considerablemente** y ahora se ajustará mejor a la forma en que se exponen las métricas en la interfaz de usuario.
  * Anteriormente exponíamos una métrica separada para cada ubicación de publicación (Publicaciones promocionadas en búsqueda, Publicaciones promocionadas en timelines, Publicaciones promocionadas en perfiles &amp; detalles de la publicación, X Audience Platform). Ahora devolveremos un conjunto estandarizado de métricas para cada una; en lugar de `promoted_tweet_timeline_impressions`, `promoted_tweet_search_impressions`, `promoted_tweets_profile_impressions`, `promoted_tweets_tpn_impressions`, estas ahora se expondrán, cuando se soliciten, dentro de una de las siguientes categorías como una única métrica, `impressions` (esto se aplica a todas las métricas):
  * `ALL_ON_TWITTER`
  * `PUBLISHER_NETWORK`
  * Cuando hagas una solicitud, obtendrás una única métrica `impressions` para que conciliar los valores con los de nuestra interfaz de usuario sea más sencillo.
  * Debes realizar dos consultas para obtener tanto los datos de `ALL_ON_TWITTER` como de `PUBLISHER_NETWORK`, ya que no pueden combinarse.
* ¡Los [endpoints de estadísticas asíncronas](/es/x-ads-api/analytics#asynchronous-analytics) ya están disponibles, a partir de los comentarios de nuestros desarrolladores!
  * Un nuevo conjunto de endpoints para solicitar estadísticas *de forma asíncrona*, ya sea para datos que no necesitas de inmediato o para extracciones de datos históricos.
  * Pon en cola un trabajo de estadísticas usando un nuevo endpoint único. Obtendremos los datos que has solicitado según lo permitan los recursos.
  * Puedes consultar un endpoint de estado del trabajo para determinar si los datos están disponibles.
  * Una vez que los datos estén disponibles, proporcionaremos un id de recogida para que descargues la respuesta JSON, que será equivalente a la respuesta del endpoint síncrono.
  * Consulta hasta **90 días** de datos correspondientes a un máximo de **20 entidades** en un solo trabajo.
* Consulta nuestra guía de migración de analytics v1, que incluye un mapeo de las métricas de v0 a las métricas de v1
* Mejoras en el Sandbox * Ahora puedes [crear varias cuentas de anuncios de prueba](/es/x-ads-api/campaign-management#post-accounts) en el entorno Sandbox. * Ahora puedes crear múltiples instrumentos de financiación para una cuenta de anuncios de prueba únicamente en el entorno Sandbox. Esto te permite probar todos nuestros tipos de instrumentos de financiación. Anteriormente solo había disponible una fuente de financiación `CREDIT_CARD` para hacer pruebas. * ¿Quieres probar una función beta? Ahora puedes activar o desactivar funciones para una cuenta en el entorno Sandbox para ajustarse a tus necesidades de prueba.
