<div id="overview">
  ## Descripción general
</div>

Cada mes, realizamos cambios y lanzamos varias funciones nuevas en la X Ads API. Estos cambios casi siempre son compatibles con versiones anteriores; sin embargo, solemos tener un puñado de cambios incompatibles cada año. Hemos recibido comentarios de desarrolladores sobre los desafíos que nuestro rápido ritmo de cambios en la Ads API plantea a sus ciclos de desarrollo a la hora de implementar nuevas funciones, gestionar obsolescencias y probar cambios. Queremos mejorar la experiencia de desarrollador al usar nuestra plataforma de Ads, por lo que introdujimos el concepto de versionar nuestros endpoints.

Algunas definiciones de conceptos sobre los que hablamos:

**Versión**: Se refiere al número de versión que se encuentra en la ruta de la URL de cualquier solicitud de Ads API, por ejemplo: GET /{version_number}/accounts. Este estilo de versionado se conoce como versionado de URI.

**Cambios incompatibles**: Los cambios incompatibles son cualquier cambio que requiera dedicar recursos de desarrollo para mantener la funcionalidad existente. Esto incluye los recursos usados para investigar los cambios que deben realizarse, determinar las funciones/endpoints que se van a marcar como obsoletos y la implementación final de todos esos cambios. Entre los cambios incompatibles se incluyen, por ejemplo:

* Eliminar un parámetro de la solicitud/respuesta de la API

* Modificar el nombre de cualquier parámetro o endpoint

* Cambio en la representación de valores (preview&#95;url → card&#95;uri)

* Cambio en el comportamiento de los endpoints (p. ej., estadísticas async vs sync)

* Agregar o modificar parámetros opcionales u obligatorios (p. ej., hacer que name sea un campo obligatorio en la solicitud)

**Deprecación**: Las versiones o productos obsoletos dejarán de tener soporte y se recomienda que los desarrolladores dejen de usar estas APIs.

**Retiro (sunset)**: Una vez que se retira (sunset) un producto o una API, el conjunto correspondiente de endpoints dejará de estar accesible a través de la API.

<div id="versioning-strategy">
  ## Estrategia de versionado
</div>

Los principios básicos de la estrategia son:

1. Todos los cambios incompatibles se agruparán en una nueva versión

2. El período de obsolescencia de las versiones existentes cuando se anuncia una nueva versión será de 6 meses

3. En cualquier momento, la API permitirá solicitudes de dos versiones simultáneamente; sin embargo, la más antigua de las dos no tendrá soporte

4. Con el fin de facilitar una adopción más rápida de nuevos productos, estos se lanzarán de forma continua (fuera del ciclo de versionado)

5. Todas las respuestas de la API contendrán un `x-current-api-version`, que se establecerá en la versión actual de la API, además de un encabezado `x-api-warn` al llamar a cualquier endpoint de la API en desuso.

En caso de que haya cambios fundamentales en los requisitos del producto que requieran un cambio incompatible en la API (por ejemplo, la eliminación de la segmentación por múltiples grupos de edad), enviaremos un aviso con 90 días de antelación para anunciar este cambio incompatible y, al menos 90 días después de que se publique el aviso, se implementará el cambio incompatible.

<div id="v5">
  ### v5
</div>

Hoy, 28 de febrero de 2019, X presenta Ads API v5, con actualizaciones centradas en permitir una mayor escala y eficiencia.

Esta versión incluye un nuevo endpoint para determinar qué entidades estuvieron activas en un periodo de tiempo determinado, estadísticas para Media Creatives (es decir, videos in-stream e imágenes en X Audience Platform), la capacidad de obtener *múltiples* cards por URI de card y una mayor flexibilidad para recuperar criterios de segmentación y otras entidades. Además, hemos corregido algunos errores y realizado actualizaciones en los nombres de parámetros y atributos de respuesta. Por último, las non-media app cards y el endpoint POST `accounts/:account_id/account_media` se han declarado obsoletos.

Como en versiones anteriores, habrá un período de transición de 6 meses para migrar a v5. El **2019-08-28**, la versión 4 de Ads API ya no estará disponible. Recomendamos a todos los socios que migren a la versión más reciente de la API lo antes posible para evitar interrupciones del servicio. *La versión 3 de Ads API ha llegado al final de su vida útil y ya no está disponible.*

#### New

**Determinación de qué entidades estuvieron activas**

El endpoint [Active Entities](/es/x-ads-api/analytics#active-entities-2) indica si las métricas de analytics para entidades de anuncios han cambiado. Diseñado para usarse junto con los endpoints de analytics, Active Entities funciona especificando un tipo de entidad y un rango de fechas —con un máximo de 90 días— y devuelve un conjunto de IDs de entidades para las que tu plataforma debe solicitar analytics. No se deben consultar IDs distintos de los devueltos en solicitudes de analytics posteriores.

Este endpoint admite los siguientes tipos de entidad: `CAMPAIGN`, `FUNDING_INSTRUMENT`, `LINE_ITEM`, `MEDIA_CREATIVE` y `PROMOTED_TWEET`.

**Estadísticas de `MEDIA_CREATIVE`**

Los endpoints de analytics de Ads API ahora proporcionan métricas para entidades Media Creative. Las Media Creatives son la forma en que se promocionan anuncios in-stream o imágenes en la X Audience Platform. La interfaz de X Ads muestra las métricas de Media Creative en las pestañas “In-stream videos” y “Display creatives”. Tanto los endpoints de analytics [síncronos](/es/x-ads-api/analytics#get-stats-accounts-account-id) como [asíncronos](/es/x-ads-api/analytics#asynchronous-analytics) ahora admiten el enum de entidad `MEDIA_CREATIVE`.

**Obtener varias cards**

Mejorando la [versión v3](https://devcommunity.x.com/t/ads-api-version-3/100732) del endpoint diseñado para recuperar una sola card por su valor de card URI, ahora es posible obtener *varias* cards usando el endpoint [GET accounts/:account&#95;id/cards/all](/es/x-ads-api/creatives#cards-fetch). Ahora, en lugar de hacer una solicitud por cada card, puedes recuperar hasta 200 cards en una sola solicitud.

Dos cosas a tener en cuenta:

1. La ruta de URL ahora es `accounts/:account_id/cards/all`. (La ruta anterior ya no está disponible). Esto se hace para mantener la coherencia con [el endpoint](/es/x-ads-api/creatives#cards-fetch) diseñado para recuperar una card por ID.
2. El parámetro de solicitud obligatorio ahora se llama card&#95;uris (en plural).

**Flexibilidad al recuperar**

El endpoint [GET accounts/:account&#95;id/targeting&#95;criteria](/es/x-ads-api/campaign-management#get-accounts-account-id-targeting-criteria) ahora admite múltiples IDs de line item. El parámetro line&#95;item&#95;ids, que acepta hasta 200 IDs, es obligatorio. Anteriormente, solo se aceptaba un único line item, lo que dificultaba la sincronización. Con este cambio, ahora es posible recuperar más segmentación en menos tiempo.

Los siguientes endpoints ahora también admiten múltiples IDs de line item, aunque el parámetro line&#95;item&#95;ids es opcional para estos.

* [GET accounts/:account&#95;id/line&#95;item&#95;apps](https://developer.x.com/en/docs/x-ads-api/campaign-management/api-reference/line-item-apps#get-accounts-account-id-line-item-apps)
* [GET accounts/:account&#95;id/media&#95;creatives](/es/x-ads-api/campaign-management#get-accounts-account-id-media-creatives)
* [GET accounts/:account&#95;id/promoted&#95;accounts](/es/x-ads-api/campaign-management#get-accounts-account-id-promoted-accounts)
* [GET accounts/:account&#95;id/preroll&#95;call&#95;to&#95;actions](/es/x-ads-api/creatives#preroll-call-to-actions)

<div id="changed">
  #### Cambios
</div>

**Recuperación de campañas y líneas de pedido en borrador**

La forma de obtener las campañas y las líneas de pedido en borrador se ha actualizado. Ahora, el parámetro with&#95;draft(boolean), cuando se establece en true, devuelve *tanto* entidades en borrador como entidades que no están en borrador. Esto es coherente con la forma en que se recuperan las entidades eliminadas (es decir, usando with&#95;deleted). Anteriormente, obtener tanto las entidades en borrador como las que no lo están requería al menos dos peticiones. Ahora, esto se puede hacer en una única llamada a la API.

| **v4** | **v5** |
| :--- | :--- | :--- |
| `draft_only` | `with_draft` |     |

**Segmentación de Network Activation Duration**

La Ads API ha resuelto un problema de visualización en el que, después de añadir segmentación de Network Activation Duration, el tipo de segmentación en la respuesta incluía un sufijo &#95;IN&#95;SEC. Hacer referencia a segundos resultaba confuso, ya que Network Activation Duration siempre se representa en meses. Esta corrección hace que la representación sea coherente y reduce la confusión.

| **v4** | **v5** |
| :--- | :--- | :--- |
| `NETWORK_ACTIVATION_DURATION_IN_SEC` | `NETWORK_ACTIVATION_DURATION` |     |

**Recuentos totales y cursores**

En v5, with&#95;total&#95;count y cursor son exclusivos. Especificar ambos en una petición devolverá el código de error EXCLUSIVE&#95;PARAMETERS. Antes de v5, with&#95;total&#95;count se ignoraba cuando se especificaba cursor. Este cambio hace que la relación sea explícita.

<div id="removed">
  #### Eliminados
</div>

Se están eliminando tres campos de las respuestas de Ads API: preview&#95;url, account&#95;id y parent&#95;ids. El esfuerzo de ingeniería necesario para estos tres es mínimo.

* En la v4, se anunció que el parámetro de respuesta preview&#95;url para las cards siempre era null. El paso final de esta migración es eliminar preview&#95;url de todas las respuestas de cards.
* El atributo de respuesta account&#95;id se está eliminando para los siguientes recursos, dado que el ID de la cuenta de anuncios ya está presente en la URL, así como en request.params. (Es intencional excluir los instrumentos de financiación de esta lista, ya que los IDs de entidad principal (parent IDs) deberían estar presentes en los objetos de respuesta, cuando sea posible, y los account IDs son entidades principales de los instrumentos de financiación).
  * Account media
  * App event providers
  * App event tags
  * Campaigns
  * Cards
  * Line items
  * Promotable users
  * Targeting criteria
* Para las solicitudes [GET accounts/:account&#95;id/targeting&#95;criteria](/es/x-ads-api/campaign-management#get-accounts-account-id-targeting-criteria), ya no devolvemos el campo parent&#95;ids, ya que siempre era un array vacío.

**Non-media app cards**

En la v5, las non-media app cards ya no son compatibles. [Anteriormente](https://devcommunity.x.com/t/deprecation-announcement-app-download-cards/87807), se dejó de permitir la creación o edición de non-media app cards. Ahora, los endpoints restantes para este recurso se están dejando obsoletos.

* Nota: Esto **no** afecta a las app download cards de imagen y video.

**Account media creates**

El endpoint POST accounts/:account&#95;id/account&#95;media ya no está disponible en la v5. Otros endpoints para este recurso *no* se ven afectados. El motivo de este cambio es que, al agregar contenido multimedia a la [Media Library](/es/x-ads-api/creatives#media-library), hay casos en los que esos recursos se agregan *automáticamente* como entidades de Account Media y, al intentar agregar un recurso ya existente al recurso Account Media, se produce un error. Esto ocurre en los siguientes casos:

* Los recursos `AMPLIFY_VIDEO` agregados a la Media Library se agregan automáticamente como recursos de Account Media con el tipo de creativo `PREROLL`.
* Las imágenes con dimensiones específicas agregadas a la Media Library se agregan automáticamente como recursos de Account Media. El tipo de creativo (por ejemplo, `INTERSTITIAL`) depende de las dimensiones de la imagen. (Para las dimensiones, consulta nuestra página de [Enumerations](/es/x-ads-api/introduction).)

<div id="v4">
  ### v4
</div>

La versión 4 de la Ads API se publica hoy, 28 de agosto de 2018.

Esta versión incluye mejoras en nuestro producto de [Audiences](/es/x-ads-api/audiences), incluida una nueva interfaz de API respaldada por un backend de procesamiento de audiencias más sólido. La versión 4 también incluye un conjunto de endpoints para administrar la configuración de usuario, cuenta e impuestos. Además, los endpoints accounts/:account&#95;id/videos quedan obsoletos. Esta versión también incluye algunos cambios menores en los nombres de parámetros y respuestas.

Al igual que con la versión 3, proporcionamos un período de transición de 6 meses. A partir del **2019-02-28**, la versión 3 de la Ads API dejará de estar disponible. Recomendamos a todos los socios que migren a la última versión de la API lo antes posible para evitar cualquier interrupción del servicio. Consulta nuestra página de [Versions](/es/x-ads-api/introduction) para obtener más detalles sobre nuestra estrategia de control de versiones.

<div id="new">
  #### Nuevo
</div>

***Audience API***

La nueva Audiences API está desarrollada sobre nuestro nuevo backend de procesamiento de audiencias, que proporciona una mayor robustez y fiabilidad. Este nuevo endpoint permitirá a los socios proporcionar múltiples tipos de identificadores de usuario para un mismo usuario, lo que significa que podremos utilizar señales adicionales para el emparejamiento. La documentación de referencia para el nuevo endpoint de Audience se puede encontrar [aquí](/es/x-ads-api/audiences). Planeamos seguir lanzando actualizaciones y mejoras de este producto durante el resto del año.

Los siguientes endpoints dejarán de estar disponibles en la v4 debido a funcionalidad redundante (seguirán funcionando en la v3 y se retirarán por completo (**sunset**) cuando la v3 ya no esté disponible):

* TON Upload:
  * GET accounts/:account&#95;id/tailored&#95;audience&#95;changes
  * GET accounts/:account&#95;id/tailored&#95;audience&#95;changes/:tailored&#95;audience&#95;change&#95;id
  * POST accounts/:account&#95;id/tailored&#95;audience&#95;changes
  * PUT accounts/:accounti&#95;d/tailored&#95;audiences/global&#95;opt&#95;out
* Real Time Audiences:
  * POST tailored&#95;audience&#95;memberships

Por último, el parámetro `list_type` se eliminará de la solicitud y la respuesta en *todos* los [endpoints de Tailored Audiences](/es/x-ads-api/audiences) en la versión 4.

***Settings Endpoints***

Ahora ofrecemos la posibilidad de que los administradores de cuentas establezcan y actualicen la configuración de usuario, de cuenta y fiscal. [User settings](/es/x-ads-api/campaign-management#get-accounts-account-id-user-settings-user-id) corresponde a las preferencias de contacto específicas del usuario para una cuenta de anuncios determinada. Usando el endpoint [PUT accounts/:account&#95;id](/es/x-ads-api/campaign-management#put-accounts-account-id), los anunciantes ahora pueden actualizar el nombre de su cuenta y el sector de actividad. Por último, los endpoints de [tax settings](/es/x-ads-api/campaign-management#get-accounts-account-id-tax-settings) permiten a los anunciantes en países donde se cobra un impuesto al valor añadido (IVA) actualizar información como el nombre de la empresa, la dirección, el ID de IVA y si la cuenta es propiedad del anunciante o de una agencia que anuncia en nombre de un anunciante.

* [PUT accounts/:account&#95;id/cards/image&#95;app&#95;download/:card&#95;id](https://developer.x.com/content/developer-twitterhttps://developer.x.com/en/docs/ads/creatives/api-reference/image-app-download#put-accounts-account-id-cards-image-app-download-card-id)
* [POST accounts/:account&#95;id/cards/video&#95;app&#95;download](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/video-app-download#post-accounts-account-id-cards-video-app-download)
* [PUT accounts/:account&#95;id/cards/video&#95;app&#95;download/:card&#95;id](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/video-app-download#put-accounts-account-id-cards-video-app-download-card-id)

Esto no afecta el comportamiento ni los valores aceptados para estos parámetros y es solo un cambio de nombre.

**`preview_url` siempre null**

Tal como se prometió en el anuncio de la v3, todas las tarjetas existentes ahora tienen un `card_uri`. Como resultado, el valor de `preview_url` siempre será `null`.

Como recordatorio, asocia una tarjeta con una publicación usando su valor `card_uri`. Consulta la siguiente solicitud de ejemplo.

$ twurl -X POST -H ads-api.x.com &quot;/4/accounts/18ce54d4x5t/tweet?text=Version 4&amp;card&#95;uri=card://958225772740714496&quot;

<div id="removed">
  #### Eliminados
</div>

**Endpoints de vídeo**

Los endpoints accounts/:account&#95;id/videos ya no estarán disponibles en v4. Este endpoint ha quedado obsoleto con la introducción de los endpoints de [Media Library](/es/x-ads-api/creatives#get-accounts-account-id-media-library). Consulta la siguiente comparación de uso.

* endpoint de vídeos de v3: `twurl -H ads-api.x.com "/3/accounts/18ce54d4x5t/videos"`

* endpoint de Media Library de v4 para vídeos: `twurl -H ads-api.x.com "/4/accounts/18ce54d4x5t/media_library?media_type=VIDEO"`

Los endpoints de Media Library tienen paridad completa con los endpoints de vídeos y también admiten funcionalidad adicional, como la capacidad de gestionar imágenes y GIF. Se solicita a los partners que usen exclusivamente Media Library para cualquier gestión de contenido multimedia.

**`as_user_id` en Tweet View**

El parámetro `as_user_id` disponible en el endpoint [GET accounts/:account&#95;id/tweet/preview/:tweet&#95;id](/es/x-ads-api/creatives#get-accounts-account-id-tweets) ya no se aceptará. La vista previa siempre se mostrará como el autor de la Publicación.

<div id="v3">
  ### v3
</div>

La versión 3 del Ads API [se lanzó](https://devcommunity.x.com/t/ads-api-version-3/100732) el 1 de febrero de 2018. La versión 2 del Ads API alcanzará el final de su vida útil el 1 de agosto de 2018.

Esta versión incluye nuestro nuevo producto Audience Intelligence, acceso a la Media Library y flujos de trabajo de tarjetas mejorados. También anunciamos la obsolescencia del endpoint [PUT accounts/:account&#95;id/targeting&#95;criteria](/es/x-ads-api/campaign-management#get-accounts-account-id-targeting-criteria). Finalmente, la versión 3 incluye algunos cambios menores en parámetros y respuestas y un límite inferior para el tamaño de los lotes.

Como con la [versión 2](https://devcommunity.x.com/t/ads-api-version-2/90360), damos a los socios **6 meses** para hacer la transición. El **2018-08-01**, la v2 del Ads API se desactivará. Recomendamos a todos los socios y desarrolladores migrar a la v3 lo antes posible.

**Audience Intelligence**

Audience Intelligence ofrece información en tiempo real sobre los principales hashtags, @handles y eventos más relevantes para una audiencia determinada en X. Por ejemplo, introduce Hombres de 18 a 34 años en EE. UU. y verás #nintendoswitch, #cardinal y @ricegum como tendencias entre esta audiencia.

Los [endpoints](/es/x-ads-api/audiences) de Audience Intelligence proporcionan la siguiente funcionalidad:

* Dada una audiencia de entrada, obtener los principales hashtags, @handles y eventos relevantes.
* Dada una audiencia de entrada, obtener información demográfica clave (como edad, género e ingresos del hogar).
* Dada una palabra clave, obtener la serie temporal del volumen de Publicaciones.

**Media Library**

La [Media Library](/es/x-ads-api/creatives#get-accounts-account-id-media-library) permite gestionar imágenes, GIFs y videos para cuentas de anuncios. Estos objetos de contenido multimedia se pueden usar en Publicaciones y para crear tarjetas. También se pueden reutilizar en múltiples creatividades, lo que elimina la necesidad de subir el mismo recurso varias veces.

Los objetos de la biblioteca se identifican mediante un media&#95;key. Las media keys son valores de tipo string con el siguiente formato: 13&#95;875943225764098048, por ejemplo. En el Ads API, estamos pasando a usar media keys para todo el contenido multimedia.

**Flujo de trabajo de tarjetas mejorado**

Todos nuestros endpoints de tarjetas ahora admiten media keys. Esto permite utilizar objetos de la Media Library para crear o actualizar tarjetas.

Además, presentamos dos nuevos endpoints para [recuperar detalles de tarjetas](/es/x-ads-api/creatives#fetching-cards). Estos endpoints se pueden utilizar para buscar tarjetas usadas en Publicaciones o Publicaciones programadas, por ejemplo, especificando el card&#95;uri o el id. Anteriormente, esto no era posible.

<div id="other-changes">
  #### Otros cambios
</div>

Además de estas nuevas funciones, incluimos los siguientes cambios en la versión 3.

**Nuevos**

* La respuesta del endpoint [GET insights/keywords/search](/es/x-ads-api/audiences) ahora incluye un atributo `related_keywords` con 30 términos relacionados con las palabras clave de entrada.

**Cambiados**

* El tamaño máximo de lote de criterios de segmentación ahora es 500.
* Los atributos de respuesta `card_uri` y `preview_url` ahora son mutuamente excluyentes. Cuando una tarjeta tiene un `card_uri`, la `preview_url` será `null`. Cuando una tarjeta no tiene un `card_uri`, solo se devolverá la `preview_url`.
  * Todas las tarjetas creadas a partir de 2018-01-29 tendrán un `card_uri`.
  * Para la versión 4, todas las tarjetas existentes tendrán un `card_uri`.
* Ya no es posible *crear* tarjetas con imágenes 5:2. Aunque las tarjetas *existentes* basadas en imágenes 5:2 seguirán funcionando, recomendamos a los partners cambiar a las relaciones de aspecto de mejor rendimiento 1.91:1 o 1:1 (cuando sean compatibles).

**Eliminados**

* El endpoint [PUT accounts/:account&#95;id/targeting&#95;criteria](/es/x-ads-api/campaign-management#get-accounts-account-id-targeting-criteria) ya no está disponible. Hemos decidido realizar este cambio porque el comportamiento de reemplazo con este endpoint causaba confusión entre los anunciantes y no era coherente con nuestros otros endpoints PUT que actualizan un único recurso a la vez. En su lugar, los partners deben usar el endpoint [POST batch/accounts/:account&#95;id/targeting&#95;criteria](/es/x-ads-api/campaign-management#post-batch-accounts-account-id-targeting-criteria), que proporciona mayor flexibilidad, incluida la posibilidad de añadir y eliminar segmentación en una sola solicitud.
* El atributo de respuesta `paused` ya no se devuelve para los instrumentos de financiación. En su lugar, consulte el atributo de respuesta `entity_status` para determinar si un instrumento de financiación está o no en pausa. Además, dado que `paused` y `cancelled` corresponden al mismo valor, `cancelled` tampoco se devuelve en la respuesta.
* Hemos eliminado el parámetro `card_id` del endpoint [GET accounts/:account&#95;id/tweet/preview](/es/x-ads-api/creatives#get-accounts-account-id-tweets).
* Dado que no es posible recuperar Publicaciones programadas eliminadas, el parámetro `with_deleted` ya no es compatible.
* El parámetro `draft_only` se ha eliminado de los siguientes endpoints, ya que estas entidades nunca pueden estar en estado de borrador:
  * [GET accounts/:account&#95;id/targeting&#95;criteria](/es/x-ads-api/campaign-management#get-accounts-account-id-targeting-criteria)
  * [GET accounts/:account&#95;id/promoted&#95;tweets](/es/x-ads-api/campaign-management#get-accounts-account-id-promoted-tweets)
  * [GET accounts/:account&#95;id/promoted&#95;accounts](/es/x-ads-api/campaign-management#get-accounts-account-id-promoted-accounts)
  * [GET accounts/:account&#95;id/media&#95;creatives](/es/x-ads-api/campaign-management#get-accounts-account-id-media-creatives)
  * [GET accounts/:account&#95;id/preroll&#95;call&#95;to&#95;actions](/es/x-ads-api/creatives#get-accounts-account-id-preroll-call-to-actions)

**Nota**

Tanto las Video Website Cards como las Publicaciones programadas han salido ahora de la versión beta. Consulte [este hilo](https://devcommunity.x.com/t/announcement-update-to-scheduled-tweets/94869) para ver los cambios que hemos realizado en las Publicaciones programadas desde su lanzamiento. * Para ofrecer una mejor estimación del alcance, el endpoint ahora tiene en cuenta el presupuesto. Ahora se requieren los siguientes parámetros:
  * [nuevo] `campaign_daily_budget_amount_local_micro`
  * `currency`
  * `bid`
  * `objective`
* El objeto de respuesta ha cambiado y ahora devuelve rangos para los valores de la respuesta.
* `infinite_count` se ha renombrado como `infinite_bid_count` para evitar confusión acerca de su propósito
* Además de `count` e `infinite_bid_count`, ahora se devolverán los siguientes nuevos datos:
  * `impressions`
  * `engagements`
  * `estimated_daily_spend_local_micro`

* El `data_type` de Tailored Audiences se ha modificado de `tailored_audiences` a `tailored_audience` en todas nuestras respuestas.
* Las audiencias personalizadas compartidas ya están disponibles en versión beta *solo mediante API*. Las audiencias personalizadas compartidas permiten que una sola audiencia se use en varias cuentas publicitarias. Usa el endpoint [POST accounts/:account&#95;id/tailored&#95;audiences/:tailored&#95;audience&#95;id/permissions](/es/x-ads-api/audiences) (y otros relacionados) para administrar los permisos de una audiencia personalizada que deseas compartir entre cuentas publicitarias.
* Mejoras significativas en la forma de recopilar estadísticas de rendimiento para cuentas de anunciantes: [analytics](/es/x-ads-api/analytics)
* Para alinearnos con nuestras [mejores prácticas](/es/x-ads-api/analytics#best-practices), ahora solo permitiremos obtener hasta **7 días** de datos mediante los [endpoints síncronos de estadísticas](/es/x-ads-api/analytics#get-stats-accounts-account-id).
* Para simplificar la obtención de métricas, hemos reemplazado el parámetro `metrics` con el nuevo parámetro `metric_groups`. Los desarrolladores solo tienen que indicar qué grupos de métricas desean que se devuelvan para una solicitud determinada.
  * Cualquier solicitud de métricas que no sean apropiadas para una entidad determinada se excluirá de la respuesta y se representará como valores `null`. *Estas métricas no se contabilizarán para tu límite de coste de analítica.*
* La respuesta se ha **simplificado significativamente** y ahora se alinea más estrechamente con la forma en que las métricas se exponen en nuestra interfaz de usuario.
  * Anteriormente exponíamos una métrica separada para cada ubicación de publicación (Promoted Tweets in Search, Promoted Tweets in Timelines, Promoted Tweets in Profiles &amp; Tweet Details, X Audience Platform). Ahora devolveremos un conjunto estandarizado de métricas para cada una (en lugar de `promoted_tweet_timeline_impressions`, `promoted_tweet_search_impressions`, `promoted_tweets_profile_impressions`, `promoted_tweets_tpn_impressions`), que se expondrán, cuando se soliciten en una de las siguientes categorías, como una única métrica, `impressions` (esto se aplica a todas las métricas):
  * `ALL_ON_TWITTER`
  * `PUBLISHER_NETWORK`
  * Cuando realices una solicitud, obtendrás una única métrica `impressions` para que resulte más sencillo hacer coincidir los valores con nuestra IU.
  * Debes hacer dos consultas para obtener tanto los datos de `ALL_ON_TWITTER` como los de `PUBLISHER_NETWORK`, ya que no se pueden combinar.
* [Los endpoints asíncronos de estadísticas](/es/x-ads-api/analytics#asynchronous-analytics) ya están disponibles, gracias a los comentarios de nuestros desarrolladores.
  * Un nuevo conjunto de endpoints para solicitar estadísticas de forma *asíncrona*, ya sea para datos que no necesitas de inmediato o para extracciones de datos históricos.
  * Pon en cola un trabajo de estadísticas usando un nuevo endpoint único. Obtendremos los datos que has solicitado según lo permitan los recursos.
  * Puedes consultar un endpoint de estado del trabajo para determinar si los datos están disponibles.
  * Una vez que los datos estén disponibles, te proporcionaremos un id de recogida para que descargues la respuesta JSON, que reflejará la respuesta del endpoint síncrono.
  * Consulta hasta **90 días** de datos de hasta **20 entidades** en un solo trabajo.
* Consulta nuestra guía de migración de analytics v1, que incluye la correspondencia entre las métricas de v0 y las métricas de v1
* Mejoras en Sandbox * Ahora puedes [crear múltiples cuentas de anuncios de prueba](/es/x-ads-api/campaign-management#post-accounts) en el entorno de Sandbox. * Ahora puedes crear múltiples instrumentos de financiación para una cuenta de anuncios de prueba únicamente en el entorno de Sandbox. Esto te permite probar todos nuestros tipos de instrumentos de financiación. Anteriormente, solo estaba disponible una fuente de financiación `CREDIT_CARD` para realizar pruebas. * ¿Quieres probar una función beta? Ahora puedes activar o desactivar funciones para una cuenta en el entorno de Sandbox para ajustarse a tus necesidades de prueba.
