<div id="typical-response-structure">
  ## Estructura típica de la respuesta
</div>

Las respuestas correctas se indican con un código HTTP de la serie 200 y una carga útil en formato JSON que contiene los objetos solicitados, creados, modificados o eliminados, junto con una indicación de cómo el servidor interpreta tu solicitud.

Si realizas una solicitud correcta, recibirás como parte de tu respuesta un nodo `request` que reproduce tu solicitud.

*Ejemplo:* [GET accounts/abcdefg/campaigns?with&#95;deleted=true](/es/x-ads-api/campaign-management#campaigns)

```json
{
  /* los datos de su respuesta... */,
  "request": {
    "params": {
      "account_id": "abcdefg",
      "with_deleted": "true"
    }
  }
}
```

El campo `data` en las respuestas JSON contendrá los objetos específicos asociados con el recurso al que accedes. El formato del nodo `data` será un array JSON cuando la respuesta pueda contener uno o más resultados. Se devolverá como un hash JSON cuando solo sea posible un resultado en la respuesta. En algunos casos poco frecuentes, puedes ver una respuesta que normalmente incluiría una colección con un hash map en su lugar. En este caso, asume que el único hash map es un objeto del mismo tipo que se especifica en el campo `type`.

<div id="error-response-structure">
  ## Estructura de la respuesta de error
</div>

Las respuestas de error se devuelven con un código HTTP que no pertenece a la serie 200. Normalmente se adjunta una respuesta JSON, pero algunos errores responderán con distintos tipos de cuerpo de contenido. En estas circunstancias, cuando no se pueda analizar la estructura de la respuesta, considera que el significado principal del código HTTP tiene prioridad. Por ejemplo, es posible que ocasionalmente veas un HTTP 404 junto con una respuesta HTML. En este caso, puedes asumir con seguridad que el contenido no se puede encontrar (HTTP 404 significa “Not Found”).

Las respuestas de error típicas siguen una estructura similar a las respuestas correctas. La naturaleza del error se comunicará en un nodo `errors` de la respuesta. El nodo `errors/code` indicará un código de error constante en MAYÚSCULAS&#95;CON&#95;GUIONES&#95;BAJOS que puedes consumir de forma programática para tomar decisiones sobre cómo resolverlo. El nodo `errors/message` indicará una descripción (normalmente) legible por humanos del error en inglés. Se pueden adjuntar campos adicionales para indicar detalles más específicos sobre el error.

```json
{
  "errors": [
    {
      "parameter": "start_time",
      "details": "fecha no válida",
      "code": "INVALID_PARAMETER",
      "value": "",
      "message": "Se esperaba un valor de tiempo, se recibió \"\" para start_time"
    }
  ],
  "request": {
    "params": {
      "account_id": "hkk5"
    }
  }
}
```

En el ejemplo anterior, se hizo una solicitud a un endpoint de analíticas con un valor no válido para el parámetro `start_time`. El `errors/code` para solicitudes con parámetros no válidos es `INVALID_PARAMETER`.
