Para admitir un número máximo más alto de campañas y la obtención eficiente de todas las entidades asociadas a una cuenta, la Advertiser API ahora admite la paginación en muchos endpoints GET. El mecanismo de paginación es fácil de usar y muy similar a la paginación basada en cursores de la API REST, tal como se describe en [Uso de cursores para navegar por colecciones](https://developer.x.com/en/docs/x-api/v1/pagination).

<div id="getting-started">
  ## Primeros pasos
</div>

Para determinadas solicitudes GET, ahora aceptamos los parámetros de consulta `cursor` y `count`, que son opcionales. Si una respuesta contiene más de `count` entidades, estos endpoints devolverán ahora las primeras `count` entidades y una clave `next_cursor` en el JSON de la respuesta.

`GET https://ads-api.x.com/5/accounts/abc1/campaigns?count=50`

```json
{
  "data": \[...\],
  "next_cursor": "c-3yvu1pzhd3i7",
  "request": {...}
}
```

Para obtener el siguiente conjunto de respuestas, debes agregar `cursor` a tus parámetros de consulta:

`GET https://ads-api.x.com/5/accounts/abc1/campaigns?cursor=c-3yvu1pzhd3i7&count=50`

```json
{
  "data": \[...\],
  "next_cursor": "c-3w3zdyg8ywan",
  "request": {...}
}
```


<div id="going-deeper">
  ## Más en profundidad
</div>

Para la mayoría de los endpoints, el valor máximo de `count` es **1,000**, el mínimo es **1** y el valor predeterminado es **200**.

El valor proporcionado por `next_cursor` siempre es una cadena de caracteres y debe considerarse opaco; la implementación está sujeta a cambios. Si se devuelven menos entidades que `count` en la página actual del conjunto de resultados, el valor de `next_cursor` será `null`.

**Nota:** los endpoints de análisis no admiten este tipo de paginación. En los endpoints de estadísticas, la paginación se realiza especificando períodos de tiempo. Consulta la documentación de esos endpoints individuales para obtener más información.
