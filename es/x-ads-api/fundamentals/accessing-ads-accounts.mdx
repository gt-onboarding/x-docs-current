<div id="user-accounts-vs-ad-accounts">
  # Cuentas de usuario vs Cuentas publicitarias
</div>

Hay dos tipos diferentes de cuentas que intervienen en el uso de la Ads API: cuentas publicitarias y cuentas de usuario de X. A lo largo de la documentación de la Ads API, el término “cuenta” generalmente se refiere a la cuenta publicitaria.

* Las cuentas publicitarias se registran en **business.x.com** y se identifican en la API mediante account&#95;id. Las cuentas publicitarias se vinculan directamente a fuentes de financiación y utilizan contenido de una o más cuentas de usuario de X como “usuarios promocionables”. Cada cuenta publicitaria puede otorgar permiso a una o más cuentas de usuario de X. La cuenta publicitaria, o “cuenta actual”, se representa en casi todas las URL que se ejecutan como un parámetro en línea :account&#95;id.
* Las cuentas de usuario de X (como @AdsAPI) se identifican por user&#95;id en la Ads API. Una o más de estas cuentas se pueden asociar con una cuenta publicitaria. La cuenta de usuario de X autenticada que realiza solicitudes en la API se denomina “usuario actual”. Un listado de las cuentas publicitarias a las que el usuario actual tiene acceso se puede obtener con [GET accounts](/es/x-ads-api/campaign-management#accounts). Los “usuarios promocionables” son identificadores (handles) de X que pueden ser promovidos por una cuenta publicitaria específica. Para obtener más detalles sobre esto, consulta [Obtaining Ads Account Access](/es/x-ads-api/introduction).

<div id="methods-for-ad-account-access">
  ## Métodos para acceder a cuentas de anuncios
</div>

Hay dos métodos que puedes usar para realizar solicitudes a la Ads API para la cuenta de un anunciante:

1. Realizar solicitudes en nombre de un anunciante (**recomendado**)
2. Realizar solicitudes usando tu cuenta a la que se le ha concedido acceso a la cuenta de un anunciante, por ejemplo, una [agencia](https://business.x.com/en/blog/agencies-manage-multiple-Twitter-Ads-accounts.html) que gestiona varias cuentas.

Este documento es una breve descripción de las diferencias entre estas opciones y debe utilizarse junto con nuestros otros recursos, como las [preguntas frecuentes sobre inicio de sesión multiusuario](https://business.x.com/en/help/troubleshooting/multi-user-login-faq.html). 

Como se describe en [Autorizar una solicitud](/es/resources/fundamentals/authentication#authorizing-a-request#authorizing-a-request), todas las solicitudes a la Ads API requieren encabezados de autorización usando [OAuth 1.0a](/es/resources/fundamentals/authentication) con un Token de acceso obtenido mediante el flujo [de 3 patas](/es/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow)[OAuth](/es/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow). Tu aplicación deberá implementar un flujo OAuth basado en la web para [obtener tokens de acceso](/es/resources/fundamentals/authentication#oauth-1-0a-2). Los desarrolladores de la Ads API nunca deben solicitar a nuestros anunciantes de X que compartan sus credenciales de inicio de sesión.

De forma predeterminada, cada [aplicación de desarrollador de X](/es/resources/fundamentals/developer-apps) contiene un **Token de acceso** estático que puede usarse para realizar solicitudes a la Ads API para la cuenta que es propietaria de la aplicación. Estas credenciales son ideales para casos de uso de una sola cuenta sin requerir un flujo OAuth de 3 patas o [basado en PIN](/es/resources/fundamentals/authentication#pin-based-authorization). Si no vas a acceder a otra cuenta de anuncios de X, usa estas credenciales de [usuario único](/es/resources/fundamentals/authentication#oauth-1-0a-2) en lugar de los pasos siguientes.

<div id="levels-of-access">
  ## Niveles de acceso
</div>

<div id="app-level-permissions">
  ### Permisos a nivel de aplicación
</div>

Cada usuario tendrá un nivel de acceso según lo solicitado en la [solicitud de acceso a la Ads API](/es/x-ads-api/getting-started/step-by-step-guide):

* **Solo conversiones**: El usuario tiene acceso a los endpoints de [Mobile](/es/x-ads-api/measurement/mobile-conversions) y [Web Conversion](/es/x-ads-api/measurement/web-conversions) con acceso de [lectura y escritura](/es/resources/fundamentals/developer-apps).
* **Acceso estándar**: El usuario tiene acceso a los endpoints de [Analytics](/es/x-ads-api/analytics), [Campaign Management](https://developer.x.com/en/docs/twitter-ads-api/campaign-management/overview), [Creatives](https://developer.x.com/en/docs/twitter-ads-api/creatives/overview), [Custom Audiences](https://developer.x.com/en/docs/twitter-ads-api/audiences/overview) y [Conversion](/es/x-ads-api/measurement/web-conversions) con acceso de [lectura y escritura](/es/resources/fundamentals/developer-apps).

**Nota:** Los desarrolladores de Ads API que solicitaron acceso antes de julio de 2023 pueden tener distintos niveles de acceso y permisos, y pueden estar limitados a cinco tokens OAuth. Consulta nuestra guía sobre [aumentar el acceso](/es/x-ads-api/getting-started/increasing-access) para obtener acceso a endpoints adicionales o levantar los límites de tokens para aplicaciones existentes.

<div id="ad-account-level-permissions">
  ### Permisos a nivel de cuenta de anuncios
</div>

Cada usuario que tenga acceso a una cuenta de anuncios tendrá un permiso específico a nivel de cuenta: **Administrador de cuenta**, **Administrador de anuncios**, **Analista de campañas**, **Analista orgánico** y **Administrador creativo**; consulta **business.x.com** para ver la documentación más reciente sobre los permisos a nivel de cuenta. Las aplicaciones deben obtener los permisos del usuario autenticado actualmente mediante el endpoint de API [Authenticated User Access](/es/x-ads-api/campaign-management#authenticated-user-access) para determinar a qué endpoints de API y funciones de anuncios pueden acceder.

**Nota:** Cualquier token de usuario utilizado con la [Conversion API](/es/x-ads-api/measurement/web-conversions) debe pertenecer a usuarios con permisos a nivel de cuenta de **Administrador de cuenta** o **Administrador de anuncios**.                                                                                                                                              

<div id="methods-of-obtaining-access-tokens">
  ## Métodos para obtener tokens de acceso
</div>

<div id="1-obtain-an-advertisers-user-access-token">
  ### 1. Obtén el token de acceso (de usuario) de un anunciante
</div>

Hay dos métodos para obtener el **token de acceso** de un anunciante. El método más común es mediante un [flujo OAuth de tres patas](/es/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) directamente desde la interfaz web de tu aplicación. Las aplicaciones que no tienen una interfaz pública accesible para los anunciantes pueden implementar un proceso OAuth [basado en PIN](/es/resources/fundamentals/authentication#pin-based-authorization). Después de que el usuario complete el flujo de tres patas, tu aplicación tendrá credenciales para realizar solicitudes a su cuenta de anuncios mediante la API.

Obtener credenciales de usuario mediante un flujo OAuth es el método que **recomendamos encarecidamente** para que la mayoría de los desarrolladores de Ads API obtengan acceso a una cuenta de anunciante. Te permite llamar a la API en nombre de un usuario y realizar acciones como ese usuario. Estos tokens no caducan, pero pueden ser revocados por el usuario en cualquier momento.

<div id="2-obtain-your-developer-access-token">
  ### 2. Obtén tu token de acceso (de desarrollador)
</div>

Esta opción requiere que el anunciante [conceda acceso a tu @username](https://business.x.com/en/help/troubleshooting/multi-user-login-faq.html) (o a varios @usernames) a su cuenta de X Ads a través de la interfaz de X en business.x.com. Los **tokens de acceso** obtenidos mediante el flujo OAuth de tres patas para tu cuenta podrán acceder a la cuenta de X Ads del anunciante.

Esto te permite llamar a la API usando los tokens OAuth de tu propio @username en lugar de los tokens OAuth del anunciante. La diferencia clave en esta opción es que solo podrás crear Publicaciones solo promocionadas si se ha otorgado a tu @username el permiso de delegación/composición de Publicaciones.

Para poder crear Publicaciones solo promocionadas en nombre del usuario promocionable `FULL` de la cuenta, también se te debe otorgar acceso para crear Publicaciones en este flujo. Esto habilitará el acceso mediante el permiso `TWEET_COMPOSER` en el endpoint [GET accounts/:account&#95;id/authenticated&#95;user&#95;access](/es/x-ads-api/campaign-management#accounts).

<div id="differences-between-these-methods">
  ## Diferencias entre estos métodos
</div>

|     |     |     |
| :--- | :--- | :--- |
|     | Token OAuth de anunciante (usuario) | Token OAuth (de desarrollador)  <br /> (@username añadido para separar la cuenta) |
| Access Ads Account | ✔   | ✔   |
| Create Posts on Behalf of User | ✔   | ✔*  |
| Manage Campaigns | ✔   | ✔   |
| Access Analytics | ✔   | ✔   |
| Create Cards on Behalf of User | ✔   | ✔   |
| Developer ability to Access  <br />via X Ads UI |     | ✔   |
| [Límites de frecuencia](/es/x-api/enterprise-gnip-2.0/fundamentals/rate-limits) | Independientes por anunciante | Independientes por cuenta de anunciante |

<div id="advertisers-access-token-via-oauth-3-legged-web-flow">
  ### Token de acceso del anunciante mediante flujo web OAuth de 3 participantes

Las capturas de pantalla descritas aquí forman parte de un ejemplo cuyo código fuente puedes ver en [https://github.com/xdevplatform/twauth-web](https://github.com/xdevplatform/twauth-web).

En algún punto de tu aplicación, querrás redirigir a X para autorizar tu aplicación.
