<div id="user-accounts-vs-ad-accounts">
  # Cuentas de usuario vs cuentas de anuncios
</div>

Hay dos tipos diferentes de cuentas que intervienen en el uso de la Ads API: cuentas de anuncios y cuentas de usuario de X. A lo largo de la documentación de la Ads API, el término “account” normalmente se refiere a la cuenta de anuncios.

* Las cuentas de anuncios se registran en **business.x.com** y se identifican en la API mediante account&#95;id. Las cuentas de anuncios se vinculan directamente a fuentes de financiación y utilizan contenido de una o más cuentas de usuario de X como “usuarios promocionables”. Cada cuenta de anuncios puede otorgar permisos a una o más cuentas de usuario de X. La cuenta de anuncios, o “cuenta actual”, está representada en casi todas las URL que se ejecutan como un parámetro en línea :account&#95;id.
* Las cuentas de usuario de X (como @AdsAPI) se identifican por user&#95;id en la Ads API. Una o más de estas cuentas se pueden asociar con una cuenta de anuncios. La cuenta de usuario de X autenticada que realiza solicitudes en la API se denomina “usuario actual”. Se puede encontrar un listado de las cuentas de anuncios a las que el usuario actual tiene acceso con [GET accounts](/es/x-ads-api/campaign-management#accounts). Los “usuarios promocionables” son identificadores de X (handles) que pueden ser promocionados por una cuenta de anuncios específica. Para más detalles sobre esto, consulta [Obtaining Ads Account Access](/es/x-ads-api/introduction).

<div id="methods-for-ad-account-access">
  ## Métodos para acceder a la cuenta de Ads
</div>

Hay dos métodos que puedes usar para realizar solicitudes a la Ads API para la cuenta de un anunciante:

1. Realizar solicitudes en nombre de un anunciante (**recomendado**)
2. Realizar solicitudes usando tu cuenta a la que se le ha otorgado acceso a la cuenta de un anunciante, por ejemplo, una [agencia](https://business.x.com/en/blog/agencies-manage-multiple-Twitter-Ads-accounts.html) que administra varias cuentas.

Este documento es una breve descripción general de las diferencias entre estas opciones y debe utilizarse junto con nuestros otros recursos, como las [preguntas frecuentes sobre el inicio de sesión multiusuario](https://business.x.com/en/help/troubleshooting/multi-user-login-faq.html). 

Como se describe en [Autorizar una solicitud](/es/resources/fundamentals/authentication#authorizing-a-request#authorizing-a-request), todas las solicitudes a la Ads API requieren encabezados de autorización que usen [OAuth 1.0a](/es/resources/fundamentals/authentication) con un token de acceso obtenido mediante el flujo [OAuth](/es/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) de [3-legged](/es/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow). Tu aplicación deberá implementar un flujo OAuth basado en la web para [obtener tokens de acceso](/es/resources/fundamentals/authentication#oauth-1-0a-2). Los desarrolladores de la Ads API nunca deben solicitar que los anunciantes de X compartan sus credenciales de inicio de sesión.

De forma predeterminada, cada [aplicación de desarrollador de X](/es/resources/fundamentals/developer-apps) contiene un **token de acceso** estático que puede utilizarse para realizar solicitudes a la Ads API para la cuenta que es propietaria de la aplicación. Estas credenciales son ideales para casos de uso de una sola cuenta sin requerir un flujo OAuth de 3-legged o basado en [PIN](/es/resources/fundamentals/authentication#pin-based-authorization). Si no estás accediendo a otra cuenta de X Ads, usa estas credenciales de [usuario único](/es/resources/fundamentals/authentication#oauth-1-0a-2) en lugar de los siguientes pasos.

<div id="levels-of-access">
  ## Niveles de acceso
</div>

<div id="app-level-permissions">
  ### Permisos a nivel de aplicación
</div>

Cada usuario tendrá un nivel de acceso según lo solicitado en la [solicitud de acceso a la Ads API](/es/x-ads-api/getting-started/step-by-step-guide):

* **Solo conversiones**: El usuario tiene acceso a los endpoints de [Mobile](/es/x-ads-api/measurement/mobile-conversions) y [Web Conversion](/es/x-ads-api/measurement/web-conversions) con acceso de [lectura y escritura](/es/resources/fundamentals/developer-apps).
* **Acceso estándar**: El usuario tiene acceso a los endpoints de [Analytics](/es/x-ads-api/analytics), [Campaign Management](https://developer.x.com/en/docs/twitter-ads-api/campaign-management/overview), [Creatives](https://developer.x.com/en/docs/twitter-ads-api/creatives/overview), [Custom Audiences](https://developer.x.com/en/docs/twitter-ads-api/audiences/overview) y [Conversion](/es/x-ads-api/measurement/web-conversions) con acceso de [lectura y escritura](/es/resources/fundamentals/developer-apps).

**Nota:** Los desarrolladores de Ads API que solicitaron acceso antes de julio de 2023 pueden tener distintos niveles de acceso y permisos, y pueden estar limitados a cinco tokens OAuth. Consulta nuestra guía sobre [aumentar el acceso](/es/x-ads-api/getting-started/increasing-access) para obtener acceso a endpoints adicionales o eliminar los límites de tokens de las aplicaciones existentes.

<div id="ad-account-level-permissions">
  ### Permisos a nivel de cuenta de Ads
</div>

Cada usuario que tenga acceso a una Ad Account tendrá un permiso específico a nivel de cuenta: **Account administrator**, **Ad manager**, **Campaign analyst**, **Organic analyst** y **Creative Manager**; consulta **business.x.com** para la documentación más reciente sobre los permisos a nivel de cuenta. Las aplicaciones deben obtener los permisos del usuario autenticado actualmente mediante el endpoint de la API [Authenticated User Access](/es/x-ads-api/campaign-management#authenticated-user-access) para determinar a qué endpoints de la API y a qué funciones de Ads pueden acceder.

**Nota:** Cualquier token de usuario utilizado con la [Conversion API](/es/x-ads-api/measurement/web-conversions) debe corresponder a usuarios con permisos a nivel de cuenta de **Account administrator** o **Ad manager**.                                                                                                                                              

<div id="methods-of-obtaining-access-tokens">
  ## Métodos para obtener tokens de acceso
</div>

<div id="1-obtain-an-advertisers-user-access-token">
  ### 1. Obtén el token de acceso de anunciante (Usuario)
</div>

Hay dos métodos para obtener el **token de acceso** de un anunciante. El método más común es a través de un [flujo OAuth de tres patas](/es/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) directamente desde tu interfaz web. Las aplicaciones que no tienen una interfaz pública accesible para los anunciantes pueden implementar un proceso de OAuth [basado en PIN](/es/resources/fundamentals/authentication#pin-based-authorization). Después de que el usuario complete el flujo de tres patas, tu aplicación tendrá credenciales para realizar solicitudes a su cuenta de Ads a través de la API.

Obtener credenciales de usuario mediante un flujo OAuth es el método que **recomendamos encarecidamente** a la mayoría de los desarrolladores de la Ads API para obtener acceso a la cuenta de un anunciante. Te permite llamar a la API en nombre de un usuario y realizar acciones como ese usuario. Estos tokens no caducan, pero el usuario puede revocarlos en cualquier momento.

<div id="2-obtain-your-developer-access-token">
  ### 2. Obtén tu token de acceso (de desarrollador)
</div>

Esta opción requiere que el anunciante [conceda acceso a tu @username](https://business.x.com/en/help/troubleshooting/multi-user-login-faq.html) (o @usernames) a su cuenta de X Ads a través de la interfaz de X en business.x.com. Los **tokens de acceso** obtenidos mediante el flujo OAuth de tres patas para tu cuenta podrán acceder a la cuenta de X Ads del anunciante.

Esto te permite realizar llamadas a la API usando los tokens OAuth de tu propio @username en lugar de los tokens OAuth del anunciante. La diferencia clave de esta opción es que solo puedes crear Publicaciones solo promocionadas si se te ha otorgado a tu @username el permiso de delegación/composición de Publicaciones.

Para obtener acceso para crear Publicaciones solo promocionadas en nombre del usuario promocionable `FULL` de la cuenta, también se te debe conceder acceso para crear Publicaciones en este flujo. Esto habilitará el acceso mediante el permiso `TWEET_COMPOSER` en el endpoint [GET accounts/:account&#95;id/authenticated&#95;user&#95;access](/es/x-ads-api/campaign-management#accounts).

<div id="differences-between-these-methods">
  ## Diferencias entre estos métodos
</div>

|     |     |     |
| :--- | :--- | :--- |
|     | Token OAuth del anunciante (usuario) | Token OAuth del desarrollador  <br /> (con @username añadido para separar la cuenta) |
| Acceder a la cuenta de X Ads | ✔   | ✔   |
| Crear Publicaciones en nombre del usuario | ✔   | ✔*  |
| Administrar campañas | ✔   | ✔   |
| Acceder a Analytics | ✔   | ✔   |
| Crear Cards en nombre del usuario | ✔   | ✔   |
| Capacidad del desarrollador para acceder  <br />a través de la interfaz de usuario de X Ads |     | ✔   |
| [Límites de frecuencia](/es/x-api/enterprise-gnip-2.0/fundamentals/rate-limits) | Independientes por anunciante | Independientes por cuenta de anunciante |

<div id="advertisers-access-token-via-oauth-3-legged-web-flow">
  ### Token de acceso del anunciante mediante el flujo web OAuth de 3 partes
</div>

El flujo estándar es web y utiliza el flujo OAuth de [autorización de 3 partes](/es/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow). Las capturas de pantalla descritas aquí forman parte de un ejemplo cuyo código fuente puedes consultar en [https://github.com/xdevplatform/twauth-web](https://github.com/xdevplatform/twauth-web).

En algún punto de tu aplicación, querrás redirigir a X para autorizar tu aplicación.
