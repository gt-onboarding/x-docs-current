<div id="ads-api-access-new-applications">
  #### **Acceso a la Ads API - Nuevas aplicaciones**
</div>

* El requisito principal para usar Conversion API es contar con una [cuenta de desarrollador](https://developer.x.com/en/portal/dashboard) y [Ads API Access](https://developer.x.com/en/docs/twitter-ads-api/getting-started). El proceso se describe en la guía [Ads API Getting Started](https://developer.x.com/en/docs/twitter-ads-api/getting-started). Ten en cuenta lo siguiente:

**Paso 1: Cuenta de desarrollador**

* Al solicitar una [cuenta de desarrollador](https://developer.x.com/en/portal/dashboard), solicita uno de nuestros [planes](https://developer.x.com/en/portal/products/) de suscripción para obtener aprobación instantánea. 
* **Nota**: Como práctica recomendada, recomendamos encarecidamente usar la cuenta oficial de X de tu empresa para crear una cuenta de desarrollador y solicitar acceso a la Ads API. Si la cuenta de desarrollador está asociada a una cuenta personal de desarrollador, no es posible transferir esas credenciales si fuera necesario. Es mejor que se gestione desde una cuenta de empresa para facilitar la administración continua y utilizar [Multi-user login](https://business.x.com/en/help/troubleshooting/multi-user-login-faq.html) según sea necesario. De lo contrario, como mínimo, la cuenta debe configurarse con opciones personalizadas (imagen de encabezado, avatar, descripción de la biografía y URL de la biografía) y usar autenticación de dos factores.

**Paso 2: Solicitud de acceso a la Ads API**

* Asegúrate de tener listo el **App** **ID** correcto para tu solicitud de acceso a la Ads API. El App ID se puede encontrar en el [Developer Portal](https://developer.x.com/en/portal/dashboard) en Projects &amp; Apps. Ejemplo: 16489123
* Solicita acceso a la Ads API poniéndote en contacto con tu representante de X.

<div id="ads-api-access-existing-applications">
  #### **Acceso a Ads API: aplicaciones existentes**
</div>

* Si ya cuentas con una aplicación de Ads API en uso activo, tanto la aplicación como los tokens de acceso existentes pueden utilizarse para Conversion API.

<div id="access-tokens">
  #### **Tokens de acceso**
</div>

* Los Tokens de acceso de usuario para el handle del usuario propietario de la aplicación de Ads API se pueden generar y obtener directamente desde el portal para desarrolladores. A esto se le llama tu “token de acceso personal” porque está pensado para utilizarse con tu propio handle de X. Puedes encontrar información general sobre autenticación y el portal para desarrolladores [aquí](/es/resources/fundamentals/authentication/overview).
* Los Tokens de acceso de usuario para handles de usuario distintos del handle propietario de la aplicación de Ads API deben generarse con un flujo OAuth de tres patas (3-legged OAuth flow). Las opciones para generar el Token de acceso con un flujo OAuth de tres patas incluyen:
  * [Línea de comandos con autorización basada en la web mediante la utilidad twurl](https://github.com/twitter/twurl#getting-started)
  * [Línea de comandos con autorización basada en PIN](https://developer.x.com/resources/fundamentals/authentication/pin-based-oauth)
  * [Flujo web personalizado que implemente el patrón de OAuth de tres patas](https://developer.x.com/resources/fundamentals/authentication/obtaining-user-access-tokens)
* Cualquier token de usuario que se use con Conversion API debe pertenecer a usuarios con nivel de acceso AD&#95;MANAGER o ACCOUNT&#95;ADMIN, lo cual se puede comprobar mediante el endpoint [authenticated&#95;user&#95;access](https://developer.x.com/en/docs/twitter-ads-api/campaign-management/api-reference/authenticated-user-access).
* **Nota:** los propios tokens (una vez creados según lo anterior) se pueden compartir con usuarios que no tengan nivel de acceso AD&#95;MANAGER o ACCOUNT&#95;ADMIN para que los utilicen.

<div id="creating-the-conversion-api-event">
  ### Crear el evento de Conversion API
</div>

Para usar Conversion API, debes crear un nuevo evento de conversión en Ads Manager o usar un evento existente que ya hayas creado y utilizado con X Pixel. Si deseas realizar la deduplicación entre los eventos de Pixel y Conversion API, debes usar el evento existente que creaste para Pixel. 

<div id="option-1-using-an-existing-conversion-event-in-ads-manager">
  ##### Opción 1: Usar un evento de conversión existente en Ads Manager
</div>

Si deseas usar un evento existente que ya estás utilizando con el píxel de X, es posible y deberás tomar el ID de evento de ese evento. Si utilizas tanto el píxel como Conversion API para el mismo evento, asegúrate de usar la clave de desduplicación tanto en el fragmento de código del píxel como en la solicitud de Conversion API (como `conversion_id`) para desduplicar eventos entre el píxel y Conversion API para el mismo evento. Consulta la sección d. Testing Events and Deduplication para obtener más información. 

<div id="option-2-creating-a-new-conversion-event-in-ads-manager">
  ##### Opción 2: Crear un nuevo evento de conversión en Ads Manager:
</div>

Es importante tener una fuente de eventos (Event Source) creada en Events Manager antes de crear un evento. Para comprobar si tienes una fuente de eventos (X Pixel) añadida a tu cuenta, ve a Events Manager y verifica si tienes X Pixel en el menú de la izquierda. 

Si todavía no tienes una fuente de eventos añadida, sigue los pasos a continuación para crear una fuente de eventos.

1. Ve a ads.x.com
2. Dirígete a la sección **Tools** en la parte superior izquierda y haz clic en **Events Manager**
3. Selecciona Add event source en la parte superior derecha para **Add an event source** si aún no tienes una fuente de eventos X Pixel añadida en la barra lateral izquierda
   1. El id de la fuente de eventos X Pixel es tu **Pixel ID**

Ahora que tienes una fuente de eventos y un Pixel ID, necesitas crear un evento dentro de esa fuente de eventos para los eventos de conversión que deseas registrar:

1. Dentro de la fuente de eventos X Pixel, selecciona **Add events** en el lado derecho
2. Selecciona Install with **Conversion API**
3. Verás el **Pixel ID** y el **Event ID** de este evento que se usarán en la API
   1. El id del evento es tu Event ID
4. Haz clic en Save y tendrás tu evento de conversión creado y listo para usar

<div id="preparing-identifiers-for-conversion-events">
  ### Preparación de identificadores para eventos de conversión
</div>

Actualmente necesitamos que se envíe al menos un identificador, como el X Click ID (`twclid`), la dirección de correo electrónico o el número de teléfono. Si se utiliza la dirección IP o el user agent, se debe enviar un segundo identificador para lograr una coincidencia de conversiones adecuada.

Enviar más identificadores generará una tasa de coincidencia de conversiones más alta.

|     |     |     |
| :--- | :--- | :--- |
| Campo de coincidencia de cliente | Formato | ¿Se requiere hashing? |
| X Click ID | Generado por X ([más información](https://business.x.com/en/help/campaign-measurement-and-analytics/conversion-tracking-for-websites.html#faq)) | No  |
| Dirección de correo electrónico | Eliminar espacios iniciales y finales | Requerido (SHA256) |
| Número de teléfono | [Estándar E164](https://en.wikipedia.org/wiki/E.164) | Requerido (SHA256) |

|     |     |     |
| :--- | :--- | :--- |
| Dirección IP | Eliminar espacios iniciales y finales | No  |
| User Agent | Eliminar espacios iniciales y finales | No  |

<div id="1-prepare-x-click-id-identifier">
  #### 1. Preparar el identificador X Click ID
</div>

Se recomienda incluir siempre el Click ID en la solicitud de conversión.

El Click ID debe extraerse del parámetro de la cadena de consulta `twclid` cuando esté disponible, después de que el usuario llegue al sitio web de destino. 

Ejemplo básico de código JavaScript:

```js
var queryString = document.location.search;
if (queryString.has('twclid') {
  twitterClickID = getParam(queryString, 'twclid');
  // Pasos recomendados: Registro e inserción en almacenamiento local
}
```

Se recomienda: 

1. Extraer siempre el valor de `twclid` cuando esté presente en los parámetros de consulta de la URL.

2. Almacenar los datos junto con los campos de formulario pertinentes o la información del evento de conversión.

Vincular el Click ID con los eventos de conversión y la información del flujo de trabajo permite casos de uso como el procesamiento por lotes, algoritmos para analizar y crear eventos de conversión basados en múltiples flujos de navegación del sitio web y cargas masivas.

La Event Source URL debe estar [codificada como URL](https://en.wikipedia.org/wiki/Percent-encoding) y está pensada para representar la página web que desencadenó el evento.

<div id="2-prepare-email-identifier">
  #### 2. Preparar identificador de correo electrónico
</div>

Los campos admitidos para la coincidencia de clientes se pueden enviar, pero deben normalizarse y, cuando sea necesario, aplicarles una función hash para proteger la privacidad.

La información debe someterse a hashing usando SHA256, sin salt. 

Por ejemplo, para una dirección de correo electrónico test@x.com, debe enviarse en formato con hash: d360d510a224510f373931ce2d6215a799f5a9c1cef221b0149b6b6b50cced62.

<div id="3-prepare-phone-identifier">
  #### 3. Preparar el identificador de teléfono
</div>

El número de teléfono debe pasarse utilizando el [estándar E.164](https://en.wikipedia.org/wiki/E.164) y se le debe aplicar un hash con SHA256, sin salt. 

Por ejemplo, para un número de teléfono de EE. UU.: +11234567890, debe enviarse en formato de hash: 1fa6b8d986d9b9cd01bf36951815158bbde9f520c0567c835dfe34783d0a4231.

<div id="4-prepare-ip-address-identifier">
  #### 4. Preparar el identificador de dirección IP
</div>

Es obligatorio enviar la dirección IP junto con otro identificador (twclid, dirección de correo electrónico, número de teléfono o user agent). No es necesario aplicar hashing a este identificador.

Este valor se escribe en notación decimal punteada, con cuatro números separados por puntos. Por ejemplo, la dirección IP de un usuario puede ser 8.25.197.25.

<div id="5-prepare-user-agent-identifier">
  #### 5. Preparar identificador de User Agent
</div>

El User Agent debe enviarse junto con otro identificador (twclid, dirección de correo electrónico, número de teléfono o dirección IP). No es necesario aplicar hashing a este identificador.

Este identificador permite que el servidor identifique la aplicación, el sistema operativo, el proveedor y/o la versión del User Agent que realiza la solicitud. Por ejemplo, este valor se puede enviar como Mozilla/5.0 (Macintosh; Intel Mac OS X 10&#95;15&#95;7) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/127.0.0.0 Safari/537.36.

<div id="constructing-the-conversion-event-request">
  ### Creación de la solicitud de evento de conversión
</div>

`POST: version/measurement/conversions/:pixel_id`

Envía eventos de conversión para una cuenta de anuncios específica. Se debe comprobar el código de respuesta para verificar que se haya procesado correctamente (HTTP 200 OK). Se recomienda contar con un mecanismo de reintento y un registro básico en caso de que se devuelvan códigos de error.

Para obtener información detallada sobre la URL del endpoint y los parámetros del cuerpo de la solicitud POST, consulta la sección de [Referencia de la API](https://developer.x.com/en/docs/twitter-ads-api/measurement/api-reference/conversions).

<div id="rate-limit">
  ### Límite de frecuencia
</div>

El límite de frecuencia será de 60.000 eventos por cuenta, por intervalo de 15 minutos.

Ten en cuenta que es posible que tu código de servidor deba implementar lógica fuera de esta llamada, incluida:

1. La instrumentación de las acciones de los usuarios (registro/logging) para poder enviar datos de conversión correctos por evento

2. Cualquier lógica necesaria para filtrar eventos de conversión de usuarios que hayan ejercido opciones de privacidad relevantes; por ejemplo, si han optado por no ser rastreados o por no permitir la venta de su información personal en el sitio web del anunciante

3. La integración con desencadenadores de eventos y páginas a fin de capturar eventos y enviar conversiones

<div id="duplication-between-pixel-and-conversion-api">
  ### Duplicación entre Pixel y Conversion API
</div>

Si quieres evitar duplicar conversiones entre solicitudes de Pixel y de Conversion API, contamos con `conversion_id` como clave de desduplicación. La desduplicación solo ocurre a nivel de evento. En otras palabras, para desduplicar entre solicitudes de Pixel y CAPI, un anunciante tiene que usar el mismo evento tanto en las solicitudes de Pixel como en las de CAPI, además de usar el mismo `conversion_id`. La desduplicación solo puede aplicarse a eventos que se reciban dentro de una ventana de 48 horas.

<div id="conversion-tracking-overview">
  ## Seguimiento de conversiones (Descripción general)
</div>

<div id="summary">
  ### Resumen
</div>

El seguimiento de conversiones permite medir la cantidad de usuarios de X que realizan una acción deseada después de ver e interactuar con tus anuncios en X. Te brinda la capacidad de medir qué campañas generan acciones como visitas al sitio web, registros y compras. Esto ofrece a los anunciantes capacidades de medición fuera de X para comprender el rendimiento de sus anuncios de respuesta directa y así poder adquirir clientes de forma rentable.

Mediante una etiqueta de conversión, los anunciantes pueden rastrear las conversiones de los usuarios y asociarlas a las campañas publicitarias en X. Esto les brinda la visibilidad necesaria para optimizar sus campañas y cumplir sus objetivos de costo por adquisición (CPA).

Hay una variedad de acciones en sitios web que un anunciante puede medir con el seguimiento de conversiones. Puede seleccionar una o más, de acuerdo con la(s) acción(es) que busca impulsar con su campaña publicitaria:

* Visita al sitio: el usuario visita una página de destino en el sitio web del anunciante
* Compra: el usuario completa la compra de un producto o servicio en el sitio web del anunciante
* Descarga: el usuario descarga un archivo, como un documento técnico o un paquete de software, desde el sitio web del anunciante
* Registro: el usuario se registra en el servicio del anunciante, en un boletín informativo o en comunicaciones por correo electrónico
* Personalizada: esta es una categoría general para una acción personalizada que no entra en ninguna de las categorías anteriores

El seguimiento de conversiones de X ofrece a los anunciantes una visión completa de la atribución de conversiones. En comparación con los sistemas de medición de terceros que los clientes pueden haber estado utilizando en lugar de la función de seguimiento de conversiones propia de X, como URL de clic únicas combinadas con etiquetas de seguimiento de terceros, la etiqueta de conversión de X ofrece a los anunciantes la capacidad de rastrear conversiones atribuidas a interacciones de la parte media y alta del embudo, como expansiones de Publicaciones, Reposts, Me gusta, respuestas y follows, así como impresiones.

<div id="faq">
  ### Preguntas frecuentes
</div>

<AccordionGroup>
  <Accordion title="¿Cómo funciona la etiqueta de seguimiento de conversiones?">
    Primero, un anunciante crea una etiqueta de conversión, que es un fragmento de código proporcionado por X, y la coloca en su sitio web. La etiqueta ya está lista para medir la conversión cuando un usuario completa la acción indicada.

    Luego, los usuarios ven el anuncio del anunciante en la aplicación de X, lo que los lleva al sitio web del anunciante y a la acción que se haya etiquetado. Si el usuario completa esa acción durante la(s) ventana(s) de atribución especificada(s) por los anunciantes durante la configuración de la etiqueta, la etiqueta reconoce que el usuario ha interactuado previamente con un anuncio de X. La etiqueta entonces se “activa” o envía una notificación a los servidores de X para que la conversión pueda atribuirse al anuncio que generó la conversión.
  </Accordion>

  <Accordion title="¿Existe alguna forma dentro del proceso de configuración de la campaña que permita al usuario seleccionar qué píxeles de seguimiento son relevantes para esa campaña?">
    No, nuestro producto no está configurado para adjuntar etiquetas de conversión específicas a campañas específicas. En su lugar, una vez que se configura una etiqueta, el sistema realiza automáticamente el seguimiento de qué anuncio generó conversiones en una determinada etiqueta.
  </Accordion>

  <Accordion title="¿Cuáles son nuestras configuraciones predeterminadas de ventana de atribución para etiquetas de conversión?">
    Ventana de atribución post-view predeterminada: 1 día

    Atribución post-engagement predeterminada: 14 días

    Estos valores predeterminados se pueden cambiar durante la configuración de la etiqueta de conversión o en cualquier momento después de crear la etiqueta. Las opciones para las ventanas de atribución post-engagement son 1, 7, 14, 30, 60 y 90 días. Las opciones para las ventanas de atribución post-view son ninguna, 1, 7, 14, 30, 60 y 90 días.
  </Accordion>

  <Accordion title="¿Cuáles son algunas ideas para creatividades y estrategias de DR eficaces que impulsen conversiones de forma efectiva?">
    Si bien los objetivos, la situación y las estrategias de cada cliente son diferentes, aquí hay algunas ideas que han funcionado para clientes que participaron en las fases alfa o beta del seguimiento de conversiones:

    **Creatividad:**

    * **Ofertas**: Combine un descuento, una promoción o una oferta de envío gratuito con la Publicación promocionada para generar más interés en la acción.
    * **Sorteos y concursos**: Especialmente para marcas reconocidas, los sorteos y concursos generaron conversiones.
    * **Experimentación con el texto de la Publicación**: Probar mayúsculas completas frente a minúsculas (FREE vs free o NOW vs now).
    * **Plazos**: Ofrezca una fecha límite para incentivar a las personas a realizar una acción inmediata (¡La oferta vence el 12 de diciembre!).
    * **Añadir fotos atractivas**: Vale la pena probar si las fotos visualmente atractivas en la creatividad de la Publicación son eficaces para impulsar conversiones; los resultados pueden variar o ser específicos de la oferta del cliente.

    **Segmentación:**

    * Segmentación por @handle y por categoría de intereses: Una alineación precisa entre el texto de la Publicación y los @handles con la audiencia prevista de la Publicación generó conversiones.
    * Uso de palabras clave de nicho pero de alto volumen: En el ámbito de los conciertos, usar palabras clave relacionadas con el artista/músico (por ejemplo, su nombre) resultó eficaz.
    * Tailored audiences: Los clientes que usaron TA web y seguimiento de conversiones juntos obtuvieron CPA más bajos que los grupos de control que usaban otros tipos de segmentación.

    Cuanto más granular sea la segmentación de su campaña, más accionables serán los resultados de conversión que se informen. Por ejemplo, es mucho más fácil optimizar una campaña con 4 palabras clave que optimizar una campaña con 50.
  </Accordion>
</AccordionGroup>

<div id="troubleshooting-and-support-for-conversion-api">
  ## Resolución de problemas y soporte para Conversion API
</div>

Para consultas sobre códigos de error tras realizar una llamada a la API, consulta la siguiente sección. Para cualquier otra consulta, no dudes en ponerte en contacto con tu representante de Twitter y trabajaremos para resolverla lo antes posible. 

<div id="error-handling-and-explanation">
  ### Gestión y explicación de errores
</div>

Una solicitud individual solo se considerará satisfactoria cuando no exista ningún error en ninguna de las conversiones que contiene. En caso de que se produzca cualquier error dentro de una conversión individual, el endpoint emitirá una lista de todos los errores aplicables.

<div id="x-ads-api-error-codes-overview">
  ### Descripción general de los códigos de error de X Ads API
</div>

A continuación se muestra una lista completa de los códigos de error en X Ads API:

[https://developer.x.com/en/docs/twitter-ads-api/response-codes](https://developer.x.com/en/docs/twitter-ads-api/response-codes)

Las respuestas satisfactorias de Conversion API se indican con un código HTTP de la serie 200 y una carga útil en formato JSON que contiene el objeto solicitado.

<div id="when-there-is-a-500-series-http-code-its-related-to-a-server-issue-instead-of-your-request-or-account-set-up-please-check-the-x-api-status-page-or-the-developer-community-forum-in-case-others-are-having-similar-issues">
  #### **Cuando recibas un código HTTP de la serie 500**, estará relacionado con un problema del servidor y no con tu solicitud o la configuración de tu cuenta. Consulta la [página de estado de X API](https://api.twitterstat.us/) o el [foro de la comunidad de desarrolladores](https://devcommunity.x.com/) para ver si otras personas están teniendo problemas similares.
</div>

<div id="when-there-is-a-400-series-http-code-the-common-cases-are">
  #### Cuando se produce un código HTTP de la serie 400, los casos más comunes son
</div>

* 400 Bad Request (la solicitud no cumple los estándares establecidos)

* 401 Unauthorized (problemas de autenticación)

* 403 Forbidden (problemas de acceso a la API asociados con esa cuenta de desarrollador)

* 404 Not Found (es posible que la URL o los parámetros no sean correctos para el endpoint)

<div id="conversion-api-error-codes">
  ### Códigos de error de la API de conversiones
</div>

<div id="400-bad-request-scenarios">
  #### Escenarios de error 400 Bad Request
</div>

| **Motivo** | **Tipo** | **Mensaje de error** |
| :--- | :--- | :--- |
| Error de identificador faltante (actualmente correo electrónico con hash o X click ID - twclid) | 400 Bad Request | Se debe proporcionar al menos un identificador de usuario |

<div id="401-unauthorized">
  ### 401 No autorizado
</div>

**Motivo**: Faltan credenciales de autenticación o son incorrectas 

**Solución**: Sigue los pasos de autenticación en la [documentación de configuración](https://developer.x.com/en/docs/twitter-ads-api/measurement/guides/conversion-api) utilizando uno de los 3 métodos de autenticación:

Los Tokens de acceso de usuario para usuarios distintos del que es propietario de la aplicación de Ads API deben generarse con un flujo OAuth de tres patas. Las opciones para generar el Token de acceso con OAuth de tres patas incluyen

* [Línea de comandos con autorización basada en la web mediante la utilidad twurl](https://github.com/twitter/twurl#getting-started)

* [Línea de comandos con autorización basada en PIN](https://developer.x.com/resources/fundamentals/authentication/pin-based-oauth)

* [Flujo web personalizado que implemente el patrón OAuth de tres patas](https://developer.x.com/resources/fundamentals/authentication/obtaining-user-access-tokens)

Cualquier Token de acceso de usuario utilizado con la Conversion API debe pertenecer a usuarios con nivel de acceso AD&#95;MANAGER o ACCOUNT&#95;ADMIN*, lo cual se puede comprobar a través del endpoint [authenticated&#95;user&#95;access](https://developer.x.com/en/docs/twitter-ads-api/campaign-management/api-reference/authenticated-user-access).

Envía eventos de conversión del sitio web para un único ID de etiqueta de evento.
