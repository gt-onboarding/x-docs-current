<div id="overview">
  ## Descripción general
</div>

Si ya tienes Aplicaciones, puedes verlas, editarlas o eliminarlas a través de la [página de Aplicaciones del portal para desarrolladores](https://developer.x.com/en/portal/projects-and-apps).

Acceder a la X API y a la X Ads API requiere un conjunto de credenciales de [autenticación](/es/resources/fundamentals/authentication), también conocidas como claves y tokens, que debes enviar con cada solicitud. Estas credenciales pueden adoptar diferentes formas según el tipo de autenticación que requiera el endpoint específico que estés usando.

A continuación se muestran las diferentes credenciales que puedes generar en tu Aplicación y cómo utilizarlas:

* **[API Key and Secret](/es/resources/fundamentals/authentication#api-key-and-secret):** Básicamente, el nombre de usuario y la contraseña de tu Aplicación. Las usarás para autenticar solicitudes que requieran [Contexto de usuario de OAuth 1.0a](/es/resources/fundamentals/authentication#oauth-1-0a-2), o para generar otros tokens, como tokens de acceso de usuario o el token de acceso de la Aplicación.

* **[Access Token and Secret](/es/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow):** En general, los tokens de acceso representan al usuario en cuyo nombre estás realizando la solicitud. Los que puedes generar a través del portal para desarrolladores representan al usuario que es propietario de la Aplicación. Los usarás para autenticar solicitudes que requieran [Contexto de usuario de OAuth 1.0a](/es/resources/fundamentals/authentication#oauth-1-0a-2). Si quieres realizar solicitudes en nombre de otro usuario, deberás utilizar el flujo OAuth de tres patas para que te autorice.

* **[Client ID and Client Secret](/es/resources/fundamentals/authentication#oauth-2-0):** Estas credenciales se utilizan para obtener un token de acceso de usuario con autenticación OAuth 2.0. De forma similar a OAuth 1.0a, los tokens de acceso de usuario se utilizan para autenticar solicitudes que proporcionan información privada de la cuenta de usuario o realizan acciones en nombre de otra cuenta, pero con un alcance granular para un mayor control sobre el acceso que tiene la aplicación cliente al usuario.

* **[App only Access Token](/es/resources/fundamentals/authentication#bearer-token-also-known-as-app-only):** Utilizarás este token cuando hagas solicitudes a endpoints que devuelvan información disponible públicamente en X.

Además de generar las claves y tokens necesarios para realizar solicitudes a la X API, también podrás configurar [permisos](/es/resources/fundamentals/developer-apps#app-permissions) de acceso, documentar el caso de uso o propósito de la Aplicación, definir una [URL de callback](/es/resources/fundamentals/developer-apps#callback-urls) y modificar otros ajustes relacionados con tu entorno de desarrollo de Aplicaciones desde el [panel de administración](https://developer.x.com/en/portal/projects-and-apps).

<div id="apps-and-projects">
  ### Aplicaciones y Proyectos
</div>

Puedes usar Aplicaciones y [Proyectos](/es/resources/fundamentals/projects) para organizar tu trabajo con la plataforma para desarrolladores de X por casos de uso. Cada Proyecto puede incluir una Aplicación con el plan Free de X API, hasta dos Aplicaciones con el plan Basic y hasta tres Aplicaciones con el plan Pro.

Si quieres acceder a los nuevos endpoints de [X API v2](/es/x-api/introduction), deberás usar claves y tokens de una Aplicación que esté asociada con un Proyecto.

Si tienes Aplicaciones que se crearon antes de que lanzáramos los Proyectos, serán visibles en la sección titulada &quot;Standalone Apps&quot;. Las Standalone Apps son Aplicaciones fuera de la estructura de Proyecto. Si vinculas una Standalone App a un Proyecto, entonces podrá realizar solicitudes a los endpoints v2.

<div id="developer-portal-dashboard">
  ### Panel del portal para desarrolladores
</div>

Puedes [visitar el panel](https://developer.x.com/content/developer-twitter/en/apps) para gestionar las Apps asociadas a tu cuenta. Para obtener más información, visita nuestra página de documentación sobre el [portal para desarrolladores](/es/resources/fundamentals/developer-portal). El panel permite a los desarrolladores realizar de forma rápida y sencilla las siguientes tareas:

* Ver tus Apps independientes existentes y los App ID asociados.
* Crear un nuevo Proyecto, App o App independiente.
* Eliminar un Proyecto o una App que no se esté usando.
* Revisar o actualizar la configuración de una App específica, incluida la actualización del nombre, la descripción, el sitio web, la URL de callback y los [permisos](/es/resources/fundamentals/developer-apps#app-permissions).
* Regenerar credenciales específicas de la App como la API Key &amp; Secret, el Token de acceso de la App y los Tokens de acceso de usuario del propietario de la App.

<div id="signing-up-for-access">
  ### Registrarse para obtener acceso
</div>

Si ya tienes aplicaciones de X, puedes verlas y editarlas desde el [panel de aplicaciones](https://developer.x.com/content/developer-twitter/en/portal/projects-and-apps) de X si has iniciado sesión en tu cuenta de X en developer.x.com. Ten en cuenta que **no** necesitas registrarte para obtener una cuenta nueva para gestionar todas las aplicaciones que se hayan creado previamente en developer.x.com.

Si necesitas crear una nueva aplicación de X, deberás haber [creado una cuenta de desarrollador](https://developer.x.com/en/portal/petition/essential/basic-info). Si eres miembro de una [cuenta de equipo](/es/resources/fundamentals/developer-portal#team-management), debes tener asignado un rol de administrador para crear una nueva aplicación.

<div id="automated-account-labeling-for-bots">
  ### Etiquetado de Cuenta Automatizada para bots
</div>

Puedes añadir una etiqueta de Cuenta Automatizada a tus cuentas de bot para que las personas en X sepan que tu bot es una cuenta automatizada. Estos bots realizan acciones programadas a través de la X API. Cuando añades una etiqueta de Cuenta Automatizada a tu bot, generas confianza con tu audiencia, legitimas tu cuenta y te diferencias de los bots de spam. Esto ayuda a que las personas en X entiendan mejor el propósito de tu cuenta cuando interactúan con tu bot.

Para añadir una etiqueta a tu cuenta de bot, sigue estos pasos:

1. Ve a la configuración de tu cuenta
2. Selecciona &quot;Tu cuenta&quot;
3. Selecciona &quot;Automatización&quot;
4. Selecciona &quot;Administrar cuenta&quot;
5. A continuación, selecciona la cuenta de X que ejecuta tu cuenta de bot
6. Introduce tu contraseña para iniciar sesión
7. Finalmente, verás la confirmación de que la etiqueta se ha aplicado a tu cuenta.

<div id="app-details">
  #### Detalles de la Aplicación
</div>

Aquí puedes editar el icono de la Aplicación, el nombre de la Aplicación, la descripción de la Aplicación, la URL de tu sitio web, las [URL de callback/URI de redirección](/es/resources/fundamentals/developer-apps#callback-urls), la URL de los términos de servicio, la URL de la política de privacidad, el nombre de la organización, la URL de la organización y el propósito o caso de uso de la Aplicación.

OAuth 2.0 y OAuth 1.0a son métodos de autenticación que permiten a los usuarios iniciar sesión en tu Aplicación con X. También permiten que tu Aplicación realice solicitudes específicas en nombre de usuarios autenticados. Puedes activar uno o ambos métodos para tu Aplicación.

Es importante mantener esta información actualizada. El nombre de tu Aplicación y la URL de tu sitio web se mostrarán como origen dentro de los metadatos de cualquier Publicación creada de forma programática por tu Aplicación. Si cambias el caso de uso de una Aplicación de X, asegúrate de actualizar el caso de uso en esta configuración a fin de garantizar que cumples con los [Términos para Desarrolladores](https://developer.x.com/content/developer-twitter/en/developer-terms/agreement-and-policy).

Si tu Aplicación tiene una etiqueta que muestra &quot;suspended&quot;, es porque tu Aplicación infringe uno o más de los [términos para desarrolladores](https://developer.x.com/en/developer-terms.html) de X, como nuestras [reglas de automatización](https://help.x.com/en/rules-and-policies/twitter-automation). El equipo de políticas de la plataforma para desarrolladores se comunicará con los desarrolladores a través de la dirección de correo electrónico configurada en la cuenta de X del propietario de la Aplicación; para revisar esta dirección de correo electrónico, consulta la [configuración de tu cuenta de X](https://x.com/settings/your_twitter_data/account). Los correos electrónicos de notificación sobre suspensiones se enviarán a esta dirección de correo electrónico con un asunto similar a &quot;Application suspension notice&quot; y contendrán información específica sobre qué hacer a continuación. Para trabajar con el equipo de X para resolver suspensiones, revisa tu correo electrónico para obtener instrucciones específicas o utiliza nuestro [formulario de ayuda de la plataforma](https://help.x.com/forms/platform).

<div id="keys-and-tokens">
  #### Claves y tokens
</div>

Dentro de una Aplicación en un Proyecto o mediante una Aplicación independiente puedes ver, regenerar o revocar los siguientes tokens:

* [API Key (Consumer Key) y API Secret (Consumer Secret)](/es/resources/fundamentals/authentication#api-key-and-secret)
* [App Access Token](/es/resources/fundamentals/authentication#bearer-token-also-known-as-app-only)
* [User Access Token y Access Token Secret](/es/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) -
  El User Access Token y el Access Token Secret disponibles dentro del portal para desarrolladores corresponden al usuario que es propietario de la Aplicación.

**Ten en cuenta**: si quieres hacer solicitudes en nombre de un usuario diferente (es decir, que no sea el usuario propietario de la Aplicación), tendrás que usar el [flujo OAuth 1.0a de tres patas](https://developer.x.com/resources/fundamentals/authentication/obtaining-user-access-tokens) o el flujo de OAuth 2.0 [Authorization Code with PKCE](/es/resources/fundamentals/authentication#oauth-2-0-authorization-code-flow-with-pkce-2) para generar un conjunto de tokens de acceso de usuario. Luego, usarás estos tokens específicos de usuario en tu solicitud a la API.

<div id="user-authentication-settings">
  #### Configuración de autenticación de usuario
</div>

Puedes configurar la autenticación de tu Aplicación para que use OAuth 1.0a u OAuth 2.0. OAuth 2.0 solo puede usarse con X API v2. OAuth 2.0 te permite elegir scopes específicos y granulares que te otorgan permisos concretos en nombre de un usuario. OAuth 1.0a puede utilizarse con X API v1.1 y v2 y utiliza una autorización amplia con ámbitos generales.

<div id="oauth-10a-application-user-permissions">
  #### Permisos de aplicación-usuario de OAuth 1.0a
</div>

Si eres el propietario de la aplicación, puedes ajustar sus permisos (solo lectura; lectura y escritura; o lectura, escritura y mensajes directos). Esto controla a qué recursos y eventos tienes acceso a través de las X API (nota: algunos recursos requieren permisos adicionales directamente de X).

También puedes activar y desactivar en esta página la capacidad de tus aplicaciones para solicitar direcciones de correo electrónico de los usuarios (esto requiere que haya una URL de Términos de servicio y una URL de Política de privacidad en la página &quot;App details&quot;).

<div id="oauth-20-type-of-app">
  #### Tipo de Aplicación OAuth 2.0
</div>

Si seleccionas OAuth 2.0 como método de autenticación de usuario, tendrás que seleccionar el tipo de Aplicación que estás creando. Tus opciones son Native App, Single Page App, Web App, Automated App o bot. Native App y Single Page App son clientes públicos, y Web App y Automated App o los bots son clientes confidenciales.

Los clientes confidenciales se autentican de forma segura con el servidor de autorización y mantienen tu secreto de cliente a salvo. Los clientes públicos son aplicaciones que normalmente se ejecutan en un navegador o en un dispositivo móvil y no pueden usar tu secreto de cliente. Si seleccionas un tipo de Aplicación que sea un cliente confidencial, se te proporcionará un secreto de cliente.

<div id="app-permissions">
  ## Permisos de la aplicación
</div>

<div id="oauth-10a-app-permissions">
  ### Permisos de la aplicación OAuth 1.0a
</div>

Los permisos de la aplicación describen el nivel de acceso para la autenticación de usuario mediante OAuth 1.0a. Los permisos de la aplicación se configuran para cada aplicación dentro de la configuración de tu [X App](/es/resources/fundamentals/developer-apps).

Hay tres niveles de permiso disponibles:

1. Solo lectura
2. Lectura y escritura
3. Lectura, escritura y acceso a Mensajes Directos

Existe un permiso adicional para solicitar la visibilidad de la dirección de correo electrónico de un usuario; este se puede combinar con cualquiera de los tres niveles indicados anteriormente.

Si se cambia un nivel de permiso, cualquier token de usuario ya emitido para esa X App debe descartarse y los usuarios deben volver a autorizar la App para que el token herede los permisos actualizados.

Una buena práctica consiste en solicitar *solo el nivel mínimo de acceso* a los datos de la cuenta de un usuario que una aplicación o servicio requiera.

<div id="read-only">
  ### Solo lectura
</div>

Este nivel de permiso permite el acceso de lectura a los recursos de X, incluidos (por ejemplo) las Publicaciones de un usuario, la timeline de inicio y la información de perfil. No permite acceder a los Mensajes Directos de un usuario ni actualizar ningún elemento u objeto.

<div id="read-and-write">
  ### Lectura y escritura
</div>

Este nivel de permisos permite acceso de lectura y escritura a los recursos de X. Además de permitir el acceso de lectura, también permite publicar Publicaciones, seguir a usuarios o actualizar elementos de la información del perfil de un usuario. También permite ocultar respuestas en nombre del usuario autenticado. Este nivel de permisos no permite ningún acceso a los Mensajes Directos (incluido lectura, escritura o eliminación).

<div id="read-write-and-access-direct-messages">
  ### Leer, escribir y acceder a mensajes directos
</div>

Este nivel de permisos incluye acceso a todo lo anterior y añade la capacidad de leer, escribir y eliminar mensajes directos en nombre de un usuario.

* [POST /2/dm&#95;conversations/:dm&#95;conversation&#95;id/messages](/es/x-api/direct-messages/send-a-new-message-to-a-dm-conversation)
* [POST /2/dm&#95;conversations/](/es/x-api/direct-messages/create-a-new-dm-conversation)
* [POST /2/dm&#95;conversations/with/:participant&#95;id/messages](/es/x-api/direct-messages/send-a-new-message-to-a-user)
* [GET /2/dm&#95;conversations/with/:user&#95;id/dm&#95;events](/es/x-api/direct-messages/get-dm-events-for-a-dm-conversation)
* [GET /2/dm&#95;conversations/:dm&#95;conversation&#95;id/dm&#95;events](/es/x-api/direct-messages/get-dm-events-for-a-dm-conversation-1)
* [GET /2/dm&#95;events](/es/x-api/direct-messages/get-recent-dm-events)

<div id="determining-permissions">
  ### Determinación de permisos
</div>

Todas las solicitudes de API autenticadas devuelven un encabezado `x-access-level` en la respuesta HTTP. El valor de ese encabezado indica el nivel de permiso actual en uso. Los valores posibles son read, read-write y read-write-directmessages.

<div id="callback-urls">
  ## URLs de callback
</div>

Los métodos de autenticación Contexto de usuario de OAuth 1.0a y OAuth 2.0 Authorization Code with PKCE permiten a los desarrolladores realizar solicitudes en nombre de diferentes usuarios de X que han seguido un flujo de inicio de sesión específico. Actualmente, hay dos flujos que puedes usar para permitir que los usuarios autoricen tu Aplicación:

* [Flujo de código de autorización de OAuth 2.0 con PKCE](/es/resources/fundamentals/authentication#oauth-2-0-authorization-code-flow-with-pkce-2)
* [Flujo OAuth de tres patas de OAuth 1.0a](/es/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) (y, por separado, el [flujo Sign in with X](/es/resources/fundamentals/authentication#log-in-with-x))

A medida que los usuarios avanzan en estos flujos, necesitan una página web o ubicación a la que se les redirija después de haber iniciado sesión correctamente y haber otorgado autorización a la Aplicación del desarrollador. Esta página web o ubicación de destino se denomina URL de callback.

Al configurar estos flujos para que sus posibles usuarios los sigan, los desarrolladores deben pasar una URL de callback en sus solicitudes a los endpoints de autenticación que forman parte de los flujos mencionados anteriormente. Por ejemplo, los desarrolladores que usen el Contexto de usuario de OAuth 1.0a deben pasar el parámetro `callback_url` al realizar una solicitud al endpoint [GET oauth/request&#95;token](/es/resources/fundamentals/authentication#post-oauth-request-token). De manera similar, los desarrolladores que usen OAuth 2.0 Authorization Code with PKCE deben pasar el parámetro `redirect_uri` con su solicitud al endpoint GET oauth2/authorize.

Además de usar estos parámetros, el desarrollador también debe asegurarse de que la URL de callback se haya agregado a la lista de URLs de callback permitidas de su [Aplicación](/es/resources/fundamentals/developer-apps), que se encuentra en la página de configuración de la Aplicación en el [portal para desarrolladores](/es/resources/fundamentals/developer-portal).

Si se configura correctamente, se redirigirá a los usuarios a la URL de callback después de iniciar sesión correctamente en X como parte de estos flujos.

<div id="things-to-keep-in-mind">
  ### Cosas que debes tener en cuenta
</div>

Cuando configures tus callback URLs, hay algunas cosas que debes tener en cuenta:

**Codifica en HTTP tus parámetros de consulta**

Como estás pasando la callback URL como un parámetro de consulta con los parámetros `callback_url` o `redirect_uri`, asegúrate de codificar la URL según HTTP.

**Límites de callback URL**

Hay un límite estricto de 10 callback URLs en el panel de Apps de X. Tu callback URL siempre debe coincidir exactamente entre la callback URL permitida que agregas al panel de Apps y el parámetro que agregas en el flujo de autorización.

Si deseas incluir datos específicos de la solicitud en la callback URL, puedes usar el parámetro `state` para almacenar datos que se incluirán después de que el usuario sea redirigido. Puedes codificar los datos en el propio parámetro `state` o usar el parámetro como un id de sesión para almacenar el estado en el servidor.

**No uses localhost como callback URL**
En lugar de usar localhost, utiliza un host personalizado en tu entorno local o http(s)://127.0.0.1

**URLs con protocolo personalizado**
Si deseas aprovechar el deep linking en dispositivos móviles, puedes usar URLs con protocolo personalizado con una parte de ruta y de dominio, como twitter://callback/path. Sin embargo, tenemos una lista de protocolos no permitidos que deberás evitar. Puedes revisar la lista de protocolos no permitidos a continuación.

<div id="error-example">
  ### Ejemplo de error
</div>

Si utilizas una URL de callback que no se haya añadido correctamente a la configuración de tu aplicación en el portal para desarrolladores, recibirás el siguiente mensaje de error:

**OAuth 1.0a**

```json HTTP 403 - Forbidden

{
  "errors": [
    {
      "code":415,"message":"Callback URL not approved for this client application. Approved callback URLs can be adjusted in your application settings."
    }
  ]
}
```

O

```xml
<?xml version="1.0" encoding="UTF-8"?>
<hash>
<error>Callback URL not approved for this client application. Approved callback URLs can be adjusted in your application settings</error>
<request>/oauth/request_token</request>
</hash>
```

**OAuth 2.0**

```json HTTP 400

{
  "error": "invalid_request",
  "error_description": "El valor proporcionado para el redirect uri no coincidió con el uri del código de autorización."
}
```

**Solución de problemas**

Si se produce un error, asegúrate de que la URL de callback que utilizas en tus solicitudes de autenticación esté codificada para HTTP y que coincida con una URL de callback que se haya añadido a la allowlist de la App cuyas claves y tokens estás utilizando en tu solicitud.
