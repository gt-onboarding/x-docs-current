Creemos que la privacidad es un derecho, no un privilegio, y forma parte de los cimientos de nuestra empresa. Al usar la plataforma de desarrolladores de X y cumplir con nuestra política para desarrolladores, desempeñas un papel fundamental para garantizar que la plataforma esté al servicio de la conversación pública en X y salvaguarde nuestro compromiso con la privacidad.

Queremos recordarte la importancia de desarrollar de forma segura para proteger tanto tus propios datos como los de los usuarios de tus aplicaciones. Es tu responsabilidad protegerte frente al riesgo de vulneraciones de seguridad y compartimos la responsabilidad de proteger a las personas que usan X. En esta página se describen las expectativas relacionadas con la creación de aplicaciones seguras y con mantener lo más seguros posible los datos y el acceso.

<Note>Ten en cuenta las tecnologías de seguridad disponibles para la plataforma de desarrolladores de X, incluidas la [autenticación](/es/resources/fundamentals/authentication), TLS y los [permisos de la app](/es/resources/fundamentals/developer-apps#app-permissions), y también, desde la perspectiva del usuario de X, el [uso de aplicaciones y sesiones de terceros](https://help.x.com/en/managing-your-account/connect-or-revoke-access-to-third-party-apps).</Note>

<div id="reporting-security-issues">
  ## Notificación de problemas de seguridad
</div>

Los usuarios de la X Developer Platform deben notificar a X en un plazo máximo de 48 horas desde la sospecha inicial de que se ha producido un incidente de seguridad, a través del [programa de notificación de vulnerabilidades de X](https://hackerone.com/twitter).

<div id="security-best-practices">
  ## Mejores prácticas de seguridad
</div>

Tenlas en cuenta al desarrollar en la plataforma para desarrolladores de X y en el resto de Internet.

<div id="security-by-design">
  ### Seguridad desde el diseño
</div>

Considera contratar profesionales de seguridad informática para realizar una auditoría del modelo de amenazas y/o una prueba de penetración. Una buena empresa de seguridad profundizará para descubrir problemas. Lee más sobre cómo X ha adoptado esta mentalidad [en esta entrada de nuestro blog](https://blog.x.com/en_us/topics/company/2019/privacy_data_protection.html).

Además, X responsabiliza a todos los socios de lo siguiente:

* Mantener el código en un repositorio seguro.
* Realizar análisis de riesgos durante todo el ciclo de vida de desarrollo de sistemas (SDLC).
* Garantizar que los problemas de seguridad se identifiquen y se mitiguen durante todo el SDLC.
* Garantizar que exista separación de entornos durante todo el proceso de SDLC.
* Garantizar que todos los errores detectados en las pruebas se corrijan, se vuelvan a probar y se cierren.

<div id="monitor-and-get-alerted">
  ### Supervisa y recibe alertas
</div>

Si crees que hay un problema con tu aplicación web, ¿cómo puedes confirmarlo? Asegúrate de mantener buenos registros (logs) y de que se te notifique sobre excepciones y errores críticos. Quizá quieras crear un panel de control con estadísticas clave para poder ver de un vistazo si algo está fallando.

<div id="create-a-reporting-channel">
  ### Crear un canal para reportar problemas
</div>

Facilita que tus usuarios se pongan en contacto contigo si detectan posibles problemas de seguridad que hayan experimentado con tu Aplicación. Si se descubre un problema que afecte a usuarios y datos de X, también es tu responsabilidad [informar de este problema a X](#report). Ten preparado un plan o procedimiento para notificar a los usuarios afectados en caso de que ocurra un incidente de seguridad.

<div id="adequate-testing">
  ### Pruebas adecuadas
</div>

Asegúrate de que tus pruebas de extremo a extremo sean exhaustivas y estén actualizadas para incluir fallos previstos en escenarios de seguridad, como el acceso no autorizado. Adopta la mentalidad de un atacante y configura pruebas de sistema diseñadas para impedir que un atacante obtenga acceso no autorizado a datos de X o haga uso de funcionalidades autorizadas.

<div id="securing-api-keys-and-tokens">
  ### Protección de claves y tokens de API
</div>

Como desarrollador en la plataforma X, tienes acceso programático tanto a tus datos como a los datos de tus usuarios almacenados por X, siempre que hayan autorizado tu aplicación de desarrollador. Todas las solicitudes a la API deben [autenticarse](/es/resources/fundamentals/authentication) usando OAuth con la clave y el secreto de tu aplicación de desarrollador y, en algunos casos, con los [Token de acceso](/es/resources/fundamentals/authentication) del usuario que otorga la autorización. Es tu responsabilidad mantener tus credenciales seguras.

Algunas prácticas recomendadas son las siguientes:

* Crear un ciclo de rotación y renovación de contraseñas/tokens.
* Cifrar siempre los datos confidenciales cuando sea necesario y evitar descifrarlos demasiado pronto en el flujo de procesamiento.
* Almacenar los Token de acceso de tus usuarios en un almacén cifrado.
* [Regenerar](/es/resources/fundamentals/authentication#regenerate-api-keys-and-tokens) o [invalidar](/es/resources/fundamentals/authentication#post-oauth2-invalidate-token) las claves y tokens si crees que se han visto comprometidos.

Para más información sobre depuración y desarrollo con OAuth para X, visita la [categoría de seguridad](https://devcommunity.x.com/c/oauth/12) del foro de la comunidad.

<div id="input-validation">
  ### Validación de datos de entrada
</div>

No des por hecho que tus usuarios te proporcionarán datos válidos y fiables. Sanitiza todos los datos que provengan de tus usuarios y que puedan terminar en solicitudes a la X API. Define una lista de permitidos (allowlist) con los tipos de datos de entrada que son aceptables para tu aplicación y descarta todo lo que no esté en esa lista de permitidos.

<div id="encrypted-communication">
  ### Comunicación cifrada
</div>

X requiere que todas las solicitudes a la API se realicen mediante TLS. La comunicación con tus propios servidores también debe cifrarse siempre que sea posible.

<div id="exposed-debugging-information">
  ### Información de depuración expuesta
</div>

Asegúrate de no exponer datos confidenciales de X ni credenciales a través de pantallas y registros de depuración. Algunos frameworks web facilitan el acceso a información de depuración si tu aplicación no está configurada correctamente. Para desarrolladores de escritorio y móviles, es fácil enviar por error una versión de compilación con opciones o símbolos de depuración habilitados. Incluye comprobaciones para estas configuraciones en tu proceso de implementación y compilación. Además, si compartes trazas de pila o volcados de errores para su análisis, asegúrate de que los datos de usuarios privados de X se hayan ocultado.

<div id="unfiltered-input-unescaped-output">
  ### Entrada sin filtrar, salida sin escapar
</div>

Un enfoque fácil de recordar para la validación de entrada es FIEO: Filter Input, Escape Output (filtrar la entrada, escapar la salida).

Filtra cualquier dato que provenga de fuera de tu aplicación, incluidos los datos de X API, los datos de cookies, los datos de formularios proporcionados por el usuario, los parámetros de la URL, los datos de las bases de datos, etc. Aplica escape a toda la salida que envíe tu aplicación, incluido el SQL enviado a tu servidor de base de datos, el HTML que envías a los navegadores de los usuarios, la salida JSON enviada a otros sistemas y los comandos enviados a programas de shell.

<div id="cross-site-scripting-xss">
  ### Cross-site scripting (XSS)
</div>

Los ataques de [XSS](https://owasp.org/) son, según prácticamente cualquier métrica, el problema de seguridad más común en la web. Si un atacante logra inyectar su propio código JavaScript en tu aplicación, puede hacer cosas maliciosas. Cualquier lugar donde almacenes y muestres datos no confiables proporcionados por el usuario debe verificarse, sanearse y escaparse en HTML. Hacer esto correctamente es difícil, porque los atacantes tienen [muchas formas distintas de llevar a cabo ataques XSS](https://www.owasp.org/index.php/Types_of_Cross-Site_Scripting). Es probable que tu lenguaje o framework de desarrollo web ya tenga un mecanismo popular y bien probado para protegerte frente a cross-site scripting; asegúrate de utilizarlo.

<div id="sql-injection">
  ### Inyección SQL
</div>

Si tu aplicación hace uso de una base de datos, debes estar al tanto de la [inyección SQL](http://en.wikipedia.org/wiki/SQL_injection). Cualquier punto donde aceptes datos de entrada es un objetivo potencial para que un atacante se salga de su campo de entrada y acceda a tu base de datos. Usa bibliotecas de acceso a bases de datos que protejan contra la inyección SQL de forma sistemática. Si te sales de ese enfoque y escribes SQL personalizado, crea pruebas exhaustivas para asegurarte de que no te estés exponiendo a este tipo de ataque. Los dos enfoques principales para defenderse contra la inyección SQL son escapar los valores antes de construir tu sentencia SQL y usar consultas parametrizadas para crear las sentencias. Se recomienda este último, ya que es menos propenso a errores de programación.

<div id="cross-site-request-forgery-csrf">
  ### Cross-site request forgery (CSRF)
</div>

¿Estás seguro de que las solicitudes que llegan a tu aplicación realmente provienen de tu propia aplicación? Los ataques de [CSRF](http://en.wikipedia.org/wiki/Cross-site_request_forgery) se aprovechan de esta falta de certeza obligando a los usuarios que han iniciado sesión en tu sitio a abrir de forma silenciosa URL que ejecutan acciones. En el caso de una aplicación de desarrollador, esto podría significar que los atacantes están usando tu aplicación para obligar a los usuarios a publicar publicaciones no deseadas o seguir cuentas de spam.

La forma más completa de tratar el CSRF es incluir un token aleatorio en cada formulario que se almacene en un lugar de confianza; si un formulario no tiene el token correcto, lanza un error. Los frameworks web modernos tienen formas sistemáticas de manejar esto, e incluso podrían estar haciéndolo por defecto si tienes suerte. Un paso preventivo sencillo (aunque de ninguna manera el único paso que deberías tomar) es hacer que cualquier acción que cree, modifique o destruya datos requiera una solicitud `POST`.

<div id="lack-of-rate-limiting">
  ### Ausencia de limitación de tasa
</div>

Utiliza CAPTCHAs cuando sea apropiado para frenar a posibles spammers y atacantes.
