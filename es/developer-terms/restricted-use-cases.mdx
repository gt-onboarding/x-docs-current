El uso de nuestra plataforma para desarrolladores requiere revisar y aceptar nuestro [Acuerdo y Política para Desarrolladores](/es/developer-terms/agreement), así como nuestras políticas relacionadas, incluidos los [Requisitos de visualización](https://developer.x.com/en/developer-terms/display-requirements.html) y las [Reglas de automatización](https://help.twitter.com/en/rules-and-policies/twitter-automation). Entre otras cosas, nuestros acuerdos y políticas ofrecen orientación sobre varios casos de uso restringidos. A continuación, proporcionamos información adicional sobre algunas de estas restricciones.

<div id="sensitive-information">
  ## Información sensible
</div>

Debes tener cuidado al usar datos de X para derivar o inferir características potencialmente sensibles sobre las personas usuarias de X. Nunca derives, infieras ni almacenes información derivada o inferida sobre un usuario de X relacionada con su:

* Salud (incluido el embarazo)
* Situación o condición financiera negativa
* Afiliación o creencias políticas
* Origen racial o étnico
* Afiliación o creencias religiosas o filosóficas
* Vida sexual u orientación sexual
* Afiliación sindical
* Presunta o real comisión de un delito

Se permite el análisis agregado de contenido de X que no almacene ningún dato personal (por ejemplo, ID de usuario, nombre de usuario y otros identificadores), siempre que dicho análisis también cumpla las leyes aplicables y todas las disposiciones del Acuerdo y la Política para Desarrolladores.

<div id="off-x-matching">
  ## Coincidencias fuera de X
</div>

Las coincidencias fuera de X consisten en asociar contenido de X, incluido un nombre de usuario de X o un id de usuario, con una persona, hogar, dispositivo, navegador u otro identificador externo a X. Un ejemplo sería asociar un nombre de usuario de X con los registros de clientes de una empresa (es decir, “John Doe” en tu registro de clientes se vincula con @johndoe en X).

Queremos que las personas se sientan cómodas creando una identidad separada y, si así lo desean, seudónima en X. Si tienes la intención de asociar cualquier información sobre un usuario de X con un identificador externo a X, exigimos que obtengas el consentimiento expreso y voluntario (opt-in) del usuario antes de realizar dicha asociación. Por ejemplo, podrías obtener este consentimiento si el usuario comparte directamente contigo su handle de X como parte del proceso de registro en tu servicio.

En situaciones en las que no cuentes con el consentimiento expreso y voluntario de un usuario para vincular su identidad en X con un identificador externo a X, exigimos que cualquier conexión que establezcas se base únicamente en información que el usuario podría razonablemente esperar que se utilice con ese fin. Si un usuario se sorprendería al saber que estás usando la información que proporcionó para vincular su cuenta de X con una identidad fuera de X, no lo hagas. Además, sin el consentimiento expreso y voluntario de una persona, solo puedes intentar hacer coincidir tus registros sobre alguien con una identidad de X basándote en:

* **Información proporcionada directamente a ti por el usuario.** Los registros sobre personas con las que no tienes una relación previa, incluidos los datos sobre personas obtenidos de terceros, no cumplen esta norma; y/o
* **Datos públicos.** “Datos públicos” en este contexto se refiere a:
  * Información sobre un usuario que obtuviste de un recurso público y de acceso general (como un directorio de miembros de una asociación profesional)
  * Información en X sobre un usuario que está disponible públicamente, incluyendo:
    * Publicaciones
    * Información de perfil, incluida la biografía del usuario y su ubicación declarada públicamente
    * Nombre para mostrar y nombre de usuario

<div id="redistribution-of-x-content">
  ## Redistribución de contenido de X
</div>

Si necesitas compartir contenido de X que hayas obtenido a través de las X API con otra parte, la mejor manera de hacerlo es compartiendo IDs de Publicaciones, IDs de Mensajes Directos y/o IDs de Usuarios, que el usuario final del contenido puede luego rehidratar (es decir, solicitar el contenido completo de la Publicación, el Usuario o el Mensaje Directo) usando las X API. Esto ayuda a garantizar que los usuarios finales del contenido de X siempre obtengan la información más actualizada directamente de nosotros.

Permitimos la redistribución limitada de contenido de X rehidratado por medios no automatizados. Si decides compartir contenido de X rehidratado con otra parte de esta manera, solo puedes compartir hasta 50.000 Objetos de Publicaciones públicas y/o Objetos de Usuarios rehidratados por destinatario y por día, y no debes poner estos datos a disposición del público (por ejemplo, como un archivo adjunto a una publicación de blog o en un repositorio público de GitHub).

Hay algunos otros puntos a tener en cuenta sobre la redistribución de contenido de X:

* Solo puedes distribuir hasta un total de 1.500.000 IDs de Publicaciones a una sola entidad dentro de un período de 30 días, a menos que hayas recibido autorización previa, expresa y por escrito de X.
* Las personas que redistribuyan IDs de Publicaciones y/o IDs de Usuarios en nombre de una institución académica con el único propósito de investigación no comercial pueden redistribuir un número ilimitado de IDs de Publicaciones y/o IDs de Usuarios.
* Para solicitar permiso para compartir contenido de X como se detalla arriba, utiliza el formulario de soporte de API Policy.

En la medida en que se te permita distribuir contenido de X a un tercero, ten en cuenta que este contenido sigue estando sujeto al Developer Agreement y a la Developer Policy, y esos terceros deben aceptar los [Terms of Service](https://twitter.com/en/tos) de X, la [Privacy Policy](https://twitter.com/en/privacy), el [Developer Agreement](/es/developer-terms/agreement) y la [Developer Policy](/es/developer-terms/policy) antes de recibir contenido de X.

<div id="multiple-applications">
  ## Múltiples aplicaciones
</div>

No se te permite registrar múltiples aplicaciones para un único caso de uso, ni para casos de uso sustancialmente similares o superpuestos. Obtén más información sobre estas políticas [aquí](https://twittercommunity.com/t/policy-clarification-multiple-applications-for-the-same-use-case/92080).

En este contexto, definimos “caso de uso” como un conjunto coherente de análisis, visualizaciones o acciones realizadas a través de una aplicación. Proporcionar el mismo servicio o aplicación a diferentes usuarios finales cuenta como un único caso de uso; sin embargo, las versiones de una herramienta o servicio de &quot;marca blanca&quot; no se consideran dentro de este mismo caso de uso. Si planeas ofrecer versiones de &quot;marca blanca&quot; de tu aplicación, debes notificárnoslo y recibir nuestra aprobación por separado. Estas reglas se aplican tanto a las aplicaciones que registras tú como a las aplicaciones registradas por los usuarios finales de tu herramienta o servicio; exigir a tus usuarios finales que registren aplicaciones con el propósito de usar tu herramienta o servicio podría derivar en medidas de cumplimiento en tu contra, contra tus aplicaciones, tus clientes y/o los usuarios finales de tu herramienta o servicio.

La única excepción a esta regla es crear instancias de desarrollo (“dev”), preproducción (“staging”) y producción (“prod”) del mismo servicio. Asegúrate de que estas aplicaciones estén claramente etiquetadas (por ejemplo, en el nombre o la descripción de la aplicación) y de que no uses aplicaciones de desarrollo o preproducción con fines de producción.

<div id="automation-spam-and-auto-responses">
  ## Automatización, spam y respuestas automáticas
</div>

El uso de las API y productos para desarrolladores de X para crear spam o participar en comportamientos de spam está prohibido. Debes revisar las [Reglas de X](https://t.co/rules) sobre spam y asegurarte de que tu aplicación no infrinja nuestras políticas ni permita que los usuarios las infrinjan.

Si tu aplicación va a utilizarse para realizar acciones de escritura en el servicio de X, como publicar Publicaciones, seguir cuentas o enviar Mensajes Directos, debes revisar cuidadosamente las [Reglas de automatización](https://support.twitter.com/articles/76915) para asegurarte de que tu servicio cumpla con nuestras directrices. En particular, debes:

* Obtener siempre el consentimiento explícito de un usuario antes de enviarle [respuestas o mensajes automatizados](https://twittercommunity.com/t/policy-clarification-automated-replies-and-mentions/94444)
* Respetar de inmediato las solicitudes de los usuarios de dejar de recibir comunicaciones tuyas
* No realizar nunca acciones masivas, agresivas o de spam, incluido el [seguimiento masivo](https://twittercommunity.com/t/policy-clarification-aggressive-following-and-inorganic-following-behavior/92769)
* No publicar nunca contenido idéntico o sustancialmente similar en [varias cuentas](https://blog.twitter.com/developer/en_us/topics/tips/2018/automation-and-the-use-of-multiple-accounts.html)

<div id="measuring-the-x-service">
  ## Medir el servicio de X
</div>

No utilices las API de X para medir la disponibilidad, el rendimiento, la funcionalidad o el uso de X con fines de benchmarking o competitivos. Por ejemplo, nunca debes usar las API de X para:

* Calcular métricas agregadas de usuarios de X, como el número total de usuarios o cuentas activas
* Calcular métricas agregadas de Publicaciones de X, como el número total de Publicaciones por día o el número de interacciones de usuarios o de cuentas
* Medir o analizar el spam o la seguridad en X, excepto según lo permitido en las [X Rules](https://help.twitter.com/en/rules-and-policies/twitter-rules)

<div id="surveillance-privacy-and-user-protection">
  ## Vigilancia, privacidad y protección de los usuarios
</div>

En X, proteger y defender la privacidad de nuestros usuarios forma parte del ADN mismo de nuestra empresa, y nuestros productos para desarrolladores y de datos reflejan ese compromiso. Creemos que los datos de X pueden ser una fuerza poderosa para el bien en el mundo — desde salvar vidas durante inundaciones en Yakarta hasta ayudar al USGS a rastrear terremotos y trabajar con la ONU para lograr los Objetivos de Desarrollo Sostenible. Sin embargo, prohibimos el uso de los datos de X y de las X APIs por parte de cualquier entidad con fines de vigilancia, o de cualquier otra manera que sea incompatible con las expectativas razonables de privacidad de nuestros usuarios. Punto.

Describimos los usos prohibidos de nuestros datos y productos para desarrolladores en el [Acuerdo para Desarrolladores](/es/developer-terms/agreement), incluidas las prohibiciones de investigar o rastrear a usuarios de X o su contenido, así como de rastrear, alertar o monitorear eventos sensibles (como protestas, manifestaciones o reuniones de organización comunitaria).

Otras categorías de actividades prohibidas en virtud de estos términos incluyen (entre otras):

* Investigar o rastrear a grupos y organizaciones sensibles, como sindicatos o grupos activistas
* Verificaciones de antecedentes o cualquier forma de evaluación o filtrado extremo
* Análisis de riesgo crediticio o de seguros
* Elaboración de perfiles individuales o segmentación psicográfica
* Reconocimiento facial

Estas políticas se aplican a todos los usuarios de nuestras APIs. Cualquier uso indebido de las X APIs para estos fines estará sujeto a acciones de cumplimiento, que pueden incluir la suspensión y la revocación del acceso.

Para obtener información adicional destinada a autoridades encargadas de la aplicación de la ley que busquen información sobre cuentas de X, visita [https://t.co/le](https://t.co/le).

Además, en este momento, X prohíbe cualquier uso de las X APIs y/o Contenido de X para ajustar mediante fine‑tuning o entrenar un modelo fundacional o de vanguardia, con la excepción de [Grok](https://help.x.com/en/using-x/about-grok).
