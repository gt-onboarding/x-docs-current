El uso de nuestra plataforma para desarrolladores requiere que revises y aceptes nuestro [Developer Agreement and Policy](/es/developer-terms/agreement), así como nuestras políticas relacionadas, incluidos los [Display Requirements](https://developer.x.com/en/developer-terms/display-requirements.html) y las [Automation Rules](https://help.twitter.com/en/rules-and-policies/twitter-automation). Entre otras cosas, nuestros acuerdos y políticas ofrecen orientación sobre varios casos de uso restringidos. A continuación, ofrecemos información adicional sobre algunas de estas restricciones.

<div id="sensitive-information">
  ## Información sensible
</div>

Debes tener cuidado al usar los datos de X para deducir o inferir características potencialmente sensibles sobre las personas usuarias de X. Nunca deduzcas ni infieras, ni almacenes información deducida o inferida, sobre la:

* Salud (incluido el embarazo)
* Situación o condición financiera negativa
* Afiliación o creencias políticas
* Origen racial o étnico
* Afiliación o creencias religiosas o filosóficas
* Vida sexual u orientación sexual
* Pertenencia a un sindicato
* Presunta o real comisión de un delito

Se permite el análisis agregado del contenido de X que no almacene ningún dato personal, como ids de usuario, nombres de usuario y otros identificadores, siempre que el análisis también cumpla con las leyes aplicables y con todas las secciones del Developer Agreement and Policy.

<div id="off-x-matching">
  ## Coincidencias fuera de X
</div>

Las coincidencias fuera de X implican asociar contenido de X, incluido un nombre de usuario de X o un id de usuario de X, con una persona, un hogar, un dispositivo, un navegador u otro identificador ajeno a X. Un ejemplo sería asociar un nombre de usuario de X con los registros de clientes de una empresa (es decir, “John Doe” en tu registro de clientes se vincula con @johndoe en X).

Queremos que las personas se sientan cómodas creando una identidad separada y, si lo desean, seudónima en X. Si tienes la intención de asociar cualquier información sobre un usuario de X con un identificador ajeno a X, exigimos que obtengas el consentimiento expreso y voluntario (opt‑in) del usuario antes de realizar la asociación. Por ejemplo, podrías obtener este consentimiento si el usuario comparte directamente contigo su nombre de usuario de X (handle) como parte de un proceso de registro en tu servicio.

En situaciones en las que no tengas el consentimiento expreso y voluntario de un usuario para vincular su identidad en X con un identificador ajeno a X, exigimos que cualquier conexión que establezcas se base únicamente en información que el usuario razonablemente esperaría que se utilizara para ese fin. Si un usuario se sorprendería al saber que estás utilizando la información que proporcionó para vincular su cuenta de X con una identidad fuera de X, no lo hagas. Además, en ausencia del consentimiento expreso y voluntario de una persona, solo puedes intentar asociar tus registros sobre alguien con una identidad en X basándote en:

* **Información proporcionada directamente a ti por el usuario.** Los registros sobre personas con las que no tienes una relación previa, incluidos datos sobre personas obtenidos de terceros, no cumplen esta condición; y/o
* **Datos públicos.** “Datos públicos” en este contexto se refiere a:
  * Información sobre un usuario que obtuviste de un recurso público y de acceso general (como un directorio de miembros de una asociación profesional)
  * Información en X sobre un usuario que está disponible públicamente, incluyendo:
    * Publicaciones
    * Información de perfil, incluida la biografía de un usuario y su ubicación declarada públicamente
    * Nombre para mostrar y nombre de usuario

<div id="redistribution-of-x-content">
  ## Redistribución de contenido de X
</div>

Si necesitas compartir contenido de X que obtuviste a través de las API de X con otra parte, la mejor forma de hacerlo es compartiendo Post IDs, Direct Message IDs y/o User IDs, que el usuario final del contenido podrá luego rehidratar (es decir, solicitar la Publicación, el Usuario o el Mensaje Directo completos) utilizando las API de X. Esto ayuda a garantizar que los usuarios finales del contenido de X siempre obtengan la información más actualizada directamente de nosotros.

Permitimos la redistribución limitada de contenido rehidratado de X por medios no automatizados. Si decides compartir contenido rehidratado de X con otra parte de esta manera, solo puedes compartir hasta 50.000 objetos públicos de Publicación rehidratados y/o objetos de Usuario por destinatario, por día, y no debes poner estos datos a disposición del público (por ejemplo, como un archivo adjunto a una publicación de blog o en un repositorio público de GitHub).

Hay algunos otros puntos a tener en cuenta sobre la redistribución de contenido de X:

* Solo puedes distribuir hasta un total de 1.500.000 Post IDs a una sola entidad dentro de un período de 30 días, a menos que hayas recibido permiso previo expreso por escrito de X.
* Las personas que redistribuyen Post IDs y/o User IDs en nombre de una institución académica con el único propósito de realizar investigación no comercial tienen permitido redistribuir un número ilimitado de Post IDs y/o User IDs.
* Para solicitar permiso para compartir contenido de X tal como se describe arriba, utiliza el formulario de soporte de API Policy.

En la medida en que tengas permitido distribuir contenido de X a un tercero, ten en cuenta que este contenido sigue sujeto al Developer Agreement y a la Developer Policy, y esos terceros deben aceptar los [Terms of Service](https://twitter.com/en/tos) de X, la [Privacy Policy](https://twitter.com/en/privacy), el [Developer Agreement](/es/developer-terms/agreement) y la [Developer Policy](/es/developer-terms/policy) antes de recibir contenido de X.

<div id="multiple-applications">
  ## Varias aplicaciones
</div>

No se permite registrar varias aplicaciones para un solo caso de uso, ni para casos de uso sustancialmente similares o superpuestos. Obtén más información sobre estas políticas [aquí](https://twittercommunity.com/t/policy-clarification-multiple-applications-for-the-same-use-case/92080).

En este contexto, definimos “caso de uso” como un conjunto coherente de análisis, visualizaciones o acciones realizadas a través de una aplicación. Ofrecer el mismo servicio o aplicación a distintos usuarios finales cuenta como un único caso de uso; sin embargo, las versiones &quot;white label&quot; de una herramienta o servicio no están incluidas. Si planeas ofrecer versiones &quot;white label&quot; de tu aplicación, debes notificárnoslo y obtener de nosotros una aprobación por separado. Estas reglas se aplican tanto a las aplicaciones que registras tú como a las aplicaciones registradas por los usuarios finales de tu herramienta o servicio; exigir que tus usuarios finales registren aplicaciones con el fin de utilizar tu herramienta o servicio podría dar lugar a medidas de aplicación en tu contra, contra tus aplicaciones, tus clientes y/o los usuarios finales de tu herramienta o servicio.

La única excepción a esta regla es crear instancias de desarrollo (“dev”), pruebas (“staging”) y producción (“prod”) del mismo servicio. Asegúrate de que estas aplicaciones estén claramente etiquetadas (por ejemplo, en el nombre o la descripción de la aplicación), y de no utilizar aplicaciones de desarrollo o de pruebas con fines de producción.

<div id="automation-spam-and-auto-responses">
  ## Automatización, spam y respuestas automáticas
</div>

Está prohibido el uso de las API y los productos para desarrolladores de X para crear spam o participar en comportamientos de tipo spam. Debes revisar las [Reglas de X](https://t.co/rules) sobre spam y asegurarte de que tu aplicación no infrinja nuestras políticas ni permita a los usuarios infringirlas.

Si tu aplicación se utilizará para realizar acciones de escritura en el servicio de X, incluyendo publicar Publicaciones, seguir cuentas o enviar Mensajes Directos, debes revisar cuidadosamente las [Reglas de automatización](https://support.twitter.com/articles/76915) para asegurarte de que tu servicio cumpla con nuestras pautas. En particular, debes:

* Obtener siempre el consentimiento explícito de un usuario antes de enviarle [respuestas o mensajes automatizados](https://twittercommunity.com/t/policy-clarification-automated-replies-and-mentions/94444)
* Respetar de inmediato las solicitudes de los usuarios de dejar de recibir comunicaciones tuyas
* Nunca realizar acciones masivas, agresivas o de tipo spam, incluido el [seguimiento masivo](https://twittercommunity.com/t/policy-clarification-aggressive-following-and-inorganic-following-behavior/92769)
* Nunca publicar contenido idéntico o sustancialmente similar en [múltiples cuentas](https://blog.twitter.com/developer/en_us/topics/tips/2018/automation-and-the-use-of-multiple-accounts.html)

<div id="measuring-the-x-service">
  ## Medir el servicio de X
</div>

No utilices las X API para medir la disponibilidad, el rendimiento, la funcionalidad ni el uso de X con fines de evaluación comparativa o con fines competitivos. Por ejemplo, nunca debes usar las X API para:

* Calcular métricas agregadas de usuarios de X, como el número total de usuarios o cuentas activas
* Calcular métricas agregadas de Publicaciones en X, como el número total de Publicaciones por día, o la cantidad de interacciones de usuarios o de cuentas
* Medir o analizar el spam o la seguridad en X, salvo lo permitido en las [Reglas de X](https://help.twitter.com/en/rules-and-policies/twitter-rules)

<div id="surveillance-privacy-and-user-protection">
  ## Vigilancia, privacidad y protección de los usuarios
</div>

En X, la protección y defensa de la privacidad de nuestros usuarios forma parte del ADN mismo de nuestra empresa, y nuestros productos de datos y para desarrolladores reflejan ese compromiso. Creemos que los datos de X pueden ser una fuerza poderosa para el bien en el mundo — desde salvar vidas durante inundaciones en Yakarta hasta ayudar al USGS a rastrear terremotos o trabajar con la ONU para lograr los Objetivos de Desarrollo Sostenible. Sin embargo, prohibimos el uso de datos de X y de las X APIs por parte de cualquier entidad con fines de vigilancia, o de cualquier otra forma que sea incompatible con las expectativas razonables de privacidad de nuestros usuarios. Sin excepciones.

Describimos los usos prohibidos de nuestros datos y productos para desarrolladores en el [Acuerdo para Desarrolladores](/es/developer-terms/agreement), incluidas las prohibiciones de investigar o rastrear a usuarios de X o su contenido, así como de rastrear, alertar o monitorear eventos sensibles (como protestas, manifestaciones o reuniones de organización comunitaria).

Otras categorías de actividades prohibidas en virtud de estos términos incluyen (entre otras):

* Investigar o rastrear a grupos y organizaciones sensibles, como sindicatos o grupos de activistas
* Verificaciones de antecedentes o cualquier forma de escrutinio extremo
* Análisis de riesgo crediticio o de seguros
* Elaboración de perfiles individuales o segmentación psicográfica
* Reconocimiento facial

Estas políticas se aplican a todos los usuarios de nuestras APIs. Cualquier uso indebido de las X APIs para estos fines podrá dar lugar a medidas de aplicación, que pueden incluir la suspensión y la revocación del acceso.

Para obtener información adicional destinada a las autoridades encargadas de hacer cumplir la ley que soliciten información sobre cuentas de X, visita [https://t.co/le](https://t.co/le).

Además, en este momento, X prohíbe cualquier uso de las X APIs y/o Contenido de X para ajustar (fine-tune) o entrenar un modelo fundacional o de vanguardia, con la excepción de [Grok](https://help.x.com/en/using-x/about-grok).
