<div id="likes-lookup-standard-v11-compared-to-x-api-v2">
  ### Búsqueda de Me gusta: estándar v1.1 en comparación con X API v2
</div>

Si has estado trabajando con el endpoint estándar v1.1 [GET favorites/list](https://developer.x.com/en/docs/x-api/v1/tweets/post-and-engage/api-reference/get-favorites-list), el objetivo de esta guía es ayudarte a entender las similitudes y diferencias entre los endpoints de búsqueda de Me gusta de la versión estándar v1.1 y de X API v2.

Con v2, también hemos introducido un nuevo endpoint de usuarios que dieron Me gusta, que te permite obtener información sobre los usuarios que dieron Me gusta a una Publicación.

* **Similitudes**
  * Autenticación
  * Límites de frecuencia
* **Diferencias**
  * URL de endpoints

  * Limitaciones de las solicitudes

  * Requisitos de la Aplicación y del Proyecto

  * Parámetros de las solicitudes

  * Nuevo formato JSON

<div id="similarities">
  #### Similitudes
</div>

**Autenticación**

Tanto los endpoints de consulta de Likes de la versión estándar v1.1 como de X API v2 usan [Contexto de usuario de OAuth 1.0a](/es/resources/fundamentals/authentication#oauth-1-0a-2) o [Bearer Token de OAuth 2.0](/es/resources/fundamentals/authentication#bearer-token-also-known-as-app-only). Por lo tanto, si antes utilizabas los endpoints estándar v1.1 [GET favorites/list endpoints](https://developer.x.com/en/docs/x-api/v1/tweets/post-and-engage/api-reference/get-favorites-list), puedes seguir usando el mismo método de autenticación si decides migrar a la versión de X API v2. 

Dependiendo de tu biblioteca o paquete de autenticación preferido, la autenticación con Bearer Token probablemente sea la forma más sencilla de empezar y se puede configurar con un simple encabezado de solicitud. Para aprender cómo generar un Bearer Token, consulta [esta guía de Bearer Token de OAuth 2.0](/es/resources/fundamentals/authentication#bearer-token-also-known-as-app-only). 
 

**Límites de frecuencia**

El endpoint estándar v1.1 GET favorites/list tiene un límite de frecuencia de 75 solicitudes por 15 minutos por usuario. El endpoint correspondiente de Publicaciones que te han gustado en v2 también tiene este mismo límite de frecuencia. Sin embargo, este endpoint de v2 también tiene un límite de frecuencia adicional de 75 solicitudes por 15 minutos por Aplicación.

<div id="differences">
  #### Diferencias
</div>

**URLs de endpoints**

* Endpoints estándar v1.1:
  * GET https://api.x.com/1.1/favorites/list.json
    (lista de Publicaciones realizadas por el usuario especificado)
  * No existe un equivalente en v1.1 para el endpoint v2 de usuarios que indican &quot;me gusta&quot;
* Endpoint de X API v2:
  * GET https://api.x.com/2/users/:id/liked&#95;tweets
    (lista de Publicaciones a las que el usuario con la id especificada les ha dado &quot;me gusta&quot;)
  * GET https://api.x.com/2/tweets/:id/liking&#95;users
    (lista de usuarios que le dieron &quot;me gusta&quot; a la Publicación con la id especificada)

**Limitaciones de las solicitudes**

El endpoint v2 de Publicaciones con &quot;me gusta&quot; te permite solicitar de 5 a 100 Publicaciones por solicitud, pero puedes solicitar todos los &quot;me gusta&quot; de una Publicación usando tokens de paginación. El endpoint v1.1 GET favorites/list también te permite recuperar todos los &quot;me gusta&quot; de Publicaciones, pero puedes recuperar de 20 a 200 Publicaciones por solicitud.

Para el endpoint v2 de usuarios que dan &quot;me gusta&quot;, estás limitado a 100 usuarios que dan &quot;me gusta&quot; por Publicación. 
 

**Requisitos de Aplicación y Proyecto**

Los endpoints de X API v2 requieren que uses credenciales de una [developer App](/es/resources/fundamentals/developer-apps) que esté asociada a un [Proyecto](/es/resources/fundamentals/developer-apps) al autenticar tus solicitudes. Todos los endpoints de X API v1.1 pueden usar credenciales de Apps independientes o de Apps asociadas a un Proyecto.

**Parámetros de la solicitud**

Los siguientes parámetros de solicitud estándar de la v1.1 aceptaban dos parámetros de consulta (user&#95;id o screen&#95;name). X API v2 solo acepta la id numérica del usuario, y debe pasarse como parte de la ruta del endpoint.

Una de las mayores diferencias entre las versiones de endpoints estándar v1.1 y X API v2 es cómo seleccionas qué campos se devuelven en tu carga útil. Para los endpoints estándar, había varios parámetros que podías usar para identificar qué campos o conjuntos de campos se devolverían en la carga útil, mientras que la versión de X API v2 simplifica estos diferentes parámetros en [fields](/es/x-api/fundamentals/fields) y [expansions](/es/x-api/fundamentals/expansions). 
 

**Nuevo formato JSON**

X API v2 introduce nuevos diseños JSON para los objetos devueltos por las APIs, incluidos los objetos de [Post](/es/x-api/fundamentals/data-dictionary#tweet) y de [user](/es/x-api/fundamentals/data-dictionary#user).

* En el nivel raíz de JSON, los endpoints estándar devuelven objetos de usuario en un array statuses, mientras que X API v2 devuelve un array data. 
* En lugar de hacer referencia a &quot;statuses&quot; Retweeted y Quoted, el JSON de X API v2 se refiere a Publicaciones Retweeted y Quoted. Muchos campos heredados y obsoletos, como contributors y user.translator&#95;type, se están eliminando. 
* En lugar de usar tanto favorites (en el objeto Post) como favourites (en el objeto user), X API v2 usa el término like. 
* X está adoptando la convención de que los valores JSON sin valor (por ejemplo, null) no se escriben en la carga útil. Los atributos de Post y user solo se incluyen si tienen valores no nulos.
   

Además de los cambios que hicimos en nuestro nuevo formato JSON, también introdujimos un nuevo conjunto de campos en el objeto Post, que incluye lo siguiente:

* [conversation&#95;id](/es/x-api/fundamentals/conversation-id)
* Dos nuevos campos de [Post annotations](/es/x-api/fundamentals/post-annotations), incluidos context y entities
* Varios campos nuevos de [metrics](/es/x-api/fundamentals/metrics)
