<div id="likes-lookup-standard-v11-compared-to-x-api-v2">
  ### Búsqueda de Me gusta: estándar v1.1 en comparación con X API v2
</div>

Si has estado trabajando con el endpoint estándar v1.1 [GET favorites/list](https://developer.x.com/en/docs/x-api/v1/tweets/post-and-engage/api-reference/get-favorites-list), el objetivo de esta guía es ayudarte a comprender las similitudes y diferencias entre los endpoints de búsqueda de Me gusta de la API estándar v1.1 y de X API v2.

Con v2, también hemos introducido un nuevo endpoint de usuarios que dieron Me gusta, que te permite obtener información sobre los usuarios que dieron Me gusta a una Publicación.

* **Similitudes**
  * Autenticación
  * Límites de frecuencia
* **Diferencias**
  * URL de los endpoints

  * Limitaciones de las solicitudes

  * Requisitos de Aplicación y Proyecto

  * Parámetros de las solicitudes

  * Nuevo formato JSON

<div id="similarities">
  #### Similitudes
</div>

**Autenticación**

Tanto los endpoints de consulta de Me gusta de la versión estándar v1.1 como de X API v2 usan [Contexto de usuario de OAuth 1.0a](/es/resources/fundamentals/authentication#oauth-1-0a-2) o [OAuth 2.0 Bearer Token](/es/resources/fundamentals/authentication#bearer-token-also-known-as-app-only). Por lo tanto, si antes utilizabas los endpoints estándar v1.1 [GET favorites/list](https://developer.x.com/en/docs/x-api/v1/tweets/post-and-engage/api-reference/get-favorites-list), puedes seguir utilizando el mismo método de autenticación si migras a la versión de X API v2, si lo deseas. 

Dependiendo de la biblioteca o paquete de autenticación que elijas, la autenticación con Bearer Token probablemente sea la forma más sencilla de empezar y puede configurarse con un encabezado de solicitud sencillo. Para aprender a generar un Bearer Token, consulta [esta guía de OAuth 2.0 Bearer Token](/es/resources/fundamentals/authentication#bearer-token-also-known-as-app-only). 
 

**Límites de frecuencia**

El endpoint estándar v1.1 GET favorites/list tiene un límite de frecuencia de 75 solicitudes cada 15 minutos por usuario. El endpoint de Publicaciones con Me gusta correspondiente en v2 también tiene este mismo límite de frecuencia. Sin embargo, este endpoint de v2 también tiene un límite de frecuencia adicional de 75 solicitudes cada 15 minutos por aplicación.

<div id="differences">
  #### Diferencias
</div>

**URLs de los endpoints**

* Endpoints estándar v1.1:
  * GET https://api.x.com/1.1/favorites/list.json
    (lista de Publicaciones que el usuario especificado ha marcado con &quot;me gusta&quot;)
  * No existe un endpoint v1.1 equivalente al endpoint v2 liking users
* Endpoints de X API v2:
  * GET https://api.x.com/2/users/:id/liked&#95;tweets
    (lista de Publicaciones que han sido marcadas con &quot;me gusta&quot; por el ID de usuario especificado)
  * GET https://api.x.com/2/tweets/:id/liking&#95;users
    (lista de usuarios que marcaron con &quot;me gusta&quot; la Publicación con el ID especificado)

**Limitaciones de la solicitud**

El endpoint v2 liked Posts te permite solicitar de 5 a 100 Publicaciones por solicitud, pero puedes solicitar todos los &quot;me gusta&quot; de una Publicación usando tokens de paginación. El endpoint v1.1 GET favorites/list también te permite obtener todos los &quot;me gusta&quot; de las Publicaciones, pero puedes extraer de 20 a 200 Publicaciones por solicitud.

Para el endpoint v2 liking users, estás limitado a 100 usuarios que dieron &quot;me gusta&quot; por Publicación. 
 

**Requisitos de la Aplicación y el Proyecto**

Los endpoints de X API v2 requieren que utilices credenciales de una [developer App](/es/resources/fundamentals/developer-apps) que esté asociada con un [Project](/es/resources/fundamentals/developer-apps) al autenticar tus solicitudes. Todos los endpoints de X API v1.1 pueden usar credenciales de Aplicaciones independientes o Aplicaciones asociadas con un Project.

**Parámetros de la solicitud**

Los siguientes parámetros de solicitud estándar v1.1 aceptaban dos parámetros de consulta (user&#95;id o screen&#95;name). X API v2 solo acepta el ID numérico de usuario, y debe pasarse como parte de la ruta del endpoint.

Una de las mayores diferencias entre las versiones de endpoints estándar v1.1 y X API v2 es cómo seleccionas qué campos se devuelven en tu payload. Para los endpoints estándar, había varios parámetros que podías usar para identificar qué campos o conjuntos de campos se devolverían en el payload, mientras que la versión X API v2 simplifica estos diferentes parámetros en [fields](/es/x-api/fundamentals/fields) y [expansions](/es/x-api/fundamentals/expansions). 
 

**Nuevo formato JSON**

X API v2 está introduciendo nuevos diseños JSON para los objetos devueltos por las API, incluidos los objetos de [Post](/es/x-api/fundamentals/data-dictionary#tweet) y de [user](/es/x-api/fundamentals/data-dictionary#user).

* En el nivel raíz de JSON, los endpoints estándar devuelven objetos de usuario en un array statuses, mientras que X API v2 devuelve un array data. 
* En lugar de hacer referencia a &quot;statuses&quot; Retweeted y Quoted, el JSON de X API v2 hace referencia a Publicaciones retuiteadas y citadas. Muchos campos heredados y obsoletos, como contributors y user.translator&#95;type, se están eliminando. 
* En lugar de usar tanto favorites (en el objeto Post) como favourites (en el objeto user), X API v2 usa el término like. 
* X está adoptando la convención de que los valores JSON sin valor (por ejemplo, null) no se incluyen en el payload. Los atributos de Post y user solo se incluyen si tienen valores distintos de null.
   

Además de los cambios que realizamos en nuestro nuevo formato JSON, también incorporamos un nuevo conjunto de campos al objeto Post, incluidos los siguientes:

* [conversation&#95;id](/es/x-api/fundamentals/conversation-id)
* Dos nuevos campos de [Post annotations](/es/x-api/fundamentals/post-annotations), incluidos context y entities
* Varios campos nuevos de [metrics](/es/x-api/fundamentals/metrics)
