<div id="standard-v11-timelines-to-x-api-v2-timelines">
  ## Cronologías estándar de v1.1 a cronologías de X API v2
</div>

Si has estado trabajando con los endpoints de cronologías de v1.1 (statuses/user&#95;timeline y statuses/mentions&#95;timeline), el objetivo de esta guía es ayudarte a entender las similitudes y diferencias entre los endpoints de cronologías estándar y los de X API v2, para que puedas migrar tu integración actual a la nueva versión.

* **Similitudes:**
  * Autenticación:
    * Contexto de usuario de OAuth 1.0a (cronología de inicio en orden cronológico inverso, cronología de Publicaciones de usuario y cronología de menciones de usuario)
    * OAuth 2.0 solo para la aplicación (cronología de Publicaciones de usuario)
  * Límite de acceso histórico: la cronología de usuario (cronología de Publicaciones de usuario) proporciona acceso a las 3200 Publicaciones más recientes; la cronología de menciones (cronología de menciones de usuario) proporciona acceso a las 800 menciones más recientes.
  * Compatibilidad con historial de edición de la Publicación y metadatos
  * Límites de frecuencia (cronología de Publicaciones de usuario)
  * Sondeo de actualización: posibilidad de obtener nuevos resultados desde el since&#95;id
  * Recorrido de cronologías mediante ID de Publicación
  * Especificaciones de resultados:
    * Orden de resultados: resultados devueltos en orden cronológico inverso
    * Posibilidad de excluir respuestas (solo cronología de Publicaciones de usuario)
    * Posibilidad de excluir Retweets (solo cronología de Publicaciones de usuario)
* **Diferencias**
  * Nueva capacidad de autenticación:
    * OAuth 2.0 solo para la aplicación (cronología de menciones de usuario)
    * Flujo de código de autorización de OAuth 2.0 con PKCE (cronología de inicio en orden cronológico inverso, cronología de Publicaciones de usuario y cronología de menciones de usuario)
  * Requisitos de acceso: requisitos de Aplicación y de Proyecto de X API v2
  * Límites de frecuencia (cronología de menciones de usuario y cronología de inicio en orden cronológico inverso)
  * Diferente límite de solicitudes y límite de volumen
  * Método de paginación adicional
    * Diferente max&#95;results (cantidad) por respuesta
  * Formato de datos de respuesta
  * Parámetros de la solicitud
    * Formato de datos personalizado en función de los parámetros de la solicitud, incluidos los campos y expansiones de v2
    * Datos adicionales disponibles: métricas, anotaciones de Publicaciones, encuestas

<div id="similarities">
  ### Similitudes
</div>

**Autenticación**

Los endpoints v1.1 `statuses/user_timeline` y la cronología de Publicaciones de usuario de X API v2 admiten tanto [Contexto de usuario de OAuth 1.0a](/es/resources/fundamentals/authentication) como [OAuth 2.0 solo para la aplicación](/es/resources/fundamentals/authentication#bearer-token-also-known-as-app-only). Por lo tanto, puedes seguir usando el mismo método de autenticación y los mismos tokens de autorización si migras a la versión de X API v2.

**Acceso histórico**

Los endpoints v1.1 `statuses/user_timeline` y la cronología de Publicaciones de usuario de X API v2 devolverán ambos las 3200 Publicaciones más recientes, incluidas las republicaciones (Retweets).

Los endpoints v1.1 `statuses/mentions_timeline` y la cronología de menciones de usuario de X API v2 pueden devolver las 800 Publicaciones más recientes.

**Compatibilidad con el historial de edición de la Publicación y metadatos**

Ambas versiones proporcionan metadatos que describen cualquier historial de edición. Consulta las [Referencias de la API de flujo filtrado](/es/x-api/posts/filtered-stream#api-reference-index) y la [página de fundamentos de edición de Publicaciones](/es/x-api/fundamentals/edit-posts) para obtener más detalles.

**Límites de frecuencia**

|     |     |
| :--- | :--- |
| **Standard v1.1** | **X API v2** |
| user&#95;timeline:<br /><br />900 solicitudes por 15 min con Contexto de usuario de OAuth 1.0a<br /><br />1500 solicitudes por 15 min con OAuth 2.0 solo para la aplicación | Cronología de Publicaciones de usuario:<br /><br />900 solicitudes por ventana de 15 minutos con Contexto de usuario de OAuth 1.0a<br /><br />1500 solicitudes por ventana de 15 minutos con OAuth 2.0 solo para la aplicación |

**Actualización mediante sondeo usando since&#95;id**

Ambas versiones permiten realizar sondeos para obtener resultados recientes utilizando el since&#95;id.

**Recorrido de cronologías por ID de Publicación**

Ambos endpoints tienen la capacidad de recorrer cronologías usando las marcas de tiempo de los ID de Publicación, según la forma en que se construyen los ID de Publicación. La funcionalidad es, en general, la misma excepto por lo siguiente:

|     |     |
| :--- | :--- |
| **Standard timelines v1.1** | **timelines v2** |
| since&#95;id (exclusivo)<br /><br />max&#95;id (inclusivo) | since&#95;id (exclusivo)<br /><br />until&#95;id (también exclusivo, frente a max&#95;id, que era inclusivo) |

**Parámetros de filtrado de respuesta**

|     |     |
| :--- | :--- |
| **Standard timelines v1.1** | **timelines v2** |
| Parámetros de filtrado de respuesta:<br /><br />* include&#95;rts<br />* exclude&#95;replies | Parámetros de filtrado de respuesta:<br /><br />* exclude=retweets,replies |
| Ejemplo <br /><br />https://api.x.com/1.1/statuses/user&#95;timeline.json?user&#95;id=2244994945&amp;include&#95;rts=0&amp;&amp;exclude&#95;replies=1 | Ejemplo:<br /><br />https://api.x.com/2/users/2244994945/tweets?max&#95;results=100&amp;exclude=retweets,replies |
| Notas:<br /><br />Para user&#95;timeline:<br /><br />* Usar include&#95;rts=0 no modifica el posible límite histórico de Publicaciones de las 3200 más recientes | Notas:<br /><br />Para la cronología de Publicaciones de usuario:<br /><br />* Usar exclude=retweets no modifica el posible límite histórico de Publicaciones de las 3200 más recientes <br />* Usar exclude=replies reduce el posible límite histórico de Publicaciones a las 800 respuestas más recientes |

<div id="differences">
  ### Diferencias
</div>

**Autenticación**

**El endpoint statuses/mentions&#95;timeline de v1.1 solo admite [OAuth 1.0a User Context](https://developer-staging.x.com/resources/fundamentals/authentication). El endpoint user mention timeline de X API v2 admite [OAuth 1.0a User Context](/es/resources/fundamentals/authentication), [OAuth 2.0 App-Only](/es/resources/fundamentals/authentication#bearer-token-also-known-as-app-only) y [OAuth 2.0 Authorization Code with PKCE](/es/resources/fundamentals/authenticationoauth-2-0/authorization-code "Este método permite que una app autorizada actúe en nombre del usuario, como si fuera el propio usuario. Normalmente se utiliza para acceder o publicar información pública para un usuario específico, y es útil cuando tu app necesita conocer la relación entre un usuario y lo que devuelve este endpoint. Haz clic para aprender a autenticarte con OAuth 2.0 Authorization Code with PKCE.") **

Si quieres aprovechar la posibilidad de acceder a métricas privadas o promocionadas mediante el endpoint de X API v2 user Post timeline, deberás usar OAuth 1.0a User Context o OAuth 2.0 Authorization Code with PKCE y pasar los tokens de acceso de usuario correspondientes al usuario que publicó la Publicación para la que quieres acceder a las métricas.
 

**URLs de endpoints**

Ten en cuenta que los endpoints de timelines de X API v2 requieren un parámetro de ruta :id para el ID de usuario.

* Endpoints estándar v1.1:
  * https://api.x.com/1.1/statuses/home&#95;timeline
  * https://api.x.com/1.1/statuses/user&#95;timeline
  * https://api.x.com/1.1/statuses/mention&#95;timeline
* Endpoint de X API v2:
  * https://api.x.com/2/users/:id/timelines/reverse&#95;chronological
  * https://api.x.com/2/users/:id/tweets
  * https://api.x.com/2/users/:id/mentions

**Requisitos de App y Proyecto**

Los endpoints de X API v2 requieren que uses credenciales de una [developer App](/es/resources/fundamentals/developer-apps) que esté asociada a un [Project](/es/resources/fundamentals/projects) al autenticar tus solicitudes. Todos los endpoints de X API v1.1 pueden usar credenciales de Apps independientes o de Apps asociadas a un proyecto. 
 

**Límites de frecuencia**

|     |     |
| :--- | :--- |
| **mentions&#95;timeline:**<br /><br />75 solicitudes por 15 minutos con OAuth 1.0a User Context | **user mention timeline:**<br /><br />180 solicitudes por ventana de 15 minutos con OAuth 1.0a User Context  <br />450 solicitudes por ventana de 15 minutos con OAuth 2.0 Bearer Token |
| **home&#95;timelime:**<br /><br />15 solicitudes cada 15 minutos  <br /><br />Se pueden obtener hasta 800 Publicaciones en el home timeline | **reverse chronological home timeline:**<br /><br />180 solicitudes cada 15 minutos<br /><br />Puedes recuperar todas las Publicaciones creadas en una timeline durante los últimos 7 días, así como las 800 más recientes independientemente de su fecha de creación. |

| Límite diario de solicitudes<br /><br />* Límite máximo de 100,000 solicitudes en un período de 24 horas. | Sin límites diarios de solicitudes, solo limitado por el volumen de consumo. |

**Límites de volumen de consumo**

|     |     |
| :--- | :--- |
| **Standard timelines v1.1** | **timelines v2** |

**Parámetros de solicitud**

|     |     |
| :--- | :--- |
| **Standard timelines v1.1** | **Timelines v2** |
| Required: user&#95;id or screen&#95;name | Requerido: el ID de usuario específico se especifica en el parámetro de ruta |
| Optional:<br /><br />count - establece la cantidad máxima de resultados devueltos por solicitud<br /><br />exclude&#95;replies - elimina las respuestas de los resultados<br /><br />Include&#95;rts - cuando se establece en 0 elimina los Retweets de los resultados<br /><br />trim&#95;user - elimina los objetos de usuario rehidratados de los resultados<br /><br />tweet&#95;mode - establece el formato de datos devuelto para los resultados, establézcalo en extended para Publicaciones de más de 140 caracteres<br /><br />since&#95;id - establece el ID de la Publicación más antigua en el resultado (exclusivo)<br /><br />max&#95;id - establece el ID de la Publicación más reciente en el resultado (inclusivo) | Opcional:<br /><br />max&#95;results - establece la cantidad máxima de resultados devueltos por solicitud<br /><br />exclude=retweets,replies - elimina Retweets o respuestas de los resultados<br /><br />tweet.fields - establece los campos del objeto de Publicación que se devolverán<br /><br />user.fields - establece los campos del objeto de usuario que se devolverán<br /><br />place.fields - establece los campos del objeto de lugar que se devolverán<br /><br />media.fields - establece los campos del objeto de contenido multimedia que se devolverán<br /><br />poll.fields - establece los campos del objeto de encuesta que se devolverán<br /><br />expansions - establece los campos y datos expandidos que se devolverán<br /><br />start&#95;time - establece la hora created&#95;at más antigua para los resultados<br /><br />end&#95;time - establece la hora created&#95;at más reciente para los resultados<br /><br />since&#95;id - establece el ID de la Publicación más antigua para los resultados (exclusivo)<br /><br />until&#95;id - establece el ID de la Publicación más reciente en el resultado (exclusivo) |

**Formato JSON de X API v2**

X API v2 introduce nuevos diseños JSON para los objetos devueltos por las API, incluidos los objetos de [Post](/es/x-api/fundamentals/data-dictionary#tweet) y de [user](/es/x-api/fundamentals/data-dictionary#user). Puedes obtener más información sobre el formato de X API v2 y cómo usar campos y expansiones visitando nuestra [guía](/es/x-api/fundamentals/data-dictionary#how-to-use-fields-and-expansions) y leyendo nuestro [diccionario de datos](/es/x-api/fundamentals/data-dictionary).

* En el nivel raíz de JSON, los endpoints estándar devuelven objetos de Publicación en una matriz statuses, mientras que X API v2 devuelve una matriz data. 
* En lugar de hacer referencia a &quot;statuses&quot; Retweeted y Quoted, el JSON de X API v2 hace referencia a Publicaciones Retuiteadas y Citadas. Muchos campos heredados y en desuso, como contributors y user.translator&#95;type, se están eliminando. 
* En lugar de usar favorites (en el objeto de Publicación) y favorites (en el objeto de usuario), X API v2 usa el término like. 
* X está adoptando la convención de que los valores JSON sin valor (por ejemplo, null) no se escriben en la carga útil. Los atributos de Publicación y de usuario solo se incluyen si tienen un valor distinto de null.
   

Una de las diferencias más importantes entre las versiones de endpoints estándar v1.1 y X API v2 es cómo seleccionas qué campos se devuelven en tu carga útil. Para los endpoints estándar, había varios parámetros que podías usar para identificar qué campos o conjuntos de campos se devolverían en la carga útil, mientras que la versión de X API v2 simplifica estos diferentes parámetros en [campos](/es/x-api/fundamentals/fields) y [expansiones](/es/x-api/fundamentals/expansions). 

* campos: los endpoints de X API v2 te permiten seleccionar qué campos se incluyen en tu payload. Por ejemplo, los objetos de Publicación, usuario, contenido multimedia, lugar y encuesta tienen cada uno una lista de campos que se pueden devolver (o no). 

* expansiones: se utilizan para expandir los objetos complementarios a los que se hace referencia en los payloads JSON de Publicación. Por ejemplo, todas las republicaciones y respuestas hacen referencia a otras Publicaciones. Al establecer expansions=referenced&#95;tweets.id, estos otros objetos de Publicación se expanden según la configuración de tweet.fields. Otros objetos, como usuarios, encuestas y contenido multimedia, también se pueden expandir. 

* conversation&#95;id

* Dos nuevos campos de [anotaciones](/es/x-api/fundamentals/post-annotations), incluidos los campos context y entities

* Varios campos nuevos de [métricas](/es/x-api/fundamentals/metrics) 
   

Hemos preparado una [guía de migración de formato de datos](/es/x-api/migrate/data-format-migration#migrating-from-standard-v1-1s-data-format-to-v2) que puede ayudarte a asignar los campos estándar de v1.1 a los campos más recientes de v2. Esta guía también te proporcionará el parámetro específico de expansión y de campo que deberás pasar con tu solicitud a v2 para devolver campos específicos.
