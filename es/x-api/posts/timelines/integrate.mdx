<div id="helpful-tools">
  ### Herramientas útiles
</div>

Antes de explorar algunos conceptos clave, te recomendamos que uses una de las siguientes herramientas o ejemplos de código para empezar a probar la funcionalidad de estos endpoints.

**Ejemplos de código**

¿Te interesa configurar estos endpoints con algo de código en tu lenguaje de programación preferido? Tenemos varios ejemplos de código disponibles que puedes usar como punto de partida en nuestra [página de GitHub](https://github.com/xdevplatform/Twitter-API-v2-sample-code).

**Librerías**

Aprovecha una de nuestras muchas [librerías de terceros de la comunidad](/es/x-api/tools-and-libraries/overview) para ayudarte a empezar. Puedes encontrar una librería que funcione con los endpoints de la v2 buscando la etiqueta de versión adecuada.

**Postman**

Postman es una excelente herramienta que puedes usar para probar estos endpoints. Cada solicitud de Postman incluye todos los parámetros de ruta y de cuerpo para ayudarte a entender rápidamente todo lo que tienes a tu disposición. Para obtener más información sobre nuestras colecciones de Postman, visita nuestra página [Uso de Postman](/es/tutorials/postman-getting-started).

<div id="key-concepts">
  ### Conceptos clave
</div>

**Autenticación**

Todos los endpoints de X API v2 requieren que las solicitudes estén [autenticadas](/es/resources/fundamentals/authentication) con un conjunto de credenciales, también conocidas como claves y tokens. Puedes usar Contexto de usuario de OAuth 1.0a u OAuth 2.0 Authorization Code with PKCE para autenticar solicitudes a estos endpoints. Puedes usar OAuth 2.0 solo para la aplicación para la cronología de Publicaciones de usuario y la cronología de menciones de usuario.

[Contexto de usuario de OAuth 1.0a](/es/resources/fundamentals/authentication#oauth-1-0a-2) requiere que utilices tus claves de API, tokens de acceso de usuario y algunos otros parámetros para [crear un encabezado de autorización](https://developer-staging.x.com/resources/fundamentals/authentication/authorizing-a-request), que luego enviarás con tu solicitud. Los tokens de acceso deben estar asociados con el usuario en cuyo nombre estás haciendo la solicitud. Si deseas generar un conjunto de tokens de acceso para otro usuario, ese usuario debe autorizar tu aplicación usando el [flujo OAuth de tres patas](https://developer-staging.x.com/resources/fundamentals/authentication/obtaining-user-access-tokens). 

Ten en cuenta que OAuth 1.0a puede ser difícil de usar. Si no estás familiarizado con este método de autenticación, te recomendamos que uses una [biblioteca](/es/x-api/tools-and-libraries/overview), utilices una herramienta como Postman o uses OAuth 2.0 para autenticar tus solicitudes. Si deseas solicitar una Publicación o métricas privadas desde estos endpoints, deberás usar Contexto de usuario de OAuth 1.0a u OAuth 2.0 Authorization Code with PKCE, lo que garantizará que tengas los permisos adecuados del usuario propietario de ese contenido.

[OAuth 2.0 solo para la aplicación](/es/resources/fundamentals/authentication#bearer-token-also-known-as-app-only) solo requiere que envíes un [OAuth 2.0 App Access Token](/es/resources/fundamentals/authentication#bearer-token-also-known-as-app-only) con tu solicitud. Puedes generar un App Access Token directamente desde una aplicación de desarrollador o generar uno usando el endpoint [POST oauth2/token](/es/resources/fundamentals/authentication#post-oauth2-token). Puedes usarlo para la cronología de Publicaciones de usuario o la cronología de menciones de usuario.

[OAuth 2.0 Authorization Code with PKCE](/es/resources/fundamentals/authentication#oauth-2-0-authorization-code-flow-with-pkce-3) permite un mayor control sobre el alcance de una aplicación y flujos de autorización en varios dispositivos. OAuth 2.0 te permite elegir scopes específicos y granulares que te otorgan permisos específicos en nombre de un usuario. 

Para habilitar OAuth 2.0 en tu aplicación, debes activarlo en la configuración de autenticación de tu aplicación, que se encuentra en la sección de configuración de la aplicación del portal para desarrolladores.

<Info>
  **Ten en cuenta**

  Si solicitas los siguientes [campos](/es/x-api/fundamentals/fields), se requiere Contexto de usuario de OAuth 1.0a u OAuth 2.0 Authorization Code: 

  * `tweet.fields.non_public_metrics`
  * `tweet.fields.promoted_metrics`
  * `tweet.fields.organic_metrics`
  * `media.fields.non_public_metrics`
  * `media.fields.promoted_metrics`
  * `media.fields.organic_metrics`
</Info>

**[](/es/x-api/fundamentals/pagination)Portal para desarrolladores, Proyectos y aplicaciones de desarrollador**

Para trabajar con cualquier endpoint de X API v2, debes [registrarte para obtener una cuenta de desarrollador](https://developer.x.com/en/portal/petition/essential/basic-info), configurar un [Proyecto](/es/resources/fundamentals/projects) dentro de esa cuenta y crear una [aplicación de desarrollador](/es/resources/fundamentals/developer-apps) dentro de ese Proyecto. Tus claves y tokens dentro de esa aplicación de desarrollador funcionarán para estos endpoints de cronologías.

**Límites de frecuencia**

Todos los días, muchos miles de desarrolladores realizan solicitudes a X API. Para ayudar a gestionar el volumen, se establecen [límites de frecuencia](/es/x-api/fundamentals/rate-limits) en cada endpoint que limitan la cantidad de solicitudes que cada desarrollador puede realizar en nombre de una aplicación o en nombre de un usuario autenticado. 

Se aplican distintos límites de frecuencia para estos endpoints según el método de autenticación que se utilice. Los límites de frecuencia a nivel de aplicación se aplican a una aplicación que realiza solicitudes usando OAuth 2.0 solo para la aplicación, mientras que el límite de frecuencia a nivel de usuario se aplica a las solicitudes que se realizan en nombre del usuario que autoriza específicamente usando Contexto de usuario de OAuth 1.0a. Estos dos límites de frecuencia se basan en la cantidad de solicitudes dentro de una ventana de 15 minutos.

Por ejemplo, una aplicación que use autenticación OAuth 2.0 solo para la aplicación para ambos endpoints de timelines puede hacer 1500 solicitudes (incluidas las solicitudes de paginación) al timeline de Publicaciones de usuario y 450 solicitudes (incluidas las solicitudes de paginación) al timeline de menciones de usuario dentro de un período de 15 minutos. Esa misma aplicación, dentro del mismo período de 15 minutos, con dos usuarios autorizados diferentes (usando Contexto de usuario de OAuth 1.0a) puede hacer 900 solicitudes (incluidas las solicitudes de paginación) al timeline de Publicaciones de usuario y 180 solicitudes (incluidas las solicitudes de paginación) al timeline de menciones de usuario por cada usuario autenticado. 

El timeline de inicio en orden cronológico inverso tiene un límite de frecuencia por usuario de 180 solicitudes por ventana de 15 minutos. Con este endpoint puedes obtener todas las Publicaciones creadas en un timeline durante los últimos 7 días, así como las 800 más recientes, independientemente de la fecha de creación.

**Campos y expansiones**

La X API v2 te permite seleccionar exactamente qué datos quieres que se devuelvan desde la API usando [campos](/es/x-api/fundamentals/fields) y [expansiones](/es/x-api/fundamentals/expansions). El parámetro de expansión te permite expandir los objetos referenciados en la carga útil. Por ejemplo, este endpoint te permite solicitar objetos de tipo encuesta (poll), lugar (place), contenido multimedia y otros objetos usando el parámetro expansions.

El parámetro fields te permite seleccionar exactamente qué campos dentro de los diferentes objetos de datos quieres recibir. De forma predeterminada, el objeto principal de Publicación devuelto por estos endpoints incluye los campos id y text. Para recibir campos adicionales como author&#95;id o public&#95;metrics, tendrás que solicitarlos específicamente usando los parámetros fields. Algunos campos importantes que quizá quieras considerar en tu integración son nuestros datos de encuestas, las [métricas](/es/x-api/fundamentals/metrics), las [anotaciones de Publicaciones](/es/x-api/fundamentals/post-annotations) y los campos de [ID de conversación](/es/x-api/fundamentals/conversation-id).

Hemos añadido una guía sobre [cómo usar campos y expansiones](/es/x-api/fundamentals/data-dictionary#how-to-use-fields-and-expansions) juntos en nuestro [diccionario de datos de X API v2](/es/x-api/fundamentals/data-dictionary).

**Métricas de Publicaciones**

Los endpoints de X API v2 te permiten solicitar métricas de Publicaciones directamente desde el objeto de Publicación devuelto, siempre que incluyas los campos adecuados en tu solicitud.

Existen algunas limitaciones con las métricas de Publicaciones que debes tener en cuenta, específicamente relacionadas con la privacidad del usuario y los siguientes campos de la respuesta:

* `tweet.fields.non_public_metrics`
* `tweet.fields.promoted_metrics`
* `tweet.fields.organic_metrics`
* `media.fields.non_public_metrics`
* `media.fields.promoted_metrics`
* `media.fields.organic_metrics`

Los campos indicados incluyen datos de métricas privadas, lo que significa que debes estar autorizado por el autor de la Publicación para recuperar estos datos en su nombre cuando uses el endpoint del timeline de Publicaciones de usuario; es decir, debes usar [Contexto de usuario de OAuth 1.0a](/es/resources/fundamentals/authentication) o [flujo de código de autorización de OAuth 2.0 con PKCE](/es/resources/fundamentals/authentication#oauth-2-0-authorization-code-flow-with-pkce-3).

Por ejemplo, para recibir `non_public_metrics` para el timeline de Publicaciones de usuario del ID de usuario 1234 tendrás que incluir en tu solicitud los tokens de acceso asociados con ese usuario. Puedes hacer que los usuarios autoricen tu aplicación y recibir un conjunto de tokens de acceso asociados a ellos usando el [flujo OAuth de tres patas](/es/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow). 

Si estás usando el timeline de menciones de usuario, los campos indicados no estarán disponibles a menos que el autor que menciona haya autorizado tu aplicación a acceder a sus datos privados de métricas y estés usando los tokens de acceso de ese usuario al hacer la solicitud con Contexto de usuario de OAuth 1.0a.

Todas las `non_public_metrics`, organic&#95;metrics y promoted&#95;metrics solo están disponibles para las Publicaciones creadas en los últimos 30 días. Esto significa que, cuando solicites los campos indicados, los resultados se ajustarán automáticamente para incluir únicamente Publicaciones de los últimos 30 días.

Si se solicitan estos campos indicados, solo se devolverán las Publicaciones cuyo autor sea el usuario autenticado; todas las demás Publicaciones recibirán un mensaje de error.

**Paginación**

Estos endpoints utilizan paginación para que las respuestas se devuelvan rápidamente. En los casos en que haya más resultados de los que se pueden enviar en una sola respuesta (hasta 100 Publicaciones en los endpoints de timelines) tendrás que paginar. Usa el parámetro max&#95;results para identificar cuántos resultados se devolverán por página y el parámetro pagination&#95;token para devolver la página siguiente de resultados. Puedes obtener más información consultando nuestra [guía de paginación](/es/x-api/fundamentals/pagination).

**Filtrado de resultados**

Estos endpoints incluyen varios parámetros que puedes usar para filtrar los resultados. Con start&#95;date y end&#95;date puedes acotar los resultados a un periodo de tiempo específico. Si prefieres usar IDs de Publicaciones para seleccionar un conjunto específico de Publicaciones, puedes usar since&#95;id y until&#95;id. La cronología de Publicaciones de usuario también tiene un parámetro exclude que puede eliminar Retweets y respuestas de tus resultados. 

**Límites de Publicaciones y volumen de Publicaciones devueltas**

Los endpoints de la cronología de Publicaciones de usuario y de la cronología de menciones de usuario tienen un límite en la cantidad de Publicaciones que pueden devolver en un mes determinado. El endpoint de la cronología de inicio en orden cronológico inverso no está sujeto a esta limitación.

Independientemente del endpoint de cronologías que uses, las Publicaciones devueltas contarán para los [límites de Publicaciones](/es/x-api/fundamentals/post-cap) a nivel de Proyecto. El uso se muestra en el portal para desarrolladores, y el “mes” comienza en el día de renovación de tu suscripción que se muestra en el [panel del portal para desarrolladores](https://developer.x.com/en/portal/dashboard). 

El endpoint de la cronología de Publicaciones de usuario solo devolverá las 3200 Publicaciones más recientes publicadas en la cronología de un usuario. Si configuras start&#95;time y end&#95;time en un periodo de tiempo que incluya Publicaciones más allá de las 3200 más recientes, recibirás una respuesta correcta, pero sin Publicaciones.

También es importante tener en cuenta que, si pasas excludes=replies en tus solicitudes a la cronología de Publicaciones de usuario, solo se devolverán las 800 Publicaciones más recientes.

El endpoint de la cronología de menciones de usuario solo devolverá las 800 menciones de Publicaciones más recientes.

El endpoint de la cronología de inicio en orden cronológico inverso devuelve las últimas 3200 Publicaciones.

**Ediciones de Publicaciones**

Las Publicaciones que sean aptas para ediciones se pueden editar hasta cinco veces durante los 30 minutos posteriores a la publicación de la Publicación original. Los endpoints de búsqueda siempre proporcionarán la versión más reciente de la Publicación. Si solo solicitas Publicaciones que se publicaron hace 30 minutos o más, siempre recibirás la versión final de la Publicación. Sin embargo, si tu caso de uso es casi en tiempo real y estás consultando Publicaciones publicadas en los últimos treinta minutos, esas Publicaciones podrían haberse editado después de que las recibieras. Estas Publicaciones se pueden rehidratar con búsqueda o con el endpoint Post Lookup para confirmar su estado final. Para obtener más información sobre cómo funcionan las ediciones de Publicaciones, consulta la página [Conceptos fundamentales sobre la edición de Publicaciones](/es/x-api/fundamentals/edit-posts).  

**Casos extremos**

* Cuando solicites métricas no públicas en el endpoint de la cronología de Publicaciones de usuario para Publicaciones con más de 30 días de antigüedad, es posible que veas un next&#95;token en la respuesta con un recuento de resultados de 0. Para evitar encontrarte con este problema, asegúrate de que el periodo de tiempo solicitado con el parámetro non&#95;public&#95;metrics esté dentro de los 30 días más recientes. Además, el valor mínimo de max&#95;results debe ser 10. Estas medidas pueden ayudar a evitar este escenario, aunque aún podría ocurrir.

* Solicitar métricas promocionadas para Publicaciones que no fueron promocionadas devuelve una respuesta vacía, en lugar de datos de Publicaciones. Nuestro equipo está trabajando actualmente para solucionar este problema.

* En el caso de un Retweet que contenga texto del cuerpo de la Publicación de más de 140 caracteres de longitud, el campo text se truncará en lugar de devolver el texto completo de la Publicación. La solución temporal es expandir la Publicación referenciada y recuperar el texto completo desde la expansión. Este es un error que corregiremos en el futuro.
