<div id="helpful-tools">
  ### Herramientas útiles
</div>

Antes de comenzar a explorar algunos conceptos clave, te recomendamos utilizar una de las siguientes herramientas o ejemplos de código para empezar a probar la funcionalidad de estos endpoints.

<div id="code-samples">
  #### Ejemplos de código
</div>

Si quieres configurar estos endpoints con código en tu lenguaje de programación preferido, ofrecemos varios ejemplos de código que puedes usar como punto de partida en nuestra [página de GitHub](https://github.com/xdevplatform/Twitter-API-v2-sample-code), incluidos un [cliente de Python](https://github.com/xdevplatform/search-tweets-python) y un [cliente de Ruby](https://github.com/xdevplatform/search-tweets-ruby).

<div id="libraries">
  #### Bibliotecas
</div>

Aprovecha una de las muchas [bibliotecas de terceros de la comunidad](/es/x-api/tools-and-libraries/overview) para ayudarte a empezar. Puedes encontrar una biblioteca que funcione con los endpoints de v2 buscando la etiqueta de versión correspondiente.

<div id="postman">
  #### Postman
</div>

Postman es una excelente herramienta que puedes usar para probar estos endpoints. Para obtener más información sobre nuestras colecciones de Postman, visita nuestra página [Uso de Postman](/es/tutorials/postman-getting-started).

  ### Conceptos clave
</div>

<div id="authentication">
  #### Autenticación
</div>

Todos los endpoints de X API v2 requieren que autentiques tus solicitudes con un conjunto de credenciales, también conocidas como claves y tokens. Puedes usar Contexto de usuario de OAuth 1.0a, OAuth 2.0 solo para la aplicación u OAuth 2.0 Authorization Code with PKCE para autenticar tus solicitudes al endpoint de búsqueda reciente. Debes usar OAuth 2.0 solo para la aplicación cuando uses el endpoint de búsqueda de archivo completo.

[OAuth 2.0 App-Only](/es/resources/fundamentals/authentication#oauth-2-0-authorization-code-flow-with-pkce-3) solo requiere que pases un [Token de acceso de App de OAuth 2.0](/es/resources/fundamentals/authentication#bearer-token-also-known-as-app-only) con tu solicitud. Puedes generar un App Access Token directamente desde una aplicación de desarrollador o generar uno usando el endpoint [POST oauth2/token](/es/resources/fundamentals/authentication#post-oauth2-token).

[Contexto de usuario de OAuth 1.0a](/es/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) requiere que utilices tus API Keys, tokens de acceso de usuario y algunos otros parámetros para [crear un encabezado de autorización](https://developer-staging.x.com/resources/fundamentals/authentication/authorizing-a-request), que luego pasarás con tu solicitud. Los tokens de acceso deben estar asociados con el usuario en cuyo nombre estás haciendo la solicitud. Si deseas generar un conjunto de tokens de acceso para otro usuario, ese usuario debe autorizar tu App usando el [flujo OAuth de tres patas](https://developer-staging.x.com/resources/fundamentals/authentication/obtaining-user-access-tokens).

Ten en cuenta que OAuth 1.0a puede ser difícil de usar. Si no estás familiarizado con este método de autenticación, te recomendamos que uses una [biblioteca](/es/x-api/tools-and-libraries/overview), que uses una herramienta como Postman o que uses OAuth 2.0 para autenticar tus solicitudes. Si deseas solicitar una Publicación o métricas privadas en estos endpoints, tendrás que usar Contexto de usuario de OAuth 1.0a u OAuth 2.0 Authorization Code with PKCE, lo que garantizará que tengas los permisos adecuados del usuario propietario de ese contenido.

[OAuth 2.0 Authorization Code with PKCE](/es/resources/fundamentals/authentication#oauth-2-0-authorization-code-flow-with-pkce-2) permite un mayor control sobre el alcance de una aplicación y sobre flujos de autorización entre varios dispositivos. OAuth 2.0 te permite elegir ámbitos específicos y granulares que te otorgan permisos concretos en nombre de un usuario.

Para habilitar OAuth 2.0 en tu App, debes habilitarlo en la configuración de autenticación de tu App, que se encuentra en la sección de configuración de la App en el portal para desarrolladores.

<Note>
  **Ten en cuenta**

  Si estás solicitando los siguientes [campos](/es/x-api/fundamentals/fields), se requiere Contexto de usuario de OAuth 1.0a u OAuth 2.0 Authorization Code:

  * tweet.fields.non&#95;public&#95;metrics
  * tweet.fields.promoted&#95;metrics
  * tweet.fields.organic&#95;metrics
  * media.fields.non&#95;public&#95;metrics
  * media.fields.promoted&#95;metrics
  * media.fields.organic&#95;metrics
</Note>

####

Portal para desarrolladores, Proyectos y aplicaciones de desarrollador

Para trabajar con cualquier endpoint de X API v2, debes [registrarte para obtener una cuenta de desarrollador](https://developer.x.com/en/portal/petition/essential/basic-info), configurar un Proyecto dentro de esa cuenta y crear una aplicación de desarrollador dentro de ese Proyecto. Tus claves y tokens dentro de esa aplicación de desarrollador funcionarán para estos endpoints de búsqueda.

Puedes usar claves y tokens de un Proyecto con cualquier [nivel de acceso](/es/x-api/getting-started/about-x-api) para realizar solicitudes al endpoint de búsqueda reciente. Sin embargo, deberás utilizar un Proyecto con el nivel de acceso Pro o Enterprise para realizar solicitudes al endpoint de búsqueda del archivo completo. Si tienes [acceso Enterprise](/es/x-api/getting-started/getting-access), tendrás acceso a funcionalidad adicional, incluida la posibilidad de usar operadores adicionales y consultas más largas.
 

<div id="rate-limits">
  #### Límites de frecuencia
</div>

Cada día, muchos miles de desarrolladores realizan solicitudes a la X API. Para ayudar a gestionar el volumen, se aplican [límites de frecuencia](/es/x-api/fundamentals/rate-limits) a cada endpoint que limitan la cantidad de solicitudes que cada desarrollador puede hacer en nombre de una aplicación o en nombre de un usuario autenticado.

Se aplican diferentes límites de frecuencia para estos endpoints según el método de autenticación que se utilice. Los límites de frecuencia a nivel de aplicación se aplican a una aplicación que realiza solicitudes usando OAuth 2.0 solo para la aplicación, mientras que el límite de frecuencia a nivel de usuario se aplica a las solicitudes realizadas en nombre del usuario que otorga la autorización usando Contexto de usuario de OAuth 1.0a u OAuth 2.0 Authorization Code with PKCE. Estos dos límites de frecuencia se basan en la frecuencia de las solicitudes dentro de una ventana de 15 minutos.

Por ejemplo, una aplicación que use autenticación OAuth 2.0 solo para la aplicación para realizar solicitudes al endpoint de búsqueda reciente puede hacer 450 solicitudes (incluidas las solicitudes de paginación) dentro de un intervalo de 15 minutos. Esa misma aplicación, dentro del mismo intervalo de 15 minutos y con dos usuarios autenticados diferentes (usando Contexto de usuario de OAuth 1.0a u OAuth 2.0 Authorization Code with PKCE) puede hacer hasta 180 solicitudes (incluidas las solicitudes de paginación) al endpoint de búsqueda reciente para cada usuario autenticado.
 

<div id="fields-and-expansions">
  #### Campos y expansiones
</div>

La X API v2 te permite seleccionar exactamente qué datos quieres recibir de la API utilizando [fields](/es/x-api/fundamentals/fields) y [expansions](/es/x-api/fundamentals/expansions). El parámetro `expansions` te permite expandir los objetos referenciados en la carga útil. Por ejemplo, este endpoint te permite solicitar objetos de tipo encuesta, lugar, contenido multimedia y otros objetos usando el parámetro `expansions`.

Los parámetros `fields` te permiten seleccionar exactamente qué campos dentro de los diferentes objetos de datos deseas recibir. De forma predeterminada, el objeto principal de la Publicación devuelto por estos endpoints incluye los campos `id` y `text` (además de `edit_history_tweet_ids` para las Publicaciones creadas después de que se lanzó esa funcionalidad). Para recibir campos adicionales como `author_id` o `public_metrics`, tendrás que solicitarlos específicamente usando los parámetros `fields`. Algunos campos importantes que quizá quieras considerar en tu integración son nuestros datos de encuestas, las [métricas](/es/x-api/fundamentals/metrics), las [anotaciones de Publicaciones](/es/x-api/fundamentals/post-annotations) y los campos de [ID de conversación](/es/x-api/fundamentals/conversation-id).

Hemos añadido una guía sobre cómo [usar fields y expansions conjuntamente](/es/x-api/fundamentals/data-dictionary#how-to-use-fields-and-expansions) en nuestro [diccionario de datos de X API v2](/es/x-api/fundamentals/data-dictionary).
 

<div id="metrics">
  #### Métricas
</div>

Los endpoints de X API v2 te permiten solicitar métricas directamente a partir de los objetos devueltos, siempre que incluyas los [campos](/es/x-api/fundamentals/fields) adecuados en tu solicitud.

Existen algunas limitaciones con las métricas de Publicaciones que debes tener en cuenta, específicamente relacionadas con la privacidad de los usuarios y con los siguientes campos de la respuesta:

* tweet.fields.non&#95;public&#95;metrics
* tweet.fields.promoted&#95;metrics
* tweet.fields.organic&#95;metrics
* media.fields.non&#95;public&#95;metrics
* media.fields.promoted&#95;metrics
* media.fields.organic&#95;metrics

Los campos indicados incluyen datos de métricas privadas, lo que significa que debes estar autorizado por el autor de la Publicación para recuperar estos datos en su nombre cuando uses el endpoint de búsqueda reciente, es decir, debes utilizar [Contexto de usuario de OAuth 1.0a](/es/resources/fundamentals/authentication). Dado que solo puedes utilizar este método de autenticación con la búsqueda reciente, no podrás recuperar estas métricas mediante el endpoint de búsqueda en el archivo completo.

Por ejemplo, para recibir `non_public_metrics` de las Publicaciones del usuario con id 1234, deberás incluir en tu solicitud los tokens de acceso asociados con ese usuario. Puedes hacer que los usuarios autoricen tu Aplicación y recibir un conjunto de tokens de acceso asociados con ellos usando el [flujo OAuth de tres patas](/es/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow).

Todas las `non_public_metrics`, `organic_metrics` y `promoted_metrics` solo están disponibles para Publicaciones creadas en los últimos 30 días. Esto significa que, cuando solicites los campos indicados, los resultados se ajustarán automáticamente para incluir solo Publicaciones de los últimos 30 días.

Si se solicitan estos campos indicados, solo se devolverán las Publicaciones creadas por el usuario autenticado; todas las demás Publicaciones generarán un mensaje de error.
 

<div id="building-search-queries">
  #### Creación de consultas de búsqueda
</div>

La característica central de estos endpoints es el uso de una única consulta de búsqueda para filtrar las Publicaciones que recibes. Estas consultas se componen de operadores que coinciden con atributos de la Publicación y del usuario, como palabras clave del mensaje, hashtags y URL. Los operadores se pueden combinar en consultas con lógica booleana y paréntesis para ayudar a refinar el comportamiento de coincidencia de la consulta.

Puedes usar nuestra guía sobre [cómo crear una consulta](/es/x-api/posts/search/integrate/build-a-query) para obtener más información.

<div id="pagination">
  #### Paginación
</div>

Estos endpoints utilizan paginación para que las respuestas se entreguen con rapidez. En los casos en que haya más resultados de los que se pueden enviar en una sola respuesta (hasta 100 Publicaciones para la búsqueda reciente y 500 para la búsqueda en todo el archivo), deberás paginar. Usa el parámetro max&#95;results para indicar cuántos resultados se devolverán por página y el parámetro pagination&#95;token para recuperar la siguiente página de resultados. Puedes obtener más información revisando nuestra [guía de paginación](/es/x-api/fundamentals/pagination).

<div id="post-caps">
  #### Límites de Publicaciones
</div>

Los endpoints de Search Posts tienen un límite en la cantidad de Publicaciones que pueden devolver en un mes determinado, independientemente de la paginación.

Independientemente del endpoint de búsqueda que utilices, las Publicaciones devueltas contarán para los [límites de Publicaciones](/es/x-api/fundamentals/post-cap) a nivel de Proyecto. El uso se muestra en el portal para desarrolladores, y el «mes» comienza el día de renovación de tu suscripción que se muestra en el [panel del portal para desarrolladores](https://developer.x.com/en/portal/dashboard).

**Ediciones de Publicaciones**

Las Publicaciones que puedan editarse se pueden editar hasta cinco veces en los 30 minutos posteriores a la publicación de la Publicación original. Los endpoints de búsqueda siempre proporcionarán la versión más reciente de la Publicación. Si solo solicitas Publicaciones que se publicaron hace 30 minutos o más, siempre recibirás la versión final de la Publicación. Sin embargo, si tienes un caso de uso casi en tiempo real y estás consultando Publicaciones publicadas en los últimos treinta minutos, es posible que esas Publicaciones se hayan editado después de que las recibieras. Estas Publicaciones se pueden rehidratar con búsqueda o con el endpoint Post Lookup para confirmar su estado final. Para obtener más información sobre cómo funcionan las ediciones de Publicaciones, consulta la página [Post edits fundamentals](/es/x-api/fundamentals/edit-posts).
