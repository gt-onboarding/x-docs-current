<div id="list-members-lookup-standard-v11-compared-to-x-api-v2">
  ### Búsqueda de miembros de Listas: estándar v1.1 comparado con X API v2
</div>

Si has trabajado con los endpoints estándar v1.1 [GET lists/members](https://developer.x.com/en/docs/twitter-api/v1/accounts-and-users/create-manage-lists/api-reference/get-lists-members) y [GET lists/memberships](https://developer.x.com/en/docs/twitter-api/v1/accounts-and-users/create-manage-lists/api-reference/get-lists-memberships), el objetivo de esta guía es ayudarte a entender las similitudes y diferencias entre los endpoints estándar v1.1 y los endpoints de miembros de Listas de X API v2.

* **Similitudes**
  * Métodos de autenticación
* **Diferencias**
  * URLs de los endpoints
  * Límites de frecuencia
  * Requisitos de aplicación y Proyecto
  * Límites en la cantidad de objetos de datos por solicitud
  * Formatos de datos de respuesta
  * Parámetros de solicitud

<div id="similarities">
  #### Similitudes
</div>

**Autenticación**

Ambas versiones del endpoint admiten tanto [Contexto de usuario de OAuth 1.0a](/es/resources/fundamentals/authentication#oauth-1-0a-2) como autenticación [solo de la Aplicación (App only)](/es/resources/fundamentals/authentication#oauth-2-0). Por lo tanto, si antes utilizabas uno de los endpoints estándar de miembros de List v1.1, puedes seguir utilizando el mismo método de autenticación si migras a la versión de X API v2.

Según la biblioteca o paquete de autenticación que utilices, la autenticación solo de la Aplicación (App only) probablemente sea la forma más sencilla de comenzar y se puede configurar con un encabezado de solicitud sencillo. Para obtener información sobre cómo generar un token de acceso de solo Aplicación, consulta [esta guía de App only](/es/resources/fundamentals/authentication#bearer-token-also-known-as-app-only).

<div id="differences">
  #### Diferencias
</div>

**URLs de endpoints**

* Endpoints estándar v1.1:
  * GET https://api.x.com/1.1/lists/members.json
    (Consulta los miembros de una Lista específica)
  * GET https://api.x.com/1.1/lists/memberships.json
    (Consulta las Listas de las que un usuario es miembro)
* Endpoint de X API v2:
  * GET https://api.x.com/2/lists/:id/members
    (Consulta los miembros de una Lista específica)

  * GET https://api.x.com/2/users/:id/list&#95;memberships
    (Consulta las Listas de las que un usuario es miembro)

**Límites de frecuencia**

|     |     |
| :--- | :--- |
| **Standard v1.1** | **X API v2** |
| /1.1/lists/members.json<br /><br />900 solicitudes por ventana de 15 minutos con Contexto de usuario de OAuth 1.0a<br /><br />15 solicitudes por ventana de 15 minutos con solo Aplicación | /2/lists/:id/members<br /><br />900 solicitudes por ventana de 15 minutos con Contexto de usuario de OAuth 1.0a<br /><br />900 solicitudes por ventana de 15 minutos con OAuth 2.0 Authorization Code con PKCE<br /><br />900 solicitudes por ventana de 15 minutos con solo Aplicación |
| /1.1/lists/memberships.json<br /><br />15 solicitudes por ventana de 15 minutos con Contexto de usuario de OAuth 1.0a<br /><br />15 solicitudes por ventana de 15 minutos con solo Aplicación | /2/users/:id/list&#95;memberships<br /><br />15 solicitudes por ventana de 15 minutos con Contexto de usuario de OAuth 1.0a<br /><br />15 solicitudes por ventana de 15 minutos con OAuth 2.0 Authorization Code con PKCE<br /><br />15 solicitudes por ventana de 15 minutos con solo Aplicación |

**Requisitos de App y Proyecto**

Los endpoints de X API v2 requieren que uses credenciales de una [aplicación de desarrollador](/es/resources/fundamentals/developer-apps) que esté asociada a un [Proyecto](/es/resources/fundamentals/projects) al autenticar tus solicitudes. Todos los endpoints de X API v1.1 pueden usar credenciales de Aplicaciones independientes o Aplicaciones asociadas con un Proyecto.

**Límites de objetos de datos por solicitud**

El endpoint estándar v1.1 /1.1/lists/members permite devolver hasta 5000 usuarios por solicitud. Los nuevos endpoints v2 permiten devolver hasta 100 usuarios por solicitud. De forma predeterminada, se devolverán 100 objetos de usuario; para cambiar la cantidad de resultados deberás pasar un parámetro de consulta max&#95;results= con un número entre 1 y 100; luego puedes pasar el next&#95;token devuelto en el cuerpo de la respuesta al parámetro de consulta pagination&#95;token en tu siguiente solicitud.

Además, el endpoint /1.1/lists/memberships permite devolver hasta 1000 Listas por solicitud. Con el reemplazo en v2, el endpoint permite hasta 100 Listas por solicitud. De forma predeterminada se devuelven 100 objetos de Lista; usa los parámetros de consulta max&#95;results= y pagination&#95;token de la misma manera que en /1.1/lists/members para cambiar la cantidad de resultados.

**Formato de los datos de respuesta**

Una de las mayores diferencias entre las versiones de endpoints estándar v1.1 y X API v2 es cómo seleccionas qué campos se devuelven en tu payload.

Para los endpoints estándar, recibes muchos de los campos de respuesta de forma predeterminada y luego tienes la opción de usar parámetros para identificar qué campos adicionales o conjuntos de campos deben devolverse en el payload.

La versión de X API v2 /users/:id/list&#95;memberships entregará los campos id y name de la Lista de forma predeterminada. Para solicitar campos u objetos adicionales, deberás usar los parámetros [fields](/es/x-api/fundamentals/fields) y [expansions](/es/x-api/fundamentals/expansions). Cualquier campo de Lista que solicites desde este endpoint se devolverá en el objeto principal de Lista. Cualquier objeto y campos expandidos se devolverán en un objeto includes dentro de tu respuesta. Luego podrás hacer coincidir cualquier objeto expandido con el objeto principal de Lista haciendo coincidir los id ubicados tanto en el objeto principal como en el expandido. 

A continuación se muestran ejemplos de posibles campos y expansiones de Lista:

* created&#95;at

* follower&#95;count

* member&#95;count

* owner&#95;id

* description

* private

|     |     |
| :--- | :--- |
| **Endpoint** | **Expansión** |
| /2/lists/:id/members | pinned&#95;tweet&#95;id |
| /2/users/:id/list&#95;memberships | owner&#95;id |

Te recomendamos leer más sobre estos nuevos parámetros en sus respectivas guías, o en nuestra guía sobre [cómo usar fields y expansions](/es/x-api/fundamentals/data-dictionary#how-to-use-fields-and-expansions). 

También hemos preparado una [guía de migración de formato de datos](/es/x-api/migrate/data-format-migration) que puede ayudarte a asignar los campos estándar de v1.1 a los nuevos campos de v2. Esta guía también te proporcionará la expansión específica y el parámetro de campo que necesitarás pasar con tu solicitud v2 para devolver campos específicos. 

Además de los cambios en cómo solicitas determinados campos, X API v2 también introduce nuevos diseños JSON para los objetos devueltos por las APIs, incluidos los objetos de [Publicación](/es/x-api/fundamentals/data-dictionary#tweet) y de [usuario](/es/x-api/fundamentals/data-dictionary#user).

* En el nivel raíz de JSON, los endpoints estándar devuelven objetos de Publicación en un array statuses, mientras que X API v2 devuelve un array data. 

* En lugar de hacer referencia a &quot;statuses&quot; retuiteados y citados, el JSON de X API v2 hace referencia a Publicaciones retuiteadas y citadas. Muchos campos heredados y obsoletos, como contributors y user.translator&#95;type, se están eliminando. 

* En lugar de usar tanto favorites (en el objeto de Publicación) como favourites (en el objeto de usuario), X API v2 usa el término like. 

* X adopta la convención de que los valores JSON vacíos (por ejemplo, null) no se escriben en la carga útil. Los atributos de Publicación y de usuario solo se incluyen si tienen valores no nulos.
