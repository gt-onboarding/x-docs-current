<div id="introduction">
  ## Introducción
</div>

Con el lanzamiento de la versión v2 de la X API, hemos adoptado un nuevo formato de respuesta de datos y un nuevo método para solicitar diferentes objetos y campos, al que simplemente llamamos formato de X API v2. 

En la sección de diferencias generales, puedes informarte sobre algunos cambios que son relevantes tanto para usuarios estándar como Enterprise. Sin embargo, también hemos preparado una guía específica para el [formato nativo estándar v1.1](https://developer.x.com/en/docs/x-api/v1/data-dictionary/overview), el [formato Native Enriched de Enterprise](/es/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#native-enriched-tweet-object) y el [formato Activity Streams de Enterprise](/es/x-api/enterprise-gnip-2.0/fundamentals/data-dictionary#activity-object), que ayuda a mapear campos y explica qué campos y expansiones debes usar para solicitar los nuevos campos de v2. 

* [Formato nativo a X API v2 (estándar v1.1)](/es/x-api/migrate/data-format-migration#migrating-from-standard-v1-1s-data-format-to-v2) 
* [Native Enriched a X API v2 (Enterprise)](/es/x-api/migrate/data-format-migration#migrating-from-native-enriched-data-format-to-v2)
* [Activity Streams a X API v2 (Enterprise)](/es/x-api/migrate/data-format-migration#migrating-from-activity-streams-data-format-to-v2)

También puede interesarte nuestra [herramienta visual de migración de formatos de datos](/es/x-api/migrate/data-format-migration#visual-data-format-migration-tool), que te ayuda a ver rápidamente las diferencias entre el [formato de datos de X API v1.1](https://developer.x.com/en/docs/x-api/v1/data-dictionary/overview) y el [formato de X API v2](/es/x-api/introduction).

<div id="general-differences">
  ### Diferencias generales
</div>

<div id="requesting-objects-and-fields">
  #### Solicitud de objetos y campos
</div>

Uno de los cambios más importantes entre los endpoints anteriores a v2 y los de v2 es que la versión más reciente solo devuelve unos pocos campos de forma predeterminada, mientras que los endpoints standard, premium y Enterprise devuelven la mayoría de los campos de forma predeterminada. La nueva versión utiliza parámetros llamados [fields](/es/x-api/fundamentals/fields) y [expansions](/es/x-api/fundamentals/expansions) para solicitar específicamente datos adicionales más allá de los valores predeterminados, lo que significa que puedes solicitar solo los datos que necesitas sin tener que procesar campos que no son relevantes para ti. 

Cualquier campo que solicites y que esté relacionado con el objeto de datos principal se devolverá en ese objeto de datos principal junto con los valores predeterminados. Sin embargo, si solicitas objetos expandidos usando el parámetro expansions, los objetos secundarios se devolverán en un nuevo objeto includes. Puedes hacer coincidir los objetos expandidos en el objeto includes con el objeto principal utilizando el campo id, que se devolverá en ambos.

Por ejemplo, si estás utilizando el endpoint v2 [Post lookup](/es/x-api/posts/lookup/introduction) e incluyes el parámetro expansions=author&#95;id en tu solicitud, recibirás el campo author&#95;id dentro del objeto Post principal, así como un objeto de usuario por cada Publicación en el objeto includes, cada uno de los cuales incluirá el campo id predeterminado que puede utilizarse para hacer coincidir el objeto de usuario con el objeto Post. A continuación se muestra un ejemplo de cómo es esto:

```
{
  "data": [
    {
      "author_id": "2244994945",
      "id": "1397568983931392004",
      "text": "The Twitter Developer Platform. Ooh la la! https://t.co/iGTdPXBfOv https://t.co/Ze8z8EODdg"
    }
  ],
  "includes": {
    "users": [
      {
        "id": "2244994945",
        "name": "Twitter Dev",
        "username": "TwitterDev"
      }
    ]
  }
}
```

<div id="updated-json-design">
  #### Diseño JSON actualizado
</div>

Además de los cambios en cómo se solicitan ciertos campos, X API v2 también introduce nuevos diseños JSON para los objetos que devuelven las API, incluidos los objetos de [Publicación](/es/x-api/fundamentals/data-dictionary#tweet) y de [usuario](/es/x-api/fundamentals/data-dictionary#user).

* En el nivel raíz de JSON, los endpoints estándar devuelven objetos de Publicación en una matriz **statuses**, mientras que X API v2 devuelve una matriz **data**. 
* En lugar de hacer referencia a &quot;statuses&quot; Retweeted y Quoted, el JSON de X API v2 hace referencia a Publicaciones Retweeted y Quoted. Muchos campos heredados y obsoletos, como **contributors** y **user.translator&#95;type**, se están eliminando. 
* En lugar de usar tanto **favorites** (en el objeto de Publicación) como **favourites** (en el objeto de usuario), X API v2 utiliza el término like. 
* X está adoptando la convención de que los valores JSON cuyo valor es nulo (por ejemplo, null) no se escriben en el payload. Los atributos de Publicación y de usuario solo se incluyen si tienen un valor distinto de null. 
   

<div id="new-v2-fields">
  #### Nuevos campos de v2
</div>

También introdujimos un nuevo conjunto de campos en el [objeto Post](/es/x-api/fundamentals/data-dictionary#tweet) que incluye lo siguiente:

* Un campo [conversation&#95;id](/es/x-api/fundamentals/conversation-id)
* Dos nuevos campos de [anotaciones](/es/x-api/fundamentals/post-annotations), incluidos context y entities
* Varios nuevos campos de [métricas](/es/x-api/fundamentals/metrics) 
* Un nuevo campo reply&#95;setting, que indica quién puede responder a una Publicación determinada

<div id="migrating-from-standard-v11s-data-format-to-v2">
  ### Migración del formato de datos estándar de v1.1 a v2
</div>

Si aún no lo has hecho, te recomendamos que leas primero la introducción de la [migración de formatos de datos](/es/x-api/migrate/data-format-migration). También puede interesarte nuestra [herramienta visual de migración de formato de datos](/es/x-api/migrate/data-format-migration#visual-data-format-migration-tool), que te ayuda a ver rápidamente las diferencias entre el [formato de datos de X API v1.1](https://developer.x.com/en/docs/x-api/v1/data-dictionary/overview) y el [formato de datos de X API v2](/es/x-api/fundamentals/data-dictionary).

El formato de datos estándar de v1.1, también conocido como formato nativo, es el formato principal que se utiliza con los endpoints de [standard v1.1](https://developer.x.com/en/docs/twitter-api/v1).

Si estás utilizando el producto premium, consulta la [guía de datos nativos enriquecidos](/es/x-api/migrate/data-format-migration#migrating-from-native-enriched-data-format-to-v2). Los clientes de Enterprise podrían estar usando datos nativos enriquecidos o flujos de actividad, según cómo estén configurados en la consola de Gnip.

<div id="migrating-from-native-enriched-data-format-to-v2">
  ### Migrar del formato de datos Native Enriched a v2
</div>

El formato de datos Native Enriched lo utilizan nuestros productos [Enterprise](/es/x-api/enterprise-gnip-2.0/enterprise-gnip).

El formato de datos Native Enriched se ha actualizado para proporcionar metadatos de Publicaciones editadas. Para obtener más información sobre los metadatos de Publicaciones editadas, consulta la página [Edit Tweets fundamentals](/es/x-api/enterprise-gnip-2.0/fundamentals/edit-tweets).

Si estás usando los endpoints estándar de v1.1, consulta la [guía de migración de estándar v1.1 a v2](/es/x-api/migrate/data-format-migration#migrating-from-standard-v1-1s-data-format-to-v2). Si estás usando los productos Enterprise con Activity Streams, también tenemos una guía [Activity Streams a v2](/es/x-api/migrate/data-format-migration#migrating-from-activity-streams-data-format-to-v2) para ti.

X API v2 introduce nuevas estructuras JSON para los objetos [Publicación](/es/x-api/fundamentals/data-dictionary#tweet) y [usuario](/es/x-api/fundamentals/data-dictionary#user).

* En el nivel raíz del JSON, el formato Native Enriched devuelve objetos de Publicación en un arreglo results, mientras que X API v2 devuelve un arreglo data. 
* En lugar de usar tanto favorites (en el objeto Publicación) como favourites (en el objeto usuario), X API v2 utiliza el término like. 
* X está adoptando la convención de que los valores JSON sin contenido (por ejemplo, null) no se escriben en la carga útil. Los atributos de Publicación y de usuario solo se incluyen si tienen valores no nulos. 
* Todos los campos id en v2 estarán en formato de cadena
   

Además de los cambios realizados en el nuevo formato JSON, también hemos introducido un nuevo conjunto de campos en el objeto Publicación, incluidos los siguientes:

* conversation&#95;id
* reply&#95;settings
* alt&#95;text en contenido multimedia
* Dos nuevos campos de [anotaciones](/es/x-api/fundamentals/post-annotations), incluidos context y entities
* Varios campos nuevos de [métricas](/es/x-api/fundamentals/metrics)
* Varios campos nuevos de [encuestas](/es/x-api/fundamentals/data-dictionary#poll)
   

Se están eliminando muchos campos heredados y obsoletos:

* contributors
* Ciertos campos entities.media y extended&#95;entities.media
* filter&#95;level
* timestamp&#95;ms
* truncated

<div id="migrating-from-activity-streams-data-format-to-v2">
  ### Migrar del formato de datos Activity Streams a v2
</div>

El formato de datos Activity Streams está disponible con nuestros productos [enterprise](/es/x-api/enterprise-gnip-2.0/enterprise-gnip).

El formato de datos Activity Streams se ha actualizado para proporcionar metadatos de Publicaciones editadas. Para obtener más información sobre los metadatos de edición de Publicaciones, consulta la página [Fundamentos de Edición de Publicaciones](/es/x-api/enterprise-gnip-2.0/fundamentals/edit-tweets).

Si estás usando los endpoints estándar de v1.1, consulta la [guía de migración de v1.1 estándar a v2](/es/x-api/migrate/overview). Si estás usando los endpoints premium o el formato Native Enriched para enterprise, consulta la [guía de Native Enriched a v2](/es/x-api/migrate/data-format-migration#migrating-from-native-enriched-data-format-to-v2).

X API v2 introduce nuevos diseños JSON para los objetos [Post](/es/x-api/fundamentals/data-dictionary#tweet) y [user](/es/x-api/fundamentals/data-dictionary#user).

* En el nivel raíz de JSON, el formato Activity Streams devuelve objetos de Publicación en un array `results`, mientras que X API v2 devuelve un array `data`. 
* En lugar de hacer referencia a “actividades” de Retweeted y Quoted, el JSON de X API v2 hace referencia a Publicaciones retuiteadas y citadas. 
* En lugar de usar tanto `favorites` (en el objeto de Publicación) como `favourites` (en el objeto user), X API v2 utiliza el término `like`. 
* Twitter adopta la convención de que los valores JSON vacíos (por ejemplo, `null`) no se incluyen en la carga útil (payload). Los atributos de Publicación y de user solo se incluyen si tienen valores no nulos. 
* Todos los campos `id` en v2 estarán en formato de cadena (`string`).
   

Además de los cambios realizados en el nuevo formato JSON, también hemos introducido un nuevo conjunto de campos en el objeto de Publicación, incluidos los siguientes:

* `conversation_id`
* `reply_settings`
* `alt_text` en contenido multimedia
* Dos nuevos campos de [annotations](/es/x-api/fundamentals/post-annotations), incluidos `context` y `entities`
* Varios campos nuevos de [metrics](/es/x-api/fundamentals/metrics)
* Varios campos nuevos de [polls](/es/x-api/fundamentals/data-dictionary#poll)
   

Muchos campos heredados y obsoletos se están eliminando o reemplazando:

* `display_text_range`
* `generator`
* `gnip`
* `link`
* `objectType`
* `provider`
* `twitter_entities.symbols` reemplazado por `data.entities.cashtags`
* Ciertos campos de `twitter_extended_entities.media` y `twitter_entities.media`
* `twitter_filter_level`
* `twitterTimeZone`
* `verb`
