> **Ten en cuenta**
>
> Hemos lanzado una nueva herramienta de compliance para X API v2 llamada [batch compliance](/es/x-api/compliance/batch-compliance). Esta nueva herramienta te permite cargar grandes conjuntos de datos de ID de Publicación o de usuario para recuperar su estado de compliance y así determinar qué datos requieren una acción para que tus conjuntos de datos cumplan con los requisitos de compliance.

> Además, tanto batch compliance como compliance firehose se han actualizado para admitir ediciones de Publicaciones. En el caso de compliance firehose, se agregó un nuevo evento `tweet_edit`. Consulta la documentación de [Compliance Data Objects](/es/x-api/enterprise-gnip-2.0/fundamentals/firehouse#compliance-data-objects) para obtener más detalles. Obtén más información sobre cómo funcionan los metadatos de Edit Post en la página [Edit Posts fundamentals](/es/x-api/fundamentals/edit-posts).

<div id="overview">
  ## Descripción general
</div>

`Enterprise`

Uno de los valores fundamentales de X es defender y respetar la voz de los usuarios. Esto incluye respetar sus expectativas e intención cuando eliminan, modifican o editan el contenido que deciden compartir en X. Creemos que esto es de importancia crítica para la salud a largo plazo de una de las mayores plataformas públicas de información en tiempo real del mundo. X pone controles en manos de sus usuarios, dando a cada persona la capacidad de controlar su propia experiencia en X. Consideramos que las empresas que reciben datos de X tienen la responsabilidad de respetar las expectativas y la intención de los usuarios finales.

Para obtener más información sobre los tipos de eventos de cumplimiento que son posibles en la plataforma X, consulta nuestro artículo [Honoring User Intent on X](/es/x-api/enterprise-gnip-2.0/fundamentals/firehouse#honoring-user-intent-on-twitter).

Cualquier desarrollador o empresa que consuma datos de X mediante una API tiene la obligación de hacer todos los esfuerzos razonables para respetar los cambios en el contenido de los usuarios. Esta obligación se extiende a eventos de usuario como eliminaciones, modificaciones y cambios en las opciones de uso compartido (por ejemplo, cuando el contenido pasa a estar protegido o retenido). Esto también incluye cuando los usuarios editan sus Publicaciones. Consulta el texto específico de la [Developer Policy](https://developer.x.com/en/developer-terms/policy) y/o de tu X Data Agreement para entender cómo esta obligación afecta tu uso de los datos de X.

X ofrece las siguientes soluciones que proporcionan información sobre estos eventos de cumplimiento de usuario y sobre si una Publicación o un Usuario determinados están disponibles públicamente o no. A continuación se presenta una breve descripción general de las soluciones y de sus patrones de integración generales:

<div id="get-statuseslookup-and-get-userslookup">
  #### GET statuses/lookup y GET users/lookup
</div>

* Formato: API REST. Consulte: [GET statuses/lookup](https://developer.x.com/en/docs/x-api/v1/tweets/post-and-engage/api-reference/get-statuses-lookup) y [GET users/lookup](https://developer.x.com/en/docs/x-api/v1/accounts-and-users/follow-search-get-users/api-reference/get-users-lookup).

* Estos endpoints siempre devuelven la versión más reciente de cualquier Publicación editada. Todos los objetos de Publicación que describen Publicaciones creadas después de que se introdujo la función de edición incluirán metadatos de edición de la Publicación; esto es así incluso para Publicaciones que no fueron editadas. 

* Para todas las Publicaciones, las solicitudes realizadas más de 30 minutos después de su creación representarán el estado final de dichas Publicaciones. 

* Entregan información de disponibilidad para Publicaciones o Usuarios específicos definidos por quien realiza la llamada como parte de la solicitud de API.

* Se pueden usar para verificaciones puntuales ad hoc sobre el estado de disponibilidad actual de un grupo específico de Publicaciones/Usuarios.

* Son ideales para clientes que necesitan una forma de comprobar el estado actual de una Publicación o Usuario específicos en un momento dado.

* Estas APIs proporcionan un mecanismo útil que los clientes pueden usar cuando necesitan comprobar la disponibilidad de una pieza de Contenido, por ejemplo cuando:
  1. Se muestran Publicaciones
  2. Se interactúa con una(s) Publicación(es) o Usuario(s) de forma 1:1
  3. Se distribuye Contenido de X a un tercero mediante una descarga de archivo permitida
  4. Se almacenan Publicaciones durante períodos prolongados

<div id="compliance-firehose-enterprise-only">
  #### Firehose de cumplimiento (solo Enterprise)
</div>

* Formato: API de streaming. Consulta: [Compliance Firehose](/es/x-api/enterprise-gnip-2.0/fundamentals/firehouse#api-reference).
* Entrega un flujo en tiempo real de [actividades de cumplimiento](/es/x-api/enterprise-gnip-2.0/fundamentals/firehouse#compliance-data-objects) en X. Estas actividades incluyen la edición de Publicaciones. 
* Puede utilizarse para mantener el estado de cumplimiento de un conjunto de datos almacenados a medida que se producen nuevos eventos de cumplimiento en la plataforma.
* Ideal para clientes que consumen y almacenan grandes volúmenes de datos de X durante períodos prolongados.

<div id="recommendations-best-practices">
  #### **Recomendaciones y mejores prácticas**
</div>

* **Cree esquemas de almacenamiento de datos que guarden el ID numérico de la Publicación y el ID del Usuario**: Los mensajes de los usuarios requieren que se tomen medidas en todas las Publicaciones de ese Usuario. Por lo tanto, dado que todos los mensajes de cumplimiento se entregan únicamente mediante IDs numéricos, es importante diseñar esquemas de almacenamiento que mantengan la relación entre Publicación y Usuario basándose en IDs numéricos. Los consumidores de datos deberán supervisar los eventos de cumplimiento tanto por ID de Publicación como por ID de Usuario y poder actualizar el almacén de datos local de forma adecuada.

* **Cree esquemas que contemplen todos los estados de cumplimiento**: Dependiendo de cómo se gestionen las actividades de cumplimiento en distintas aplicaciones, puede ser necesario agregar otros metadatos al almacén de datos. Por ejemplo, los consumidores de datos pueden decidir agregar metadatos a una base de datos existente para facilitar la restricción de la visualización de contenido en los países afectados por un mensaje status&#95;withheld.

* **Gestión de eliminaciones de Retweets**: Los Retweets son un tipo especial de Publicación en la que el mensaje original está anidado en un objeto dentro del Retweet. En este caso, hay dos IDs de Publicación referenciados en un Retweet: el ID del Retweet y el ID del mensaje original (incluido en el objeto anidado). Cuando se elimina un mensaje original, se emite un mensaje de eliminación de Publicación para el ID original. Los eventos de eliminación de Publicación normalmente desencadenan eventos de eliminación para todos los Retweets. Sin embargo, en algunos casos no se envían todos y los sistemas de cliente deben ser tolerantes frente a eliminaciones de Retweets incompletas. La eliminación del ID original debería ser suficiente para eliminar todos los Retweets posteriores. Es una buena práctica hacer referencia al ID de Publicación original al almacenar Retweets y eliminar todos los Retweets referenciados cuando se reciban eventos de eliminación de Publicaciones.

  #### **API de Compliance Firehose**
</div>

Los tipos posibles de eventos de compliance incluirán eventos de Publicación (o “status”) y eventos de usuario, para los cuales se describen varios tipos a continuación.  

Ten en cuenta lo siguiente:

* Obtén más información sobre los estados de usuario [aquí](https://support.x.com/articles/14016) y sobre nuestra política para desarrolladores en relación con las Publicaciones eliminadas [aquí](https://dev.x.com/overview/terms/policy#3.Update_Respect_Users_Control_and_Privacy).
* Compliance Firehose se ha actualizado para proporcionar eventos `tweet_edit`. 
* Varios eventos de eliminación, protección y suspensión de usuario no son necesariamente permanentes y pueden alternar entre estados indefinidamente. Estos incluyen: user&#95;delete, user&#95;undelete, user&#95;protect, user&#95;unprotect y user&#95;suspend, user&#95;unsuspend.
* Los eventos user&#95;delete son seguidos por status&#95;deletes 30 días después solo si el usuario no ha elegido realizar un user&#95;undelete de su cuenta. Es posible que un user&#95;delete sea revertido por el usuario y que las eliminaciones de todas sus Publicaciones 30 días después no se produzcan.
* user&#95;suspend es una acción que permanece vigente a menos que el usuario sea objeto de un evento user&#95;unsuspend. Estos no están sujetos a cambios asociados a un período de 30 días.

Consulta la columna &quot;Recommended Action&quot; para entender cómo procesar cada tipo de evento con el fin de respetar la privacidad e intención del usuario final.

| Original Message Type | Object | Permanent (Yes/No) | Recommended Action |
| :--- | :--- | :--- | :--- |

| user&#95;protect | User | No  | Suprime o elimina todas las Publicaciones del usuario asociado. |

| user&#95;suspend | User | No  | Suprime o elimina todas las Publicaciones del usuario asociado. |

<div id="integrating-compliance-firehose">
  ### Integración con Compliance Firehose
</div>

Compliance Firehose es una API de streaming en tiempo real que proporciona eventos de cumplimiento que ocurren en la plataforma X. Para comprender los eventos de cumplimiento y cómo se generan en X, consulta nuestro artículo, [Honoring User Intent on X](/es/x-api/enterprise-gnip-2.0/fundamentals/firehouse#honoring-user-intent-on-twitter).

Es importante tener en cuenta que los eventos de Publicación y de Usuario se entregan de forma independiente y que cada uno debe procesarse de manera independiente (es decir, la eliminación de una Publicación no implica un evento de Usuario, y viceversa). Varios eventos de Usuario no son necesariamente permanentes y pueden alternar entre estados indefinidamente. Estos incluyen: user&#95;delete, user&#95;undelete, user&#95;protect, user&#95;unprotect y user&#95;suspend, user&#95;unsuspend.

Los eventos user&#95;delete van seguidos de eventos status&#95;delete 30 días después, solo si el usuario no ha seleccionado user&#95;undelete para su cuenta. Es posible que un user&#95;delete sea revertido por el usuario y que 30 días después no se produzcan las eliminaciones de todas sus Publicaciones.

user&#95;suspend es una acción que permanece vigente a menos que el usuario esté sujeto a un evento user&#95;unsuspend. Estos no están sujetos a ningún cambio asociado a un período de 30 días.

Nunca es apropiado mostrar eventos de cumplimiento directamente a los usuarios de tu software ni incorporarlos de otro modo en tus productos o experiencias para clientes. Están destinados únicamente a mantener el cumplimiento y respetar las acciones de los usuarios de X.

<div id="integrating-with-the-compliance-firehose">
  #### **Integración con el Compliance Firehose**
</div>

Para integrar el Compliance Firehose en tu sistema, deberás crear una integración que pueda hacer lo siguiente:

1. Establecer una conexión de streaming con cada partición de la API de Compliance Firehose
2. Manejar volúmenes de datos elevados: desacoplar la ingesta del flujo del procesamiento adicional usando procesos asíncronos
3. Volver a conectarse automáticamente a las particiones del flujo cuando se produzca una desconexión por cualquier motivo
4. Procesar los eventos de cumplimiento que sean relevantes para los datos de Publicación y Usuario que has almacenado, de acuerdo con las orientaciones presentadas anteriormente

<div id="honoring-user-intent-on-twitter">
  ### Respetar la intención de los usuarios en X
</div>

Creemos que respetar la privacidad y la intención de las personas usuarias de X es de importancia crítica para la salud a largo plazo de una de las plataformas de información pública y en tiempo real más grandes del mundo. X pone los controles de privacidad en manos de sus usuarios, dándoles la capacidad de controlar su propia experiencia en X. Como empresas que consumen datos de X, tenemos una responsabilidad colectiva de respetar la privacidad y las acciones de los usuarios finales para mantener este entorno de confianza y respeto.

Hay una variedad de situaciones que pueden afectar a las Publicaciones y a las cuentas de usuario y que influyen en cómo se muestran en la plataforma. Las acciones que afectan la privacidad y la intención se definen tanto a nivel de Estado (Publicación) como de Usuario. Estas acciones incluyen:

<div id="user">
  #### Usuario
</div>

| Acción | Descripción |
| :--- | :--- |
| Proteger cuenta | Un usuario de X puede proteger o quitar la protección de su cuenta en cualquier momento. Las cuentas protegidas requieren la aprobación manual por parte del usuario de cada persona a la que se le permite ver las Publicaciones de su cuenta.   <br />Para obtener más información, consulta [Publicaciones públicas y protegidas](https://support.x.com/articles/14016-about-public-and-protected-tweets). |
| Eliminar cuenta | Un usuario de X puede decidir eliminar su cuenta y todas las Publicaciones asociadas en cualquier momento. X conserva la información de la cuenta durante 30 días después de la eliminación en caso de que el usuario decida recuperarla y, de ese modo, reactivar su cuenta. |
| Scrub Geo | Un usuario de X puede eliminar todos los datos de ubicación de sus Publicaciones anteriores en cualquier momento. A esto se le conoce como “scrub geo”. |
| Suspender cuenta | X se reserva el derecho de suspender las cuentas que infrinjan las Reglas de X o si se sospecha que una cuenta ha sido pirateada o comprometida. Solo X puede revertir (levantar la suspensión de) las suspensiones de cuentas. |
| Retener cuenta | X se reserva el derecho de restringir de manera reactiva el acceso a cierto contenido en un país específico. Solo X puede levantar la retención de una cuenta.   <br />Para obtener más información, consulta [Contenido retenido por país](https://support.x.com/articles/20169222-country-withheld-content). |

<div id="status">
  #### Estado
</div>

| Acción | Descripción |
| :--- | :--- |
| Delete Status | Un usuario de X puede eliminar un estado en cualquier momento. Los estados eliminados no se pueden revertir y se eliminan de forma permanente. |
| Withhold Status | X se reserva el derecho de restringir de forma reactiva el acceso a cierto contenido en un país específico de vez en cuando. Solo X puede levantar la retención de un estado retenido.   <br />Para obtener más información, consulta [Country Withheld Content](https://support.x.com/articles/20169222-country-withheld-content). |

<div id="keeping-track-of-user-and-status-changes">
  #### Seguimiento de cambios en User y Status
</div>

El estado de un User o de un Status puede cambiar en cualquier momento debido a una de las acciones anteriores, y esto afecta la forma en que se espera que los consumidores de datos de X gestionen la disponibilidad y la privacidad de todo el contenido asociado. Cuando se producen estas acciones, se envía un mensaje de cumplimiento correspondiente que indica que el estado de un User o de un Status ha cambiado.

<div id="other-recommendations-best-practices">
  #### Otras recomendaciones y mejores prácticas
</div>

* **Cree esquemas de almacenamiento de datos que almacenen el ID numérico de la Publicación y el ID de usuario**: Los mensajes de usuario requieren que se tomen medidas con respecto a todas las Publicaciones de ese usuario. Por lo tanto, dado que todos los mensajes de cumplimiento se entregan únicamente por ID numérico, es importante diseñar esquemas de almacenamiento que mantengan la relación entre la Publicación y el usuario a partir de ID numéricos. Los consumidores de datos deberán supervisar los eventos de cumplimiento tanto por ID de Publicación como por ID de usuario y poder actualizar adecuadamente el almacén de datos local.

* **Cree esquemas que aborden todos los estados de cumplimiento**: Dependiendo de cómo se gestionen las actividades de cumplimiento en varias aplicaciones, puede ser necesario agregar otros metadatos al almacén de datos. Por ejemplo, los consumidores de datos pueden decidir agregar metadatos a una base de datos existente para facilitar la restricción de la visualización de contenido en países afectados por un mensaje `status_withheld`.

* **Gestión de la eliminación de Retweets**: Los Retweets son un tipo especial de Publicación en la que el mensaje original está anidado en un objeto dentro del Retweet. En este caso, se hace referencia a dos ID de Publicación en un Retweet: el ID del Retweet y el ID del mensaje original (incluido en el objeto anidado). Cuando se elimina un mensaje original, se emite un mensaje de eliminación de Publicación para el ID original. Los mensajes de eliminación posteriores NO se emiten para todos los Retweets. La eliminación del ID original debería bastar para eliminar todos los Retweets posteriores.
