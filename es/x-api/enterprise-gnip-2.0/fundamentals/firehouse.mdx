> **Ten en cuenta**
>
> Hemos lanzado una nueva herramienta de compliance para X API v2 llamada [batch compliance](/es/x-api/compliance/batch-compliance). Esta nueva herramienta te permite cargar grandes conjuntos de datos de IDs de Publicación o de usuario para recuperar su estado de compliance y así determinar qué datos requieren acción para que tus conjuntos de datos cumplan con las políticas de compliance.

> Además, tanto batch compliance como compliance firehose se han actualizado para admitir ediciones de Publicaciones. Para compliance firehose se añadió un nuevo evento `tweet_edit`. Consulta la documentación de [Compliance Data Objects](/es/x-api/enterprise-gnip-2.0/fundamentals/firehouse#compliance-data-objects) para más detalles. Obtén más información sobre cómo funciona la metainformación de edición de Publicaciones en la página [Fundamentos de edición de Publicaciones](/es/x-api/fundamentals/edit-posts).

<div id="overview">
  ## Descripción general
</div>

`Enterprise`

Uno de los valores fundamentales de X es defender y respetar la voz de los usuarios. Esto incluye respetar sus expectativas e intención cuando eliminan, modifican o editan el contenido que deciden compartir en X. Consideramos que esto es de importancia crítica para la salud a largo plazo de una de las plataformas públicas de información en tiempo real más grandes del mundo. X pone los controles en manos de sus usuarios, dándoles la posibilidad de controlar su propia experiencia en X. Consideramos que las empresas que reciben datos de X tienen la responsabilidad de respetar las expectativas y la intención de los usuarios finales.

Para obtener más información sobre los tipos de eventos de cumplimiento que son posibles en la plataforma de X, consulta nuestro artículo [Respetar la intención del usuario en X](/es/x-api/enterprise-gnip-2.0/fundamentals/firehouse#honoring-user-intent-on-twitter).

Cualquier desarrollador o empresa que consuma datos de X a través de una API tiene la obligación de hacer todos los esfuerzos razonables para respetar los cambios en el contenido de los usuarios. Esta obligación se extiende a eventos del usuario como eliminaciones, modificaciones y cambios en las opciones para compartir contenido (por ejemplo, cuando el contenido pasa a estar protegido o retenido). Esto también incluye cuando los usuarios editan sus Publicaciones. Consulta el texto específico en la [Política para desarrolladores](https://developer.x.com/en/developer-terms/policy) y/o en tu Acuerdo de datos de X para comprender cómo esta obligación afecta tu uso de los datos de X.

X ofrece las siguientes soluciones que proporcionan información sobre estos eventos de cumplimiento de usuarios y sobre si una Publicación o Usuario específico está disponible públicamente o no. A continuación, se presenta una breve descripción de las soluciones y sus patrones generales de integración:

<div id="get-statuseslookup-and-get-userslookup">
  #### GET statuses/lookup y GET users/lookup
</div>

* Formato: API REST. Ver: [GET statuses/lookup](https://developer.x.com/en/docs/x-api/v1/tweets/post-and-engage/api-reference/get-statuses-lookup) y [GET users/lookup](https://developer.x.com/en/docs/x-api/v1/accounts-and-users/follow-search-get-users/api-reference/get-users-lookup).

* Estos endpoints siempre devuelven la última versión de cualquier edición de una Publicación. Todos los objetos de Publicación que describen Publicaciones creadas después de que se introdujo la función de edición incluirán metadatos de edición de Publicación. Esto se aplica incluso a Publicaciones que no fueron editadas.

* Para todas las Publicaciones, las solicitudes de Publicaciones realizadas más de 30 minutos después de su creación representarán el estado final de todas las Publicaciones.

* Entregan información de disponibilidad para Publicaciones o Usuarios específicos, según lo definido por quien realiza la llamada como parte de la solicitud de API.

* Se pueden usar para comprobaciones puntuales ad hoc sobre el estado de disponibilidad actual de un grupo específico de Publicaciones o Usuarios.

* Son ideales para clientes que necesitan una forma de comprobar el estado actual de una Publicación o Usuario específico en un momento determinado.

* Estas APIs proporcionan un mecanismo útil que pueden utilizar los clientes que necesitan comprobar la disponibilidad de un elemento de Contenido, por ejemplo, cuando:
  1. Muestran Publicaciones
  2. Interactúan con una o varias Publicaciones o Usuarios de forma 1:1
  3. Distribuyen Contenido de X a un tercero mediante una descarga de archivo permitida
  4. Almacenan Publicaciones durante períodos prolongados

<div id="compliance-firehose-enterprise-only">
  #### Compliance Firehose (solo para Enterprise)
</div>

* Formato: API de streaming. Consulta: [Compliance Firehose](/es/x-api/enterprise-gnip-2.0/fundamentals/firehouse#api-reference).
* Entrega un flujo en tiempo real de [actividades de cumplimiento](/es/x-api/enterprise-gnip-2.0/fundamentals/firehouse#compliance-data-objects) en X. Estas actividades incluyen cuando se editan las Publicaciones.
* Puede utilizarse para mantener el estado de cumplimiento de un conjunto de datos almacenados a medida que se producen nuevos eventos de cumplimiento en la plataforma.
* Ideal para clientes que consumen y almacenan grandes volúmenes de datos de X durante largos periodos de tiempo.

<div id="recommendations-best-practices">
  #### **Recomendaciones y mejores prácticas**
</div>

* **Cree esquemas de almacenamiento de datos que almacenen el ID numérico de la Publicación y el ID del Usuario**: Los mensajes de los usuarios requieren que se tomen medidas sobre todas las Publicaciones de ese Usuario. Por lo tanto, dado que todos los mensajes de cumplimiento se entregan solo por ID numérico, es importante diseñar esquemas de almacenamiento que mantengan la relación entre la Publicación y el Usuario basándose en IDs numéricos. Los consumidores de datos deberán supervisar los eventos de cumplimiento tanto por ID de Publicación como por ID de Usuario y poder actualizar el almacén de datos local de forma adecuada.

* **Cree esquemas que contemplen todos los estados de cumplimiento**: Dependiendo de cómo se aborden las actividades de cumplimiento en distintas aplicaciones, puede ser necesario agregar otros metadatos al almacén de datos. Por ejemplo, los consumidores de datos pueden decidir agregar metadatos a una base de datos existente para facilitar la restricción de la visualización de contenido en países afectados por un mensaje `status_withheld`.

* **Gestión de eliminaciones de Retweets**: Los Retweets son un tipo especial de Publicación en la que el mensaje original está anidado en un objeto dentro del Retweet. En este caso, hay dos IDs de Publicación referenciados en un Retweet: el ID del Retweet y el ID del mensaje original (incluido en el objeto anidado). Cuando se elimina un mensaje original, se emite un mensaje de eliminación de Publicación para el ID original. Los eventos de eliminación de Publicaciones normalmente desencadenan eventos de eliminación para todos los Retweets. Sin embargo, en algunos casos no se envían todos y los sistemas cliente deben ser tolerantes ante eliminaciones de Retweets incompletas. La eliminación del ID original debería ser suficiente para eliminar todos los Retweets posteriores. Es una práctica recomendada hacer referencia al ID de la Publicación original al almacenar Retweets y eliminar todos los Retweets referenciados al recibir eventos de eliminación de Publicaciones.

<div id="compliance-firehose-api">
  #### **API de Compliance Firehose**
</div>

Los posibles tipos de eventos de cumplimiento incluirán eventos de Publicación (o “status”) y eventos de Usuario, para los cuales hay varios tipos descritos a continuación.  

Tenga en cuenta:

* Lea más sobre los estados de la cuenta de usuario [aquí](https://support.x.com/articles/14016) y sobre nuestra política para desarrolladores en relación con las Publicaciones eliminadas [aquí](https://dev.x.com/overview/terms/policy#3.Update_Respect_Users_Control_and_Privacy).
* El Compliance Firehose se ha actualizado para proporcionar eventos `tweet_edit`. 
* Varios eventos de eliminación, protección y suspensión de Usuario no son necesariamente permanentes y pueden alternar entre estados indefinidamente. Estos incluyen: user&#95;delete, user&#95;undelete, user&#95;protect, user&#95;unprotect y user&#95;suspend, user&#95;unsuspend.
* Los user&#95;deletes van seguidos de status&#95;deletes 30 días después solo si el usuario no ha decidido user&#95;undelete su cuenta. Es posible que un user&#95;delete sea revertido por el usuario y que las eliminaciones de todas sus Publicaciones 30 días después no se produzcan.
* user&#95;suspend es una acción que sigue siendo válida a menos que el usuario sea objeto de un evento user&#95;unsuspend. Estos no están sujetos a ningún cambio en un período de 30 días.

Consulte la columna ‘Acción recomendada’ para entender cómo procesar cada tipo de evento a fin de respetar la privacidad y la intención del usuario final.

| :--- | :--- | :--- | :--- |
| delete | Status | Sí | Eliminar la Publicación asociada. |

| user&#95;protect | User | No  | Suprimir o eliminar todas las Publicaciones del usuario asociado. |

<div id="integrating-compliance-firehose">
  ### Integración con Compliance Firehose
</div>

Compliance Firehose es una API de streaming en tiempo real que entrega eventos de cumplimiento que se producen en la plataforma X. Para comprender los eventos de cumplimiento y cómo se generan en X, consulta nuestro artículo [Honrar la intención del usuario en X](/es/x-api/enterprise-gnip-2.0/fundamentals/firehouse#honoring-user-intent-on-twitter).

Es importante tener en cuenta que los eventos de Publicación y de usuario se entregan de forma independiente y que cada uno debe procesarse por separado (es decir, la eliminación de una Publicación no implica un evento de usuario y viceversa). Varios eventos de usuario no son necesariamente permanentes y pueden alternar entre estados indefinidamente. Estos incluyen: user&#95;delete, user&#95;undelete, user&#95;protect, user&#95;unprotect y user&#95;suspend, user&#95;unsuspend.

Los user&#95;deletes van seguidos de status&#95;deletes 30 días después solo si el usuario no ha optado por realizar un user&#95;undelete de su cuenta. Es posible que un user&#95;delete sea revertido por el usuario y que las eliminaciones de todas sus Publicaciones 30 días después no se produzcan.

User&#95;suspend es una acción que permanece activa a menos que el usuario sea objeto de un evento user&#95;unsuspend. Estos no están sujetos a ningún cambio asociado a un período de 30 días.

Nunca es adecuado mostrar directamente eventos de cumplimiento a los usuarios de tu software ni incorporarlos de otro modo en tus productos o experiencias de cliente. Están destinados únicamente a mantener el cumplimiento y a respetar las acciones de los usuarios de X.

<div id="integrating-with-the-compliance-firehose">
  #### **Integración con el Compliance Firehose**
</div>

Para integrar el Compliance Firehose en tu sistema, debes crear una integración que pueda hacer lo siguiente:

1. Establecer una conexión de streaming con cada partición de la API de streaming del Compliance Firehose
2. Gestionar volúmenes altos de datos: desacoplar la ingesta del stream del procesamiento adicional utilizando procesos asíncronos
3. Volver a conectarse automáticamente a las particiones del stream cuando se desconecte por cualquier motivo
4. Procesar los eventos de cumplimiento que sean relevantes para los datos de Publicación y de Usuario que hayas almacenado, de acuerdo con la guía presentada anteriormente

<div id="honoring-user-intent-on-twitter">
  ### Respetar la intención del usuario en X
</div>

Creemos que respetar la privacidad y la intención de los usuarios de X es fundamental para la salud a largo plazo de una de las plataformas de información pública y en tiempo real más grandes del mundo. X pone los controles de privacidad en manos de sus usuarios, dándoles a las personas la posibilidad de controlar su propia experiencia en X. Como clientes empresariales de los datos de X, tenemos la responsabilidad colectiva de respetar la privacidad y las acciones de los usuarios finales para mantener este entorno de confianza y respeto.

Hay diversas cosas que pueden suceder a las Publicaciones y a las cuentas de usuario que afectan la forma en que se muestran en la plataforma. Las acciones que afectan la privacidad y la intención se definen tanto a nivel de Estado (Publicación) como a nivel de Usuario. Estas acciones incluyen:

<div id="user">
  #### Usuario
</div>

| Acción | Descripción |
| :--- | :--- |
| Proteger cuenta | Un usuario de X puede proteger o desproteger su cuenta en cualquier momento. Las cuentas protegidas requieren la aprobación manual, por parte del usuario, de cada persona a la que se le permite ver las Publicaciones de su cuenta.   <br />Para obtener más información, consulta [Acerca de las Publicaciones públicas y protegidas](https://support.x.com/articles/14016-about-public-and-protected-tweets). |
| Eliminar cuenta | Un usuario de X puede decidir eliminar su cuenta y todos los mensajes de estado asociados en cualquier momento. X conserva la información de la cuenta durante 30 días después de la eliminación en caso de que el usuario decida recuperarla y, en efecto, reactivar la cuenta. |
| Eliminar datos de ubicación | Un usuario de X puede eliminar todos los datos de ubicación de Publicaciones anteriores en cualquier momento. Esto se conoce como “scrub geo”. |
| Suspender cuenta | X se reserva el derecho de suspender las cuentas que infrinjan las Reglas de X o si se sospecha que una cuenta ha sido hackeada o comprometida. Las suspensiones de cuenta solo pueden revertirse (levantarse) por X. |
| Retener cuenta | X se reserva el derecho de restringir de forma reactiva el acceso a cierto contenido en un país específico de vez en cuando. Una cuenta retenida solo puede dejar de estar retenida por X.   <br />Para obtener más información, consulta [Contenido retenido por país](https://support.x.com/articles/20169222-country-withheld-content). |

<div id="status">
  #### Estado
</div>

| Acción | Descripción |
| :--- | :--- |
| Eliminar estado | Un usuario de X puede eliminar un estado en cualquier momento. Los estados eliminados no se pueden revertir y se eliminan de forma permanente. |
| Retener estado | X se reserva el derecho de retener de forma reactiva el acceso a determinado contenido en un país específico, ocasionalmente. Solo X puede levantar la retención de un estado.   <br />Para obtener más información, consulta [Country Withheld Content](https://support.x.com/articles/20169222-country-withheld-content). |

<div id="keeping-track-of-user-and-status-changes">
  #### Seguimiento de cambios en Usuarios y Estados
</div>

El estado de un Usuario o de un Estado puede cambiar en cualquier momento debido a una de las acciones anteriores, y esto afecta la forma en que se espera que los consumidores de datos de X gestionen la disponibilidad y la privacidad de todo el contenido asociado. Cuando ocurren estas acciones, se envía un mensaje de cumplimiento normativo correspondiente que indica que el estado de un Usuario o de un Estado ha cambiado.

  #### Códigos de respuesta

Las siguientes respuestas pueden ser devueltas por la API para estas solicitudes. La mayoría de los códigos de error se devuelven con una cadena de texto con detalles adicionales en el cuerpo. Para respuestas distintas de 200, los clientes deben intentar reconectarse.

| Status | Text | Definition |
| :--- | :--- | :--- |
| 200 | Success | La conexión se abrió correctamente y se enviarán nuevas actividades a medida que vayan llegando. |
| 401 | Unauthorized | La autenticación HTTP falló debido a credenciales no válidos. Inicia sesión en console.gnip.com con tus credenciales para asegurarte de que las estés usando correctamente en tu solicitud. |
| 406 | Not Acceptable | Generalmente, esto ocurre cuando tu cliente no incluye correctamente los encabezados para aceptar la codificación gzip desde el flujo, pero también puede ocurrir en otras circunstancias. Contendrá un mensaje JSON similar a: &quot;Esta conexión requiere compresión. Para habilitar la compresión, envía un encabezado &#39;Accept-Encoding: gzip&#39; en tu solicitud y prepárate para descomprimir el flujo a medida que se lea en el lado del cliente.&quot; |
| 429 | Rate Limited | Tu aplicación ha excedido el límite de solicitudes de conexión. |
| 503 | Service Unavailable | Problema del servidor de Twitter. Vuelve a conectarte usando un patrón de retroceso exponencial. Si no se ha publicado ningún aviso sobre este problema en la [X API Status Page](https://api.twitterstat.us/), ponte en contacto con el equipo de soporte. |

<div id="other-recommendations-best-practices">
  #### Otras recomendaciones y mejores prácticas
</div>

* **Cree esquemas de almacenamiento de datos que guarden el ID numérico de la Publicación y el ID de usuario**: Los mensajes de usuario requieren que se tomen medidas sobre todas las Publicaciones de ese usuario. Por lo tanto, dado que todos los mensajes de cumplimiento se entregan únicamente por ID numérico, es importante diseñar esquemas de almacenamiento que mantengan la relación entre la Publicación y el usuario basándose en IDs numéricos. Los consumidores de datos deberán supervisar los eventos de cumplimiento tanto por el ID de la Publicación como por el ID de usuario y poder actualizar el almacén de datos local de forma adecuada.

* **Cree esquemas que contemplen todos los estados de cumplimiento**: Dependiendo de cómo se gestionen las actividades de cumplimiento en distintas aplicaciones, puede ser necesario añadir otros metadatos al almacén de datos. Por ejemplo, los consumidores de datos pueden decidir añadir metadatos a una base de datos existente para facilitar la restricción de la visualización del contenido en los países afectados por un mensaje `status_withheld`.

* **Gestión de eliminaciones de Retuit**: Los Retuits son un tipo especial de Publicación en el que el mensaje original está anidado en un objeto dentro del Retuit. En este caso, hay dos IDs de Publicación referenciados en un Retuit: el ID del Retuit y el ID del mensaje original (incluido en el objeto anidado). Cuando se elimina un mensaje original, se emite un mensaje de eliminación de Publicación para el ID original. No se emiten mensajes de eliminación adicionales para cada uno de los Retuits. La eliminación del ID original debería ser suficiente para eliminar todos los Retuits posteriores.
