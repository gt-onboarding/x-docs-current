title: Límites de frecuencia
keywords: ["límites de frecuencia", "límites de frecuencia de la API", "límites de solicitudes", "limitación de frecuencia", "restricción de tráfico", "ventana de 15 minutos", "errores de límites de frecuencia"]
---

Cada día, miles de desarrolladores realizan solicitudes a la X API. Para ayudar a gestionar el gran volumen de estas solicitudes, se establecen límites en la cantidad de solicitudes que se pueden realizar. Estos límites ayudan a proporcionar la API confiable y escalable en la que nuestra comunidad de desarrolladores confía.

El número máximo de solicitudes permitidas se basa en un intervalo de tiempo, normalmente durante un período o ventana temporales específicos. El intervalo más común es de quince minutos. Por ejemplo, un endpoint con un límite de 900 solicitudes por 15 minutos permite hasta 900 solicitudes en cualquier intervalo de 15 minutos.

Los límites de frecuencia dependen del método de autenticación. Por ejemplo, si se usa [Contexto de usuario de OAuth 1.0a](/es/resources/fundamentals/authentication), cada conjunto de [Tokens de acceso](/es/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) de los usuarios tiene su propio límite de frecuencia por período. Alternativamente, si se usa [OAuth 2.0 Bearer Token](/es/resources/fundamentals/authentication#oauth-2-0), tu aplicación tendrá su propio límite independiente por período de tiempo. Cuando se exceden estos límites, la API devuelve un error.

<div id="rate-limits-and-authentication-method">
  ## Límites de frecuencia y método de autenticación
</div>

Los límites de frecuencia se establecen tanto a nivel de la aplicación de desarrollador como del token de acceso de usuario:

* **[OAuth 2.0 Bearer Token](/es/resources/fundamentals/authentication/oauth-2-0/application-only): Límite de frecuencia de la aplicación**
  Este método te permite realizar una cierta cantidad de solicitudes en nombre de tu aplicación de desarrollador. Al usar este método de autenticación, los límites se determinan por las solicitudes realizadas con un Bearer Token.
  * Ejemplo: Con un límite de 450 solicitudes por intervalo de 15 minutos, puedes realizar 450 solicitudes en nombre de tu aplicación dentro de ese intervalo.

* **[OAuth 1.0a User Context](/es/resources/fundamentals/authentication/oauth-1-0a/obtaining-user-access-tokens): Límite de frecuencia de usuario**
  Este método permite realizar solicitudes en nombre de un usuario de X identificado por el token de acceso de usuario. Por ejemplo, si se recuperan métricas privadas de Publicaciones, autentícate con tokens de acceso de usuario para ese usuario, generados mediante el [flujo OAuth de tres patas](/es/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow).
  * Ejemplo: Con un límite de 900 solicitudes por 15 minutos por usuario, puedes realizar hasta 900 solicitudes por usuario en ese período de tiempo.

<div id="http-headers-and-response-codes">
  ## Encabezados HTTP y códigos de respuesta
</div>

Usa los encabezados HTTP para entender cuál es la situación de tu aplicación dentro de un límite de solicitudes determinado, según la solicitud más reciente realizada.

* `x-rate-limit-limit`: límite máximo de solicitudes para el endpoint
* `x-rate-limit-remaining`: solicitudes restantes en la ventana de 15 minutos
* `x-rate-limit-reset`: tiempo restante antes de que se restablezca el límite de solicitudes (en segundos epoch UTC)

<div id="error-responses">
  ### Respuestas de error
</div>

Si una aplicación supera el límite de frecuencia de solicitudes para un endpoint, la API devolverá una respuesta [HTTP 429 “Too Many Requests”](http://tools.ietf.org/html/rfc6585) con el siguiente mensaje de error en el cuerpo de la respuesta:

```json
{ "errors": [ { "code": 88, "message": "Rate limit exceeded" } ] }
```

<div id="recovering-from-a-rate-limit">
  ## Recuperarse de un límite de frecuencia
</div>

Cuando se superan estos límites de frecuencia, el endpoint devuelve un error 429 &quot;Too many requests&quot;. Como se analiza a continuación, cuando se producen errores de límite de frecuencia, una buena práctica es examinar los encabezados HTTP que indican cuándo se restablece el límite y pausar las solicitudes hasta ese momento.<br />Cuando se produce un error de &quot;too many requests&quot; o de limitación de frecuencia, es necesario reducir la frecuencia con la que se realizan las solicitudes. Cuando se genera un error de límite de frecuencia, se puede comprobar el encabezado HTTP `x-rate-limit-reset:` para saber cuándo se restablecerá la limitación de frecuencia.<br /><br />Otro patrón común se basa en la espera exponencial (exponential backoff), donde el tiempo entre solicitudes comienza siendo pequeño (por ejemplo, unos pocos segundos) y luego se duplica antes de cada nuevo intento. Esto continúa hasta que una solicitud tiene éxito o se alcanza un tiempo máximo razonable entre solicitudes (por ejemplo, unos pocos minutos).<br /><br />Idealmente, la aplicación cliente es consciente de los límites de frecuencia existentes y puede pausar las solicitudes hasta que expire la ventana que se ha superado. Si excedes un límite de 15 minutos, tiene sentido esperar uno o dos minutos antes de volver a intentarlo.<br /><br />Ten en cuenta que, más allá de estos límites en el número de solicitudes, el nivel de acceso Standard Basic proporciona hasta 500 000 Publicaciones por mes a partir de los endpoints de búsqueda reciente y flujo filtrado. Si has superado el límite mensual en el número de Publicaciones, entonces tiene más sentido que tu Aplicación muestre una notificación, conozca su día de renovación mensual y espere para reanudar las solicitudes hasta ese día.

<div id="tips-to-avoid-being-rate-limited">
  ### Consejos para evitar límites de frecuencia
</div>

Los siguientes consejos están pensados para ayudarte a programar de forma defensiva y reducir la posibilidad de que se te apliquen límites de frecuencia. Algunas funciones de la aplicación que quizá quieras ofrecer son simplemente inviables debido a estos límites, especialmente en lo relativo a la actualidad de los resultados. Si el objetivo de tu aplicación es disponer de información en tiempo real, revisa los endpoints de streams filtrados y muestreados.

<div id="caching">
  #### Almacenamiento en caché
</div>

Almacena las respuestas de la API si prevés un uso frecuente. En lugar de llamar a la API en cada carga de página, almacena la respuesta en caché de forma local.

<div id="prioritize-active-users">
  #### Prioriza a los usuarios activos
</div>

Si tu sitio realiza un seguimiento de muchos usuarios de X (por ejemplo, al obtener su estado actual o estadísticas sobre su uso de X), considera solicitar datos únicamente de los usuarios que hayan iniciado sesión recientemente en tu sitio.

<div id="adapt-to-the-search-results">
  #### Adáptate a los resultados de búsqueda
</div>

Si tu aplicación supervisa un volumen alto de términos de búsqueda, realiza consultas con menos frecuencia en aquellas búsquedas que no arrojan resultados que en las que sí los arrojan. Al usar una estrategia de backoff puedes mantener actualizadas las consultas populares sin desperdiciar recursos de procesamiento solicitando consultas que casi nunca cambian. Como alternativa, considera usar el endpoint filtered stream y aplicar filtros con tus consultas de búsqueda.

<div id="denylist">
  #### Lista de bloqueo
</div>

Si tú o tu aplicación han sido incluidos en la lista de bloqueo y crees que ha habido un error, puedes usar nuestros [formularios de Platform Support](https://support.x.com/forms/platform) para solicitar asistencia.
