title: Límites de frecuencia
keywords: ["límites de frecuencia", "límites de frecuencia de la API", "límites de solicitudes", "limitación de frecuencia", "restricción de solicitudes", "ventana de 15 minutos", "errores de límites de frecuencia"]
---

Cada día, muchos miles de desarrolladores realizan solicitudes a la X API. Para ayudar a gestionar el gran volumen de estas solicitudes, se establecen límites en la cantidad de solicitudes que se pueden realizar. Estos límites ayudan a ofrecer una API confiable y escalable en la que nuestra comunidad de desarrolladores confía.

El número máximo de solicitudes permitidas se basa en un intervalo de tiempo, normalmente durante un período o ventana de tiempo específicos. El intervalo más común es de quince minutos. Por ejemplo, un endpoint con un límite de 900 solicitudes por 15 minutos permite hasta 900 solicitudes en cualquier intervalo de 15 minutos.

Los límites de frecuencia dependen del método de autenticación. Por ejemplo, si usas [Contexto de usuario de OAuth 1.0a](/es/resources/fundamentals/authentication), cada conjunto de [tokens de acceso](/es/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) de los usuarios tiene su propio límite de frecuencia por período. Alternativamente, si usas [OAuth 2.0 Bearer Token](/es/resources/fundamentals/authentication#oauth-2-0), tu aplicación tendrá su propio límite independiente por período de tiempo. Cuando se exceden estos límites, la API devuelve un error.

<div id="rate-limits-and-authentication-method">
  ## Límites de frecuencia y método de autenticación
</div>

Los límites de frecuencia se establecen tanto a nivel de la aplicación de desarrollador como del token de acceso de usuario:

* **[OAuth 2.0 Bearer Token](/es/resources/fundamentals/authentication/oauth-2-0/application-only): Límite de frecuencia de la aplicación**
  Este método te permite realizar un cierto número de solicitudes en nombre de tu aplicación de desarrollador. Cuando utilizas este método de autenticación, los límites se determinan por las solicitudes realizadas con un Bearer Token.
  * Ejemplo: Con un límite de 450 solicitudes por intervalo de 15 minutos, puedes realizar 450 solicitudes en nombre de tu aplicación dentro de ese intervalo.

* **[Contexto de usuario de OAuth 1.0a](/es/resources/fundamentals/authentication/oauth-1-0a/obtaining-user-access-tokens): Límite de frecuencia de usuario**
  Este método permite realizar solicitudes en nombre de un usuario de X identificado por el token de acceso de usuario. Por ejemplo, si recuperas métricas privadas de Publicaciones, autentícate con tokens de acceso de usuario para ese usuario, generados mediante el [flujo OAuth de tres patas](/es/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow).
  * Ejemplo: Con un límite de 900 solicitudes por 15 minutos por usuario, puedes realizar hasta 900 solicitudes por usuario en ese período de tiempo.

<div id="http-headers-and-response-codes">
  ## Encabezados HTTP y códigos de respuesta
</div>

Usa los encabezados HTTP para entender en qué punto se encuentra tu aplicación respecto a un límite de frecuencia determinado, en función de la solicitud más reciente realizada.

* `x-rate-limit-limit`: límite máximo de solicitudes para el endpoint
* `x-rate-limit-remaining`: solicitudes restantes para la ventana de 15 minutos
* `x-rate-limit-reset`: tiempo restante antes de que se restablezca el límite de frecuencia (en segundos desde la época UTC)

<div id="error-responses">
  ### Respuestas de error
</div>

Si una aplicación excede el límite de frecuencia de un endpoint, la API devolverá una respuesta [HTTP 429 “Too Many Requests”](http://tools.ietf.org/html/rfc6585) con el siguiente mensaje de error en el cuerpo de la respuesta:

```json
{ "errors": [ { "code": 88, "message": "Rate limit exceeded" } ] }
```

<div id="recovering-from-a-rate-limit">
  ## Recuperación tras alcanzar un límite de frecuencia
</div>

Cuando se superan estos límites de frecuencia, el endpoint devuelve un error 429 &quot;Too many requests&quot;. Como se explica a continuación, cuando se producen errores de límite de frecuencia, una práctica recomendada es examinar las cabeceras HTTP que indican cuándo se reinicia el límite y pausar las solicitudes hasta entonces.<br />Cuando se produce un error de &quot;too many requests&quot; o de límite de frecuencia, es necesario reducir la frecuencia con la que se realizan solicitudes. Cuando se alcanza un error de límite de frecuencia, se puede comprobar la cabecera HTTP `x-rate-limit-reset:` para saber cuándo se restablecerá el límite de frecuencia<br /><br />. Otro patrón habitual se basa en el backoff exponencial (exponential backoff), en el que el tiempo entre solicitudes empieza siendo pequeño (por ejemplo, unos segundos) y luego se duplica antes de cada reintento. Esto se mantiene hasta que una solicitud tiene éxito o se alcanza un tiempo máximo razonable entre solicitudes (por ejemplo, unos minutos).<br /><br /> Idealmente, el cliente debería ser consciente de los límites de frecuencia existentes y poder pausar las solicitudes hasta que expire la ventana que se ha superado. Si superas un límite de 15 minutos, entonces tiene sentido esperar uno o dos minutos antes de volver a intentarlo.<br /><br /> Ten en cuenta que, más allá de estos límites en el número de solicitudes, el nivel de acceso Standard Basic proporciona hasta 500.000 Publicaciones al mes mediante los endpoints de búsqueda reciente y flujo filtrado. Si has superado el límite mensual en el número de Publicaciones, entonces tiene más sentido que tu aplicación muestre una notificación, conozca su día de renovación mensual y detenga las solicitudes hasta ese día.

<div id="tips-to-avoid-being-rate-limited">
  ### Consejos para evitar límites de frecuencia
</div>

Los siguientes consejos están pensados para ayudarte a programar de forma defensiva y reducir la probabilidad de que se apliquen límites de frecuencia. Algunas funciones de la aplicación que quizá quieras ofrecer son simplemente imposibles debido a estos límites, especialmente en lo que respecta a la actualidad de los resultados. Si tu aplicación requiere información en tiempo real, consulta los endpoints de streaming filtrado y muestreado.

<div id="caching">
  #### Caché
</div>

Almacena en caché las respuestas de la API cuando preveas un uso frecuente. En lugar de llamar a la API en cada carga de página, guarda la respuesta localmente.

<div id="prioritize-active-users">
  #### Prioriza usuarios activos
</div>

Si tu sitio hace un seguimiento de muchos usuarios de X (por ejemplo, consultando su estado actual o estadísticas sobre su uso de X), considera solicitar datos únicamente de los usuarios que hayan iniciado sesión recientemente en tu sitio.

<div id="adapt-to-the-search-results">
  #### Adáptate a los resultados de búsqueda
</div>

Si tu aplicación supervisa un gran volumen de términos de búsqueda, realiza consultas con menos frecuencia para las búsquedas que no arrojan resultados que para aquellas que sí los tienen. Al aplicar un backoff puedes mantenerte al día en las consultas que son populares sin desperdiciar recursos solicitando consultas que casi nunca cambian. Como alternativa, considera usar el filtered stream endpoint y aplicar tus consultas de búsqueda como filtros.

</div>

Si tú o tu aplicación estáis en la lista de denegación y crees que ha habido un error, puedes usar nuestros [formularios de soporte de la plataforma](https://support.x.com/forms/platform) para solicitar asistencia.
