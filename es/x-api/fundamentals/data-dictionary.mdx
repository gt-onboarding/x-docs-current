<div id="tweet">
  ### Publicación
</div>

Las publicaciones son el bloque de construcción básico de todo en Twitter. El objeto de Publicación tiene una larga lista de campos de “nivel raíz”, como `id`, `text` y `created_at`. Los objetos de Publicación también son el objeto “principal” de varios objetos hijo, incluidos `user`, `media`, `poll` y `place`. Usa el parámetro de campos `tweet.fields` cuando solicites estos campos de nivel raíz en el objeto de Publicación.

El objeto de Publicación se puede encontrar y ampliar en el recurso de usuario. También se pueden encontrar y ampliar publicaciones adicionales relacionadas con la publicación solicitada en el recurso de Publicación. El objeto está disponible como expansión con `?expansions=pinned_tweet_id` en el recurso de usuario o con `?expansions=referenced_tweets.id` en el recurso de Publicación para obtener el objeto solo con los campos predeterminados. Usa la expansión junto con el parámetro de campos `tweet.fields` cuando solicites campos adicionales para completar el objeto.

**Recuperar un objeto de Publicación**

**Solicitud de ejemplo**

En la siguiente solicitud, solicitamos campos de la Publicación en el endpoint [Tweets lookup](/es/resources/fundamentals/rate-limits). Asegúrate de reemplazar `$BEARER_TOKEN` por tu propio [Bearer Token](/es/resources/fundamentals/authentication) generado.

```bash
curl --request GET 'https://api.x.com/2/tweets?ids=1212092628029698048&tweet.fields=attachments,author_id,context_annotations,created_at,entities,geo,id,in_reply_to_user_id,lang,possibly_sensitive,public_metrics,referenced_tweets,text,withheld&expansions=referenced_tweets.id' --header 'Authorization: Bearer $BEARER_TOKEN'
```

<div id="user">
  ### User
</div>

El objeto user contiene metadatos de la cuenta de usuario de Twitter que describen al usuario al que se hace referencia. El objeto user es el objeto principal que se devuelve en el endpoint [users lookup](/es/x-api/users/lookup/introduction). Al solicitar campos de usuario adicionales en este endpoint, simplemente utiliza el parámetro `user.fields`.

El objeto user también puede encontrarse como un objeto secundario y expandirse en el objeto Publicación. El objeto está disponible para expansión mediante `?expansions=author_id` o `?expansions=in_reply_to_user_id` para obtener el objeto condensado con solo los campos predeterminados. Usa la expansión junto con el parámetro `user.fields` cuando solicites campos adicionales para completar el objeto.

<div id="space">
  ### Space
</div>

Los Spaces permiten la expresión y la interacción mediante conversaciones de audio en vivo. El diccionario de datos de Space contiene metadatos relevantes sobre un Space; todos los detalles se actualizan en tiempo real.

Los objetos de usuario se pueden encontrar y expandir en el recurso de usuario. Estos objetos están disponibles para su expansión añadiendo al menos uno de `host_ids`, `creator_id`, `speaker_ids`, `mentioned_user_ids` al parámetro de consulta `expansions`.

A diferencia de las Publicaciones, los Spaces son efímeros y dejan de estar disponibles después de que finalizan o cuando se cancelan por parte de su creador. Cuando tu aplicación maneja datos de Spaces, eres responsable de devolver la información más actualizada y debes eliminar los datos que ya no estén disponibles en la plataforma. Los [endpoints de búsqueda de Spaces](/es/x-api/spaces/lookup/introduction) pueden ayudarte a garantizar que respetas las expectativas y la intención de los usuarios.

<div id="direct-message-events">
  ### Eventos de Mensaje Directo
</div>

Las conversaciones de Mensajes Directos (DM) están compuestas por eventos. La X API v2 actualmente admite tres tipos de eventos: MessageCreate, ParticipantsJoin y ParticipantsLeave.

Los objetos de eventos de DM se devuelven mediante los endpoints de [búsqueda de Mensajes Directos](/es/x-api/direct-messages/lookup/introduction), y se crea un evento MessageCreate cuando los Mensajes Directos se crean correctamente con los endpoints de [Administrar Mensajes Directos](/es/x-api/direct-messages/lookup/introduction).

Al solicitar eventos de DM, se incluyen tres atributos predeterminados del objeto de evento, o campos: id, event&#95;type y text. Para recibir campos de eventos adicionales, usa el parámetro [fields](/es/x-api/fundamentals/fields) dm&#95;event.fields para seleccionar otros. Otros campos de eventos disponibles incluyen lo siguiente: dm&#95;conversation&#95;id, created&#95;at, sender&#95;id, attachments, participant&#95;ids y referenced&#95;tweets.

Varios de estos campos proporcionan los ID de otros objetos de X relacionados con el evento de Mensaje Directo:

* sender&#95;id - El ID de la cuenta que envió el mensaje o que invitó a un participante a una conversación de grupo
* partricipants&#95;ids - Un array de ID de cuenta. Para los eventos ParticipantsJoin y ParticipantsLeave, este array contendrá un único ID de la cuenta que creó el evento
* attachments - Proporciona ID de contenido multimedia para el contenido que el remitente ha subido a X
* referenced&#95;tweets - Si se encuentra una URL de una Publicación en el campo text, el ID de esa Publicación se incluye en la respuesta

Las [expansions](/es/x-api/fundamentals/expansions) sender&#95;id, participant&#95;ids, referenced&#95;tweets.id y attachments.media&#95;keys están disponibles para expandir estos ID de objetos de X.

<div id="how-to-use-fields-and-expansions">
  ## Cómo usar campos y expansiones
</div>

De forma predeterminada, los objetos de datos de la X API v2 incluyen una pequeña cantidad de campos predeterminados cuando realizas una solicitud sin usar los parámetros [fields](/es/x-api/fundamentals/fields) o [expansions](/es/x-api/fundamentals/expansions). Esta guía te mostrará cómo usar los parámetros de consulta `fields` y `expansions` en tu solicitud para recibir objetos y campos adicionales en tu respuesta.

En esta guía, solicitaremos varios campos presentes en la siguiente captura de pantalla de una Publicación.
 

![Esta imagen incluye una captura de pantalla de una Publicación de @X. Puedes ver el texto de la Publicación, el nombre de usuario, la fecha y hora de publicación, la fuente y las métricas públicas. También incluye un video.](https://cdn.cms-twdigitalassets.com/content/dam/developer-twitter/ScreenShotDataDictionaryGuide.png.twimg.1920.png)

Como puedes ver en la captura de pantalla, hay varios elementos visibles de información relacionados con la Publicación, incluida la persona autora de la Publicación, las métricas de la Publicación, la marca de tiempo de creación, el video y la cantidad de visualizaciones del video. También hay varios datos que no son visibles en la captura de pantalla, pero que igualmente están disponibles para solicitar. 

Al realizar una solicitud a la API, la respuesta predeterminada es sencilla y contiene solo los campos predeterminados de la Publicación (id y text). También solo recibirás el objeto principal que se devuelve con el endpoint específico que estés usando, y no los objetos de datos asociados que puedan estar relacionados con el objeto principal.

Esta simplicidad, junto con los parámetros `fields` y `expansions`, te permite solicitar únicamente aquellos campos que necesitas, según tu caso de uso. 
 

<div id="requesting-additional-fields-and-objects">
  #### Solicitud de campos y objetos adicionales.
</div>

En primer lugar, solicitaremos un [objeto Publicación](/es/x-api/fundamentals/data-dictionary#tweet) mediante un ID de Publicación y el [endpoint GET /tweets](/es/x-api/posts/lookup/introduction).

Solicitud:

```bash
curl --request GET --url 'https://api.x.com/2/tweets?ids=1260294888811347969' \
  --header 'Authorization: Bearer $BEARER_TOKEN'
```

Respuesta:

```
{
    "data": [
        {
            "id": "1260294888811347969",
            "text": "No te pierdas las Publicaciones sobre tu Publicación. \n\nAhora en iOS, puedes ver los Retuits con comentarios en un solo lugar. https://t.co/oanjZfzC6y"
        }
    ]
}
```

La siguiente guía paso a paso te mostrará cómo recuperar los datos adicionales que se pueden ver en la captura de pantalla.

1. Identifica los campos adicionales que quieras solicitar usando nuestro [modelo de objetos](/es/x-api/fundamentals/data-dictionary#tweet) o revisando la lista de campos en las páginas de referencia de la API de cada endpoint.

   En este caso, solicitaremos los siguientes campos adicionales:
   attachments, author&#95;id, created&#95;at, public&#95;metrics.

2. Crea el parámetro de consulta `tweet.fields` con los campos anteriores como su valor, usando una lista separada por comas:
   `?tweet.fields=attachments,author_id,created_at,public_metrics`

3. Agrega el parámetro de consulta a la solicitud GET /tweets que realizaste anteriormente.

Solicitud:

`curl --request GET --url 'https://api.x.com/2/tweets?ids=1260294888811347969&tweet.fields=attachments,author_id,created_at,public_metrics' \   --header 'Authorization: Bearer $BEARER_TOKEN'`

Respuesta:

```
{
    "data": [
        {
            "id": "1260294888811347969",
            "text": "No te pierdas las Publicaciones sobre tu Publicación. \n\nAhora en iOS, puedes ver los Retuits con comentarios en un solo lugar. https://t.co/oanjZfzC6y",
            "author_id": "783214",
            "public_metrics": {
                "retweet_count": 5219,
                "reply_count": 1828,
                "like_count": 17141,
                "quote_count": 3255
            },
            "attachments": {
                "media_keys": [

                ]
            },
            "created_at": "2020-05-12T19:44:51.000Z"

}
```

?expansions=attachments.media&#95;keys

Solicitud:

```bash
curl --request GET --url 'https://api.x.com/2/tweets?ids=1260294888811347969&tweet.fields=attachments,author_id,created_at,public_metrics&expansions=attachments.media_keys' \
  --header 'Authorization: Bearer $BEARER_TOKEN'
```

```

            "id": "1260294888811347969",
            "text": "No te pierdas las Publicaciones sobre tu Publicación. \n\nAhora en iOS, puedes ver las Republicaciones con comentarios en un solo lugar. https://t.co/oanjZfzC6y",

                "reply_count": 1828,

                "quote_count": 3255

                "media_keys": [

                ]
            },
            "author_id": "783214"

        "media": [
            {
                "media_key": "13_1260294804770041858",
                "type": "video"
            }
        ]
    }

```

5. Por último, vamos a solicitar el número de visualizaciones y la duración del video. Estos no son campos predeterminados, por lo que tenemos que solicitarlos específicamente. Utiliza el parámetro `media.fields` con los valores separados por comas `public_metrics` y `duration_ms` en tu solicitud.

?media.fields=public&#95;metrics,duration&#95;ms

Solicitud:
 

`curl --request GET --url 'https://api.x.com/2/tweets?ids=1260294888811347969&tweet.fields=attachments,author_id,created_at,public_metrics&expansions=attachments.media_keys&media.fields=duration_ms,public_metrics' --header 'Authorization: Bearer $BEARER_TOKEN'`

            "attachments": {
                "media_keys": [
                    "13_1260294804770041858"
                ]
            }
        }
    ],
    "includes": {
        "media": [
            {

En total, incluimos los siguientes parámetros en este ejemplo:

* ids=1260294888811347969
* tweet.fields=attachments,author&#95;id,created&#95;at,public&#95;metrics
* expansions=attachments.media&#95;keys
* media.fields=public&#95;metrics,duration&#95;ms
   

Al combinarlos, la cadena de consulta completa se ve así:

```
?ids=1260294888811347969&tweet.fields=attachments,author\_id,created\_at,public\_metrics&expansions=attachments.media\_keys&media.fields=public\_metrics,duration\_ms
```
