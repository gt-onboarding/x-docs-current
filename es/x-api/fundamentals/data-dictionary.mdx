### Tweet

Las Publicaciones son el bloque de construcción básico de todo en X. El objeto de Publicación (Tweet) tiene una larga lista de campos de «nivel raíz», como `id`, `text` y `created_at`. Los objetos de Publicación también son el objeto «padre» de varios objetos hijo, incluidos `user`, `media`, `poll` y `place`. Usa el parámetro de campos `tweet.fields` cuando solicites estos campos de nivel raíz en el objeto de Publicación.

El objeto de Publicación se puede encontrar y ampliar en el recurso de usuario. También se pueden encontrar y ampliar Publicaciones adicionales relacionadas con la Publicación solicitada en el recurso Tweet. El objeto se puede ampliar con `?expansions=pinned_tweet_id` en el recurso de usuario o con `?expansions=referenced_tweets.id` en el recurso Tweet para obtener el objeto solo con los campos predeterminados. Usa esa expansión junto con el parámetro de campos `tweet.fields` cuando solicites campos adicionales para completar el objeto.

**Obtención de un objeto de Publicación**

**Solicitud de ejemplo**

En la siguiente solicitud se especifican campos para la Publicación en el endpoint de [Tweets lookup](/es/resources/fundamentals/rate-limits). Asegúrate de reemplazar `$BEARER_TOKEN` por tu propio [Bearer Token](/es/resources/fundamentals/authentication) generado.

```bash
curl --request GET 'https://api.x.com/2/tweets?ids=1212092628029698048&tweet.fields=attachments,author_id,context_annotations,created_at,entities,geo,id,in_reply_to_user_id,lang,possibly_sensitive,public_metrics,referenced_tweets,text,withheld&expansions=referenced_tweets.id' --header 'Authorization: Bearer $BEARER_TOKEN'
```

<div id="user">
  ### Usuario
</div>

El objeto de usuario contiene metadatos de la cuenta de usuario de Twitter que describen al usuario de referencia. El objeto de usuario es el objeto principal que se devuelve en el endpoint de [búsqueda de usuarios](/es/x-api/users/lookup/introduction). Al solicitar campos de usuario adicionales en este endpoint, simplemente usa el parámetro `user.fields`.

El objeto de usuario también se puede encontrar como un objeto hijo y expandirse dentro del objeto Publicación. El objeto está disponible para expansión con `?expansions=author_id` o `?expansions=in_reply_to_user_id` para obtener el objeto condensado solo con los campos predeterminados. Usa la expansión junto con el parámetro `user.fields` cuando solicites campos adicionales para completar el objeto.

<div id="space">
  ### Space
</div>

Los Spaces permiten la expresión y la interacción a través de conversaciones de audio en vivo. El diccionario de datos del Space contiene metadatos relevantes sobre un Space; todos los detalles se actualizan en tiempo real.

Los objetos de usuario se pueden encontrar y expandir en el recurso de usuario. Estos objetos están disponibles para su expansión añadiendo al menos uno de `host_ids`, `creator_id`, `speaker_ids`, `mentioned_user_ids` al parámetro de consulta `expansions`.

A diferencia de las Publicaciones, los Spaces son efímeros y dejan de estar disponibles después de que terminan o cuando su creador los cancela. Cuando tu aplicación gestione datos de Spaces, eres responsable de devolver la información más actualizada y debes eliminar los datos que ya no estén disponibles en la plataforma. Los [endpoints de consulta de Spaces](/es/x-api/spaces/lookup/introduction) pueden ayudarte a asegurarte de que respetas las expectativas e intenciones de los usuarios.

<div id="direct-message-events">
  ### Eventos de Mensaje Directo
</div>

Las conversaciones de Mensajes Directos (DM) se componen de eventos. La X API v2 actualmente admite tres tipos de eventos: MessageCreate, ParticipantsJoin y ParticipantsLeave.

Los objetos de eventos de DM se devuelven mediante los endpoints de [consulta de Mensajes Directos](/es/x-api/direct-messages/lookup/introduction), y se crea un evento MessageCreate cuando los Mensajes Directos se crean correctamente con los endpoints de [Administrar Mensajes Directos](/es/x-api/direct-messages/lookup/introduction).

Al solicitar eventos de DM, se incluyen tres atributos de objeto de evento predeterminados, o campos: id, event&#95;type y text. Para recibir campos de eventos adicionales, usa el parámetro [fields](/es/x-api/fundamentals/fields) dm&#95;event.fields para seleccionar otros. Otros campos de eventos disponibles incluyen lo siguiente: dm&#95;conversation&#95;id, created&#95;at, sender&#95;id, attachments, participant&#95;ids y referenced&#95;tweets.

Varios de estos campos proporcionan los identificadores de otros objetos de X relacionados con el evento de Mensaje Directo:

* sender&#95;id - El identificador de la cuenta que envió el mensaje o que invitó a un participante a una conversación de grupo
* participant&#95;ids - Una matriz de identificadores de cuentas. Para los eventos ParticipantsJoin y ParticipantsLeave, esta matriz contendrá un único identificador de la cuenta que creó el evento
* attachments - Proporciona identificadores de contenido multimedia para contenido que el remitente ha subido a X
* referenced&#95;tweets - Si se encuentra una URL de una Publicación en el campo text, el identificador de esa Publicación se incluye en la respuesta

Las expansiones sender&#95;id, participant&#95;ids, referenced&#95;tweets.id y attachments.media&#95;keys están disponibles para ampliar estos identificadores de objetos de X. Consulta también las [expansiones](/es/x-api/fundamentals/expansions) para más detalles.

<div id="how-to-use-fields-and-expansions">
  ## Cómo usar campos y expansiones
</div>

De forma predeterminada, los objetos de datos de la X API v2 incluyen una pequeña cantidad de campos predeterminados cuando haces una solicitud sin usar los parámetros [fields](/es/x-api/fundamentals/fields) o [expansions](/es/x-api/fundamentals/expansions). Esta guía te mostrará cómo usar los parámetros de consulta `fields` y `expansions` en tu solicitud para recibir objetos y campos adicionales en tu respuesta.

En esta guía, solicitaremos varios campos en la siguiente captura de pantalla de una Publicación.
 

![Esta imagen incluye una captura de pantalla de una Publicación publicada por @X. Puedes ver el texto de la Publicación, el nombre de usuario, la fecha y hora de publicación, la fuente y las métricas públicas. También incluye un video. ](https://cdn.cms-twdigitalassets.com/content/dam/developer-twitter/ScreenShotDataDictionaryGuide.png.twimg.1920.png)

Como puedes ver en la captura de pantalla, hay varios elementos de información visibles relacionados con la Publicación, incluida la autora o el autor de la Publicación, las métricas de la Publicación, la marca de tiempo de creación, el video y la cantidad de vistas del video. También hay varios elementos de datos que no son visibles en la captura de pantalla, pero que aún se pueden solicitar. 

Al hacer una solicitud a la API, la respuesta predeterminada es sencilla y contiene solo los campos predeterminados de la Publicación (`id` y `text`). También recibirás únicamente el objeto primario que se devuelve con el endpoint específico que estés utilizando, y no recibirás ninguno de los objetos de datos asociados que puedan estar relacionados con el objeto primario.

Esta simplicidad, junto con los parámetros `fields` y `expansions`, te permite solicitar únicamente aquellos campos que necesitas, según tu caso de uso.

Solicitud:

```bash
curl --request GET --url 'https://api.x.com/2/tweets?ids=1260294888811347969' \
  --header 'Authorization: Bearer $BEARER_TOKEN'
```

Respuesta:

   En este caso, solicitaremos los siguientes campos adicionales:
   attachments, author&#95;id, created&#95;at, public&#95;metrics.

2. Crea el parámetro de consulta `tweet.fields` con los campos anteriores como valor, usando una lista separada por comas:
   `?tweet.fields=attachments,author_id,created_at,public_metrics`

3. Añade el parámetro de consulta a la solicitud GET /tweets que realizaste anteriormente.

Solicitud:

`curl --request GET --url 'https://api.x.com/2/tweets?ids=1260294888811347969&tweet.fields=attachments,author_id,created_at,public_metrics' \   --header 'Authorization: Bearer $BEARER_TOKEN'`
