La X API proporciona los siguientes códigos de respuesta y error para ayudar a comprender y depurar en tiempo real. Usa la guía de depuración y el índice de errores que aparecen a continuación para obtener contexto adicional.

Las respuestas satisfactorias se indican con un código HTTP de la serie 200 y un cuerpo de respuesta en formato JSON que contiene los objetos solicitados, creados, modificados o eliminados, junto con una expresión de la interpretación que hace el servidor de tu solicitud.

Las respuestas de error se devuelven con un código HTTP que no pertenece a la serie 200. Los distintos códigos de error indican diferentes motivos de un error. La X API intenta devolver [códigos de estado HTTP](https://en.wikipedia.org/wiki/List_of_HTTP_status_codes) apropiados para cada solicitud.

Cuando se produce un error durante una solicitud, se devuelve información detallada sobre el error en el cuerpo de la respuesta para ayudar a diagnosticar el problema. Un campo type, que es un URI, indica la naturaleza del problema, mientras que los campos adicionales proporcionan detalles sobre el mismo. Los campos type, title y detail siempre se devolverán en estos cuerpos (consulta la tabla a continuación). Cualquier campo adicional, como en el ejemplo siguiente, variará según el tipo de error.

```json
{
  "client_id": "101010101",
  "required_enrollment": "Standard Basic",
  "registration_url": "https://developer.twitter.com/en/account",
  "title": "Cliente prohibido",
  "detail": "Esta solicitud debe realizarse con una cuenta de desarrollador aprobada que esté registrada en el endpoint solicitado. Obtenga más información en nuestra documentación.",
  "reason": "client-not-enrolled",
  "type": "https://api.x.com/2/problems/client-forbidden"
}
```

<div id="partial-errors">
  ## Errores parciales
</div>

En algunos casos es posible que veas los errores detallados arriba en una respuesta que devolvió un código de estado 200. En esos casos, el endpoint está diseñado para devolver los datos que pueda y, al mismo tiempo, proporcionar errores detallados sobre lo que no pudo devolver.

Por ejemplo, el [endpoint de consulta de Publicaciones](/es/x-api/posts/lookup/introduction) permite que una [aplicación de desarrollador](/es/resources/fundamentals/developer-apps#overview) de X solicite más de un id. Si algunas de esas Publicaciones están disponibles, pero una de ellas ha sido eliminada, las Publicaciones disponibles se devolverían en el campo data de la respuesta. Se devolvería un campo errors adicional en el payload, indicando qué Publicaciones solicitadas no se pudieron devolver. Se utiliza el mismo formato que para los errores de solicitud completa para facilitar el diagnóstico de problemas.

<div id="error-information">
  ### Información sobre errores
</div>

Cada tipo de problema indica la naturaleza del problema encontrado. También puedes encontrar una lista completa de posibles problemas en nuestra [especificación de la API](https://api.x.com/2/openapi.json).

La X API intenta devolver [códigos de estado HTTP](https://en.wikipedia.org/wiki/List_of_HTTP_status_codes) adecuados para cada solicitud.

| Título | Tipo | Descripción |
| :--- | :--- | :--- |
| Problema genérico | `about:blank` | Un problema genérico sin información adicional más allá de la proporcionada por el código de estado HTTP. |
| Problema de solicitud no válida | `https://api.X.com/2/problems/invalid-request` | Un problema que indica que esta solicitud no es válida. Si tu solicitud incluye un cuerpo POST, asegúrate de que el contenido sea JSON válido y coincida con la [especificación OpenAPI](https://api.x.com/2/openapi.json). |
| Problema de recurso no encontrado | `https://api.X.com/2/problems/resource-not-found` | Un problema que indica que una Publicación, Usuario, etc. determinado no existe. |
| Problema de recurso no autorizado | `https://api.X.com/2/problems/not-authorized-for-resource` | Un problema que indica que no tienes permiso para ver una Publicación, Usuario, etc. en particular. |
| Problema de cliente prohibido | `https://api.X.com/2/problems/client-forbidden` | Un problema que indica que tu cliente tiene prohibido realizar esta solicitud. |
| Problema de recurso no permitido | `https://api.X.com/2/problems/disallowed-resource` | Un problema que indica que el recurso solicitado infringe las normas de esta API. |
| Problema de autenticación no admitida | `https://api.X.com/2/problems/unsupported-authentication` | Un problema que indica que la autenticación utilizada no es compatible. |
| Problema de límite de uso alcanzado | `https://api.X.com/2/problems/usage-capped` | Un problema que indica que se ha superado un límite de uso. |
| Problema de excepción de conexión | `https://api.X.com/2/problems/streaming-connection` | Un problema que indica que hay algo mal con la conexión. |
| Problema de cliente desconectado | `https://api.X.com/2/problems/client-disconnected` | Tu cliente se ha desconectado. |
| Problema de desconexión operativa | `https://api.X.com/2/problems/operational-disconnect` | Has sido desconectado por motivos operativos. |
| Problema de límite de reglas | `https://api.X.com/2/problems/rule-cap` | Has superado el número máximo de reglas. |
| Problema de longitud de regla | `https://api.X.com/2/problems/rule-length` | Has superado el número máximo de caracteres permitido en tu consulta o regla según tu nivel de acceso. Consulta los [niveles de acceso](/es/x-api/getting-started/about-x-api#x-api-access-levels-and-versions). |
| Problema de reglas no válidas | `https://api.X.com/2/problems/invalid-rules` | La regla que has enviado no es válida. |
| Problema de reglas duplicadas | `https://api.X.com/2/problems/duplicate-rules` | La regla que has enviado es un duplicado. |

<div id="troubleshooting-tips">
  ## Consejos para la solución de problemas
</div>

<Accordion title="Depurar tu código">
  Al crear una aplicación, es normal encontrarse con errores o problemas inesperados de vez en cuando. A continuación se incluyen algunos consejos sobre cómo puedes depurar tu código:

  Empieza por dividir el problema en partes más pequeñas para identificar dónde está el fallo. Por ejemplo, si estás integrando un endpoint REST o de streaming, el problema podría deberse a cualquiera de los siguientes elementos:

  * El endpoint requiere una [autenticación](/es/resources/fundamentals/authentication/overview) adecuada.
  * El endpoint requiere parámetros y encabezados válidos. Cualquier regla de filtrado debe construirse usando los operadores correctos y la [sintaxis](/es/x-api/enterprise-gnip-2.0/fundamentals/rules-filtering#building-rules-and-queries) adecuada.
  * El URI del endpoint debe ser correcto y, en el caso de endpoints REST, se debe usar el método HTTP correcto.
  * Los datos o el recurso a los que intentas acceder no están disponibles para ti (por ejemplo, los datos privados solo están disponibles para usuarios autenticados).
  * Tu paquete de datos actual te da acceso solo a determinados endpoints y a límites de frecuencia específicos. Consulta tu [portal para desarrolladores](https://developer.x.com/en/portal/products) para obtener más detalles.
  * Tu código utiliza una biblioteca de terceros para integrar el endpoint en tu código.
  * Tu código necesita analizar correctamente la respuesta del endpoint.

  Lee el mensaje de error adjunto. Esto debería darte una buena indicación de cuál es el problema. Utiliza las tablas de la [sección de códigos de error y respuesta](#x-api-http-status-codes) para obtener consejos de solución de problemas específicos para cada código de error.

  Para endpoints REST, puedes usar un cliente REST como [Postman](https://www.postman.com/) o [Insomnia](https://insomnia.rest/) para validar los pasos del (1) al (5) anteriores (revisa nuestra guía [&quot;Introducción a Postman&quot;](/es/tutorials/postman-getting-started)). Si la solicitud con el cliente REST devuelve un código de estado de éxito 200, puedes asumir que el problema está en tu código o en la biblioteca que estás usando, y no en la solicitud al endpoint en sí.

  Para endpoints de streaming, puedes usar cURL (una herramienta de línea de comandos para enviar o recibir solicitudes utilizando la sintaxis de URL) para comprobar si el problema está en la solicitud al endpoint (pasos (1) a (5) anteriores) o en tu propio código (pasos (6) a (7) anteriores).
</Accordion>

<Accordion title="Problemas de autenticación">
  * Verifica que estás utilizando el [método de autenticación](/es/resources/fundamentals/authentication/overview) adecuado que requiere el endpoint. Esto se puede identificar en la página de referencia de la API del endpoint.
  * Verifica que tus credenciales de autenticación sean correctas. Puedes comprobar o regenerar las claves y tokens de tu App en la [sección Apps](https://developer.x.com/en/portal/projects-and-apps) del panel del portal para desarrolladores (en “Details”).
  * Verifica que hayas [autorizado correctamente tu solicitud OAuth 1.0a](/es/resources/fundamentals/authentication/oauth-1-0a/authorizing-a-request) con `oauth_nonce`, `oauth_signature` y `oauth_timestamp` para tu solicitud.
  * Si el problema persiste, considera utilizar una biblioteca de OAuth, un cliente REST como [Postman](https://www.postman.com/) o [Insomnia](https://insomnia.rest/), o [xurl](https://github.com/xdevplatform/xurl).

  Consulta nuestra [guía sobre autenticación](/es/resources/fundamentals/authentication/overview) para obtener información adicional sobre todos estos puntos.
</Accordion>

<Accordion title="Problemas de límites de frecuencia y límite de Publicaciones">
  Un error 429 puede producirse si alcanzas un límite de frecuencia para un endpoint determinado o si alcanzas un límite de Publicaciones.

  Si se devuelve el error específico &quot;Too many requests&quot;, alcanzaste el límite de frecuencia del endpoint. En otras palabras, has superado la cantidad máxima de solicitudes permitidas para un endpoint en un período de tiempo determinado.

  Ten en cuenta que los límites de frecuencia se establecen a nivel de Aplicación y a nivel de usuario.

  * El **nivel de X App** indica la cantidad de solicitudes permitidas cuando utilizas OAuth 2.0 solo para la aplicación, donde los límites de frecuencia se determinan globalmente para toda la Aplicación. Por ejemplo, si un método permite 15 solicitudes por ventana de límite de frecuencia, entonces te permite hacer 15 solicitudes por ventana en nombre de tu X App. Este límite se considera completamente separado del límite a nivel de usuario. Lee más en nuestra guía sobre [OAuth 2.0 solo para la aplicación](/es/resources/fundamentals/authentication/oauth-2-0/application-only).
  * El **nivel de contexto de usuario** indica la cantidad de solicitudes que se pueden hacer por Token de acceso de usuario cuando se utiliza Contexto de usuario de OAuth 1.0a u OAuth 2.0 Authorization Code with PKCE. Por ejemplo, si un método permite 15 solicitudes por ventana de límite de frecuencia, entonces permite 15 solicitudes por ventana y por Token de acceso de usuario. Lee más en nuestra guía sobre cómo obtener los Tokens de acceso de un usuario con [OAuth 1.0a](/es/resources/fundamentals/authentication/oauth-1-0a/obtaining-user-access-tokens) y [OAuth 2.0](/es/resources/fundamentals/authentication/oauth-2-0/user-access-token).

  Comienza comprobando los límites de frecuencia para el endpoint que estás utilizando. Puedes encontrar esta información en la página de referencia de la API del endpoint y en el nuevo panel del portal para desarrolladores.

  Revisa nuestra documentación para obtener información adicional sobre límites de frecuencia, incluido cómo usar encabezados HTTP para seguir el estado de tu Aplicación respecto a un límite de frecuencia determinado, cómo recuperarte de un código de error 429 por límite de frecuencia y consejos para evitar que se apliquen límites de frecuencia desde el principio:

  * [Límites de frecuencia para endpoints v2](/es/x-api/fundamentals/rate-limits)

  Si has recibido el error específico &quot;Usage cap exceeded: Monthly product cap&quot;, significa que has alcanzado el límite de Publicaciones para tu nivel de acceso. Encontrarás muchos detalles sobre cuáles son estos [límites de Publicaciones](/es/x-api/fundamentals/post-cap) en nuestra página de documentación.
</Accordion>

<Accordion title="Problemas de Publicaciones faltantes">
  Sigue los pasos a continuación si esperabas que se devolviera una Publicación, pero el endpoint no la entregó.

  * Revisa tu regla para asegurarte de que estás utilizando los [operadores](/es/x-api/enterprise-gnip-2.0/fundamentals/rules-filtering#enterprise-operators) y la [sintaxis](/es/x-api/enterprise-gnip-2.0/fundamentals/rules-filtering#building-rules-and-queries) correctos. Divide la regla en cláusulas más pequeñas para asegurarte de que estás utilizando la sintaxis correcta.
  * Si la cuenta desde la que se envió la Publicación estaba [protegida](https://help.twitter.com/en/safety-and-security/public-and-protected-tweets) en el momento en que se creó la Publicación, la Publicación no se devolverá, incluso si la cuenta es pública en el momento de la solicitud al endpoint. Normalmente puedes comprobar esto usando la [Búsqueda avanzada de X](https://x.com/search-advanced): si una Publicación no aparece usando la funcionalidad de Búsqueda avanzada de X, debes asumir que no será devuelta por el endpoint.

  Los siguientes pasos se aplican solo a endpoints de streaming:

  * ¿Estabas conectado al stream cuando se envió la Publicación? Recuerda que la marca de tiempo entregada en el objeto de la Publicación indica la hora en UTC. Si se produjo una desconexión cuando se envió la Publicación, revisa las [funciones de recuperación y redundancia](/es/x-api/posts/filtered-stream/integrate/recovery-and-redundancy-features#recovery-and-redundancy) disponibles para rellenar cualquier dato que se haya perdido.
  * ¿Tu regla estaba activa cuando se envió la Publicación? Recuerda que la marca de tiempo entregada en el objeto de la Publicación indica la hora en UTC.
</Accordion>

<Accordion title="Problemas de desconexión por búfer lleno">
  Es posible que recibas uno de los siguientes errores cuando tu stream no puede seguir el ritmo al que entregamos los datos y tu Aplicación no está consumiendo los datos del stream con la suficiente rapidez:

  ```
  This stream has been disconnected because your client was unable to keep up with us.
  ```

  ```
  This stream has been disconnected for operational reasons.
  ```

  Permitimos que la entrega se retrase durante un período de tiempo y tenemos un búfer temporal de preparación para cada stream de nuestro lado; pero si no te pones al día, iniciamos una desconexión para que puedas volver a conectarte en el punto temporal actual. Ten en cuenta que esto puede provocar pérdida de datos (para los datos que estén en el búfer en el momento de la desconexión por búfer lleno).

  Esto puede ocurrir en torno a grandes picos en el volumen de datos. En general, recomendamos usar un proceso de almacenamiento en búfer para consumir datos rápidamente que esté separado del proceso de procesamiento.

  Si eres un cliente Enterprise que utiliza endpoints v1.1, puedes obtener más información sobre cómo optimizar tu Aplicación para evitar desconexiones como esta en nuestros artículos sobre [conexión](/es/x-api/enterprise-gnip-2.0/powertrack-api#disconnections-explained) y sobre cómo consumir datos de streaming [aquí](https://developer.x.com/en/docs/tutorials/consuming-streaming-data) y [aquí](/es/x-api/enterprise-gnip-2.0/powertrack-api#planning-for-high-volume-social-data-events).

  Hay una variedad de herramientas disponibles para recuperar Publicaciones perdidas debido a una desconexión, incluidas las que se enumeran a continuación. Ten en cuenta que las siguientes herramientas solo están disponibles con endpoints v1.1 en el nivel de acceso Enterprise.

  * [**Redundant Connections**](/es/x-api/enterprise-gnip-2.0/powertrack-api#recovery-and-redundancy-features): con múltiples conexiones, consume el stream desde varios servidores para evitar pérdida de datos cuando una se desconecta.
  * [**Replay**](/es/x-api/enterprise-gnip-2.0/powertrack-api#recovery-and-redundancy-features): recupera datos de los últimos 5 días mediante un stream separado.
  * [**Backfill**](/es/x-api/enterprise-gnip-2.0/powertrack-api#recovery-and-redundancy-features): vuelve a conectarte en un plazo de 5 minutos y comienza desde donde lo dejaste.
  * **Historical PowerTrack**: recupera datos de todo el archivo de X.
</Accordion>

<div id="when-you-post-a-question-make-sure-to-include-the-following-information">
  ### Cuando publiques una pregunta, asegúrate de incluir la siguiente información
</div>

* Una descripción del problema
* La llamada a la API que estás realizando (incluye los encabezados, si es posible)
* La respuesta de X devuelta (incluye cualquier mensaje de error)
* Qué esperabas recibir en su lugar
* Lista de pasos seguidos para solucionar el problema
* Lista de pasos necesarios para reproducir el problema
* Si corresponde, el período de tiempo durante el cual se produjo el problema
* Si corresponde, el ID de la Aplicación, el ID de la Publicación, etc.
* Cualquier ejemplo de código o captura de pantalla relevante

Incluye únicamente un tema/pregunta por publicación.

Si tienes solicitudes de funcionalidades nuevas o comentarios, envíalos a través del [Formulario de comentarios de la plataforma para desarrolladores de X](https://t.co/devfeedback).

Para problemas relacionados con las políticas, como la suspensión de una aplicación, ponte en contacto con el [soporte de políticas](https://help.twitter.com/forms/platform).

Para problemas relacionados con X, como inicio de sesión y soporte de cuenta, utiliza el [Centro de ayuda de X](https://help.twitter.com/en/contact-us).
