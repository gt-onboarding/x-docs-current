<p>Junto con X API v2, lanzamos una nueva estrategia de control de versiones que permite a los desarrolladores comprender mejor cuándo deben esperar cambios en las API públicas de X y cuándo necesitarán migrar a nuevas versiones. </p>
<p>Se notificará a los desarrolladores sobre obsolescencias, retiradas, cambios e incorporaciones a la X API a través de nuestros <a href="https://developer.x.com/en/updates/stay-informed">canales de comunicación</a>, de modo que puedan planificar adecuadamente la incorporación de estos cambios en su hoja de ruta de desarrollo. Todos los cambios en la API se documentarán en el registro de cambios.</p>
<p>Actualmente, la X API tiene tres versiones diferentes. Recomendamos encarecidamente utilizar X API v2 al planificar tu integración, salvo que la funcionalidad que requiere tu caso de uso aún no esté disponible en v2. </p>
<p>Para obtener más información sobre cada versión, visita las siguientes páginas:</p>

  <li>Gnip 2.0 (<a href="/es/x-api/enterprise-gnip-2.0/enterprise-gnip">Enterprise</a>)</li>
</ul>

<div id="versioning-strategy">
  ## Estrategia de versionado
</div>

<p>El versionado de la X API se representará mediante números de versión declarados en la ruta de acceso de nuestros endpoints:</p>
<p>`https://api.x.com/2/tweets`</p>
<p>Nuestro objetivo es lanzar versiones mayores de la X API cuando sea necesario, pero no con una frecuencia superior a una vez cada 12 meses. Se lanzará una versión mayor cuando se introduzcan cambios incompatibles en la API. Publicaremos guías de migración al lanzar una nueva versión mayor para ayudar a los desarrolladores a migrar a la nueva versión. </p>
<p>Un cambio incompatible requiere que los desarrolladores modifiquen su código para mantener la funcionalidad existente en su aplicación. Los cambios compatibles serán aditivos y se implementarán en la versión más reciente cuando estén listos, sin requerir trabajo por parte del desarrollador a menos que este desee aprovechar la nueva funcionalidad.</p>
<p>Si es necesario implementar un cambio incompatible a mitad de ciclo (por razones de seguridad o privacidad), este cambio se realizará en la versión más reciente.</p>

<div id="breaking-changes">
  ### Cambios incompatibles
</div>

<p>Estos cambios requieren que los desarrolladores modifiquen su código para mantener la funcionalidad existente de su aplicación.</p>

<ul>
  <li>Adición de un nuevo parámetro obligatorio</li>
  <li>Eliminación de un endpoint existente</li>
  <li>Eliminación de cualquier campo en la respuesta (ya sea obligatorio u opcional)</li>
  <li>Eliminación de un parámetro de consulta</li>
  <li>Reestructuración del formato de entrada o salida (por ejemplo, convertir un campo de nivel superior en un subcampo o cambiar la ubicación de los errores para que aparezcan en línea)</li>
  <li>Cambio del nombre o del tipo de datos de un parámetro de entrada existente o de un valor de salida</li>
  <li>Cambio del nombre de un campo</li>
  <li>Cambio del nombre de un recurso</li>
  <li>Cambio de un código de respuesta</li>
  <li>Cambio de tipos de error</li>
  <li>Cambios en los ámbitos de autorización existentes</li>
</ul>

<div id="non-breaking-changes">
  ### Cambios no incompatibles
</div>

<ul>
  <li>Incorporación de un nuevo endpoint</li>
  <li>Incorporación de un nuevo parámetro opcional</li>
  <li>Incorporación de un nuevo campo de respuesta</li>
  <li>Cambio de texto en mensajes de error</li>
  <li>Disponibilidad de nuevos scopes</li>
  <li>“Anulación” de campos (poner el valor de un campo en null por motivos de privacidad/seguridad como alternativa a eliminarlo por completo)</li>
</ul>

<div id="deprecation-and-retirement">
  ### Obsolescencia y retirada
</div>

<p>En primer lugar, esta es nuestra definición de lo que significan obsolescencia y retirada para la X API:</p>

<ul>
  <li><b>Obsolescencia:</b> La funcionalidad deja de estar soportada por el equipo. No se lanzará nueva funcionalidad relacionada con esa característica y, si hay errores o problemas con el producto, las probabilidades de que trabajemos en una solución son extremadamente bajas. </li>
  <li><b>Retirada:</b> La funcionalidad dejará de estar disponible.</li>
</ul>

<p>En la mayoría de los casos, en cuanto se publica una nueva versión, la versión anterior se marcará como obsoleta. Las versiones permanecerán en estado de obsolescencia durante un período de tiempo, después del cual se retirarán. </p>
<p>Por favor, <a href="https://developer.x.com/en/updates/stay-informed">mantente al día</a> para obtener más información sobre futuras obsolescencias y retiradas.</p>
