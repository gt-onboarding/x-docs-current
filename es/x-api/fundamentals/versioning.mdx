<p>Junto con X API v2, lanzamos una nueva estrategia de control de versiones que permite a los desarrolladores comprender mejor cuándo deben esperar cambios en las APIs públicas de X y cuándo necesitarán migrar a nuevas versiones. </p>
<p>Los desarrolladores serán notificados de las deprecaciones, retiradas, cambios y adiciones a X API a través de nuestros <a href="https://developer.x.com/en/updates/stay-informed">canales de comunicación</a>, de modo que puedan planificar adecuadamente cómo incorporar estos cambios en su hoja de ruta de desarrollo. Todos los cambios en la API se registrarán en el registro de cambios (changelog).</p>
<p>Actualmente, X API tiene tres versiones diferentes. Recomendamos encarecidamente que los usuarios utilicen X API v2 al planificar su integración, a menos que aún no hayamos lanzado en v2 la funcionalidad que requiere su caso de uso. </p>
<p>Para obtener más información sobre cada versión, visite las siguientes páginas:</p>

  <li>Gnip 2.0 (<a href="/es/x-api/enterprise-gnip-2.0/enterprise-gnip">Enterprise</a>)</li>

<div id="versioning-strategy">
  ## Estrategia de versionado
</div>

<p>El versionado de la X API se representará mediante números de versión declarados en la ruta de acceso de los endpoints:</p>
<p>`https://api.x.com/2/tweets`</p>
<p>Nuestro objetivo es lanzar versiones principales de la X API cuando sea necesario, pero como máximo una vez cada 12 meses. Se publicará una versión principal cuando se introduzcan cambios incompatibles en la API. Publicaremos guías de migración al lanzar una nueva versión principal para ayudar a los desarrolladores a migrar a la nueva versión. </p>
<p>Un cambio incompatible requiere que los desarrolladores modifiquen su código para mantener la funcionalidad existente en su aplicación. Los cambios compatibles serán aditivos y se implementarán en la versión más reciente cuando estén listos, sin requerir trabajo adicional por parte de los desarrolladores, a menos que quieran aprovechar la nueva funcionalidad.</p>
<p>Si es necesario implementar un cambio incompatible a mitad de ciclo (por motivos de seguridad o privacidad), este cambio se realizará en la versión más reciente.</p>

<div id="breaking-changes">
  ### Cambios incompatibles
</div>

<p>Estos cambios requieren que los desarrolladores modifiquen su código para mantener la funcionalidad existente de su aplicación.</p>

<ul>
  <li>Incorporación de un nuevo parámetro obligatorio</li>
  <li>Eliminación de un endpoint existente</li>
  <li>Eliminación de cualquier campo en la respuesta (ya sea obligatorio u opcional)</li>
  <li>Eliminación de un parámetro de consulta</li>
  <li>Reestructuración del formato de entrada o salida (por ejemplo, convertir un campo de nivel superior en un subcampo o cambiar la ubicación de los errores para que aparezcan en línea)</li>
  <li>Cambio del nombre o del tipo de datos de un parámetro de entrada existente o de un valor de salida</li>
  <li>Cambio del nombre de un campo</li>
  <li>Cambio del nombre de un recurso</li>
  <li>Cambio de un código de respuesta</li>
  <li>Cambio de tipos de error</li>
  <li>Cambios en los ámbitos de autorización existentes</li>
</ul>

<div id="non-breaking-changes">
  ### Cambios retrocompatibles
</div>

<ul>
  <li>Adición de un nuevo endpoint</li>
  <li>Adición de un nuevo parámetro opcional</li>
  <li>Adición de un nuevo campo de respuesta</li>
  <li>Cambio de texto en mensajes de error</li>
  <li>Disponibilidad de nuevos scopes</li>
  <li>Establecer campos en null (cambiar el valor de un campo a null por motivos de privacidad o seguridad como alternativa a eliminarlo por completo)</li>
</ul>

<div id="deprecation-and-retirement">
  ### Obsolescencia y retirada
</div>

<p>En primer lugar, esta es nuestra definición de lo que significan la obsolescencia y la retirada en el contexto de la X API:</p>

<ul>
  <li><b>Obsolescencia:</b> La funcionalidad deja de contar con soporte por parte del equipo. No se incorporará ninguna funcionalidad nueva relacionada con esa característica y, si hay errores o problemas con el producto, las probabilidades de que implementemos una corrección son extremadamente bajas. </li>
  <li><b>Retirada:</b> La funcionalidad dejará de estar disponible.</li>
</ul>

<p>En la mayoría de los casos, tan pronto como se publique una nueva versión, la versión anterior se marcará como obsoleta. Las versiones permanecerán en estado obsoleto durante un período de tiempo, tras el cual se retirarán. </p>
<p><a href="https://developer.x.com/en/updates/stay-informed">Mantente informado</a> para obtener más información sobre futuras obsolescencias y retiradas.</p>
