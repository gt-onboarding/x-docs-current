  ### Cambio en el comportamiento de los eventos de MD

  Hoy realizamos un cambio en el comportamiento de los eventos de MD de X API v2. Con este cambio, el array participant&#95;ids solo se incluirá para los eventos ParticipantsJoin y ParticipantsLeave, e incluirá únicamente a las personas que se unieron o salieron en el momento del evento. El evento MessageCreate ya no incluirá el array participant&#95;ids. Más información [aquí](https://devcommunity.x.com/t/upcoming-change-to-x-api-dm-events-endpoints-behavior/241841).
</Update>

<Update label="3 ago 2022" description="X API v2">
  ### Adición de alias de operadores de filtrado

  Se añadieron operadores de filtrado/consulta &quot;aliases&quot; a los endpoints de búsqueda y filtered stream, en Enterprise, premium y v2. En muchos casos, el operador se refiere a Publicaciones en lugar de statuses, por ejemplo, &quot;in&#95;reply&#95;to&#95;tweet&#95;id&quot; como alias de &quot;in&#95;reply&#95;to&#95;status&#95;id&quot;. En otros casos, el nuevo alias de operador aporta más claridad al filtrado realizado. Por ejemplo, dado que las Publicaciones solo pueden tener un adjunto de video, se proporciona un nuevo alias &quot;has:video&#95;link&quot; para &quot;has:videos&quot;. Consulta la [tabla de operadores del flujo filtrado de v2](https://developer.x.com/en/docs/twitter-api/tweets/filtered-stream/integrate/build-a-rule#list) para ver más ejemplos.
</Update>

<Update label="Dec 14, 2021" description="X API v2: Autenticación">
  ### Compatibilidad con la autenticación OAuth 2.0

  A partir de hoy, todos los desarrolladores pueden autenticarse con OAuth 2.0 seleccionando OAuth 2.0 como método de autenticación en el portal para desarrolladores. Hemos añadido compatibilidad para clientes confidenciales y públicos y para que todos los endpoints relevantes de v2 utilicen este método de autenticación como parte de este lanzamiento.

  OAuth 2.0 es un protocolo de autorización estándar de la industria que proporciona a los desarrolladores mayor control sobre los alcances de una aplicación y mejora los flujos de autorización en varios dispositivos. En otras palabras, los desarrolladores que crean aplicaciones para las personas en X tendrán más control sobre la información que su aplicación solicita a sus usuarios, de modo que solo tengan que pedir a sus usuarios finales los datos e información que necesitan.

  Este protocolo de autorización moderno te permitirá presentar a tus usuarios finales un flujo de consentimiento más eficiente para autorizar tu aplicación, que solo muestre los alcances específicos que les hayas solicitado.

  Para obtener más información sobre este lanzamiento, visita nuestro anuncio en el foro.
</Update>

<Update label="Nov 15, 2021" description="X API v2: Acceso anticipado">
  ### Actualizaciones importantes de la plataforma a X API v2

  **Actualizaciones importantes de la plataforma: X API v2 es ahora la versión principal, nuevos niveles de acceso Essential y Elevated, y cambios en las políticas**

  Hoy anunciamos que X API v2 es ahora la versión principal de X API. Hemos lanzado suficientes endpoints y funcionalidades en X API v2 para satisfacer las necesidades del 90 % de todas las aplicaciones existentes creadas sobre X API.

  Hoy lanzamos los siguientes cambios para mejorar aún más la experiencia de los desarrolladores con X API v2:

  * Acceso Essential rápido y gratuito a la API, y acceso Elevated gratuito para los desarrolladores cuyas propuestas de uso hayan sido aprobadas. Tenemos muchos más detalles sobre estos niveles de acceso en nuestra página Acerca de X API.

  * Estamos eliminando redacción en nuestra Política para Desarrolladores que restringía cómo desarrollar con las funciones principales de X y limitaba la cantidad de usuarios que puedes admitir a través de tu aplicación.

    * Estamos abriendo nuestra plataforma para animarte a crear herramientas y productos que hagan que X sea mejor, más saludable y que amplíen la conversación pública. [Consulta una lista de soluciones que nos encantaría que crearas](https://developer.x.com/en/docs/twitter-api/what-to-build).

    * En concreto, hemos eliminado términos que restringían la replicación de la experiencia de X, incluidas las funciones principales de X, así como términos que requerían permiso para tener un alto número de tokens de usuario.

    * Sabemos que crear soluciones que ayuden a las personas en X a menudo significa que un desarrollador tenga que crear (o replicar) algunas de las cosas que están disponibles en X. Estos cambios en nuestra Política para Desarrolladores están pensados para aportar claridad al ecosistema de desarrolladores y proporcionar una plataforma de API abierta que facilite a los desarrolladores crear, innovar y tener un impacto en la conversación pública.

  Entramos en muchos más detalles sobre este lanzamiento y los lanzamientos de X API que nos han llevado a este momento en nuestra [publicación en el foro](https://devcommunity.x.com/t/ushering-in-a-new-era-for-the-twitter-developer-platform-with-the-twitter-api-v2/162087).

  Además de explorar las entradas anteriores del registro de cambios, hemos reunido algunos recursos clave que puedes revisar para entender todo lo que se ha lanzado en X API v2 hasta la fecha:

  * [Mapa de endpoints de X API](/es/x-api/migrate/x-api-endpoint-map)

  * [Acerca de X API](/es/x-api/getting-started/about-x-api)
</Update>

<Update label="18 de diciembre de 2020" description="Campo de configuración de respuestas de conversación para publicaciones en X API v2">
  ### Adición del campo reply&#95;settings

  Hoy lanzamos un nuevo campo en el objeto Post llamado `reply_settings`. Este campo está disponible en todos los endpoints de X API v2 que devuelven Publicaciones, incluidos post lookup, hide replies, recent search, sampled stream, filtered stream, follows lookup, los endpoints de user post timeline y user mention timeline, o mediante [expansiones](/es/x-api/fundamentals/expansions) del objeto Post en cualquier endpoint.

  Este campo indica cómo el autor de la Publicación ha permitido que otros respondan a sus Publicaciones, ya sea que pueda hacerlo todo el mundo, solo las personas que menciona en la Publicación o las cuentas a las que está siguiendo. Los valores devueltos por el campo `reply_settings` incluyen `everyone`, `mentionedUsers` y `following`. Si no se configuran controles de conversación para la Publicación, este campo seguirá apareciendo en el objeto Post con el valor `everyone`.
</Update>

<Update label="10 de mayo de 2018" description="X API: estándar v1.1">
  ### Cambios en la representación de la carga útil de publicaciones citadas

  Hoy estamos incorporando un cambio en la forma en que se representarán las URL en la carga útil de publicaciones citadas.

  Estamos añadiendo una nueva entidad llamada &quot;quoted&#95;status&#95;permalink&quot; para garantizar que se pueda hacer referencia a la publicación citada y eliminaremos el enlace t.co del campo &quot;text&quot; de la publicación citada.

  Con el nuevo formato, el enlace t.co ya no se añade al campo &quot;text&quot; de la publicación citada en el objeto &quot;quoted&#95;status&quot;.

  Además, ya no incluiremos los campos t.co url, expanded&#95;url y display&#95;url de la publicación citada en la entidad ‘urls’. En su lugar, se añadirán a un nuevo objeto &quot;quoted&#95;status&#95;permalink&quot; con los atributos de URL &quot;url&quot;, &quot;expanded&quot; y &quot;display&quot;. Esto significa que el objeto &quot;quoted&#95;status&#95;permalink&quot; se reservará para el enlace t.co que apunta de vuelta a la publicación citada, mientras que la matriz &quot;entities.urls&quot; se puede usar para identificar cualquier enlace compartido dentro de la publicación original o de la cita.

  Consulta nuestra [publicación en el foro](https://devcommunity.x.com/t/updating-how-urls-are-rendered-in-the-quote-tweet-payload/105473) y la [documentación actualizada](https://devcommunity.x.com/t/upcoming-changes-to-the-developer-platform/104603) para obtener más detalles.
</Update>

  La API de búsqueda de 30 días (tanto 30 días como archivo completo) dejará de admitir el operador url&#95;contains: dentro de 30 días a partir de hoy, o después del 4 de agosto de 2017 (el soporte de url&#95;contains: en realidad terminará con el primer despliegue después de esa fecha). La retirada de este operador se anunció como parte de la migración a Gnip 2.0 en agosto de 2016. Cuando finalice el soporte para el operador, cualquier consulta que use url&#95;contains: será rechazada por no ser válida.
</Update>

<Update label="May 5, 2017" description="X API: Standard v1.1">
  ### Parámetros opcionales para comandos de DM

  El parámetro opcional `enable_dm_commands` de statuses/update permitirá que las aplicaciones eliminen anticipadamente la compatibilidad con los comandos de DM, antes de que termine el período de transición. El valor predeterminado es *true* (es decir, el comportamiento heredado actual), pero se puede establecer en *false* para obtener el nuevo comportamiento posterior al 1 de noviembre. El parámetro opcional `fail_dm_commands` de statuses/update hará que los comandos de DM devuelvan HTTP 403 (código de error 151) desde la API cuando se establezca en *true*. El valor predeterminado es *false*. Lee más sobre esta actualización [en nuestro foro](https://devcommunity.x.com/t/retiring-legacy-dm-commands-on-the-standard-tweet-api/86653).
</Update>

<Update label="Feb 22, 2017" description="X API: Enterprise v1.1">
  ### Nuevos enriquecimientos y cambios en la carga útil

  ¡Nuevo enriquecimiento! Los metadatos de encuestas ahora están disponibles a través de nuestras APIs Enterprise. Consulta la [documentación](/es/x-api/enterprise-gnip-2.0/fundamentals/account-activity#migration-introduction) para más detalles.

  Nuevo campo en la carga útil: el campo Image Alt-Text (decription) puede estar presente en las secciones entities.media[] y extended&#95;entities.media[] de la carga útil si un usuario elige la opción &quot;Add description&quot; al agregar una foto a una publicación. Esto solo está disponible en formato nativo enriquecido.

  Formato de carga útil predeterminado: los flujos nuevos creados en los productos Gnip ahora tendrán como valor predeterminado el formato JSON original.
</Update>
