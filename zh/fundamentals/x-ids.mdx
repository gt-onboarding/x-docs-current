在 X 上的每个对象——帖子、私信（Direct Message）、用户、列表等——都有一个唯一的 ID。

在平台刚上线之初，这些 ID 数字足够小，可以按顺序生成。随着时间推移，为了应对平台的增长，ID 从 32 位扩展到 64 位。如今，X 的 ID 是基于时间而非顺序的唯一 64 位无符号整数。完整的 ID 由时间戳、worker 编号和序列号组成。X 开发了一个名为 &quot;Snowflake&quot; 的内部服务，用于一致地生成这些 ID（[在 X 博客上了解更多](https://blog.x.com/engineering/en_us/a/2010/announcing-snowflake.html)）。

像 64 位这样大的整数，可能会在那些用少于 64 位来表示整数的编程语言中引发问题。以 JavaScript 为例，其中整数大小被限制为 53 位。为了解决这一问题，在最初的 X API（v1/1.1）设计中，ID 值会以两种格式返回：既作为整数，也作为字符串。

```json
{"id": 10765432100123456789, "id_str": "10765432100123456789"}
```

如果你在浏览器的 JavaScript 控制台中运行命令 `(10765432100123456789).toString()`，结果会是 `"10765432100123458000"` —— 由于这种转换，64 位整数会丢失精度（这有时被称为“munging”，即对一段数据进行破坏性的更改）。

在 X API 1.1 及更早版本中，你应始终使用数字的字符串表示形式以避免精度丢失。

在较新版本的 API 中，所有大整数值默认都以字符串形式表示。
