在 X 上的每个对象——帖子、私信、用户、列表等——都有一个唯一的 ID。

在平台最初阶段，这些 ID 所对应的数值足够小，可以按顺序生成。随着时间推移，为了适应增长，ID 从 32 位升级到 64 位。如今，X 的 ID 是基于时间的唯一 64 位无符号整数，而不再是顺序生成。完整的 ID 由时间戳、工作节点号（worker number）和序列号组成。X 开发了一个名为 “Snowflake” 的内部服务，用于持续稳定地生成这些 ID（[可在 X 博客上阅读更多相关内容](https://blog.x.com/engineering/en_us/a/2010/announcing-snowflake.html)）。

像 64 位这样大的整数，可能会在仅用少于 64 位来表示整数的编程语言中引发问题。例如，在 JavaScript 中，整数最多只能使用 53 位来表示。为了解决这一问题，在最初的 X API（v1/1.1）设计中，ID 值会以两种格式返回：既包含整数形式，也包含字符串形式。

```json
{"id": 10765432100123456789, "id_str": "10765432100123456789"}
```

如果你在浏览器的 JavaScript 控制台中运行命令 `(10765432100123456789).toString()`，结果将是 `"10765432100123458000"` —— 由于这种转换，64 位整数会丢失精度（这有时被称为 &quot;munging&quot; —— 对数据的破坏性修改）。

在 X API 的 1.1 及之前版本中，你应始终使用数字的字符串表示形式来避免精度丢失。

在较新版本的 API 中，所有较大的整数值默认都表示为字符串。
