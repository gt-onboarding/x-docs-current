<p>随着 X API v2 的推出，我们引入了一项新的版本管理策略，帮助开发者更好地了解何时可能会对 X 公共 API 进行变更，以及何时需要迁移到新版本。 </p>
<p>我们会通过 <a href="https://developer.x.com/en/updates/stay-informed">沟通渠道</a> 向开发者告知 X API 的弃用、下线、变更和新增内容，以便他们在开发路线图中合理规划并适配这些变更。所有对 API 的更改都会记录在变更日志中。</p>
<p>X API 目前有三个不同的版本。除非你的用例所需的功能尚未在 v2 中推出，否则我们强烈建议你在规划集成时使用 X API v2。 </p>
<p>若想进一步了解各个版本，请访问以下页面：</p>

<ul>
  <li><a href="/zh/x-api/introduction">X API v2</a></li>
  <li>Gnip 2.0（<a href="/zh/x-api/enterprise-gnip-2.0/enterprise-gnip">Enterprise</a>）</li>
</ul>

<div id="versioning-strategy">
  ## 版本策略
</div>

<p>X API 的版本将通过在端点路由路径中声明的版本号来表示：</p>
<p>`https://api.x.com/2/tweets`</p>
<p>我们会在必要时发布 X API 的主版本（major version），但频率不会高于每 12 个月一次。当在 API 中引入破坏性变更（breaking changes）时，将发布新的主版本。我们会在推出新的主版本时发布迁移指南，帮助开发者迁移到新版本。</p>
<p>破坏性变更会要求开发者修改其代码，以维持其应用中现有功能的正常运行。非破坏性变更将是增量式的，并在准备就绪后部署到最新版本，对于开发者而言不需要进行任何额外工作，除非希望利用这些新功能。</p>
<p>如果必须在版本周期中途推出破坏性变更（出于安全或隐私原因），该变更将会应用到最新版本。</p>

<div id="breaking-changes">
  ### 破坏性变更
</div>

<p>这些变更要求开发者修改代码，以维持其应用的现有功能。</p>

<ul>
  <li>添加新的必需参数</li>
  <li>删除现有端点</li>
  <li>删除响应中的任意字段（无论是必需还是可选）</li>
  <li>删除查询参数</li>
  <li>重构输入或输出格式（例如，将顶级字段变为子字段，或者将错误的位置改为内联返回）</li>
  <li>更改现有输入参数或输出值的名称或数据类型</li>
  <li>更改字段名称</li>
  <li>更改资源名称</li>
  <li>更改响应状态码</li>
  <li>更改错误类型</li>
  <li>对现有授权范围的更改</li>
</ul>

<div id="non-breaking-changes">
  ### 非破坏性变更
</div>

<ul>
  <li>添加新的 endpoint</li>
  <li>添加新的可选参数</li>
  <li>添加新的响应字段</li>
  <li>修改错误消息中的文本</li>
  <li>提供新的 scope</li>
  <li>字段“置空”（出于隐私/安全原因将某个值更改为 null，作为完全移除该字段的替代方案）</li>
</ul>

<div id="deprecation-and-retirement">
  ### 弃用和停用
</div>

<p>首先，这是我们针对 X API 中“弃用”和“停用”的定义：</p>

<ul>
  <li><b>弃用（Deprecation）：</b>该功能不再由团队提供支持。不会再针对该功能发布任何新的功能，如果产品存在任何缺陷或问题，我们着手修复的可能性极低。</li>
  <li><b>停用（Retired）：</b>该功能将不再可访问。</li>
</ul>

<p>在大多数情况下，一旦新版本发布，上一版本会立即被标记为弃用。版本会在弃用状态下保留一段时间，之后将被停用。</p>
<p>请<a href="https://developer.x.com/en/updates/stay-informed">保持关注</a>以了解未来的弃用和停用计划。</p>
