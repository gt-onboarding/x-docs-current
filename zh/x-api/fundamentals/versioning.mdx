<p>随着 X API v2 的推出，我们发布了一套新的版本管理策略，帮助开发者更好地了解何时会对 X 的公共 API 进行更改，以及何时需要迁移到新版本。 </p>
<p>开发者将通过我们的 <a href="https://developer.x.com/en/updates/stay-informed">沟通渠道</a> 获知对 X API 的弃用、停用、变更和新增内容，从而能够在开发路线图中合理规划，适配这些变化。对 API 的所有更改都会记录在更新日志中。</p>
<p>目前 X API 提供三个不同的版本。除非你的用例所需的功能尚未在 X API v2 中提供，否则我们强烈建议你在规划集成时使用 X API v2。 </p>
<p>要进一步了解各版本，请访问以下页面：</p>

<ul>

  <li>Gnip 2.0（<a href="/zh/x-api/enterprise-gnip-2.0/enterprise-gnip">Enterprise</a>）</li>
</ul>

<div id="versioning-strategy">
  ## 版本管理策略
</div>

<p>X API 的版本将通过在端点路由路径中声明的版本号来表示：</p>
<p>`https://api.x.com/2/tweets`</p>
<p>我们会在必要时发布 X API 的主要版本，但频率不会高于每 12 个月一次。当在 API 中引入不兼容变更时，将发布一个主要版本。在推出新的主要版本时，我们会发布迁移指南，帮助开发者迁移到新版本。 </p>
<p>不兼容变更会要求开发者修改代码，以在其应用中保持现有功能。兼容变更将以增量方式添加，并在准备就绪时发布到最新版本中；除非你希望利用新功能，否则开发者无需进行任何改动。</p>
<p>如果出于安全或隐私原因必须在发布周期中途推出不兼容变更，该变更将应用到最新版本。</p>

<div id="breaking-changes">
  ### 破坏性变更
</div>

<p>这些变更要求开发者修改代码，以保持其应用的现有功能。</p>

<ul>
  <li>新增一个必填参数</li>
  <li>移除现有的 endpoint</li>
  <li>移除响应中的任意字段（无论是必填还是可选）</li>
  <li>移除查询参数</li>
  <li>调整输入或输出格式（例如，将顶层字段改为子字段，或将错误改为以内联方式返回）</li>
  <li>更改现有输入参数或输出值的名称或数据类型</li>
  <li>更改字段名称</li>
  <li>更改资源名称</li>
  <li>更改响应状态码</li>
  <li>更改错误类型</li>
  <li>更改现有授权作用域（scope）</li>
</ul>

<div id="non-breaking-changes">
  ### 非破坏性更改
</div>

<ul>
  <li>新增端点</li>
  <li>新增可选参数</li>
  <li>新增响应字段</li>
  <li>修改错误消息文本</li>
  <li>新增可用的 scope（权限范围）</li>
  <li>字段“置为 null”（将某个字段的值更改为 null，以满足隐私/安全需求，作为不完全移除该字段的替代方案）</li>
</ul>

<div id="deprecation-and-retirement">
  ### 弃用和退役
</div>

<p>首先，以下是我们在 X API 中对“弃用”和“退役”的定义：</p>

<ul>
  <li><b>弃用：</b>该功能不再由团队支持。不会再针对该功能开发或发布任何新功能，如果产品存在任何缺陷或问题，我们评估并修复该问题的可能性极低。 </li>
  <li><b>退役：</b>该功能将不再可访问。</li>
</ul>

<p>在大多数情况下，一旦新版本发布，上一版本就会被标记为已弃用。版本会在弃用状态下保留一段时间，之后将被退役。 </p>
<p>请<a href="https://developer.x.com/en/updates/stay-informed">保持关注</a>以了解更多未来的弃用和退役信息。</p>
