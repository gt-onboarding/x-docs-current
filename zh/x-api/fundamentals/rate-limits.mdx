keywords: ["速率限制", "API 速率限制", "请求限制", "速率限制机制", "限流", "15 分钟时间窗", "速率限制错误"]
---

每天都有成千上万的开发者向 X API 发起请求。为了管理如此巨大的请求量，我们会对可发起的请求数量设置限制。这些限制有助于为我们的开发者社区提供可靠且可扩展的 API。

允许的最大请求数是基于时间间隔设定的，通常是在指定的一段时间或时间窗口内。最常见的间隔是 15 分钟。例如，一个端点如果限制为每 15 分钟 900 个请求，则意味着在任意连续 15 分钟内最多可以发送 900 个请求。

速率限制取决于所使用的身份验证方法。例如，如果使用 [OAuth 1.0a 用户上下文](/zh/resources/fundamentals/authentication)，每组用户的 [Access Tokens](/zh/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) 在每个时间段内都有各自的速率限制。或者，如果使用 [OAuth 2.0 Bearer Token](/zh/resources/fundamentals/authentication#oauth-2-0)，你的应用在每个时间段内会有自己单独的限制。当超出这些限制时，将返回错误。

<div id="rate-limits-and-authentication-method">
  ## 速率限制和身份验证方式
</div>

速率限制分别在开发者应用级别和用户访问令牌级别设置：

* **[OAuth 2.0 Bearer Token](/zh/resources/fundamentals/authentication/oauth-2-0/application-only)：应用速率限制**
  此方法允许你代表你的开发者应用发起一定数量的请求。使用这种身份验证方式时，限制由使用 Bearer Token 发起的请求数决定。
  * 示例：在限制为每 15 分钟 450 个请求的情况下，你可以在该时间间隔内代表你的应用发起 450 个请求。

* **[OAuth 1.0a User Context](/zh/resources/fundamentals/authentication/oauth-1-0a/obtaining-user-access-tokens)：用户速率限制**
  此方法允许你代表由用户访问令牌标识的 X 用户发起请求。例如，如果要获取帖子上的私有指标，请使用通过 [三方 OAuth 流程](/zh/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) 为该用户生成的用户访问令牌进行身份验证。
  * 示例：在限制为每位用户每 15 分钟 900 个请求的情况下，你可以在该时间段内对每位用户最多发起 900 个请求。

<div id="http-headers-and-response-codes">
  ## HTTP 头信息和响应代码
</div>

根据最近一次请求，使用 HTTP 头信息了解你的应用在当前速率限制中的使用情况。

* `x-rate-limit-limit`：该端点的速率限制上限
* `x-rate-limit-remaining`：当前 15 分钟窗口内剩余的请求数
* `x-rate-limit-reset`：距离速率限制重置的剩余时间（以 UTC 纪元秒表示）

<div id="error-responses">
  ### 错误响应
</div>

如果应用在某个端点超出了速率限制，API 将返回 [HTTP 429 “Too Many Requests”](http://tools.ietf.org/html/rfc6585) 响应，并在响应正文中包含以下错误信息：

```json
{ "errors": [ { "code": 88, "message": "Rate limit exceeded" } ] }
```

<div id="recovering-from-a-rate-limit">
  ## 从速率限制中恢复
</div>

当超过这些速率限制时，端点会返回 429 “Too many requests”（请求过多）错误。如下所述，当发生速率限制错误时，最佳实践是检查指示限制何时重置的 HTTP 头部，并在此之前暂停请求。<br />当出现 “too many requests” 或速率限制错误时，需要降低发出请求的频率。当遇到速率限制错误时，可以检查 `x-rate-limit-reset:` HTTP 头部，以了解速率限制何时会重置。<br /><br />另一种常见模式是采用指数退避策略，即请求之间的时间间隔一开始较短（例如几秒钟），然后在每次重试前加倍。持续这样做，直到请求成功，或者达到一个合理的请求间最大时间间隔（例如几分钟）。<br /><br /> 理想情况下，客户端能够自动识别现有的速率限制，并在当前超出的时间窗口过期之前暂停请求。如果你超过的是 15 分钟的限制，那么在重试前等待一两分钟是合理的。<br /><br /> 请注意，除了这些对请求次数的限制之外，Standard Basic 访问级别还通过最近搜索和过滤流端点，每月最多提供 500,000 条帖子。如果你已经超过了每月的帖子数量限制，那么对你的应用而言，更合理的做法是发出通知，记录每月的开通日，并在该日期之前暂停请求。

<div id="tips-to-avoid-being-rate-limited">
  ### 避免触发速率限制的提示
</div>

下面这些提示旨在帮助你以防御性方式编写代码，降低遭遇速率限制的可能性。由于速率限制的存在，你可能无法实现某些希望在应用中提供的功能，尤其是对结果实时性要求较高的功能。如果你的应用以实时信息为目标，请考虑使用 filtered stream 和 sampled stream 端点。

<div id="caching">
  #### 缓存
</div>

如果预计会频繁使用，请将 API 响应存储下来。与其在每次页面加载时都调用 API，不如将响应缓存在本地。

<div id="prioritize-active-users">
  #### 优先处理活跃用户
</div>

如果你的网站需要跟踪大量 X 用户（例如获取他们的当前状态或关于其 X 使用情况的统计信息），请考虑只为那些最近在你的网站上登录过的用户请求数据。

<div id="adapt-to-the-search-results">
  #### 根据搜索结果进行调整
</div>

如果你的应用需要监控大量搜索词，对于没有结果的搜索词，查询频率应当低于有结果的搜索词。通过使用退避策略，你可以既保持对热门查询的及时更新，又避免在极少变化的查询上浪费请求资源。或者，可以考虑使用 filtered stream 端点，并使用你的搜索查询来进行过滤。

<div id="denylist">
  #### 封禁名单
</div>

如果某个应用滥用速率限制，它将被封禁。被封禁的应用将无法从 X API 获取响应。如果你或你的应用已被封禁，并认为这是一个错误，你可以通过我们的[平台支持表单](https://support.x.com/forms/platform)请求协助。请包含以下信息：

1. 说明你认为你的应用被封禁的原因。
2. 如果你现在已不再触发速率限制，请详细描述你是如何解决该问题的。
