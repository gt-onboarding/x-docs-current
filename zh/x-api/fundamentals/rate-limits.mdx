---
title: 速率限制
keywords: ["rate limits", "API rate limits", "request limits", "rate limiting", "throttling", "15 minute window", "rate limit errors"]
---

每天都有成千上万的开发者向 X API 发起请求。为了管理如此庞大的请求量，我们对在特定时间内可以发起的请求数量进行了限制。这些限制有助于为我们的开发者社区提供可靠且可扩展的 API 服务。

允许的最大请求数基于一个时间区间，通常是一段指定的时间或时间窗口。最常见的时间区间是 15 分钟。例如，一个端点如果限制为每 15 分钟最多 900 次请求，就意味着在任意 15 分钟区间内最多可以发起 900 次请求。

速率限制取决于所使用的身份验证方式。例如，如果使用 [OAuth 1.0a User Context](/zh/resources/fundamentals/authentication)，每组用户的 [Access Tokens](/zh/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow) 在每个时间周期内都有各自独立的速率限制。或者，如果使用 [OAuth 2.0 Bearer Token](/zh/resources/fundamentals/authentication#oauth-2-0)，你的应用在每个时间周期内也会有自己独立的限制。当超出这些限制时，将返回错误。

<div id="rate-limits-and-authentication-method">
  ## 速率限制和身份验证方式
</div>

速率限制分别在开发者应用和用户访问令牌两个层级设定：

* **[OAuth 2.0 Bearer Token](/zh/resources/fundamentals/authentication/oauth-2-0/application-only)：应用速率限制**
  此方式允许你代表你的开发者应用发出一定数量的请求。使用这种身份验证方式时，限制是根据使用 Bearer Token 发起的请求数来确定的。
  * 示例：在限制为每 15 分钟 450 次请求的情况下，你可以在该时间段内代表你的应用发出 450 次请求。

* **[OAuth 1.0a User Context](/zh/resources/fundamentals/authentication/oauth-1-0a/obtaining-user-access-tokens)：用户速率限制**
  此方式允许代表由用户访问令牌标识的 X 用户发出请求。例如，如果要获取帖子中的私有指标，请为该用户使用通过[三方 OAuth 流程](/zh/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow)生成的用户访问令牌进行身份验证。
  * 示例：在限制为每个用户每 15 分钟 900 次请求的情况下，你在该时间段内最多可以为每个用户发出 900 次请求。

<div id="http-headers-and-response-codes">
  ## HTTP 标头和响应代码
</div>

使用 HTTP 标头来了解基于最近一次请求，你的应用在给定速率限制中的使用情况。

* `x-rate-limit-limit`: 该端点的速率限制上限
* `x-rate-limit-remaining`: 15 分钟时间窗内剩余的请求数
* `x-rate-limit-reset`: 距离速率限制重置的剩余时间（以 UTC 纪元秒为单位）

<div id="error-responses">
  ### 错误响应
</div>

如果应用超出了某个端点的速率限制，API 会返回一个 [HTTP 429 “Too Many Requests”](http://tools.ietf.org/html/rfc6585) 响应，并在响应正文中包含以下错误信息：

```json
{ "errors": [ { "code": 88, "message": "Rate limit exceeded" } ] }
```

<div id="recovering-from-a-rate-limit">
  ## 从速率限制中恢复
</div>

当超过这些速率限制时，端点会返回 429 “Too many requests” 错误。如下所述，当出现速率限制错误时，最佳实践是检查指示限制何时重置的 HTTP 头部，并在此之前暂停发送请求。<br />当出现 “too many requests” 或速率限制错误时，需要降低发起请求的频率。当触发速率限制错误时，可以检查 `x-rate-limit-reset:` HTTP 头部，以了解速率限制何时会重置。<br /><br />另一种常见模式是使用指数退避（exponential backoff），即请求之间的时间间隔一开始很短（例如几秒钟），然后在每次重试前加倍。如此反复，直到请求成功，或者达到某个合理的最大请求间隔时间（例如几分钟）。<br /><br /> 理想情况下，客户端应能自动感知现有的速率限制，并在当前超出的时间窗口到期前暂停请求。如果你超出了 15 分钟限额，那么在重试前等待一到两分钟是合理的。<br /><br /> 请注意，除了这些请求次数限制之外，Standard Basic 访问级别还通过最近搜索（recent search）和 filtered stream 端点每月最多提供 500,000 条帖子。如果你超出了每月帖子数量限制，那么让你的应用触发通知，记录其每月的订阅起始日，并在该日期之前暂停请求会更合理。

<div id="tips-to-avoid-being-rate-limited">
  ### 避免触发速率限制的提示
</div>

下面这些提示旨在帮助你采用防御性编程方式编写代码，并降低触发速率限制的可能性。受速率限制机制影响，你可能想为应用提供的某些功能在实践中根本无法实现，尤其是对结果“新鲜度”的要求。如果你的应用以获取实时信息为目标，请考虑使用 filtered 和 sampled 流式端点。

<div id="caching">
  #### 缓存
</div>

如果预计会频繁使用某个接口的结果，可以将 API 响应存储下来。与其在每次页面加载时都调用 API，不如将响应缓存在本地。

<div id="prioritize-active-users">
  #### 优先处理活跃用户
</div>

如果你的网站需要跟踪大量 X 用户（例如获取他们的当前状态或关于其 X 使用情况的统计信息），请考虑仅为最近在你的网站上登录过的用户发起数据请求。

<div id="adapt-to-the-search-results">
  #### 根据搜索结果进行调整
</div>

如果你的应用需要监控大量搜索词，对于那些没有结果的搜索，相比有结果的搜索，应当降低其查询频率。通过使用退避策略，你可以及时获取热门查询的最新结果，同时避免在几乎不会变化的查询上浪费请求资源。或者，你也可以考虑使用 filtered stream endpoint，并使用你的搜索查询来设置过滤条件。

<div id="denylist">
  #### 封禁列表
</div>

如果某个应用滥用速率限制，该应用将被加入封禁列表。被封禁的应用将无法从 X API 获取任何响应。如果你或你的应用被封禁，并且认为这是一个错误，你可以使用我们的[平台支持表单](https://support.x.com/forms/platform)寻求帮助。请提供以下信息：

1. 说明你认为你的应用为何被封禁。
2. 如果你现在已经不再触发速率限制，请详细描述你是如何解决该问题的。
