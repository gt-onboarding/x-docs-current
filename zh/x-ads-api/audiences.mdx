## 自定义受众

<div id="overview">
  ### 概览
</div>

合作伙伴有多种方式创建[自定义受众](https://business.x.com/en/targeting/tailored-audiences.html)。

* [受众 API（CRM）](#crm)
* [网页](#web)
* [移动端](#mobile)
* [灵活受众](#flexible)

请注意，你无法在定向中排除相似自定义受众。此外，你也不能在同一个广告明细项（广告组）中同时定向一个自定义受众和一个自定义受众的相似受众。

**受众管理**

受众可以通过受众合作伙伴和 Ads API 合作伙伴进行管理。我们在 API 中提供了一系列端点，用于访问和维护自定义受众。

对于自定义受众信息，我们提供两个端点：

* [GET accounts/:account&#95;id/custom&#95;audiences](/zh/x-ads-api/audiences)
* [GET accounts/:account&#95;id/custom&#95;audiences/:custom&#95;audience&#95;id](/zh/x-ads-api/audiences)

有关如何上传和管理受众的更多详细信息，请参阅[受众 API 指南](/zh/x-ads-api/audiences)。

**处理时间**

一般来说，受众变更会以批处理方式进行，每 6–8 小时运行一次。在受众变更处理期间，正在更新的现有受众不会受到影响。我们不建议在这一时间范围内针对同一受众进行超过一次的新增操作和一次的移除操作。

**定向**

只有当某个受众至少匹配到 100 名在过去 90 天内活跃于 X 自有及运营客户端的用户时，该受众才能被用于定向。[GET accounts/:account&#95;id/custom&#95;audiences/:custom&#95;audience&#95;id](/zh/x-ads-api/audiences) 会指明某个受众是否由于匹配用户过少而无法用于定向。

<div id="audiences-faq">
  ### 受众常见问题解答[](#real-time-tailored-audiences "此标题的固定链接")
</div>

**问：我们推送了大量数据，为什么受众规模仍然显示为 TOO&#95;SMALL？**

答：目前数据会被实时加入到受众中，但用于计算并提供受众规模的处理任务只会在一段时间之后运行。正确的受众规模应在数小时后显示在 UI 中。

**问：我们已经完成发送受众数据，并等待了 24 小时或更久，但仍然无法投放到该受众——下一步该怎么做？**

答：请确认以下事项：

* 传递的用户 ID 是否正确且格式未损坏。
* 传递的受众名称是否正确，并且与之前的成员更新相匹配。
* 请确认 POST 命令的响应是否正常。
* 请确认 ID Sync 像素是否已按要求正确实现，并且如 ID Sync 流程所述，有足够多的用户访问了相关站点以完成用户映射。未被映射的用户即使出现在成员更新中，也不会被转换为可定向用户。

如果以上全部已确认无误且工作正常，请联系 X 产品联系人，并尽可能详细地提供信息（参见 [Guide to Partner Inbounds](/zh/x-ads-api/introduction) 了解推荐的信息示例）。

**问：我们可以调用该 endpoint 多少次，以及应该使用什么算法？**

答：我们强烈建议只用增量变更来调用我们的系统，切勿重复发送完整的受众成员列表。系统已经过测试，具备足够的吞吐量来处理全球最大网站之一的增量数据更新。首次上传受众时应谨慎控制节奏，预期首次上传将需要相当长的时间才能完成。

**问：受众用于定向时的最小规模是多少？**

* 受众的最小规模为 100 名用户（匹配后）。如果匹配到的受众少于 500 名用户，则该受众在 X Ads UI 中将不可用于定向。

**问：处理受众文件需要多长时间？受众文件在 X 用户界面中可用需要多长时间？**

* 通常处理受众文件需要 4–6 小时，但这取决于文件大小。文件一旦处理完成，这些受众就会在 X Ads UI 中可用。

**问：匹配率是如何计算的？**

* 匹配率 = 90 天内活跃的 X 用户数 / 提供的用户数

**问：我们如何测试一个受众文件是否正常工作？**

* 你可以提供一个测试受众文件，并使用“keltonlynn”作为广告主 handle。之后我们可以验证该文件是否能够被正确摄取并加载到 X 界面中。

**问：什么是合作伙伴用户标识符（`p_user_id`）？**

* 这是你们公司用于唯一标识每位客户的标识符。

**问：什么是标准 ID？**

* 它可以是电子邮件地址、设备 ID、X @handle 或 ID。

**问：如何获取 HMAC Key？**

* 我们会通过加密电子邮件提供该密钥。请将你的公共 PGP 密钥发送到 [mpp-inquiry@x.com](mailto:mpp-inquiry%x.com)，我们会先给你发送一封测试邮件以验证一切正常。验证通过后，我们会将 HMAC Key 发送给你。

**问：如何使用提供的 HMAC Key 验证哈希过程是否成功？**

* X 会提供一个测试文件（包含示例电子邮件地址、设备 ID 等）以及一个对应的哈希结果文件，你可以据此比对验证自己的结果。

**问：完整数据匹配文件是否有大小限制？**

* 不，完整数据匹配文件没有大小限制。

**问：处理完整数据匹配文件需要多长时间？**

* 一旦 X 收到该文件，处理大约需要 1 天时间。

本文档说明 Custom Audiences CRM 合作伙伴的集成细节，包括文件格式和数据交换流程。

**摘要**

Company 将代表某个客户向 X 提供一份经过哈希处理的常见用户标识符列表（例如电子邮件地址）或合作伙伴用户 ID，以便执行匿名匹配并生成用于定向的 X 用户 ID 列表。这些用于定向的细分人群将关联到广告主在 ads.x.com 广告系列设置中、由文件名指定的特定 @handle。

来自 Company 的所有文件都将通过 IronBox（www.golockbox.com）上的安全包传输给 X，使用的是 X 授予 Company 的专用账户。X 将提供对 IronBox 的访问权限。IronBox API 的文档可在 [https://secure.goironcloud.com/Docs/Help/](https://secure.goironcloud.com/Docs/Help/) 查阅。

<div id="partner-id-matching-requirements">
  #### 合作方 ID 匹配要求
</div>

如果公司使用自身的标准 ID 系统来跟踪用户（即不使用电子邮件地址、设备 id、X 用户 ID 等常见用户标识符），则推荐采用以下流程。

**1. 全量数据匹配**

初始阶段，公司将通过单个文件向 X 提供包含所有用户记录的完整列表，这些记录中包含与 X 共享的唯一通用用户标识符，以执行一次全量数据匹配，并由 X 生成并存储合作方 ID（`p_user_id`）到 X ID（`tw_id`）的映射。此操作将每 2–3 个月定期执行一次，以确保正确维护。匹配完成后，X 将通过电子邮件向公司分享基于该文件的基线匹配率。

该文件的格式应为：

名称约定：FullDataMatch.[CompanyName].txt

哈希算法：HMAC&#95;SHA-256

格式：

第 1 列：通用用户标识符的 HMAC 哈希值

第 2 列：合作方用户 ID（对每个用户唯一，但在文件中可重复出现）

列分隔符（CSV）：使用逗号分隔哈希后的通用用户标识符和合作方 ID

按行分隔的值

* 示例：如果用户记录 A 的合作方用户 ID 为 1，通用用户标识符为 1、2 和 3：

|     |     |
| :--- | :--- |
| 通用用户标识符 1 | p&#95;user&#95;1 |
| 通用用户标识符 2 | p&#95;user&#95;1 |
| 通用用户标识符 3 | p&#95;user&#95;1 |

* 有关通用用户标识符的哈希指引，请参见下方“哈希说明”部分

**2. 自定义细分列表**

公司将以 `p_user_id` 列表的形式提供用户列表，用于为客户在 X 上创建自定义受众以进行定向投放。

* 按行分隔的值
* `p_user_id`
* （与上文 1. 全量数据匹配部分中提供的内容相同。如果在全量数据匹配中提供的值是哈希后的，则公司将在受众文件中提供相同的哈希值；如果提供的值未哈希，则公司将提供未哈希的值。）
* 仅包含合作方用户 ID 列，不需要包含通用用户标识符
* 每行一个 `p_user_id`，无需额外分隔符或列

<div id="standard-matching-requirements">
  #### 标准匹配要求
</div>

如果公司没有使用统一的标准 ID 来映射所有客户用户标识符，建议采用以下流程。

**自定义分群列表**

公司将代表其客户，直接向 X 提供经过哈希处理的通用用户标识符列表，用于创建自定义受众。

此文件的格式应为：

* 行分隔的值（每行一个值）
* 通用用户标识符的哈希值（例如电子邮件地址）
* 遵循下文所述的文件命名规范
* 遵循下文“哈希处理说明”中关于电子邮件地址的哈希处理指引

<div id="custom-segment-list-file-naming-operations">
  #### 自定义分段列表文件命名与操作
</div>

文件的操作将由其文件名决定，可用操作及通用文件命名规范如下：audiencename&#95;partnername.handle.operation.filetype

* audiencename：自定义受众的名称。此字段是在 ads.x.com 广告系列设置界面中选择受众时显示的名称，例如 brand&#95;loyalty&#95;card&#95;holders。
* partnername：代表广告主上传数据的公司的名称，例如 company&#95;name。
* handle：将拥有自定义受众访问权限的 X 账号（@handle），例如 @pepsi、@dietpepsi
* operation：new、add、remove、removeall、replace（详见下文）
* ：标准 Unix 纪元时间（秒），用于确保上传的每个受众文件都是唯一的
* filetype：文件应为 *.txt 文本格式

<div id="creating-and-updating-audiences">
  #### 创建和更新受众
</div>

使用单个文件创建一个新受众，例如：loyalty&#95;card&#95;holders&#95;partnername.pepsi.new.txt

Add - 将某个列表中的匹配项添加到现有受众，例如：loyalty&#95;card&#95;holders&#95;partnername.pepsi.add..txt

Remove - 从现有受众中移除某个列表中的匹配项，例如：loyalty&#95;card&#95;holders&#95;partnername.pepsi.remove..txt

Remove All - 将定期更新的累积列表中产生的匹配项，从该客户的所有受众中移除（即客户的选择退出列表 Opt Out List）。例如：partnername.pepsi.removeall.txt

* 这可用于存放从该广告主处选择退出的用户的完整列表。
* X 仅以此文件中提供的最新列表为准，并会在提供并处理该文件时，对所有已匹配上的现有及未来受众生效。

Replace - 移除一个现有受众，并用一个新的受众列表替换它。例如：loyalty&#95;card&#95;holders&#95;partnername.pepsi.replace..txt

Overall Company Opt-Out - 公司将提供一个累积的选择退出（Opt-Out）文件，用于移除根据公司选择退出政策已选择退出的用户。

X 仅以此公司选择退出文件中提供的最新列表为准，并会在提供并处理该文件时，对所有已匹配上的现有及未来受众中的 X 用户生效。公司选择退出文件的格式如下：例如：partnername.removeall.txt

Delete - 从当前受众列表中移除一个现有受众，例如：loyalty&#95;card&#95;holders&#95;partnername.pepsi.delete.txt

<div id="hashing-directions">
  #### 哈希指引
</div>

X 将通过 PGP 安全地共享一个经过 Base64 编码的生产密钥，用于对常见用户标识符（如电子邮件地址）进行哈希计算。公司应对该密钥进行 Base64 解码，以生成一个 32 字节的密钥，用于执行哈希。

示例 Base64 编码密钥：

BrQvOg+dACBUmKjRiNxZgJLh6zydjS0ZOv80FelTNzM=

示例 Base64 解码密钥：

/:� TшY

规范化处理：公司将在对常见用户标识符进行哈希前执行基本规范化处理（设备 ID 除外，参见“设备 ID 规范化”部分）。

<div id="e-mail-normalization">
  #### 电子邮件规范化
</div>

即去除前后空格，并将电子邮件地址全部转换为小写。

例如：原始电子邮件地址：testemail&#95;Organisational&#95;baseball+884`@`It92I6Ev2B`.`Com

规范化后：testemail&#95;organisational&#95;baseball+884`@`it92i6ev2b`.`com

哈希值：74d9584eded0ad1e5572a1c1849f3716751d371d6117a6155dad5363f4b4fbec

注意：编码后的 HMAC 和密钥的字符数可能会根据输入和编码方式而变化，因此具体字符数并不固定。

<div id="device-id-normalization">
  #### 设备 ID 规范化
</div>

我们对设备 ID 的哈希有一致的要求：使用 SHA-256 哈希算法以及我们提供给数据合作伙伴的通用盐值。我们会像处理电子邮件地址一样去除空格，但不会对 IDFA/Android ID 进行小写转换，必须使用 IDFA/Android ID 的原始格式。

下面是 iOS 和 Android 设备 ID 在哈希前的原始格式示例：

iOS IDFA：DD99CFF7-6186-4602-9DF2-ED3FD0B2D431

Android ID：b5bf2122961b3595

哈希后的 iOS IDFA：134fb8cd95c7fd42e2793f469a447198ca5f990968db2dbadad70e723ed9750b

哈希后的 Android ID：130dddff1939f229476f50bc8adab8fcb7e3525b0e9604fe8effc15e68cee4a4

<div id="x-user-id-normalization">
  #### X 用户 ID 规范化
</div>

X ID 仍将进行哈希处理，因为用于数据分组的数据——例如由 @handles 构成的 Customer List——即使不是个人可识别信息（PII），对广告主来说也仍然是私有的。对于 X ID 的哈希计算，我们仍采用相同要求：使用 SHA-256 哈希算法，以及我们提供给数据合作伙伴的通用盐值。应从 X ID/`@`username 中移除空格，但 User ID 不需要进行规范化。为实现规范化，应将 @username 转为小写。同时，@ 符号本身不应作为 username 的一部分包含在内。

原始 ID 格式如下：

* User ID: 27674040
* @username: testusername

哈希后的 User ID: bf6b57d4e861e83bea8bbed2b800b251a64c95468ee6e8cb07c3368c9ed45e85

哈希后的 @username: 12201ae78ad1afa907c7112d17f498154ffb0bf9ea523f5390e072a06d7d9812

**数据标准化**

**设备 ID**：

* IDFA - 使用带连字符的小写格式；例如：`4b61639e-47cc-4056-a16a-c8217e029462`
* AdID - 需要与设备上的原始格式一致，不要全部大写，保留连字符；例如：`2f5f5391-3e45-4d02-b645-4575a08f86e`
* Android id - 需要与设备上的原始格式一致，不要全部大写，且不包含连字符或空格；例如：`af3802a465767e36`

**电子邮件地址**：

* 使用小写，去除前后空格；例如：`support@x.com`

**X 用户名**：

* 不包含 @，使用小写，并去除前后空格；例如：`jack`

**X 用户 ID**：

* 标准整数；例如：`143567`

**数据哈希**

每一行数据都必须使用 `SHA256` 进行哈希，且不得使用盐值（salt）。另外，最终输出的哈希值必须为小写。例如：49e0be2aeccfb51a8dee4c945c8a70a9ac500cf6f5cb08112575f74db9b1470d，**而不是**49E0BE2AECCFB51A8DEE4C945C8A70A9AC500CF6F5CB08112575F74DB9B1470D

```
# hasing user @AdsAPI using python
import hashlib
hashlib.sha256("adsapi".encode()).hexdigest()

#output
49e0be2aeccfb51a8dee4c945c8a70a9ac500cf6f5cb08112575f74db9b1470d
```

更多与哈希相关的代码示例可在 [github.com/xdevplatform/ads-platform-tools](https://github.com/xdevplatform/ads-platform-tools) 找到。

**信息**

合作伙伴会代表广告主发送一组 ID（`p_user_ids`）用于定向。此操作通过 ID 同步（ID Sync）流程完成，该流程会构建 `p_user_ids` 与 X 用户 ID 之间的映射。然后使用该映射生成可用于定向的 X 用户 ID 列表。这些自定义受众将在广告主特定的 @handle 下可用，具体由 ads.x.com 上“Custom Audiences Web”广告系列设置中的标签指定。

X 将提供一个可部署在合作伙伴标签和站点上的安全像素代码，用于将这些 ID（`p_user_ids`）与 X 用户 ID 进行匹配。ID Sync 流程完成后，合作伙伴将创建定向文件，并通过一个 HTTPS 端点提供给 X。X 会定期获取并导入这些定向文件，然后在 X 界面中提供使用。

**X 安全像素**

X 安全像素的形式如下：

[https://analytics.x.com/i/adsct?p&#95;user&#95;id=xyz&amp;p&#95;id=123](https://analytics.x.com/i/adsct?p\\_user\\_id=xyz\&p_id=123)

**`p_user_id`** - xyz 表示由合作伙伴提供的合作伙伴用户 ID

**`p_id`** - 123 表示合作伙伴的唯一 ID（由 X 提供）

**合作伙伴 HTTPS 端点和定向用户文件**

合作伙伴需要向 X 提供一个 HTTPS 端点以及用于定期获取并导入定向文件的凭证（用户名/密码）。示例 HTTPS 端点形式如下：

```
https://<partnerdomain>/twitter/partner_targeting_%Y-%M-%D.tsv.gz
```

%Y - 年格式代码 (YYYY)

%M - 月格式代码 (MM)

%D - 日格式代码 (DD)

传输的数据将包含以下文件：

1. Partner Targeting User File
2. Targeting Conversion File

所有文件均为 TSV 格式，其中每一行中的各个字段之间以制表符分隔。有效的字段值本身绝不会包含制表符。

**允许的 X IP 范围：**

以下是可被授权访问 Partner Endpoint 的 IP 段范围。

* 199.16.156.0/22
* 199.59.148.0/22

**Partner Targeting User File：**

| **Column Number** | **Column Name**   | **Column Type** | **Description**                                                                                                                             |
| :---------------- | :---------------- | :-------------- | :------------------------------------------------------------------------------------------------------------------------------------------ |
| 1                 | partner id        | string          | “partner id” 是 X 提供给 Partner 的 ID，用于唯一标识每个 Partner。                                                                                         |
| 2                 | advertiser id     | string          | “advertiser id” 是广告主的 @handle。                                                                                                              |
| 3                 | p&#95;user&#95;id | string          | “p&#95;user&#95;id” 是 Partner 用来标识用户的唯一 ID。                                                                                                 |
| 3                 | confidence score  | integer         | “confidence score”（置信度评分）为可选字段。我们建议的取值范围为 0–100。如果用例是重定向，则 confidence score 为 “100” 表示该用户是被直接重定向的。0–99 之间的任何评分都表示对相似受众的置信度水平。               |
| 4                 | segment label     | string          | “segment label”（分群标签）为可选字段。Partner 可以使用 “segment label” 来指定例如产品类别等信息。我们建议使用此 “segment label”，因为它是在 ads.x.com UI 中 Custom Audiences 的人类可读名称。 |

**说明：**

每次我们收到新的 Partner Targeting File 时，除非另有约定，我们都期望该文件是 Partner 推荐我们投放目标用户的完整列表，而不是增量列表。我们会与每个 Partner 协商此 Partner Targeting File 的交付频率。如果在预期时间内没有收到 Partner Targeting File，我们将继续使用上一版本，并设定预先定义的过期时间。

<div id="audience-api-integration">
  ## Audience API 集成
</div>

### 概述

Audience API 作为 Ads API [v4](/zh/x-ads-api/introduction) 的一部分推出，对旧版 Audiences 端点进行了多项改进。这个新端点由全新的 Audience 处理后端提供支持，在稳定性、健壮性和可靠性方面都有显著提升。本指南旨在重点说明 Audience API 与传统 Audience 上传和管理流程之间的差异。 

你可以在 [Audience API](/zh/x-ads-api/audiences) 参考文档页面查看详细说明。 

**注意**：所有 Audience 用户数据在上传前都必须进行 SHA-256 哈希处理。关于更多细节、可接受的用户标识符类型以及数据规范化要求，请参阅 [user data](/zh/x-ads-api/audiences) 页面。

**Audience 功能变更**

自 v4 起，Custom Audience 引入了以下变更，一旦 Ads API v3 下线，任何被弃用的端点都将不再可用：

* **已弃用** TON Upload：
  * GET accounts/:account&#95;id/custom&#95;audience&#95;changes
  * GET accounts/:account&#95;id/custom&#95;audience&#95;changes/:custom&#95;audience&#95;change&#95;id
  * POST accounts/:account&#95;id/custom&#95;audience&#95;changes
  * PUT accounts/:account&#95;id/custom&#95;audiences/global&#95;opt&#95;out
* **已弃用** Real Time Audiences：
  * POST custom&#95;audience&#95;memberships
* Custom Audience：
  * `list_type` 参数将从所有 [Custom Audience](/zh/x-ads-api/audiences) 端点的请求和响应中移除。该参数之前用于标识 Audience 的用户标识符类型（例如 email、X User ID 等），但现在 Audience 能够针对同一个 Audience 接受多种用户标识符，因此该参数已不再需要。
* 通用：
  * Audience 回溯窗口已更新为匹配过去 90 天内活跃的用户（此前为 30 天）
  * 受众可被定向投放所需的最小匹配用户数已降低为 100 个用户（此前为 500 个用户）

<Note>
  **先决条件**

  * Ads API 访问权限
  * 若要访问 Audience 端点，你需要被加入 allowlist。请填写此表单，并在最初接受时间早于 2018-08-01 的情况下，接受新的 [X Ads Products and Services Agreement](/zh/x-ads-api/introduction)
</Note>

**Audience 上传流程**

下表列出了旧版与新版 Audience 创建流程之间的主要差异，更多详细信息将在后文进一步说明：

| 流程步骤 | Audience API | （已弃用）TON Upload |
| :--- | :--- | :--- |
| 创建空白 Audience | 可通过 [POST custom&#95;audience endpoint]/x-ads-api/audiences 创建 | 可通过 [POST custom&#95;audience endpoint]/x-ads-api/audiences 创建 |
| 添加新用户 | 使用 `operation_type` 为 `Update` 搭配 [Audience endpoint](/zh/x-ads-api/audiences) | 使用 `operation` 为 `ADD` 搭配 [POST custom&#95;audience&#95;changes](/zh/x-ads-api/audiences) 端点 |
| 移除用户 | 使用 `operation_type` 为 `Delete` 搭配 [Audience endpoint](/zh/x-ads-api/audiences) | 使用 `operation` 为 `REMOVE` 搭配 [POST custom&#95;audience&#95;changes](/zh/x-ads-api/audiences) 端点 |
| 为用户设置退出（Opt-out） | 使用 `operation_type` 为 `Delete` 搭配 [Audience endpoint](/zh/x-ads-api/audiences)，并提供该用户所属的相应 `custom_audience_id` | 使用 [Global opt-out endpoint](/zh/x-ads-api/audiences) |

**注意** 任何通过 TON Upload 路径进行更新或退出操作的 Audience，都必须先通过 [TON Upload](/zh/x-ads-api/audiences) 端点上传相应列表，并使用 [custom&#95;audience&#95;changes](/zh/x-ads-api/audiences) 端点将其关联到某个 Audience。

**速率限制**

Audience API 端点的速率限制为每个账号每 1 分钟 1500 次调用。单个 payload 中可发送的用户数量没有限制。对 payload 的唯一约束为：

1. 操作总数：最多 2500 个操作

2. 最大 payload 大小：5,000,000 字节

**Audience 用户管理**

<div id="remove-users-from-an-audience">
  #### 从受众中移除用户
</div>

与上文添加用户的流程类似，也可以通过如下请求将用户从受众中移除：

POST [https://ads-api.x.com/11/accounts/18ce54d4x5t/custom&#95;audiences/1nmth/users](https://ads-api.x.com/11/accounts/18ce54d4x5t/custom_audiences/1nmth/users)

```

    [
      {
        "operation_type": "Delete",
        "params": {
          "effective_at": "2018-05-15T00:00:00Z",
          "expires_at": "2019-01-01T07:00:00Z",
          "users": [
            {
              "email": [
                "abc@x.com"
              ],
              "twitter_id": [
                "783214",
                "1225933934"
              ]
            },
            {
              "email": [
                "edf@x.com"
              ],
              "twitter_id": [
                "121291606",
                "17874544"
              ]
            }
          ]
        }
      }
    ]
```

`operation_type` 必须设置为 `Delete`，系统会根据在将用户添加到受众时所使用的任意键来匹配用户。比如，如果某个用户是通过 `email` 和 `twitter_id` 添加到受众中的，那么之后可以使用这些键中的任意一个来移除该用户，即可以使用 `email` 或 `twitter_id`，或两者同时使用。

此外，在同一个请求中既可以向受众添加用户，也可以从受众中移除用户。该 endpoint 支持在单个请求中使用多个 `operation_type`。

<div id="opt-out-users">
  #### 选择退出的用户
</div>

随着全局选择退出端点被弃用，合作伙伴必须对已从任意受众中选择退出的用户执行 `Delete` 操作。可以通过以下几种方式实现：

1. 跟踪每个用户属于哪些受众，并从各个受众中分别移除这些用户。
2. 将该用户从与某个 Ads 账户关联的**所有**受众中移除。

**通用最佳实践**

* 我们强烈建议以接近实时的小批量方式调用此端点，以避免出现处理时间更长的突增队列，从而整体上减少对我们系统的不必要负载。这样也可以确保这些用户更快可用于广告系列定向。
* 一次成功的 API 调用会返回 `success_count` 和 `total_count`，对应请求中接收到的 `user` 对象的数量。
* 此端点具有原子性，也就是说，要么整个请求全部成功；如果出现任何错误，则整个请求都会失败。出现错误响应时，建议 API 使用方先修复错误，然后携带完整负载重试该请求。 
* 如果调用失败，建议合作伙伴在重试时使用[指数退避](https://en.wikipedia.org/wiki/Exponential_backoff)策略。例如，第一次失败后立即重试，第二次失败后在 1 分钟后重试，第三次连续失败后在 5 分钟后重试，依此类推。

  #### POST accounts/:account&#95;id/custom&#95;audiences/:custom&#95;audience&#95;id/users[](#post-accounts-account-id-custom-audiences-custom-audience-id-users "Permalink to this headline")

此端点允许合作伙伴为给定的 `custom_audience_id` 添加、更新和移除用户。该端点也支持为每个用户提供多种用户标识符类型。

请求中 `users` 字段中提供的所有数据，**除** `partner_user_id` 外，必须使用 `SHA256` 进行哈希处理，并进行[规范化](/zh/x-ads-api/audiences#twitter-user-id-normalization)。

**批量请求**

* 当前的最大批量大小为 `2500`（**仅针对该端点**）。批量大小由每个请求中的操作数量（`Update`/`Delete`）决定。例如，在一个数组中包含超过 2500 个操作对象（`{"operation_type": "Update/Delete", [..] }`）会导致错误。
* 此端点可接受的最大请求 POST 正文大小为 `5,000,000` 字节。
* 此端点的速率限制为每 1 分钟窗口 1500 次。
* 所有参数都在请求正文中发送，并且必须使用 `Content-Type` 为 `application/json`。
* 批量请求作为一个整体成功或失败，对于错误和成功的所有 API 响应都会保留与初始请求相同的条目顺序。

**批量响应**

Ads API 返回的响应包含两个字段：`success_count` 和 `total_count`。这两个值必须始终相等，它们表示后端已处理的请求记录数。如果请求正文中发送的记录数 **不** 等于 `success_count` 和 `total_count`，应视为错误情况，需要重试。

**批量错误**

* 请求级错误（例如，超过最大批量大小）会在响应中的 `errors` 对象下返回。
* 条目级错误（例如，缺少必需参数）会在响应中的 `operation_errors` 对象下返回。
* `operation_errors` 中错误的索引对应输入条目中的索引，并附带相应的错误信息。

允许批量创建自定义受众。有关受众的信息，请参阅 [Custom Audiences 概览](/zh/x-ads-api/audiences) 页面。

**注意：** 此批量接口目前处于**封闭测试**阶段，仅对部分广告主开放。在此测试期间，只能创建基于移动自定义受众的 Flexible Audiences。

**批量请求**

* 当前最大批处理大小为 10。
* 所有参数都在请求正文中发送，并且需要将 `Content-Type` 设置为 `application/json`。
* 批量请求会作为一个整体一起失败或一起成功，且所有 API 响应（无论错误还是成功）都会保留与初始请求相同的条目顺序。

**批量响应**

批量 API 响应返回一个有序的条目集合。除此之外，其结构与相应的单条目接口完全相同。

**批量错误**

* 请求级错误（例如，超过最大批处理大小）会在响应中的 `errors` 对象下显示。
