<div id="ads-api-access-new-applications">
  #### **Ads API 访问权限 - 新应用**
</div>

* 使用 Conversion API 的首要前提是拥有[开发者账号](https://developer.x.com/en/portal/dashboard)以及[Ads API 访问权限](https://developer.x.com/en/docs/twitter-ads-api/getting-started)。具体流程详见 [Ads API 入门](https://developer.x.com/en/docs/twitter-ads-api/getting-started)指南。请注意以下事项：

**步骤 1：开发者账号**

* 在申请[开发者账号](https://developer.x.com/en/portal/dashboard)时，请申请我们的某个订阅[套餐](https://developer.x.com/en/portal/products/)，以便即时获批。 
* **注意**：作为最佳实践，我们强烈建议使用公司官方的 X 账号（handle）来创建开发者账号并申请 Ads API 访问权限。如果开发者账号关联的是个人开发者的 handle，则在需要时无法转移这些凭证。最好将其归属在公司账号下以便持续管理，并视需要使用 [Multi-user login](https://business.x.com/en/help/troubleshooting/multi-user-login-faq.html)。否则，至少应确保账号已配置为非默认设置（头图、头像、个人简介和个人简介 URL），并启用双重身份验证（Two-Factor Authentication）。

**步骤 2：Ads API 申请**

* 请确保在提交 Ads API 申请时准备好正确的 **App ID**。App ID 可以在[开发者门户](https://developer.x.com/en/portal/dashboard)的“Projects &amp; Apps”下找到。例如：16489123
* 通过联系您的 X 客户代表来申请 Ads API 访问权限。

<div id="ads-api-access-existing-applications">
  #### **Ads API 访问权限 - 现有应用**
</div>

* 如果您已经拥有一个正在使用的 Ads API 应用，则该应用及其现有访问令牌都可以用于 Conversion API。

<div id="access-tokens">
  #### **访问令牌**
</div>

* 拥有 Ads API 应用的用户账号（handle）所对应的用户访问令牌可以直接在开发者门户中生成和获取。由于该令牌旨在用于你自己的 X 账号（handle），因此称为“个人访问令牌”。有关身份验证和开发者门户的整体信息，请参见[此处](/zh/resources/fundamentals/authentication/overview)。
* 对于除 Ads API 应用所有者账号以外的其他用户账号，其用户访问令牌必须通过三方 OAuth 流程生成。使用三方 OAuth 生成访问令牌的方式包括：
  * [通过 twurl 工具在命令行中使用基于 Web 的授权](https://github.com/twitter/twurl#getting-started)
  * [通过命令行使用基于 PIN 的授权](https://developer.x.com/resources/fundamentals/authentication/pin-based-oauth)
  * [实现三方 OAuth 模式的自定义 Web 流程](https://developer.x.com/resources/fundamentals/authentication/obtaining-user-access-tokens)
* 与 Conversion API 搭配使用的任何用户令牌，都必须属于具有 AD&#95;MANAGER 或 ACCOUNT&#95;ADMIN 访问级别的用户，可通过 [authenticated&#95;user&#95;access](https://developer.x.com/en/docs/twitter-ads-api/campaign-management/api-reference/authenticated-user-access) 端点进行检查。
* **注意：** 令牌本身（按上述方式创建之后）可以共享给没有 AD&#95;MANAGER 或 ACCOUNT&#95;ADMIN 访问级别的用户供其使用。

<div id="creating-the-conversion-api-event">
  ### 创建 Conversion API 事件
</div>

要使用 Conversion API，你需要在 Ads Manager 中创建一个新的转化事件，或使用已经创建并与 X Pixel 搭配使用的现有事件。如果你希望在 Pixel 事件和 Conversion API 事件之间进行去重处理，则需要复用之前为 Pixel 创建的那个现有事件。 

<div id="option-1-using-an-existing-conversion-event-in-ads-manager">
  ##### 选项 1：在 Ads Manager 中使用现有的转化事件
</div>

如果你想继续使用当前已经与 X pixel 搭配的现有事件，可以这么做，你需要从该事件中获取 Event ID。若你在同一事件上同时使用 pixel 和 Conversion API，请确保在 Pixel 代码片段和 Conversion API 请求中都使用去重键（在请求中作为 `conversion_id`），以对 Pixel 和 Conversion API 之间的同一事件进行去重。更多信息请参见 d. Testing Events and Deduplication 一节。 

<div id="option-2-creating-a-new-conversion-event-in-ads-manager">
  ##### 选项 2：在 Ads Manager 中创建新的转化事件：
</div>

在创建事件之前，务必先在 Events Manager 中创建一个 Event Source。要验证你的账号中是否已添加 Event Source（X Pixel），请前往 Events Manager，并查看左侧菜单中是否有 X Pixel。 

如果你尚未添加 Event Source，请按照以下步骤创建 Event Source。

1. 访问 ads.x.com
2. 在左上角导航到 **Tools** 区域并点击 **Events Manager**
3. 在右上角选择 **Add event source** 以 **Add an event source**，如果你在左侧侧边栏中尚未添加 X Pixel Event Source
   1. X Pixel Event Source 的 ID 就是你的 **Pixel ID**

现在你已经有了 Event Source 和 Pixel ID。你需要在该 Event Source 内创建一个事件，用于追踪你希望监测的转化事件：

1. 在 X Pixel Event Source 内，选择右侧的 **Add events**
2. 选择通过 **Conversion API** 安装
3. 你会看到该事件的 **Pixel ID** 和 **Event ID**，它们将用于 API 中
   1. 该事件的 ID 就是你的 Event ID
4. 点击 Save，此时你的转化事件已创建完成，可以开始使用

<div id="preparing-identifiers-for-conversion-events">
  ### 为转化事件准备标识符
</div>

目前，至少需要传递一个标识符，例如 Click ID (`twclid`)、电子邮件地址或电话号码。若使用 IP 地址或 User Agent，则必须同时发送第二个标识符，才能正确进行转化匹配。

传递更多标识符可以提高转化匹配率。

|     |     |     |
| :--- | :--- | :--- |
| 客户匹配字段 | 格式 | 是否需要哈希处理？ |
| X Click ID | 由 X 生成（[了解更多](https://business.x.com/en/help/campaign-measurement-and-analytics/conversion-tracking-for-websites.html#faq)） | 否  |
| 电子邮件地址 | 去除首尾空格 | 需要（SHA256） |
| 电话号码 | [E164 标准](https://en.wikipedia.org/wiki/E.164) | 需要（SHA256） |

|     |     |     |
| :--- | :--- | :--- |
| IP 地址 | 去除首尾空格 | 否  |
| User Agent | 去除首尾空格 | 否  |

<div id="1-prepare-x-click-id-identifier">
  #### 1. 准备 X Click ID 标识符
</div>

建议在转化请求中始终包含 Click ID。

当用户导航到目标网站后，如果可用，应从查询字符串参数 twclid 中解析出 Click ID。 

基本 JavaScript 代码示例：

```js
var queryString = document.location.search;
if (queryString.has('twclid') {
  twitterClickID = getParam(queryString, 'twclid');
  // 建议的后续步骤：记录日志，插入到本地存储
}
```

建议执行以下操作： 

1. 当 URL 查询参数中包含 `twclid` 值时，请始终对其进行解析。

2. 将该数据与相关的表单字段或转化事件信息一起存储。

将 Click ID 与转化事件和工作流信息关联起来，可以支持诸如批处理、基于多条网站导航路径分析并创建转化事件的算法，以及批量上传等场景。

Event Source URL 应进行[URL 编码](https://en.wikipedia.org/wiki/Percent-encoding)，用于表示触发该事件的网页。

<div id="2-prepare-email-identifier">
  #### 2. 准备电子邮件标识符
</div>

支持的客户匹配字段可以传递，但应先进行规范化处理，并在需要时进行哈希以保护隐私。

信息必须使用 SHA256 算法进行哈希，且不能加盐。 

例如，对于电子邮件地址 test@x.com，应以哈希后的格式发送给我们：d360d510a224510f373931ce2d6215a799f5a9c1cef221b0149b6b6b50cced62。

<div id="3-prepare-phone-identifier">
  #### 3. 准备电话标识符
</div>

电话号码应按 [E.164 标准](https://en.wikipedia.org/wiki/E.164) 提供，并且必须使用不加盐的 SHA256 进行哈希处理。 

例如，对于美国电话号码：+11234567890，应以哈希后的格式发送给我们：1fa6b8d986d9b9cd01bf36951815158bbde9f520c0567c835dfe34783d0a4231。

<div id="4-prepare-ip-address-identifier">
  #### 4. 准备 IP 地址标识符
</div>

IP 地址必须与另一个标识符（twclid、电子邮件地址、电话号码或 user agent）一起传递。此标识符不需要进行哈希处理。

该值使用点分十进制表示法，由四个用点分隔的数字组成。举例来说，某个用户的 IP 地址可能是 8.25.197.25。

<div id="5-prepare-user-agent-identifier">
  #### 5. 准备 User Agent 标识符
</div>

User Agent 必须与另一个标识符（twclid、电子邮件地址、电话号码或 IP 地址）一同传递。此标识符不需要进行哈希处理。

通过该标识符，服务器可以识别发出请求的用户代理所使用的应用、操作系统、供应商和/或版本。例如，该值可以设置为：Mozilla/5.0 (Macintosh; Intel Mac OS X 10&#95;15&#95;7) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/127.0.0.0 Safari/537.36。

<div id="constructing-the-conversion-event-request">
  ### 构建转化事件请求
</div>

`POST: version/measurement/conversions/:pixel_id`

为特定广告账户发送转化事件。应检查响应状态码是否表示成功（HTTP 200 OK）。建议实现重试机制并进行基础日志记录，以防返回错误码。

有关该端点 URL 和 POST 请求体参数的详细信息，请参阅 [API 参考](https://developer.x.com/en/docs/twitter-ads-api/measurement/api-reference/conversions) 部分。

<div id="rate-limit">
  ### 速率限制
</div>

速率限制为每 15 分钟，每个账号最多 60,000 个事件。

请注意，你的服务器代码可能需要在此次请求调用之外实现逻辑，包括：

1. 对用户行为进行埋点（日志记录），以便能够为每个事件发送正确的转化数据

2. 实现任何必要的逻辑，用于过滤已行使相关隐私选择的用户的转化事件——例如，当他们在广告主网站上选择退出被跟踪或个人信息被出售时

3. 与事件触发器和页面进行集成，以便捕获事件并发送转化数据

<div id="testing-events">
  ### 测试事件
</div>

当你的事件成功接收到转化事件后，在 12 到 24 小时内，Ads Manager 的 Conversion Tracking 页面中，“Single event web tag” 的状态应显示为 TRACKING。通过 Conversion API 发送转化不会影响正在投放中的广告系列。 

你也可以通过以下方式按标签 ID 查看该转化事件的分析结果：

* 通过 Ads Manager 导出数据（[网站转化跟踪分析帮助页面](https://business.x.com/en/help/campaign-measurement-and-analytics/conversion-tracking-for-websites.html)）

* 通过 Ads API 导出数据（segmentation&#95;type=CONVERSION&#95;TAGS） 

<div id="duplication-between-pixel-and-conversion-api">
  ### Pixel 与 Conversion API 之间的去重
</div>

如果你希望在 Pixel 和 Conversion API 请求之间对转化进行去重，我们提供了 `conversion_id` 作为去重键。去重只会在事件级别进行。换句话说，要在 Pixel 和 CAPI 请求之间进行去重，广告主必须在 Pixel 和 CAPI 请求中使用相同的事件，并且使用相同的 `conversion_id`。去重只能针对在 48 小时内接收到的事件进行。

<div id="conversion-tracking-overview">
  ## 转化跟踪（概览）
</div>

<div id="summary">
  ### 摘要
</div>

转化跟踪功能可以帮助你衡量 X 用户在看到并与您在 X 上的广告互动后执行目标操作的人数。它可以让你了解哪些广告活动带来了网站访问、注册和购买等行为。通过这种方式，广告主可以在 X 站外衡量其效果广告的表现，从而更有效率、更具成本效益地获客。

通过使用转化标签，广告主可以跟踪用户转化并将其回溯到在 X 上投放的广告活动。这为他们提供了足够的数据可见性，以便优化广告活动，从而实现其单次获客成本（CPA）目标。

通过转化跟踪，广告主可以衡量多种网站行为。他们可以根据广告活动希望带动的行为，选择一个或多个操作类型：

* 站点访问：用户访问广告主网站上的落地页
* 购买：用户在广告主的网站上完成商品或服务的购买
* 下载：用户从广告主的网站下载文件，如白皮书或软件包
* 注册：用户注册广告主的服务、订阅新闻邮件或电子邮件沟通
* 自定义：统称不属于上述任何类别的自定义行为

X 的转化跟踪为广告主提供了完整的转化归因视角。与某些客户可能用来替代 X 自身转化跟踪能力的第三方衡量系统（例如与第三方跟踪标签配合使用的唯一点击 URL）相比，X 的转化标签能够让广告主跟踪归因于漏斗中部和顶部互动（如帖子展开、转发、点赞、回复和关注）以及展示所带来的转化。

<div id="faq">
  ### 常见问题
</div>

<AccordionGroup>
  <Accordion title="转化跟踪标签是如何工作的？">
    首先，广告主会创建一个转化标签，它是一段由 X 提供的代码片段，并将其放置在自己的网站上。当用户完成指定操作时，该标签就可以对转化进行测量。

    随后，用户会在 X 客户端中看到广告主的广告，从而进入广告主的网站并完成已被标记的操作。如果用户在广告主在标签设置期间指定的归因窗口内完成该操作，则该标签会识别出该用户之前与 X 广告有过互动。然后标签会“触发”，即向 X 的服务器发送通知，以便将这次转化归因到产生该转化的广告上。
  </Accordion>

  <Accordion title="在广告系列设置过程中，用户是否可以选择哪些跟踪像素与该广告系列相关？">
    不可以，我们的产品目前并不支持将特定转化标签绑定到特定广告系列。相反，一旦标签设置完成，系统会自动记录是哪条广告为某个标签带来了转化。
  </Accordion>

  <Accordion title="转化标签的默认归因窗口设置是什么？">
    默认曝光后归因窗口：1 天

    默认互动后归因窗口：14 天

    这些默认值可以在创建转化标签时或标签创建后的任意时间进行更改。互动后归因窗口的可选值为 1、7、14、30、60 和 90 天。曝光后归因窗口的可选值为“无”、1、7、14、30、60 和 90 天。
  </Accordion>

  <Accordion title="有哪些有效的效果导向型（DR）创意和策略，可以高效推动转化？">
    尽管每个客户的目标、情况和策略各不相同，但以下做法在参与转化跟踪 Alpha 或 Beta 测试的客户中已被证明有效：

    **创意：**

    * **优惠**：将折扣、促销或免运费优惠与推广帖子（Promoted Tweet）搭配使用，以提高用户对目标行为的兴趣
    * **抽奖和竞赛**：尤其是对于知名品牌，抽奖和竞赛有助于推动转化
    * **帖子文案实验**：测试全大写和小写（例如 FREE 与 free 或 NOW 与 now）
    * **截止日期**：设置截止日期以鼓励人们立即采取行动（优惠截至 12 月 12 日！）
    * **添加吸引人的照片**：值得测试在帖子创意中加入视觉上具有吸引力的照片是否有助于提升转化；结果可能因客户的产品或服务而异。

    **定向：**

    * @handle（@账号）定向和兴趣类别定向：帖子文案与 @handle 与帖子目标受众高度匹配，有助于推动转化
    * 使用小众但高搜索量的关键词：在演唱会相关场景中，使用与艺人/音乐人相关的关键词（例如他们的名字）被证明是有效的
    * 定向受众（Tailored Audiences）：同时使用 TA 网页和转化跟踪的客户，相比使用其他定向方式的对照组，获得了更低的每次行动成本（CPA）

    广告系列细分越精细，你获得的转化报告结果就越具有可操作性。比如，优化包含 4 个关键词的广告系列，远比优化包含 50 个关键词的广告系列要容易得多。
  </Accordion>
</AccordionGroup>

<div id="error-handling-and-explanation">
  ### 错误处理与说明
</div>

单个请求只有在其包含的所有转化均无错误时才会成功。如果某个单独转化中存在任何错误，端点将返回一个包含所有适用错误的列表。

<div id="x-ads-api-error-codes-overview">
  ### X Ads API 错误代码概览
</div>

以下是 Ads API 错误代码的完整列表：

[https://developer.x.com/en/docs/twitter-ads-api/response-codes](https://developer.x.com/en/docs/twitter-ads-api/response-codes)

成功的 Conversion API 响应会以 200 系列 HTTP 状态码表示，并返回一个 JSON 格式的响应体，其中包含所请求的对象。

<div id="when-there-is-a-500-series-http-code-its-related-to-a-server-issue-instead-of-your-request-or-account-set-up-please-check-the-x-api-status-page-or-the-developer-community-forum-in-case-others-are-having-similar-issues">
  #### **当返回 500 系列 HTTP 状态码时**，通常是服务器端问题，而不是你的请求或账号设置导致的。请查看 [X API 状态页](https://api.twitterstat.us/) 或 [开发者社区论坛](https://devcommunity.x.com/)，以确认是否有其他人遇到类似问题。
</div>

<div id="when-there-is-a-400-series-http-code-the-common-cases-are">
  #### 当出现 4xx HTTP 状态码时，常见情况包括
</div>

* 400 Bad Request（请求不符合规范）

* 401 Unauthorized（认证问题）

* 403 Forbidden（与该开发者账号关联的 API 访问问题）

* 404 Not Found（该接口的 URL 或参数可能不正确）

<div id="401-unauthorized">
  ### 401 未授权
</div>

**原因**：缺少或不正确的身份验证凭据 

**解决方案**：按照[设置文档](https://developer.x.com/en/docs/twitter-ads-api/measurement/guides/conversion-api)中的身份验证步骤操作，使用以下三种身份验证方法之一：

对于非 Ads API 应用所有者的用户账号，其用户访问令牌必须通过三方 OAuth 流程生成。使用三方 OAuth 生成访问令牌的方式包括：

* [通过 twurl 实用工具在命令行中进行基于 Web 的授权](https://github.com/twitter/twurl#getting-started)

* [命令行中的基于 PIN 的授权](https://developer.x.com/resources/fundamentals/authentication/pin-based-oauth)

* [实现三方 OAuth 模式的自定义 Web 流程](https://developer.x.com/resources/fundamentals/authentication/obtaining-user-access-tokens)

用于 Conversion API 的任何用户令牌，其对应用户必须具备 AD&#95;MANAGER 或 ACCOUNT&#95;ADMIN 访问级别*，可通过 [authenticated&#95;user&#95;access](https://developer.x.com/en/docs/twitter-ads-api/campaign-management/api-reference/authenticated-user-access) 端点进行检查。

`POST version/measurement/conversions/:pixel_id`

为单个 Event Tag ID 发送网站转化事件。

应检查响应状态码以确认请求是否成功（HTTP 200 OK）。建议在返回错误码时，配备重试机制和基本日志记录。

速率限制为每个账户每 15 分钟最多 100,000 次请求（每个请求允许最多 500 个事件）。
