<div id="overview">
  ## 概览
</div>

创意素材是指在广告活动中可以推广的任何实体。帖子可以包含文本、图片、GIF、视频或卡片。卡片可以包含图片或视频。

图片、GIF 或视频类创意素材可以通过 [POST media/upload](/zh/x-api/media/upload-media#post-media-upload) 上传——这是一个仅支持图片的简单上传端点——或通过 POST media/upload（分片）端点上传。这些可以添加到

卡片：

* POST accounts/:account&#95;id/cards
  帖子：
* POST accounts/:account&#95;id/tweets - 要将卡片添加到帖子，请使用 card&#95;uri 参数。
  定时帖子：
* POST accounts/:account&#95;id/scheduled&#95;tweets

有关卡片的更多详细信息，请参阅 Cards 页面。Promoted Video 页面提供了将视频与卡片或帖子关联的详细信息。

<div id="cards">
  ### 卡片
</div>

Ads API 支持多种卡片类型，可用于帖子中，并可在广告系列中进行推广。**注意**：一旦作为帖子发布，卡片详情将会公开可见。这可能包括持有该卡片的用户的信息。

<div id="image">
  #### 图片
</div>

以下图片规范适用于 [Cards](/zh/x-ads-api/creatives#cards-2) 中使用的素材。图片大小必须不超过 3MB，且宽度至少为 800 像素。此外，我们支持以下宽高比：

* Website：1:1 和 1.91:1
* Image App Download：1:1 和 1.91:1
* Poll：1.91:1
* Image Conversation：1.91:1
* Image Direct Message：1.91:1

我们支持以下图片格式：.bmp、.jpeg 和 .png。

<div id="video">
  #### 视频
</div>

以下视频规格适用于在 [Cards](/zh/x-ads-api/creatives#cards-2) 中使用的素材。我们支持以下宽高比（宽:高）。

* 视频网站：16:9 和 1:1
* 视频应用下载：16:9 和 1:1
* 投票：16:9
* 视频会话：16:9
* 视频私信：16:9

<div id="promoted-video">
  ### 推广视频
</div>

本文档简要概述了通过 Ads API 上传和推广视频的流程。

Ads API 支持在 [帖子](/zh/x-ads-api/creatives#tweets-2) 中以及以下类型的卡片中使用推广视频：

* [视频网站卡片](https://devcommunity.x.com/t/ads-api-version-11/168814)
* [视频应用下载卡片](https://devcommunity.x.com/t/ads-api-version-11/168814)
* [视频会话卡片](/zh/x-ads-api/creatives#video-conversation-cards)

首先，使用 [POST media/upload (chunked)](https://developer.x.com/en/docs/media/upload-media/api-reference/post-media-upload-init.html) 端点上传视频。然后使用 `media_id`，通过 POST accounts/:account&#95;id/videos 端点将该视频与广告账户关联。视频的 `id`，有时称为 `media_key`，将在后续请求中使用。它是一个以整数开头、后跟下划线并以一个 long 型数值结尾的字符串。例如：`13_875943225764098048`。

<div id="promoted-video-in-tweets">
  #### 帖子中的推广视频
</div>

要创建一个帖子，请使用 [POST accounts/:account&#95;id/tweet](/zh/x-ads-api/creatives#get-accounts-account-id-tweets) 端点，并提供该视频的 `id`。在此步骤中，你还可以提供视频标题、描述和号召性用语（CTA）。这些信息会展示给用户查看。

<div id="promoted-video-in-cards">
  #### 卡片中的推广视频
</div>

“Video App Download” 卡片和 “Video Conversation” 卡片支持添加一张封面图片。使用 [POST media/upload](https://developer.x.com/en/docs/media/upload-media/api-reference/post-media-upload.html) 端点上传一张图片，以便在这些卡片中使用。

使用以下任一端点创建卡片：

* [POST accounts/:account&#95;id/cards/video&#95;website](https://devcommunity.x.com/t/ads-api-version-11/168814)
* [POST accounts/:account&#95;id/cards/video&#95;app&#95;download](https://devcommunity.x.com/t/ads-api-version-11/168814)
* [POST accounts/:account&#95;id/cards/video&#95;conversation](https://devcommunity.x.com/t/ads-api-version-11/168814)

同时传入视频的 `id`，以及可选的图片 `media_id`（作为封面图片）。

最后，使用 POST accounts/:account&#95;id/tweet 端点创建帖子。通过 `card_uri` 参数将卡片附加到帖子上。

<div id="general-information">
  #### 常规说明
</div>

有关通过 API 上传视频的详细指南，请参阅 [视频上传指南](https://developer.x.com/en/docs/media/upload-media/uploading-media.html)。

视频也可以作为前贴片素材进行推广。有关详细说明，请参阅[视频观看量前贴片目标指南](/zh/x-ads-api/campaign-management#video-views-preroll-objective)。

* （自 2015-10-22 起）在上传将用于推广内容的视频时，对 [POST media/upload (chunked)](https://developer.x.com/en/docs/media/upload-media/api-reference/post-media-upload-init.html) 端点发起的所有 `INIT` 命令请求，都必须将 `media_category` 参数设置为 `amplify_video`。使用此新参数可确保视频以异步方式预处理，并为在推广内容中使用做好准备。在视频上传完成后，可以使用 `STATUS` 命令检查异步处理是否已完成。
* 当前允许的推广视频最长为 10 分钟，文件大小不超过 500MB。
* 上传的视频应为 mp4 或 mov 格式。
* 上传视频通常会较快处理完毕，但处理时间可能因视频时长和文件大小而异。
* 上传的海报图像应为 png 或 jpg 格式。没有宽高比或尺寸要求，但海报图像会被调整以适配视频播放器。

<div id="scheduled-tweets">
  ### 定时帖子
</div>

<div id="introduction">
  #### 简介
</div>

预定帖子功能允许广告主或用户创建一个帖子，并安排在将来的某个时间发布。除了能够创建和管理这些帖子之外，API 还支持将这些帖子与一个 line item 关联，以便在帖子发布后进行推广。这样，广告主就可以预先创建原生帖子，并在任何关键活动开始前提前规划他们的活动创意。例如，将一个帖子创意预先排期为在新产品发布公告时立即上线。

Scheduled Tweets API 端点提供的全部功能如下：

* 创建、修改和查看新创建的预定帖子
* 将一个预定帖子与 line item 关联
* 查询和管理现有的预定帖子
* 一旦预定帖子上线，检索已上线帖子的 `id`

* [GET accounts/:account&#95;id/scheduled&#95;promoted&#95;tweets/:scheduled&#95;promoted&#95;tweet&#95;id](/zh/x-ads-api/campaign-management#get-accounts-account-id-scheduled-promoted-tweets-scheduled-promoted-tweet-id)（使用其 `id` 查找预定推广帖子）

  #### 预定帖子视图
</div>

* [GET accounts/:account&#95;id/scheduled&#95;tweets/preview/:scheduled&#95;tweet&#95;id](/zh/x-ads-api/creatives#scheduled-tweets-2)（查看现有预定帖子）

由于预定帖子与“已发布”帖子是彼此独立的实体，在创建或编辑这些帖子时会执行两套不同的验证。第一套验证规则会在创建预定帖子步骤中运行，具体包括：

<div id="scheduled-tweet-create">
  #### 创建计划帖子：
</div>

* 验证已认证用户是否有权限为指定 @handle 创建自然帖子。要创建仅推广帖子，已认证用户必须是具备[帖子创作权限](https://developer.x.com/content/developer-twitter/en/docs/ads/campaign-management/api-reference/authenticated-user-access#get-accounts-account-id-authenticated-user-access)的账号用户。
* 验证在距离 `scheduled_at` 时间点前后 15 分钟的时间窗口内，计划创建的帖子不超过 30 条。`SCHEDULED_TWEET_LIMIT_EXCEEDED` 错误消息表示在同一个未来的 15 分钟时间段内安排了过多的计划帖子。广告主需要删除一条已有的计划帖子，或将 `scheduled_at` 时间调整得更早或更晚。

<div id="scheduled-tweet-goes-live">
  #### 定时帖子正式发布：
</div>

* 这些验证规则会在 scheduled&#95;at 指定的时间运行，与通过 API 创建常规帖子时应用的规则完全相同。举例来说，如果一个定时帖子同时包含图片和 GIF，则该定时帖子不会发布，且 scheduled&#95;status 会被设置为 FAILED。

<div id="workflow">
  #### 工作流程
</div>

**创建新的定时帖子**

可以使用 [POST accounts/:account&#95;id/scheduled&#95;tweets](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/scheduled-tweets#post-accounts-account-id-scheduled-tweets) 端点来创建新的定时帖子。该端点要求提供以下参数：`scheduled_at` 时间，以及在帖子中未包含任何媒体实体时的帖子 `text`。此外，此端点还提供了一些附加选项，允许你通过 `as_user_id` 参数代表另一个 @handle 创建定时帖子，并可以添加卡片（`card_uri`）和媒体（`media_ids`）。注意，一个帖子只能包含同一类型的实体，即视频、GIF 或图像三者之一。`nullcast` 参数用于控制该帖子是否为 “Promoted-Only” 帖子。所有新创建的定时帖子默认都是 “Promoted-Only”（`nullcast=true`）。如果 `nullcast=false`，则会创建自然（非推广）的定时帖子。

成功创建定时帖子后，响应中会包含一个 `id` 字段，它表示该定时帖子本身的唯一标识符。除了此字段，还会返回另一个名为 `tweet_id` 的字段。该字段初始为 `null`，但一旦帖子发布，该字段就会被填充为该已发布帖子的 ID。

```
twurl -H 'ads-api.x.com' -X POST "/6/accounts/:account_id/scheduled_tweets" -d 'scheduled_at=2017-12-24T23:59:00Z&text=Happy Holidays!'
```

这会创建以下定时帖子：


一旦该预定发布的帖子上线，`tweet_id` 字段就会被填入该“上线”帖子的 ID。

**将预定发布帖子关联到广告单元（line item）**

虽然预定发布帖子可用于创建自然帖子（Organic Tweets），我们也允许合作伙伴创建“仅推广”（Promoted-Only，`nullcast=true`）帖子，这两种帖子都可以关联到一个广告单元。为支持该用例，我们还提供了一个 [POST accounts/:account&#95;id/scheduled&#95;promoted&#95;tweets](https://developer.x.com/content/developer-twitter/en/docs/ads/campaign-management/api-reference/scheduled-promoted-tweets#post-accounts-account-id-scheduled-promoted-tweets) 端点。此端点在一次 API 调用中只允许将单个预定推广帖子关联到一个广告单元。若要将多个预定发布帖子关联到同一个广告单元，则需要进行多次 API 调用。

请注意，无法修改现有的预定推广帖子。

```
twurl -H 'ads-api.x.com' -X POST "/6/accounts/:account_id/scheduled_promoted_tweets" -d 'line_item_id=a44qc&scheduled_tweet_id=917507899668099072'
```

```json
{
  "data": {
    "line_item_id": "a44qc",
    "id": "26576",
    "created_at": "2017-10-09T22:24:16Z",
    "updated_at": "2017-10-09T22:24:16Z",
    "scheduled_tweet_id": "917507899668099072",
    "tweet_id": null,
    "deleted": false
  },
  "request": {
    "params": {
      "line_item_id": "a44qc",
      "scheduled_tweet_id": 917507899668099100,
      "account_id": "aaaaa"
    }
  }
}
```

此端点仅用于在指定的 Scheduled Tweet 与某个 line item 之间创建关联。一旦 campaign/line item 的投放时间进入当前时间范围，该 line item 将自动开始投放对应的“live” Tweet。虽然在此步骤中我们会验证 Scheduled Tweet 是否处于 `SCHEDULED` 状态，以及给定的 Scheduled Tweet 是否对指定的 objective 有效，但不会执行其他验证。任何针对 line item 和 Scheduled Tweet 的其余验证规则都会在 Tweet 变为“live”时运行。

为确保 campaign 的投放不出现问题，建议将 Scheduled Tweet 的 `scheduled_at` 时间设置在 campaign/line item 投放时间开始之前。

例如，如果 Scheduled Tweet 被设置为在 campaign 开始日期之后才上线（并且仅有一个 Tweet 关联到一个 line item），那么 campaign 的状态会是 `ACTIVE`，但由于 Scheduled Tweet 尚未上线，将没有可用于投放的创意。

**Scheduled Tweet 管理**

其余这一组端点允许 API 使用方管理其所有的 Scheduled Tweet 和 Scheduled Promoted Tweet。可以使用这些 API 返回所有 Scheduled Tweet 的列表（可选择按给定状态过滤），以及通过 `id` 查询指定的 Scheduled Tweet。

<div id="what-happens-when-a-scheduled-tweet-goes-live">
  #### 定时帖子发布时会发生什么情况？
</div>

当某个定时帖子即将发布时，换句话说在 `scheduled_at` 时间点，将进行以下更新：

* 会创建“上线”的帖子，但这可能会有最长 1 秒的延迟
* `tweet_id` 会被添加到以下实体：
* 定时帖子
* 推广定时帖子
* 会创建一个新的推广帖子实体

<div id="best-practices">
  #### 最佳实践
</div>

在创建或推广计划发布的帖子时，建议遵循以下最佳实践：

* 在创建计划发布的帖子时，确保该帖子是有效的（例如，一个帖子只能包含图片、视频或 GIF 中的一种，而不能同时包含多种）
* 确保广告投放时间段（即 `start_time` 和 `end_time`）与计划发布帖子的 `scheduled_at` 时间保持一致
* 计划发布的帖子不应被安排在超过一年之后（365 天）
* 目前计划发布的帖子不支持帖子预览功能（即在创建之前预览计划发布帖子的功能）

### 媒体库

<div id="introduction">
  #### 简介
</div>

Media Library 端点提供了管理 X Ads 账户中图片、GIF 和视频的功能。库中的媒体素材可用于帖子及卡片的创建，也可以在多个[创意](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/overview)中重复使用，从而无需多次上传相同的素材。

<div id="api-endpoints">
  #### API 端点
</div>

* [POST media/upload](https://developer.x.com/content/developer-twitter/en/docs/media/upload-media/api-reference/post-media-upload#post-media-upload)（上传媒体）或 [POST media/upload (chunked)](https://developer.x.com/content/developer-twitter/en/docs/media/upload-media/api-reference/post-media-upload-init#post-media-upload-init)（分块上传媒体）
* [POST accounts/:account&#95;id/media&#95;library](https://developer.x.com/x-ads-api/creatives#get-accounts-account-id-media-library#post-accounts-account-id-media-library)（将媒体添加到媒体库）

<div id="adding-to-the-library">
  #### 添加到媒体库
</div>

将媒体添加到媒体库需要两个步骤。首先，使用 [POST media/upload](https://developer.x.com/content/developer-twitter/en/docs/media/upload-media/api-reference/post-media-upload#post-media-upload) 端点或 [POST media/upload (chunked)](https://developer.x.com/content/developer-twitter/en/docs/media/upload-media/api-reference/post-media-upload-init#post-media-upload-init) 这一组端点上传素材。（有关多部分上传流程的详细信息，请参阅［分片媒体上传］([https://developer.x.com/content/developer-twitter/en/docs/media/upload-media/uploading-media/chunked-media-upload](https://developer.x.com/content/developer-twitter/en/docs/media/upload-media/uploading-media/chunked-media-upload)) 指南。）

```
twurl -X POST -H upload.x.com "/1.1/media/upload.json?additional_owners=756201191646691328" --file latte.jpeg --file-field "media"
```

```json

   "size":74194,
```

<div id="request-parameters">
  #### 请求参数
</div>

所有 Media Library 的 POST 请求都需要一个媒体标识符。此值会在上传步骤中返回。使用 `media_id`（如上例所示）时，还必须指定 `media_category`。有四种可能的类别值：AMPLIFY&#95;VIDEO、TWEET&#95;GIF、TWEET&#95;IMAGE 和 TWEET&#95;VIDEO。

可以为 Media Library 中的对象选择性地设置 `name` 和 `file_name` 值。这些属性可帮助用户区分素材库中的不同媒体变体。

对于视频，还可以设置标题和描述。这些值旨在在调用 [POST accounts/:account&#95;id/tweet](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/tweets#post-accounts-account-id-tweet) 端点时，作为 `video_title` 和 `video_description` 请求参数传入。在帖子中，此文本会显示在视频下方。

<div id="attributes">
  #### 属性
</div>

Media Library 正式引入了 `media_key` 概念。它是媒体库中对象的唯一标识符。Media key 是如下格式的字符串值：13&#95;875943225764098048。我们的所有 card 端点都完全支持它们。

此外，Media Library 的响应中包含以字符串形式表示的 `media_id`。该字段会出现在当前尚不接受 media key 的资源中：[Tweets](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/tweets#post-accounts-account-id-tweet)*、[Tweet preview](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/tweets#get-accounts-account-id-tweet-preview)* 和 [Scheduled Tweets](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/scheduled-tweets#post-accounts-account-id-scheduled-tweets)。我们正致力于在所有端点全面支持 media key。

`aspect_ratio` 属性会针对 GIF 和视频返回。它可用于为只接受特定纵横比的 card 过滤可用媒体。

*这些端点支持 `video_id` 参数，它是一个 media key。

<div id="identifying-cards">

<div id="introduction">
  #### 简介
</div>

Card 是一种可自定义的广告格式，会使用媒体内容，并且可以与网站、应用或号召性用语（call to action）关联，以推动特定的用户互动，例如发起私信（Direct Message）。它们可以附加到帖子、计划发布的帖子或帖子草稿中。

Card 可以通过两种方式之一在帖子对象中被引用：通过 Card 的 `card_uri`，或通过其 `preview_url`。下面给出了每种方式的示例值。

| :--- | :--- |
| card://1043282691834048513 | https://cards.x.com/cards/18ce54d4x5t/68w3s |

<div id="identifying-media">
  ### 媒体识别
</div>

<div id="introduction">
  #### 简介
</div>

媒体——图片、GIF 和视频——可以添加到帖子和卡片中。此外，视频可以用作贴片广告素材，图片可以在 [X Audience Platform](https://developer.x.com/en/docs/ads/measurement/overview/twitter-audience-platform) 上推广。本节介绍如何在这些实体中查找媒体引用。

媒体标识符有两种类型：ID 和 key。下面给出了每种类型的示例值。

| **Media ID** | **Media key** |
| :--- | :--- |
| 1029825579531807971 | 13&#95;1029825579531807971 |

媒体 key 是在 ID 前添加一个数字前缀和下划线组成的。

<div id="image">
  #### 图片
</div>

下表显示了当前每个与图片相关的资源在响应中可用的标识符类型，以及对应的属性名称。

| **Resource** | **Identifier** | **Attribute(s)** |
| :--- | :--- | :--- |
| Image cards | None |  |
| 帖子（Tweet） | Both | `entities["media"]["id_str"]`  `entities["media"]["media_key"]` |
| 预定帖子（Scheduled Tweet） | Both | `media_ids` and `media_keys` |
| 帖子草稿（Draft Tweet） | Both | `media_ids` and `media_keys` |
| 账号媒体（Account Media） | None |  |
| 媒体库（Media Library） | Both | `media_id` and `media_key` |

Image cards 和账号媒体中的图片不包含对任何媒体标识符的引用。帖子只包含媒体 ID。预定帖子和帖子草稿同时包含媒体 ID 和 media key。媒体库也会返回这两者。

对于帖子，entities[&quot;media&quot;] 数组中对象里的 id 和 id&#95;str 字段对应媒体 ID。对于包含多张图片的帖子，每个媒体实体的引用只能在 extended&#95;entities[&quot;media&quot;] 中找到。

除了对标识符的引用之外，通常还需要能够访问图片的 URL。

| **Resource** | **Attribute(s)** | **Format** |
| :--- | :--- | :--- |
| Image cards | `image` | .jpg |
| 帖子（Tweet）* | `entities["media"][0]["media_url"]` or `extended_entities["media"][i]["media_url"]` | .jpg |
| 预定帖子（Scheduled Tweet） | None |  |
| 帖子草稿（Draft Tweet） | None |  |
| 账号媒体（Account Media） | `media_url` | .jpg |
| 媒体库（Media Library） | `media_url` | .jpg |

* 该 URL 的具体位置取决于帖子是包含单张图片还是多张图片。

所有 Image cards 都包含一个名为 image 的响应属性，其中包含 X 图片 URL（对于 image app download cards，该属性名称为 wide&#95;app&#95;image）。

对于帖子，media URL 的位置既取决于媒体类型，也取决于所使用的 endpoint。对于只包含一张图片的帖子，URL 可在 entities[&quot;media&quot;][0][&quot;media&#95;url&quot;] 中找到，这对 Ads API 和 Standard API 都适用。而当帖子包含多张图片时，这些 URL 只能在 extended&#95;entities[&quot;media&quot;][i][&quot;media&#95;url&quot;] 中找到，该字段仅在 Standard API 中可用。

<div id="video">
  #### 视频
</div>

下表显示了当前每种与视频相关的资源在响应中可用的标识符类型，以及相应的属性名称。

| **资源** | **标识符** | **属性** |
| :--- | :--- | :--- |
| 视频卡片 | 可能为任一类型 | `video_content_id` |
| 视频投票卡片 | 无 |  |
| 帖子 | 两者都有 | `entities["media"]["id_str"]`  `entities["media"]["media_key"]` |
| 计划帖子 | 两者都有 | `media_ids` and `media_keys` |
| 草稿帖子 | 两者都有 | `media_ids` and `media_keys` |
| 账号媒体 | 媒体 key | `video_id` |
| 媒体库 | 两者都有 | `media_id` and `media_key` |

虽然视频卡片（带视频的投票卡片除外）包含一个 video&#95;content&#95;id 响应属性，但其返回值的类型并不一致。在某些情况下它是媒体 ID，在其他情况下则是媒体 key。

下表显示了如何访问视频的 URL。

| **资源** | **属性** | **格式** |
| :--- | :--- | :--- |
| 视频卡片 | `video_url` and `video_hls_url` | .vmap  .m3u8 |
| 带视频的帖子 | `extended_entities["media"][i]["video_info"]["variants"][j]["url"]` | .mp4 |
| 计划帖子 | 无 |  |
| 草稿帖子 | 无 |  |
| 账号媒体 | 无 |  |
| 媒体库 | `media_url` | .mp4 |

视频卡片在响应中包含 video&#95;url 和 video&#95;hls&#95;url 属性，其对应的 URL 分别为 .vmap 和 .m3u8。

<div id="media-library">
  #### 媒体库
</div>

有时你需要获取某个媒体资源的额外信息。比如在视频卡片的用例中，需要获取 mp4 URL，而不是 vmap URL。这类信息可以在媒体库中获得。关于可用信息的详细说明，请参阅我们的[媒体库指南](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/guides/media-library)。广告账户的 FULL promotable user 所属的大多数素材都可以在媒体库中找到，但也有一些例外。

**获取媒体**

如上所述，图片卡片不包含对 media id 或 media key 的引用。因此，无法通过媒体库获取这些卡片的素材。[Account Media](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/account-media#account-media) 中的图片也是如此。

在创建视频卡片之前，要求其视频素材已先作为媒体库（或其前身 Videos 资源）的一部分存在。因此，这些素材始终可以在媒体库中检索到。Account Media 中的 PREROLL 素材也是如此。

最后，帖子中的媒体始终可以保证存在于媒体库中。

下表总结了在媒体库中可以检索到哪些素材，并考虑了资源响应中是否包含可用于查找的标识符。

| **资源** | **是否在媒体库中** |
| :--- | :--- |
| 图片卡片 | 否 |
| 视频卡片 | 是* |
| 帖子（任何媒体）** | 是 |
| 已排期的帖子 | 是 |
| 帖子草稿 | 是 |
| Account Media 图片 | 否 |
| Account Media 视频（`PREROLL`） | 是 |

* 适用于 `video_content_id` 为 media key 的卡片。当该值为 media id 时，素材仍然存在于媒体库中，但检索它需要在其前面加上数字前缀和下划线。\
** 帖子只返回 media id。虽然可以保证素材存在于媒体库中，但获取它需要在其前面加上数字前缀和下划线。

**与 Account Media 的交互**

在以下两种情况下，添加到媒体库中的媒体素材会自动添加到 Account Media 资源中。

* 当一个 AMPLIFY&#95;VIDEO 素材被添加到媒体库时，它会自动作为 PREROLL 创意类型的 Account Media 素材被添加。
* 当具有特定尺寸的图片（参见我们[枚举页面](https://developer.x.com/content/developer-twitter/en/docs/ads/general/overview/enums)中的“Creative Types”）被添加到媒体库时，它们会自动作为 Account Media 素材被添加。其创意类型（例如 INTERSTITIAL）取决于图片尺寸。

<div id="tweets">
  ### 帖子
</div>

<div id="introduction">
  #### 简介
</div>

X Ads API 支持三种类型的帖子：已发布、计划发布和草稿。

<div id="nullcasted-tweets">
  #### 未公开投放的帖子（Nullcasted Tweets）
</div>

帖子可以是未公开投放的（又称“仅推广”）或自然帖。未公开投放的帖子在发布后不会显示在用户的公开时间线中，但它们仍然是公开可见的。自然帖则会投放给用户的关注者，并显示在用户的公开时间线中。

**创建帖子**

这三个创建帖子的端点都支持一个布尔值类型的 `nullcast` 参数，为 API 使用者提供创建未公开投放帖子或自然帖的选项。未公开投放的帖子可以由该用户本人创建，也可以由任何被授权代表该用户创建帖子的主体创建。自然帖只能由[具有完整推广资格的用户](https://developer.x.com/content/developer-twitter/en/docs/tutorials/ads-api-hierarchy-terminology#promotable-users)创建。

**更新帖子**

对于计划发布和草稿状态的帖子，可以更新其 `nullcast` 属性。对于计划发布的帖子，可以在该帖子的 `scheduled_at` 时间之前进行编辑。草稿帖子可以无限期编辑。但一旦帖子发布，就无法再将其从未公开投放修改为自然帖，或反之亦然。

<div id="promoting-tweets">
  #### 推广帖子
</div>

只有已发布和已排期的帖子可以被推广。这些帖子可以是 nullcast 帖子或自然帖；对此没有任何限制。广告主可以推广自己的帖子，也可以在获得许可的情况下推广其他用户的帖子。（参见：[Promoting another user&#39;s Tweets](https://developer.x.com/content/developer-twitter/en/docs/tutorials/promoting-another-users-content) 了解更多信息。）

在单个广告系列中可以推广多个帖子。同样，一个帖子也可以在一个或多个广告系列中被推广。

要推广已发布的帖子，请使用 [POST accounts/:account&#95;id/promoted&#95;tweets 端点](https://developer.x.com/content/developer-twitter/en/docs/ads/campaign-management/api-reference/promoted-tweets#post-accounts-account-id-promoted-tweets)。这会将已发布的帖子关联到一个 line item。要推广已排期的帖子，请使用 [POST accounts/:account&#95;id/scheduled&#95;promoted&#95;tweets 端点](https://developer.x.com/content/developer-twitter/en/docs/ads/campaign-management/api-reference/scheduled-promoted-tweets#post-accounts-account-id-scheduled-promoted-tweets)。

<div id="tweet-ids">
  #### 帖子 ID
</div>

已发布、已安排和草稿帖子的 ID 都是[数值型](https://developer.x.com/content/developer-twitter/en/docs/basics/twitter-ids)，它们是 64 位无符号整数。例如，下面这个已发布帖子的 ID 是 1166476031668015104。

<div id="carousels">
  ### 轮播
</div>

<div id="introduction">
  #### 简介
</div>

X Ads API 支持创建和检索视频轮播和图片轮播。轮播是一种卡片类型，可以包含 2 到 6 个媒体资源。轮播卡片可以将用户引导至一个网站，或鼓励他们安装移动应用。关于轮播、其优势、最佳实践以及常见问题的更多信息，请参阅我们的 [Carousel Ads on X](https://business.x.com/en/advertising/carousels.html) 页面。

与其他卡片类型一样，轮播可以用于帖子中，而这些帖子随后可以被推广。工作流程与您已经熟悉的流程相同：

1. 上传媒体
2. 创建卡片
3. 创建帖子
4. 推广帖子

唯一的区别在于卡片的创建方式。其他卡片创建请求接受查询参数，而轮播卡片的创建请求只接受 JSON POST 请求体。

<div id="endpoints">
  #### 端点
</div>

Ads API 支持创建和检索轮播卡片。

要创建任意类型的轮播卡片，请使用 [POST accounts/:account&#95;id/cards](https://developer.x.com/en/docs/twitter-ads-api/creatives/api-reference/cards#post-accounts-account-id-cards) 端点。要检索轮播卡片，请使用 [GET accounts/:account&#95;id/cards](https://developer.x.com/en/docs/twitter-ads-api/creatives/api-reference/cards#get-accounts-account-id-cards) 端点。

<div id="json-post-body">
  #### JSON POST Body
</div>

轮播是通过两个组件来创建的。第一个组件用于指定将要使用的媒体资源，第二个组件用于指定网站或应用的信息。

具体来说，轮播卡片按以下组件顺序创建：

* 一个 `SWIPEABLE_MEDIA` 组件，用于接收一个 media key 数组
* *下列组件中的一个*：
* 一个 `DETAILS` 组件，用于指定网站信息
* 一个 `BUTTON` 组件，用于指定应用信息

`SWIPEABLE_MEDIA` 组件必须包含一个 `media_keys` 数组，你可以在其中指定 2 到 6 张图片或视频。传入 media key 的顺序决定它们被渲染的顺序。

```
{
  "type": "SWIPEABLE_MEDIA",
  "media_keys": [
    "13_1089771253848666112",
    "13_1191948012077092867"
  ]
}
```

提醒一下，你可以通过向 [GET accounts/:account&#95;id/media&#95;library](https://developer.x.com/en/docs/twitter-ads-api/creatives/api-reference/media-library#get-accounts-account-id-media-library) 端点发起请求来获取 media key。

第二个组件对象的组成取决于你是希望将用户引导至网站，还是鼓励他们安装应用。下表对这两个选项进行了汇总。（**注意**：下表列出的所有键都是必需的。）

|        | **网站**                                                             | **应用**                                            |
| :----- | :----------------------------------------------------------------- | :------------------------------------------------ |
| 指定组件类型 | `"type": "DETAILS"`                                                | `"type": "BUTTON"`                                |
| 标题/标签  | `"title": "X"`                                                     | `"label": { "type": "ENUM", "value": "INSTALL" }` |
| 目标     | `"destination": { "type": "WEBSITE", "url": "https://www.x.com" }` | `"destination": { "type": "APP", ... }`           |

综合以上内容，下面展示了一个网站轮播的示例 JSON POST 请求体。

```
{
  "name": "website carousel",
  "components": [
    {
      "type": "SWIPEABLE_MEDIA",
      "media_keys": [
        "13_1089771253848666112",
        "13_1191948012077092867"
      ]
    },
    {
      "type": "DETAILS",
      "title": "X",
      "destination": {
        "type": "WEBSITE",
        "url": "https://www.x.com"
      }
    }
  ]
}
```

`BUTTON` 组件中的应用目标对象需要指定国家代码以及至少一个应用标识符。它们还可以选择性地包含深度链接。有关这些字段的说明，请参阅 [参考文档](https://developer.x.com/en/docs/twitter-ads-api/creatives/api-reference/cards#app-destination)。

综合以上内容，下面展示了一个应用轮播 JSON POST 请求体示例。

```
{
  "name": "app carousel",
  "components": [
    {
      "type": "SWIPEABLE_MEDIA",
      "media_keys": [
        "13_1089771253848666112",
        "13_1191948012077092867"
      ]
    },
    {
      "type": "BUTTON",
      "label": {
        "type": "ENUM",
        "value": "INSTALL"
      },
      "destination": {
        "type": "APP",
        "country_code": "US",
        "googleplay_app_id": "com.twitter.android",
        "iphone_app_id": "333903271"
      }
    }
  ]
}
```

<div id="creative-metadata-tagging">
  ### 创意元数据标注
</div>

<div id="introduction">
  #### 简介
</div>

本指南面向创意合作伙伴、代理商和创意开发者，用于为在 X 广告活动中使用的素材添加标签，从而更好地了解单个素材的价值和表现。

**注意：** 媒体素材只能由创建该媒体素材的合作伙伴或开发者进行标记。如果媒体素材的使用者并未创建该媒体素材，则不要对其实施元数据标记。

Creative Metadata Tagging 功能可为由 Creative Partners 创建的图片和视频提供归因，无论该素材在 X 上由何处、由何种实体上传。为在创意素材与创意合作伙伴之间建立关联，本功能采用 [XMP](https://exiftool.org/TagNames/XMP.html) 标准。

<div id="tagging-creative-assets">
  #### 为创意素材添加标签
</div>

下表展示了当前在每个与图片相关的资源响应中可用的标识符类型，以及对应的属性名称。要为创意素材添加标签，需要使用标注工具。推荐使用 [ExifTool](https://exiftool.org/)，它是一个跨平台的 Perl 库以及一个用于读取、写入和编辑元信息的[命令行应用程序](https://exiftool.org/exiftool_pod.html)。查看所有支持的[文件类型](https://exiftool.org/#supported)。

请按照提供的[安装 ExifTool 的说明](https://exiftool.org/install.html)进行操作。也可以通过 [Homebrew](https://formulae.brew.sh/) 提供的软件包来简化安装过程，它为 macOS 和 Linux 提供了 [exiftool 安装命令](https://formulae.brew.sh/formula/exiftool)。在命令行中输入 `exiftool -ver`，如果输出工具的版本号，则表示安装成功。有关 ExifTool 命令参数的更多信息，请参见 [ExifTool 文档](https://exiftool.org/)。

创意合作伙伴可以在新的或现有的创意素材上，将其 X app&#95;id 赋值给 contributor XMP 标签和 date 标签来设置元数据标签。当[上传媒体](https://developer.x.com/en/docs/twitter-api/v1/media/upload-media/overview)时，这些创意素材将遵循现有的大小限制。 

**注意：** X 对 contributor XMP 标签的使用可确保元数据仅记录专用于 X 上广告活动的取值。

`exiftool -contributor="<YOUR APP ID>" -creative_file.jpg`

<div id="post-accountsaccount_idcards">
  #### POST accounts/:account&#95;id/cards
</div>

创建一个与指定账户关联的新卡片（card）。

创建卡片的请求只接受 JSON 格式的 POST 请求体。`Content-Type` 必须设置为 `application/json`。

有关详细的用法示例，请参阅我们的[轮播卡片指南](https://developer.x.com/en/docs/twitter-ads-api/creatives/guides/carousels)。

<div id="resource-url">
  #### 资源 URL
</div>

`https://ads-api.x.com/12/accounts/:account_id/cards`

<div id="parameters">
  #### 参数
</div>

JSON POST 请求体必须包含卡片的 `name` 字段和一个 `components` 数组。组件以对象的形式表示，用于描述该卡片面向广告主的属性。

下面的示例展示了有效负载的一般结构（但其中包含无效信息）。

```
{
  "name": "some name",
  "components": [
    {
      "type": "TYPE_ENUM",
      "key": "value"
    }
  ]
}
```

下面是关于组件的更多信息。

| Name                              | Description                                                                                                                                                                                                   |
| :-------------------------------- | :------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| account&#95;id   *required*       | 所使用账号的标识符。出现在资源路径中，并且通常是除 [GET accounts](https://developer.x.com/en/docs/ads/campaign-management/api-reference/accounts#get-accounts) 之外所有 Advertiser API 请求必需的参数。指定的账号必须与已认证用户关联。 类型：string 示例：`18ce54d4x5t` |
| name   *required*                 | 卡片的名称。最大长度：80 个字符。 类型：string 示例：`component based card`                                                                                                                                                        |
| components   *sometimes required* | 描述用于创建卡片的组件。更多信息见下文。不能与 `slides` 一起指定。**注意**：组件的顺序很重要。类型：array of objects                                                                                                                                     |
| slides   *sometimes required*     | 使用这个“数组的数组”来创建 Multi-Destination Carousels。将每张卡片描述为组件的分组。每个 slide 都应当是卡片的完整表示。不能与 `components` 一起指定。**注意**：每个 slide 的顺序很重要。类型：array of array                                                                  |

<div id="components">
  #### 组件
</div>

每个组件都必须包含一个 `type` 字段，用于确定对象的架构。Ads API 支持以下组件类型，并按基于媒体和基于描述的组件进行分组。

* 媒体：
* `MEDIA`：单个视频或图片
* `SWIPEABLE_MEDIA`：2-6 个视频或图片
* 描述：
* `DETAILS`
* `BUTTON`

每个组件都有一组必需字段（除了 `type` 键之外）。这些字段列在下表中。

| 组件 `type`         | 字段                      | 值类型              |
| :---------------- | :---------------------- | :--------------- |
| `MEDIA`           | `media_key`             | string           |
| `SWIPEABLE_MEDIA` | `media_keys`            | array of strings |
| `DETAILS`         | `title`   `destination` | string   object  |
| `BUTTON`          | `label`   `destination` | object   object  |

下面是在 `components` 数组中的一个 `BUTTON` 组件示例（有意省略 `name` 键）。（省略号表示在这些位置还需要提供更多信息。）

```
{
  "components": [
    {
      "type": "BUTTON",
      "label": {
        ...
      },
      "destination": {
        ...
      }
    }
  ]
}
```

指定组件对象的顺序决定了它们从上到下的渲染顺序。Cards 必须使用一个基于媒体的组件，并搭配一个 `DETAILS` 或 `BUTTON` 组件来创建。基于描述的组件会渲染在媒体组件下方，并具有关联的目标地址，可以是 URL 或移动应用。

**Label**

Labels 定义显示在按钮上的文本，因此仅适用于 `BUTTON` 组件。Label 对象有两个必需的键：`type` 和 `value`。`type` 必须设置为 `ENUM`，而 `value` 可以是以下之一：`BOOK`、`CONNECT`、`INSTALL`、`OPEN`、`ORDER`、`PLAY` 或 `SHOP`。

在前一个示例的基础上，下面展示了 `BUTTON` 组件中的 `label` 对象。

```
{
  "components": [
    {
      "type": "BUTTON",
      "label": {
        "type": "ENUM",
        "value": "INSTALL"
      },
      "destination": {
        ...
      }
    }
  ]
}
```

**目标位置**

目标位置是广告主希望将用户引导至的地方。它们在 `DETAILS` 或 `BUTTON` 组件中始终为必填项。目标位置有两种类型：`WEBSITE` 或 `APP`。

**注意**：网站目标位置只能与 `DETAILS` 组件一起使用，应用目标位置只能与 `BUTTON` 组件一起使用。

**网站目标位置**

| Name              | Description                                                                 |
| :---------------- | :-------------------------------------------------------------------------- |

<div id="get-accountsaccount_idtweets">
  #### GET accounts/:account&#95;id/tweets
</div>

检索该账户的完全可推广用户（默认）或 `user_id` 参数中指定用户的帖子详情。该用户可以是账户下的任意[可推广用户](https://developer.x.com/en/docs/ads/campaign-management/api-reference/promotable-users)。

<div id="post-accountsaccount_idtweet">
  #### POST accounts/:account&#95;id/tweet
</div>

为账号的完整可推广用户（默认）或在 `as_user_id` 参数中指定的用户创建一条帖子。支持空投放（nullcast，默认）和自然帖子的创建。空投放帖子不会出现在公共时间线中，也不会展示给关注者。这两种类型都可以用于广告系列。

如果已认证用户不是此账号上的 `FULL` 可推广用户，请先调用 [GET accounts/:account&#95;id/authenticated&#95;user&#95;access](https://developer.x.com/en/docs/ads/campaign-management/api-reference/authenticated-user-access#get-accounts-account-id-authenticated-user-access) 端点，以确定其是否拥有代表该用户发帖的权限。权限值为 `TWEET_COMPOSER` 表明该用户可以使用此端点代表 `FULL` [可推广用户](https://developer.x.com/en/docs/ads/campaign-management/api-reference/promotable-users#get-accounts-account-id-promotable-users) 创建空投放帖子。

当使用 [upload.x.com 端点](https://developer.x.com/rest/reference/post/media/upload) 上传媒体时，请为 `additional_owners` 参数传入与调用此端点时 `as_user_id` 相同的 `user_id` 值。
