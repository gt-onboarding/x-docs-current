<div id="overview">
  ## 概览
</div>

创意素材（Creatives）是指在广告活动中可以被推广的任何实体。帖子可以包含文本、图片、GIF、视频或卡片。卡片可以包含图片或视频。

图片、GIF 或视频类创意素材可通过 [POST media/upload](/zh/x-api/media/upload-media#post-media-upload) 上传——这是一个仅支持图片的简单上传端点——或者通过 POST media/upload（分片上传）端点上传。这些可以添加到

卡片：

* POST accounts/:account&#95;id/cards
  帖子：
* POST accounts/:account&#95;id/tweets - 要将卡片添加到帖子中，请使用 `card_uri` 参数。
  预定发布的帖子：
* POST accounts/:account&#95;id/scheduled&#95;tweets

如需了解卡片的更多详情，请参阅 Cards 页面。Promoted Video 页面提供了将视频与卡片或帖子关联的详细信息。

<div id="cards">
  ### 卡片
</div>

Ads API 支持多种可用于帖子中的卡片类型，这些帖子随后可以在广告活动中推广。**注意**：一旦帖子发布，卡片详情将会公开可见，其中可能包括卡片所属用户的相关信息。

<div id="image">
  #### 图片
</div>

以下图片规范适用于 [Cards](/zh/x-ads-api/creatives#cards-2) 中使用的素材。图片大小必须不超过 3MB，且宽度至少为 800px。此外，我们支持以下宽高比：

* 网站卡片：1:1 和 1.91:1
* 图片应用下载卡片：1:1 和 1.91:1
* 投票卡片：1.91:1
* 图片会话卡片：1.91:1
* 图片私信卡片：1.91:1

我们支持以下图片格式：.bmp、.jpeg 和 .png。

<div id="video">
  #### 视频
</div>

以下视频规格适用于在 [Cards](/zh/x-ads-api/creatives#cards-2) 中使用的素材。我们支持以下宽高比。

* 视频网站：16:9 和 1:1
* 视频应用下载：16:9 和 1:1
* 投票：16:9
* 视频对话：16:9
* 视频私信：16:9

<div id="promoted-video">
  ### 推广视频
</div>

本文档简要概述了通过 Ads API 上传和推广视频的流程。

Ads API 支持在[帖子](/zh/x-ads-api/creatives#tweets-2)以及以下卡片中使用 Promoted Video：

* [视频网站卡片](https://devcommunity.x.com/t/ads-api-version-11/168814)
* [视频应用下载卡片](https://devcommunity.x.com/t/ads-api-version-11/168814)
* [视频对话卡片](/zh/x-ads-api/creatives#video-conversation-cards)

首先，使用 [POST media/upload (chunked)](https://developer.x.com/en/docs/media/upload-media/api-reference/post-media-upload-init.html) 端点上传视频。然后使用 `media_id`，通过 POST accounts/:account&#95;id/videos 端点将该视频与广告账户关联。视频的 `id`（有时称为 `media_key`）将在后续请求中使用。该字符串以一个整数值开头，后接下划线，并以一个长整型数值结尾。例如：`13_875943225764098048`。

<div id="promoted-video-in-tweets">
  #### 帖子中的推广视频
</div>

要创建帖子，请使用 [POST accounts/:account&#95;id/tweet](/zh/x-ads-api/creatives#get-accounts-account-id-tweets) 端点，并提供该视频的 `id`。在此步骤中，你还可以提供视频标题、描述以及号召性用语（CTA）。这些值会直接展示给用户。

<div id="promoted-video-in-cards">
  #### 卡片中的推广视频
</div>

Video App Download 和 Video Conversation 卡片支持添加海报图像。使用 [POST media/upload](https://developer.x.com/en/docs/media/upload-media/api-reference/post-media-upload.html) 端点上传一张用于这些卡片的图像。

使用以下端点之一创建卡片：

* [POST accounts/:account&#95;id/cards/video&#95;website](https://devcommunity.x.com/t/ads-api-version-11/168814)
* [POST accounts/:account&#95;id/cards/video&#95;app&#95;download](https://devcommunity.x.com/t/ads-api-version-11/168814)
* [POST accounts/:account&#95;id/cards/video&#95;conversation](https://devcommunity.x.com/t/ads-api-version-11/168814)

并使用视频的 `id`，以及可选的图像 `media_id`（作为海报图像）。

最后，使用 POST accounts/:account&#95;id/tweet 端点创建帖子。通过 `card_uri` 参数将卡片附加到帖子上。

<div id="general-information">
  #### 一般信息
</div>

有关通过 API 上传视频的详细指南，请参阅[视频上传指南](https://developer.x.com/en/docs/media/upload-media/uploading-media.html)。

视频也可以作为前贴片素材进行推广。有关详细说明，请参阅[视频观看量前贴片目标指南](/zh/x-ads-api/campaign-management#video-views-preroll-objective)。

* （自 2015-10-22 起）在上传将用于推广内容的视频时，所有发往 [POST media/upload (chunked)](https://developer.x.com/en/docs/media/upload-media/api-reference/post-media-upload-init.html) 端点的 `INIT` 命令请求，其 `media_category` 参数必须设置为 `amplify_video`。使用该新参数可确保视频被异步预处理，并为在推广内容中使用做好准备。上传视频后，可以使用 `STATUS` 命令检查异步处理是否完成。
* 目前允许的推广视频最大长度为 10 分钟，且文件大小须不超过 500MB。
* 上传的视频应为 mp4 或 mov 格式。
* 上传的视频通常会较快完成处理，但处理时间可能会因视频长度和文件大小而有所不同。
* 上传的封面图像应为 png 或 jpg 格式。对长宽比或尺寸没有要求，但封面图像会调整以适配视频播放器。

### 定时发布帖子

<div id="introduction">
  #### 介绍
</div>

Scheduled Tweets 功能允许广告主或用户创建一个可以安排在未来某个时间发布的帖子。除了可以创建和管理这些帖子之外，API 还支持将这些帖子与广告单元（line item）关联，以便在帖子发布后进行推广。这样，广告主就可以提前创建原生帖子，并在任何关键活动开始前规划他们的活动素材。例如，可以预先设置一个帖子创意，使其在新产品发布的同时立即发布。

Scheduled Tweets API 端点提供的完整功能集如下所示：

* 创建、修改和查看新创建的预定帖子
* 将一个 Scheduled Tweet 与广告单元（line item）关联
* 查询和管理现有的预定帖子
* 一旦预定帖子发布，获取已发布帖子的 `id`

<div id="api-endpoints">
  #### API 端点
</div>

与上述功能相关的全部端点如下：

<div id="scheduled-tweet-management">
  #### 定时帖子管理
</div>

* [GET accounts/:account&#95;id/scheduled&#95;tweets](/zh/x-ads-api/creatives#get-accounts-account-id-scheduled-tweets)（检索所有定时帖子的列表）
* [GET accounts/:account&#95;id/scheduled&#95;tweets/:scheduled&#95;tweet&#95;id](/zh/x-ads-api/creatives#get-accounts-account-id-scheduled-tweets)（使用其 `id` 查询特定定时帖子）
* [POST accounts/:account&#95;id/scheduled&#95;tweets](/zh/x-ads-api/creatives#post-accounts-account-id-scheduled-tweets)（创建新的定时帖子）
* [PUT accounts/:account&#95;id/scheduled&#95;tweets/:scheduled&#95;tweet&#95;id](/zh/x-ads-api/creatives#example-request-39)（修改现有定时帖子）
* [DELETE accounts/:account&#95;id/scheduled&#95;tweets/:scheduled&#95;tweet&#95;id](/zh/x-ads-api/creatives#example-request-40)（使用其 `id` 删除定时帖子）
* [GET accounts/:account&#95;id/scheduled&#95;tweets/preview/:scheduled&#95;tweet&#95;id](/zh/x-ads-api/creatives#scheduled-tweets-2)（预览现有定时帖子）

<div id="scheduled-promoted-tweets">
  #### 预定推广帖子
</div>

* [GET accounts/:account&#95;id/scheduled&#95;promoted&#95;tweets](/zh/x-ads-api/campaign-management#get-accounts-account-id-scheduled-promoted-tweets)（获取所有预定推广帖子的列表）
* [GET accounts/:account&#95;id/scheduled&#95;promoted&#95;tweets/:scheduled&#95;promoted&#95;tweet&#95;id](/zh/x-ads-api/campaign-management#get-accounts-account-id-scheduled-promoted-tweets-scheduled-promoted-tweet-id)（根据其 `id` 查询预定推广帖子）
* [POST accounts/:account&#95;id/scheduled&#95;promoted&#95;tweets](/zh/x-ads-api/campaign-management#post-accounts-account-id-scheduled-promoted-tweets)（创建新的预定推广帖子）
* [DELETE accounts/:account&#95;id/scheduled&#95;promoted&#95;tweets/:scheduled&#95;promoted&#95;tweet&#95;id](/zh/x-ads-api/campaign-management#delete-accounts-account-id-scheduled-promoted-tweets-scheduled-promoted-tweet-id)（根据其 `id` 删除现有的预定推广帖子）

<div id="scheduled-tweet-view">
  #### 预定帖子视图
</div>

* [GET accounts/:account&#95;id/scheduled&#95;tweets/preview/:scheduled&#95;tweet&#95;id](/zh/x-ads-api/creatives#scheduled-tweets-2)（查看现有的预定帖子）

由于预定帖子作为独立于“实时”帖子的实体存在，对这些帖子进行创建或编辑时会运行两套不同的验证。第一套验证规则会在创建预定帖子的步骤中执行，具体包括：

<div id="scheduled-tweet-create">
  #### 定时帖子创建：
</div>

* 验证已认证用户是否有权限为指定的 @handle 创建自然帖子。要创建“仅推广”帖子，已认证用户必须是拥有 [帖子创作权限](https://developer.x.com/content/developer-twitter/en/docs/ads/campaign-management/api-reference/authenticated-user-access#get-accounts-account-id-authenticated-user-access) 的账号用户。
* 验证在距 `scheduled_at` 时间前后 15 分钟的时间窗口内，预定创建的帖子数量不超过 30 条。出现 SCHEDULED&#95;TWEET&#95;LIMIT&#95;EXCEEDED 错误消息表示在同一未来 15 分钟时间段内安排了过多的定时帖子。广告主需要删除一条现有的定时帖子，或将 `scheduled_at` 时间向前或向后调整。

<div id="scheduled-tweet-goes-live">
  #### 定时帖子发布：
</div>

* 这些验证规则会在 scheduled&#95;at 指定的时间点运行，并且与通过 API 创建常规帖子时应用的规则完全相同。例如，如果定时帖子同时包含图片和 GIF，则该定时帖子不会发布，其 scheduled&#95;status 将被设置为 FAILED。

<div id="workflow">
  #### 工作流
</div>

**创建新的预定帖子**

可以使用 [POST accounts/:account&#95;id/scheduled&#95;tweets](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/scheduled-tweets#post-accounts-account-id-scheduled-tweets) 端点创建新的预定帖子。此端点具有以下必需参数：`scheduled_at` 时间；如果帖子中未包含任何媒体实体，还必须提供帖子 `text`。此外，该端点还提供了一些附加选项，使你能够通过 `as_user_id` 参数代表其他 @handle 创建预定帖子，并可添加卡片（`card_uri`）和媒体（`media_ids`）。请注意，一个帖子只能包含同一类型的实体，即视频、GIF 或图片中的一种。`nullcast` 参数用于控制该帖子是否为“仅推广”帖子。所有新创建的预定帖子默认都是“仅推广”（`nullcast=true`）。如果 `nullcast=false`，则会创建一个自然（Organic）预定帖子。

一旦预定帖子创建成功，响应中将包含一个 `id` 字段，表示预定帖子本身的唯一标识符。除了此字段外，还会返回一个名为 `tweet_id` 的字段。该字段最初为 `null`，在帖子正式发布后，该字段会被填充为该“已发布”帖子的 ID。

```
twurl -H 'ads-api.x.com' -X POST "/6/accounts/:account_id/scheduled_tweets" -d 'scheduled_at=2017-12-24T23:59:00Z&text=Happy Holidays!'
```

这将创建如下计划发布的帖子：

```json
{
  "request": {
    "params": {
      "text": "Happy Holidays!",
      "scheduled_at": "2017-12-24T23:59:00Z"
    }
  },
  "data": {
    "completed_at": null,
    "id_str": "917507899668099072",
    "text": "Happy Holidays!",

    "tweet_id": null,

```

一旦此预定帖子发布上线，`tweet_id` 字段就会被填入该“实时”帖子的 id。

**将预定帖子关联到 line item（广告单元）**

虽然预定帖子可用于创建自然（Organic）帖子，我们也允许合作伙伴创建“仅推广”（Promoted-Only，`nullcast=true`）帖子，这两种类型都可以关联到某个 line item。为此，我们还提供了一个 [POST accounts/:account&#95;id/scheduled&#95;promoted&#95;tweets](https://developer.x.com/content/developer-twitter/en/docs/ads/campaign-management/api-reference/scheduled-promoted-tweets#post-accounts-account-id-scheduled-promoted-tweets) 端点。该端点在一次 API 调用中只允许将一条预定推广帖子关联到一个 line item。若要将多条预定帖子关联到同一个 line item，则需要进行多次 API 调用。

请注意，无法修改已经存在的预定推广帖子。

```
twurl -H 'ads-api.x.com' -X POST "/6/accounts/:account_id/scheduled_promoted_tweets" -d 'line_item_id=a44qc&scheduled_tweet_id=917507899668099072'
```

```json
{
  "data": {
    "line_item_id": "a44qc",
    "id": "26576",
    "created_at": "2017-10-09T22:24:16Z",
    "updated_at": "2017-10-09T22:24:16Z",
    "scheduled_tweet_id": "917507899668099072",
    "tweet_id": null,
    "deleted": false
  },
  "request": {
    "params": {
      "line_item_id": "a44qc",
      "scheduled_tweet_id": 917507899668099100,
      "account_id": "aaaaa"
    }
  }
}
```

此 endpoint 只是为指定的预定帖子（Scheduled Tweet）与某个 line item 创建一条关联。一旦 campaign/line item 的投放日期进入当前时间范围，该 line item 将自动开始投放对应的“上线”帖子。虽然在此步骤中我们会验证预定帖子是否处于 `SCHEDULED` 状态，以及该预定帖子是否适用于给定的投放目标，但不会执行其他校验。任何适用于该 line item 和预定帖子的其余校验规则，会在帖子实际“上线”时执行。

为确保广告投放不会出现问题，建议将预定帖子的 `scheduled_at` 时间设置在 campaign/line item 投放日期开始之前。

例如，假设预定帖子设置为在 campaign 开始日期之后才上线（并且只有一个帖子与一个 line item 关联），那么 campaign 将处于 `ACTIVE` 状态，但由于预定帖子尚未上线，将没有任何创意可用于投放。

**预定帖子管理（Scheduled Tweet Management）**

其余的一组 endpoints 允许 API 使用者管理其所有预定帖子（Scheduled Tweets）和预定推广帖子（Scheduled Promoted Tweets）。这些 API 可用于返回所有预定帖子的列表，可选地按给定状态进行筛选，也可用于通过 `id` 查询指定的预定帖子。

<div id="what-happens-when-a-scheduled-tweet-goes-live">
  #### 当已排程帖子正式发布时会发生什么？
</div>

当某条已排程帖子即将发布（也就是在 `scheduled_at` 时间点）时，将进行以下更新：

* 系统会创建对应的“正式发布”帖子，但这一步最多可能有 1 秒的延迟
* `tweet_id` 会被添加到以下实体中：
* 已排程帖子
* 推广的已排程帖子
* 会创建一个新的推广帖子实体

<div id="best-practices">
  #### 最佳实践
</div>

在创建或推广预定发布的帖子时，建议遵循以下最佳实践：

* 在创建预定发布的帖子时，确保该帖子是有效的（例如，一个帖子只能包含图片、视频或 GIF 三者之一，而不能是三者的任意组合）
* 确保广告系列的投放日期（即 `start_time` 和 `end_time`）与该预定发布帖子的 `scheduled_at` 时间保持一致
* 预定发布的帖子不应被安排在距离当前时间一年（365 天）之后
* 目前不支持预定发布帖子的预览功能（即在创建前预览预定发布帖子的功能）

### 媒体库

<div id="introduction">
  #### 介绍
</div>

Media Library 端点支持管理 X Ads 账户的图片、GIF 和视频。媒体库中的素材可用于帖子和卡片的创建，并且可以在多个[创意](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/overview)中重复使用，从而避免多次上传相同的素材。

<div id="api-endpoints">
  #### API 端点
</div>

* [POST media/upload](https://developer.x.com/content/developer-twitter/en/docs/media/upload-media/api-reference/post-media-upload#post-media-upload) 或 [POST media/upload (chunked)](https://developer.x.com/content/developer-twitter/en/docs/media/upload-media/api-reference/post-media-upload-init#post-media-upload-init)（上传媒体）
* [POST accounts/:account&#95;id/media&#95;library](https://developer.x.com/x-ads-api/creatives#get-accounts-account-id-media-library#post-accounts-account-id-media-library)（将媒体添加到媒体库）

<div id="adding-to-the-library">
  #### 添加到媒体库
</div>

将媒体添加到媒体库需要两个步骤。首先，使用 [POST media/upload](https://developer.x.com/content/developer-twitter/en/docs/media/upload-media/api-reference/post-media-upload#post-media-upload) 端点或 [POST media/upload (chunked)](https://developer.x.com/content/developer-twitter/en/docs/media/upload-media/api-reference/post-media-upload-init#post-media-upload-init) 这一组端点上传资源。（有关我们的分块上传流程的详细信息，请参阅 [Chunked media upload](https://developer.x.com/content/developer-twitter/en/docs/media/upload-media/uploading-media/chunked-media-upload) 指南。）

```
twurl -X POST -H upload.x.com "/1.1/media/upload.json?additional_owners=756201191646691328" --file latte.jpeg --file-field "media"
```

```json

   "media_id":966947208837742592,
   "media_id_string":"966947208837742592",
   "size":74194,
   "expires_after_secs":86400,
   "image":{
      "image_type":"image/jpeg",
      "w":800,
      "h":418

}
```

<div id="request-parameters">
  #### 请求参数
</div>

所有发往 Media Library 的 POST 请求都需要一个媒体标识符。该值会在上传步骤中返回。使用 media&#95;id（如上例所示）时，还必须指定一个 media&#95;category。media&#95;category 共有四种可能的取值：AMPLIFY&#95;VIDEO、TWEET&#95;GIF、TWEET&#95;IMAGE 和 TWEET&#95;VIDEO。

另外，可以为 Media Library 中的对象设置 name 和 file&#95;name 值。这些属性可帮助用户区分媒体库中同一媒体的不同变体。

对于视频，还可以设置标题和描述。这些值应作为 video&#95;title 和 video&#95;description 请求参数，与 [POST accounts/:account&#95;id/tweet](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/tweets#post-accounts-account-id-tweet) 端点一同传递。在帖子中，这些文本会显示在视频下方。

<div id="attributes">
  #### 属性
</div>

Media Library 中正式引入了 `media_key` 概念。它是媒体库中对象的唯一标识符。media key 是一种字符串形式的值，格式如下：13&#95;875943225764098048。在我们所有卡片相关的端点中均已完全支持使用这些 media key。

此外，Media Library 的响应还包含 `media_id`，以字符串形式表示。对于当前尚不接受 media key 的资源，将包含该字段：[帖子](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/tweets#post-accounts-account-id-tweet)*、[帖子预览](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/tweets#get-accounts-account-id-tweet-preview)* 和 [计划发布的帖子](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/scheduled-tweets#post-accounts-account-id-scheduled-tweets)。我们正致力于在所有资源中全面支持 media key。

`aspect_ratio` 属性会在 GIF 和视频的响应中返回。它可用于筛选媒体，以便用于只接受特定纵横比的卡片中。

*这些端点支持 `video_id` 参数，它是一个 media key。

<div id="identifying-cards">
  ### 识别 Card 卡片
</div>

<div id="introduction">
  #### 介绍
</div>

Card 是一种可自定义的广告格式，它使用媒体，并且可以与网站、应用或号召性用语（call to action）关联，以推动特定的用户互动，例如发起私信（Direct Message）。它们可以附加到帖子、计划发送的帖子或草稿帖子上。

在帖子对象中，可以通过两种方式引用 Card：使用 Card 的 `card_uri` 或其 `preview_url`。每种方式的示例值如下所示。

| card&#95;uri | preview&#95;url |
| :--- | :--- |
| card://1043282691834048513 | https://cards.x.com/cards/18ce54d4x5t/68w3s |

**注意**：从 Ads API 版本 3 开始，对于新创建的 Card，在 cards 响应中仅会生成并返回 `card_uri`。

**注意**：从 Ads API 版本 5 开始，cards 响应中将不再返回 `preview_url`。

在帖子对象的响应中，引用类型取决于创建该帖子的方式。换句话说，如果帖子是使用 `card_uri` 请求参数创建的，则响应中会出现该 `card_uri` 值。另一方面，如果在帖子文本中包含了 `preview_url`，则响应中会出现该预览 URL。

<div id="identifying-media">
  ### 媒体识别
</div>

#### 简介

媒体（图片、GIF 和视频）可以添加到帖子和卡片中。此外，视频可以用作前贴片广告素材，图片可以在 [X Audience Platform](https://developer.x.com/en/docs/ads/measurement/overview/twitter-audience-platform) 上进行推广。本节介绍如何在这些实体中查找媒体引用。

有两种类型的媒体标识符：ID 和 key。每种类型的示例值如下所示。

| **媒体 ID** | **媒体 key** |
| :--- | :--- |
| 1029825579531807971 | 13&#95;1029825579531807971 |

媒体 key 由 ID 前加上一个数字前缀和一个下划线构成。

<div id="image">
  #### 图片
</div>

下表显示了当前在每个与图片相关的资源的响应中可用的标识符类型，以及对应的属性名称。

| **Resource** | **Identifier** | **Attribute(s)** |
| :--- | :--- | :--- |
| Image cards | None |  |
| Tweet | Both | `entities["media"]["id_str"]`  `entities["media"]["media_key"]` |
| Scheduled Tweet | Both | `media_ids` and `media_keys` |
| Draft Tweet | Both | `media_ids` and `media_keys` |
| Account Media | None |  |
| Media Library | Both | `media_id` and `media_key` |

Image cards 和 Account Media 中的图片不包含对任何媒体标识符的引用。帖子（Tweet）只包含媒体 ID。计划发布和草稿帖子同时包含媒体 ID 和媒体 key。Media Library 也同时返回这两者。

对于帖子，`entities["media"]` 数组中对象的 `id` 和 `id_str` 字段对应于媒体 ID。对于包含多张图片的帖子，每个媒体实体的引用只能在 `extended_entities["media"]` 中找到。

除了标识符引用之外，通常还需要能够获取图片的 URL。

| **Resource** | **Attribute(s)** | **Format** |
| :--- | :--- | :--- |
| Image cards | `image` | .jpg |
| Tweet* | `entities["media"][0]["media_url"]` or `extended_entities["media"][i]["media_url"]` | .jpg |
| Scheduled Tweet | None |  |
| Draft Tweet | None |  |
| Account Media | `media_url` | .jpg |
| Media Library | `media_url` | .jpg |

* 该 URL 的位置取决于帖子是包含单张图片还是多张图片。

所有 image cards 都包含一个 `image` 响应属性，其中包含 X 图片 URL。（对于 image app download cards，此属性名为 `wide_app_image`。）

对于帖子，media URL 的位置取决于媒体类型以及所使用的端点。对于只包含一张图片的帖子，URL 可以在 `entities["media"][0]["media_url"]` 中找到，这对 Ads API 和 Standard API 都适用。但是，当帖子包含多张图片时，这些 URL 只能在 `extended_entities["media"][i]["media_url"]` 中找到。这仅在 Standard API 中可用。

<div id="video">
  #### 视频
</div>

下表显示了当前在各个与视频相关的资源响应中可用的标识符类型，以及对应的属性名称。

| **资源** | **标识符** | **属性** |
| :--- | :--- | :--- |
| 视频卡片 | 可能是任一类型 | `video_content_id` |
| 视频投票卡片 | 无 |  |
| 帖子 | 两者都有 | `entities["media"]["id_str"]`  `entities["media"]["media_key"]` |
| 计划发布的帖子 | 两者都有 | `media_ids` 和 `media_keys` |
| 草稿帖子 | 两者都有 | `media_ids` 和 `media_keys` |
| 账号媒体 | 媒体键 | `video_id` |
| 媒体库 | 两者都有 | `media_id` 和 `media_key` |

虽然视频卡片（带视频的投票卡片除外）包含一个 `video_content_id` 响应属性，但返回值的类型并不一致。有些情况下，它是媒体ID；在另一些情况下，它是媒体键。

关于如何访问视频URL的信息如下所示。

| **资源** | **属性** | **格式** |
| :--- | :--- | :--- |
| 视频卡片 | `video_url` 和 `video_hls_url` | .vmap  .m3u8 |
| 带视频的帖子 | `extended_entities["media"][i]["video_info"]["variants"][j]["url"]` | .mp4 |
| 计划发布的帖子 | 无 |  |
| 草稿帖子 | 无 |  |
| 账号媒体 | 无 |  |
| 媒体库 | `media_url` | .mp4 |

视频卡片包含 `video_url` 和 `video_hls_url` 响应属性，分别对应 .vmap 和 .m3u8 URL。

<div id="media-library">
  #### 媒体库
</div>

有时需要获取有关媒体资产的更多信息。比如，对于视频卡片，你可能需要获取 mp4 URL，而不是 vmap URL。这些信息可以在媒体库中获取。关于可用信息的详细内容，请参阅我们的[媒体库指南](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/guides/media-library)。大多数属于该广告账户的 FULL 级可推广用户的资产都可以在媒体库中找到，但也有一些例外。

**获取媒体**

如上所述，图片卡片不包含对媒体 id 或 media key 的引用。因此，无法通过媒体库获取其资产。[Account Media](https://developer.x.com/content/developer-twitter/en/docs/ads/creatives/api-reference/account-media#account-media) 中的图片也同样如此。

在创建视频卡片之前，要求其视频资产必须已经在媒体库中（或在其前身 Videos 资源中）。因此，这些资产始终可以在媒体库中检索到。Account Media 中的 PREROLL 资产也同样如此。

最后，帖子中的媒体始终保证存在于媒体库中。

下表总结了在考虑资源响应是否包含可用于查找的标识符时，哪些资产可以在媒体库中检索。

| **Resource** | **In the Media Library** |
| :--- | :--- |
| Image cards | No |
| Video cards | Yes* |
| Tweets (any media)** | Yes |
| Scheduled Tweets | Yes |
| Draft Tweets | Yes |
| Account media images | No |
| Account media videos (`PREROLL`) | Yes |

* 对于其 `video_content_id` 为 media key 的卡片。当该值是 media ID 时，资产仍然存在于媒体库中，但检索它需要在其前添加一个数字前缀和下划线。\
** 帖子只返回 media ID。虽然资产保证存在于媒体库中，但获取它需要在其前添加一个数字前缀和下划线。

**与 Account Media 的交互**

在以下两种情况下，添加到媒体库的媒体资产会自动添加到 Account Media 资源中。

* 当 AMPLIFY&#95;VIDEO 资产被添加到媒体库时，它会作为 PREROLL 创意类型的 Account Media 资产自动添加。
* 当具有特定尺寸的图片（参见我们[枚举页面](https://developer.x.com/content/developer-twitter/en/docs/ads/general/overview/enums)中的 “Creative Types”）被添加到媒体库时，它们会被自动添加为 Account Media 资产。创意类型（例如 INTERSTITIAL）取决于图片尺寸。

<div id="tweets">
  ### 帖子
</div>

#### 简介

X Ads API 支持三种类型的帖子：已发布、预定发布和草稿。

<div id="nullcasted-tweets">
  #### Nullcasted 帖子
</div>

帖子可以是仅推广（nullcasted，亦称“Promoted-only”）或自然的。仅推广帖子在发布后不会出现在用户的公开时间线上，但它们依然是公开的。另一方面，自然帖子会展示给用户的关注者，并会出现在用户的公开时间线上。

**创建帖子**

三个创建帖子的端点都支持一个布尔类型的 `nullcast` 参数，使 API 使用者可以选择创建仅推广或自然帖子。仅推广帖子可以由用户本人创建，也可以由任何被授权代替该用户创建帖子的主体创建。自然帖子只能由[完全可推广用户](https://developer.x.com/content/developer-twitter/en/docs/tutorials/ads-api-hierarchy-terminology#promotable-users)创建。

**更新帖子**

可以更新计划发布和草稿帖子的 `nullcast` 属性。对于计划发布的帖子，可以在该帖子的 `scheduled_at` 时间之前进行编辑。草稿帖子可以无限期编辑。但一旦发布，就无法将帖子从仅推广更改为自然，或反之。

<div id="promoting-tweets">
  #### 推广帖子
</div>

只有已发布和预定发布的帖子可以被推广。它们可以是 nullcasted 或自然（organic）的内容；对此没有限制。只要事先获得许可，广告主可以推广自己的帖子，也可以推广其他用户的帖子。（参见：[Promoting another user&#39;s Tweets](https://developer.x.com/content/developer-twitter/en/docs/tutorials/promoting-another-users-content) 了解更多信息。）

在单个广告系列中可以推广多条帖子。同样，一条帖子也可以在一个或多个广告系列中被推广。

要推广已发布的帖子，请使用 [POST accounts/:account&#95;id/promoted&#95;tweets endpoint](https://developer.x.com/content/developer-twitter/en/docs/ads/campaign-management/api-reference/promoted-tweets#post-accounts-account-id-promoted-tweets) 端点。这会将已发布的帖子与某个广告项目（line item）关联。要推广预定发布的帖子，请使用 [POST accounts/:account&#95;id/scheduled&#95;promoted&#95;tweets](https://developer.x.com/content/developer-twitter/en/docs/ads/campaign-management/api-reference/scheduled-promoted-tweets#post-accounts-account-id-scheduled-promoted-tweets) 端点。

<div id="tweet-ids">
  #### 帖子 ID
</div>

已发布、已排期和草稿帖子的 ID 是[数值型](https://developer.x.com/content/developer-twitter/en/docs/basics/twitter-ids)——它们是 64 位无符号整数。例如，下面这个已发布帖子的 ID 是 1166476031668015104。

<div id="carousels">
  ### 轮播
</div>

<div id="introduction">
  #### 介绍
</div>

X Ads API 支持创建和获取视频轮播和图片轮播。轮播是一种卡片类型，可以包含 2 到 6 个媒体素材。轮播卡片可以将用户引导至某个网站，或鼓励他们安装移动应用。有关轮播、其优势、最佳实践和常见问题的更多信息，请参阅我们的 [Carousel Ads on X](https://business.x.com/en/advertising/carousels.html) 页面。

轮播与任何其他卡片类型一样，可以用于帖子，这些帖子随后可以被推广。工作流程与您已经熟悉的流程相同：

1. 上传媒体
2. 创建卡片
3. 创建帖子
4. 推广帖子

唯一的区别在于卡片的创建方式。其他卡片创建请求接受查询参数，而轮播卡片创建请求只接受 JSON 格式的 POST 请求体。

<div id="endpoints">
  #### 端点
</div>

Ads API 支持创建和检索轮播卡片。

要创建任意类型的轮播卡片，请使用 [POST accounts/:account&#95;id/cards](https://developer.x.com/en/docs/twitter-ads-api/creatives/api-reference/cards#post-accounts-account-id-cards) 端点。要检索轮播卡片，请使用 [GET accounts/:account&#95;id/cards](https://developer.x.com/en/docs/twitter-ads-api/creatives/api-reference/cards#get-accounts-account-id-cards) 端点。

<div id="json-post-body">
  #### JSON POST Body
</div>

轮播是由两个组件创建的。第一个组件指定要使用的媒体资源，第二个组件指定与网站或应用相关的信息。

具体来说，一个轮播卡片按以下顺序由这些组件构成：

* 一个 `SWIPEABLE_MEDIA` 组件，它接收一个 media key 数组
* *以下组件中的一个*：
* 一个 `DETAILS` 组件，用于指定网站信息
* 一个 `BUTTON` 组件，用于指定应用信息

`SWIPEABLE_MEDIA` 组件必须包含一个 `media_keys` 数组，你可以在其中指定 2 到 6 个图片或视频。传入 media key 的顺序决定了它们的渲染顺序。

```
{
  "type": "SWIPEABLE_MEDIA",
  "media_keys": [
    "13_1089771253848666112",
    "13_1191948012077092867"
  ]
}
```

提醒一下，你可以通过向 [GET accounts/:account&#95;id/media&#95;library](https://developer.x.com/en/docs/twitter-ads-api/creatives/api-reference/media-library#get-accounts-account-id-media-library) 端点发起请求来获取 media key。

第二个组件对象的结构取决于你是希望将用户引导至网站，还是鼓励他们安装应用。下表对这两种选项进行了总结。（**注意**：表中列出的所有键都是必需的。）

|        | **网站**                                                             | **应用**                                            |
| :----- | :----------------------------------------------------------------- | :------------------------------------------------ |
| 指定组件类型 | `"type": "DETAILS"`                                                | `"type": "BUTTON"`                                |
| 标题/标签  | `"title": "X"`                                                     | `"label": { "type": "ENUM", "value": "INSTALL" }` |
| 目标     | `"destination": { "type": "WEBSITE", "url": "https://www.x.com" }` | `"destination": { "type": "APP", ... }`           |

综合以上设置，下面展示了一个网站轮播的 JSON POST 请求体示例。

```
{
  "name": "website carousel",
  "components": [
    {
      "type": "SWIPEABLE_MEDIA",
      "media_keys": [
        "13_1089771253848666112",
        "13_1191948012077092867"
      ]
    },
    {
      "type": "DETAILS",
      "title": "X",
      "destination": {
        "type": "WEBSITE",
        "url": "https://www.x.com"
      }
    }
  ]
}
```

`BUTTON` 组件中的应用目标对象需要一个国家代码以及至少一个应用标识符。它们还可以可选地包含深度链接。有关这些字段的说明，请参阅[参考文档](https://developer.x.com/en/docs/twitter-ads-api/creatives/api-reference/cards#app-destination)。

综合以上内容，下面展示了一个应用轮播的 JSON POST 请求正文示例。

```
{
  "name": "app carousel",
  "components": [
    {
      "type": "SWIPEABLE_MEDIA",
      "media_keys": [
        "13_1089771253848666112",
        "13_1191948012077092867"
      ]
    },
    {
      "type": "BUTTON",
      "label": {
        "type": "ENUM",
        "value": "INSTALL"
      },
      "destination": {
        "type": "APP",
        "country_code": "US",
        "googleplay_app_id": "com.twitter.android",
        "iphone_app_id": "333903271"
      }
    }
  ]
}
```

<div id="creative-metadata-tagging">
  ### 创意元数据标记
</div>

#### 简介

本指南面向创意合作伙伴、代理商和创意开发者，帮助你为 X 活动中使用的素材添加标签，从而更好地了解单个素材的价值和表现。

**注意：** 媒体素材只能由创建该媒体素材的合作伙伴或开发者添加标签。如果媒体素材的使用者并未创建该媒体素材，请勿进行元数据标记。

Creative Metadata Tagging 能够为由创意合作伙伴创建的图片和视频提供归因，而不受素材上传到 X 的具体位置或上传主体的限制。为了在创意素材与创意合作伙伴之间建立关联，我们采用了 [XMP](https://exiftool.org/TagNames/XMP.html) 标准。

<div id="tagging-creative-assets">

</div>

请按照提供的[ExifTool 安装说明](https://exiftool.org/install.html)进行操作。[Homebrew](https://formulae.brew.sh/) 也提供了软件包，通过提供适用于 macOS 和 Linux 的 [exiftool 安装命令](https://formulae.brew.sh/formula/exiftool)进一步简化安装流程。通过在命令行中输入 exiftool -ver 并查看返回的工具版本号，确认工具是否已正确安装。在 [ExifTool 文档](https://exiftool.org/) 中了解更多 ExifTool 命令参数的信息。

<div id="post-accountsaccount_idcards">
  #### POST accounts/:account&#95;id/cards
</div>

创建一个与指定账户关联的新卡片。

创建卡片的请求只接受 JSON 格式的 POST 请求体。`Content-Type` 必须设置为 `application/json`。

有关详细的使用示例，请参阅我们的[轮播指南](https://developer.x.com/en/docs/twitter-ads-api/creatives/guides/carousels)。

<div id="resource-url">
  #### 资源 URL
</div>

`https://ads-api.x.com/12/accounts/:account_id/cards`

#### 参数

JSON POST 请求正文必须包含卡片的 `name` 和一个 `components` 数组。组件以对象形式表示，用于描述该卡片面向广告主的属性。

下面的示例展示了有效负载的一般结构（其中的信息不可直接使用）。

```
{
  "name": "some name",
  "components": [
    {
      "type": "TYPE_ENUM",
      "key": "value"
    }
  ]
}
```

下面是关于组件的更多信息。

| Name                              | Description                                                                                                                                                                                                         |
| :-------------------------------- | :------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| account&#95;id   *required*       | 所使用账户的标识符。出现在资源路径中，并且通常是除 [GET accounts](https://developer.x.com/en/docs/ads/campaign-management/api-reference/accounts#get-accounts) 之外所有 Advertiser API 请求的必需参数。指定的账户必须与已通过身份验证的用户关联。 类型：string  示例：`18ce54d4x5t` |
| name   *required*                 | 卡片的名称。最大长度：80 个字符。 类型：string  示例：`component based card`                                                                                                                                                             |
| components   *sometimes required* | 描述用于创建卡片的组件。更多信息见下文。不能与 `slides` 同时指定。**注意**：组件的顺序很重要。 类型：array of objects                                                                                                                                          |
| slides   *sometimes required*     | 使用该数组的数组来创建 Multi-Destination Carousels。将每张卡片描述为组件的分组。每个 slide 都应是对一张卡片的完整表示。不能与 `components` 同时指定。**注意**：各个 slide 的顺序很重要。 类型：array of array                                                                        |

<div id="components">
  #### 组件
</div>

每个组件都必须包含一个 `type` 字段，用于确定对象的 schema。Ads API 支持以下组件类型，分为基于媒体和基于描述的组件。

* 媒体：
* `MEDIA`：单个视频或图片
* `SWIPEABLE_MEDIA`：2–6 个视频或图片
* 描述：
* `DETAILS`
* `BUTTON`

每个组件都有一组必需字段（除了 `type` 键之外）。这些字段列在下表中。

| 组件 `type`         | 字段                      | 值类型              |
| :---------------- | :---------------------- | :--------------- |
| `MEDIA`           | `media_key`             | string           |
| `SWIPEABLE_MEDIA` | `media_keys`            | array of strings |
| `DETAILS`         | `title`   `destination` | string   object  |
| `BUTTON`          | `label`   `destination` | object   object  |

下面是在 `components` 数组中的一个 `BUTTON` 组件示例（故意省略 `name` 键）。（省略号表示需要在这些位置补充更多信息。）

```
{
  "components": [
    {
      "type": "BUTTON",
      "label": {
        ...
      },
      "destination": {
        ...
      }
    }
  ]
}
```

指定组件对象的顺序决定了它们自上而下的渲染顺序。Cards 必须由一个基于媒体的组件以及一个 `DETAILS` 或 `BUTTON` 组件创建。基于描述的组件会渲染在媒体下方，并且具有关联的目标，要么是 URL，要么是移动应用。

**Label**

Label 定义显示在按钮上的文本，因此只适用于 `BUTTON` 组件。Label 对象有两个必填键：`type` 和 `value`。`type` 必须设置为 `ENUM`，而 `value` 可以是：`BOOK`、`CONNECT`、`INSTALL`、`OPEN`、`ORDER`、`PLAY` 或 `SHOP` 之一。

在前一个示例的基础上，下面展示了 `BUTTON` 组件中的 `label` 对象。

```
{
  "components": [
    {
      "type": "BUTTON",
      "label": {
        "type": "ENUM",
        "value": "INSTALL"
      },
      "destination": {
        ...
      }
    }
  ]
}
```

**目标地址**

目标地址是广告主希望将用户引导至的去向。在 `DETAILS` 或 `BUTTON` 组件中始终为必填项。目标地址类型有两种：`WEBSITE` 或 `APP`。

**注意**：网站目标地址只能与 `DETAILS` 组件一起使用，而 App 目标地址只能与 `BUTTON` 组件一起使用。

**网站目标地址**

| Name              | Description                                                                  |
| :---------------- | :--------------------------------------------------------------------------- |
| type   *required* | 目标地址的类型，用于决定其架构。 类型：enum  可能的取值：`WEBSITE`                                    |
| url   *required*  | 将用户重定向到的网站的 URL。 类型：string  示例：`https://devcommunity.x.com/c/advertiser-api` |

**App 目标地址**

| Name                                                        | Description                                                                                                                  |
| :---------------------------------------------------------- | :--------------------------------------------------------------------------------------------------------------------------- |
| type   *required*                                           | 目标地址的类型，用于决定其架构。 类型：enum  可能的取值：`APP`                                                                                        |

<div id="post-accountsaccount_idtweet">
  #### POST accounts/:account&#95;id/tweet
</div>

为该账户的完整可推广用户（默认）或 `as_user_id` 参数中指定的用户创建帖子。支持创建非公开投放（默认）和自然帖子。非公开投放帖子不会出现在公共时间线上，也不会推送给关注者。这两种类型都可以用于广告投放活动。

如果已认证用户不是此账户的 `FULL` 可推广用户，请通过调用 [GET accounts/:account&#95;id/authenticated&#95;user&#95;access](https://developer.x.com/en/docs/ads/campaign-management/api-reference/authenticated-user-access#get-accounts-account-id-authenticated-user-access) 端点来确定其是否有权限代表该用户发帖。权限值 `TWEET_COMPOSER` 表明该用户可以使用此端点代表 `FULL` [可推广用户](https://developer.x.com/en/docs/ads/campaign-management/api-reference/promotable-users#get-accounts-account-id-promotable-users) 创建非公开投放帖子。

当使用 [upload.x.com 端点](https://developer.x.com/rest/reference/post/media/upload) 处理媒体时，请将 `additional_owners` 参数中的 `user_id` 值设置为与调用此端点时传入的 `as_user_id` 相同的值。
