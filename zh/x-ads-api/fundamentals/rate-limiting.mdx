Advertiser API 的速率限制方式与 REST API v1.1 类似，相关文档请参阅：[REST API Rate Limiting in v1.1](https://developer.x.com/content/developer-twitter/en/docs/basics/rate-limits)。与 REST API v1.1 不同的是，这里没有提供按 endpoint 列出的可编程索引。各 endpoint 的速率限制和重置时间窗口通过 HTTP 响应头返回。Ads API 中的所有速率限制均使用 OAuth 1.0A。

<div id="user-level-and-ad-account-level-limits">
  ## 用户级别和广告账户级别限制
</div>

速率限制分为两种类型：用户令牌级别和广告账户级别。一部分端点启用了广告账户级别速率限制。用户令牌是你用于认证并调用 Ads API 的 [OAuth 访问令牌](/zh/resources/fundamentals/authentication#obtaining-access-tokens-using-3-legged-oauth-flow)。每个用户令牌可以访问一个或多个广告账户。

当响应头中返回了广告账户级别速率限制时，开发者应当使用广告账户级别速率限制；仅在找不到广告账户级别限制时才使用用户级别限制。

用户级别速率限制通过以下响应头体现：**x-rate-limit-limit**、**x-rate-limit-remaining** 和 **x-rate-limit-reset**

对于启用了广告账户级别速率限制的端点，其速率限制通过以下响应头体现：**x-account-rate-limit-limit**、**x-account-rate-limit-remaining** 和 **x-account-rate-limit-reset**。

广告账户级别速率限制仅适用于 GET 端点，以便应用在单个用户令牌可访问多个广告账户的场景下，同步实体数据（例如 campaign 或 line item 对象）。写入操作不保证使用相同的广告账户级别速率限制。

对于采用广告账户级别速率限制的端点，用户级别的速率限制会被设置为一个较高的值，用来表示整个应用的全局配额。在可用的情况下，应优先使用广告账户级别速率限制来控制你的请求量。

<div id="best-practices">
  ## 最佳实践
</div>

1. 在你的数据库中保存上次同步的时间戳，在请求数据时（在适用的情况下）使用 `sort_by=updated_at-desc` 选项来获取数据，这样当你处理到早于上次同步时间戳的数据时，就可以停止同步过程，从而避免对相同数据进行重复同步。

2. 在一次请求中获取多个实体：某些端点允许你指定一个用逗号分隔的值列表，以检索多条类似的数据。这可以减少你发起的调用总次数，从而更高效地利用速率限制。

3. 在请求中使用最大的 “count” 值：对于某些端点（例如 [GET accounts/:account&#95;id/targeting&#95;criteria](/zh/x-ads-api/campaign-management#targeting-criteria)），强烈建议使用最大 count 值进行调用，以返回 1000 个对象，而不是默认的 200 个。

<div id="faqs">
  ## 常见问题解答
</div>

*是否可以为某个广告账户或我们的应用提高速率限制？*

我们通常无法提高速率限制，这些限制的设定已经足以支持规模最大的广告账户。请首先落实本文件中列出的最佳实践；如果在此之后速率限制仍然影响到你扩展规模或实现业务目标的能力，请联系你的 X Ads API 对接人，并提供完整的用例说明和相关请求的详细信息。

<div id="scopes-for-this-doc">
  ### 本文档中的范围说明
</div>

* Category：所有属于该类别的端点在每个时间窗口内共享同一组分配的速率限制。
