<div id="typical-response-structure">
  ## 典型响应结构
</div>

成功的响应会以 200 系列 HTTP 状态码表示，并带有一个 JSON 负载，其中包含你所请求、创建、修改或删除的对象，以及服务器对该请求的理解结果。

如果你发出了成功的请求，你会在响应中收到一个 `request` 节点，用于回显你的请求。

*示例：* [GET accounts/abcdefg/campaigns?with&#95;deleted=true](/zh/x-ads-api/campaign-management#campaigns)

```json
{
  /* 您的响应数据... */,
  "request": {
    "params": {
      "account_id": "abcdefg",
      "with_deleted": "true"
    }
  }
}
```

JSON 响应中的 `data` 字段将包含与所访问资源关联的具体对象。当响应可能包含一个或多个结果时，`data` 节点的格式将是 JSON 数组；当响应中只可能有一个结果时，它将作为 JSON 哈希返回。在某些少见情况下，你可能会在通常应返回集合的响应中，看到只返回了一个哈希映射。在这种情况下，可以认为该单个哈希映射表示一个对象，其类型与 `type` 字段中指定的类型相同。

<div id="error-response-structure">
  ## 错误响应结构
</div>

错误响应会携带非 200 段的 HTTP 状态码。通常会附带 JSON 格式的响应体，但某些错误会返回不同类型的响应体。在无法解析响应结构的情况下，应优先依据 HTTP 状态码本身的含义进行判断。例如，你有时可能会看到返回 HTTP 404 并带有 HTML 响应体。在这种情况下，可以放心地认为请求的内容不存在（HTTP 404 的含义是 “Not Found”）。

典型的错误响应在结构上与成功响应类似。错误的具体信息会包含在响应的 `errors` 节点中。`errors/code` 节点会提供一个 CAPS&#95;CASE 格式的常量错误码，你可以在代码中使用它来决定如何处理和恢复。`errors/message` 节点通常会提供一段英文的人类可读错误描述。还可能附加其他字段，用于提供关于错误的更细粒度信息。

```json
{
  "errors": [
    {
      "parameter": "start_time",
      "details": "invalid date",
      "code": "INVALID_PARAMETER",
      "value": "",
      "message": "Expected time, got \"\" for start_time"
    }
  ],
  "request": {
    "params": {
      "account_id": "hkk5"
    }
  }
}
```

在上述示例中，向某个 analytics 端点发起的请求中使用了无效的 `start_time` 参数值。对于包含无效参数的请求，其 `errors/code` 为 `INVALID_PARAMETER`。
